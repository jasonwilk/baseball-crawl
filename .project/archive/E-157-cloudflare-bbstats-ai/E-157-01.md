# E-157-01: Production Docker Compose Configuration

## Epic
[E-157: Cloudflare Production Deployment for bbstats.ai](epic.md)

## Status
`DONE`

## Description
After this story is complete, the Docker Compose stack will be production-ready for bbstats.ai: Traefik routes traffic for both `baseball.localhost` (dev) and `bbstats.ai` (production), the Traefik dashboard is disabled by default (dev-only via override), cloudflared is pinned to a stable version, and `.env.example` documents all production-specific values for bbstats.ai.

## Context
The base docker-compose.yml currently includes dev-only features (Traefik insecure API, dashboard port) and a dev-only Host rule. For production deployment, the base config must be production-safe, with dev conveniences moved to the override file. The `.env.example` needs bbstats.ai-specific production values documented so an operator can configure the production `.env` without guessing.

## Acceptance Criteria
- [ ] **AC-1**: Traefik routes requests with `Host: bbstats.ai` to the app service, per TN-4 (dual-host rule). `Host: baseball.localhost` continues to work for local dev.
- [ ] **AC-2**: The base docker-compose.yml does not expose the Traefik dashboard, the Traefik host port (`8000:80`), or enable `--api.insecure`, per TN-5. Dev-only Traefik features (insecure API, dashboard port, app traffic host port) are in docker-compose.override.yml.example (and the existing docker-compose.override.yml if present). The override `command` array for traefik duplicates ALL base command flags plus dev-only additions (Docker Compose replaces `command`, does not merge -- see TN-5). The app service's `127.0.0.1:8001:8000` direct binding is preserved in the base docker-compose.yml (this is the devcontainer health-check path, not a Traefik port).
- [ ] **AC-3**: The cloudflared service image is pinned to a specific version tag (not `latest`), per TN-5.
- [ ] **AC-4**: `.env.example` documents all bbstats.ai production values per TN-6, including inline comments explaining each variable's production vs. dev behavior. The `CF_ACCESS_CLIENT_ID` / `CF_ACCESS_CLIENT_SECRET` variables are annotated as optional (not needed when CF Access is not enforcing auth). The `CLOUDFLARE_API_TOKEN` comment is updated per TN-7 to reflect it is for programmatic Cloudflare API use only (not referenced by `.mcp.json`, which uses OAuth).
- [ ] **AC-5**: All three compose files (docker-compose.yml, docker-compose.override.yml, docker-compose.override.yml.example) are syntactically valid YAML (verifiable by inspection or `python -c "import yaml, pathlib; yaml.safe_load(pathlib.Path('docker-compose.yml').read_text())"` -- note: `docker compose` commands are not available in the epic worktree per worktree isolation constraints).

## Technical Approach
The Traefik router label needs a dual-host OR condition so the same compose file works in both environments. Three items move from the base compose to the override: `--api.insecure=true`, the `8180:8080` dashboard port, and the `8000:80` app traffic port (in production, all traffic arrives via tunnel → Docker-internal traefik:80). The override file already exists for profiling cloudflared; it can be extended with the Traefik dev features. Pin cloudflared to the current stable release tag. Update `.env.example` comments and example values to reference bbstats.ai where the existing examples use `baseball.<your-domain>` or `baseball.localhost`. See TN-5 for the Docker Compose `command` replace-not-merge constraint -- the override must duplicate all base traefik command flags.
