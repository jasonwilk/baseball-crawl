# E-157-02: Production Deployment Documentation Update

## Epic
[E-157: Cloudflare Production Deployment for bbstats.ai](epic.md)

## Status
`DONE`

## Description
After this story is complete, both deployment documentation files (`docs/production-deployment.md` and `docs/cloudflare-access-setup.md`) will be updated for the bbstats.ai deployment: concrete domain values replace generic placeholders, the auth model section reflects no-CF-Access-blocking, the scoped API token is documented, and an operator can follow the runbook end-to-end to deploy at bbstats.ai.

## Context
The existing docs use `baseball.<your-domain>` as a placeholder throughout and describe a CF Access blocking model (policies, service tokens, email OTP). With bbstats.ai as the production domain and no CF Access enforcement, these docs need a targeted update. The CF setup guide also needs the scoped API token documented (purpose, permissions, rotation) and the tunnel naming convention (bbstats-ai). These are the ops docs that Jason (the operator) will follow on the production server.

## Acceptance Criteria
- [ ] **AC-1**: `docs/production-deployment.md` uses `bbstats.ai` as the concrete production domain throughout (no `<your-domain>` placeholders for the primary deployment). Generic instructions for alternative domains may remain as secondary notes.
- [ ] **AC-2**: `docs/production-deployment.md` Step 3 (Cloudflare Tunnel) references the tunnel name `bbstats-ai` and the public hostname configuration per TN-2. The `api.baseball` subdomain entry is removed from the public hostname table (single domain, no API subdomain for bbstats.ai). Step 3 includes a prominent callout (warning box or bold prerequisite) cross-referencing the mandatory pre-go-live policy removal step in `docs/cloudflare-access-setup.md` -- the operator must remove the two blocking CF Access policies before bringing the tunnel live, or all non-Jason traffic will be blocked. Step 2.3 (.env) lists production values per TN-6.
- [ ] **AC-3**: `docs/cloudflare-access-setup.md` documents the no-CF-Access-blocking auth model per TN-3. Includes an explicit **pre-go-live step**: remove the two existing blocking policies ("WARP Allow (jason@wilkinson.nu)" and "Google Auth Policy (jason@wilkinson.nu)") from the bbstats-ai Access app before the tunnel goes live. Explains why (CF Access enforces any active policy, blocking all non-matching traffic). Recommends preserving the Access app shell (policy-free) for future CF Email OTP adoption. The existing sections on Access policies and service tokens are retained as reference material but clearly marked as "not active for bbstats.ai" (or equivalent disambiguation).
- [ ] **AC-4**: The scoped API token is documented per TN-7: its permissions, zone scope, account scope (Jason's account only), distinction from both the tunnel token and the OAuth-based Cloudflare MCP, `.env` variable name (`CLOUDFLARE_API_TOKEN`), and token rotation guidance.
- [ ] **AC-5**: `docs/production-deployment.md` verification steps (Step 5) reference `https://bbstats.ai` and remove references to CF Access authentication on the health check or dashboard access.
- [ ] **AC-6**: If `mg.bbstats.ai` (or any new Mailgun sending domain) is referenced, the docs note that Mailgun DNS verification (SPF/DKIM TXT records) is a prerequisite before magic link emails will deliver. Silent delivery failure otherwise.
- [ ] **AC-7**: Both docs have an updated "Last updated" footer referencing E-157-02.

## Technical Approach
Targeted edits to both existing docs. Replace placeholder domains with `bbstats.ai`. Revise the CF Access sections to reflect the passive Access model. Add the scoped API token documentation. Update verification steps. Keep the docs as operator-facing runbooks -- concise, copy-pasteable commands, no agent jargon.
