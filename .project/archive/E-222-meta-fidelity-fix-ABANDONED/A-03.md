## Description
After this story is complete, `.claude/hooks/fidelity-write-guard.sh` will exist and be registered in `.claude/settings.json` as a PreToolUse hook on Write/Edit operations. The hook blocks PM from writing negative-assertion claims about consultations to epic files when unread relay messages are present in the active team inbox. The hook is coarse (cannot attribute a specific agent) and fails open on schema parse errors or missing inbox dirs.

## Context
The previous PM instance wrote "Expert consultations: none required for planning" despite two long expert relays sitting unread in its inbox. This hook is a mechanical forcing function that blocks that exact failure mode from recurring. Consumer Check 2 in D-01 is the semantic partner for deeper content verification; this hook is the syntactic blocker at the Write/Edit boundary. SE empirically verified the inbox schema during planning.

## Acceptance Criteria
- [ ] **AC-1**: `.claude/hooks/fidelity-write-guard.sh` exists and is approximately 50 lines of bash, modeled on the existing `.claude/hooks/worktree-guard.sh` pattern.
- [ ] **AC-2**: Hook runs as PreToolUse on Write/Edit tool calls. Reads JSON from stdin and extracts `tool_input.file_path` and `tool_input.new_content` (or the Edit-tool equivalent field).
- [ ] **AC-3**: Hook blocks only when ALL of the following are true:
  - `file_path` matches `epics/E-*/*.md` or `.project/archive/E-*/*.md`
  - Content matches regex `consultation(s)? not required|expert(s)? input not needed|no (expert )?consultations?`
  - At least one file under `/home/vscode/.claude/teams/*/inboxes/*.json` has `read: true` messages with `text` length > 500 chars AND `timestamp` within the last 30 minutes
  - Exactly one team directory is active
- [ ] **AC-4**: Hook fails OPEN on any of: JSON parse errors, missing inbox directory, missing teams directory, empty inbox files. (Write/Edit must succeed when the hook cannot verify state.)
- [ ] **AC-5**: Hook is registered in `.claude/settings.json` under PreToolUse handlers for Write and Edit.
- [ ] **AC-6**: A comment at the top of the hook documents the attribution limitation (in-process teammates share session; hook cannot attribute specific agent; Consumer Check 2 in D-01 is the semantic partner).
- [ ] **AC-7**: A comment at the top of the hook documents the platform fragility (inboxes path/schema are CC internals; fail-open preserves writes on version bumps).
- [ ] **AC-8**: Empirical test: removing the inboxes directory entirely does NOT block a test write to an epic file (fail-open behavior verified).

## Technical Approach

Model the hook on `.claude/hooks/worktree-guard.sh`. Structure:

1. Read JSON from stdin.
2. Parse `tool_input.file_path` and `tool_input.new_content` (or Edit `new_string`).
3. Check `file_path` against the epic-path regex; if not a match, exit 0 (allow).
4. Check content against the negative-assertion regex; if no match, exit 0 (allow).
5. Enumerate `/home/vscode/.claude/teams/*/inboxes/*.json`; if directory missing or no teams, exit 0 (fail-open).
6. If exactly one team directory is active, scan inbox JSON for unread (`read: true` per SE-verified schema — note: in CC inbox schema, `read: true` indicates a consumed-but-present relay; confirm exact field semantics before committing) messages with `text` length > 500 and `timestamp` within 30 min.
7. If a matching message exists, exit 2 with a stderr message explaining the block and pointing to `fidelity-protocol.md`; otherwise exit 0.

Register the hook in `.claude/settings.json` alongside existing PreToolUse handlers. Implementer reads both `worktree-guard.sh` and the current `settings.json` to identify the registration pattern.

**Cross-reference note on AC-3 inbox semantics**: SE empirically verified the schema during planning; implementer confirms by re-reading a live inbox file under `/home/vscode/.claude/teams/*/inboxes/` before committing the regex/field checks. If the schema does not match planning's assumption, escalate rather than guess.
