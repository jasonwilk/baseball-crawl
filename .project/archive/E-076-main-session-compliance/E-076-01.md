# E-076-01: Create Agent Team Compliance Rules File

## Epic
[E-076: Main Session Compliance Guardrails](epic.md)

## Status
`DONE`

## Description
After this story is complete, a new rules file at `.claude/rules/agent-team-compliance.md` will contain three pattern-action checkpoints that the main session (and all agents) must follow: (1) Explicit Team Request -- ensuring TeamCreate is used when the user names multiple agents, (2) Explicit Consultation Directive -- ensuring named agents are actually spawned when the user requests consultation, and (3) Anti-Fabrication Rule -- ensuring that when a spawned agent cannot reach a third agent, the main session spawns the missing agent rather than filling in the answer itself.

## Context
The main session has repeatedly violated explicit user requests to form teams and consult specific agents. Six prior epics (E-015, E-021, E-047, E-056, E-059, E-065) addressed PM-side compliance and dispatch workflow structure, but none added procedural checkpoints for the main session's own behavior during ad-hoc user requests. The pattern from E-029/E-047/E-059 is that prose rules fail while procedural checkpoints succeed. Architecture confirmed by claude-architect consultation during epic formation.

## Acceptance Criteria
- [ ] **AC-1**: File `.claude/rules/agent-team-compliance.md` exists with YAML frontmatter `paths: ["**"]` (loads for all agents on every interaction).
- [ ] **AC-2**: The file contains a **"Why These Rules Exist"** rationale section that includes both: (a) historical context explaining that six prior epics (E-015, E-021, E-047, E-056, E-059, E-065) attempted to fix this class of compliance failure by targeting PM behavior and dispatch mechanics, but the main session itself was never given procedural checkpoints, and (b) a reframing statement: when a user names specific agents, they are requesting those agents' judgment -- not any correct answer. The user may want a different perspective, domain expertise encoded in the agent's system prompt and memory, or a verifiable record that the named expert participated. Substituting your own answer -- even if correct -- violates the user's intent.
- [ ] **AC-3**: The file contains a mandatory preamble instruction (e.g., "User Request Classification -- execute before responding") that positions the pattern-matching at the decision point, before tool selection.
- [ ] **AC-4**: The file contains **Pattern 1: Explicit Team Request** with: (a) trigger -- user names 2+ agents/roles in a team-formation context, (b) required action -- use Agent Teams (TeamCreate + spawn each named agent), (c) prohibited -- spawning one agent and asking it to consult the others; using Task tool for individual spawns when a team was requested.
- [ ] **AC-5**: The file contains **Pattern 2: Explicit Consultation Directive** with: (a) trigger -- user says "consult [agent]", "work with [agent]", "ask [agent]", or names a specific agent as a required participant, (b) required action -- actually spawn that agent (via Agent Teams if Pattern 1 also applies, or Task tool otherwise), send the question, wait for the response, (c) prohibited -- answering on the agent's behalf, claiming to have consulted without spawning, paraphrasing what you think the agent would say.
- [ ] **AC-6**: The file contains **Pattern 3: Anti-Fabrication Rule** with: (a) trigger -- a spawned agent reports it cannot reach another agent (e.g., PM says "I can't spawn SE"), (b) required action -- the main session spawns the missing agent directly and relays the question (the main session has full spawning capability), (c) prohibited -- answering the question yourself, telling the requesting agent "I'll handle it" and then fabricating the answer.
- [ ] **AC-7**: All three patterns use the trigger/required/prohibited format consistently. Each pattern has all three components.
- [ ] **AC-8**: The file contains a brief **"Scope"** note stating that these patterns apply to explicit agent naming only -- not to domain-implied consultation (which is governed by the PM's advisory Consultation Triggers table). This prevents the patterns from becoming overly broad.

## Technical Approach
The implementing agent should create a new rules file following the conventions of existing files in `.claude/rules/` (YAML frontmatter with paths glob, markdown body). The three patterns address distinct failure modes and should be separate sections. The format (trigger/required action/prohibited) was recommended by claude-architect during consultation. Reference `/workspaces/baseball-crawl/.claude/rules/dispatch-pattern.md` as the closest analogue in structure and enforcement style, and `/workspaces/baseball-crawl/.claude/agent-memory/product-manager/lessons-learned.md` (lines 59-78) for the history of consultation compliance fixes.

## Notes
The trigger/required/prohibited format was recommended by claude-architect because: (1) it is pattern-matching (agent scans for specific phrases, not vague intent), (2) it provides dual guidance (what TO do and what NOT to do), (3) it is positional (preamble places it at the decision point before tool selection). This is the format that has worked best in the project's context-layer enforcement rules.

The example phrases in AC-5 ("consult [agent]", "work with [agent]", "ask [agent]") are illustrative, not exhaustive. The trigger definition in the rules file should use the broader framing ("names a specific agent as a required participant") as the primary definition, with the phrases as examples. A user saying "have the architect look at this" should also match.
