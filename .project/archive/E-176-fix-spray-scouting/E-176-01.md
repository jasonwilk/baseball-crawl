# E-176-01: Boxscore-UUID Fallback in Scouting Spray Crawler

## Epic
[E-176: Fix Spray Charts for Scouting Reports and Opponent Pipeline](epic.md)

## Status
`DONE`

## Description
After this story is complete, the `ScoutingSprayChartCrawler` will successfully crawl spray chart data for tracked opponents that have `gc_uuid=NULL`. When the database lookup returns NULL, the crawler will fall back to extracting per-game opponent UUIDs from cached boxscore files and use those UUIDs to call the spray endpoint. Since the endpoint returns both teams' data regardless of which team's UUID is in the path, this produces complete spray data for both teams in each game.

## Context
This is the core fix for the spray chart pipeline. Currently, `crawl_team()` calls `_lookup_gc_uuid(public_id)` and returns an empty `CrawlResult` if the result is NULL. The entire spray crawl is skipped for any tracked team without a `gc_uuid`. The boxscore-UUID fallback is a proven approach -- the report generator already implements it in `_build_boxscore_uuid_map()` (commit e802bae), but in the wrong place (inline in the generator). This story puts the fallback in the right place: the crawler itself.

## Acceptance Criteria
- [ ] **AC-1**: Given a tracked team with `gc_uuid=NULL` and cached boxscore files containing UUID keys, when `crawl_team()` is called for that team, then spray chart JSON files are written to `data/raw/{season_id}/scouting/{public_id}/spray/{event_id}.json` for each completed game that has a boxscore with a UUID key.
- [ ] **AC-2**: Given a tracked team with `gc_uuid=NULL` and NO cached boxscore files, when `crawl_team()` is called, then the crawler logs an INFO message and returns an empty `CrawlResult` (no error, no crash).
- [ ] **AC-3**: Given a tracked team with a valid `gc_uuid` in the database, when `crawl_team()` is called, then the existing direct-UUID path is used (no boxscore scanning). The fallback does not change behavior for teams that already have `gc_uuid`.
- [ ] **AC-4**: The boxscore-UUID extraction uses the UUID regex per TN-1 to distinguish UUID keys from `public_id` slug keys in boxscore JSON top-level keys.
- [ ] **AC-5**: `CredentialExpiredError` propagates immediately from the fallback path (same as the existing direct-UUID path).
- [ ] **AC-6**: Individual game API errors in the fallback path are caught, logged, and counted (same error handling pattern as the existing `_crawl_team_season` method).
- [ ] **AC-7**: Idempotency is preserved -- if a spray file already exists for a game, it is skipped regardless of whether the direct-UUID or fallback path is used.
- [ ] **AC-8**: New tests cover: (a) fallback path with boxscore UUID extraction, (b) fallback with no boxscores, (c) direct-UUID path unchanged, (d) mixed scenario where some games have boxscores and some don't.

## Technical Approach
The `crawl_team()` method currently returns early when `gc_uuid` is NULL. The change adds a fallback branch: when `gc_uuid` is NULL, scan the boxscore directory for the team, extract per-game opponent UUIDs per TN-1, and call the spray endpoint using those UUIDs. The per-game crawl loop reuses the existing `_crawl_team_season` pattern (or a shared helper) for file writing, error handling, and idempotency. The boxscore directory path follows the existing scouting layout at `data/raw/{season_id}/scouting/{public_id}/boxscores/`.
