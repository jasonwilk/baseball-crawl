# E-176-03: Opportunistic gc_uuid Resolution Cascade

## Epic
[E-176: Fix Spray Charts for Scouting Reports and Opponent Pipeline](epic.md)

## Status
`DONE`

## Description
After this story is complete, a new `gc_uuid_resolver` module will provide a resolution cascade that discovers and stores `gc_uuid` values for tracked teams using only cached data (member-team boxscores, opponents.json) plus a search API fallback. The resolver is called during the scouting pipeline so that `gc_uuid` is populated opportunistically, enabling the direct-UUID spray crawl path for future runs and eliminating the need for the boxscore-UUID fallback on subsequent crawls.

## Context
The boxscore-UUID fallback (Story 01) solves the immediate problem of crawling spray data without `gc_uuid`. But it requires scanning boxscores on every crawl run and makes per-game API calls with different UUIDs. Resolving `gc_uuid` once and storing it on the team row is more efficient long-term -- subsequent crawls use the direct path. The resolution cascade per TN-2 uses three tiers of decreasing reliability, and the first two tiers require zero API calls.

## Acceptance Criteria
- [ ] **AC-1**: A new module at `src/gamechanger/resolvers/gc_uuid_resolver.py` exports a function that accepts a team_id (int), public_id (str), team name (str), season_year (int|None), a DB connection, a data_root (Path), and an optional GameChangerClient. It returns the resolved `gc_uuid` (str) or None.
- [ ] **AC-2**: **Tier 1 -- member-team boxscores**: The resolver scans member team boxscore directories (`data/raw/*/teams/*/boxscores/*.json`). For each boxscore, it extracts the UUID key that does NOT match the member team's own `gc_uuid` (= the opponent's UUID per TN-1). It then cross-references with the `games` table: looks up the game by `games.game_id` (which stores the event_id value; the boxscore filename stem is the event_id) and verifies that the target tracked team's `teams.id` appears as `home_team_id` or `away_team_id`. When confirmed, the extracted opponent UUID is the target team's `gc_uuid`, which is returned and stored on the team row.
- [ ] **AC-3**: **Tier 2 -- progenitor_team_id**: The resolver scans `opponents.json` files (`data/raw/*/teams/*/opponents.json`) for entries where the `name` field matches the target team's name (case-insensitive). When `progenitor_team_id` is non-NULL on a matching entry, it is returned and stored as `gc_uuid`. Note: opponents.json entries contain `root_team_id`, `owning_team_id`, `name`, `is_hidden`, `progenitor_team_id` -- there is no `public_id` field, so matching is by name only.
- [ ] **AC-4**: **Tier 3 -- POST /search**: When the first two tiers fail, a `GameChangerClient` is provided, AND `season_year` is not None, the resolver strips classification suffixes ("Varsity", "JV", "Reserve", "Freshman", "Reserve/Freshman") from the team name and calls POST /search with the shortened name. A single unambiguous match for the correct season year is accepted. This tier is skipped when no client is provided OR when `season_year` is None (search results cannot be validated without a season year).
- [ ] **AC-5**: The resolved `gc_uuid` is stored using `UPDATE teams SET gc_uuid = ? WHERE id = ? AND gc_uuid IS NULL` -- never overwrites an existing value.
- [ ] **AC-6**: The cascade stops at the first successful tier. Tiers 1 and 2 make zero API calls. Only tier 3 makes an API call.
- [ ] **AC-7**: All tiers handle errors gracefully (log warning, continue to next tier or return None). No tier raises exceptions to the caller except `CredentialExpiredError` from tier 3.
- [ ] **AC-8**: Tests cover: (a) tier 1 success, (b) tier 2 success, (c) tier 3 success with shortened name, (d) all tiers fail returns None, (e) existing gc_uuid is never overwritten, (f) tier 3 skipped when no client provided, (g) tier 3 skipped when season_year is None.
- [ ] **AC-9**: The scouting CLI's `_scout_live` function in `src/cli/data.py` calls the resolver for tracked teams with `gc_uuid IS NULL` after the main scouting crawl/load completes and before the spray crawl stage begins.
- [ ] **AC-10**: A test verifies that `_scout_live` invokes the resolver at the correct pipeline stage (after main crawl/load, before spray crawl).

## Technical Approach
Create `src/gamechanger/resolvers/gc_uuid_resolver.py` (new `resolvers` subpackage with `__init__.py`). The main function implements the three-tier cascade per TN-2. Tier 1 iterates member team boxscore directories, reads JSON files, extracts UUID keys, and joins with the `games` table to find the target team. Tier 2 reads opponents.json files and matches by team name or public_id. Tier 3 reuses the search API pattern but with name-shortening logic. The scouting CLI (`_scout_live` in `src/cli/data.py`) should call the resolver for teams with `gc_uuid IS NULL` before the spray crawl stage.

## Notes
- The `_resolve_gc_uuid_via_search()` function in `src/reports/generator.py` (to be removed in Story 02) is a reference for the tier 3 search logic, but it uses exact name matching which fails for compound names. The new version must use shortened/stripped names.
- Classification suffixes to strip: "Varsity", "JV", "Freshman", "Reserve", "Reserve/Freshman", plus any trailing whitespace. The stripping should be iterative (e.g., "Lincoln Northeast Reserve/Freshman Rockets" -> try "Lincoln Northeast Rockets" first).
- The `resolvers` subpackage is new. Follow the existing package pattern (e.g., `src/gamechanger/crawlers/`, `src/gamechanger/loaders/`).
- **Timing prerequisite**: Tier 1 requires the `games` table to be populated with the target team's `teams.id` in `home_team_id` or `away_team_id`. This means the resolver must run AFTER the main scouting crawl/load (which creates game rows), not before. The scouting CLI (`_scout_live`) already has the correct ordering: main crawl/load → resolver → spray crawl.
