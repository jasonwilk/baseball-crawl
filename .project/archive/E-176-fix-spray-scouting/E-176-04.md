# E-176-04: Spray Pipeline Coverage for Direct-Add Teams

## Epic
[E-176: Fix Spray Charts for Scouting Reports and Opponent Pipeline](epic.md)

## Status
`DONE`

## Description
After this story is complete, `ScoutingSprayChartCrawler.crawl_all()` will discover and crawl spray data for ALL tracked teams with a `public_id`, not just those with an `opponent_links` entry. Teams added via the admin "generate report" flow (which creates a `teams` row but no `opponent_links` entry) will be included in `bb data scout` spray crawl runs.

## Context
The scouting spray crawler's `crawl_all()` method discovers teams via `opponent_links`. Teams added through the admin "generate report" flow bypass `opponent_links` entirely -- they get a `teams` row with `membership_type=tracked` but no link. These teams are invisible to the batch spray pipeline. See TN-4 for the query fix.

## Acceptance Criteria
- [ ] **AC-1**: Given a tracked team with `public_id` but no `opponent_links` row, when `crawl_all()` is called, then that team is included in the spray crawl (its spray files are fetched or skipped-as-cached like any other team).
- [ ] **AC-2**: Given a tracked team that IS in `opponent_links` with `is_hidden=1`, when `crawl_all()` is called, then that team is still excluded (hidden teams remain hidden regardless of the new query path).
- [ ] **AC-3**: Given a tracked team that appears in BOTH `opponent_links` and the `teams` table, when `crawl_all()` is called, then that team is processed exactly once (no duplicate crawls).
- [ ] **AC-4**: The query extension per TN-4 uses UNION to combine `opponent_links` public_ids with tracked teams that have no `opponent_links` entry, while excluding teams hidden via `is_hidden=1`.
- [ ] **AC-5**: Tests cover: (a) direct-add team included, (b) hidden opponent excluded, (c) no duplicate processing for teams in both sources, (d) existing opponent_links-only behavior unchanged.

## Technical Approach
Modify the SQL query in `crawl_all()` per TN-4. The current query selects from `opponent_links` only. Add a UNION branch that includes tracked teams not present in `opponent_links`. The second branch must also exclude teams that have an `opponent_links` row with `is_hidden=1` to preserve hidden-team filtering. UNION naturally deduplicates teams that appear in both sources.

## Notes
- `opponent_links` and `team_opponents` are two SEPARATE tables (both tables, neither is a view). `team_opponents` is the opponent relationship registry (`our_team_id` → `opponent_team_id`). `opponent_links` maps GC opponents endpoint entries to resolved teams and carries `is_hidden`, `public_id`, and `resolved_team_id`. The UNION's second branch must exclude teams that have an `opponent_links` row with `is_hidden=1` to preserve hidden-team filtering. See TN-4 for the full constraint.
- Member teams (`membership_type='member'`) are never included -- the WHERE clause filters to `tracked` only.
