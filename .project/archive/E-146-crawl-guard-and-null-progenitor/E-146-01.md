# E-146-01: Remove OpponentCrawler Phase 2 Roster Fetch

## Epic
[E-146: Crawl Pipeline Guards and Null-Progenitor Auto-Resolution](epic.md)

## Status
`DONE`

## Description
After this story is complete, `OpponentCrawler.crawl_all()` will no longer attempt to fetch rosters for opponent teams via the authenticated `GET /teams/{uuid}/players` endpoint. Phase 2 is removed entirely -- the scouting pipeline already handles opponent rosters via the public-path endpoint. The opponent crawl summary log is preserved with registry-level statistics.

## Context
The OpponentCrawler currently has two phases: Phase 1 fetches the opponent registry (needed), Phase 2 fetches rosters for each opponent via an authenticated endpoint (always 403s for non-owned teams). Phase 2 is entirely redundant with the scouting pipeline, which uses `GET /teams/public/{public_id}/players` (no ownership required). Every run wastes API calls and generates noisy warning logs. Per Technical Notes TN-1 in the epic.

## Acceptance Criteria
- [ ] **AC-1**: `OpponentCrawler.crawl_all()` no longer calls `RosterCrawler.crawl_team()` or `GET /teams/{uuid}/players` for any opponent team.
- [ ] **AC-2**: `OpponentCrawler.crawl_all()` still fetches and writes the opponent registry (`opponents.json`) for each member team (Phase 1 unchanged).
- [ ] **AC-3**: The opponent crawl produces a summary log line after Phase 1 with: total opponents, count with `progenitor_team_id`, count without, count hidden. (These counters existed in the removed Phase 2 method; they must be preserved in the Phase 1 loop or a new summary step.)
- [ ] **AC-4**: The `RosterCrawler` import and `self._roster_crawler` instantiation are removed from `OpponentCrawler` if no longer used.
- [ ] **AC-5**: Existing tests for `OpponentCrawler` are updated to reflect Phase 2 removal. New or updated tests confirm that `crawl_all()` does not make authenticated roster calls for non-owned teams.
- [ ] **AC-6**: The `CrawlResult` returned by `crawl_all()` reflects only Phase 1 counts (files_written/skipped/errors from registry fetches only).

## Technical Approach
Remove the Phase 2 code path from `OpponentCrawler`. The opponent registry fetch (Phase 1) is preserved. The summary statistics that were computed during Phase 2's iteration over opponents should be moved into Phase 1's iteration loop or a dedicated post-registry summary method, so the useful logging (progenitor counts, hidden counts) is not lost.
