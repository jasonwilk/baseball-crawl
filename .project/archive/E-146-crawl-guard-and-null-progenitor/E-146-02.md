## Description
After this story is complete, `OpponentResolver` will have a `resolve_unlinked()` method that attempts an experimental resolution of null-progenitor opponents via the followâ†’bridgeâ†’unfollow flow. `GameChangerClient` will gain `post()` and `delete()` methods to support the write operations. For each unlinked opponent with a `root_team_id`, the method follows the team, fetches the `public_id` via the bridge endpoint, stores the resolution, and unfollows. This is an experimental extension of the documented opponent resolution flow -- `docs/api/flows/opponent-resolution.md` describes null-progenitor opponents as requiring manual linking, and whether `root_team_id` works with follow/bridge endpoints is untested. Failures at any step are handled gracefully -- the opponent stays unlinked and the error is logged.

## Context
~14% of opponents have `progenitor_team_id: null` and cannot be auto-resolved through the standard chain. These opponents have `root_team_id` (a UUID that GC accepts in some team-scoped endpoints). The experimental flow: `POST /teams/{root_team_id}/follow` â†’ `GET /teams/{root_team_id}/public-team-profile-id` â†’ store `public_id` â†’ unfollow. Whether `root_team_id` works with follow/bridge is untested -- the code must be defensive. Per Technical Notes TN-2, TN-3, TN-5, TN-6 in the epic.

## Acceptance Criteria
- [ ] **AC-1**: `OpponentResolver.resolve_unlinked()` queries `opponent_links` for `SELECT DISTINCT root_team_id FROM opponent_links WHERE public_id IS NULL AND resolution_method IS NULL AND is_hidden = 0` -- one follow/bridge/unfollow cycle per distinct `root_team_id`, not per row.
- [ ] **AC-2**: `GameChangerClient` gains `post()` and `delete()` methods that follow the same pattern as `get()` (auth headers via `_ensure_access_token()`, error hierarchy). `post()` must handle 204 No Content as success (no JSON parsing). `delete()` must handle both 204 No Content (no body) and 200 with text body (e.g., `"OK"`) as success. Both methods are needed for the follow (POST) and unfollow (DELETE) calls.
- [ ] **AC-3**: For each eligible opponent, the method attempts the followâ†’bridgeâ†’unfollow cycle using `root_team_id` as the team UUID:
  - Follow: `POST /teams/{root_team_id}/follow` (expects 204)
  - Bridge: `GET /teams/{root_team_id}/public-team-profile-id` (expects `{"id": "<public_id>"}`)
  - Unfollow: `DELETE /teams/{root_team_id}/users/{user_id}` then `DELETE /me/relationship-requests/{root_team_id}` (best-effort)
- [ ] **AC-4**: On successful bridge, ALL `opponent_links` rows sharing that `root_team_id` (across multiple `our_team_id` values) are updated with the obtained `public_id`, `resolution_method='follow-bridge'`, and `resolved_at`. This is a fan-out UPDATE (not an INSERT), since the rows already exist from the initial `resolve()` pass. No `teams` row is created -- the scouting pipeline's `_ensure_team_row(public_id=...)` handles team row creation when it first encounters the opponent during `scout_all()`.
- [ ] **AC-5**: If the follow step fails (non-204 response, network error), the opponent is skipped and the error is logged at WARNING level. No unfollow is attempted.
- [ ] **AC-6**: If the bridge step fails (403, error, missing `id` field), the error is logged and the method proceeds to unfollow (cleanup) before continuing to the next opponent.
- [ ] **AC-7**: If the unfollow step fails, a WARNING is logged but the overall resolution is not marked as failed (the `public_id` was already stored successfully).
- [ ] **AC-8**: A minimum 2-second delay is applied between each follow/bridge/unfollow cycle per Technical Notes TN-2.
- [ ] **AC-9**: The authenticated user's UUID for the unfollow step is fetched once at the start of `resolve_unlinked()` via `GET /me/user` per Technical Notes TN-5.
- [ ] **AC-10**: `resolve_unlinked()` returns a `ResolveResult` with counts per distinct `root_team_id` attempt (not per `opponent_links` row): `resolved` for successful follow-bridge resolutions, `errors` for unexpected exceptions. A new field `follow_bridge_failed: int` is added to `ResolveResult` for opponents that were attempted but failed at the follow or bridge step. (The existing `unlinked` field retains its original meaning of "no progenitor_team_id, stored as-is" and is not reused here.)
- [ ] **AC-11**: Manual links (`resolution_method='manual'`) are never overwritten -- the fan-out UPDATE's WHERE clause excludes rows where `resolution_method IS NOT NULL` (manual and auto-resolved rows are already resolved).
- [ ] **AC-12**: Tests cover: `post()` and `delete()` on `GameChangerClient` (auth headers, error handling), successful full cycle with fan-out to multiple `our_team_id` rows (mock all API endpoints), follow failure (skip gracefully), bridge failure (unfollow attempted), unfollow failure (resolution still counted as success), manual link protection (UPDATE skips manual rows).

## Technical Approach
Add `post()` and `delete()` methods to `GameChangerClient`, following the same auth/retry/error pattern as `get()`. Then add `resolve_unlinked()` as a new method on the existing `OpponentResolver` class. The `ResolveResult` dataclass gets one new field (`follow_bridge_failed`). The user ID for unfollow is fetched from `GET /me/user` once at method entry.

On successful bridge, store `public_id` via a simple fan-out UPDATE on `opponent_links` (not the INSERT-based `_upsert_resolved()`), because the rows already exist and the update must touch ALL rows sharing a `root_team_id`. No `teams` row is created by this method -- the scouting pipeline handles that downstream via `_ensure_team_row(public_id=...)`.

The API endpoints called by this method are already documented in `docs/api/endpoints/`: `post-teams-team_id-follow.md`, `get-teams-team_id-public-team-profile-id.md`, `delete-teams-team_id-users-user_id.md`, `delete-me-relationship-requests-team_id.md`.
