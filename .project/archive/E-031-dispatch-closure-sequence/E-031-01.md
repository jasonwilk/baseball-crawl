# E-031-01: Expand dispatch closure in rules and PM definition

## Epic
[E-031: Dispatch Closure Sequence](../E-031-dispatch-closure-sequence/epic.md)

## Status
`DONE`

## Description
After this story is complete, both `/.claude/rules/dispatch-pattern.md` and `/.claude/agents/product-manager.md` will contain an expanded dispatch closure sequence that mandates validation, status updates, archiving, user summary, and commit offer before and after team shutdown. The two files will be consistent with each other and with the canonical closure sequence defined in the epic's Technical Notes.

## Context
The dispatch closure step is currently a one-liner in both files. This has led to incomplete closures where the PM spins down the team without presenting a summary, or without explicitly archiving the epic. The epic's Technical Notes contain the canonical closure sequence that must be reflected in both files.

## Acceptance Criteria
- [ ] **AC-1**: `/.claude/rules/dispatch-pattern.md` "The Dispatch Flow" replaces its current step 9 ("When all stories are done, PM marks the epic `COMPLETED`, shuts down teammates, and deletes the team.") with a multi-step closure sequence covering: (a) validate all ACs, (b) update all statuses and epic History, (c) archive the epic directory, (d) update PM memory, (e) review ideas backlog, (f) present summary to user, (g) after team shutdown offer to scan and commit.
- [ ] **AC-2**: `/.claude/agents/product-manager.md` Dispatch Procedure step 11 ("Close. When all stories are done, follow the 'Completing an epic' checklist in the Atomic Status Update Protocol. Then shut down teammates and delete the team.") is expanded to describe the same closure sequence, referencing the Atomic Status Update Protocol where appropriate but adding the user summary and commit offer steps explicitly.
- [ ] **AC-3**: Both files describe the same closure steps in the same order. The dispatch-pattern.md version may be more concise (it is a shared reference); the product-manager.md version may include more PM-specific detail (it is the PM's own procedure). But neither omits a step that the other includes.
- [ ] **AC-4**: The closure sequence in both files explicitly notes that the PM has no Bash tool and must request the archive directory move from an implementing agent still on the team before shutting them down.
- [ ] **AC-5**: The closure sequence in both files explicitly states that commit must NOT happen automatically -- the user must approve.
- [ ] **AC-6**: No other sections of either file are changed beyond what is necessary to support the expanded closure sequence (e.g., if a numbered list needs renumbering, that is fine, but do not rewrite unrelated sections).

## Technical Approach
1. Read the epic's Technical Notes for the canonical closure sequence.
2. Read `/.claude/rules/dispatch-pattern.md` and identify step 9 in "The Dispatch Flow" section.
3. Replace step 9 with an expanded closure sequence (steps 9a through 9g or a sub-list). Renumber subsequent steps if needed (there are no steps after 9 currently).
4. Read `/.claude/agents/product-manager.md` and identify step 11 in the "Dispatch Procedure" section.
5. Replace step 11 with an expanded closure sequence. Reference the Atomic Status Update Protocol's "Completing an epic" checklist for the status-update steps, but add the user summary (step 6 in the canonical sequence) and commit offer (step 7) explicitly.
6. Cross-check both files to ensure consistency.
