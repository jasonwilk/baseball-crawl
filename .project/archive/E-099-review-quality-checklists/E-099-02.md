# E-099-02: Add Error-Path Testing Requirement to testing.md

## Epic
[E-099: Targeted Review Quality Checklists](epic.md)

## Status
`DONE`

## Description
After this story is complete, `.claude/rules/testing.md` will contain an explicit "Error-Path Testing" section that requires tests for failure paths in orchestration code (CLI commands, pipeline runners, loaders). This makes error-path coverage a documented project convention rather than an implicit expectation, enabling the code-reviewer to flag missing error-path tests as MUST FIX convention violations.

## Context
E-097's CLI tests covered the happy path and ACs (dry-run, --team flag) but no test exercised what happens when `loader.load_team()` fails. The testing rules say "include edge cases: empty data, malformed input, missing fields" but don't mention error-path testing for orchestration code -- the scenario where a called function fails and the caller must propagate that failure correctly. Adding this to `testing.md` makes it a documented convention that both SE and CR can reference.

## Acceptance Criteria
- [ ] **AC-1**: `testing.md` contains a new "Error-Path Testing" section after the existing "Test Scope Discovery" section.
- [ ] **AC-2**: The section states the requirement: when code calls a function that can fail (returns errors, raises exceptions, or returns a status indicating failure), write at least one test where that function fails, and verify the caller handles the failure correctly (propagates error, sets appropriate status, returns non-zero exit code, does not print misleading success messages).
- [ ] **AC-3**: The section calls out orchestration code (CLI commands, pipeline runners) as the primary scope -- code where multiple steps chain together and a failure in one step must be visible to the operator.
- [ ] **AC-4**: The section includes a concrete example pattern (mock the dependency to raise or return a failure indicator, assert exit code and output).
- [ ] **AC-5**: No other sections of `testing.md` are changed. Existing content (test scope discovery, subprocess smoke tests, general testing rules) is preserved exactly.

## Technical Approach
Read `/.claude/rules/testing.md`. Add a new "Error-Path Testing" section after the "Test Scope Discovery" section. Keep it concise -- one requirement statement, one scope note (orchestration code), one example pattern. Reference the epic Technical Notes for the specific bug class definition.
