# E-048-01: Create proxy/ infrastructure

## Epic
[E-048: Migrate mitmproxy to Host-Based Standalone Proxy](epic.md)

## Status
`DONE`

## Description
After this story is complete, the `proxy/` folder will be a fully self-contained mitmproxy deployment that runs on the Mac host. It will have its own `docker-compose.yml`, environment configuration, control scripts, gitignore, directory scaffolding, and a README with complete setup instructions. The operator can `cd proxy && ./start.sh` to launch the proxy.

## Context
The proxy currently runs inside the devcontainer's Docker-in-Docker stack via a `--profile proxy` flag in the project's `docker-compose.yml`. This causes networking problems because ports must be forwarded through VS Code to reach the Mac host. Running the proxy directly on the Mac host eliminates this entirely. A working prototype exists at `proxy-host/` -- use it as a reference but build the final version fresh in `proxy/`.

## Acceptance Criteria
- [ ] **AC-1**: `proxy/docker-compose.yml` defines a single `mitmproxy` service using the `mitmproxy/mitmproxy` image with `mitmweb` entrypoint. Ports: `0.0.0.0:8080:8080` (proxy, all interfaces for iPhone), `127.0.0.1:8081:8081` (mitmweb UI, localhost only). Volumes: `./certs:/home/mitmproxy/.mitmproxy` (CA cert persistence), `..:/app` (project root for addon imports and .env writes). Environment: `PYTHONPATH=/app`. Scripts loaded via `--scripts /app/proxy/addons/loader.py`. Web host set to `0.0.0.0`.
- [ ] **AC-2**: `proxy/docker-compose.yml` passes `MITMWEB_PASSWORD` from the environment to the container (via `env_file: .env`). The mitmweb entrypoint command includes `--set web_password=${MITMWEB_PASSWORD:-}`. When `MITMWEB_PASSWORD` is set in `proxy/.env`, mitmweb requires that password for UI access. When unset or empty, the empty string triggers mitmweb's default random-token behavior (token URL printed to logs).
- [ ] **AC-3**: `proxy/.env.example` documents `MITMWEB_PASSWORD` with a comment explaining it is optional (mitmweb generates a token-based URL if unset).
- [ ] **AC-4**: `proxy/.gitignore` ignores `certs/`, `data/`, `.env`, and `__pycache__/`.
- [ ] **AC-5**: `proxy/certs/.gitkeep` and `proxy/data/.gitkeep` exist so the directories are tracked but their contents are ignored.
- [ ] **AC-6**: `proxy/start.sh` is executable, `cd`s to its own directory, creates `certs/` if missing, runs `docker compose up -d`, and prints: proxy address, mitmweb UI URL, LAN IP (detected via `ipconfig getifaddr en0` on macOS, with fallback), iPhone setup steps, and a note about `host.docker.internal:8080` for devcontainer access.
- [ ] **AC-7**: `proxy/stop.sh` is executable, `cd`s to its own directory, runs `docker compose down`, and prints confirmation.
- [ ] **AC-8**: `proxy/status.sh` is executable, `cd`s to its own directory, runs `docker compose ps`, and shows port 8080 listeners via `lsof`.
- [ ] **AC-9**: `proxy/logs.sh` is executable, `cd`s to its own directory, runs `docker compose logs -f mitmproxy`.
- [ ] **AC-10**: `proxy/README.md` documents: what the proxy does, how to start/stop/check status, iPhone setup (proxy config + CA cert install + trust toggle), how the devcontainer reaches the proxy via `host.docker.internal:8080`, where data goes (`proxy/data/`), where certs persist (`proxy/certs/`), and that credentials are written to the project root `.env`. Include a troubleshooting section covering: port 8080 in use, iPhone can't connect, certificate not trusted.
- [ ] **AC-11**: From inside the devcontainer (with the proxy running on the Mac host), `curl -sx http://host.docker.internal:8080 http://mitm.it` returns a non-empty HTML response (the mitmproxy certificate download page). This validates the devcontainer-to-host-proxy network path that the entire architecture depends on.

## Notes
- The `..:/app` volume mount in docker-compose.yml means the proxy compose file must be run from the `proxy/` directory (or with `-f proxy/docker-compose.yml` and adjusted paths). The scripts handle this by `cd`-ing to their own directory.
- The `proxy/.env` is separate from the project root `.env`. The proxy `.env` only contains proxy-specific config (like MITMWEB_PASSWORD). The credential extractor writes to the project root `.env` via the `/app/.env` mount path.
- The `mitmproxy/mitmproxy` Docker image ships Python 3.14 while the devcontainer runs 3.13. Addons execute under 3.14 at runtime. Low practical risk since all addon code uses stdlib only, but `proxy/README.md` should document this version gap so operators are aware.
