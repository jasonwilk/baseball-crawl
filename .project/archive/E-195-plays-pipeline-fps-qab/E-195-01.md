# E-195-01: Plays Crawler

## Epic
[E-195: Plays Data Ingestion Pipeline -- FPS% and QAB](epic.md)

## Status
`DONE`

## Description
After this story is complete, a `PlaysCrawler` will fetch and cache raw plays JSON for every completed game across all configured member teams. The crawler reads game-summaries files (already written by `ScheduleCrawler`) to discover `event_id` values, fetches the plays endpoint for each completed game not already cached, and writes raw JSON to disk following the existing crawler pattern.

## Context
This is the data acquisition layer for the plays pipeline. The plays endpoint is CONFIRMED and not ownership-gated -- it works for any game with a known `event_id`. The crawler follows the same pattern as `GameStatsCrawler` (read game-summaries, filter to completed games, fetch per-game data from `/game-stream-processing/{event_id}/...`, skip if cached). The raw JSON on disk serves as the input for the parser and loader in subsequent stories.

## Acceptance Criteria
- [ ] **AC-1**: Given a team with completed games in game-summaries, when `PlaysCrawler.crawl_all()` runs, then a JSON file is written to `data/raw/{season}/teams/{gc_uuid}/plays/{event_id}.json` for each completed game not already cached.
- [ ] **AC-2**: Given a plays JSON file already exists on disk for a game, when the crawler runs again, then that game is skipped (existence-only idempotency check, matching existing crawler patterns).
- [ ] **AC-3**: Given a game-summaries file is missing for a team, when the crawler runs, then the team is skipped with a warning log (not a crash).
- [ ] **AC-4**: Given an API error on an individual game fetch, when the crawler encounters it, then the error is logged and counted but crawling continues for remaining games. `CredentialExpiredError` propagates (not caught).
- [ ] **AC-5**: The crawler returns a `CrawlResult` with accurate `files_written`, `files_skipped`, and `errors` counts.
- [ ] **AC-6**: The Accept header matches the plays endpoint spec per `docs/api/endpoints/get-game-stream-processing-event_id-plays.md`: `application/vnd.gc.com.event_plays+json; version=0.0.0`.
- [ ] **AC-7**: The path parameter uses `event_id` (from game-summaries top-level field), per TN-6. Never `game_stream.id`.
- [ ] **AC-8**: Unit tests cover: successful crawl with file write, skip-if-cached, missing game-summaries handling, API error handling. Tests mock HTTP -- no real API calls.

## Technical Approach
Follow the `GameStatsCrawler` pattern in `src/gamechanger/crawlers/game_stats.py` -- the closest existing crawler since it also targets the `/game-stream-processing/{event_id}/...` endpoint structure. The crawler reads game-summaries JSON files from disk, filters to completed games, and fetches plays data per game. Key difference: the Accept header is `application/vnd.gc.com.event_plays+json; version=0.0.0`. Reference `docs/api/flows/plays-ingestion.md` Sections 2-3 for endpoint details and ID chain.
