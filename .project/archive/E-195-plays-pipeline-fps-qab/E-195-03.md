## Description
After this story is complete, a `PlaysLoader` will read cached plays JSON files from disk, parse them via `PlaysParser`, and write the resulting play and event records to the `plays` and `play_events` database tables. The loader handles player stub creation for unknown player IDs, whole-game idempotency, game FK validation, and per-game error isolation.

## Context
The loader is a thin orchestration layer between the parser (E-195-02) and the database. It follows the existing loader pattern (e.g., `GameLoader.load_all()`): iterate over cached JSON files, parse each, write to DB. The parser does the heavy lifting; the loader handles DB mechanics (transactions, FK safety, idempotency). The plays JSON files are named by `event_id`, and `event_id = games.game_id` (the same ID used as the `games` table primary key).

## Acceptance Criteria
- [ ] **AC-1**: Given cached plays JSON files on disk and a database with the plays/play_events schema, when `PlaysLoader.load_all(team_dir: Path)` runs, then parsed play records and their events are inserted into the `plays` and `play_events` tables.
- [ ] **AC-2**: Whole-game idempotency per TN-7: before processing a game, the loader checks if any `plays` row exists for that `game_id`. If yes, the entire game is skipped. Re-running the loader for already-loaded games produces zero new rows.
- [ ] **AC-3**: Unknown `batter_id` or `pitcher_id` values that do not exist in the `players` table get a stub player row inserted before the play row, per the FK-safe orphan handling pattern documented in CLAUDE.md.
- [ ] **AC-4**: Given a parse error on an individual game file, when the loader encounters it, then the error is logged, the game is skipped, and loading continues for remaining games. The loader does not crash on a single bad file.
- [ ] **AC-5**: The loader processes games within a single DB transaction per game (all plays and events for one game commit together or roll back together).
- [ ] **AC-6**: The loader returns a `LoadResult` (from `src/gamechanger/loaders/__init__.py`) using the existing 3-field contract: `loaded` (plays inserted), `skipped` (games skipped -- already loaded or missing FK), `errors` (games with parse/insert errors).
- [ ] **AC-7**: Game FK guard per TN-7: the loader verifies that each `game_id` exists in the `games` table before inserting plays. Games not yet loaded by the game loader are skipped with a warning log.
- [ ] **AC-8**: Unit tests cover: successful load with DB verification, whole-game idempotent re-load (zero new rows), stub player creation for unknown IDs, parse error isolation, game FK guard (skip when game not in games table). Tests use an in-memory SQLite database with migration 009 applied.

## Technical Approach
Follow the `GameLoader` pattern in `src/gamechanger/loaders/game_loader.py` for the `load_all()` method structure. The loader iterates over `data/raw/{season}/teams/{gc_uuid}/plays/{event_id}.json` files, calls `PlaysParser.parse_game()` on each, and writes results to DB. For play_events, the loader must first insert the parent `plays` row to get the `play_id` (via `cursor.lastrowid`), then insert child `play_events` rows referencing that `play_id`.

- The `game_id` in the plays JSON filename IS the `event_id` which equals `games.game_id`. No ID translation is needed.
