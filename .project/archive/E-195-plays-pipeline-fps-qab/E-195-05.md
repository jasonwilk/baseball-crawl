# E-195-05: Own-Team FPS/QAB Validation

## Epic
[E-195: Plays Data Ingestion Pipeline -- FPS% and QAB](epic.md)

## Status
`DONE`

## Description
After this story is complete, a validation script will exist that compares derived FPS and QAB counts from the plays pipeline against GC's season-stats API values for own teams. The script is designed to run post-dispatch by the operator (it requires `data/app.db` which is not available in worktrees). The story deliverable is the script with unit tests; the operator runs it after dispatch to produce the validation results artifact.

## Context
Own teams already have FPS (pitching) and QAB (batting) as season aggregates from the season-stats API (`player_season_pitching.fps`, `player_season_batting.qab`). These serve as ground truth for validating our plays-derived values. The validation compares our pipeline's output against GC's pre-computed stats. A 5% tolerance accounts for scorekeeper data quality and edge-case definitional differences (per TN-9).

## Acceptance Criteria
- [ ] **AC-1**: A validation script at `scripts/validate_plays_stats.py` compares derived FPS count per pitcher (from `SUM(is_first_pitch_strike) WHERE outcome NOT IN ('Hit By Pitch', 'Intentional Walk')` grouped by pitcher_id) against `player_season_pitching.fps` for each pitcher with data in both sources.
- [ ] **AC-2**: The script compares derived QAB count per batter (from `SUM(is_qab)` grouped by batter_id) against `player_season_batting.qab` for each batter with data in both sources.
- [ ] **AC-3**: The script produces a summary report showing: per-player comparison (derived vs GC value, absolute difference, percentage difference), overall match rate, and any players exceeding 5% tolerance.
- [ ] **AC-4**: If any player exceeds 5% tolerance, the script outputs diagnostic detail: which games contribute the discrepancy, sample plays where the derived flag may differ from GC's calculation.
- [ ] **AC-5**: The script reports plays data coverage: percentage of completed games (from the `games` table) that have plays data in the `plays` table, flagging any completed games with zero plays as potential scorekeeper gaps.
- [ ] **AC-6**: The script writes output to `/.project/research/E-195-validation-results.md`. This output is produced when the operator runs the script post-dispatch (not during dispatch).
- [ ] **AC-7**: Unit tests verify the script's comparison logic using synthetic test data (in-memory SQLite with known FPS/QAB values inserted into both plays and season-stats tables). Tests do NOT require real game data.
- [ ] **AC-8**: If systematic discrepancies are found during the operator's post-dispatch run, the findings are documented in the results artifact with enough detail for a follow-up fix. The validation story does NOT fix parser bugs -- it identifies them.

## Technical Approach
Create `scripts/validate_plays_stats.py` as a validation script. The script connects to the database, runs aggregation queries on the `plays` table, and compares against `player_season_pitching` / `player_season_batting` tables. Unit tests verify the comparison and reporting logic using synthetic data. The actual validation run happens post-dispatch when the operator runs the script in the devcontainer.
