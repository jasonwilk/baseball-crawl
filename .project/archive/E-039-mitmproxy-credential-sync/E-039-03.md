# E-039-03: Header Capture and Parity Report

## Epic
[E-039: mitmproxy Credential Sync and API Discovery](epic.md)

## Status
`DONE`

## Description
After this story is complete, a mitmproxy addon captures the full header set from every GameChanger request, groups headers by traffic source (iOS app vs. web browser), and produces a parity report comparing captured headers against the project's canonical `BROWSER_HEADERS` in `src/http/headers.py`. The report highlights headers that are missing, extra, or have different values -- making it trivial to keep the project's HTTP fingerprint in sync with real traffic.

## Context
The project's `BROWSER_HEADERS` dict in `src/http/headers.py` was manually captured from a single Chrome 131 curl command. It goes stale as Chrome updates and may not match what the iOS GameChanger app sends. This addon automates the comparison so the operator can see at a glance whether the project's headers are current. The addon does NOT auto-update `BROWSER_HEADERS` -- it produces a report that the operator reviews.

## Acceptance Criteria
- [ ] **AC-1**: Given the mitmproxy service is running with the header capture addon loaded, when a GameChanger request flows through the proxy, then the full request header set is captured and stored in memory, grouped by the source returned by `gc_filter.detect_source()` (`"ios"`, `"web"`, or `"unknown"`).
- [ ] **AC-2**: Given headers have been captured from at least one request, when the operator accesses the parity report (at `data/mitmproxy/header-report.json`), then it contains: (a) the latest captured headers for each source, (b) a diff against `BROWSER_HEADERS` showing missing, extra, and changed headers.
- [ ] **AC-3**: The parity report JSON structure is a top-level object with a `sources` key containing a list of per-source report objects. Each report object includes: `source` (`"ios"`/`"web"`/`"unknown"`), `captured_headers` (dict), `browser_headers` (dict -- the project canonical set), `missing_in_captured` (list of keys in BROWSER_HEADERS but not in captured), `extra_in_captured` (list of keys in captured but not in BROWSER_HEADERS), `value_differences` (list of `{"key": ..., "captured": ..., "canonical": ...}` objects). Top-level also includes `generated_at` (ISO 8601 timestamp). Example structure:
  ```json
  {
    "generated_at": "2026-03-04T12:00:00Z",
    "sources": [
      {
        "source": "web",
        "captured_headers": {"Accept": "text/html", ...},
        "browser_headers": {"Accept": "text/html,application/xhtml+xml,...", ...},
        "missing_in_captured": ["DNT"],
        "extra_in_captured": ["X-Custom"],
        "value_differences": [{"key": "Accept", "captured": "text/html", "canonical": "text/html,application/xhtml+xml,..."}]
      }
    ]
  }
  ```
- [ ] **AC-4**: Given requests from multiple traffic sources (e.g., iOS device, web browser) flow through the proxy, when the report is generated, then it contains a separate entry in the `sources` list for each source that has been observed (only sources with captured traffic appear).
- [ ] **AC-5**: Credential headers (`gc-token`, `gc-device-id`, `gc-signature`, `gc-app-name`, `cookie`) are excluded from the parity report (they are handled by the credential extractor, not the header capture addon).
- [ ] **AC-6**: The report file is overwritten on each update (latest snapshot, not append-only).
- [ ] **AC-7**: Unit tests exist for the header diffing logic (comparing two header dicts and producing the structured diff). Tests do NOT require a running mitmproxy instance.

## Technical Approach
1. Create `mitmproxy/addons/header_capture.py` as a mitmproxy addon class.
2. In the `request()` hook:
   a. Use `gc_filter.is_gamechanger_domain()` (from `mitmproxy/addons/gc_filter.py`, created in E-039-01) to filter to GameChanger domains.
   b. Use `gc_filter.detect_source()` to detect traffic source from User-Agent.
   c. Store the full header dict in a per-source dict (overwriting previous -- we want the latest, not a history).
   d. Exclude credential headers from storage.
3. After updating the stored headers, generate the parity report:
   a. Import `BROWSER_HEADERS` from `src.http.headers` (at `/workspaces/baseball-crawl/src/http/headers.py`). The project root is mounted at `/app` in the mitmproxy container and must be on `sys.path` (same mechanism as E-039-02 -- configured in E-039-01's volume mounts).
   b. Compute the three-way diff (missing, extra, changed).
   c. Write the report to `/app/data/mitmproxy/header-report.json`.
4. Write unit tests in `tests/test_mitmproxy/test_header_capture.py`.

## Notes
- The report is a JSON file (not markdown) so it can be consumed programmatically if needed later.
- Per-request headers like `Content-Length` or `Host` should be excluded from the diff (they are request-specific, not fingerprint-relevant). The addon should maintain a skip-list for these.
- The `data/mitmproxy/` directory is git-ignored (added in E-039-01).
