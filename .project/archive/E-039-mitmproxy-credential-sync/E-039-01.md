# E-039-01: mitmproxy Docker Compose Service and Shared Filter

## Epic
[E-039: mitmproxy Credential Sync and API Discovery](epic.md)

## Status
`DONE`

## Description
After this story is complete, mitmproxy runs as an optional Docker Compose service behind the `proxy` profile. `docker compose --profile proxy up -d` starts the proxy alongside the app stack. The proxy listens on port 8080 (proxy traffic) and 8081 (mitmweb UI), both bound to `0.0.0.0` for LAN access. The Traefik dashboard moves from port 8080 to 8180 to avoid the conflict. CA certificates persist across container restarts via a named volume. A shared `gc_filter.py` utility module is also created as infrastructure for all addon stories.

## Context
This is the infrastructure foundation for all mitmproxy addons. The proxy must be reachable from the iPhone on the LAN, so it binds to `0.0.0.0` rather than `127.0.0.1`. The profile approach ensures the proxy does not start by default and has zero impact on users who do not need it. The shared `gc_filter.py` module is created here (not in a later refactoring story) so that addon stories 02/03/04 can import domain filtering and source detection from day one, avoiding duplicate inline implementations.

## Acceptance Criteria
- [ ] **AC-1**: Given the `docker-compose.yml` file, when `docker compose up -d` is run (no profile flag), then only `app`, `traefik`, and `cloudflared` services start -- mitmproxy does NOT start.
- [ ] **AC-2**: Given the `docker-compose.yml` file, when `docker compose --profile proxy up -d` is run, then the `mitmproxy` service starts alongside the existing services. `docker compose ps` shows the mitmproxy service as `Up` (or `running`). A healthcheck is defined on the service (e.g., `curl -s http://localhost:8081` or a `wget` probe to the mitmweb UI) so that `docker compose ps` reports a health status.
- [ ] **AC-3**: Given the mitmproxy service is running, when an HTTP request is sent through `<host-ip>:8080` as a proxy, then the request is proxied successfully (verify with `curl -x http://localhost:8080 http://httpbin.org/ip`).
- [ ] **AC-4**: Given the mitmproxy service is running, when `http://localhost:8081` is opened in a browser, then the mitmweb UI loads and shows intercepted traffic.
- [ ] **AC-5**: Given the mitmproxy service is stopped and restarted, when the CA certificate fingerprint is checked, then it matches the previous run (certificates persist via volume).
- [ ] **AC-6**: Given the docker-compose.yml file, the Traefik dashboard is accessible on port 8180 (not 8080). Existing Traefik routing (`baseball.localhost:8000`) continues to work.
- [ ] **AC-7**: Given `devcontainer.json`, ports 8080, 8081, and 8180 are listed in `forwardPorts`.
- [ ] **AC-8**: The `data/mitmproxy/` directory is covered by existing `.gitignore` rules (the `data/` entry already handles this -- verify, do not add a redundant entry).
- [ ] **AC-9**: The `mitmproxy/` top-level directory exists with an `addons/__init__.py` file.
- [ ] **AC-10**: A shared module at `mitmproxy/addons/gc_filter.py` exports: (a) `is_gamechanger_domain(host: str) -> bool` that matches `*.gc.com` and `*.gamechanger.com`, (b) `detect_source(user_agent: str) -> str` returning `"ios"`, `"web"`, or `"unknown"` per the detection rules in epic Technical Notes.
- [ ] **AC-11**: `detect_source()` checks iOS patterns (`GameChanger/`, `CFNetwork/`, `Darwin/`) before web patterns (`Chrome/`, `Firefox/`, `Safari/`) so Safari-on-iOS is correctly classified as `"ios"`. All checks are case-insensitive.
- [ ] **AC-12**: Unit tests exist for `is_gamechanger_domain()` and `detect_source()` covering: gc.com, api.gc.com, gamechanger.com, non-GC domains, iOS User-Agents, Chrome User-Agents, Safari-on-iOS User-Agents (should be `"ios"`), empty User-Agent.
- [ ] **AC-13**: The mitmproxy service in `docker-compose.yml` mounts the project root at `/app` (read-only for source, read-write for `.env` and `data/`) so addon scripts can import project modules (`src.gamechanger.credential_parser`, `src.http.headers`) and write to `/app/.env` and `/app/data/mitmproxy/`.

## Technical Approach
1. Add a `mitmproxy` service to `docker-compose.yml` using the `mitmproxy/mitmproxy` image (or `mitmweb` variant per R-01 findings). Set `profiles: [proxy]`. Mount a named volume for cert persistence (`mitmproxy-certs:/home/mitmproxy/.mitmproxy`). Mount the project root at `/app` so addons can import project modules and write to `.env` and `data/mitmproxy/`. Mount `./mitmproxy/addons` for addon script loading. Bind ports `0.0.0.0:8080:8080` and `0.0.0.0:8081:8081`.
2. Change the Traefik dashboard port from `8080:8080` to `8180:8080`.
3. Update `devcontainer.json` `forwardPorts` to include 8081 and 8180 (8080 is already there).
4. Verify `data/mitmproxy/` is covered by existing `.gitignore` `data/` rule (do not add redundant entry).
5. Create `mitmproxy/addons/__init__.py`.
6. Create `mitmproxy/addons/gc_filter.py` with `is_gamechanger_domain()` and `detect_source()` as pure functions (no mitmproxy imports). See epic Technical Notes for detection rules.
7. Write unit tests in `tests/test_mitmproxy/test_gc_filter.py`. Create `tests/test_mitmproxy/__init__.py`.
8. If R-01 findings indicate a custom Dockerfile is needed, create `mitmproxy/Dockerfile` with the necessary additions.

## Notes
- The mitmproxy variant (mitmweb vs. mitmdump) will be determined by E-039-R-01. Default assumption is `mitmweb` for the web UI.
- The proxy port (8080) is a well-known proxy port that iOS can easily connect to. Changing it would add friction.
- If R-01 reveals that `python-dotenv` is not in the mitmproxy image, this story should add a `mitmproxy/Dockerfile` that installs it.
- `gc_filter.py` has no mitmproxy imports -- it is pure Python utility functions. This makes it easy to test without mocking mitmproxy internals. Safari on iOS includes "Safari/" in the User-Agent but also includes "Mobile/" and platform indicators; `detect_source()` must check iOS patterns FIRST to correctly classify it as `"ios"`.
