# E-039-02: GameChanger Credential Extraction Addon

## Epic
[E-039: mitmproxy Credential Sync and API Discovery](epic.md)

## Status
`DONE`

## Description
After this story is complete, a mitmproxy addon watches all proxied traffic for GameChanger authentication headers (`gc-token`, `gc-device-id`, `gc-signature`, `gc-app-name`) and automatically writes them to the project `.env` file using the existing `merge_env_file()` function. Credentials appear in `.env` within seconds of a GameChanger request flowing through the proxy -- no manual curl-copy needed.

## Context
This is the core value proposition of the mitmproxy integration: eliminating the manual credential rotation workflow. Today the user must copy a curl command from Chrome DevTools, paste it into a file, and run a script. With this addon, simply opening the GameChanger app or website while the proxy is active keeps credentials fresh. The addon reuses `src/gamechanger/credential_parser.merge_env_file()` so `.env` format and merge behavior are identical to the existing `refresh_credentials.py` workflow.

## Acceptance Criteria
- [ ] **AC-1**: Given the mitmproxy service is running with the credential extractor addon loaded, when an HTTP request to `*.gc.com` or `*.gamechanger.com` with a `gc-token` header flows through the proxy, then the `.env` file is updated with `GAMECHANGER_AUTH_TOKEN=<token-value>` within 5 seconds.
- [ ] **AC-2**: Given a request with `gc-token`, `gc-device-id`, and `gc-app-name` headers, when captured, then all three are written to `.env` as `GAMECHANGER_AUTH_TOKEN`, `GAMECHANGER_DEVICE_ID`, and `GAMECHANGER_APP_NAME` respectively.
- [ ] **AC-3**: Given a request with a `gc-signature` header, when captured, then it is written to `.env` as `GAMECHANGER_SIGNATURE` (new key -- not currently used by the client, but captured for future use).
- [ ] **AC-4**: Given credentials already exist in `.env` with other non-credential keys (e.g., `CLOUDFLARE_TUNNEL_TOKEN`), when new credentials are written, then existing non-credential keys are preserved (merge, not overwrite).
- [ ] **AC-5**: Given a non-GameChanger request (e.g., to `google.com`) flows through the proxy, when the addon processes it, then no credentials are extracted and `.env` is not modified.
- [ ] **AC-6**: Given duplicate requests with the same `gc-token` value, when captured, then the addon writes to `.env` at most once per unique token value (deduplication to avoid redundant disk writes).
- [ ] **AC-7**: The addon logs (to mitmproxy's log system) each credential update with the traffic source and keys written, never the values (e.g., "Credentials updated from ios source: GAMECHANGER_AUTH_TOKEN, GAMECHANGER_DEVICE_ID"). Credentials in `.env` are NOT source-tagged -- one `GAMECHANGER_AUTH_TOKEN` key regardless of source.
- [ ] **AC-8**: Unit tests exist for the credential extraction logic (header-to-env-key mapping, domain filtering, deduplication) using mocked flow objects. Tests do NOT require a running mitmproxy instance.

## Technical Approach
1. Create `mitmproxy/addons/credential_extractor.py` as a mitmproxy addon class.
2. Implement a `response()` or `request()` hook that:
   a. Uses `gc_filter.is_gamechanger_domain()` (from `mitmproxy/addons/gc_filter.py`, created in E-039-01) to check the request host.
   b. Uses `gc_filter.detect_source()` to determine traffic source for logging.
   c. Extracts credential headers using the `_CREDENTIAL_HEADERS` mapping (same as `/workspaces/baseball-crawl/src/gamechanger/credential_parser.py` but reading from flow objects, not curl strings).
   d. Adds `gc-signature` -> `GAMECHANGER_SIGNATURE` to the mapping.
   e. Compares extracted values against a local cache (dict) to skip writes when nothing changed.
   f. Calls `merge_env_file("/app/.env", ...)` from `src.gamechanger.credential_parser` (at `/workspaces/baseball-crawl/src/gamechanger/credential_parser.py`) to write credentials. The project root is mounted at `/app` in the mitmproxy container (configured in E-039-01).
3. The addon must be loadable by mitmproxy's `--scripts` flag. The project root is on `sys.path` via the `/app` mount (E-039-01).
4. Write unit tests in `tests/test_mitmproxy/test_credential_extractor.py`.

## Notes
- The `gc-signature` header is captured opportunistically. It expires within minutes and cannot currently be regenerated programmatically. Capturing it in `.env` is informational -- it may help future reverse-engineering efforts.
- The addon must handle the case where `.env` does not exist yet (first run). `merge_env_file()` already handles this.
- The mitmproxy Docker container runs as a non-root user. Ensure file permissions on `.env` allow writes.
