# E-016-02: Rewrite Orchestrator Agent Definition

## Epic
[E-016: Evolve Project Manager to Product Manager](./epic.md)

## Status
`DONE`

## Description
After this story is complete, the orchestrator agent definition at `/.claude/agents/orchestrator.md` will specify `model: sonnet` (upgraded from `model: haiku`), include `Read`, `Glob`, and `Grep` in its tools list (enabling it to read project state before routing), and contain an explicit routing table with anti-patterns. The orchestrator will NOT have Write, Edit, or Bash tools -- this is structural enforcement of "route, don't modify." The dispatch routing rule will be simplified to one sentence.

## Context
The current orchestrator is 171 lines running on haiku with only the Task tool. It cannot read files, so it routes purely on keyword matching against the user's request text. This means it cannot check whether an epic is READY before routing a dispatch request, cannot verify story statuses, and cannot understand project structure. Upgrading to sonnet with read-only file tools makes it a genuinely smart router.

The dispatch routing section currently has a lengthy example showing the exact text to tell the user. This can be simplified now that dispatch-pattern.md (modified by E-016-04) contains the canonical rule.

## Acceptance Criteria

- [ ] **AC-1**: The file `/.claude/agents/orchestrator.md` exists and the frontmatter contains `model: sonnet` (changed from `model: haiku`).
- [ ] **AC-2**: The frontmatter `tools` list contains exactly: `Task`, `Read`, `Glob`, `Grep`. It does NOT contain `Write`, `Edit`, `Bash`, or any other tool.
- [ ] **AC-3**: The file contains an explicit routing table (markdown table or structured list) mapping request categories to target agents. At minimum these categories are covered: work-initiation/dispatch, project planning/backlog, API exploration, coaching/domain, agent infrastructure, general implementation.
- [ ] **AC-4**: The file contains an anti-patterns section (or equivalent) listing at least 3 things the orchestrator must NOT do. At minimum: (1) never attempt to do work itself, (2) never route dispatch requests through a Task call to PM, (3) never summarize the user's request when relaying -- pass verbatim. The verbatim relay rule must include: "Pass the user's request verbatim when routing to PM. Do NOT paraphrase or pre-interpret. If the request is ambiguous, ask the user -- do not smooth it over by summarizing." (Inlined from multi-agent-patterns skill, Risk Point 1.)
- [ ] **AC-5**: The dispatch routing rule is present and states that dispatch requests ("start epic X", "execute story X") must NOT be wrapped in a Task call to the PM. The rule references `/.claude/rules/dispatch-pattern.md` and does NOT contain the phrase "Dispatch Manifest".
- [ ] **AC-6**: The file contains a "file-based routing" section or rule that instructs the orchestrator to read relevant files before making routing decisions, using its Read/Glob/Grep tools. Must include at minimum: "Check epic.md Status field before routing a dispatch request" as a concrete example. Must also include the principle: "Read the minimum needed for the routing decision. The orchestrator reads to route, not to understand deeply." (Inlined from context-fundamentals and filesystem-context skills.)
- [ ] **AC-7**: The frontmatter `description` field accurately describes the orchestrator's new capabilities (sonnet model, can read files, smart routing).
- [ ] **AC-8**: The file does NOT contain the string "Dispatch Manifest" anywhere.

## Technical Approach

1. Read the current `orchestrator.md` fully (171 lines).
2. Rewrite the file with the following structure:
   - **Frontmatter**: Update model to sonnet, add Read/Glob/Grep to tools list.
   - **Identity**: Smart router with file-reading capability. Still NOT a worker.
   - **Routing Table**: Explicit table mapping request types to agents. Include the current direct-routing exceptions.
   - **File-Based Routing**: New section. Before routing, the orchestrator should read relevant files to understand project state. Examples: check epic status before routing dispatch, read story file to understand context.
   - **Anti-Patterns**: New section. What the orchestrator must never do.
   - **Dispatch Rule**: Simplified. One rule: "Dispatch requests require PM in main session. Tell the user to invoke PM directly. See dispatch-pattern.md."
   - **Response Format**: Brief guidance on response format (route and relay, stay out of the way).
3. Remove the lengthy dispatch routing example that currently spans ~20 lines.
4. Verify all acceptance criteria are met.

## Notes
- The orchestrator's current "Available Agents" section with detailed examples per agent is useful for keyword routing but may be redundant once the orchestrator can read files. Consider keeping a compact version of the agent descriptions but removing the verbose example lists.
- Key constraint: the orchestrator must NOT have Write/Edit/Bash. This is intentional. If it could edit, it would be tempted to "fix things" instead of routing. The tool restriction is the enforcement mechanism.
- The `color: cyan` field should be preserved from the current definition.
