# E-008-03: Decision Gate -- User Review and Approval of Intent/Context Layer Recommendation

## Epic
[E-008: Intent/Context Layer Design](epic.md)

## Status
`DONE`

## Description
After this story is complete, the user's decision on the intent/context layer recommendation will be explicitly recorded in a decision log entry, and the project's next action will be clearly defined. This story is a structured pause: no agent work proceeds past this point until the user has reviewed the recommendation and stated a position. The outcome is either (a) an approved path with any user-specified constraints captured, ready for a follow-on epic, or (b) a documented rejection with rationale and a defined loop-back action.

## Context
E-008-02 produces a recommendation document written for two audiences: the user (strategic decision) and an implementing agent (execution guide). Before any implementation work begins, the user must explicitly endorse the recommended path. Without this gate, an implementing agent could begin building the wrong system, or build the right system with misaligned constraints, requiring costly rework.

This story exists because the recommendation is a design opinion, not a directive. The user is the operator and the final decision-maker. This gate makes that authority explicit and creates a durable record of what was decided, why, and on what conditions -- so future agents and the PM have ground truth when writing or executing implementation stories.

The decision gate also handles the rejection case cleanly. If the user disagrees with the recommendation, this story captures the rejection rationale and specifies the loop-back: either challenge a specific assumption in the analysis (return to E-008-01), pursue an alternative option that was not sufficiently analyzed (write a new research spike), or defer the decision entirely with an explicit review date.

## Acceptance Criteria
- [ ] **AC-1**: Given the recommendation document at `/.project/research/E-008-intent-context-layer-recommendation.md`, when this story is reached, then the PM confirms the document exists and is complete (all eight sections present) before presenting the decision prompt to the user.
- [ ] **AC-2**: Given the decision prompt, when the user is asked to review, then the prompt surfaces: (a) the name of the recommended approach, (b) a one-sentence rationale summary, (c) the estimated implementation scope, and (d) the three decision options available (Approve / Reject with loop-back / Defer).
- [ ] **AC-3**: Given the user selects "Approve", when the decision is recorded, then a decision log entry is written to `/.project/research/E-008-decision-log.md` containing: date, decision = "APPROVED", the name of the approved approach, and any constraints or modifications the user specified before implementation proceeds.
- [ ] **AC-4**: Given the user selects "Approve", when the decision is recorded, then the epic.md Status is updated to `COMPLETED` and the epic is moved to `/.project/archive/E-008-intent-context-layer-design/` as part of closure.
- [ ] **AC-5**: Given the user selects "Approve", when the decision is recorded, then the PM signals to the user that a follow-on implementation epic (E-009 or next available) can be created, and offers to create it immediately.
- [ ] **AC-6**: Given the user selects "Reject", when the decision is recorded, then a decision log entry is written containing: date, decision = "REJECTED", the rejection rationale as stated by the user, and one of the following loop-back actions: (i) return to E-008-01 with a specified revised scope, (ii) commission a new research spike targeting the alternative the user prefers, or (iii) defer with a specific review date.
- [ ] **AC-7**: Given the user selects "Reject" with loop-back action (i) or (ii), when the decision is recorded, then the relevant story or spike is reset to TODO (or created if new) and E-008 Status is updated to `ACTIVE` to reflect ongoing work.
- [ ] **AC-8**: Given the user selects "Defer", when the decision is recorded, then a decision log entry is written containing: date, decision = "DEFERRED", the reason for deferral, and a review date. The epic Status is updated to `BLOCKED` with a note pointing to the review date.
- [ ] **AC-9**: Given the user approves with constraints, when the decision is recorded, then the constraints are written verbatim into the decision log entry under a "Constraints for Implementation" section, and the PM notes them in the follow-on epic brief.

## Technical Approach
This story is executed by the PM, not by an implementing agent. It is a human-in-the-loop checkpoint. The PM's role is:

1. Confirm `/.project/research/E-008-intent-context-layer-recommendation.md` exists and is complete.
2. Read the "Recommended Approach" and "Implementation Sketch" sections of the recommendation document.
3. Present the decision summary and options to the user in a clear, compact format.
4. Receive the user's decision (Approve / Reject / Defer).
5. If Approve: record the decision log, capture any constraints, close the epic, signal the follow-on.
6. If Reject: record the decision log, establish the loop-back action, update story/epic statuses.
7. If Defer: record the decision log, set a review date, update the epic to BLOCKED.

The decision log file (`/.project/research/E-008-decision-log.md`) is a new file created by this story. If multiple rounds occur (e.g., reject -> revise -> approve), each round appends a new dated entry to the log -- the log is append-only, never overwritten.

### Decision Prompt Format (to present to user)
```
E-008 Intent/Context Layer -- Decision Required

Recommended approach: [name from recommendation doc]
Rationale (summary): [one sentence from recommendation doc]
Implementation scope: [story count estimate from recommendation doc]

Your options:
  A) Approve -- proceed to implementation epic
  B) Reject -- return to analysis or pursue an alternative
  C) Defer -- revisit on [date]

Before choosing A, note any constraints or modifications you want applied
to the implementation (e.g., "do not add files to src/", "keep changes
confined to .claude/", "pilot on one agent before rolling out to all").
```

## Notes
- This story is owned by the PM, not claude-architect. The PM runs the checkpoint and records the decision; no implementation agent is involved.
- The decision log is a project record, not a temporary file. It belongs in `/.project/research/` alongside the recommendation and comparison documents.
- If the user approves with zero constraints, the "Constraints for Implementation" section of the log entry should say "None -- proceed as specified in the recommendation document."
- The loop-back paths in the Reject case are not failure states -- they are valid outcomes. A well-reasoned rejection that produces a sharper research question is progress.
- When offering to create the follow-on epic (AC-5), the PM should ask the user whether they want it created immediately or at a later session.
