# E-027-01: Connect Devcontainer to Compose Network

## Epic
[E-027: Devcontainer-to-Compose Networking](./epic.md)

## Status
`DONE`

## Description
After this story is complete, the devcontainer can reach the FastAPI app container by a stable hostname (e.g., `http://app:8000/health`) without manual IP lookups. The app's Docker Compose network is accessible from inside the devcontainer, and `docker logs` / `docker compose logs` work for reading app output.

## Context
Claude Code runs inside the devcontainer and needs to troubleshoot the FastAPI app -- checking health, curling endpoints, reading logs. Currently the app is only reachable by its ephemeral container IP, which changes on every restart. This story fixes the networking so the app is always reachable by name.

The devcontainer uses Docker-in-Docker (`ghcr.io/devcontainers/features/docker-in-docker:2`), which means the Docker daemon runs inside the devcontainer. Compose containers are children of this daemon. The networking solution must account for this topology -- the devcontainer is not itself a container from its Docker daemon's perspective; it is the host.

## Acceptance Criteria
- [x] **AC-1**: After `docker compose up -d` completes inside the devcontainer, the app is reachable by a stable hostname from the devcontainer shell. Specifically, `curl http://localhost:8001/health` returns a 200 response. The hostname does not change across container restarts.
- [x] **AC-2**: `docker compose logs app` (run from the workspace root inside the devcontainer) returns the app's stdout/stderr output.
- [x] **AC-3**: `docker compose logs -f app` streams live logs (follow mode works).
- [x] **AC-4**: The solution survives a full `docker compose down && docker compose up -d` cycle -- the stable hostname still works after teardown and recreation.
- [x] **AC-5**: No changes to the app's Python code or Dockerfile. Changes are limited to `docker-compose.yml` only.

## Technical Approach
Investigate the network topology inside the docker-in-docker devcontainer. Since the DinD Docker daemon runs inside the devcontainer, the Compose bridge network is local to that daemon. The devcontainer shell can likely reach containers via `localhost` port mappings or by connecting to the bridge network.

Possible approaches (in order of simplicity):
1. **Use `host.docker.internal` or `localhost` with port mappings** -- if the Compose services expose ports, they may already be reachable from the devcontainer shell via `localhost:8000`. Check if this already works and just needs documentation.
2. **Add the devcontainer to the Compose network** -- use `docker network connect` in a `postStartCommand` or equivalent lifecycle hook.
3. **Use a named network in `docker-compose.yml`** with `external: true` and configure the devcontainer to join it.

Start with the simplest approach that satisfies AC-1 through AC-4. If `localhost:8000` already works from the devcontainer shell (because DinD maps ports to the devcontainer's localhost), the solution may just be verifying and documenting that fact -- plus ensuring `docker compose logs` works (AC-2/AC-3).
