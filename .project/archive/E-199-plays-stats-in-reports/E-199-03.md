# E-199-03: Cascade-Delete Team Data on Report Deletion

## Epic
[E-199: Plays-Derived Stats in Standalone Reports](epic.md)

## Status
`DONE`

## Description
After this story is complete, manually deleting a report via the admin UI will cascade-delete the associated team row and all its dependent data -- but only if the team is not independently tracked. This prevents report-generated data from accumulating as orphaned rows. Automatic expiry-triggered deletion is out of scope (expired reports are currently only hidden/404'd). This is a data destruction operation and must be thoroughly tested.

## Context
Report generation creates persistent DB data (team rows, season stats, spray charts, plays, rosters) for ephemeral reports (14-day expiry). Without cleanup, this data accumulates indefinitely for teams that are not independently tracked. The cleanup must be conditional: if the team is also tracked through the normal pipeline (has `team_opponents` links or `is_active=1`), its data must be preserved.

The existing `_delete_report()` function in `src/api/routes/admin.py` deletes only the `reports` row and HTML file. This story extends it to cascade-delete team data when safe. The existing `_cleanup_orphan_teams()` function in `src/reports/generator.py` provides the FK-safe deletion order pattern but doesn't handle `plays`, `play_events`, or `reconciliation_discrepancies` tables.

## Acceptance Criteria
- [ ] **AC-1**: Given a report-only team (`is_active = 0`, no `team_opponents` rows referencing this team_id in either direction, no other `reports` rows referencing this team_id, no shared games with tracked teams per TN-5), when the report is deleted, then the team row and ALL dependent data are deleted in FK-safe order per TN-6: play_events, plays, reconciliation_discrepancies, player_game_batting, player_game_pitching, spray_charts, games, team_rosters, player_season_batting, player_season_pitching, scouting_runs, crawl_jobs, coaching_assignments, user_team_access, and the team row itself. Any `opponent_links` rows where `resolved_team_id` matches are un-resolved (NULLed) rather than deleted.
- [ ] **AC-2**: Given a report team that is also tracked via `team_opponents` (has rows in `team_opponents` as either `our_team_id` or `opponent_team_id`), when the report is deleted, then only the `reports` row and HTML file are deleted. The team and ALL its data are preserved.
- [ ] **AC-3**: Given a report team that was later activated (`is_active = 1`), when the report is deleted, then only the `reports` row and HTML file are deleted. The team and ALL its data are preserved.
- [ ] **AC-4**: Given a report team that appears as an opponent in games involving a tracked team (a team that appears in `team_opponents`), when the report is deleted, then team data is NOT deleted (shared-game guard prevents cascade). Only the `reports` row and HTML file are deleted. This prevents cascade-deleting per-game data for tracked opponents who shared games with the report team.
- [ ] **AC-5**: Given multiple reports reference the same team_id, when one report is deleted, then the team data is NOT deleted (other reports still reference it). Team cleanup only occurs when the LAST report for that team is deleted and all other guard conditions pass.
- [ ] **AC-6**: Given a report team with no games, no plays, no stats, and no roster (empty team row only), when the report is deleted and the team is eligible for cleanup, then the team row is deleted without errors.
- [ ] **AC-7**: The FK-safe cascade delete order handles all tables including `play_events`, `plays`, and `reconciliation_discrepancies`, per TN-6.
- [ ] **AC-8**: Tests cover all six scenarios: (1) clean cascade delete of report-only team with full data, (2) preservation when tracked via `team_opponents`, (3) preservation when `is_active=1`, (4) preservation when shared games exist with tracked teams, (5) multi-report guard, (6) empty-team-row edge case. Each test verifies row counts in ALL affected tables before and after deletion.

## Technical Approach
Extend `_delete_report()` in `src/api/routes/admin.py`. Read `team_id` from the `reports` row BEFORE deleting the report row (the FK is lost after deletion). After deleting the report row and HTML file, check whether the team is eligible for cleanup (guard conditions per TN-5). If eligible, perform the cascade delete. The existing `_cleanup_orphan_teams()` in `src/reports/generator.py` already implements the FK-safe cascade for the orphan cleanup case but is missing plays-related tables and guard conditions. Consolidate: extend `_cleanup_orphan_teams()` to handle all tables per TN-6, then call it from both the report deletion path and the existing orphan cleanup path. This avoids duplicating cascade logic across two files.

## Notes
- The `reports` table has `team_id` FK. Use this to identify the associated team after report deletion.
- The guard condition must check `team_opponents` in BOTH directions -- the team could appear as `our_team_id` or `opponent_team_id`.
- The multi-report guard requires checking `SELECT COUNT(*) FROM reports WHERE team_id = ? AND id != ?` before cascading.
- No FK cascades exist in the schema -- all deletes must be explicit SQL statements in the correct order.
- This is a data destruction operation. The test suite is the safety proof. Each test must verify row counts in ALL affected tables (not just the team row) to prove the cascade is complete AND the guard conditions are airtight.
- AC-4 (shared games): if a report team played games against a tracked team, the shared-game guard prevents cascade deletion entirely. This avoids deleting per-game data (player_game_batting, player_game_pitching, spray_charts) for tracked opponents who shared games with the report team. The guard query: check if any game involving this team also involves a team that appears in `team_opponents` (as either `our_team_id` or `opponent_team_id`).
- `opponent_links`: When an opponent link was resolved to the report team (via `resolved_team_id`), un-resolve the link (NULL out `resolved_team_id`, `resolution_method`, `resolved_at`) rather than deleting the row. This preserves the opponent name and allows re-resolution later. The guard condition checks `team_opponents` (not `opponent_links`), so un-resolved links don't block deletion.
