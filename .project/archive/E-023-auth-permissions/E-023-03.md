# E-023-03: Passkey Registration and Login

## Epic
[E-023: Authentication and Team-Level Permissions](epic.md)

## Status
`DONE`

## Description
After this story is complete, a coach who has logged in via magic link can register a passkey (WebAuthn credential) on their device. On subsequent visits, the login page offers a "Use passkey" button that authenticates via biometric/PIN without needing an email. Both the passkey flow and the magic link flow land in the same session-creation path established by E-023-02.

## Context
Passkeys are an optional upgrade over magic link login. The first time a coach logs in, they use a magic link. After that, they can register a passkey on their phone or laptop for instant future access. This is especially valuable for coaches who check the dashboard frequently from the same device -- a fingerprint tap is faster than waiting for an email.

The WebAuthn ceremony requires JavaScript on the client side (for `navigator.credentials.create()` and `navigator.credentials.get()`). The server side uses `py_webauthn` to generate options and verify responses.

This story modifies the login page created by E-023-02 to add the passkey option, and adds new routes for the registration and authentication ceremonies.

## Acceptance Criteria

### Passkey registration
- [ ] **AC-1**: After a successful magic link verification (E-023-02's `GET /auth/verify` redirect), if the user has no registered passkeys, redirect to a post-login interstitial page (`src/api/templates/auth/passkey_prompt.html`) that offers "Register this device for quick login" (link to `GET /auth/passkey/register`) and "Skip for now" (link to `/dashboard`). If the user already has a passkey, redirect directly to `/dashboard`. This interstitial is a separate template in the auth directory -- it does NOT modify the dashboard template (to avoid file conflicts with E-023-04).
- [ ] **AC-2**: `GET /auth/passkey/register` requires an active session (redirect to `/auth/login` if not authenticated). It renders an HTML page (`src/api/templates/auth/passkey_register.html`) that: (a) calls `py_webauthn.generate_registration_options()` server-side with `rp_id` from `WEBAUTHN_RP_ID` env var (default: `localhost`), `rp_name="LSB Baseball"`, and the user's email + user_id, (b) embeds the resulting options as a JSON blob in a `<script>` tag or data attribute, and (c) includes inline JavaScript that reads the embedded options and calls `navigator.credentials.create()`. This is a single rendered page, not a separate JSON API endpoint.
- [ ] **AC-3**: `POST /auth/passkey/register` receives the attestation response from the browser, verifies it via `py_webauthn.verify_registration_response()`, and stores the resulting `credential_id`, `public_key`, and `sign_count` in the `passkey_credentials` table linked to the user. On success, redirect to `/dashboard` with a flash message "Device registered for quick login." (Flash message via query parameter, e.g., `/dashboard?msg=Device+registered+for+quick+login`.)
- [ ] **AC-4**: A user can register multiple passkeys (e.g., phone + laptop). Each registration creates a separate row in `passkey_credentials`.
- [ ] **AC-5**: If registration verification fails (tampered response, browser error), return a user-friendly error page with a "Try again" link.

### Passkey login
- [ ] **AC-6**: The login page (`GET /auth/login`) includes a "Sign in with passkey" button (in addition to the email form). This button is visible to all users, even if they have not registered a passkey (the browser will handle the "no credential found" case).
- [ ] **AC-7**: Clicking "Sign in with passkey" triggers JavaScript that calls `GET /auth/passkey/login/options` to get authentication options. The server calls `py_webauthn.generate_authentication_options()` with `rp_id` and `allow_credentials=[]` (empty -- let the browser's credential manager handle discovery). The options are returned as JSON.
- [ ] **AC-8**: The browser calls `navigator.credentials.get()` with the options. The resulting assertion response is POSTed to `POST /auth/passkey/login/verify`.
- [ ] **AC-9**: `POST /auth/passkey/login/verify` verifies the assertion via `py_webauthn.verify_authentication_response()`, updates `sign_count` in the DB, creates a session (same path as magic link verify -- generate token, hash, store, set cookie), and redirects to `/dashboard`.
- [ ] **AC-10**: If passkey verification fails, return a JSON error response that the client JavaScript displays as "Passkey authentication failed. Please try again or use email login."

### WebAuthn challenge state
- [ ] **AC-11**: Registration and authentication challenges are stored server-side only (never in a client-side cookie). For registration (user has an active session): store the challenge in the session row or a dedicated column. For authentication (no session yet): store the challenge in a short-lived server-side store (e.g., a `webauthn_challenges` DB table with a TTL, or an in-memory dict keyed by challenge ID -- implementing agent's choice). The challenge must match between the options generation step and the verification step to prevent replay attacks.

## Technical Approach
Add passkey routes to the existing auth router or create a sub-router. The WebAuthn ceremony has two phases (registration, authentication), each with an options-generation step and a verification step.

**Challenge storage**: For registration (user has a session), store the challenge in the user's session row (add a `challenge` TEXT column to `sessions` if needed). Store the challenge bytes (base64-encoded) during options generation, read it back during verification, then clear it. For passkey login (no session yet), use a short-lived server-side store: either a `webauthn_challenges` DB table with a TTL or an in-memory dict keyed by challenge ID. Do NOT use client-side cookies for challenge storage (per AC-11). The implementing agent should choose the simplest server-side approach that works.

**Client-side JavaScript**: The passkey registration and login pages need JavaScript to call the WebAuthn browser APIs. Keep it minimal and inline (no build step). Use the `@simplewebauthn/browser` npm package via CDN OR write raw `navigator.credentials` calls. The implementing agent should choose whichever is simpler -- raw calls are only ~20 lines of JS.

**py_webauthn usage pattern:**
```python
from webauthn import (
    generate_registration_options,
    verify_registration_response,
    generate_authentication_options,
    verify_authentication_response,
    options_to_json,
)
from webauthn.helpers.structs import (
    AuthenticatorSelectionCriteria,
    ResidentKeyRequirement,
    UserVerificationRequirement,
)
```

## Notes
- Passkey support depends on the browser. All modern browsers (Chrome, Safari, Firefox) support WebAuthn. If a coach's browser does not support it, the "Sign in with passkey" button simply will not work and they fall back to magic link. No special detection or fallback logic is needed.
- The `allow_credentials=[]` approach in AC-7 relies on discoverable credentials (resident keys). When generating registration options, set `resident_key=ResidentKeyRequirement.PREFERRED` so the credential is stored on the device and can be discovered without the server providing a credential ID list.
- WebAuthn requires HTTPS in production (localhost is exempted by browsers for development). The Cloudflare Tunnel already provides HTTPS.
- The challenge storage mechanism is left to the implementing agent's judgment within the server-side-only constraint. Options: session row column (for registration) or ephemeral DB table / in-memory dict (for login). No client-side cookies for challenges. The key constraint is that it must survive the round-trip between options generation and verification (typically < 60 seconds).
- The passkey registration CTA uses a dedicated interstitial template (`passkey_prompt.html`) in the auth template directory, not the dashboard template. This avoids file conflicts with E-023-04 and preserves parallel execution capability. The interstitial is shown only after magic link verification when the user has no registered passkeys.
