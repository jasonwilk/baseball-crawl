# E-023-02: Magic Link Login Flow

## Epic
[E-023: Authentication and Team-Level Permissions](epic.md)

## Status
`DONE`

## Description
After this story is complete, any user (coach or admin) can log in by entering their email address on a login page, receiving a magic link via email (or stdout in dev), clicking the link, and landing on the dashboard with an active session. The session middleware protects all dashboard and admin routes, redirecting unauthenticated users to the login page. A dev bypass using `DEV_USER_EMAIL` allows local development without email infrastructure. This is the single login flow for ALL users -- there is no separate admin login path.

## Context
This is the core authentication story. It establishes the session infrastructure that all other auth stories build on (passkeys in E-023-03, dashboard scoping in E-023-04, admin in E-023-05). The magic link flow is the primary login method for ALL users (coaches and admins alike) -- passkeys (E-023-03) are an optional upgrade on top of this.

The session middleware must be in place before any route can use `request.state.user` or `request.state.permitted_teams`. Admin routes (E-023-05) use this same session middleware plus an `is_admin` guard -- there is no separate admin auth mechanism in the app. Cloudflare Access policy handles network-level protection of `/admin` routes externally.

## Acceptance Criteria

### Login page
- [ ] **AC-1**: `GET /auth/login` renders a login page with an email input field and a "Send magic link" submit button. If the user already has a valid session, redirect to `/dashboard` instead of showing the login page.
- [ ] **AC-2**: `POST /auth/login` with a valid email (exists in `users` table) generates a magic link token, stores its SHA-256 hash in `magic_link_tokens` with a 15-minute expiry, and either sends the link via Mailgun or logs it to stdout (see AC-11/AC-12). After processing, render the same confirmation page as AC-3: "If this email is registered, you will receive a login link." (The user-visible response is identical for known and unknown emails to prevent enumeration.)
- [ ] **AC-3**: `POST /auth/login` with an email NOT in the `users` table renders a message: "If this email is registered, you will receive a login link." (Do not reveal whether the email exists -- same response for known and unknown emails to prevent enumeration.)
- [ ] **AC-4**: The magic link URL format is `{APP_URL}/auth/verify?token={raw_token}` where `raw_token` is a `secrets.token_urlsafe(32)` value (43 characters, URL-safe base64).

### Token verification
- [ ] **AC-5**: `GET /auth/verify?token=<raw_token>` hashes the token with SHA-256, looks it up in `magic_link_tokens`, and validates: (a) the token exists, (b) `expires_at` is in the future, (c) `used_at` is NULL. If all pass, mark the token as used (`used_at = datetime('now')`), create a session, set the session cookie, and redirect to `/dashboard`.
- [ ] **AC-6**: If the token is invalid, expired, or already used, `GET /auth/verify` renders an error page: "This login link is invalid or has expired. Please request a new one." with a link back to `/auth/login`.

### Session management
- [ ] **AC-7**: Session creation: generate a `secrets.token_hex(32)` value (64 hex characters), store its SHA-256 hash in the `sessions` table with a 7-day expiry, set an HTTP-only cookie named `session` with the raw token value, `Max-Age=604800`, `SameSite=Lax`, `Path=/`. Set `Secure` flag when not in dev mode.
- [ ] **AC-8**: Session middleware: on every request to a protected route (including `/dashboard/*` and `/admin/*`), read the `session` cookie, hash it, look up in `sessions` table, verify not expired. If valid, attach `request.state.user` (dict: `user_id`, `email`, `display_name`, `is_admin`) and `request.state.permitted_teams` (list of `team_id` strings from `user_team_access`; all `is_owned=1` team_ids if admin). If invalid or missing, redirect to `/auth/login`. Note: `/admin/*` routes are additionally protected at the network layer by Cloudflare Access (WARP required) -- but that is external to the app. The session middleware treats `/admin` routes the same as `/dashboard` routes.
- [ ] **AC-9**: These paths are excluded from session checks: `/auth/*`, `/health`, `/static/*` (if static files are served).
- [ ] **AC-10**: `GET /auth/logout` clears the session cookie, deletes the session row from the DB, and redirects to `/auth/login`.

### Email sending
- [ ] **AC-11**: When `MAILGUN_API_KEY` environment variable is set, the magic link is sent via Mailgun's API (`POST https://api.mailgun.net/v3/{MAILGUN_DOMAIN}/messages`) using `httpx` (already in `requirements.txt`). The email subject is "Your login link for LSB Baseball" and the body contains the magic link URL. Auth is HTTP Basic with username `api` and password `MAILGUN_API_KEY`.
- [ ] **AC-12**: When `MAILGUN_API_KEY` is NOT set, the magic link URL is logged to stdout at INFO level in this format: `[DEV] Magic link for {email}: {url}`. No email is sent.

### Dev bypass
- [ ] **AC-13**: When `DEV_USER_EMAIL` environment variable is set, the session middleware auto-creates a session for that email on every request (bypassing the login page). If the user does not exist in the DB, it is auto-created with `is_admin=1`, `display_name="Dev Admin"`, and team access to all `is_owned=1` teams.
- [ ] **AC-14**: The dev bypass is logged at startup: `[DEV] Auth bypass active for: {email}`.

### Registration in main.py
- [ ] **AC-15**: The session middleware is registered in `src/api/main.py` before route handlers.
- [ ] **AC-16**: The auth router (`/auth/*` routes) is registered in `src/api/main.py`.

### Tests
- [ ] **AC-17**: Tests cover: (a) login page renders, (b) POST login with known email creates a token, (c) POST login with unknown email shows same response (no enumeration), (d) valid token verification creates session and sets cookie, (e) expired token is rejected, (f) used token is rejected, (g) session middleware allows valid session, (h) session middleware redirects invalid session to login, (i) logout clears session, (j) dev bypass auto-creates user and session, (k) health endpoint works without session.

## Technical Approach
Create three new files: `src/api/auth.py` (session middleware), `src/api/routes/auth.py` (login/verify/logout routes), and `src/api/email.py` (Mailgun helper).

**`src/api/auth.py` -- Session middleware:**
Use Starlette's `BaseHTTPMiddleware`. On each request:
1. Check if path is excluded (`/auth/`, `/health`, `/static/`).
2. If `DEV_USER_EMAIL` is set, handle dev bypass.
3. Otherwise, read `session` cookie, hash, look up in DB, check expiry.
4. If valid, attach user + teams to `request.state`.
5. If invalid, redirect to `/auth/login`.

**`src/api/routes/auth.py` -- Auth routes:**
FastAPI `APIRouter` with prefix `/auth`. Routes: `GET /login`, `POST /login`, `GET /verify`, `GET /logout`.

**`src/api/email.py` -- Mailgun helper:**
A simple function `send_magic_link_email(to_email: str, magic_link_url: str) -> bool` that either calls Mailgun or logs to stdout.

Token generation uses `secrets.token_urlsafe(32)` for magic links and `secrets.token_hex(32)` for sessions. Hashing uses `hashlib.sha256`.

For `request.state.permitted_teams` when the user is admin, query `SELECT team_id FROM teams WHERE is_owned = 1`.

Database access: use the existing `src/api/db.py` `get_connection()` function.

Reference files:
- `src/api/main.py` -- where to register middleware and routers
- `src/api/db.py` -- database connection helper
- `migrations/003_auth.sql` -- schema (created by E-023-01)

## Notes
- The "same response for known/unknown email" pattern (AC-3) is a deliberate security measure. The timing difference between the two paths (one sends an email, one does not) is a theoretical side channel, but not worth optimizing for at this scale.
- The session middleware must handle the case where the database has not been migrated yet (no `users` table). In that scenario, the middleware should log a warning and allow the request through (graceful degradation during initial setup).
- The login page HTML should be minimal but usable on mobile. Coaches will often use their phones. Simple CSS, large tap targets, no JavaScript required for the email flow (JS is needed only for passkeys in E-023-03).
- Keep the middleware simple. No caching. Every request does a fresh DB lookup. At 1-5 concurrent users and SQLite on the same machine, this is fast enough.
- `APP_URL` environment variable (e.g., `http://localhost:8000` in dev, `https://baseball.example.com` in prod) is used to construct the magic link URL. Default to `http://localhost:8000`.
- The session middleware is the ONLY auth mechanism in the app. Admin routes do NOT have a separate auth path (no CF JWT header inspection). Cloudflare Access policy protects `/admin` at the network layer (outside the app). The app just checks the session cookie and the `is_admin` flag.
- **Story size**: This story is large (login UI, magic link issuance, verification, session middleware, logout, Mailgun integration, dev bypass, app registration, env changes, test files). It is kept as a single story because these concerns are tightly coupled -- splitting them would create artificial dependencies between login, session, and email subsystems that must work together to deliver any testable increment. The implementing agent should work through the ACs methodically.
- **Existing test impact**: `tests/test_dashboard.py` contains unauthenticated dashboard tests that will fail once the session middleware is registered. This story must update those tests to work with auth (e.g., set `DEV_USER_EMAIL` in the test environment, or inject a session cookie). This is listed in "Files to Create or Modify" above.
