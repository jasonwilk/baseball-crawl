# E-023-05: Admin Page for Managing Coaches

## Epic
[E-023: Authentication and Team-Level Permissions](epic.md)

## Status
`DONE`

## Description
After this story is complete, Jason can manage coach accounts and team assignments through a web page at `/admin/users`. The admin routes are protected by two layers: (1) Cloudflare Access policy at the network layer requires WARP to reach `/admin/*` -- this is external to the app, (2) the app's session middleware + an `is_admin` guard. The app does NOT inspect any Cloudflare headers. The admin page lists all users, shows their team assignments, and provides forms to add new users, remove users, and modify team assignments. Jason is seeded as the first admin user.

## Context
Jason needs a way to manage who can see what. Rather than editing the database directly, this story provides a minimal admin interface. The admin page uses the SAME session-based auth as the rest of the app (magic link / passkey login), plus a simple `is_admin` guard. Cloudflare Access policy (external to the app) requires WARP to reach `/admin/*` at the network layer, but the app does not inspect any Cloudflare headers -- it just checks the session cookie and the `is_admin` flag.

The admin page answers: "How does a new coach get access?" (Jason adds their email and assigns them to teams) and "How do I revoke access?" (Jason removes the user or their team assignments).

In dev mode, the `DEV_USER_EMAIL` bypass auto-creates an admin session, so admin routes work locally without WARP or Cloudflare.

## Acceptance Criteria

- [ ] **AC-1**: All `/admin/*` routes require an active session (via the session middleware from E-023-02) AND `is_admin=1` on the session user. If no session, redirect to `/auth/login`. Do NOT return a bare JSON error -- FastAPI's default `HTTPException(403)` returns JSON, which is not appropriate for a browser-facing page. The app does NOT inspect any Cloudflare headers (`Cf-Access-Jwt-Assertion`, etc.) -- network-level protection of `/admin` is Cloudflare's job, not the app's.

- [ ] **AC-3**: The admin guard is a simple FastAPI dependency or decorator that reads `request.state.user["is_admin"]` (already set by session middleware) and returns 403 if not admin. This is the ONLY app-level check on admin routes -- no CF header parsing, no JWT decoding.

### User management

## Technical Approach
Create `src/api/routes/admin.py` with a FastAPI `APIRouter` using the prefix `/admin`.

**Admin guard**: Write a FastAPI dependency function `require_admin` that:
1. Reads `request.state.user` (already set by session middleware from E-023-02).
2. If no user (middleware would have redirected, but defensive check), redirect to `/auth/login`.
3. If `user["is_admin"] != 1`, raise HTTP 403.
4. Return the user dict.

This is intentionally simple -- the session middleware already handles authentication (who is the user?), the admin guard only handles authorization (are they an admin?). No Cloudflare header inspection, no JWT decoding. Cloudflare Access policy protects `/admin` at the network layer externally.

```python
from fastapi import Depends, Request
from fastapi.responses import HTMLResponse, RedirectResponse
from fastapi.templating import Jinja2Templates

templates = Jinja2Templates(directory="src/api/templates")

async def require_admin(request: Request):
    """Dependency that requires an admin session. Returns 403 page if not admin."""
    user = getattr(request.state, "user", None)
    if not user:
        # Session middleware should have redirected, but be defensive
        return RedirectResponse(url="/auth/login", status_code=302)
    if not user.get("is_admin"):
        # Render the forbidden template (not a bare HTTPException, which returns JSON)
        return templates.TemplateResponse(
            "errors/forbidden.html", {"request": request}, status_code=403
        )
    return user
```

Note: The implementing agent should choose the cleanest FastAPI pattern for the admin guard (dependency vs. middleware). The key requirement is that non-admin users see a rendered HTML forbidden page (`src/api/templates/errors/forbidden.html`), not a JSON error response. The template should be a simple page with a message like "You do not have permission to access this page" and a link back to the dashboard.

Database operations use the existing `src/api/db.py` `get_connection()` function. All CRUD is standard SQL against the `users`, `user_team_access`, `sessions`, `magic_link_tokens`, and `passkey_credentials` tables.

For team assignments, query `SELECT team_id, name FROM teams WHERE is_owned = 1` to populate the checkbox list. Only Lincoln-owned teams appear (not opponent teams).

Flash messages: use a simple query parameter pattern (`/admin/users?msg=User+added+successfully`) rather than adding session infrastructure. Keep it dead simple.

Templates use Jinja2, matching the existing dashboard template patterns.

## Notes
- The admin page is intentionally spartan. No fancy UI, no JavaScript framework. HTML forms with POST actions, server-side rendering via Jinja2. This matches the "dead simple" constraint.
- The admin page does NOT manage Cloudflare Access policies. Jason configures Cloudflare Access separately (via the Cloudflare dashboard) to require WARP for `/admin/*` paths. The admin page only manages which users exist in the app and which teams they can see.
- **Cloudflare Access prerequisite**: Before deploying, Jason must create a Cloudflare Access policy that restricts `/admin/*` to WARP-authenticated users. This is a one-time Cloudflare dashboard configuration, not an app-level concern. Without this policy, `/admin` routes would be accessible to anyone with a valid app session + `is_admin` flag. Document this prerequisite in the deployment runbook or `.env.example`.
- Flash messages via query parameters are not ideal for security-sensitive operations (the message is visible in the URL bar), but for an admin-only page showing "User added successfully," it is perfectly adequate. No sensitive data appears in the flash message.
- The self-delete prevention (AC-6) is a simple server-side check: compare the user_id being deleted with the admin's user_id from `request.state.user`.
- Cascade delete (AC-6) must clean up all tables that reference `user_id`: `user_team_access`, `sessions`, `magic_link_tokens`, `passkey_credentials`. Use explicit DELETE statements rather than relying on SQLite's foreign key cascade (which requires `ON DELETE CASCADE` at table creation time -- not specified in the migration).
- The admin link on the dashboard (AC-10) modifies `team_stats.html`, which E-023-04 also modifies. This is why E-023-05 depends on E-023-04 completing first.
