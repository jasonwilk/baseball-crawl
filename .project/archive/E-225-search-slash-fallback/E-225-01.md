# E-225-01: Shared search helper with punctuation-normalization fallback

## Epic
[E-225: POST /search Punctuation Normalization Fallback](./epic.md)

## Status
`DONE`

## Description
After this story is complete, `src/gamechanger/search.py` will expose a `search_teams_by_name(client, team_name, *, start_at_page=0)` helper that performs a single `POST /search` call by team name and transparently retries with a normalized name (non-word non-space characters replaced by spaces under a Unicode-aware regex, whitespace collapsed) when the first attempt returns zero hits for a name containing any `[^\w ]` character. This is the sole building block used by Story 2 to replace the four duplicated call sites; this story does not yet migrate any caller.

## Context
GameChanger's `POST /search` backend returns zero hits for team names containing certain punctuation characters (confirmed: `/`, straight apostrophe `'` U+0027, `%`, `#`), plus an additional Unicode trap where the indexed canonical form uses a curly apostrophe (U+2019) and queries using a straight apostrophe fail to match. Four call sites in `src/` currently duplicate the same `client.post_json("/search", body={"name": ...}, ...)` pattern with a shared content-type constant, each one vulnerable to the same bug. Rather than patch each site independently, this story creates a single shared helper; Story 2 migrates the call sites onto it. Writing the helper first with full unit tests gives Story 2 a stable, proven primitive to adopt in a mechanical substitution.

See epic Technical Notes TN-1 (helper contract), TN-2 (fallback trigger conditions), TN-3 (normalization behavior), TN-4 (content-type consolidation), TN-5 (error handling), TN-7 (testing pattern), TN-8 (Unicode apostrophe trap coverage), and TN-11 (edge cases).

## Acceptance Criteria
- [ ] **AC-1**: `src/gamechanger/search.py` exists and exports `search_teams_by_name(client, team_name, *, start_at_page=0) -> list[dict]` per Technical Notes TN-1. The `_SEARCH_CONTENT_TYPE` constant is defined in this module. A test `test_signature_keyword_only_start_at_page` asserts via `inspect.signature(search_teams_by_name).parameters` that the function has the parameters `client` (positional-or-keyword), `team_name` (positional-or-keyword), and `start_at_page` (keyword-only with default `0`). This locks the exact parameter name `start_at_page` so Story 2's mocks and call-site migrations do not silently desync if an implementer picks a different name.
- [ ] **AC-2**: Given a gate-clean `team_name` (i.e., contains only `\w` word characters and spaces), when the helper is called, then exactly one `POST /search` call is issued with `body={"name": team_name}` AND `params={"start_at_page": 0, "search_source": "search"}` AND `content_type=_SEARCH_CONTENT_TYPE`, and the returned `hits` list is passed through unchanged.
- [ ] **AC-3**: Given a `team_name` containing at least one `[^\w ]` character AND the first `POST /search` returns zero hits, when the helper runs, then a second `POST /search` is issued with the normalized name per Technical Notes TN-3 and the second response's `hits` is returned. Tests MUST assert the exact body name on BOTH mocked calls using `mock_post_json.call_args_list[0].kwargs["body"]["name"] == <raw>` and `mock_post_json.call_args_list[1].kwargs["body"]["name"] == <exact normalized expected>`. Apply to all trigger-character test cases: `/` (e.g., `"Lincoln Northwest JV/Reserve Falcons"` -> `"Lincoln Northwest JV Reserve Falcons"`), `'` U+0027 (e.g., `"O'Connor Academy Varsity"` -> `"O Connor Academy Varsity"`), `%` (e.g., `"Team%20 Varsity"` -> `"Team 20 Varsity"`), and `#` (e.g., `"Team#1 Varsity"` -> `"Team 1 Varsity"`).
- [ ] **AC-4**: Given a `team_name` containing a `[^\w ]` character AND the first `POST /search` returns non-zero hits, when the helper runs, then NO second call is issued and the first response's `hits` is returned (fallback only fires on empty first attempt -- per Technical Notes TN-2). Test MUST assert `mock_post_json.call_count == 1`.
- [ ] **AC-5**: Given a `team_name` where both attempts return zero hits, when the helper runs, then the helper returns an empty list without raising. Test MUST assert `mock_post_json.call_count == 2` AND assert the exact body name on each call (`[0]` = raw, `[1]` = normalized).
- [ ] **AC-6**: Given the `POST /search` call raises `CredentialExpiredError` (or any other exception), when the helper runs, then the exception propagates to the caller unchanged (helper does not catch, log, or swallow -- per Technical Notes TN-5).
- [ ] **AC-7**: Unit tests in `tests/test_gamechanger_search.py` cover AC-2 through AC-6 and the specific fallback-trigger characters (`/`, `'` U+0027, `%`, `#`). All fallback-path tests assert exact body strings on both mocked calls per AC-3. The test module includes a comment documenting the GC-side storage convention (curly apostrophe is canonical) per Technical Notes TN-8.
- [ ] **AC-8**: Two explicit Unicode-apostrophe test cases per Technical Notes TN-8 exist: (a) a straight-apostrophe name `"O'Connor Academy Varsity"` (U+0027) triggers the fallback and recovers hits -- test asserts `mock_post_json.call_count == 2` AND `call_args_list[1].kwargs["body"]["name"] == "O Connor Academy Varsity"`; (b) a curly-apostrophe name `"Kearney A\u2019s 10U"` (U+2019) returns hits on the first attempt and the fallback never fires -- test asserts `mock_post_json.call_count == 1` AND `call_args_list[0].kwargs["body"]["name"] == "Kearney A\u2019s 10U"` (curly apostrophe passed through unchanged).
- [ ] **AC-9**: For a caller-supplied `start_at_page=N` (test with N=2), tests assert `mock_post_json.call_args_list[0].kwargs["params"]["start_at_page"] == 2` AND `mock_post_json.call_args_list[1].kwargs["params"]["start_at_page"] == 2`. The fallback MUST NOT silently reset to page 0 when invoked from a paginating caller. This is critical because `_resolve_gc_uuid` paginates -- a silent page-0 reset would return page-0 results as if they were page-N, causing wrong-data bugs far worse than the original zero-hits problem.
- [ ] **AC-10**: Negative regression for gate correctness. A test `test_clean_multiword_name_empty_hits_no_fallback` with input `"Lincoln Northwest Falcons"` and mocked empty hits on first attempt asserts `mock_post_json.call_count == 1`. This protects against future simplification of the gate regex dropping the space (e.g., changing `[^\w ]` to `[^\w]`), which would cause every multi-word clean name with empty hits to trigger a spurious second API call.
- [ ] **AC-11**: Exact normalization output is locked. A test `test_normalization_exact_output` with input `"Lincoln // JV  Team"` (two slashes, two spaces between JV and Team) asserts `mock_post_json.call_args_list[1].kwargs["body"]["name"] == "Lincoln JV Team"` after empty first-attempt. A second case with input `"Lincoln\tJV\nTeam"` (tab and newline whitespace) asserts the normalized output is `"Lincoln JV Team"` -- covers TN-3 step 2's whitespace-collapse contract for non-space whitespace.

## Technical Approach
Create a new module `src/gamechanger/search.py` with:

- `from __future__ import annotations` per Python style rules.
- A module-level constant `_SEARCH_CONTENT_TYPE` (verbatim from the four existing locations).
- A module-level regex pattern compiled once for the gate-detection and normalization steps using `re.UNICODE`. The detection test can be a single `re.search(r"[^\w ]", name)` call against the name; the normalization uses `re.sub(r"[^\w ]+", " ", name)` plus whitespace collapse and strip.
- The `search_teams_by_name` function with type hints on every parameter and the return type. The `start_at_page` keyword-only parameter must be named exactly `start_at_page` per AC-1.
- A module-level `logger = logging.getLogger(__name__)`.
- The function takes a `GameChangerClient` instance. Importing `GameChangerClient` should follow the existing pattern in the four current call sites (consult `src/gamechanger/client.py` for the canonical import path and how the four existing call sites reference the client type).
- The function issues the first `client.post_json("/search", body={"name": team_name}, params={"start_at_page": start_at_page, "search_source": "search"}, content_type=_SEARCH_CONTENT_TYPE)` call, inspects `result.get("hits", [])`, and returns that list when it is non-empty OR when the name is gate-clean (contains no `[^\w ]` character).
- Only when both conditions hold (first attempt hits empty AND name contains at least one `[^\w ]` character) does it issue the second call with the normalized name per Technical Notes TN-3. The same `start_at_page` value is threaded to the fallback call. It returns the second attempt's hits (which may also be empty).
- A debug-level log entry when the fallback fires, naming the original name and the normalized name. Do not log at INFO or higher -- this is an internal retry, not an operator-visible event.

Definition of "gate-clean": per Technical Notes TN-2, a name is gate-clean when it contains no `[^\w ]` character -- i.e., only Unicode word characters (`\w`) and literal spaces. A name that is not gate-clean matches the gate and may trigger the fallback when the first attempt returns zero hits.

Write tests in `tests/test_gamechanger_search.py` using a `Mock()` for the client whose `post_json.side_effect` is a function that branches on the `body` argument. All fallback-path tests MUST assert exact body strings on both mocked calls per AC-3. Include at minimum:

- `test_signature_keyword_only_start_at_page` (AC-1 signature lock)
- `test_clean_name_single_call_hits_passed_through` (AC-2)
- `test_clean_name_single_call_empty_hits_returned` (AC-2)
- `test_clean_multiword_name_empty_hits_no_fallback` (AC-10 -- negative regression)
- `test_slash_name_first_empty_triggers_normalized_fallback_returns_hits` (AC-3; asserts exact `"Lincoln Northwest JV/Reserve Falcons"` -> `"Lincoln Northwest JV Reserve Falcons"`)
- `test_percent_name_first_empty_triggers_normalized_fallback_returns_hits` (AC-3; asserts exact normalized output)
- `test_hash_name_first_empty_triggers_normalized_fallback_returns_hits` (AC-3; asserts exact normalized output)
- `test_straight_apostrophe_name_first_empty_triggers_normalized_fallback_returns_hits` (AC-8a; asserts exact `"O'Connor Academy Varsity"` -> `"O Connor Academy Varsity"`)
- `test_curly_apostrophe_name_first_hits_no_fallback` (AC-8b; asserts single call with curly apostrophe passed through unchanged)
- `test_punctuation_name_first_nonempty_no_fallback` (AC-4)
- `test_punctuation_name_both_attempts_empty_returns_empty_list` (AC-5)
- `test_credential_expired_error_propagates` (AC-6)
- `test_start_at_page_passed_through_to_both_attempts` (AC-9; caller passes `start_at_page=2`, test asserts both mocked calls have `params["start_at_page"] == 2`)
- `test_normalization_exact_output` (AC-11; input `"Lincoln // JV  Team"` -> `"Lincoln JV Team"` on normalized body)
- `test_normalization_collapses_tab_and_newline` (AC-11; input `"Lincoln\tJV\nTeam"` -> `"Lincoln JV Team"` on normalized body)
- `test_accented_letter_preserved_in_normalization` (TN-3 guardrail; input `"Gonzàlez Varsity/JV"` -> `"Gonzàlez Varsity JV"` on normalized body, confirming accents are preserved under `re.UNICODE`)
- `test_empty_string_name_single_call_no_fallback`

The mock payload for the user-impact fallback match should use api-scout's confirmed probe output (Technical Notes TN-7): a hit with `result.public_id == "yecaUcoSVpJa"`, `result.id == "ac053e2c-ee27-4f55-9b16-ed77c1bdfebb"`, `result.name == "Lincoln Northwest JV/Reserve Falcons"`. The mock for the curly-apostrophe case should use a fabricated payload whose `result.name` contains the U+2019 character (do not rely on implicit Unicode normalization in the test source; use an explicit `"\u2019"` escape).

A module-docstring comment in `tests/test_gamechanger_search.py` explains the GC storage convention (curly apostrophe is canonical; straight apostrophe is a known-trap input) so future readers understand the shape of case `test_curly_apostrophe_name_first_hits_no_fallback`.

No call site is modified in this story. Existing duplicates of `_SEARCH_CONTENT_TYPE` across the four call-site files remain untouched until Story 2.

## Notes
- The helper intentionally does NOT paginate. Pagination control stays at each call site.
- The helper intentionally does NOT apply any post-search filtering (public_id match, season year match, name exact-match). Those filters live at each call site and remain unchanged in Story 2.
- The normalization regex class `[^\w ]` with `re.UNICODE` is Unicode-word-aware by design. Accented letters (`é`, `ñ`, `ü`), non-Latin scripts, and Unicode digits are preserved across the fallback. Only punctuation and non-word symbols are replaced by spaces. This matches GC's confirmed server-side diacritic-folding behavior -- per Technical Notes TN-3.
- The fallback is a lossy recovery, not a round-trippable transformation. A name normalized from `"O'Connor Academy"` to `"O Connor Academy"` cannot be reversed. That is acceptable because the downstream call-site filters compare against the API's returned `result.name` (canonical form), not against the normalized query form.
