# E-158-03: Spray Chart Loader

## Epic
[E-158: Spray Chart Pipeline and Dashboard Integration](./epic.md)

## Status
`DONE`

## Description
After this story is complete, the load pipeline will parse raw spray chart JSON files and insert ball-in-play events into the `spray_charts` table. The loader handles both offensive and defensive chart types, maps player UUIDs to existing `player_id` records, determines correct `team_id` ownership, and uses idempotent ingestion via `INSERT OR IGNORE` on `event_gc_id`. Both own-team and opponent player events are loaded from each game response.

## Context
The crawler (E-158-02) saves one JSON file per game containing `spray_chart_data.offense` and `spray_chart_data.defense` keyed by player UUID. Each event has a stable GC UUID (`id` field) that serves as the idempotency key. The schema migration (E-158-01) adds the `event_gc_id` column and UNIQUE index needed for this.

Per Technical Notes TN-2, each event's `attributes.defenders` array contains one or more fielder entries with location data. Store only the primary defender (first in the array). Per TN-12, the `spray_chart_data` field may be `null` when the scorekeeper did not record hit locations â€” handle gracefully.

## Acceptance Criteria
- [ ] **AC-1**: A new `SprayChartLoader` class exists at `src/gamechanger/loaders/spray_chart_loader.py` following existing loader patterns (constructor takes `db: sqlite3.Connection`).
- [ ] **AC-2**: The loader extracts events from both `spray_chart_data.offense` and `spray_chart_data.defense` sections, mapping to `chart_type` values `'offensive'` and `'defensive'` respectively.
- [ ] **AC-3**: For each event, the loader stores the following columns: `game_id`, `player_id`, `team_id`, `pitcher_id` (always NULL), `chart_type`, `play_type`, `play_result`, `x` and `y` (from the first defender's location), `fielder_position` (from the first defender), `error` (from the first defender, as 0/1 integer), `event_gc_id` (the event's `id` field), `created_at_ms` (the event's `createdAt` field), `season_id`.
- [ ] **AC-3a**: `game_id` is the `event_id` extracted from the filename (matching `games.game_id` PK). Do NOT use `stream_id`.
- [ ] **AC-3b**: `team_id` is resolved per Technical Notes TN-10: join `games` â†’ `home_team_id`/`away_team_id`, then match `player_id` (the player UUID key from the API response) against `team_rosters`. If the player is not found in either team's roster, assign `team_id` = the opposing team's ID (i.e., whichever of `home_team_id`/`away_team_id` is NOT the crawling team derived from the file path's `gc_uuid`) per TN-10 step 4.
- [ ] **AC-3c**: `season_id` is the `{season}` path segment from `data/raw/{season}/...` stored directly as text (e.g., `2026-spring-hs`) per Technical Notes TN-11. No parsing needed â€” the path segment IS the season_id.
- [ ] **AC-4**: Ingestion uses `INSERT OR IGNORE` keyed on the UNIQUE `event_gc_id` column per Technical Notes TN-6. Re-running the loader for the same game produces zero new inserts.
- [ ] **AC-5**: Events where `spray_chart_data` is `null`, where `spray_chart_data.offense`/`defense` is missing or an empty dict, or where a defender has no `location.x`/`location.y` are all skipped gracefully (logged, not errored). Events with an empty `defenders[]` array (e.g., over-the-fence home runs) are stored with NULL `x`, `y`, `fielder_position`, and `error` â€” the renderer handles these via the center-zone fallback (TN-4).
- [ ] **AC-6**: Players not found in the `players` table get a stub row inserted (`player_id` = UUID from API, `first_name` = 'Unknown', `last_name` = 'Unknown', all other columns NULL) before the spray chart row. The spray chart row's `team_id` is assigned per AC-3b (including the orphan fallback to the opposing team). Log a WARNING with the orphaned player UUID. This is the FK-safe orphan handling pattern used by existing loaders.
- [ ] **AC-7**: The loader returns a `LoadResult` with accurate `loaded`, `skipped`, and `errors` counts.
- [ ] **AC-8**: A `_run_spray_chart_loader()` helper function is added to `src/pipeline/load.py` and appended to the `_LOADERS` list as `("spray-chart", _run_spray_chart_loader)` after the `"season-stats"` entry. This ordering guarantees the `games` table is populated (game loader runs before season-stats) before TN-10's team_id resolution queries it. `"spray-chart"` is also added to the `_LOADER_CHOICES` list in `src/cli/data.py` so `bb data load --loader spray-chart` works.
- [ ] **AC-9**: Tests verify: idempotent re-ingestion (second run = 0 new rows), null spray data handling, player stub creation for unknown players, correct chart_type mapping, and events with empty `defenders[]` array are stored with NULL `x`, `y`, `fielder_position`, and `error` (not skipped).

## Technical Approach
Follow the existing loader patterns (e.g., `GameLoader`). The loader iterates spray JSON files from `data/raw/{season}/teams/{gc_uuid}/spray/{event_id}.json` (written by E-158-02), parses the nested event structure, and inserts rows. The `game_id` is the `event_id` extracted from the filename (matching `games.game_id` PK).

For `team_id` determination per TN-10: use the `games` table (`home_team_id`/`away_team_id`) and `team_rosters` to match each player UUID to their team. The `players` table has no `team_id` column â€” do not attempt to look up team from `players` directly.

Reference the spike's fetch script at `.project/research/spray-chart-spike/fetch.py` for the API response parsing pattern, but adapt to the loader architecture.
