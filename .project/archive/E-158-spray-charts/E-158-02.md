# E-158-02: Spray Chart Crawler

## Epic
[E-158: Spray Chart Pipeline and Dashboard Integration](./epic.md)

## Status
`DONE`

## Description
After this story is complete, the crawl pipeline will fetch and cache raw player-stats JSON (containing spray chart data) for every completed game, writing one file per game to `data/raw/{season}/teams/{gc_uuid}/spray/{event_id}.json`. The crawler integrates into the existing crawl orchestrator and runs after `game-stats` in the pipeline order.

## Context
The spray chart data comes from a different endpoint than boxscores (`player-stats` vs `boxscore`). It follows the same pattern as `GameStatsCrawler`: read game-summaries to find completed games, fetch per-game data, cache to disk. The endpoint is fully documented at `docs/api/endpoints/get-teams-team_id-schedule-events-event_id-player-stats.md`. One API call returns both teams' spray data.

Completed game spray data never changes, so the idempotency check is existence-only (same as `GameStatsCrawler` — if the file exists, skip).

## Technical Approach
Follow the `GameStatsCrawler` pattern closely. The crawler reads game-summaries files from disk (written by `ScheduleCrawler`), identifies completed games, and fetches their player-stats. The output path differs from boxscores (`spray/` subdirectory instead of `games/`). Registration in `crawl.py` requires adding the import and a new entry to `_CRAWLER_NAMES` and `_build_crawlers()`.

Reference the spike's fetch script at `.project/research/spray-chart-spike/fetch.py` for the API call pattern, but adapt to the crawler class architecture (constructor injection, CrawlResult, etc.).
