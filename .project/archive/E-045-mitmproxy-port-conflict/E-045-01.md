# E-045-01: Remove mitmproxy ports from devcontainer forwarding

## Epic
[E-045: Resolve mitmproxy Port Conflict with Devcontainer](epic.md)

## Status
`DONE`

## Description
After this story is complete, the devcontainer no longer forwards ports 8080 and 8081 via VS Code, eliminating the conflict with Docker Compose's own `0.0.0.0` bindings for the mitmproxy service. Docker Compose remains the sole mechanism for exposing mitmproxy ports to the LAN, and a clarifying comment in `docker-compose.yml` explains why these ports must not be duplicated in devcontainer forwarding.

## Context
VS Code's `forwardPorts` in `devcontainer.json` currently includes 8080 and 8081. These conflict with OrbStack/Docker's port mappings from `docker-compose.yml` (`0.0.0.0:8080:8080` and `0.0.0.0:8081:8081`). Both mechanisms bind to the host's port 8080, causing the iPhone to sometimes connect to VS Code's forwarding tunnel instead of Docker's mapping -- resulting in the Traefik dashboard appearing instead of mitmproxy's `mitm.it` certificate page.

The fix is simple: remove 8080 and 8081 from `forwardPorts`. Docker Compose's `0.0.0.0` binding handles LAN exposure. The mitmweb UI (8081) is also accessible via `localhost:8081` through Docker's binding, so VS Code forwarding adds no value.

## Technical Approach
1. Read `/workspaces/baseball-crawl/.devcontainer/devcontainer.json`.
2. Change `"forwardPorts": [8000, 8080, 8081, 8180]` to `"forwardPorts": [8000, 8180]`.
3. Add a comment explaining the exclusion. Since JSON does not support comments, use the `//` convention that VS Code's JSON-with-comments parser supports (devcontainer.json is parsed as JSONC by VS Code).
4. Read `/workspaces/baseball-crawl/docker-compose.yml`.
5. Add a comment to the mitmproxy ports section warning against duplicating these in devcontainer forwardPorts.
6. Run `pytest` to confirm no regressions.
