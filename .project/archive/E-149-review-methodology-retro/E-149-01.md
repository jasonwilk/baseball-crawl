# E-149-01: CR Agent Definition -- Bug Pattern Checklist Expansion

## Epic
[E-149: Review Methodology Retro and Gap-Filling](epic.md)

## Status
`DONE`

## Description
After this story is complete, the code-reviewer agent definition will contain 6 new bug pattern checklist items plus 1 enhancement to the existing Status lifecycle item, addressing the gap categories identified in the E-147/E-148 retro. The CR will have explicit instructions to cross-reference callers, API endpoint docs, migration files, function docstrings, and prose references -- expanding its review scope beyond the Files Changed list.

## Context
The E-147/E-148 retro identified that 16 of 18 Codex findings trace back to CR's rubric lacking cross-referencing instructions (13 non-context-layer findings plus 3 context-layer findings addressed via checklist extensions). CR reviewed changed files in isolation and never looked at callers, API docs, migration schemas, or unchanged functions that interact with changed code. See epic TN-1 through TN-3 for the full finding catalog and proposed checklist items.

## Acceptance Criteria
- [ ] **AC-1**: The Bug Pattern Checklist in `.claude/agents/code-reviewer.md` contains a **Caller audit** item per TN-3 item 1. The item instructs CR to grep for callers of any function whose signature, return type, or observable behavior changed, and verify each caller remains correct. Includes the semantic sibling extension and stale prose reference sweep sub-clauses.
- [ ] **AC-2**: The Bug Pattern Checklist contains an **API field contract** item per TN-3 item 2. The item instructs CR to cross-reference API field paths against `docs/api/endpoints/`, verify correct endpoint variant usage (authenticated vs public), and confirm required headers are included.
- [ ] **AC-3**: The Bug Pattern Checklist contains a **Function contract preservation** item per TN-3 item 3. The item instructs CR to compare rewritten functions against their docstrings, type hints, and documented behavioral guarantees.
- [ ] **AC-4**: The Bug Pattern Checklist contains a **Deploy-time safety** item per TN-3 item 4. The item instructs CR to verify new column references exist in the current migration set, that migration numbers are sequential, and that Python code handles NULL for new columns on existing tables.
- [ ] **AC-5**: The Bug Pattern Checklist contains a **Remediation regression guard** item per TN-3 item 5. The item instructs CR to apply the full checklist to Round 2 remediation code, not just verify the original finding is fixed.
- [ ] **AC-6**: The Bug Pattern Checklist contains a **Test-validates-spec** item per TN-3 item 6. The item instructs CR to verify test mocks match the authoritative spec, not the implementation under test.
- [ ] **AC-7**: The existing **Status lifecycle** checklist item is extended per TN-3 item 7 to cover in-memory flags and booleans (not just database status writes). The extension verifies flags are set only AFTER the gated operation succeeds.
- [ ] **AC-8**: Each new/enhanced checklist item includes a "Catches:" annotation listing which retro findings it addresses (per TN-7 regression test matrix), so the provenance is traceable. Format example: `*Catches: E-147 findings 1, 4 (wrong API field path, missing vendor header)*`.
- [ ] **AC-9**: The existing Bug Pattern Checklist items (SQL query scope, Return value consumption) are unchanged beyond the Status lifecycle extension in AC-7.
- [ ] **AC-10**: The Step 2 (Load context) list is updated to include conditional context loading: "If the review assignment includes API endpoint files, also read those files" and "If the review assignment includes migration files, also read those files." The instruction references files provided in the assignment (by Story 02), not files CR discovers independently.

## Technical Approach
The implementer (architect) should read the current CR agent definition, identify where the Bug Pattern Checklist sits in the rubric structure, and add the 6 new items plus 1 enhancement in the style of the existing checklist items (SQL query scope, Return value consumption, Status lifecycle). Items 1-6 are new; item 7 is an extension to the existing Status lifecycle item. Each item should be concrete and procedural -- tell CR exactly what to do, not just what to look for. See TN-3 for the detailed item descriptions. The Step 2 context loading additions should be conditional -- triggered by the structured context fields in the CR assignment (added by Story 02), not by CR's independent discovery -- to avoid loading irrelevant files for every review.
