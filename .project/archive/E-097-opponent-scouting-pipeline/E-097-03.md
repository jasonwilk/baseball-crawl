# E-097-03: Opponent Scouting Crawler

## Epic
[E-097: Opponent Scouting Data Pipeline](epic.md)

## Status
`DONE`

## Description
After this story is complete, the system will have a scouting crawler (`src/gamechanger/crawlers/scouting.py`) that, given an opponent's `public_id`, fetches their game schedule, player roster, and per-game boxscores via public GameChanger endpoints. A loader (`src/gamechanger/loaders/scouting_loader.py`) will persist the crawled data into the existing database tables and compute season aggregates from boxscores. A new `bb data scout` CLI command will let the operator trigger scouting runs for all opponents with a `public_id` or a specific team.

## Context
The scouting chain has been verified end-to-end (2026-03-12) using public endpoints on an unfollowed team (`8O8bTolVfb9A`). The entire flow works with `public_id` alone -- no UUID required, no following required. The public `/games` endpoint returns game `id` values that work directly as `game_stream_id` for boxscores (no bridge call needed). The schedule step requires NO authentication. The `season-stats` endpoint returns Forbidden for non-owned teams, so season aggregates must be computed from boxscores. The existing crawler/loader patterns in `src/gamechanger/crawlers/` and `src/gamechanger/loaders/` provide the architectural template. The existing data tables already support opponent data. The `scouting_runs` table and `teams.gc_uuid` column (story 02) provide idempotency tracking and UUID opportunism.

## Acceptance Criteria

### Crawler
- [ ] **AC-1**: A module `src/gamechanger/crawlers/scouting.py` exists with a `ScoutingCrawler` class that accepts a `GameChangerClient`, config, and database connection. It exposes a `scout_team(public_id: str, season_id: str)` method and a `scout_all()` method. The `public_id` is the primary input; UUID is not required.
- [ ] **AC-2**: `scout_team()` executes the public-endpoint scouting chain: (1) fetch game schedule via `GET /public/teams/{public_id}/games` (NO auth -- use `get_public()` method), filter to `game_status == "completed"`, extract `id` field as `game_stream_id`; (2) fetch roster via `GET /teams/public/{public_id}/players` (gc-token); (3) fetch boxscores for each game via `GET /game-stream-processing/{game_id}/boxscore` (gc-token). Season-stats is NOT called (returns Forbidden for non-owned teams). The `game-summaries` endpoint is NOT used.
- [ ] **AC-3**: `scout_all()` queries `opponent_links` for all opponents with a `public_id` (WHERE `public_id IS NOT NULL`) and calls `scout_team()` for each. Opponents with recent scouting runs (check `scouting_runs` table, configurable freshness threshold) are skipped.
- [ ] **AC-4**: The crawler writes raw API responses to `data/raw/{season}/scouting/{public_id}/` (games.json, roster.json, boxscores/{game_stream_id}.json) following the existing raw-data pattern.
- [ ] **AC-5**: The crawler creates/updates a `scouting_runs` row (status: running -> completed/failed, with counts of games found, games crawled, players found). On first run, `first_fetched` is set; on every run, `last_checked` is updated to the current timestamp.
- [ ] **AC-6**: The crawler handles `CredentialExpiredError` and `ForbiddenError` gracefully -- logs a warning and skips the opponent rather than crashing the entire run.

### Loader
- [ ] **AC-7**: A module `src/gamechanger/loaders/scouting_loader.py` exists that delegates per-game loading to the existing `GameLoader` (`src/gamechanger/loaders/game_loader.py`) using `GameLoader.load_file()` directly (NOT `load_all()`, which reads game_summaries.json). The internal `_GameSummaryEntry` namedtuple in `game_loader.py` must be renamed to `GameSummaryEntry` and made public/importable. Pass the scouted opponent's `team_id` as `owned_team_id` to `GameLoader`. The scouting loader's additional responsibilities: roster loading into `players`/`team_rosters`, `scouting_runs` tracking, and season aggregate computation (AC-8a).
- [ ] **AC-8**: The loader uses the FK-safe stub-player pattern for any unknown player_ids encountered in boxscores (consistent with existing loaders -- see CLAUDE.md).
- [ ] **AC-8a**: The loader computes season aggregate stats by querying `player_game_batting` and `player_game_pitching` rows for the scouted team's players, summing counting stats (batting: AB, H, doubles, triples, HR, RBI, BB, SO, SB; pitching: ip_outs, H, ER, BB, SO, HR), and upserting into `player_season_batting` and `player_season_pitching`. Rate stats (AVG, OBP, ERA, WHIP) are NOT stored — they are computed at display time from counting stat columns. Home/away split columns are left NULL for scouting-computed season stats (deferred to a future story).
- [ ] **AC-9**: The loader is idempotent -- re-running with the same raw data does not create duplicate rows. For `scouting_runs`, use `ON CONFLICT(team_id, season_id, run_type) DO UPDATE` (preserving `first_fetched` per E-097-02 AC-4). For other tables (players, games, stats), use `INSERT OR REPLACE` or `ON CONFLICT` as appropriate to each table's constraints.

## Technical Approach
Follow the patterns established by the existing crawlers and loaders. Key references:
- Crawler pattern: `src/gamechanger/crawlers/opponent.py` (two-phase crawl, raw file output, freshness skip)
- **Game loader (PRIMARY reference)**: `src/gamechanger/loaders/game_loader.py` — reuse `GameLoader.load_file()` directly for per-game boxscore loading (NOT `load_all()`). It handles boxscore parsing, player stubs, game records, batting/pitching line upsert, asymmetric team key detection, and home/away assignment. The `_GameSummaryEntry` namedtuple must be renamed to `GameSummaryEntry` (public/importable) -- already touching this file for the IP bug fix.
- Season stats loader: `src/gamechanger/loaders/season_stats_loader.py` — reference for `_ip_to_ip_outs()` and season stat upsert patterns (but the scouting loader aggregates from DB rows, not from the API)
- CLI pattern: `src/cli/data.py` (existing `bb data crawl` and `bb data load` commands)
- Client: `src/gamechanger/client.py` (`GameChangerClient` with `get()`, `get_paginated()`, and new `get_public()`)

**Public endpoint usage**: The schedule endpoint (`/public/teams/{public_id}/games`) requires NO authentication -- use the new `get_public()` method. The roster (`/teams/public/{public_id}/players`) and boxscore endpoints require gc-token -- use the existing `get()` method. Note the inverted URL patterns: `/public/teams/` for schedule, `/teams/public/` for roster.

**Public games field mapping**: The response fields differ from `game-summaries`. Key fields: `id` (= game_stream_id for boxscores), `game_status` (filter to `"completed"`), `score.team`/`score.opponent_team` (scoring data). No pagination has been observed.

**Boxscore structure**: The response uses `players[]` + `groups[]` (lineup/pitching categories) with per-player `stats[]` arrays. Player identity is in `players[]`, stat entries reference by `player_id`. The existing `game_loader.py` handles this correctly — do NOT reimplement the parsing.

**Season aggregate computation**: Query `player_game_batting` and `player_game_pitching` tables for the scouted team's players, sum counting stats across all games, and upsert into `player_season_batting`/`player_season_pitching`. Rate stats are NOT stored — they are computed at display time. Home/away split columns are left NULL (deferred).

**UUID opportunism**: When `_detect_team_keys()` finds a UUID key in the boxscore response, check if the corresponding team's `teams.gc_uuid` is NULL and update it if so. This is a write-through pattern that never blocks the scouting flow.

**season_id derivation rule**: Derive `season_id` from the earliest game date in the schedule response: extract the year, use `{year}-spring-hs` as the default (e.g., `2025-spring-hs`). This matches all current opponents (high school spring ball). If the schedule has no games, skip the opponent. Non-spring and non-HS season types are out of scope for this epic. The `--season` CLI flag provides a manual override escape hatch for edge cases.

## Notes
- Verified test data: Lincoln Southwest Varsity 2025 (28 games, ~15 players) and unfollowed team `8O8bTolVfb9A` (24 games, 19 players).
- Rate limiting: the crawler will make 2 + N API calls per opponent (1 unauthenticated schedule + 1 roster + N boxscores, where N = completed games). For 10 opponents with 25 games each, that is ~270 calls. Existing rate limiting in `GameChangerClient` (1-2s delay between requests) should be sufficient. The crawler should log progress so the operator can monitor.
- Boxscores are NOT optional in this pipeline -- they are the source of truth for season aggregates since `season-stats` is Forbidden for non-owned teams. Every scouting run must fetch boxscores to compute season stats.
- **IP is float decimal innings** in the boxscore response (e.g., 3.333... = 3⅓ IP = 10 outs). Convert via `round(float(IP) * 3)`. See AC-11a for the pre-existing bug fix.
- **GameLoader reuse**: `_detect_team_keys()` detects by format (slug vs UUID pattern) -- it works as-is for the scouting path, no changes needed.
- **Public games returns only completed games**: This is exactly what the scouting pipeline needs (stats from finished games). If future use cases need upcoming/in-progress games, the authenticated `game-summaries` fallback exists.
