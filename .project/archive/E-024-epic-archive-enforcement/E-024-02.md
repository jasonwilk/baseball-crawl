# E-024-02: PreToolUse Hook: Block Commits with Unarchived Completed Epics

## Epic
[E-024: Epic Archive Enforcement](epic.md)

## Status
`DONE`

## Description
After this story is complete, a PreToolUse hook will block any `git commit` command when completed or abandoned epics remain in the `/epics/` directory. This provides technical enforcement of the archive requirement, following the same two-layer defense pattern established by E-019 (PII safety) and E-022 (safety scan hardening).

## Context
The PM already has prose instructions to archive completed epics, but E-022 proved that prose alone is insufficient (lesson from E-021: "Prose prohibitions without technical enforcement are insufficient"). The project has an established hook pattern: `/.claude/hooks/pii-check.sh` intercepts `git commit` via PreToolUse, scans for violations, and outputs JSON denial if problems are found. This story creates an analogous hook for epic archive enforcement.

## Acceptance Criteria
- [ ] **AC-1**: A new file `/.claude/hooks/epic-archive-check.sh` exists, is executable, and follows the same structure as `/.claude/hooks/pii-check.sh` (reads JSON stdin via jq, checks for git commit pattern, outputs JSON denial or exits silently).
- [ ] **AC-2**: The hook scans all `epic.md` files under `$CLAUDE_PROJECT_DIR/epics/` for status `COMPLETED` or `ABANDONED` (backtick-wrapped, e.g., `` `COMPLETED` ``).
- [ ] **AC-3**: Given an `epic.md` file in `/epics/` with status `COMPLETED`, when a `git commit` command is intercepted, then the hook outputs a JSON denial with `permissionDecision: "deny"` and the denial reason identifies which epic(s) need archiving.
- [ ] **AC-4**: Given an `epic.md` file in `/epics/` with status `ABANDONED`, when a `git commit` command is intercepted, then the hook outputs a JSON denial (same behavior as COMPLETED).
- [ ] **AC-5**: Given no epic.md files in `/epics/` have status COMPLETED or ABANDONED, when a `git commit` command is intercepted, then the hook exits silently (exit 0, no output), allowing the commit to proceed.
- [ ] **AC-6**: Given `jq` is not installed, the hook exits 0 silently (fail open, same as PII hook).
- [ ] **AC-7**: Given no `/epics/` directory exists or it contains no epic.md files, the hook exits 0 silently.
- [ ] **AC-8**: The hook is registered in `/.claude/settings.json` under `hooks.PreToolUse` as a second entry in the existing `Bash` matcher's hooks array, with `timeout: 10` and an appropriate `statusMessage`.
- [ ] **AC-9**: The hook is documented in `/.claude/hooks/README.md` following the same format as the existing PII Check section.

## Technical Approach
1. Read `/.claude/hooks/pii-check.sh` as the reference implementation for the hook structure.
2. Create `/.claude/hooks/epic-archive-check.sh`:
   - Parse stdin JSON with `jq` to extract the command
   - Check if command matches `git commit` pattern: `(^|[;&|]\s*)git\s+commit`
   - If not git commit: `exit 0`
   - Scan for stale epics: `grep -rl '`COMPLETED`\|`ABANDONED`' "$CLAUDE_PROJECT_DIR"/epics/*/epic.md 2>/dev/null`
   - If matches found: build a denial message listing the epic directories that need archiving, output JSON denial
   - If no matches: `exit 0`
3. Read `/.claude/settings.json` and add the new hook as a second entry in the existing `hooks.PreToolUse[0].hooks` array (the Bash matcher).
4. Read `/.claude/hooks/README.md` and add a documentation section for the new hook.
