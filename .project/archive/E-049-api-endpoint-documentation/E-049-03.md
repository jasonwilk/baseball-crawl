# E-049-03: Document team, event, and game-stream endpoint schemas from bulk payloads

## Epic
[E-049: API Endpoint Documentation and Dual-Header System](epic.md)

## Status
`DONE`

## Description
After this story is complete, the `docs/gamechanger-api.md` API spec will contain full response schema documentation for the remaining bulk-collected endpoints: team-scoped endpoints not yet documented, event-scoped metadata endpoints, and game-stream narrative/viewer endpoints. This covers 22 of the 37 raw payloads (the `/me/*` and `/organizations/*` families are handled by E-049-01 and E-049-02 respectively).

## Context
Many team-scoped endpoints (`/teams/{id}/players`, `/teams/{id}/schedule`, etc.) are already documented. This story focuses on the NEW team endpoints discovered via proxy (`opponents/players`, `relationships`, `external-associations`, `notification-setting`, `scoped-features`) plus the game-stream narrative endpoints (`recap-story`, `viewer-payload-lite`, `events`) and event metadata. The `team-opponents-players.json` payload is very large (102K+ tokens) and represents a high-value bulk opponent roster endpoint.

## Technical Approach
1. Read each raw payload file for team-scoped, event-scoped, and game-stream families
2. For `team-opponents-players.json` (102K+ tokens), read only the first 50-100 lines to capture the schema structure
3. For `gamestream-recap-story.json`, document the structured narrative format carefully -- the typed segment system (`team`, `player`, `text`) with embedded UUIDs is unique and valuable
4. Cross-reference with `proxy/data/endpoint-log.jsonl` for request metadata
5. For already-documented endpoints, compare the raw payload against existing documentation to identify schema drift
6. Redact all PII in example snippets

## Notes
- Raw payload locations: all files in `data/raw/bulk-20260305-234522/` not covered by E-049-01 or E-049-02
- The `gamestream-recap-story.json` format is remarkable -- it's essentially a templated sports article with structured references to teams and players. Each player segment includes `id`, `name`, `short_name`, `team_id`, `num`, and `is_first_mention`. This could be a source for generating human-readable scouting reports.
- `team-opponents-players.json` is the paginated bulk opponent roster -- 102K+ tokens suggests hundreds of player records. Document the per-player schema, pagination structure, and total count.
- `event-player-stats.json` is already fully documented (CONFIRMED in API spec) -- skip it
