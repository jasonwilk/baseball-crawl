# E-049-06: Document mobile credential capture workflow

## Epic
[E-049: API Endpoint Documentation and Dual-Header System](epic.md)

## Status
`DONE`

## Description
After this story is complete, `docs/admin/mitmproxy-guide.md` will contain a dedicated section explaining how to capture mobile app credentials (gc-token, gc-device-id) from the iOS GameChanger app via mitmproxy, how those credentials differ from web browser credentials, and how to install them in the system for use with the mobile header profile.

## Context
The mitmproxy infrastructure (E-039, E-048) is already in place and was used to capture the iOS app traffic that discovered the 50+ new endpoints. However, the workflow for extracting and using mobile credentials is not documented. Web credentials come from browser DevTools (copy as curl). Mobile credentials come from mitmproxy interception of iOS app HTTPS traffic. The two credential sets share the same `gc-token` JWT format but differ in device-id and app-version headers.

## Acceptance Criteria
- [ ] **AC-1**: `docs/admin/mitmproxy-guide.md` contains a new section "Mobile Credential Capture" explaining the end-to-end workflow: (1) start mitmproxy on the Mac host, (2) configure the iOS device to use the proxy, (3) install the mitmproxy CA certificate on the device, (4) open the GameChanger app and sign in, (5) identify the credential headers in the captured traffic
- [ ] **AC-2**: The section documents which headers to extract from mobile traffic for credential storage: `gc-token`, `gc-device-id`, and optionally `gc-app-version` (noting the mobile value `2026.7.0.0` vs web `0.0.0`)
- [ ] **AC-3**: The section explains how to store mobile credentials in `.env` for use with the mobile header profile, including any new env var names needed (e.g., `GC_MOBILE_DEVICE_ID` if different from the web device ID, or a note that the same `gc-device-id` may work for both)
- [ ] **AC-4**: The section includes a comparison table: "Web vs Mobile Credential Capture" showing the source (browser DevTools vs mitmproxy), the auth headers (same gc-token JWT), the device-id (may differ), and the credential lifetime (same 14-day token)
- [ ] **AC-5**: The section notes security considerations: the mitmproxy CA cert installation enables MITM of all HTTPS traffic on the device, not just GameChanger -- include a warning to remove the CA cert after capture is complete
- [ ] **AC-6**: The section references the existing mitmproxy startup/shutdown commands (`cd proxy && ./start.sh`, `./stop.sh`, `./status.sh`) and the credential extraction addon at `proxy/addons/credential_extractor.py`

## Notes
- The mitmproxy CA cert installation on iOS requires: Settings > General > About > Certificate Trust Settings > enable the mitmproxy cert. This may change across iOS versions -- note the iOS version used for the documented workflow (iOS 26.x based on proxy capture).
- An open question is whether the mobile and web `gc-device-id` values can be interchanged. If the server ties the device-id to the app type, using a web device-id with mobile headers (or vice versa) might fail. Document this as "unverified -- test when mobile credentials are available."
