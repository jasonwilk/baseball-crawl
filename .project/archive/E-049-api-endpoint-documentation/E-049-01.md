# E-049-01: Document /me/* endpoint schemas from bulk payloads

## Epic
[E-049: API Endpoint Documentation and Dual-Header System](epic.md)

## Status
`DONE`

## Description
After this story is complete, the `docs/gamechanger-api.md` API spec will contain full response schema documentation for all `/me/*` endpoints whose raw payloads were captured in the bulk collection. Each endpoint will have field types, nullability, array structures, and coaching-relevance notes. Endpoints that returned errors will be documented with their error responses and the context in which they failed.

## Context
The bulk collection at `data/raw/bulk-20260305-234522/` contains 7 `/me/*` payloads. Some of these endpoints (`/me/user`) are already documented -- verify for schema drift. Others (`/me/associated-players`, `/me/archived-teams`) are entirely new and contain high-value data for player tracking. One (`/me/related-organizations`) returned an HTTP 500 error with web headers.

## Acceptance Criteria
- [ ] **AC-1**: `docs/gamechanger-api.md` contains a full response schema section for `GET /me/teams-summary` with field types and notes, based on `data/raw/bulk-20260305-234522/me-teams-summary.json`
- [ ] **AC-2**: `docs/gamechanger-api.md` contains a full response schema section for `GET /me/associated-players` documenting the `teams` map, `players` map (with `first_name`, `last_name`, `team_id`), and `associations` array structure, based on `data/raw/bulk-20260305-234522/me-associated-players.json`. PII fields explicitly noted.
- [ ] **AC-3**: `docs/gamechanger-api.md` contains a full response schema section for `GET /me/archived-teams` confirming it returns the same schema as `GET /me/teams` (bare JSON array of team objects), based on `data/raw/bulk-20260305-234522/me-archived-teams.json`
- [ ] **AC-4**: `docs/gamechanger-api.md` documents `GET /me/related-organizations` as returning HTTP 500 with web browser headers, with the exact error message (`Cannot read properties of undefined (reading 'page_starts_at')`) and a note that it succeeded via iOS app proxy capture
- [ ] **AC-5**: `docs/gamechanger-api.md` contains brief schema documentation for `GET /me/widgets` and `GET /me/schedule` based on their raw payloads, with coaching-relevance assessment (both are low-priority but should be documented for completeness)
- [ ] **AC-6**: The existing `GET /me/user` documentation is reviewed against `data/raw/bulk-20260305-234522/me-user.json` and any schema differences are noted (new fields, missing fields, type changes)
- [ ] **AC-7**: All new endpoint sections follow the existing documentation format in `docs/gamechanger-api.md` (status badge, field table, example JSON with redacted PII, key observations)
- [ ] **AC-8**: Each endpoint section documents the `Accept` header value where known from individual curl captures already recorded in `docs/gamechanger-api.md` (e.g., vendor-typed accepts like `event_box_score+json`), or notes "Accept header: not yet confirmed" if unknown. Note: `proxy/data/endpoint-log.jsonl` contains `request_content_type` (the request Content-Type), NOT the Accept header -- do not conflate the two fields.

## Technical Approach
1. Read each raw payload file in `data/raw/bulk-20260305-234522/` for the `/me/*` family
2. Cross-reference with `proxy/data/endpoint-log.jsonl` for request metadata (status codes, content-types, query parameters)
3. For each endpoint, add a new subsection to the existing "Me (additional)" section in `docs/gamechanger-api.md`
4. Promote endpoints from `OBSERVED (proxy log only)` to `CONFIRMED (web headers, schema documented)` where appropriate -- or note `HTTP 500 (web headers)` for the failing endpoints
5. Redact all PII (player names, user IDs, emails) in example snippets
6. For `me-schedule.json`, the file is large (34K+ tokens) -- read the first 100 lines to capture the schema structure, no need to document every event
