# E-002-05: Crawl opponent team data for all scheduled games

## Epic
[E-002: GameChanger Data Ingestion Pipeline](epic.md)

## Status
`DONE`

## Description
After this story is complete, the system will fetch opponent registry data and roster data for every opponent team that appears in the opponent registries of our configured teams, writing raw JSON to `data/raw/{season}/teams/{opponent_team_id}/`. Additionally, the crawler writes the full opponent registry for each owned team to `data/raw/{season}/teams/{team_id}/opponents.json`. This is what enables the "competitive intelligence" use case: scouting opponents before games.

## Context
When Lincoln plays a game, the opponent's team ID appears in the game-summaries data. This crawler uses those opponent IDs to fetch public roster data for those teams.

**Two data sources for opponent discovery**:
1. **`GET /teams/{team_id}/opponents`** (primary): Returns the full opponent registry with `progenitor_team_id` (canonical UUID) and `name` for each opponent. Supports pagination. `is_hidden` flag distinguishes active vs stale entries. This endpoint provides the most complete opponent list including teams from prior seasons.
2. **Game-summaries `game_stream.opponent_id`** (secondary): Each completed game record includes the opponent's team UUID. This provides opponent IDs for games actually played.

Both sources yield the same canonical UUIDs (`progenitor_team_id` == `pregame_data.opponent_id` from schedule, confirmed). The opponents endpoint is preferred because it includes all opponents (not just those with completed games) and provides team names directly.

**Confirmed**: The `/teams/{team_id}/players` endpoint accepts opponent team UUIDs directly (documented in `docs/gamechanger-api.md`). Opponent rosters are accessible.

## Acceptance Criteria
- [ ] **AC-1**: Given configured owned teams, when `OpponentCrawler().crawl_all()`, then for each owned team, the opponents endpoint is called (`GET /teams/{team_id}/opponents`, with pagination) and the full response is written to `data/raw/{season}/teams/{team_id}/opponents.json`.
- [ ] **AC-2**: From the opponents response, unique `progenitor_team_id` values (where present and `is_hidden: false`) are extracted. For each unique opponent team UUID (excluding IDs already in `config/teams.yaml` as owned teams), an attempt is made to fetch the opponent's roster via `/teams/{progenitor_team_id}/players`.
- [ ] **AC-3**: Data for opponents is written to the same directory structure as owned teams: `data/raw/{season}/teams/{opponent_team_id}/roster.json`.
- [ ] **AC-4**: Given an opponent whose data is inaccessible (403 Forbidden), the error is logged at WARNING level (not ERROR -- this is expected behavior), and the crawl continues. The opponent is not retried.
- [ ] **AC-5**: Given an opponent record without `progenitor_team_id` (manually-entered or placeholder opponents, ~14% of records), the opponent is skipped with an INFO log noting the name and that no canonical team UUID is available.
- [ ] **AC-6**: Opponent roster files follow the same idempotency rules as equivalent files for owned teams (24-hour freshness). The opponents.json registry file uses the same freshness check.
- [ ] **AC-7**: After the run, the crawler logs: total unique opponents found (from registry), opponents with progenitor_team_id, opponents without progenitor_team_id, opponents hidden (is_hidden: true, skipped), opponents successfully crawled (roster fetched), opponents with access denied, opponents with unexpected errors.

## Technical Approach
`src/gamechanger/crawlers/opponent.py`. Two-phase approach:

**Phase 1 -- Fetch opponent registry:**
For each owned team, call `GET /teams/{team_id}/opponents` (with pagination via `x-next-page` header / `start_at` query param). Write the full response to `data/raw/{season}/teams/{team_id}/opponents.json`. Accept header: `application/vnd.gc.com.opponent_team:list+json; version=0.0.0`.

**Phase 2 -- Fetch opponent rosters:**
Extract unique `progenitor_team_id` values from all opponents.json files (filtering out `is_hidden: true` and entries without `progenitor_team_id`). For each unique opponent UUID (excluding owned teams), call `/teams/{progenitor_team_id}/players` to fetch the roster.

The `OpponentCrawler` should reuse `RosterCrawler` logic for the roster-fetching phase rather than duplicating it. Consider passing opponent team IDs as a parameter to `RosterCrawler.crawl_team()`.

`crawl_all()` returns a `CrawlResult` (defined in E-002-01, `src/gamechanger/crawlers/__init__.py`).

## Notes
This story is the highest-risk story in E-002 because it depends on GameChanger's access controls for non-owned teams. If opponent roster data is entirely locked down, this story should adjust to capture only the opponents registry (opponents.json) and note that opponent roster data must be sourced from boxscore data (E-002-03 captures both teams' batting/pitching lines in every game).

**Opponent registry value even without roster access**: The opponents.json file is valuable on its own -- it provides opponent team names and canonical UUIDs that can be used to identify teams in boxscore responses (where the opponent key is a UUID). Even if roster fetching fails for all opponents, the registry crawl succeeds independently.

**Public endpoint fallback**: If authenticated roster calls fail for opponents, the public roster endpoint (`GET /teams/public/{public_id}/players`, no auth needed) is available if we can resolve the opponent's `public_id`. The UUID-to-public_id bridge (`GET /teams/{team_id}/public-team-profile-id`) requires auth but only needs to be called once per opponent. This fallback is not in scope for this story but is documented as a future option.
