# E-002-11: Distinguish 401 from 403 in GameChangerClient

## Epic
[E-002: GameChanger Data Ingestion Pipeline](epic.md)

## Status
`DONE`

## Description
After this story is complete, the `GameChangerClient` will raise distinct exceptions for 401 (token expired, global, unrecoverable) and 403 (forbidden, per-resource, possibly expected). Callers like `OpponentCrawler` can then correctly distinguish between a globally dead token and a per-opponent access denial, avoiding the current bug where a token expiry mid-crawl is silently misclassified as benign per-opponent access denial.

## Context
This is a codex review remediation story (P2 should-fix). Currently `GameChangerClient` raises the same `CredentialExpiredError` for both 401 and 403 responses (lines 145-150 in `get_paginated()`, lines 244-249 in `_get_with_retries()`). The `OpponentCrawler` catches `CredentialExpiredError` at line 253 and treats it as "expected per-opponent access denied" (WARNING log, continue to next opponent). This is correct for 403 but wrong for 401: if the token expires mid-run, every subsequent opponent also gets a 401, and the crawler logs each one as a benign warning and iterates uselessly through the remaining opponents.

The fix has two parts:
1. **Client**: Split `CredentialExpiredError` into two exceptions (or add a subclass) so callers can distinguish 401 from 403.
2. **OpponentCrawler**: Catch 403 (continue) separately from 401 (abort or re-raise).

## Acceptance Criteria
- [ ] **AC-1**: `GameChangerClient` raises a distinct exception for 401 vs 403. Two approaches are acceptable: (a) new `ForbiddenError` subclass of `CredentialExpiredError` for 403 (backward-compatible -- existing `except CredentialExpiredError` still catches both), or (b) two independent exceptions. The implementing agent should choose (a) for backward compatibility.
- [ ] **AC-2**: Given a 401 response during opponent roster crawling, when `OpponentCrawler` catches it, then the crawler logs an ERROR and aborts immediately (re-raises or returns early). It does NOT continue to the next opponent.
- [ ] **AC-3**: Given a 403 response during opponent roster crawling, when `OpponentCrawler` catches it, then the existing behavior is preserved: WARNING log, increment `access_denied` counter, continue to next opponent.
- [ ] **AC-4**: Given a 401 response during a non-paginated `get()` call, the same distinct exception is raised (consistent behavior across `get()` and `get_paginated()`).
- [ ] **AC-5**: Existing tests that catch `CredentialExpiredError` continue to pass (backward compatibility via subclass or import alias). New tests cover: 401 raises the token-expired exception, 403 raises the forbidden exception, `OpponentCrawler` aborts on 401 but continues on 403.

## Technical Approach
**Part 1 -- Client exception split** (`src/gamechanger/client.py`):

Add a new `ForbiddenError(CredentialExpiredError)` subclass. Update the 401/403 handling in both `get_paginated()` and `_get_with_retries()`:
- 401 -> raise `CredentialExpiredError` (unchanged)
- 403 -> raise `ForbiddenError` (new subclass)

Because `ForbiddenError` inherits from `CredentialExpiredError`, any existing `except CredentialExpiredError` clause still catches 403 responses -- backward compatible.

**Part 2 -- OpponentCrawler fix** (`src/gamechanger/crawlers/opponent.py`):

In `_crawl_opponent_rosters()`, change the exception handling at line 253:
```python
# Before:
except CredentialExpiredError as exc:
    # Treats all auth errors as per-opponent access denial

# After:
except ForbiddenError as exc:
    # 403 -- per-opponent access denial, expected and benign
    ...
except CredentialExpiredError as exc:
    # 401 -- token expired, abort the crawl
    logger.error("Token expired during opponent crawl: %s", exc)
    raise
```

The order matters: `ForbiddenError` (subclass) must be caught before `CredentialExpiredError` (parent).
