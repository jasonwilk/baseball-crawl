# E-002-07a: Load raw game JSON into database

## Epic
[E-002: GameChanger Data Ingestion Pipeline](epic.md)

## Status
`DONE`

## Description
After this story is complete, the system will have a `GameLoader` that reads boxscore JSON files from `data/raw/` and upserts game records and per-player batting/pitching lines into the appropriate database tables. Coaches will be able to query game results and individual player game performances from SQL.

## Context
This loader handles three related record types from the same raw files:
1. **Game records** (one row per game in the `games` table)
2. **Per-player batting lines** (rows in `player_game_batting`)
3. **Per-player pitching lines** (rows in `player_game_pitching`)

These come from the per-game boxscore files written by E-002-03 (`GET /game-stream-processing/{game_stream_id}/boxscore`).

### Boxscore Response Shape (confirmed 2026-03-04)

The boxscore response is a JSON object with **team identifiers as top-level keys** (two keys total: own team + opponent). Key formats are asymmetric: own team uses a public_id slug (alphanumeric), opponent uses a UUID.

Per team:
- `players[]` -- full roster context (all rostered players, not just lineup)
- `groups[]` -- array of stat group objects:
  - `category: "lineup"` (batting): main stats `{AB, R, H, RBI, BB, SO}` per player + sparse `extra[]` for 2B, 3B, HR, TB, HBP, SB, CS, E
  - `category: "pitching"`: main stats `{IP, H, R, ER, BB, SO}` per player + sparse `extra[]` for WP, HBP, #P, TS, BF
- `team_stats` per group: aggregate totals (AB, R, H, RBI, BB, SO for lineup; IP, H, R, ER, BB, SO for pitching)
- Batting order is implicit (list position in `stats[]`). `player_text` has position(s). `is_primary` distinguishes starters from subs.
- Pitching: `player_text` has decision (W/L/SV/""). Listed in order pitched.

### ID Mapping: game_stream_id -> game_id

Raw boxscore files are named `{game_stream_id}.json`. The `games.game_id` PK uses `event_id` (= `game_stream.game_id` from game-summaries). The loader must map between these IDs. Options:
1. Read game-summaries data from disk to build a `game_stream_id -> event_id` mapping.
2. Use a mapping file written by E-002-03 or E-002-08.

The implementing agent should read game-summaries files to build this mapping, since those files are already on disk (written by E-002-02).

### Field Mapping: Boxscore -> Schema

| Boxscore field | Schema column (`player_game_batting`) | Notes |
|---|---|---|
| stats.AB | ab | Direct |
| stats.H | h | Direct |
| stats.RBI | rbi | Direct |
| stats.BB | bb | Direct |
| stats.SO | so | Direct |
| extra 2B | doubles | Sparse -- absent means 0 |
| extra 3B | triples | Sparse -- absent means 0 |
| extra HR | hr | Sparse -- absent means 0 |
| extra SB | sb | Sparse -- absent means 0 |
| stats.R | (not in schema) | Runs scored per player -- schema lacks this column |

| Boxscore field | Schema column (`player_game_pitching`) | Notes |
|---|---|---|
| stats.IP | ip_outs | **Conversion needed** -- schema stores outs (3 per inning). See Open Questions. |
| stats.H | h | Direct |
| stats.ER | er | Direct |
| stats.BB | bb | Direct |
| stats.SO | so | Direct |
| extra WP | (not in schema) | Wild pitches -- schema lacks this column |
| extra HBP | (not in schema) | Hit batters -- schema lacks this column |
| extra #P | (not in schema) | Pitch count -- schema lacks this column for game level |
| extra TS | (not in schema) | Strikes thrown -- schema lacks this column for game level |
| extra BF | (not in schema) | Batters faced -- schema lacks this column for game level |

**Schema gaps**: The `player_game_batting` table lacks a `runs` column, and `player_game_pitching` lacks columns for WP, HBP, #P, TS, BF. These stats are available in the boxscore but cannot be stored in the current schema. The loader should log a DEBUG note about unmappable fields. Schema expansion can be addressed in a future story if needed.

## Acceptance Criteria
- [ ] **AC-1**: Given `games/{game_stream_id}.json` files (boxscore responses), when `GameLoader().load_all()`, then each completed game is upserted into the `games` table with: `game_id` (mapped from `event_id` via game-summaries), `game_date`, `home_team_id`, `away_team_id`, `home_score`, `away_score`, `season_id`. (Exact column names per the delivered `migrations/001_initial_schema.sql`.)
- [ ] **AC-2**: Given the same game files, player batting lines are upserted into `player_game_batting` and pitching lines into `player_game_pitching`: one row per (player_id, game_id). Running twice is idempotent. Sparse extras (2B, 3B, HR, SB for batting) are reconstructed from the `extra[]` array; absent values are stored as 0.
- [ ] **AC-3**: The loader returns a `LoadResult` (loaded, skipped, errors counts).
- [ ] **AC-4**: Given a game record where a `player_id` does not exist in the `players` table, the loader inserts a stub player row (`player_id`, `first_name='Unknown'`, `last_name='Unknown'`) before inserting the stat row, and logs a WARNING including the player_id. The stat row is written successfully. (FK-safe stub-player pattern.)
- [ ] **AC-5**: Given the same game appearing under multiple team directories (see E-002-03 Notes on dedup), the loader upserts by `game_id` so duplicate raw files produce the same database state.
- [ ] **AC-6**: The loader correctly handles the asymmetric team key format in boxscore responses (own team = public_id slug, opponent = UUID). Both teams' batting and pitching lines are loaded regardless of key format.
- [ ] **AC-7**: The `games` table `home_team_id` and `away_team_id` are populated by cross-referencing with game-summaries data (`home_away` field). The boxscore response itself does not include home/away designation.
- [ ] **AC-8**: Before inserting `games` or player stat rows, the loader ensures prerequisite FK rows exist: (a) `teams` rows for both `home_team_id` and `away_team_id` (upsert with name from boxscore or game-summaries if available; opponent teams get `is_owned=0`), (b) a `seasons` row for the `season_id`. This prevents FK constraint violations on the `games` table.

## Technical Approach
`src/gamechanger/loaders/game_loader.py` -- `GameLoader` class. Follows the same pattern as `RosterLoader` (E-002-06). Extract field mapping to separate `_map_batting()` and `_map_pitching()` methods for testability.

**Asymmetric key handling**: Detect whether a top-level key is a UUID (contains dashes, 36 chars) or a public_id slug (alphanumeric, no dashes). This determines which team is "own" vs "opponent" in the response.

**Home/away resolution**: The boxscore does not contain `home_away`. Read game-summaries data from disk to get `home_away` for each game. Combined with `game_stream.opponent_id`, this resolves which team is home and which is away.

**IP conversion**: The boxscore `IP` field may need conversion to `ip_outs`. If IP represents whole innings, multiply by 3. If IP represents outs directly, no conversion. The implementing agent should inspect actual data and handle accordingly, with a clear comment documenting the assumption.

The implementing agent must read the delivered `migrations/001_initial_schema.sql` to get exact column names and types. Do not rely on table/column names assumed in this story file.

## Notes
- This story does NOT load season-aggregate stats -- that is E-002-07b.
- The loaders do NOT make HTTP calls. DoD requirement about mocking HTTP does not apply -- mock the database or use in-memory SQLite instead.
- Orphaned player IDs are handled via the stub-player pattern: insert a minimal player row before the stat row to satisfy FK constraints. Stub players are identifiable by `first_name='Unknown'` and `last_name='Unknown'` and are backfilled by a later roster crawl.
- Stats available in the boxscore but not in the current schema (batting: R; pitching: WP, HBP, #P, TS, BF) are silently dropped with a DEBUG log. Schema expansion for these fields is a future concern.
