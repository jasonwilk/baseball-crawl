# E-002-09: Wire GameLoader and SeasonStatsLoader into load.py

## Epic
[E-002: GameChanger Data Ingestion Pipeline](epic.md)

## Status
`DONE`

## Description
After this story is complete, `scripts/load.py` will orchestrate all three loaders (roster, game, season-stats) in dependency order. Currently only the roster loader is wired up -- `GameLoader` and `SeasonStatsLoader` exist but are not registered in the loader registry, so running `python scripts/load.py` leaves the `games`, `player_game_*`, and `player_season_*` tables untouched.

## Context
This is a codex review remediation story (P1 blocker). E-002-08 delivered `scripts/load.py` with a loader registry pattern, but only the roster loader was registered. The game loader (E-002-07a) and season stats loader (E-002-07b) were delivered separately and never wired in. The orchestrator contract requires all available loaders to be orchestrated.

The existing pattern is clear: each loader has a `_run_*_loader()` wrapper function that iterates over teams/files, calls the loader, and returns an aggregated `LoadResult`. The roster loader wrapper at lines 49-77 is the template.

## Acceptance Criteria
- [ ] **AC-1**: `scripts/load.py` registers three loaders in dependency order: `roster`, `game`, `season-stats`. The `_LOADERS` list at line 82 contains all three entries.
- [ ] **AC-2**: Given raw game JSON files on disk (from E-002-03), when `python scripts/load.py` (or `python scripts/load.py --loader game`), then `GameLoader` is invoked and game/player-game records are loaded into the database.
- [ ] **AC-3**: Given raw season-stats JSON files on disk (from E-002-04), when `python scripts/load.py` (or `python scripts/load.py --loader season-stats`), then `SeasonStatsLoader` is invoked and player-season records are loaded into the database.
- [ ] **AC-4**: The `--loader` filter accepts `roster`, `game`, and `season-stats` as valid choices. The module docstring and `--help` output reflect all three loader names.
- [ ] **AC-5**: Running `python scripts/load.py --dry-run` lists all three loaders in order.
- [ ] **AC-6**: Existing roster loader tests continue to pass. New tests cover: game loader wiring (called with correct args), season-stats loader wiring (called with correct args), loader ordering in `_LOADERS`, `--loader game` and `--loader season-stats` filter paths.

## Technical Approach
Edit `scripts/load.py`:

1. Import `GameLoader` from `src.gamechanger.loaders.game_loader` and `SeasonStatsLoader` from `src.gamechanger.loaders.season_stats_loader`.
2. Add `_run_game_loader(db, config, data_root) -> LoadResult` wrapper. It should iterate over owned teams, find the team's `games/` directory, and call `GameLoader(db, config.season, team.id).load_all(team_dir)` for each. The `GameLoader` constructor requires `(db, season_id, owned_team_id)` -- read `/workspaces/baseball-crawl/src/gamechanger/loaders/game_loader.py` to confirm the exact signature.
3. Add `_run_season_stats_loader(db, config, data_root) -> LoadResult` wrapper. It should iterate over owned teams, find `stats.json`, and call `SeasonStatsLoader(db).load_file(path, team_id, config.season)`. Read `/workspaces/baseball-crawl/src/gamechanger/loaders/season_stats_loader.py` to confirm the exact signature.
4. Register all three in `_LOADERS` in dependency order: roster first (players must exist), then game (game records + per-game stats), then season-stats (season aggregates).
5. Update the module docstring to list all three loader names.

Add tests in `tests/test_scripts/test_load_orchestrator.py` (extend existing file).
