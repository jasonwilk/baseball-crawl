# E-002-03: Crawl game stats (box score) and write raw JSON

## Epic
[E-002: GameChanger Data Ingestion Pipeline](epic.md)

## Status
`DONE`

## Description
After this story is complete, the system will have a `GameStatsCrawler` that reads game-summaries data from previously crawled files, fetches per-game box score data for each completed game via `GET /game-stream-processing/{game_stream_id}/boxscore`, and writes raw JSON to `data/raw/{season}/teams/{team_id}/games/{game_stream_id}.json`. This is the most data-dense crawl in the pipeline and provides per-player batting lines, pitching lines, and sparse extra stats (doubles, triples, HR, SB, pitch counts, etc.) for both teams in every completed game.

## Context
Game stats are the core product for coaching intelligence. Box scores contain per-batter performance (AB, R, H, RBI, BB, SO + sparse extras) and per-pitcher performance (IP, H, R, ER, BB, SO + sparse extras), plus team totals. The boxscore endpoint returns data for BOTH teams in a single response, making it the primary source for scouting opponents at the game level.

This crawler reads game-summaries files written by E-002-02 rather than calling the game-summaries API directly, which keeps crawlers decoupled.

**Critical ID mapping**: The boxscore endpoint uses `game_stream.id` from the game-summaries response -- NOT `event_id`, NOT `game_stream.game_id`. In game-summaries: `event_id == game_stream.game_id` always, but `game_stream.id != game_stream.game_id` always. The schedule endpoint does NOT expose `game_stream.id` -- you must go through game-summaries. See `docs/gamechanger-api.md` (GET /game-stream-processing/{game_stream_id}/boxscore) for the full endpoint specification.

## Acceptance Criteria
- [ ] **AC-1**: Given game-summaries files for all configured teams, when `GameStatsCrawler().crawl_all()`, then for each completed game in each game-summaries file, a file is written to `data/raw/{season}/teams/{team_id}/games/{game_stream_id}.json`, where `game_stream_id` is the `game_stream.id` value from the game-summaries record.
- [ ] **AC-2**: Only games with `game_status: "completed"` (from the game-summaries file at `data/raw/{season}/teams/{team_id}/game_summaries.json`, written by E-002-02) are crawled. Scheduled or cancelled games are skipped with an INFO log.
- [ ] **AC-3**: Idempotency: if a game file already exists (any age -- completed game stats do not change), it is not re-fetched.
- [ ] **AC-4**: The written JSON is the unmodified API response from `GET /game-stream-processing/{game_stream_id}/boxscore`.
- [ ] **AC-5**: Given an API error for a specific game, the error is logged with the `game_stream_id` and `event_id` and crawl continues.
- [ ] **AC-6**: At the end of a run, the crawler logs a summary: total games found, games already cached, games fetched, games errored.
- [ ] **AC-7**: The Accept header `application/vnd.gc.com.event_box_score+json; version=0.0.0` is used for the boxscore request. Auth headers (`gc-token`, `gc-device-id`) are included via `GameChangerClient`.

## Technical Approach
`src/gamechanger/crawlers/game_stats.py`. The crawler:

1. Reads game-summaries JSON from `data/raw/{season}/teams/{team_id}/game_summaries.json` (written by E-002-02).
2. Filters to `game_status: "completed"` records.
3. Extracts `game_stream.id` from each completed game record (this is the API path parameter).
4. For each `game_stream_id`, checks if `data/raw/{season}/teams/{team_id}/games/{game_stream_id}.json` exists. If it does, skip (AC-3).
5. Calls `GET /game-stream-processing/{game_stream_id}/boxscore` with the correct Accept header.
6. Writes the raw response to disk.

**API endpoint**: `GET /game-stream-processing/{game_stream_id}/boxscore`
- Path parameter: `game_stream.id` from game-summaries (NOT `event_id`)
- Accept header: `application/vnd.gc.com.event_box_score+json; version=0.0.0`
- Auth: `gc-token` + `gc-device-id` via `GameChangerClient`
- No `gc-user-action` header required (confirmed absent from capture)

**Response shape**: JSON object with team identifiers as top-level keys (asymmetric: own team = public_id slug, opponent = UUID). Each team has `players[]` (full roster), `groups[]` (lineup + pitching stat groups). See `docs/gamechanger-api.md` for complete schema.

**Mapping game_stream_id back to event_id**: The crawler should also persist a lightweight mapping of `game_stream_id -> event_id` (extractable from game-summaries data where `event_id == game_stream.game_id`). This mapping is needed by the loader (E-002-07a) to populate the `games.game_id` column (which uses `event_id`). Implementation options: include the mapping in the manifest, or have the loader read game-summaries directly.

`crawl_all()` returns a `CrawlResult` (defined in E-002-01, `src/gamechanger/crawlers/__init__.py`).

## Notes
The game file is keyed on `{team_id}/{game_stream_id}` because the same game may appear in two teams' game-summaries (if both teams are in `config/teams.yaml`). Write the file under the team that "owns" the game-summaries entry to avoid coordination complexity. If both teams are ours, writing it twice (same content) is acceptable.

**Dedup implication for E-002-07a**: When loading game data from raw files, the same game may appear under multiple team directories. The loader must handle this (upsert by game_id ensures idempotency).

**IP representation**: The boxscore returns `IP` as integer values in the pitching stats. Whether these represent whole innings or outs needs confirmation from a game with a partial inning. The loader (E-002-07a) will handle any necessary conversion to the schema's `ip_outs` convention.

**Sparse extras pattern**: Some stats (2B, 3B, HR, TB, SB, CS, HBP, E for batting; WP, HBP, #P, TS, BF for pitching) are in a sparse `extra` array where only players with non-zero values appear. Zero-value players are omitted entirely. The loader must handle this reconstruction.
