# E-002-10: Add 5xx retry/backoff to get_paginated()

## Epic
[E-002: GameChanger Data Ingestion Pipeline](epic.md)

## Status
`DONE`

## Description
After this story is complete, `GameChangerClient.get_paginated()` will retry on 5xx server errors with exponential backoff, matching the behavior already implemented in `get()`. Currently `get_paginated()` raises `GameChangerAPIError` immediately on any non-200/401/403/429 response, meaning a single transient 500 error on any page fails the entire paginated crawl.

## Context
This is a codex review remediation story (P1 blocker). The `get()` method (lines 208-286 in `client.py`) has proper 3-retry exponential backoff for 5xx errors via `_get_with_retries()`. The `get_paginated()` method (lines 85-170) handles 200, 401/403, and 429, but falls through to a bare `raise GameChangerAPIError` on line 165 for all other status codes including 5xx. The docstring at line 110 promises "GameChangerAPIError: On 5xx responses after retries" but the implementation does not retry.

Both `ScheduleCrawler` and `OpponentCrawler` use `get_paginated()` for their paginated endpoints. A transient 500 on page 2 of game-summaries would fail the entire schedule crawl.

## Acceptance Criteria
- [ ] **AC-1**: Given a 5xx response on a paginated request, when `get_paginated()` is called, then the request is retried up to 3 times with exponential backoff (1s, 2s, 4s delays) before raising `GameChangerAPIError`. This matches the existing `_get_with_retries()` behavior.
- [ ] **AC-2**: Given a 5xx that succeeds on retry, the pagination continues from the same page (the retry replays the failed page, not the entire sequence).
- [ ] **AC-3**: Given a non-5xx, non-200, non-401/403, non-429 status code (e.g., 418), `get_paginated()` raises `GameChangerAPIError` immediately without retrying. Only 5xx codes trigger retry.
- [ ] **AC-4**: Retry attempts are logged at WARNING level with the status code, URL, attempt number, and backoff duration (matching the existing `_get_with_retries()` log format).
- [ ] **AC-5**: Existing tests for `get_paginated()` continue to pass. New tests cover: 5xx retry succeeds on second attempt, 5xx exhausts all retries and raises, non-5xx error raises immediately, retry logs at WARNING level.

## Technical Approach
Refactor `get_paginated()` in `src/gamechanger/client.py` to use retry logic for 5xx responses. Two approaches:

**Option A (preferred)**: Extract the retry logic from `_get_with_retries()` into a reusable pattern and apply it within the `get_paginated()` while loop. The per-page request (lines 124-126) gets wrapped in a retry loop that handles 5xx responses with exponential backoff.

**Option B**: Refactor `get_paginated()` to use `_get_with_retries()` for each individual page request. This is cleaner but requires `_get_with_retries()` to support the extra headers and pagination context that `get_paginated()` adds.

The implementing agent should choose whichever approach results in less code duplication and cleaner tests.
