# E-063-01: DEV_USER_EMAIL Production Guard

## Epic
[E-063: Dashboard Auth Hardening](./epic.md)

## Status
`DONE`

## Description
After this story is complete, the application will refuse to start when `DEV_USER_EMAIL` is set and `APP_ENV=production`. This prevents an accidental `.env` misconfiguration from silently disabling all authentication and granting admin access to every request.

## Context
The dev bypass in `SessionMiddleware` auto-creates an admin user and attaches it to every request when `DEV_USER_EMAIL` is set. There is no guard preventing this from running in production. Since the bypass grants `is_admin=1`, a misconfigured production `.env` silently disables all authentication AND authorization. SE consultation recommended a startup-time check that raises `RuntimeError` -- surfacing the problem immediately rather than silently denying at request time.

## Acceptance Criteria
- [ ] **AC-1**: Given `DEV_USER_EMAIL=anything` and `APP_ENV=production` in the environment, when the app starts, then `SessionMiddleware.__init__` raises `RuntimeError` with a message indicating the misconfiguration.
- [ ] **AC-2**: Given `DEV_USER_EMAIL=anything` and `APP_ENV=development` (or unset), when the app starts, then the existing dev bypass behavior is unchanged (info log, auto-session).
- [ ] **AC-3**: Given `DEV_USER_EMAIL` is empty or unset and `APP_ENV=production`, when the app starts, then startup proceeds normally with no error.
- [ ] **AC-4**: Tests cover all three scenarios (production+set, dev+set, production+unset) by patching environment variables and asserting on `RuntimeError` or successful init.

## Technical Approach
The check belongs in `SessionMiddleware.__init__` in `src/api/auth.py`, where the `DEV_USER_EMAIL` env var is already read and logged. The guard should check both env vars and raise before the info log line. Tests should patch `os.environ` and instantiate the middleware class.
