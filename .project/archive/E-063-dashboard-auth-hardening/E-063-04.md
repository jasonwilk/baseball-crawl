# E-063-04: Invalidate Stale Magic Link Tokens

## Epic
[E-063: Dashboard Auth Hardening](./epic.md)

## Status
`DONE`

## Description
After this story is complete, when a new magic link is issued for a user, all prior unused magic link tokens for that user will be invalidated. This prevents old intercepted tokens from being usable after the user has requested a fresh link.

## Context
Currently, `POST /auth/login` creates a new `magic_link_tokens` row each time a user requests a login link. Old tokens remain valid until their 15-minute TTL expires. If a user requests 5 links (e.g., retrying because email was slow), all 5 remain usable. An attacker who intercepts one old link can use it even after the user logged in with a newer one. SE consultation recommended invalidating (setting `used_at`) all prior unused tokens for the same user when a new link is issued.

## Acceptance Criteria
- [ ] **AC-1**: Given a user has an existing unused magic link token, when a new magic link is issued for that user via `POST /auth/login`, then the prior token's `used_at` is set to the current timestamp (making it unusable).
- [ ] **AC-2**: Given a user requests a magic link and then requests another, when the user tries to verify the first (older) token via `GET /auth/verify`, then verification fails (token is marked used).
- [ ] **AC-3**: Given a user requests a magic link and then requests another, when the user verifies the second (newer) token, then verification succeeds and a session is created.
- [ ] **AC-4**: Tests cover the invalidation behavior: issue token A, issue token B, verify A fails, verify B succeeds.

## Technical Approach
In `src/api/routes/auth.py`, in the `post_login` function, add an `UPDATE` statement before the `INSERT` that sets `used_at = datetime('now')` on all rows in `magic_link_tokens` where `user_id` matches and `used_at IS NULL`. This should be within the same transaction as the new token insert.
