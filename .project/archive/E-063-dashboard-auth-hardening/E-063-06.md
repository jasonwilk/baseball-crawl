# E-063-06: Fail Closed on Missing Auth Tables

## Epic
[E-063: Dashboard Auth Hardening](./epic.md)

## Status
`DONE`

## Description
After this story is complete, when the auth tables are missing (migrations not applied), the session middleware will return a 503 Service Unavailable instead of allowing requests through unauthenticated. This prevents dashboard pages from rendering without a user context when the app starts in a broken state.

## Context
The session middleware in `src/api/auth.py` has two `OperationalError` catch blocks (lines 272-279 for the dev bypass path, lines 289-296 for the cookie validation path). Both catch "no such table" errors and call `call_next(request)` -- passing the request through with no `request.state.user` set. Admin routes happen to redirect to login because `_require_admin()` checks `if not user:`, but dashboard routes render with an empty user dict, making `/dashboard` accessible without authentication when migrations haven't been applied. SE consultation confirmed this is the root cause of the reported bug where dashboard was accessible without auth.

## Acceptance Criteria
- [ ] **AC-1**: Given the `sessions` or `users` table does not exist, when any protected route is requested (not in `/auth/*`, `/health`, `/static/*`), then the middleware returns a 503 status code with a plain-text body (e.g., "Service temporarily unavailable").
- [ ] **AC-2**: The 503 behavior applies to both the dev bypass path and the cookie validation path in `SessionMiddleware.dispatch`.
- [ ] **AC-3**: Given the `sessions` and `users` tables exist, when a protected route is requested, then normal auth behavior is unchanged (session validation, redirect to login on failure).
- [ ] **AC-4**: Other `OperationalError` types (not "no such table") continue to be re-raised as before.
- [ ] **AC-5**: Tests cover: (a) missing table returns 503, (b) normal auth still works when tables exist, (c) non-table OperationalErrors propagate.

## Technical Approach
In `src/api/auth.py`, in the `SessionMiddleware.dispatch` method, replace the two `OperationalError` catch blocks that currently call `call_next(request)` with blocks that return a `Response` with status 503. Use `starlette.responses.PlainTextResponse` or `Response` with appropriate content type. Keep the `if "no such table" in str(exc).lower()` discriminator to distinguish schema errors from other `OperationalError`s.
