# E-063-05: Magic Link Rate Limiting

## Epic
[E-063: Dashboard Auth Hardening](./epic.md)

## Status
`DONE`

## Description
After this story is complete, the magic link issuance endpoint will enforce a per-user cooldown period. If a user (by email) requests a magic link within the cooldown window of their last issuance, no new link is sent but the same "check your email" response is shown (to prevent user enumeration).

## Context
Currently `POST /auth/login` issues a magic link for any known email on every POST with no rate limiting. An attacker could flood a user's inbox with magic link emails, abusing the email sending service. SE consultation recommended tracking the last issuance time and enforcing a cooldown (e.g., 60 seconds), returning the same confirmation page regardless of whether a link was actually sent.

This story depends on E-063-04 (stale token invalidation) because the rate limiting query checks the `magic_link_tokens` table and the invalidation behavior from E-063-04 ensures only one active token exists per user at a time.

## Acceptance Criteria
- [ ] **AC-1**: Given a user requested a magic link less than 60 seconds ago (strictly less than), when they submit `POST /auth/login` again, then no new magic link is sent (no email, no new `magic_link_tokens` row), but the "check your email" page is still returned (same response as a successful issuance).
- [ ] **AC-2**: Given a user requested a magic link 60 or more seconds ago, when they submit `POST /auth/login`, then a new magic link is issued normally.
- [ ] **AC-3**: Given an unknown email address, when they submit `POST /auth/login`, then the "check your email" page is returned (unchanged behavior -- no user enumeration).
- [ ] **AC-4**: The cooldown check uses the `created_at` column on `magic_link_tokens` (already exists in `migrations/003_auth.sql`) for the most recent row for the user. No new columns or migrations are needed.
- [ ] **AC-5**: Tests cover: (a) first request succeeds, (b) second request within cooldown is suppressed (no email sent), (c) request after cooldown succeeds.

## Technical Approach
In `src/api/routes/auth.py`, in the `post_login` function, after looking up the user, query `magic_link_tokens` for the most recent row for that `user_id` and check if its `created_at` is within the cooldown window (< 60 seconds ago). If so, skip the INSERT and email send but still return the same `check_email.html` template. The `created_at` column already exists on `magic_link_tokens` (`migrations/003_auth.sql`).
