# E-034-03: Create Spec-Review Rubric and Shell Wrapper

## Epic
[E-034: Codex Review Integration](epic.md)

## Status
`DONE`

## Description
After this story is complete, the repo will contain a checked-in Codex spec-review rubric at `.project/codex-spec-review.md` and an executable shell wrapper at `scripts/codex-spec-review.sh` that invokes Codex to review planning artifacts (epics and stories) against project workflow contracts. The PM (via an implementing agent) will be able to request a spec review of any epic directory before marking it READY.

## Context
The code-review lane (E-034-01) handles reviewing code changes. This story creates a second, distinct review lane for planning artifacts. Spec review and code review have fundamentally different evaluation criteria: spec review checks acceptance criteria clarity, dependency correctness, agent routing, story sizing, and workflow compliance -- not bugs, tests, or code style.

The PM cannot run scripts directly (no Bash tool). The intended workflow is: PM requests a spec review by dispatching to a `general-dev` agent, who runs the helper and returns Codex's output. The PM then incorporates findings into the epic before setting READY. E-034-04 handles the PM workflow integration.

See the epic Technical Notes (Lane B: Spec Review) for rubric content principles and CLI mode guidance.

## Acceptance Criteria
- [ ] **AC-1**: A file exists at `.project/codex-spec-review.md` containing a stable rubric for reviewing planning artifacts (epics and stories), not application code.
- [ ] **AC-2**: The rubric instructs Codex to evaluate, at minimum: (a) acceptance-criteria clarity and testability, (b) dependency correctness and sequencing, (c) file-conflict / parallel-execution risks, (d) story sizing and vertical slicing, (e) agent-routing correctness, (f) mismatch between epic claims and current repo reality, (g) missing expert consultation where PM workflow would normally require it, (h) unclear Definition of Done for implementing agents, (i) weak distinction between implemented reality and planned future state, (j) explicit "no findings" statement when the spec is clean.
- [ ] **AC-3**: The rubric instructs Codex to read `/workspaces/baseball-crawl/CLAUDE.md`, `/.claude/rules/workflow-discipline.md`, `/.claude/rules/dispatch-pattern.md`, and `/.claude/agents/product-manager.md` for project workflow context.
- [ ] **AC-4**: A file exists at `scripts/codex-spec-review.sh` that is executable (`chmod +x`).
- [ ] **AC-5**: The helper accepts a required positional argument: the path to an epic directory (e.g., `epics/E-034-codex-review`). It validates the directory exists and contains an `epic.md` file, failing with a clear error if not.
- [ ] **AC-6**: The helper includes the stable rubric plus the contents of all `.md` files in the target epic directory in the Codex prompt or review packet.
- [ ] **AC-7**: The helper accepts an optional `--note "text"` flag (or `--note-file /path/to/file`) allowing the caller to include a small runtime context handoff summarizing what the epic is trying to accomplish, what changed, and what the PM is unsure about. When provided, the note is included in the Codex prompt. When omitted, the review proceeds without it.
- [ ] **AC-8**: The runtime note mechanism is ephemeral -- no files are created or persisted by the helper beyond its stdout/stderr output. No Codex memory system is created.
- [ ] **AC-9**: The implementation uses the correct Codex CLI entry point for this use case. The implementing agent MUST verify the actual installed CLI syntax (run `codex --help`, `codex exec --help`, or equivalent) before finalizing the helper. Document the chosen CLI mode in a comment at the top of the script.
- [ ] **AC-10**: Neither the rubric nor the helper contains hard-coded model names. Model selection is delegated to the user's Codex configuration.
- [ ] **AC-11**: The helper uses `set -euo pipefail` for safe bash execution.
- [ ] **AC-12**: The helper handles the "codex not installed / unavailable" path with a clear, intentional error message (exit non-zero, message to stderr).

## Technical Approach
1. Run `codex --help` to survey available subcommands. Determine the correct CLI entry point for non-diff review of file contents. Likely candidates: `codex exec`, `codex` with appropriate flags, or piping assembled content to the CLI.
2. Create `.project/codex-spec-review.md` with the spec-review rubric. Keep it concise and focused on planning quality, not code quality. See the epic Technical Notes (Lane B: Spec Review) for the evaluation criteria checklist.
3. Create `scripts/codex-spec-review.sh` as a thin wrapper. The script should:
   - Accept a required epic directory path argument
   - Validate the directory and presence of `epic.md`
   - Accept an optional `--note "text"` or `--note-file /path` flag
   - Read all `.md` files in the target directory
   - Assemble a one-shot prompt: rubric + epic/story file contents + optional runtime note
   - Pass the assembled prompt to the Codex CLI via the mechanism determined in step 1
   - Use `set -euo pipefail`
   - Handle missing `codex` command gracefully
4. Make the script executable with `chmod +x`.
