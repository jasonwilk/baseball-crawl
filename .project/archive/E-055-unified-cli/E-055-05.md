# E-055-05: Database Commands (`bb db`)

## Epic
[E-055: Unified Operator CLI](epic.md)

## Status
`DONE`

## Description
After this story is complete, the operator can manage the database through `bb db backup` and `bb db reset`. These wrap the existing `scripts/backup_db.py` and `scripts/reset_dev_db.py` with consistent Typer interfaces.

## Context
Database operations are less frequent than credential or data tasks but critical when needed. Grouping them under `bb db` makes them discoverable and keeps the operator from searching through the scripts directory.

## Acceptance Criteria
- [ ] **AC-1**: `src/cli/db.py` exists and defines a Typer sub-app mounted on the main app as `bb db`.
- [ ] **AC-2**: `bb db backup` creates a timestamped backup of the SQLite database. Supports `--db-path` flag. Exit codes match `backup_db.py`.
- [ ] **AC-3**: `bb db reset` drops and recreates the dev database with seed data. Supports `--db-path` and `--force` flags. Exit codes match `reset_dev_db.py`.
- [ ] **AC-4**: `bb db reset` without `--force` in a production environment (APP_ENV=production) exits with an error and actionable message.
- [ ] **AC-5**: `bb db reset` without `--force` prompts for interactive confirmation ("This will destroy and recreate the database. Confirm? [y/N]") in ALL environments (dev and production). `--force` skips the prompt (for scripted use). Use `typer.confirm()`. The production guard (AC-4) fires BEFORE the confirmation prompt.
- [ ] **AC-6**: `bb db --help` lists both commands with descriptions.
- [ ] **AC-7**: Tests using `CliRunner` verify argument mapping for both commands (mock underlying functions). Include a test that `bb db reset` without `--force` triggers the confirmation prompt.
- [ ] **AC-8**: All existing tests continue to pass.

## Technical Approach
1. Create `src/cli/db.py` with a `typer.Typer(help="Database operations.")` sub-app.
2. `backup` command imports and calls `scripts.backup_db.backup_database(db_path)`. Catches `FileNotFoundError` and converts to non-zero exit with actionable message. On success, `backup_database()` returns the `Path` to the backup file -- print it and exit 0.
3. `reset` command imports and calls `scripts.reset_dev_db.reset_database(db_path, force)`. **Note**: `reset_database()` returns `tuple[int, int]` (tables_created, rows_inserted), NOT an exit code. The CLI must call it, print the summary, and exit 0 on success. Catches `SystemExit` (from production guard) and `FileNotFoundError` (from missing seed file) to convert to appropriate exit codes with messages.
4. The sub-app mount in `src/cli/__init__.py` is pre-wired by E-055-01 -- do not modify `__init__.py`.

Reference files:
- `/workspaces/baseball-crawl/scripts/backup_db.py` (has `backup_database(db_path: Path | None = None) -> Path`, raises `FileNotFoundError`)
- `/workspaces/baseball-crawl/scripts/reset_dev_db.py` (has `reset_database(db_path: Path | None = None, force: bool = False) -> tuple[int, int]`, raises `SystemExit` on production guard, `FileNotFoundError` on missing seed)

## Notes
- The `reset` command is destructive. The help text should make this clear: "Drop and recreate the database. All data will be lost."
- The `--force` flag behavior mirrors the production guard in `reset_dev_db.py`.
- **`backup_database()` raises `FileNotFoundError`** (when the DB file does not exist) rather than returning a non-zero exit code. The CLI wrapper must catch this exception and convert it to a non-zero exit with an actionable error message (e.g., "Database not found at {path}. Run `bb data sync` first.").
