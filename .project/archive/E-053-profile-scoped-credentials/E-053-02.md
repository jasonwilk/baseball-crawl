# E-053-02: Profile-aware credential loading in GameChangerClient

## Epic
[E-053: Profile-Scoped Credentials](epic.md)

## Status
`DONE`

## Description
After this story is complete, `GameChangerClient` will load credentials from profile-scoped `.env` keys only. A client instantiated with `profile="web"` reads `GAMECHANGER_AUTH_TOKEN_WEB`; if absent, it raises `ConfigurationError`. No fallback to flat keys. Same pattern for `"mobile"` with `_MOBILE` suffix.

## Context
`GameChangerClient.__init__` already accepts a `profile` parameter that controls header selection and proxy routing. Credential loading (`_load_credentials`) is the last profile-unaware piece -- it reads flat keys only. This story closes that gap by making credential loading profile-scoped with no fallback to flat keys. It has no dependency on E-053-01 because it only changes how credentials are *read*, not how they are *written*.

## Acceptance Criteria
- [ ] **AC-1**: Given `.env` contains `GAMECHANGER_AUTH_TOKEN_WEB=token_w`, when `GameChangerClient(profile="web")` is instantiated, then it uses `token_w`.
- [ ] **AC-2**: Given `.env` does NOT contain `GAMECHANGER_AUTH_TOKEN_WEB` (even if flat `GAMECHANGER_AUTH_TOKEN` is present), when `GameChangerClient(profile="web")` is instantiated, then it raises `ConfigurationError` naming the missing `GAMECHANGER_AUTH_TOKEN_WEB` key. No fallback.
- [ ] **AC-3**: Given `.env` contains `GAMECHANGER_AUTH_TOKEN_MOBILE=token_m` and `GAMECHANGER_DEVICE_ID_MOBILE=device_m`, when `GameChangerClient(profile="mobile")` is instantiated, then it uses the `_MOBILE` keys.
- [ ] **AC-4**: Given `.env` is missing `GAMECHANGER_AUTH_TOKEN_MOBILE`, when `GameChangerClient(profile="mobile")` is instantiated, then it raises `ConfigurationError` listing `GAMECHANGER_AUTH_TOKEN_MOBILE`.
- [ ] **AC-5**: Profile-scoped keys apply to `AUTH_TOKEN`, `DEVICE_ID`, `APP_NAME`, and `SIGNATURE`. `BASE_URL` remains unsuffixed (same API host for both profiles). `APP_NAME` remains optional (current behavior preserved -- web defaults to `"web"`, mobile omits).
- [ ] **AC-6**: The `_REQUIRED_KEYS` tuple is replaced with a function that computes the required key names from the profile (e.g., `_required_keys(profile="web")` returns `("GAMECHANGER_AUTH_TOKEN_WEB", "GAMECHANGER_DEVICE_ID_WEB", "GAMECHANGER_BASE_URL")`).
- [ ] **AC-7**: Existing tests that use flat credential keys are updated to use profile-scoped keys.
- [ ] **AC-8**: Tests cover: profile-scoped keys present, missing profile-scoped key (no fallback), both profiles independently.
- [ ] **AC-9**: `credential_parser.py` and `refresh_credentials.py` are updated to write `_WEB` suffixed keys (e.g., `GAMECHANGER_AUTH_TOKEN_WEB`) instead of flat keys (e.g., `GAMECHANGER_AUTH_TOKEN`). The curl-paste credential path is inherently web-only. Tests verify that refreshed credentials use `_WEB` suffixed key names.

## Technical Approach
- Refactor `_load_credentials` to accept the profile string (available from `__init__`).
- For each credential key, construct the profile-scoped name: `{KEY}_{PROFILE_SUFFIX}`. Profile suffix: `"web"` -> `"_WEB"`, `"mobile"` -> `"_MOBILE"`. No fallback to unsuffixed keys.
- Keep `BASE_URL` unsuffixed -- it is the same for both profiles. Only `AUTH_TOKEN`, `DEVICE_ID`, `APP_NAME`, and `SIGNATURE` are profile-scoped.
- The `ConfigurationError` message should name the profile-scoped key that was expected (so the operator knows which key to set).
- Note: these scripts will eventually be wrapped by E-055's unified `bb` CLI (separate epic, no dependency).

## Notes
- `GAMECHANGER_BASE_URL` is the same API host regardless of profile -- do not scope it. Only auth credentials (`AUTH_TOKEN`, `DEVICE_ID`) and fingerprint metadata (`APP_NAME`, `SIGNATURE`) vary by profile.
- `check_credentials.py` and `bootstrap.py` also have a `_REQUIRED_KEYS` tuple that checks flat keys -- that is handled by E-053-03, not this story.
