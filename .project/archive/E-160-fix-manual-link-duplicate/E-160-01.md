# E-160-01: Fix save_manual_opponent_link to Update Teams and Set resolved_team_id

## Epic
[E-160: Fix save_manual_opponent_link Duplicate Team Bug](epic.md)

## Status
`DONE`

## Description
After this story is complete, manually connecting an opponent via the admin UI will also update the `teams` table with the opponent's `public_id` and set `resolved_team_id` on the `opponent_links` row. This prevents the scouting pipeline from creating duplicate team rows when it later encounters the same `public_id`.

## Context
The manual connect flow (`POST /admin/opponents/{link_id}/connect`) calls `save_manual_opponent_link` which only updates `opponent_links`. The auto-resolve path handles both `teams` and `opponent_links` updates correctly. This story brings the manual path to parity. See epic Technical Notes TN-1 through TN-5 for full root cause analysis and disconnect symmetry requirements.

## Acceptance Criteria
- [ ] **AC-1**: Given an opponent with an existing stub team row (found via `team_opponents` junction OR direct `teams` name match per TN-2 two-tier lookup) with `public_id IS NULL`, when the admin manually connects it via the connect flow, then the stub team's `public_id` column is updated to the linked value AND `opponent_links.resolved_team_id` is set to that stub team's INTEGER PK. This covers both schedule-loader stubs (which have `team_opponents` rows) and discovery-path stubs (created by `bulk_create_opponents` without `team_opponents` rows).
- [ ] **AC-2**: Given an opponent with an existing stub team row whose `public_id` is already set (non-NULL) and DIFFERS from the admin-entered slug, when the admin manually connects it: (a) if the new slug is NOT owned by any other `teams` row, then `teams.public_id` IS updated to the new slug (with a WARNING logged) AND `opponent_links.resolved_team_id` IS set; (b) if the new slug IS already owned by another `teams` row, then `teams.public_id` is left unchanged, a WARNING is logged ("slug already owned — manual merge required"), and `opponent_links.resolved_team_id` IS still set. If the slugs match, no `teams.public_id` update is needed. Per Technical Notes TN-3.
- [ ] **AC-3**: Given an opponent with no matching stub team (neither via `team_opponents` nor direct name match), when the admin manually connects it, then `resolved_team_id` remains NULL (graceful degradation — no crash, no new team created), per Technical Notes TN-2 step 5.
- [ ] **AC-4**: Given a stub team with `public_id` set by the connect flow (AC-1 case), a subsequent `INSERT OR IGNORE INTO teams` with the same `public_id` produces no new row (team count stays at 1). Verified via a direct DB-layer test — no ScoutingCrawler integration test required.
- [ ] **AC-5**: Given a manually-linked opponent, when the admin disconnects the link (`POST /admin/opponents/{link_id}/disconnect`), then `teams.public_id` is cleared on the stub team (set to NULL) so a subsequent reconnect can write the corrected value — BUT only if no other `opponent_links` rows reference the same `resolved_team_id`. If other links exist, `teams.public_id` is preserved. Per Technical Notes TN-5.
- [ ] **AC-6**: Given a disconnect-then-reconnect scenario (AC-5 followed by AC-1), the reconnect correctly sets the new `public_id` on the stub team (AC-5 cleared it to NULL, so AC-1's NULL→new path applies cleanly without triggering TN-3's overwrite WARNING).
- [ ] **AC-7**: Given multiple tracked team stubs with the same name and `public_id IS NULL`, the connect flow selects the one with the highest `id` (most recently created), per TN-2 tie-break rule.
- [ ] **AC-8**: All existing tests pass. New tests cover AC-1 through AC-7 scenarios (including AC-4 duplicate-prevention, AC-5 shared-reference guard, and AC-7 multi-match tie-break).

## Technical Approach
The fix centers on `save_manual_opponent_link` in `src/api/db.py`. The function currently takes `(link_id, public_id)` and only updates `opponent_links`. It needs to be extended to also find the existing stub team via `team_opponents` (joining on `opponent_team_id` to `teams` where name and membership_type match) and wire up `public_id` and `resolved_team_id`. The caller in `src/api/routes/admin.py` (`connect_opponent`) already has the full `link` dict with `our_team_id` and `opponent_name` and should pass the necessary context.

Key constraints: when the stub already has a `public_id`, apply the TN-3 overwrite rule (update with collision guard). If no stub is found, degrade gracefully (leave `resolved_team_id = NULL`).

The disconnect path (`disconnect_opponent_link`) also needs updating per TN-5: when `resolution_method = 'manual'`, clear `teams.public_id` on the stub team — but only if no other `opponent_links` rows reference the same `resolved_team_id` (shared-reference guard). This ensures the disconnect→reconnect correction flow works cleanly.
