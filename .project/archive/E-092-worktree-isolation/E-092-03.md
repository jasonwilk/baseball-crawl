# E-092-03: Update implement skill for worktree lifecycle

## Epic
[E-092: Git Worktree Isolation for Agent Teams](epic.md)

## Status
`DONE`

## Description
After this story is complete, the implement skill (`SKILL.md`) will spawn implementing agents with `isolation: "worktree"`, include a merge-back step after review approval and before cascade, add worktree cleanup to the closure sequence, and exempt context-layer stories from worktree isolation.

## Context
The implement skill is the primary dispatch procedure reference. It must align with the updated dispatch pattern (E-092-02). The skill defines the concrete phases and steps that the main session follows during dispatch. This story adds worktree awareness to Phase 2 (spawning), Phase 3 (coordination loop -- merge-back between review and cascade), and Phase 5 (closure -- verify merges, cleanup).

## Acceptance Criteria
- [ ] **AC-1**: Phase 2 Step 2 (spawn implementing agents) documents passing `isolation: "worktree"` when spawning implementing agents
- [ ] **AC-2**: Phase 2 Step 2 documents that the code-reviewer is NOT spawned with `isolation: "worktree"`
- [ ] **AC-3**: Phase 3 includes a new merge-back step with explicit sequencing: APPROVED -> `git merge --no-ff <branch>` -> remove worktree -> delete branch -> mark DONE -> cascade. A story is NOT marked DONE until its branch is successfully merged
- [ ] **AC-4**: Phase 3 documents the conflict escalation path: if merge fails, story remains IN_PROGRESS, cascade blocked for dependent stories only, user resolves in main checkout, then main session completes the merge-back sequence
- [ ] **AC-5**: Phase 3 Step 2 (route stories to agents) documents the context-layer exception: stories routed to claude-architect for context-layer work are spawned WITHOUT `isolation: "worktree"`
- [ ] **AC-6**: The spawn context message template in Phase 3 Step 4 tells agents they are working in a worktree and references `.claude/rules/worktree-isolation.md`
- [ ] **AC-7**: Phase 5 Step 1 (validate all work) includes verifying all worktree branches have been merged into the current branch (a `git branch` check) and a safety-net sweep for orphaned worktrees (`git worktree list --porcelain`; force-remove any found). Worktree removal is per-story at merge-back time -- the closure check catches only orphans from error paths
- [ ] **AC-8**: The Workflow Summary diagram is updated to reflect the merge-back step
- [ ] **AC-9**: The Anti-Patterns section includes: "Do not spawn context-layer stories with worktree isolation"
- [ ] **AC-10**: The review assignment template (Phase 3 Step 5) includes the worktree path when the implementer worked in a worktree, so the code-reviewer knows where to read files, run tests, and run `git diff`

## Technical Approach
Read the current `/.claude/skills/implement/SKILL.md` and the updated `/.claude/rules/dispatch-pattern.md` (from E-092-02). Integrate worktree steps at the specific phase/step locations described in the ACs. The merge-back step is a new sub-step within the existing Phase 3 review loop -- it fits between "reviewer returns APPROVED" and "mark DONE + cascade."
