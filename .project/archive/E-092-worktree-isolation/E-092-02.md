# E-092-02: Update dispatch pattern for worktree dispatch

## Epic
[E-092: Git Worktree Isolation for Agent Teams](epic.md)

## Status
`DONE`

## Description
After this story is complete, the dispatch pattern (`dispatch-pattern.md`) will document worktree-native dispatch: implementing agents are spawned with `isolation: "worktree"`, the main session tracks worktree paths and branches, a per-story merge-back step exists after review approval, and context-layer stories are explicitly exempt from worktree isolation.

## Context
The dispatch pattern is the canonical reference for how agent teams work during epic execution. It currently assumes all agents share the main checkout. This story adds worktree awareness throughout: spawning, coordination, merge-back, conflict escalation, and the context-layer exception. The implement skill (E-092-03) depends on this story because it references dispatch-pattern.md as the canonical model.

## Acceptance Criteria
- [ ] **AC-1**: The "How Dispatch Works" section documents that implementing agents are spawned with `isolation: "worktree"` by default
- [ ] **AC-2**: A new "Worktree Dispatch" subsection (or equivalent) covers: (a) spawn with isolation parameter, (b) main session tracking of worktree path/branch per story, (c) merge-back protocol, (d) conflict escalation to user
- [ ] **AC-3**: The merge-back protocol specifies the full sequence: APPROVED -> `git merge --no-ff <branch>` from main checkout -> remove worktree -> delete branch -> mark DONE -> cascade. On merge conflict: story remains IN_PROGRESS, cascade blocked for dependent stories only, user resolves conflict in main checkout, then main session completes the sequence
- [ ] **AC-4**: The context-layer routing rule is extended: stories routed to claude-architect for context-layer work are spawned WITHOUT `isolation: "worktree"`
- [ ] **AC-5**: The "Team Composition" section documents that the code-reviewer is NOT spawned with worktree isolation -- it reads from implementers' worktree paths
- [ ] **AC-6**: A note documents that migration stories must not run concurrently (migration number conflicts)
- [ ] **AC-7**: The closure sequence step "Validate all work" includes verifying all worktree branches have been merged (a `git branch` check). Worktree removal is per-story (at merge-back time, not closure), so the closure check is a safety-net sweep: `git worktree list --porcelain` to catch any orphans, force-remove if found
- [ ] **AC-8**: The worktree dispatch subsection documents that the main session SHOULD NOT assign concurrent stories with overlapping "Files to Create or Modify" sections to parallel worktrees -- if overlap is detected during routing, those stories are serialized (assigned sequentially, not concurrently)

## Technical Approach
This is an update to an existing context-layer file. Read the current `/.claude/rules/dispatch-pattern.md` and add worktree-specific guidance at the appropriate points throughout the document. The changes should integrate naturally into the existing structure rather than being a bolted-on appendix. Key integration points: spawning (Team Composition, Spawning Scenarios), coordination (The Dispatch Flow steps 4-6), and closure (Closure Sequence step 9).
