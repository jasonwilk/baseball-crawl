# E-062-R-01: Prototype Endpoint File Format

## Epic
[E-062: Split API Documentation into Per-Endpoint Files](epic.md)

## Status
`DONE`

## Objective
Validate the structured-markdown-with-YAML-frontmatter approach by converting a representative sample of endpoints from the current monolith. Produce a finalized file format specification, naming convention, and migration plan that the implementation stories can follow without ambiguity.

## Time Box
Single focused agent session. The research direction is clear from prior deep research -- this spike validates with concrete prototyping, not open-ended exploration.

## Research Questions
1. Does the YAML frontmatter schema proposed in epic Technical Notes capture all metadata currently present across all three documentation tiers (fully documented, partially documented, minimal)?
2. What is the right file naming convention for edge-case paths (e.g., `/teams/public/{public_id}/players` which uses an inverted URL pattern vs. `/public/teams/{public_id}`)?
3. What cross-reference patterns exist in the current monolith, and how should they be expressed as file links in the split structure?
4. How should web vs. mobile profile-specific information be structured within an endpoint file?
5. What is the right boundary between "global reference file" content and "per-endpoint" content? (e.g., does the header profiles comparison table belong in `headers.md` only, or should each endpoint file repeat its specific Accept header?)

## Findings

Prototyped three endpoint files spanning the documentation quality spectrum:

1. **`get-teams-team_id-schedule.md`** (Tier 1 / CONFIRMED): Fully documented with complete YAML frontmatter, schema tables, two JSON examples (timed game event, full-day event), and Known Limitations. Confirms the format handles complex endpoints with location polymorphism and datetime format switching.

2. **`get-teams-team_id-lineup-recommendation.md`** (Tier 2 / CONFIRMED but Accept unconfirmed): Demonstrates `accept: null` pattern and how to document gaps in frontmatter notes without blocking file creation. `profiles.web.notes` carries the caveat inline.

3. **`get-organizations-org_id-opponent-players.md`** (Tier 3 / PARTIAL): HTTP 500 on web profile, HTTP 200 observed on mobile. Demonstrates `status: PARTIAL`, `caveats` field, `Suspected Required Headers` section with investigation note, and fallback strategy documentation.

Key schema refinements made during prototyping:
- `profiles.{profile}.status` (confirmed/unverified/partial/observed/not_applicable) replaces simple boolean -- captures behavioral differences, not just presence
- `PARTIAL` status added to vocabulary for endpoints that work only with specific parameters not yet confirmed
- `caveats` array field added for blocking issues that prevent full documentation
- Tag vocabulary finalized at 19 controlled terms; see format spec for full list
- Accept header is always per-endpoint (never global), even when it matches a common pattern
- Inverted URL pattern (`/teams/public/{public_id}/players`) handled by mirroring the exact path into the filename

Complete endpoint inventory (88 endpoints + web-routes reference file) is in the format spec, with each endpoint assigned to E-062-02, E-062-03, or E-062-06.

## Recommendation

See `/workspaces/baseball-crawl/.project/research/E-062-format-spec.md` Section 8 for explicit answers to all five research questions. Summary:

- **RQ-1 (frontmatter completeness)**: Yes, with additions: `profiles.{profile}.status/.notes`, `caveats` array, `PARTIAL` status, controlled tag vocabulary
- **RQ-2 (inverted URL naming)**: Mirror exact path segments into filename; `get-teams-public-public_id-players.md` vs `get-public-teams-public_id.md` -- the filename difference self-documents the API quirk
- **RQ-3 (cross-references)**: `see_also` frontmatter for machine-readable links; inline "ID chain" blocks for narrative flows; body text links for related schemas
- **RQ-4 (web/mobile profiles)**: `profiles.web.*` and `profiles.mobile.*` with status + notes; full headers listed only for web profile (the primary implementation target); mobile-specific behavior captured in notes
- **RQ-5 (global vs. per-endpoint boundary)**: Accept header always per-endpoint; standard browser headers (`User-Agent`, `DNT`, `sec-*`) always global (in `headers.md`); `gc-user-action` per-endpoint when present
