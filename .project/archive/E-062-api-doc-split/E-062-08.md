# E-062-08: API Docs Claude Rule + Ingest-Endpoint Skill Rewrite

## Epic
[E-062: Split API Documentation into Per-Endpoint Files](epic.md)

## Status
`DONE`

## Description
After this story is complete, agents will have a dedicated rule (`.claude/rules/api-docs.md`) governing how they interact with the per-endpoint API documentation structure, the ingest-endpoint skill will be fully rewritten to create/update per-endpoint files instead of appending to the monolith, and the documentation ownership table will reflect the new structure. The rule enforces loading discipline (index first, specific files only) and accuracy standards. The skill embeds the complete format spec inline so it is self-contained.

## Context
E-062 replaces the monolithic `docs/gamechanger-api.md` with a per-endpoint file structure. Two context-layer changes are needed to make this sustainable:

1. **A rule** that teaches agents the new structure, prevents bulk-loading 88 endpoint files, and enforces accuracy standards for frontmatter fields like `last_confirmed`.

2. **A skill rewrite** that changes the ingest-endpoint workflow from "append to monolith" to "create/update a per-endpoint file with YAML frontmatter." The current skill references the monolith throughout and its Phase 1 instructions are incompatible with the new structure.

This story absorbs E-062-05's AC-7 (ingest-endpoint skill rewrite) and AC-8 (documentation.md ownership update) to avoid wasted work -- E-062-08 performs a full skill rewrite that supersedes the path-swap approach in E-062-05.

Expert consultation performed: CA (rule scope, loading discipline, skill rewrite approach, overlap resolution), UXD (index scannability -- informing rule's loading discipline guidance).

## Acceptance Criteria

### Rule: `.claude/rules/api-docs.md`
- [ ] **AC-1**: New file `.claude/rules/api-docs.md` exists with glob paths scoped to `docs/api/**` (rule loads only when agents touch API doc files, not every session).
- [ ] **AC-2**: Rule documents the **canonical location and structure** of API documentation: `docs/api/README.md` (index), `docs/api/endpoints/` (per-endpoint files), `docs/api/*.md` (global reference files: auth, headers, pagination, content-type, base-url, error-handling).
- [ ] **AC-3**: Rule enforces **loading discipline** with prescriptive language:
  - MUST read `docs/api/README.md` first to identify needed endpoint files
  - MUST NOT glob-read or bulk-load all files in `docs/api/endpoints/`

- [ ] **AC-4**: Rule documents the **accuracy standard**: Accept headers, auth requirements, and status values in frontmatter must match live verification. The `last_confirmed` field must only be updated after live verification (not during migrations or edits that don't re-verify the endpoint).

- [ ] **AC-6**: Rule documents **file naming convention** (method-path-segments.md with param names, no curly braces).

## Technical Approach
The rule file uses YAML frontmatter with `paths: ["docs/api/**"]` for glob-triggered loading. It is structured as a reference document that agents consult when reading or writing API endpoint files.

The skill rewrite preserves the two-phase structure (api-scout then claude-architect) but replaces all monolith references with per-endpoint file operations. The embedded format spec is extracted from `/.project/research/E-062-format-spec.md` Sections 1-5 and condensed to the essential fields needed by api-scout during ingestion.

The E-062-05 story file has already had AC-7 and AC-8 removed during prior spec review triage. AC-13 verifies this is still the case.
