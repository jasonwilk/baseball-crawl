# E-229-03: Wire `run_plays_stage` into `run_scouting_sync` (web)

## Epic
[E-229: Plays in `bb data scout` parity](./epic.md)

## Status
`DONE`

## Description
After this story is complete, the web scout pipeline (`run_scouting_sync`) invokes the shared `run_plays_stage` helper after the post-spray dedup sweep. Plays-stage failures (per-game errors, auth expiry, reconcile failures) are non-fatal: the operator sees a structured prefix in `crawl_jobs.error_message` on partial failure, and `crawl_jobs.status` is unaffected by plays outcome. Plays runs as additive enrichment, mirroring the spray-stage pattern that already exists in `run_scouting_sync`.

## Context
The web scout entry point is `run_scouting_sync` in `src/pipeline/trigger.py:637-761`. Today the function flow is:
1. Auth refresh (line 664-672)
2. Season-year heal (line 674-677)
3. Scouting crawl + load (line 681-728)
4. `_mark_job_terminal('completed')` + `_update_last_synced` (lines 727-728)
5. Spray crawl + load via `_run_spray_stages` (line 730-735)
6. Post-spray dedup sweep (line 737-750)

This story slots the plays stage at step 7, AFTER post-spray dedup. The pre-existing timing oddity in step 4 (status marked `completed` before spray runs) is NOT fixed in this epic per the SE-revised analysis (see History entry for the Y2 retraction). Plays follows the same additive-enrichment pattern that spray already established -- error reporting via `crawl_jobs.error_message` prefix; status timing unchanged.

The auth-error pattern for plays follows the helper's contract: `CredentialExpiredError` is caught internally, returned as `auth_expired=True` in `PlaysStageResult`. The web wrapper logs a warning naming the deferred game count. The `crawl_jobs.error_message` column receives a structured prefix per epic Technical Notes "Operator visibility" section when partial failure occurs (loaded < attempted).

## Technical Approach
Two changes to `src/pipeline/trigger.py`:

1. Add `run_plays_stage` import.
2. Inside the `if load_result.errors:` ... `else:` block at line 720-750, after the post-spray dedup sweep at line 737-750, insert the plays-stage call. Connection management is unchanged. The plays-stage call slots inside the same `with closing(sqlite3.connect(...))` block (line 681) -- helper accepts that connection.

The terminal status writes at lines 727-728 (`_mark_job_terminal('completed')` + `_update_last_synced`) are NOT moved. The pre-existing timing oddity (status marked completed before later additive stages run) is preserved as-is per the Y2 retraction (see epic History). Captured as follow-up idea #6 in E-229-05 AC-10.

Connection management is unchanged. WAL + foreign keys are already set on the shared connection at lines 682-683.

For the `error_message` UPDATE on partial failure: the existing flow already wrote `error_message=None` via `_mark_job_terminal(conn, crawl_job_id, "completed", None)` at line 727. The plays wrapper UPDATEs this column directly via `UPDATE crawl_jobs SET error_message = ? WHERE id = ?` when plays returns non-clean. This is a follow-on UPDATE, not a re-call of `_mark_job_terminal` (which would also re-write `status` and `completed_at` -- not desired).

Structured prefix format per epic Tech Notes "Operator visibility" rule: comma-joined fragments of `"plays: {loaded}/{attempted} loaded"` plus non-zero `"{errored} errored"` and/or `"{N} deferred (auth)"`. When `result` is fully clean (zero errored, zero deferred), no UPDATE is issued -- existing `error_message=None` stands. Spray/dedup pre-existing silent-swallow behavior is unchanged -- those errors do NOT contribute to the prefix; only the plays-stage portion is included.

The early-return failure path at lines 705-707 ("crawl_result.errors > 0 and games_crawled == 0") is preserved -- it correctly marks the job `failed` before any of the additive stages run.
