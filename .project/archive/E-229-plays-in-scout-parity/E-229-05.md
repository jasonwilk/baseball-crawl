# E-229-05: Parity-invariant test `test_scout_plays_parity.py`

## Epic
[E-229: Plays in `bb data scout` parity](./epic.md)

## Status
`DONE`

## Description
After this story is complete, the CLAUDE.md scouting-pipeline-parity invariant for the plays stage is encoded as an executable test. The test runs both the CLI scout path (`_scout_live` via `typer.testing.CliRunner`) and the web scout path (`run_scouting_sync`) against identical fixture inputs, then asserts equivalent DB state on the three plays-touching tables: `plays`, `play_events`, `reconciliation_discrepancies`. The test makes the parity contract grep-able for future reviewers asking "where is the parity contract enforced?"

## Context
The CLAUDE.md "Scouting pipeline parity" architectural invariant says: "`run_scouting_sync` (web) and `bb data scout` (CLI) MUST produce equivalent data artifacts -- all five stages in both paths." Today the invariant is enforced by code review and integration tests on each path independently. This story adds a single executable test that fails when the two paths diverge.

The test relies on the shared fixtures factored into `tests/conftest.py` by E-229-01 (`mock_gc_client_with_plays`, `seed_boxscore_for_plays`, `plays_json_factory`, `seed_scout_result_skeleton`) plus per-path scaffolding from `tests/test_cli_data_scout_plays.py` (E-229-02) and `tests/test_pipeline_scouting_sync_plays.py` (E-229-03).

## Technical Approach
Two test instances run end-to-end with the same fixtures:

1. **CLI run**: invoke `bb data scout --team <public_id>` via `typer.testing.CliRunner` against a fresh per-test tmp_path SQLite file seeded with the test fixtures. Mock HTTP via the shared `mock_gc_client_with_plays` fixture.
2. **Web run**: invoke `run_scouting_sync(team_id, public_id, crawl_job_id)` directly against another fresh per-test tmp_path SQLite file seeded with identical fixtures. Mock HTTP via the same shared fixture.

Per AC-7: both DBs are tmp_path-based, NOT `:memory:`. Each path runs with its own DB connection on its own tmp_path file. Use `monkeypatch.setattr` on `src.cli.data._resolve_db_path` (CLI) and `src.pipeline.trigger.get_db_path` (web -- the imported binding, per Python import-binding semantics; existing `tests/test_trigger.py:214` is the canonical example).

After both runs complete, query the three tables on each DB, normalize (sort by natural key, exclude excluded columns), and compare. Use `assert cli_rows == web_rows` for plays and reconciliation_discrepancies; for play_events, build the natural-key join and compare.

The test does NOT need to verify CLI exit code or `crawl_jobs.status` -- those are path-specific surfaces and are covered by the per-path test files (E-229-02, E-229-03). This test exclusively asserts equivalent DB artifacts.

For test isolation: each path runs with its own DB connection, populated by an identical `seed_scout_result_skeleton` invocation. Time-sensitive columns (`created_at`, `completed_at`, `started_at`) are stripped before comparison via either freezegun or column projection in the comparison query.

For the diff-formatter helper (AC-9): a small pure function that takes `(cli_rows: list[tuple], web_rows: list[tuple], table_name: str)` and returns a string. When the rows match, returns empty string. When they differ, returns a multi-line diff naming the table, row counts on each side, and either the missing rows or the columns/values that mismatched. The unit tests in AC-9 verify the formatter on synthetic inputs without exercising the full parity test path.
