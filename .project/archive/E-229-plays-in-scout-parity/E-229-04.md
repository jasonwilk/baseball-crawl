# E-229-04: Migrate report generator to `run_plays_stage`; delete `_crawl_and_load_plays`

## Epic
[E-229: Plays in `bb data scout` parity](./epic.md)

## Status
`DONE`

## Description
After this story is complete, `src/reports/generator.py` invokes the shared `run_plays_stage` helper instead of its private `_crawl_and_load_plays` function. The private function `_crawl_and_load_plays` is deleted; the inline call site at lines 1091-1110 is REPLACED by a shared-helper call wrapped in an `except Exception`-only defensive envelope (the now-dead `except CredentialExpiredError` block is removed -- the helper's contract guarantees no auth re-raise). There is now exactly one implementation of plays-stage orchestration across all three caller sites (CLI scout, web scout, report generator).

For tracked-team reports where scout has already loaded plays, the helper's per-game idempotency check makes the call cheap (zero HTTP, zero new rows). For non-tracked-team reports (`bb report generate <public_id>` for any GC slug per `.claude/rules/scouting-data-flows.md`), the helper performs the full crawl + load + reconcile -- preserving today's behavior for that case.

## Context
Today, `src/reports/generator.py:524-643` defines a private `_crawl_and_load_plays` helper that duplicates what E-229-01 lifts into the shared module. The inline call site at `src/reports/generator.py:1091-1110` (inside `generate_report`) invokes that private helper as Step 4d of report generation. Once E-229-01 ships, the private helper is redundant: its semantics are exactly what the shared helper provides, and keeping both creates a drift surface.

The migration is mechanical: replace the private call with a shared-helper call using the same arguments, then delete the private function. The shared helper's `PlaysStageResult` is consumed for logging only -- the report generator does not write to `crawl_jobs` (reports use the `reports` table) and does not need the dataclass for state threading.

## Technical Approach
The migration is a delete + replace. Steps:

1. Verify no other site in `src/` imports `_crawl_and_load_plays` (private function -- should have zero external callers, but grep to confirm).
2. Verify no other site in `src/reports/generator.py` references `_PLAYS_ACCEPT` outside of `_crawl_and_load_plays` (should be zero -- grep to confirm before deleting the constant).
3. Replace the call at lines 1091-1110 with a `run_plays_stage(...)` call. Keep the existing log-line summary; emit it from the returned `PlaysStageResult`.
4. Delete the private function (lines 524-643), any private sub-helpers that are unused after the deletion, and the `_PLAYS_ACCEPT` constant at line 57.
5. Delete the now-dead `except CredentialExpiredError` block from the call-site error handling. Retain only `except Exception` as a defensive wrapper.
6. Update imports: remove `from src.gamechanger.loaders.plays_loader import PlaysLoader` if it's no longer referenced after deletion (it may still be referenced elsewhere in the file -- check before removing). Add `from src.gamechanger.pipelines import run_plays_stage`.

The retained `except Exception` defensive wrapper around the helper call exists because report generation has its own outer error envelope (`_fail_report` for fatal failures); the inner defensive wrapper turns any unexpected helper exception into a non-fatal warning so the report still renders. This is parity with how spray failures are handled in the same function.

The test in AC-6 is a monkeypatch test, not an integration test. Mock `src.gamechanger.pipelines.run_plays_stage` (or its import alias inside `generator.py`) and assert the call signature. The integration coverage already exists in `tests/test_plays_stage.py` (story 01) and the parity test (story 05).
