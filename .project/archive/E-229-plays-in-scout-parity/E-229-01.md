# E-229-01: Implement `run_plays_stage` shared helper module

## Epic
[E-229: Plays in `bb data scout` parity](./epic.md)

## Status
`DONE`

## Description
After this story is complete, the system will have a single shared module `src/gamechanger/pipelines/plays_stage.py` that exposes `run_plays_stage(...)`, the canonical pipeline orchestration for crawling, loading, and reconciling plays for a given list of game IDs from a given team perspective. The helper is the sole entry point for plays loading from the CLI scout path, the web scout path, and the report generator. The helper is non-fatal: it catches `CredentialExpiredError` mid-stage and returns a structured `PlaysStageResult` so callers can log a warning and continue. The helper assumes upstream scout stages (boxscore load via `game_loader`) have already populated `game_perspectives` for every loaded game; the helper does NOT itself write to `game_perspectives` (per DE recommendation -- the `PlaysLoader` `game_perspectives` write gap is captured as a separate idea, not in scope here).

## Context
Today, plays loading is implemented inline inside `src/reports/generator.py:524-643` (private `_crawl_and_load_plays`). The CLI scout (`_scout_live` in `src/cli/data.py`) and the web scout (`run_scouting_sync` in `src/pipeline/trigger.py`) both skip plays entirely. This story establishes the shared module that subsequent stories wire into the three caller sites.

The helper signature, return shape, idempotency semantics, and error handling are defined in epic Technical Notes. This story implements the module and its unit tests; subsequent stories wire the three callers.

## Technical Approach
The shared helper is a lift-and-shift of the existing `src/reports/generator.py::_crawl_and_load_plays` (lines 524-643), generalized to accept `perspective_team_id` and `public_id` keyword-only arguments instead of being closed over the report's local variables. Internals follow the existing pattern: per-game `client.get("/game-stream-processing/{game_id}/plays", accept=_PLAYS_ACCEPT)`, write to `tempfile.TemporaryDirectory()` as `{game_id}.json`, instantiate `PlaysLoader(conn, owned_team_ref=...)` with a `TeamRef` whose `id` is the `perspective_team_id`, call `loader.load_all(tmp_dir)`, then per-game `reconcile_game(conn, game_id, dry_run=False, perspective_team_id=perspective_team_id)`.

The `_PLAYS_ACCEPT` constant is defined module-private inside `src/gamechanger/pipelines/plays_stage.py`. Story E-229-04 deletes the corresponding `_PLAYS_ACCEPT` constant from `src/reports/generator.py:57` once the shared helper is in use.

**Parameter naming**: the perspective-tagging argument is `perspective_team_id`, not `team_id`. Inside the helper, this value flows into `PlaysLoader(owned_team_ref=...)` AND into `reconcile_game(perspective_team_id=...)`. Naming it `perspective_team_id` at the helper boundary forces correct caller mental model and matches the perspective-provenance MUST constraint terminology.

The helper does NOT open or close the connection -- it accepts an open `sqlite3.Connection` from the caller. Per Technical Notes, the docstring explicitly warns that the first plays-loader commit will commit any uncommitted writes from earlier scout steps; callers must pass a clean connection. The helper does NOT call `PRAGMA foreign_keys=ON` -- caller responsibility, documented in the docstring to prevent the helper from masking caller bugs.

**Reconcile selection**: `PlaysLoader.load_all()` returns aggregate `LoadResult(loaded, skipped, errors)` -- NOT per-game outcomes. To determine which games to reconcile, the helper performs a post-load DB probe per game: `SELECT 1 FROM plays WHERE game_id = ? AND perspective_team_id = ? LIMIT 1`. Only games where the probe returns a row receive `reconcile_game(...)`. This mirrors the existing pattern in `src/reports/generator.py:619-633`.

`PlaysLoader._load_game()` already enforces whole-game perspective-aware idempotency (`SELECT 1 FROM plays WHERE game_id = ? AND perspective_team_id = ? LIMIT 1`); the helper does NOT add an outer DB idempotency check beyond a pre-fetch HTTP-skip optimization (matches existing `src/reports/generator.py:556-565` pattern). The pre-fetch skip avoids wasted HTTP traffic on rerun; the loader's gate is the authoritative idempotency guard.

`PlaysStageResult` fields surface the data needed by callers for operator-facing logging:
- `attempted`, `loaded`, `skipped`, `errored`, `reconcile_errors`, `auth_expired`, `deferred_game_ids`.
- The dataclass is the helper's only return; no other state is threaded through `crawl_jobs` (per epic Q3 decision).

## Notes
The helper signature deliberately keeps `perspective_team_id: int` and `public_id: str` flat (no `TeamRef`); `public_id` is log-only. This matches SE's Q4 N1 recommendation. The parameter rename from `team_id` to `perspective_team_id` is per DE's Q2 footgun-prevention recommendation. A future refactor that moves to in-memory `PlaysLoader` (eliminating the tempdir hack) is out of scope for E-229 per SE Q4 N3 and is captured as a separate idea (filed by E-229-05's closing AC). The `PlaysLoader._load_game()` `game_perspectives` write gap is captured as a separate idea per DE Q3 -- the helper does not patch around it; callers are responsible for ensuring upstream boxscore load has populated `game_perspectives` first (which the scout pipeline does via `game_loader.py`).
