# E-007-03: PM Prompt -- Dispatch Mode

## Epic
[E-007: Orchestrator Workflow Discipline](epic.md)

## Status
`DONE`

## Description
After this story is complete, the project-manager agent prompt will include a "Dispatch Mode" section that describes exactly how PM executes a "start epic X" or "execute story Y" directive. This turns PM from a pure planning agent into a two-mode agent: Planning Mode (existing behavior) and Dispatch Mode (new). In Dispatch Mode, PM reads the epic, finds TODO stories with satisfied dependencies, dispatches each via Task tool with a standard context block, updates story statuses, and tracks completion. This gives PM the authority and the instructions to act as the true hub of work execution.

## Context
The current PM prompt explicitly says "You do NOT assign work to agents unless asked." This was a correct constraint for a PM focused only on planning, but the workflow contract requires PM to be the dispatch hub. The new instruction replaces that constraint with a two-mode model: when the user asks PM to plan, PM plans. When the user asks PM to execute, PM dispatches. The Dispatch Mode instruction must be specific enough that PM can execute it without asking the user "which agent should I use for this story?"

## Acceptance Criteria

- [ ] **AC-1**: The PM prompt contains a new "Dispatch Mode" section (or equivalently titled section) that describes when Dispatch Mode is triggered: user says "start epic E-NNN", "execute story E-NNN-SS", "dispatch the epic", or equivalent.
- [ ] **AC-2**: Dispatch Mode includes a numbered procedure with these steps exactly:
  1. Read the epic directory (`/epics/E-NNN-*/epic.md` and all story files).
  2. Identify stories with `Status: TODO` whose blocking dependencies are satisfied (all blocked-by stories are DONE).
  3. Update each eligible story file from `Status: TODO` to `Status: IN_PROGRESS` before dispatching.
  4. Dispatch each eligible story via Task tool with the standard context block (see AC-4).
  5. After each dispatched task completes, verify the story's acceptance criteria are met.
  6. Update the story file to `Status: DONE`.
  7. Check for newly unblocked stories and repeat from step 2 until all stories are DONE or BLOCKED.
- [ ] **AC-3**: Dispatch Mode includes an agent-selection rule: given a story's content, PM must select the correct implementing agent. The rule specifies: Python implementation -> general-dev; DB schema / D1 / migrations -> data-engineer; API exploration -> api-scout; agent config / CLAUDE.md / rules -> claude-architect.
- [ ] **AC-4**: Dispatch Mode specifies the standard context block that PM must include in every Task tool call: (a) absolute path to the story file, (b) full contents of the story file, (c) the "Technical Notes" section of the parent epic, (d) names and statuses of all dependency stories.
- [ ] **AC-5**: The "What You Do NOT Do" section removes or replaces the line "You do NOT assign work to agents unless asked" -- replacing it with "You do NOT assign work to agents unless in Dispatch Mode (triggered by an explicit execute/start/dispatch directive from the user or orchestrator)."
- [ ] **AC-6**: The existing Planning Mode behavior (writing epics, stories, ideas) is unchanged. Dispatch Mode is an addition, not a replacement.

## Technical Approach
Edit `/.claude/agents/project-manager.md`. Add a new "Dispatch Mode" section after the "System of Work" section (or as a subsection of it). Update the "What You Do NOT Do" section as described in AC-5. Do not remove or reorganize existing sections.

The Dispatch Mode section should be concrete and procedural. A model reading it should be able to dispatch a story without asking a clarifying question. Include a short example showing the Task tool call structure.

Example Task tool invocation structure for PM to follow:
```

    [Technical Notes section]

    - E-NNN-01: [title] -- DONE

    Read the story file, satisfy all acceptance criteria, and update the story
    Status to DONE when complete.
```

## Notes
The "What You Do NOT Do" replacement in AC-5 is important: the old constraint "unless asked" was intended to prevent PM from being proactively pushy. The new constraint "unless in Dispatch Mode" preserves that intent (PM doesn't go off and start dispatching unprompted) while enabling the dispatch capability when explicitly triggered.

The example Task tool invocation in Technical Approach is a model -- adjust wording as needed, but the four required elements (story path, story contents, epic Technical Notes, dependency statuses) are non-negotiable.
