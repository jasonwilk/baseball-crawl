# E-007-08: Epic Dispatch Gate -- READY Status and Promotion Ritual

## Epic
[E-007: Orchestrator Workflow Discipline](epic.md)

## Status
`DONE`

## Description
After this story is complete, the epic status system has an explicit `READY` status between `DRAFT` and `ACTIVE` that serves as the PM's formal dispatch gate. An epic cannot be dispatched (stories cannot move to IN_PROGRESS) until the PM has explicitly set it to `READY`. This prevents premature dispatch of incompletely refined epics and gives orchestrators and agents a clear, unambiguous signal for whether an epic is dispatchable.

## Context
E-007 established "Status: TODO in the story file" as the PM's authorization signal for individual stories. That is correct and stays. What E-007 did not address is the epic-level gate: how does an orchestrator or PM know whether an epic as a whole is ready to be dispatched?

The current epic status system is:
- `DRAFT` -- being written. Not ready for execution.
- `ACTIVE` -- has stories ready for execution.
- `BLOCKED` -- all remaining stories are blocked.
- `COMPLETED` -- all stories done.
- `ABANDONED` -- will not be completed.

The problem: `ACTIVE` is ambiguous. An epic can be set to `ACTIVE` when the PM first writes stories, but those stories may still have vague acceptance criteria, unresolved open questions, or missing expert consultation. Nothing in the current system requires the PM to do a final review pass before the orchestrator dispatches the epic.

The gap this creates: a user says "start epic E-002" and the orchestrator routes to PM. PM reads the epic, finds stories with Status: TODO, and dispatches them -- even if the PM never consulted api-scout about what endpoints actually exist, or never consulted baseball-coach about what stats are needed. The epic was `ACTIVE` but not genuinely ready.

### The Fix: READY Status

Insert a new status `READY` between `DRAFT` and the dispatching phase. The ritual:
1. PM creates the epic (DRAFT).
2. PM runs Refinement Mode: consults experts, writes stories, sets ACs.
3. PM does a final Quality Checklist review.
4. PM explicitly sets the epic to `READY`.
5. Only then can dispatch be triggered.

`ACTIVE` becomes the status for epics that are currently being executed (at least one story is IN_PROGRESS). This makes the status system clearly track where the epic is in its lifecycle.

### New Status System

| Status | Meaning | Who Sets It |
|--------|---------|-------------|
| `DRAFT` | Being written. Not ready for execution. No stories yet, or stories without complete ACs. | PM |
| `READY` | PM has completed Refinement Mode. Stories have testable ACs. Expert consultation done. Safe to dispatch. | PM (explicit act) |
| `ACTIVE` | Execution underway. At least one story is IN_PROGRESS. | PM (set when first story dispatched) |
| `BLOCKED` | All remaining TODO stories are blocked on unresolved dependencies. | PM |
| `COMPLETED` | All stories DONE. Ready for archive. | PM |
| `ABANDONED` | Will not be completed. Reason documented. | PM |

### What "Setting READY" Requires

Before the PM may set an epic to `READY`, the following must be true:
1. All stories have Status: TODO (none are empty stubs without ACs).
2. All acceptance criteria are specific and testable (no "should work" language).
3. Expert consultation has been completed (or explicitly noted as not required).
4. The Quality Checklist in the PM prompt has been run against all stories.
5. Open Questions section has either answers or explicit "will be resolved during execution" notes.

This is not a bureaucratic checklist -- it is the PM confirming that an agent could pick up any TODO story and execute it without guessing.

### Orchestrator Enforcement

The orchestrator should refuse to dispatch an epic that is not `READY` or `ACTIVE`. When routing "start epic E-NNN" to PM, the PM must check the epic status before dispatching. If the epic is `DRAFT`, PM enters Refinement Mode first and tells the user the epic is not yet ready.

### Placement in Dispatch Mode

The existing Dispatch Mode step 1 is "Read the epic directory." Insert a new step 0 or modify step 1:

```
### Dispatch Mode Precondition Check

Before identifying eligible stories, verify the epic status:
- If Status is `READY` or `ACTIVE`: proceed with dispatch.
- If Status is `DRAFT`: do NOT dispatch. Tell the user: "Epic E-NNN is in DRAFT status and has not been marked READY. I need to complete refinement before dispatching. [Enter Refinement Mode or ask user how to proceed.]"
- If Status is `BLOCKED`: do NOT dispatch. Explain which dependencies are unresolved.
```
