# E-007-09: PM Decision Gates -- Evaluation Criteria and Synthesis Ritual

## Epic
[E-007: Orchestrator Workflow Discipline](epic.md)

## Status
`DONE`

## Description
After this story is complete, `project-manager.md` documents the Decision Gate pattern: a first-class PM activity for epics whose stories collectively produce evaluation results that must be synthesized before the PM can close the epic or authorize next steps. A decision gate is not a story that implements something -- it is a PM-owned synthesis step that reads all completed story outputs, applies a declared set of evaluation criteria, and records a formal recommendation or decision. The pattern describes when to use a gate, how to define criteria up front, where criteria live in the epic, what the synthesis output looks like, and how the decision is recorded and communicated.

## Context

E-007's work established two PM modes: Refinement Mode (epic formation, expert consultation) and Dispatch Mode (story execution). Both modes operate on implementation stories -- stories that produce code, docs, or config. Neither mode addresses a third pattern that has now emerged in practice: the evaluation epic.

An evaluation epic (like E-008) works differently:
1. Stories are research tasks, not implementation tasks.
2. Stories apply a shared set of evaluation criteria to different options.
3. When all story research is done, the PM must synthesize the results and make a formal recommendation before the epic can close or authorize next steps (e.g., E-009).

The gap: neither Refinement Mode nor Dispatch Mode describes this synthesis activity. Dispatch Mode says "when all stories are DONE, mark the epic COMPLETED." That is wrong for evaluation epics -- completing the research stories does not close the epic. The PM must still synthesize and decide. Without a documented pattern, the PM has no guidance on:

- When to declare that an epic needs a decision gate (not all epics do)
- How to define evaluation criteria up front so all research stories apply them consistently
- Where in the epic structure the criteria live
- What a synthesis document looks like and where it goes
- What the gate outcomes are (approve, reject, defer) and what each triggers
- How the decision gets recorded so it persists across sessions

The concrete motivating example is E-008: intent/context layer design. E-008 defined evaluation criteria (token efficiency, agent onboarding clarity, maintenance burden, workflow compatibility, implementation complexity, reversibility) that all research stories must apply. After research stories complete, the PM synthesizes results, makes a recommendation, and decides whether to authorize E-009 or loop back. E-008-03 is a formal decision gate story, owned by the PM.

This pattern needs to be named and documented in `project-manager.md` so the PM applies it consistently in future evaluation epics.

- [ ] **AC-2**: The section specifies when a decision gate is appropriate. - [ ] **AC-3**: The section specifies that evaluation criteria are defined during Refinement Mode -- before stories are written -- and live in the epic's Technical Notes section. ## Technical Approach

This is a pure prompt-editing story. One file is modified: `/.claude/agents/project-manager.md`.

The Decision Gates section belongs inside or immediately after Refinement Mode. Rationale: the gate is defined during refinement (criteria are set up front), even though the gate itself executes during or after dispatch. Placing it between Refinement Mode and Dispatch Mode makes the three activities read in natural sequence:

1. Refinement Mode -- form the epic, define criteria if evaluation epic, consult experts
2. Decision Gates -- (if applicable) synthesize research results, record decision
3. Dispatch Mode -- execute implementation stories

### Suggested Section Structure

```markdown
## Decision Gates

A decision gate is a PM-owned synthesis step that appears in evaluation epics -- epics whose stories are research or options-comparison tasks rather than implementation tasks. The gate is the final story in such an epic. It reads all completed research outputs, evaluates them against declared criteria, and records a formal recommendation before the epic closes or the next epic is authorized.

Decision gates are NOT bureaucracy added to every epic. Most epics -- those whose stories build, configure, or test something -- do not need a gate. Use a gate only when:
1. The epic's stories are research, evaluation, or options-comparison tasks.
2. Stories are expected to apply a shared set of evaluation criteria.
3. The PM must make an explicit recommendation or go/no-go decision before authorizing next steps.

### Defining Criteria (Refinement Phase)

Evaluation criteria are defined during Refinement Mode, before stories are written. They live in the epic's Technical Notes section under a heading like "Evaluation Criteria". Each criterion must be specific enough that two agents evaluating the same option against it would reach the same conclusion. Vague criteria ("is simple", "is maintainable") are not acceptable -- define what measurable signal would indicate the criterion is met or failed.

Example criteria format:
- **Token efficiency**: Does the approach add fewer than X lines to shared agent context? Rank options by token cost.
- **Agent onboarding clarity**: Given only the epic's context and the approach, could a new agent (with no session history) execute a story correctly on the first attempt?
- **Reversibility**: If adopted and later abandoned, can the change be undone in under one story?

### The Gate Story

The gate story is:
- Owned by PM (not an implementing agent)
- Numbered as the last story in the epic's sequence (e.g., E-NNN-03 if there are two research stories)
- Status: TODO initially; blocked until all research stories are DONE; not dispatched to an implementing agent -- the PM executes it directly
- Produces a written synthesis document (see below)

In the epic's Stories table, the gate story's Dependencies column lists all research stories it depends on. Example:

| E-NNN-03 | Decision gate: options synthesis and recommendation | TODO | E-NNN-01, E-NNN-02 | PM |

### Gate Execution

When all research stories are DONE, the PM:
1. Reads every research story's output (the artifact or analysis each produced).
2. Scores or compares each option against every evaluation criterion from Technical Notes.
3. Produces a synthesis document at `/.project/research/E-NNN-[slug]-decision-log.md`.
4. Records the decision with one of three outcomes:

**APPROVED** -- The recommendation is clear and the path forward is authorized.
- Mark the gate story DONE.
- Mark the epic COMPLETED.
- If a follow-on epic was gated on this decision, note in the log that it is now authorized.
- Tell the user: "Decision gate closed. Recommendation: [X]. [Next epic / next step] is now authorized."

**REJECTED** -- The research is insufficient or the options are all unacceptable. Loop back.
- Mark the gate story BLOCKED.
- Identify what additional research is needed and create new stories or amend existing ones.
- Epic stays ACTIVE.
- Tell the user: "Decision gate returned to research. Reason: [Y]. Adding [new story or revised story]."

**DEFERRED** -- A decision is premature. External information is needed or timing is wrong.
- Mark the gate story BLOCKED with an explicit review date.
- Move the epic to BLOCKED status.
- Append an entry to the decision log explaining the deferral and what must change before revisiting.
- Tell the user: "Decision deferred. Reason: [Z]. Epic BLOCKED until [condition or date]."

### Decision Log Format

The decision log at `/.project/research/E-NNN-[slug]-decision-log.md` is append-only. Each entry contains:
- Date
- Gate story ID
- Outcome (APPROVED / REJECTED / DEFERRED)
- Criteria scores or comparison table
- Recommendation and rationale
- What happens next

Never delete or edit old entries. If a gate loops back (REJECTED) and re-executes, append a new entry below the previous one.
```

## Notes
- The motivating example is E-008 (intent/context layer design). E-008-03 is the first actual decision gate story in this project. The pattern being documented here is the generalized form of what E-008-03 does.
- The criteria for E-008 are: token efficiency, agent onboarding clarity, maintenance burden, workflow compatibility, implementation complexity, reversibility. These live in E-008's Technical Notes section.
- Decision gates should be rare. Most epics in this project are implementation epics (E-001 through E-006) and need no gate. The pattern exists for the class of epics that are genuinely evaluative.
- The PM executes gate stories directly -- they are not dispatched to an implementing agent. The PM reads the research outputs and writes the synthesis. This is consistent with how E-007-01 through E-007-06 were executed: the PM is both spec-writer and executor for PM-domain work.
