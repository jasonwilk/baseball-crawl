# E-007-02: Orchestrator Prompt -- PM-as-Gatekeeper

## Epic
[E-007: Orchestrator Workflow Discipline](epic.md)

## Status
`DONE`

## Description
After this story is complete, the orchestrator agent prompt will explicitly encode PM as the mandatory gatekeeper for all work-initiation requests. The orchestrator will refuse to route implementation requests directly to general-dev or data-engineer when no PM-approved story backs the work. It will also explicitly name the "direct-routing exceptions" (api-scout, baseball-coach, claude-architect) that may be invoked without PM intermediation. The net effect: a user saying "start working on the data crawler" routes to PM, not to general-dev.

## Context
The current orchestrator prompt routes to PM for project planning requests, but does not have an explicit rule about implementation requests. The "Routing Rules" section says "when in doubt, ask the user" -- which is insufficient. The desired behavior is a positive assertion: work-initiation requests ALWAYS go to PM, and the orchestrator knows what "work-initiation" means. Direct-routing exceptions (api-scout, baseball-coach, claude-architect) remain valid because they are exploratory/consultative agents, not implementing agents.

## Acceptance Criteria

- [ ] **AC-1**: Given the orchestrator prompt, when a human reads the "Routing Rules" section, they find an explicit rule stating that work-initiation requests (any request asking an agent to write code, build a feature, implement a story, or start an epic) MUST be routed to project-manager first, not to general-dev or data-engineer.
- [ ] **AC-2**: The prompt names the "direct-routing exceptions" by agent name: `api-scout`, `baseball-coach`, and `claude-architect` may be invoked directly without PM intermediation, with a brief rationale for each.
- [ ] **AC-3**: The prompt defines "work-initiation request" clearly enough that a model reading it could correctly classify test cases (see Notes below for the five test scenarios -- each must be addressed by the rule as written).
- [ ] **AC-4**: The prompt includes an example routing for "start epic X" showing the Task call going to project-manager with the instruction to dispatch the epic.
- [ ] **AC-5**: The existing routing logic for non-implementation requests (e.g., "what stats do coaches care about?" -> baseball-coach) is unchanged.
- [ ] **AC-6**: The updated prompt does not lengthen the orchestrator response times by adding unnecessary verbosity. The new rules section should be under 150 words.

## Technical Approach
Edit `/.claude/agents/orchestrator.md`. Locate the "Routing Rules" section. Add a new numbered rule (before or after the existing rules, whichever makes it most prominent) for the PM-as-gatekeeper principle. Add a new "Direct-Routing Exceptions" subsection after the Routing Rules. Add one routing example for "start epic X" in the Examples section.

Do NOT rewrite the existing routing rules -- only add. Do NOT change the orchestrator model, color, or tools configuration.

The new rule should read approximately:
> **Work-initiation requests go to PM first.** Any request that asks an agent to write code, implement a feature, build a component, execute a story, or start an epic MUST be routed to project-manager before going to any implementing agent. The project-manager reviews the relevant stories and dispatches work. Exception: api-scout, baseball-coach, and claude-architect may be invoked directly (they are consultative/exploratory, not implementing agents).
