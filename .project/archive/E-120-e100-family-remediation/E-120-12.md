# E-120-12: Scouting load public_id type guard

## Epic
[E-120: E-100 Family Code Review Remediation](epic.md)

## Status
`DONE`

## Description
After this story is complete, the scouting load command will skip teams with no `public_id` explicitly (with a clear warning) instead of silently falling back to an integer team_id as the directory name, which would never match crawled data.

## Context
At `src/cli/data.py:388`, `effective_pub_id = pub_id or team_id` falls back to an integer `team_id` when `public_id` is None. The resulting scouting directory path (e.g., `data/2026-spring-hs/scouting/42`) never matches any crawled data directory (which uses public_id slugs like `lsb-varsity`). The `is_dir()` guard on the next line catches this gracefully, but the warning message shows the misleading integer path. The fix is to skip early with a clear message when `public_id` is missing.

## Acceptance Criteria
- [ ] **AC-1**: When `public_id` is None for a team in the scouting load loop (`src/cli/data.py`, `_run_scouting_load` function), the team is skipped with a warning that clearly states the team has no public_id (not a misleading path-not-found warning with an integer directory name)
- [ ] **AC-2**: The `effective_pub_id = pub_id or team_id` fallback pattern at `data.py:388` is removed
- [ ] **AC-3**: All existing tests pass

## Technical Approach
In `src/cli/data.py`, replace the `effective_pub_id = pub_id or team_id` pattern with an early guard: if `pub_id` is None, log a descriptive warning and `continue` to the next team.
