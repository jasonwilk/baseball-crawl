# E-120-09: Parameter shadowing cleanup

## Epic
[E-120: E-100 Family Code Review Remediation](epic.md)

## Status
`DONE`

## Description
After this story is complete, function parameters that shadow Python builtins (`filter`, `id`) will be renamed to domain-specific names, eliminating static analysis warnings and improving code clarity.

## Context
Two parameter shadowing issues: (1) `db.py:831` defines `_opponent_links_where(our_team_id, filter)` where `filter` shadows the Python builtin. Should be renamed to something like `link_filter` or `status_filter`. (2) `admin.py:1150` and `admin.py:1189` use `id` as a path parameter name, shadowing the Python builtin. Should be renamed to `team_id` for clarity and to match the domain model. All callers and URL path references must be updated consistently.

## Acceptance Criteria
- [ ] **AC-1**: The `filter` parameter in `_opponent_links_where` at `db.py` is renamed to a non-shadowing name. All callers of this function are updated to use the new parameter name.
- [ ] **AC-2**: The `id` path parameter in `edit_team_form` (GET `/teams/{id}/edit`) and `update_team` (POST `/teams/{id}/edit`) at `admin.py` is renamed to `team_id`. The FastAPI route path template, function signature, and all internal references within those functions are updated consistently.
- [ ] **AC-3**: All existing tests pass (including any tests that reference these parameter names or URL paths)

## Technical Approach
1. For AC-1: Rename `filter` to a descriptive alternative in `_opponent_links_where` and update all call sites. Grep for `_opponent_links_where` to find all callers.
2. For AC-2: Change `{id}` to `{team_id}` in the route decorator paths, rename the parameter in function signatures, and update all internal usages. Verify no tests or templates construct URLs with the old parameter name.
