# E-120-10: XSS escaping regression test for query parameters

## Epic
[E-120: E-100 Family Code Review Remediation](epic.md)

## Status
`DONE`

## Description
After this story is complete, a regression test will verify that user-controlled query parameters (`?msg=`, `?error=`) rendered in admin HTML templates are properly escaped, preventing XSS if auto-escaping is ever accidentally disabled.

## Context
Admin routes accept `?msg=` and `?error=` query parameters and render their values in HTML templates (e.g., after team add/edit redirects). Jinja2 auto-escapes by default, but there is no test verifying this behavior. If auto-escaping were accidentally disabled (e.g., by a `| safe` filter or `{% autoescape false %}` block), XSS would be possible. A single regression test closes this gap.

## Acceptance Criteria
- [ ] **AC-1**: A test sends a request to an admin route with a `?msg=` query parameter containing an HTML/script payload (e.g., `<script>alert(1)</script>`) and asserts that the response body contains the escaped form (e.g., `&lt;script&gt;`) rather than the raw HTML
- [ ] **AC-2**: The test covers at least the team list page (`/admin/teams?msg=...`) since that is the primary redirect target for msg/error params
- [ ] **AC-3**: All existing tests pass

## Technical Approach
Add a test function in the appropriate admin test file. Use the existing test client to GET `/admin/teams?msg=<script>alert(1)</script>` and assert that `<script>` does not appear unescaped in the response body, while `&lt;script&gt;` (or the absence of raw `<script>`) does appear.
