# E-066-01: ZSH Default Shell and Env Bootstrap

## Epic
[E-066: Devcontainer Terminal Setup](epic.md)

## Status
`DONE`

## Description
After this story is complete, new interactive terminals in the devcontainer open in ZSH instead of Bash. The environment bootstrap script injects shared exports into both `.bashrc` and `.zshrc`, so environment variables (e.g., `BRIGHT_DATA_TOKEN`) are available in both shells. All existing bash automation (hooks, scripts, Claude Code's Bash tool) continues to work without regression.

## Context
The `devcontainers/base:ubuntu` image already has ZSH installed with Oh My Zsh pre-configured (theme: `devcontainers`, plugins: `git`, auto-update disabled). The default shell is Bash. SE confirmed that `chsh` in postCreateCommand is the correct approach -- do NOT add the `common-utils` feature (it's already baked into the base image and re-adding causes conflicts). The `chsh` command does not interfere with the existing Oh My Zsh installation. The env bootstrap script currently only injects into `.bashrc`; it must also inject into `.zshrc` for ZSH sessions to see the same environment variables.

## Acceptance Criteria
- [ ] **AC-1**: `devcontainer.json` postCreateCommand begins with `sudo chsh -s /usr/bin/zsh vscode` (before other commands in the chain)
- [ ] **AC-2**: `post-create-env.sh` creates `~/.zshrc` if it does not exist (via `touch`) before the idempotency check
- [ ] **AC-3**: `post-create-env.sh` injects the same MARKER-delimited export block into both `~/.bashrc` and `~/.zshrc`
- [ ] **AC-4**: The `.bashrc` injection still prepends before the interactive guard (non-interactive bash must see the exports)
- [ ] **AC-5**: `postCreateCommand` clones `zsh-autosuggestions` and `zsh-syntax-highlighting` into the Oh My Zsh custom plugins directory (`${ZSH_CUSTOM:-~/.oh-my-zsh/custom}/plugins/`)
- [ ] **AC-6**: The Oh My Zsh `plugins` line in `~/.zshrc` is updated to `plugins=(git z zsh-autosuggestions zsh-syntax-highlighting)`
- [ ] **AC-7**: Existing tests pass (`pytest` -- no regressions)

## Technical Approach
Two files change. In `devcontainer.json`, the postCreateCommand needs three additions: `chsh` at the front of the command chain, shallow git clones of `zsh-autosuggestions` and `zsh-syntax-highlighting` into Oh My Zsh's custom plugins directory, and a sed command to update the plugins line in `~/.zshrc`. In `post-create-env.sh`, the script currently handles only `.bashrc` -- it needs a parallel block for `.zshrc` using the same MARKER pattern. SE flagged that `.zshrc` may not exist when the script runs, so it must be created first.

The plugin clones use `--depth=1` for minimal footprint. Do NOT use the `devcontainers-extra/features/zsh-plugins` community feature -- it adds a fragile third-party dependency. The git-clone approach is simpler and more transparent.

See epic Technical Notes for the SE consultation details on why `chsh` over `common-utils`, the `.zshrc` touch requirement, and the plugin installation approach.

## Notes
- The `common-utils` feature is already in the base image. Do not add it to the features list.
- Oh My Zsh is pre-installed at `~/.oh-my-zsh/` with the `devcontainers` theme (shows git branch) and `git` plugin. Auto-update is already disabled. The `chsh` approach preserves all of this.
- The `z` plugin ships with Oh My Zsh (no external deps). It provides frecency-based directory jumping, useful for navigating between `src/`, `tests/`, `scripts/`, `data/`.
- `zsh-autosuggestions` shows ghost text from command history as you type. `zsh-syntax-highlighting` colors valid/invalid commands in real time. Both are high-value for a bash-to-zsh transition. Combined startup latency is ~20-50ms (imperceptible).
- Do NOT use the `devcontainers-extra/features/zsh-plugins` community feature -- it uses fragile sed and adds unnecessary third-party dependency.
- Claude Code's Bash tool is unaffected by this change -- it runs bash directly regardless of the user's interactive shell.
