# E-066-05: Terminal and Shell User Guide

## Epic
[E-066: Devcontainer Terminal Setup](epic.md)

## Status
`DONE`

## Description
After this story is complete, a practical reference guide at `docs/admin/terminal-guide.md` helps the operator get productive with the new ZSH and tmux setup. The guide covers what changed, ZSH essentials for bash users, tmux basics, connecting to the devcontainer from iTerm2, and the three operating modes. The audience is a technically capable operator who is new to ZSH and tmux but comfortable with containers and the command line.

## Context
The operator is transitioning from default bash to ZSH as the interactive shell, with tmux newly available. He needs confidence that the new tools work correctly -- especially connecting from iTerm2 outside VS Code for heavy Agent Teams sessions. This is an operator-facing admin document in `docs/admin/`, which is the docs-writer's domain.

The guide complements but does not duplicate E-066-04's CLAUDE.md updates. CLAUDE.md documents the dual-shell contract and mode taxonomy for agents and contributors (terse, context-layer). This guide is for the human operator (practical, task-oriented).

## Technical Approach
This is a documentation-only story producing one new file in `docs/admin/`. The guide should be structured for quick scanning (headers, tables, short code blocks) rather than narrative prose. The operator knows containers and command lines -- he just hasn't used ZSH or tmux before.

Key sources for content:
- Epic Technical Notes (plugin list, tmux config settings, dual-shell contract, three modes, session convention)
- `.devcontainer/tmux.conf` (the actual keybindings and settings -- reference after E-066-03 is complete)

The iTerm2 connection workflow should reference the `devcontainer exec` CLI or `docker exec` as mechanisms for attaching to the devcontainer from a host terminal. The devcontainer is managed by VS Code / the devcontainer CLI -- it is NOT defined in the project's `docker-compose.yml` (which only defines `app`, `traefik`, and `cloudflared`). The container name can be found via `docker ps` while VS Code has the devcontainer running. When using `docker exec`, the command must include `-u vscode` to land as the correct user (e.g., `docker exec -it -u vscode <container> zsh`). Without `-u vscode`, the operator lands as root and misses the environment variable setup from the `vscode` user's `.zshrc`.
