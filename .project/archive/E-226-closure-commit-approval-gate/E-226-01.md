# E-226-01: Enforce closure commit approval gate in implement skill

## Epic
[E-226: Closure Commit Approval Gate Enforcement](../E-226-closure-commit-approval-gate/epic.md)

## Status
`DONE`

## Description
After this story is complete, the implement skill's Phase 5 closure sequence will enforce an explicit user-approval gate as a first-class numbered step before the closure `feat(E-NNN)` commit. The gate step names the exact command to present, the exact approval words, and spells out what does not count as approval. The archive `chore(E-NNN)` commit (Phase 5 Step 11) is not given its own approval gate -- instead it gets a preflight `git status --porcelain` anomaly check and is unchained from `git add -A` so anomalies can be surfaced between staging and committing. The Workflow Summary and Anti-Pattern #5 cross-reference the new structure accurately.

## Context
During E-225 dispatch, the main session ran `git commit` at closure without pausing to present the staged diff or wait for approval. The rule already exists in `.claude/skills/implement/SKILL.md` but lives in a trailing prose paragraph below the closure-sequence numbered list, making the 8-step sequence read as a continuous run-through. The archive commit (Step 11) has no gate at all -- it is a chained `git add -A && git commit`.

The fix is entirely contained in `.claude/skills/implement/SKILL.md`. See epic Technical Notes TN-1 through TN-7 for the target shape of each change.
