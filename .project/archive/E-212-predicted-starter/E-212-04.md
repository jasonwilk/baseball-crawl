# E-212-04: Report Generator Integration and Template Rendering

## Epic
[E-212: Predicted Starter](epic.md)

## Status
`DONE`

## Description
After this story is complete, the report generation pipeline will include the predicted starter stage, and the scouting report HTML template will render the predicted starter section with four confidence-dependent rendering modes, a rest/availability table, and an optional LLM narrative block. The section appears in every generated report between the exec summary and Recent Form.

## Context
This story wires the data query (E-212-01), deterministic engine (E-212-02), and optionally the LLM layer (E-212-03) into the existing `generate_report()` pipeline and renders the results in the report template. The template must handle four confidence tiers (high, moderate, low/committee, suppress) plus an additive LLM narrative block. The rest/availability table is always present when pitching data exists. This is the integration and presentation story -- the analysis logic is already built.

## Acceptance Criteria
- [ ] **AC-1**: `generate_report()` in `src/reports/generator.py` calls `get_pitching_history(team_id, season_id, db=conn)`, `build_pitcher_profiles()`, and `get_pitching_workload()` (shared functions from `src/api/db.py`) during the Step 5 query phase, then calls `compute_starter_prediction(pitcher_profiles, pitching_history, workload=pitching_workload)` (from E-212-02) to produce the Tier 1 prediction. All DB calls in this stage reuse the existing `conn` from `generate_report()` via the `db=conn` parameter per TN-5. The `get_pitching_workload()` call reuses the existing function already called for the pitching workload display -- its output is passed to the engine for rest table population.
- [ ] **AC-2**: When `OPENROUTER_API_KEY` is set, `generate_report()` attempts `enrich_prediction()` (from E-212-03), passing available W-L records (scouted team's record from the public team profile data already fetched; user's team record when generating in an opponent context). When the key is not set, enrichment is skipped without logging a warning. When the call fails, a warning is logged and the report continues with Tier 1 only -- matching the non-fatal stage pattern per TN-5.
- [ ] **AC-3**: The `data` dict passed to `render_report()` includes `starter_prediction` and `enriched_prediction`. Contract: when `get_pitching_history()` returns an empty list (team has 0 pitching rows), the engine is never called and `starter_prediction = None` -- the template renders only the section header with a "No pitching data available" note (no rest table, no candidate cards -- the rest table would be empty). When `get_pitching_history()` returns 1+ rows, the engine is called and `starter_prediction` is always a `StarterPrediction` dataclass (possibly with `confidence = "suppress"` for 1-3 games). `enriched_prediction` is None when Tier 2 is skipped or fails, `EnrichedPrediction` when successful.
- [ ] **AC-4**: The scouting report template renders a "Predicted Starter" section per TN-4:
  - Section header using `.section-header` style
  - Placed after exec summary, before Recent Form section
  - Inside `.stats-content` wrapper (landscape page)
- [ ] **AC-5**: **High confidence rendering**: Primary candidate card with blue left-border accent, plain-English reasoning (NO percentage displayed), full game log table (last 3-5 starts with Date, IP, #P, K, BB, Decision), GS badge.
- [ ] **AC-6**: **Moderate confidence rendering**: Primary candidate card (blue accent) with a caveat line + secondary candidate card (standard border, condensed game log). No percentage displayed.
- [ ] **AC-7**: **Low/Committee rendering**: No blue-accented card. Text label indicating committee or multiple candidates. Up to 3 candidate cards at equal visual weight (all standard border), each with condensed game log.
- [ ] **AC-8**: **Suppress rendering** (confidence = "suppress"): No candidate cards. Only the rest/availability table with a contextual note from `data_note` (e.g., "Fewer than 4 games available -- insufficient data for starter prediction").
- [ ] **AC-9**: **Rest/availability table** always renders when `rest_table` is non-empty: compact table for up to 3 arms (1-2 starters by GS + highest-appearance relievers) with Name, GS, Last Outing (date), Days Rest, Last Outing (#P), 7d Workload columns. Uses existing report table styling at 8pt. Appears below candidate cards (or alone when suppressed).
- [ ] **AC-9a**: **Bullpen order** renders when `bullpen_order` is non-empty: a single line below the rest table listing up to 3 relievers ranked by first-relief-appearance frequency (e.g., "Likely bullpen order: Verkamp (4/6), Dewing, Schmidt"). Uses `.sort-annotation` style. Omitted entirely when empty.
- [ ] **AC-10**: When `enriched_prediction` is present, a "Scouting Analysis" narrative block renders below the rest table per TN-4: gray left-border (`border-left: 3px solid #d1d5db`), sub-header at `.key-player-label` weight, narrative text at `8.5pt/1.5`, optional bullpen sequence paragraph. When `enriched_prediction` is None, no trace of the narrative block exists in the HTML.
- [ ] **AC-11**: Disclaimer line renders at section bottom using `.sort-annotation` style. Text is "Based on rotation pattern, rest days, and recent workload. Actual starter may differ." when `enriched_prediction` is None; text is "Based on rotation pattern, rest days, recent workload, and AI-assisted analysis. Actual starter may differ." when `enriched_prediction` is present.
- [ ] **AC-12**: Game log dates use the existing `data-date` attribute pattern for JS relative-date upgrade in browser viewing.
- [ ] **AC-13**: **Mobile responsiveness**: Lower-priority elements (secondary candidate game logs, rest table columns beyond Name/GS/Days Rest) use the existing `mob-hide-extra` CSS class per TN-4. The primary candidate card and its reasoning line must remain visible on mobile.
- [ ] **AC-14**: Print safety: `break-inside: avoid` on the section container and narrative block. No dark background colors. Card backgrounds use `#f9fafb`.
- [ ] **AC-15**: `render_report()` in `src/reports/renderer.py` passes `starter_prediction` and `enriched_prediction` to the Jinja2 template context.
- [ ] **AC-16**: Tests verify: report renders correctly for each of the four confidence tiers (high, moderate, low/committee, suppress), report renders correctly with LLM narrative present, LLM failure during generation produces a valid report with Tier 1 prediction only, rest table renders in all modes, mobile classes applied correctly on lower-priority elements.

## Technical Approach
The generator integration follows TN-5: insert the prediction stage after existing queries and before the `data = {}` assembly. The template uses Jinja2 conditionals branching on `starter_prediction.confidence` to select the rendering mode. The rest table renders independently of the prediction confidence tier. CSS for the new section reuses existing report conventions (`.section-header`, `.depth-badge`, `.sort-annotation`, `.key-players`-style card containers).
