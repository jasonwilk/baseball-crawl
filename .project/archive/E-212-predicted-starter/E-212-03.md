# E-212-03: OpenRouter LLM Client and Starter Analysis (Tier 2)

## Epic
[E-212: Predicted Starter](epic.md)

## Status
`DONE`

## Description
After this story is complete, a reusable OpenRouter HTTP client will exist in `src/llm/`, and a report-specific analysis module will use it to enrich the Tier 1 deterministic prediction with LLM-generated narrative, matchup context, and bullpen sequence analysis. The enrichment is optional -- it is skipped silently when `OPENROUTER_API_KEY` is not set, and failures are non-fatal.

## Context
This is the project's first LLM integration. The OpenRouter client is deliberately general-purpose (future LLM features can reuse it). The report-specific prompt engineering and response parsing live in `src/reports/llm_analysis.py`, keeping the "how to talk to OpenRouter" separate from "what to ask about pitching rotations." The enrichment follows the sequential pattern: Tier 1 produces `StarterPrediction`, Tier 2 wraps it in `EnrichedPrediction` with narrative fields.

## Acceptance Criteria
- [ ] **AC-1**: A package `src/llm/` exists with `__init__.py` and `openrouter.py`. The client function `query_openrouter(messages, model, max_tokens, temperature) -> dict` sends a chat completion request to `https://openrouter.ai/api/v1/chat/completions` using `httpx` directly.
- [ ] **AC-2**: The client reads `OPENROUTER_API_KEY` from the environment. When the key is absent, calling code can detect this before attempting a request (e.g., a helper function `is_llm_available() -> bool`).
- [ ] **AC-3**: The client reads `OPENROUTER_MODEL` from the environment with a default value. Timeout is 30 seconds. No retries. Failures raise a specific exception (e.g., `LLMError`) with a descriptive message.
- [ ] **AC-4**: The client uses a plain `httpx.Client()` (not `create_session()` from `src/http/session.py`). This is a documented exception -- OpenRouter is a standard API, not GameChanger.
- [ ] **AC-5**: A module `src/reports/llm_analysis.py` exists with a function `enrich_prediction(prediction: StarterPrediction, pitching_history: list[dict], *, team_record: str | None = None, opponent_record: str | None = None) -> EnrichedPrediction` that builds a prompt, calls OpenRouter, and parses the response. The `team_record` and `opponent_record` parameters accept W-L strings (e.g., "15-3") for inclusion in the LLM prompt per AC-6.
- [ ] **AC-6**: The prompt includes structured pitching data (formatted tables, not prose), the Tier 1 prediction results, and both teams' W-L records when available (scouted team's record always; user's team record when the report is generated in an opponent context). W-L is the only opponent-strength signal available (no conference/non-conference or strength-of-schedule data from GameChanger). The LLM may note matchup-strength elevation as a possibility but should not overstate its predictive value -- coaches' deployment decisions based on opponent strength are real (~25-30% of games at competitive HS level) but unpredictable from record alone. The response is requested as JSON with `narrative`, `bullpen_sequence`, and optional `confidence_adjustment` fields. **Contract**: `confidence_adjustment` is requested in the prompt to improve narrative quality (gives the LLM room to express agreement/disagreement with Tier 1), but is intentionally discarded after parsing -- the deterministic confidence tier from Tier 1 is authoritative for rendering decisions. `EnrichedPrediction` does not store `confidence_adjustment`.
- [ ] **AC-7**: `EnrichedPrediction` dataclass contains: `base` (the original `StarterPrediction`), `narrative` (string), `bullpen_sequence` (string or None), `model_used` (string for attribution).
- [ ] **AC-8**: When the LLM response is malformed (not valid JSON, missing required fields), `enrich_prediction` raises `LLMError` rather than returning corrupt data.
- [ ] **AC-9**: Tests for `openrouter.py`: mock httpx transport, verify correct headers (`Authorization: Bearer ...`), correct body shape, and correct error handling for 401/429/500/timeout responses.
- [ ] **AC-10**: Tests for `llm_analysis.py`: mock `query_openrouter`, verify prompt construction includes pitcher data and Tier 1 prediction, verify response parsing extracts the right fields, verify malformed response raises `LLMError`.

## Technical Approach
The OpenRouter client is a thin wrapper around httpx POST. The analysis module builds a system prompt (coaching context, output format instructions) and a user prompt (structured pitching data + Tier 1 results), calls the client, and parses the JSON response. Error handling follows the non-fatal pattern per TN-3 and TN-5 -- the caller in `generate_report()` catches exceptions and falls back to Tier 1.

## Notes
- Token cost is negligible: ~2-3K input + ~500 output per report = ~$0.001 at Haiku pricing.
- The prompt should use structured data format (tables/JSON) rather than prose descriptions of stats -- LLMs parse structured data more reliably.
- `OPENROUTER_API_KEY` must NEVER appear in logs, code, or commit history per project security rules.
- Per coach: LLM narrative is most valuable at moderate confidence (explains nuance). At high confidence it's nice to have. At low confidence the LLM should explain the ambiguity, not manufacture a prediction. The prompt should include the confidence tier so the LLM can calibrate its tone.
- Tournament detection hint: if the last 3+ games are on consecutive or near-consecutive days, the LLM should note "predictions less reliable in tournament play" but not suppress the analysis.
