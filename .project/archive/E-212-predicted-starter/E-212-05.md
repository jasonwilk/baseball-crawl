# E-212-05: Dashboard Opponent Detail Integration

## Epic
[E-212: Predicted Starter](epic.md)

## Status
`DONE`

## Description
After this story is complete, the dashboard opponent detail page (`/dashboard/opponents/{opponent_team_id}`) and its print variant will include a "Predicted Starter" section using the shared pitching history query and deterministic engine. The section renders identically to the standalone report's Tier 1 presentation (four confidence modes + rest table) but uses dashboard CSS conventions. No LLM enrichment on the dashboard per TN-6.

## Context
The predicted starter feature serves two scouting surfaces: standalone reports (E-212-04) and the dashboard opponent detail page (this story). Both use the same shared query functions (`get_pitching_history()`, `build_pitcher_profiles()` from `src/api/db.py`) and the same deterministic engine (`compute_starter_prediction()` from `src/reports/starter_prediction.py`). The dashboard is a live page load, so the 30-second LLM call is not acceptable here -- Tier 1 only.

## Acceptance Criteria
- [ ] **AC-1**: Both the `opponent_detail` route handler AND its print variant in `src/api/routes/dashboard.py` call `get_pitching_history()`, `build_pitcher_profiles()`, and `get_pitching_workload()` (from `src/api/db.py`), then `compute_starter_prediction(pitcher_profiles, pitching_history, workload=pitching_workload)` (from `src/reports/starter_prediction.py`), and pass the result to the template context.
- [ ] **AC-2**: The DB calls use `run_in_threadpool` matching the existing async pattern in the route handler.
- [ ] **AC-3**: The `opponent_detail.html` template renders a "Predicted Starter" section with the same four confidence modes as the standalone report per TN-4:
  - High: named starter card with reasoning, full game log, GS badge
  - Moderate: named starter with caveat + alternative card
  - Low/Committee: equal-weight candidate cards, "committee" label
  - Suppress: rest table only with contextual note
- [ ] **AC-4**: The rest/availability table renders in all modes when `rest_table` is non-empty (up to 3 arms: 1-2 starters by GS + highest-appearance relievers). The "Last Outing" date column is server-rendered as relative (e.g., "5d ago") matching the existing `rest_display` pattern. The bullpen order line renders below the rest table when `bullpen_order` is non-empty (same format as the report: ranked relievers with frequency counts).
- [ ] **AC-5**: The section is placed immediately after the "Their Pitchers" summary cards -- before "Team Batting" and "Last Meeting." This matches the report's information hierarchy: pitching staff overview, then "who's starting?" as the first actionable intelligence.
- [ ] **AC-6**: The print variant (`/dashboard/opponents/{opponent_team_id}/print`) includes the predicted starter section.
- [ ] **AC-7**: No LLM enrichment is attempted on the dashboard. The template does not include the "Scouting Analysis" narrative block or any LLM-related conditionals.
- [ ] **AC-8**: No percentages are displayed. Plain-English reasoning only, matching the standalone report convention.
- [ ] **AC-9**: The section uses Tailwind utility classes matching the existing opponent detail page styling (border, padding, text-size classes -- not the standalone report's inline styles). Game log dates are server-rendered as relative (e.g., "5d ago") matching the existing `rest_display` pattern -- do NOT introduce client-side JS date conversion on the dashboard for this section.
- [ ] **AC-10**: When `starter_prediction` is None (meaning `get_pitching_history()` returned 0 rows -- the engine was never called), the entire section is omitted from the page. This differs from standalone reports, which always show the section header with a "No pitching data available" note. The dashboard omits empty sections for cleaner UX.
- [ ] **AC-11**: **Mobile responsiveness**: On screens narrower than 640px, secondary candidate cards (#2, #3) show only the reasoning line -- game log details are hidden via Tailwind responsive classes (`hidden sm:block` or similar). The primary candidate card always shows the full game log. The rest table uses an `overflow-x-auto` wrapper matching the dashboard's existing table pattern.
- [ ] **AC-12**: **Print safety**: The print variant renders the predicted starter section with `break-inside: avoid` on the section container and candidate cards, preventing page breaks within the prediction section. If the Tailwind CDN version does not support `break-inside-avoid`, add a `<style>` block with the print rule.
- [ ] **AC-13**: Tests verify: opponent detail page renders with prediction data, opponent detail page renders without prediction data (no pitching history), prediction failure does not break the page.

## Technical Approach
The route handler adds the prediction computation between the existing `_fetch_opponent_detail_data()` call and the template render. The prediction is a lightweight synchronous computation (one DB query + pure Python engine) with negligible latency impact. The template section is a new `{% include %}` partial or inline block, using dashboard CSS classes.

## Notes
- This story can execute in parallel with E-212-03 (LLM) and E-212-04 (report integration) since they share no files. Both E-212-04 and E-212-05 depend on E-212-02 but not on each other.
- The dashboard opponent detail already calls `get_pitching_workload()` from `src/api/db.py` -- the predicted starter query follows the same shared function pattern.
- Dashboard CSS conventions differ from standalone report inline styles. Tailwind equivalents for TN-4 visual elements:
  - Blue accent (report: `border-left: 3px solid #1e3a5f`): `border-l-4 border-blue-900`
  - Card container (report: `#f9fafb` background): `bg-white rounded shadow border border-gray-200 p-4` (matches existing dashboard card pattern)
  - GS badge (report: `.depth-badge`): `text-xs font-semibold text-gray-500 bg-gray-100 rounded px-1`
  - Reasoning text: `text-sm text-gray-700`
  - Disclaimer (report: `.sort-annotation`): `text-xs italic text-gray-500`
- Game log dates: server-rendered relative format ("5d ago") matching existing `rest_display` pattern. The dashboard does NOT use the report's client-side JS relative-date pattern (`data-date` attribute). Both surfaces show relative dates to coaches but via different mechanisms.
