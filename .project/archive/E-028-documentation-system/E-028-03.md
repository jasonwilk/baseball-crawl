# E-028-03: Admin/developer documentation -- initial content

## Epic
[E-028: Documentation System and Docs-Writer Agent](epic.md)

## Status
`DONE`

## Description
After this story is complete, the `docs/admin/` directory contains a comprehensive set of documentation for Jason as the system operator and developer. The docs cover architecture, getting started, operations, and the agent ecosystem -- everything a technically competent person needs to understand and operate the system.

## Context
Today the project has six technical docs in `docs/` but no high-level overview of how the system works. A new developer or returning operator would need to read CLAUDE.md, all agent definitions, and multiple epic files to piece together the architecture. This story creates the admin documentation that makes the system understandable at a glance and operable with confidence.

## Acceptance Criteria
- [ ] **AC-1**: `docs/admin/README.md` exists as an index page that links to all admin docs with a one-sentence description of each.
- [ ] **AC-2**: `docs/admin/architecture.md` exists and contains:
  - (a) System overview paragraph describing what baseball-crawl does and for whom
  - (b) Component list: FastAPI app, SQLite database, Docker Compose stack, Cloudflare Tunnel, agent ecosystem
  - (c) Data flow description: GameChanger API -> crawl -> parse -> SQLite -> FastAPI/Jinja2 dashboard
  - (d) Directory structure overview (src/, docs/, epics/, migrations/, scripts/, .claude/, data/, tests/)
  - (e) Tech stack summary with rationale references (Python, SQLite, FastAPI+Jinja2, Docker Compose)
  - (f) Cross-references to existing detailed docs (gamechanger-api.md, http-integration-guide.md, cloudflare-access-setup.md)
- [ ] **AC-3**: `docs/admin/getting-started.md` exists and contains:
  - (a) Prerequisites (devcontainer or local Python, Docker, Git)
  - (b) Clone and setup steps (install hooks, install dependencies)
  - (c) How to start the development stack (Docker Compose)
  - (d) How to seed the development database
  - (e) How to verify the system is working (health check, dashboard)
  - (f) How to run tests
  - (g) References to credential management (refresh_credentials.py, .env pattern)
- [ ] **AC-4**: `docs/admin/operations.md` exists and contains:
  - (a) Deployment overview (Docker Compose on home Linux server + Cloudflare Tunnel)
  - (b) Credential rotation procedure (GameChanger tokens, Cloudflare service tokens)
  - (c) Database backup and restore (cross-reference to existing docs/database-restore.md)
  - (d) Common troubleshooting scenarios
  - (e) Monitoring approach (Docker Compose logs, health endpoints)
- [ ] **AC-5**: `docs/admin/agent-guide.md` exists and contains:
  - (a) Overview of the agent ecosystem (what it is, why it exists)
  - (b) Table of all agents with name, role, model, and one-line description
  - (c) How to work with agents (product-manager as entry point for work, direct-routing exceptions)
  - (d) Epic/story workflow from an operator perspective (how to request work, how to check status)
  - (e) How to create a new agent (reference to claude-architect)
- [ ] **AC-6**: Every documentation file includes a "Last updated" date and the story reference (E-028-03) at the top or bottom.
- [ ] **AC-7**: Documentation is accurate against the current state of the codebase -- file paths, commands, and configuration references are verified by reading the actual project files.
- [ ] **AC-8**: No credentials, tokens, or real data appear in any documentation file.

## Technical Approach
This story is assigned to the docs-writer agent. The agent must read the following source files to produce accurate documentation:

- `/workspaces/baseball-crawl/CLAUDE.md` -- Project conventions, tech stack, architecture decisions
- `/workspaces/baseball-crawl/docker-compose.yml` -- Stack configuration
- `/workspaces/baseball-crawl/.devcontainer/devcontainer.json` -- Dev environment config
- `/workspaces/baseball-crawl/src/api/main.py` -- FastAPI app entry point
- `/workspaces/baseball-crawl/migrations/001_initial_schema.sql` -- Database schema
- `/workspaces/baseball-crawl/migrations/apply_migrations.py` -- Migration runner
- `/workspaces/baseball-crawl/scripts/` -- All utility scripts
- `/workspaces/baseball-crawl/.claude/agents/` -- All agent definitions (scan for ecosystem table)
- `/workspaces/baseball-crawl/docs/` -- All existing docs (for cross-referencing)
- `/workspaces/baseball-crawl/.claude/rules/` -- Rules files (for agent-guide context)
- `/workspaces/baseball-crawl/pyproject.toml` -- Python project config
- `/workspaces/baseball-crawl/requirements.txt` -- Dependencies

Write documentation that is useful for a technically competent human (Jason) who understands Python, Docker, and SQL but may not have the full project context loaded. Avoid agent-speak (epics, stories, acceptance criteria) in the getting-started and operations guides -- write for a human operator.
