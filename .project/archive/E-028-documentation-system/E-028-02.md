# E-028-02: Documentation maintenance rules and workflow integration

## Epic
[E-028: Documentation System and Docs-Writer Agent](epic.md)

## Status
`DONE`

## Description
After this story is complete, a rules file at `.claude/rules/documentation.md` defines when documentation must be updated, who owns which docs, how staleness is detected, and -- critically -- how documentation updates are enforced as a required step in the epic lifecycle. Documentation updates are not optional afterthoughts; they are a mandatory part of closing an epic, wired directly into the PM's workflow and the dispatch pattern.

## Context
Documentation that is written once and never maintained is worse than no documentation -- it creates false confidence. Conventions that rely on someone remembering to check docs are equally fragile. This story establishes documentation maintenance as a **required workflow step** -- baked into the PM's epic completion checklist and the dispatch pattern -- so that documentation updates are triggered automatically whenever an epic changes how the system works.

The rule file defines both the conventions (what, who, when) and the workflow integration (the PM must assess documentation impact at epic close-out, and the docs-writer is dispatched when needed). E-028-05 then wires these requirements into the existing workflow infrastructure files (dispatch-pattern.md, workflow-discipline.md, and the PM agent definition).

## Acceptance Criteria
- [ ] **AC-1**: File `.claude/rules/documentation.md` exists and is a valid Claude Code rules file (markdown format with YAML frontmatter, consistent with existing rules in `.claude/rules/`).
- [ ] **AC-2**: The rules file defines **update triggers** -- specific events that require documentation updates. At minimum: (a) new feature or endpoint ships, (b) architecture or deployment config changes, (c) new agent created or existing agent modified, (d) database schema changes, (e) new epic completes that changes how the system works.
- [ ] **AC-3**: The rules file defines **documentation ownership** -- which agent or role is responsible for which docs. At minimum: (a) docs-writer owns `docs/admin/` and `docs/coaching/`, (b) api-scout owns `docs/gamechanger-api.md`, (c) docs-writer references but does not modify agent-maintained docs, (d) claude-architect owns agent definitions and CLAUDE.md.
- [ ] **AC-4**: The rules file defines a **staleness detection convention**: each documentation file should include a "Last updated" date and reference to the epic/story that produced or last modified it. Files not updated in 90+ days should be reviewed when their domain area changes.
- [ ] **AC-5**: The rules file defines the **documentation directory structure**: `docs/admin/` for operator/developer docs, `docs/coaching/` for end-user docs, existing files in `docs/` root remain as agent-maintained references.
- [ ] **AC-6**: The rules file defines a **mandatory documentation assessment step** that the PM must perform when completing any epic. The rule must specify: (a) PM reviews the epic's scope against the update triggers from AC-2, (b) if any trigger fires, PM creates and dispatches a documentation update task to docs-writer before archiving the epic, (c) if no trigger fires, PM records "No documentation impact" in the epic's History section, (d) the assessment happens between "all stories DONE" and "archive the epic" in the completion flow.
- [ ] **AC-7**: The rules file defines the **documentation update task format**: PM provides docs-writer with (a) what changed (epic ID and summary), (b) which docs are affected (specific files in `docs/admin/` or `docs/coaching/`), (c) what needs updating (new content, revised content, or removal of stale content). This is a lightweight dispatch, not a full story.
- [ ] **AC-8**: The rules file is concise (no longer than existing rules files like `.claude/rules/pii-safety.md` or `.claude/rules/testing.md`) and uses imperative language consistent with other rules files.

Key design decisions from epic Technical Notes:
- Two audience directories: `docs/admin/` and `docs/coaching/`
- Existing docs in `docs/` root stay in place, owned by their respective agents
- Staleness detection via "Last updated" date and epic/story reference
- docs-writer is the primary maintainer but other agents have ownership of specific files
- The documentation assessment is a PM responsibility, not an implementing agent responsibility
- docs-writer is dispatched by the PM when the assessment identifies doc impact -- this is the automatic trigger mechanism
