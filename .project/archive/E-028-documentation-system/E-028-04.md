# E-028-04: End-user documentation -- structure and initial content

## Epic
[E-028: Documentation System and Docs-Writer Agent](epic.md)

## Status
`DONE`

## Description
After this story is complete, the `docs/coaching/` directory contains documentation for the coaching staff audience. Since the dashboard (E-004) is still in DRAFT, this story establishes the documentation structure and writes content that is meaningful now: what the system does for coaches, a statistics glossary with interpretation guidance, and placeholder pages for features not yet built.

## Context
The coaching staff are the end-users of this system. They will consume dashboards and scouting reports but have no technical knowledge of the underlying platform. Documentation must explain what they see, what the numbers mean, and how to use the information for game prep. Since the dashboard does not exist yet, this story focuses on the statistics glossary (which is useful regardless of presentation format) and the structural foundation that will be filled in as features ship.

**Baseball-coach consultation**: Before writing statistics content, the docs-writer must consult the baseball-coach agent (via Task tool or by reading the baseball-coach agent's domain knowledge) to ensure: (a) statistics are explained in coaching terms, not sabermetric jargon, (b) sample size caveats are included for every stat, (c) the priority of stats matches what coaches actually use for game prep, (d) the technical level of explanation is appropriate for high school coaching staff.

## Acceptance Criteria
- [ ] **AC-1**: `docs/coaching/README.md` exists as a landing page that explains in plain language: (a) what the baseball-crawl system does for coaches, (b) who maintains it (Jason), (c) how coaches will access it (dashboard -- coming soon), (d) links to other coaching docs.
- [ ] **AC-2**: `docs/coaching/understanding-stats.md` exists and contains:
  - (a) Batting statistics glossary: OBP, SLG, K%, BB%, BABIP -- each with a plain-language definition, the formula, and an interpretation tip (e.g., "A .350 OBP means the batter reaches base about 1 in 3 plate appearances")
  - (b) Pitching statistics glossary: K/9, BB/9, K/BB ratio, FIP (if applicable) -- same format
  - (c) Sample size warnings section explaining why stats from 20 plate appearances are less reliable than stats from 80, with specific thresholds for when to trust each stat
  - (d) Splits explanation: what home/away and L/R splits mean and when they are useful
  - (e) Language appropriate for coaching staff who understand baseball but may not be familiar with sabermetrics
- [ ] **AC-3**: `docs/coaching/scouting-reports.md` exists as a placeholder page with: (a) a description of what scouting reports will contain when the dashboard ships, (b) a "Coming soon" notice, (c) a section header structure that can be filled in when E-004 delivers the feature.
- [ ] **AC-4**: Every documentation file includes a "Last updated" date and the story reference (E-028-04).
- [ ] **AC-5**: Statistics definitions and sample size caveats are consistent with the thresholds established in the baseball-coach agent definition: 20 PA minimum for batting stats, 15 IP minimum for pitching stats.
- [ ] **AC-6**: No technical jargon (Python, SQL, Docker, API, agent) appears in coaching documentation. The audience is coaches, not developers.

## Technical Approach
This story is assigned to the docs-writer agent. Before writing statistics content, the agent should:

1. Read `/workspaces/baseball-crawl/.claude/agents/baseball-coach.md` -- sections "Key Baseball Analytics Knowledge", "Sample Size Warnings", and "What Coaches Actually Use" contain the domain knowledge needed to write accurate coaching-level content.
2. Read `/workspaces/baseball-crawl/CLAUDE.md` -- section "Key Metrics We Track" defines the statistics the system will compute.
3. If the baseball-coach agent is available for consultation (via Task tool), ask: "What statistics need the most explanation for high school coaching staff? What is their typical level of familiarity with sabermetrics? Should the glossary be organized by importance or alphabetically?"

The coaching documentation should read like a guide a head coach would hand to an assistant coach who is new to data-driven game prep. Concrete examples beat abstract definitions. Use hypothetical LSB player examples (fake names) to illustrate stats in context.
