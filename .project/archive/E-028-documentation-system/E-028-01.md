# E-028-01: Create docs-writer agent definition

## Epic
[E-028: Documentation System and Docs-Writer Agent](epic.md)

## Status
`DONE`

## Description
After this story is complete, a fully defined `docs-writer` agent exists in the agent ecosystem. The agent definition follows the canonical section skeleton, has appropriate tool access, and is ready to be dispatched for documentation work. The docs-writer specializes in maintaining human-readable documentation for both the admin/developer audience and the coaching staff audience.

## Context
The project has six agents but none dedicated to documentation. Technical knowledge is scattered across CLAUDE.md, agent definitions, and ad-hoc docs in `docs/`. A dedicated docs-writer agent ensures documentation is treated as a first-class concern with clear ownership, conventions, and quality standards. This story creates the agent; subsequent stories use it.

## Acceptance Criteria
- [ ] **AC-1**: File `.claude/agents/docs-writer.md` exists with valid YAML frontmatter (name, description, model, color, memory, tools) and a markdown body following the canonical section skeleton (Identity, Core Responsibilities, Anti-Patterns, Error Handling, Inter-Agent Coordination, Skill References, Memory).
- [ ] **AC-2**: The `description` field is 1-3 concise sentences suitable for agent routing -- no embedded examples, no multi-paragraph blocks.
- [ ] **AC-3**: The `model` field is `sonnet`, the `color` field is `purple`, and the `tools` list includes exactly: Read, Write, Edit, Glob, Grep (no Bash, no WebFetch).
- [ ] **AC-4**: The Identity section establishes the agent as a documentation specialist for two audiences: admin/developer (Jason) and coaching staff (end-users).
- [ ] **AC-5**: Core Responsibilities cover at minimum: (a) writing and maintaining admin/developer documentation in `docs/admin/`, (b) writing and maintaining end-user documentation in `docs/coaching/`, (c) reading source code and agent definitions to produce accurate technical content, (d) consulting baseball-coach for coaching terminology and end-user needs.
- [ ] **AC-6**: Anti-Patterns include at minimum: (a) never writes application code or tests, (b) never modifies agent definitions, (c) never modifies `docs/gamechanger-api.md` (api-scout territory), (d) never invents statistics or coaching advice without baseball-coach validation.
- [ ] **AC-7**: Inter-Agent Coordination section describes relationships with at minimum: product-manager (dispatches documentation work to docs-writer), baseball-coach (consulted for coaching content accuracy), api-scout (owns API spec -- docs-writer references but does not modify).
- [ ] **AC-8**: Memory section includes a memory directory path and instructions for what to save (documentation decisions, audience feedback, content structure patterns).
- [ ] **AC-9**: The agent definition does not duplicate content from CLAUDE.md -- it references CLAUDE.md for shared conventions.

## Technical Approach
This is a claude-architect story. Read the existing agent definitions at `/workspaces/baseball-crawl/.claude/agents/` for the canonical pattern (especially `baseball-coach.md` and `general-dev.md` as examples of well-structured agents). Follow the canonical section skeleton defined in the claude-architect agent's Standards section.

Key design decisions from epic Technical Notes:
- Model: sonnet (structured writing, not deep reasoning)
- Color: purple (unused in current ecosystem)
- Tools: Read, Write, Edit, Glob, Grep only
- Two audience directories: `docs/admin/` and `docs/coaching/`

The docs-writer is an implementing agent (requires story reference before beginning work), not a direct-routing exception. PM dispatches docs work via stories.

Create the agent memory directory at `/workspaces/baseball-crawl/.claude/agent-memory/docs-writer/` with an initial `MEMORY.md`.
