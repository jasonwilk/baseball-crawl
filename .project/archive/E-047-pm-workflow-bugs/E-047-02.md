# E-047-02: Dispatch Authorization Gate

## Epic
[E-047: PM Workflow Bugs](epic.md)

## Status
`DONE`

## Description
After this story is complete, three files will contain coordinated rules that prevent the PM from auto-dispatching stories after marking an epic READY. The PM will be required to present the finished epic to the user and wait for explicit dispatch authorization. "Define the epic" and "execute the epic" will be clearly documented as separate user actions. A compound-request exception will allow the user to authorize both planning and dispatch in a single statement (e.g., "define and execute").

## Context
The PM currently has no rule preventing it from chaining plan mode into dispatch mode. When asked to "define an epic," the PM sometimes marks it READY and immediately begins dispatching stories without returning to the user for authorization. The Dispatch Flow assumes step 1 is "User requests dispatch," but nothing enforces this boundary when the PM itself just finished formation. Three files need coordinated changes to close this gap.

- [ ] **AC-4**: `.claude/rules/workflow-discipline.md` -- a new "Dispatch Authorization Gate" section exists (after the Epic READY Gate section) stating that READY and dispatch are separate actions, and listing plan-mode phrases that do NOT authorize dispatch.

This prevents forced round-trip confirmations when the user's intent is clear.
