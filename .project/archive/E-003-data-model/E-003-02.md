# E-003-02: Coaching assignments migration

## Epic
[E-003: Data Model and Storage Schema](epic.md)

## Status
`DONE`

## Description
After this story is complete, a `coaching_assignments` table exists in the database, linking coaches (users) to teams and seasons with a role designation. This is a domain concept ("Coach Smith is head coach of JV in Spring 2026"), separate from the auth/access control tables in E-023. The migration applies cleanly after both the data model migration (001) and the auth migration (003).

## Context
Coaching assignments are a core domain relationship that the dashboard and scouting reports will use. A coach can be assigned to multiple teams in a season (e.g., head coach of Varsity, volunteer assistant for JV). The same coach can have different roles in different seasons.

This table has a cross-epic dependency: it FKs to `users(user_id)` from E-023-01's `003_auth.sql` migration, and to `teams(team_id)` and `seasons(season_id)` from E-003-01's `001_initial_schema.sql`. Therefore this migration is numbered 004 -- it must run after both 001 and 003.

The `coaching_assignments` table is distinct from E-023's `user_team_access` table:
- `user_team_access` controls what a user can SEE on the dashboard (auth/access layer)
- `coaching_assignments` records the coaching relationship (domain layer) -- used for scouting reports, lineup cards, historical records

## Technical Approach
Create `migrations/004_coaching_assignments.sql` following the conventions established in `001_initial_schema.sql`:
- Same header comment style
- Same naming conventions (`_id` suffix for PKs, `TEXT NOT NULL DEFAULT (datetime('now'))` for timestamps)
- Use `CREATE TABLE IF NOT EXISTS` and `CREATE INDEX IF NOT EXISTS` for idempotency

Write tests in `tests/test_coaching_assignments.py` that apply migrations 001 + 003 + 004 to an in-memory SQLite database and verify FK enforcement, uniqueness, and the multi-role scenario.

The test will need to create prerequisite data: a season (in seasons table), a team (in teams table), and a user (in users table) before inserting coaching_assignments rows.

## Notes
- Role values are convention-based, not constraint-enforced. The accepted values are: `head_coach`, `assistant`, `volunteer`. No SQL CHECK constraint -- if new roles emerge (e.g., `pitching_coach`), they can be added without a migration.
- This table is intentionally separate from `user_team_access`. A coach might be assigned to a team (domain) but not yet have dashboard access (auth) -- or vice versa. The two tables serve different purposes and are maintained independently.
- If the user later wants to auto-populate `user_team_access` from `coaching_assignments`, that's a feature story, not a schema story.
