# E-003-01: Rewrite core schema migration with seasons, crawl config, and pitching

## Epic
[E-003: Data Model and Storage Schema](epic.md)

## Status
`DONE`

## Description
After this story is complete, `migrations/001_initial_schema.sql` will contain the finalized data model: `seasons` as a first-class entity, `teams` with crawl configuration columns, `team_rosters` and `games` using `season_id` FKs instead of bare text, `player_season_batting` with expanded splits, and a new `player_season_pitching` table. The migration applies cleanly to a fresh SQLite database. Any existing dev database must be deleted and recreated.

## Context
The original `001_initial_schema.sql` was written before several key design decisions were finalized. It lacks: a `seasons` table (uses bare `season TEXT` columns), crawl configuration on `teams` (`source`, `is_active`, `last_synced`), the `player_season_pitching` table, and expanded splits on `player_season_batting`. Since the project is pre-production with no real data, the cleanest approach is to rewrite the migration in place rather than writing ALTER TABLE migrations.

This is the foundation story -- every other story in E-003 and the load stories in E-002 depend on this schema being correct.

## Technical Approach
Rewrite `migrations/001_initial_schema.sql` in place. Follow the existing conventions from the current file (header comment style, naming conventions, inline comments). Reference the full schema specification in the epic's Technical Notes section at `/workspaces/baseball-crawl/.project/archive/E-003-data-model/epic.md`.

Key differences from the current file:
1. Add `seasons` table (must be created BEFORE tables that FK to it)
2. Add `source`, `is_active`, `last_synced` columns to `teams`
3. Change `season TEXT` to `season_id TEXT REFERENCES seasons(season_id)` in `team_rosters`, `games`, `player_season_batting`
4. Expand `player_season_batting` splits to include hr, bb, so per split group
5. Add `player_season_pitching` table with splits and pitch counts
6. Add index on `teams(is_active)` and `player_season_pitching(player_id, season_id)`
7. Use consistent naming: pitching batter splits are `vs_lhb`/`vs_rhb` (vs left/right-handed BATTERS), batting pitcher splits are `vs_lhp`/`vs_rhp` (vs left/right-handed PITCHERS)

Write tests in `tests/test_schema.py` that apply the migration to an in-memory SQLite database and verify table existence, FK enforcement, and the crawl config query.

Important: Since this rewrites an existing migration that may have been applied to dev databases, the implementing agent should note in the migration header comment that any existing `data/app.db` must be deleted and recreated. The `_migrations` table tracks applied filenames only (not content hashes), so the filename `001_initial_schema.sql` still matches the existing `_migrations` entry -- the migration runner will skip the rewritten file on an already-migrated database. Deleting and recreating the database is the only way to apply the new schema.

## Notes
- The `players`, `player_game_batting`, and `player_game_pitching` tables are structurally unchanged from the original migration. Only their surrounding context (new tables, updated FKs in other tables) changes.
- Migration 002 slot was previously reserved for the stats schema but is now unused. E-023 auth migration stays at 003.
- The `player_season_pitching` splits use `vs_lhb`/`vs_rhb` (batters), while `player_season_batting` splits use `vs_lhp`/`vs_rhp` (pitchers). This is intentional -- batting splits are against pitcher handedness, pitching splits are against batter handedness.
- All ID fields use `TEXT` -- GameChanger IDs are opaque strings (confirmed by E-001-03).
- `is_active` on teams defaults to 1, meaning newly discovered teams are crawled by default. The operator can set `is_active=0` to exclude teams.
- The dev seed (`data/seeds/seed_dev.sql`) and its tests (`tests/test_seed.py`) must be updated in the same story because `scripts/reset_dev_db.py` always runs migrations then loads the seed. Leaving the seed on the old schema would break the dev reset flow and cause test regressions. E-003-04 creates a *separate* test fixture (`tests/fixtures/seed.sql`) for query validation -- it does not modify `seed_dev.sql`.
