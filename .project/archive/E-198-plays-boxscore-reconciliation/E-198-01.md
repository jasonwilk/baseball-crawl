# E-198-01: Reconciliation Detection Engine + Schema

## Epic
[E-198: Plays-vs-Boxscore Reconciliation Engine](epic.md)

## Status
`DONE`

## Description
After this story is complete, the system will have a reconciliation detection engine that compares plays-derived aggregates against boxscore ground truth for every game, logs structured discrepancy records to a new `reconciliation_discrepancies` table, and exposes a `bb data reconcile` CLI command (bare command = dry-run mode) that reports the discrepancy distribution across all games. Detection covers 11 pitcher signals, 5 core batter signals (detection only, no correction), and 3 game-level sanity checks. No corrections are applied -- this is detection and logging only, producing the data needed to inform the correction algorithm in E-198-02.

## Context
The plays pipeline (E-195) has ~55% FPS match rate and ~53.8% QAB match rate. The boxscore data is already loaded into `player_game_pitching` and `player_game_batting` by the game_loader. This story builds the detection infrastructure: schema, engine core, pitcher signal checkers, batter signal checkers (to explain the QAB gap), game-level sanity checks, and the CLI dry-run command. The discrepancy distribution from this story's output directly informs the correction algorithm design in E-198-02.

## Acceptance Criteria
- [ ] **AC-1**: Migration 012 creates the `reconciliation_discrepancies` table per the schema in Technical Notes, including:
  - `team_id INTEGER NOT NULL REFERENCES teams(id)` column scoping each row to one team
  - 5-value status CHECK constraint (`MATCH`, `CORRECTABLE`, `CORRECTED`, `AMBIGUOUS`, `UNCORRECTABLE`)
  - UNIQUE constraint on `(run_id, game_id, team_id, player_id, signal_name)`
  - `'__game__'` sentinel convention for game-level signals' `player_id`
- [ ] **AC-2**: `src/reconciliation/engine.py` implements `reconcile_game(conn, game_id, dry_run=True)` that:
  - Skips games with no rows in the `plays` table (logs a warning, returns early)
  - Loads boxscore data from `player_game_pitching` and `player_game_batting` for both teams in the game
  - Loads plays data from `plays` and `play_events` for the game
  - Extracts pitcher appearance order from the cached boxscore JSON file (per Pitcher Order Extraction in Technical Notes; tries `{game_id}.json` then `{game_stream_id}.json` for backward compat), falling back to `player_game_pitching` ordered by `id` with a warning if no JSON found
  - Compares 11 pitcher signals (1A-1I, 1K, 1L per Technical Notes; 1J R/ER deferred), 5 batter signals (2A-2E), and 3 game-level signals (game_runs, game_hits, game_pa_count; game_errors dropped)
  - Writes one `reconciliation_discrepancies` row per signal per player per team (game-level signals use `'__game__'` as player_id, one row per team)
  - Returns a summary dataclass with signal match counts
- [ ] **AC-3**: Each signal checker derives the plays-side value per the formulas in the Technical Notes signal tables (Pitcher Signals, Batter Detection Signals, Game-Level Sanity Checks sections). Key implementation notes:
  - IP/outs (1C): status always AMBIGUOUS (did_outs_change undercounts non-PA outs)
  - WP (1K): status always AMBIGUOUS (template coverage incomplete)
  - All batter signals: status MATCH or AMBIGUOUS only (no correction path)
  - PROBABILISTIC is a confidence annotation in the signal table, NOT a status value; maps to AMBIGUOUS
- [ ] **AC-4**: Batter signal checkers (detection only, no correction) cover AB, H, SO, BB, HBP per batter per the formulas in Technical Notes Batter Detection Signals table.
- [ ] **AC-5**: Game-level sanity checks implemented per Technical Notes, using DB aggregates (not JSON team_stats), one discrepancy row per signal per team:
  - `game_pa_count`: `SUM(player_game_pitching.bf)` for the team's pitchers vs total plays count for that team's half-innings
  - `game_runs`: `SUM(player_game_batting.r)` for the team vs plays-derived runs (final play's home_score/away_score)
  - `game_hits`: `SUM(player_game_batting.h)` for the team vs plays-derived hit outcomes
  - Note: `game_errors` dropped from Phase 1 (boxscore errors are fielding-perspective, plays errors are batting-perspective; no clean cross-check)
- [ ] **AC-6**: Status assignment uses only the 5 schema values (`MATCH`, `CORRECTABLE`, `CORRECTED`, `AMBIGUOUS`, `UNCORRECTABLE`): `MATCH` when delta == 0; `CORRECTABLE` when pitcher order or BF divergence indicates a reassignable boundary; `AMBIGUOUS` for IP/outs, WP (PROBABILISTIC-confidence signals), and multi-candidate corrections; `UNCORRECTABLE` for missing data. Batter signals use `MATCH` or `AMBIGUOUS` only.
- [ ] **AC-7**: `bb data reconcile` CLI command (default: dry-run) processes all games with plays data and prints a summary: total games, games with all signals matching, games with correctable pitcher errors, games with ambiguous errors, per-signal match rates for pitcher and batter signals separately.
- [ ] **AC-8**: `bb data reconcile --game-id <id>` processes a single game with verbose per-signal output.
- [ ] **AC-9**: Tests cover: signal detection for a game with perfect pitcher attribution (all MATCH), signal detection for a game with known pitcher misattribution (CORRECTABLE signals), batter signal detection, game-level sanity checks (per-team rows), skip behavior for games without plays data, CLI smoke test.

## Technical Approach
The engine reads from `player_game_pitching` and `player_game_batting` (boxscore ground truth), `plays` + `play_events` (plays-derived data), and cached boxscore JSON files for pitcher order extraction. The reconciliation module lives at `src/reconciliation/` as a separate package. The detection engine iterates through pitcher, batter, and game-level signals per Technical Notes. Each comparison produces a discrepancy row scoped to one team. The CLI command in `src/cli/data.py` adds a `reconcile` subcommand. See epic Technical Notes for the full signal tables, schema, pitcher order extraction, and architecture.

## Notes
- **Boxscore file paths** (verified against crawler code 2026-04-01): Member teams: `data/raw/{season_id}/teams/{gc_uuid}/games/{game_id}.json` (filename = `event_id` = `games.game_id`). Backward compat: also try `{game_stream_id}.json` in the same directory (the game_loader's dual-key index supports both). Scouting teams: `data/raw/{season_id}/scouting/{public_id}/boxscores/{game_stream_id}.json`. See Technical Notes Pitcher Order Extraction for full lookup procedure and join chain.
- Pitcher order MUST be extracted from cached boxscore JSON (not DB AUTOINCREMENT order) per Technical Notes. The pitching group `stats` array order is authoritative.
- Sparse extras (BF, #P, TS, WP, HBP) are only present for non-zero values in the boxscore; absent means 0
- E-195 validation results at `/.project/research/E-195-validation-results.md` provide per-pitcher diagnostics useful for test case design
- Total strikes (signal 1H) requires joining `play_events` to count strike pitch results per pitcher -- this crosses the plays/play_events boundary
- SO outcome strings (`'Strikeout'`, `'Dropped 3rd Strike'`) must be empirically validated against boxscore SO counts on the first detection run. If mismatches are systematic, the outcome strings may need adjustment.
- The batter detection pass may reveal whether the 53.8% QAB gap is from batter misattribution, outcome classification differences, or QAB definition differences between our logic and GC's. This diagnostic insight is a key output of story 01.
- R/ER (signal 1J) is deferred from Phase 1 -- requires inherited runner tracking not present in plays data.
- **Catcher's Interference escaping**: The outcome string `"Catcher's Interference"` contains an apostrophe. Use parameterized queries (not string interpolation) to avoid SQL escaping issues in the AB exclusion list.
- **AB exclusion list validation**: Like SO outcomes, the AB exclusion list (`Walk`, `Hit By Pitch`, `Sacrifice Fly`, `Sacrifice Bunt`, `Catcher's Interference`, `Intentional Walk`) should be empirically validated against boxscore AB counts on the first detection run.
