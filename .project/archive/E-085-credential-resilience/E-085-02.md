# E-085-02: Enhanced `bb creds check` Output

## Epic
[E-085: Credential Resilience and Diagnostic Improvements](epic.md)

## Status
`DONE`

## Description
After this story is complete, `bb creds check` will provide a comprehensive diagnostic report covering credential presence, token expiry status (decoded locally from the JWT), API reachability via `GET /me/user`, Bright Data proxy configuration status, and clear actionable next steps. The output will use Rich formatting for readability.

## Context
Currently `bb creds check` calls `check_single_profile()` which instantiates `GameChangerClient` and calls `GET /me/user`. The output is a single line: "valid -- logged in as NAME" or an error message. This is insufficient for operators diagnosing issues -- they don't know whether the problem is missing credentials, expired tokens, proxy misconfiguration, or network issues. The enhanced version should report each diagnostic dimension separately so the operator can act on the specific problem.

The Bright Data proxy check logic already exists in `src/http/proxy_check.py` (used by `bb proxy check`). The JWT decode logic for extracting `exp` already exists in `src/cli/creds.py` (`_decode_jwt_exp()`).

## Acceptance Criteria
- [ ] **AC-1**: Given all credentials present and valid, `bb creds check --profile web` outputs a multi-section report using `[OK]`/`[!!]`/`[XX]`/`[--]` status indicators, showing: (a) credential presence (which keys are set), (b) refresh token expiry (decoded locally -- "expires in N days" or "expired N days ago"), (c) API health check result ("GET /me/user -> 200, logged in as NAME" -- display name only, no PII), (d) proxy status (configured/not configured, and if configured, routing correctly or not)
- [ ] **AC-2**: Given missing credentials, the output clearly lists which `.env` keys are missing without revealing any values
- [ ] **AC-3**: Given an expired refresh token (decodable but `exp` is in the past), the output reports the token as expired with the expiry date, AND still attempts the API health check (because the login fallback from E-085-01 may recover it, or the access token may still be cached)
- [ ] **AC-4**: Given proxy is configured (`PROXY_ENABLED=true` + `PROXY_URL_WEB` set), the output includes proxy routing status (reuse `check_proxy_routing()` from `src/http/proxy_check.py`)
- [ ] **AC-5**: Given proxy is not configured, the output shows "Proxy: not configured" (informational, not an error)
- [ ] **AC-6**: The test endpoint (`GET /me/user`) is displayed in the output so operators know what was tested
- [ ] **AC-7**: Multi-profile mode (`bb creds check` with no `--profile`) outputs the same enhanced report for each profile
- [ ] **AC-8**: Exit codes are preserved: 0 = valid, 1 = expired/error, 2 = missing credentials
- [ ] **AC-9**: No credential values (tokens, passwords, proxy URLs) appear in the output -- only key names, status indicators, and decoded metadata (expiry dates, user display names)
- [ ] **AC-10**: All new code has test coverage. Tests mock HTTP calls and `.env` reads.

## Technical Approach
The enhanced check combines several existing capabilities:
- JWT `exp` decoding: `_decode_jwt_exp()` in `src/cli/creds.py` (or factor into a shared utility)
- API health check: existing `check_single_profile()` in `src/gamechanger/credentials.py`
- Proxy routing check: `check_proxy_routing()` from `src/http/proxy_check.py`
- Credential presence: `dotenv_values()` against the known key names

The current `credentials.py` `check_single_profile()` function returns `(exit_code, message)` -- a single string. The enhanced version needs to return richer structured data so the CLI layer can format it properly with Rich.

## Notes
- The `_decode_jwt_exp()` function currently lives in `src/cli/creds.py`. If it's useful in `credentials.py` too, consider moving it to a shared location (implementer discretion).
- `check_proxy_routing()` already handles the "not configured" case via `ProxyCheckOutcome.NOT_CONFIGURED` -- reuse that.
- Be careful with the proxy check timeout -- it makes real HTTP calls to an IP-echo service. Consider whether this should be opt-in (e.g., `--include-proxy` flag) or always-on. PM recommendation: always-on since it's fast and operators expect a complete diagnostic.
- UXD visual language: use `[OK]` (green), `[!!]` (yellow), `[XX]` (red), `[--]` (dim) status indicators. Sectioned panel layout. Mobile token health should always show `[!!]` -- honest about no auto-refresh.
- api-scout: `GET /me/user` only accepts access tokens (not refresh). Response contains PII fields -- display only user's display name, never log the full response.
- `ProxyCheckOutcome.PASS_UNVERIFIED` (proxy succeeded but direct baseline failed, so anonymization is unconfirmed) should render as `[!!]` (yellow) with message "routing unverified (direct baseline unavailable)". This aligns with the shared visual language in E-085-03.
