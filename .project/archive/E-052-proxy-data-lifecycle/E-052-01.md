# E-052-01: Session directory structure and start/stop lifecycle

## Epic
[E-052: Proxy Data Lifecycle](epic.md)

## Status
`IN_PROGRESS`

## Description
After this story is complete, every `start.sh` invocation creates a new timestamped session directory under `proxy/data/sessions/`, writes a `session.json` metadata file with `status: "active"`, and sets a `current` symlink. `stop.sh` finalizes the session by updating `session.json` with `stopped_at`, `status: "closed"`, and leaving the `current` symlink in place (pointing to the now-closed session). The session ID is passed to the mitmproxy container via `PROXY_SESSION_DIR` so addons can route output there (addon changes are in E-052-02).

## Context
Today `start.sh` and `stop.sh` are simple wrappers around `docker compose up/down`. They have no concept of a capture session. This story adds the session lifecycle scaffolding that all other stories depend on.

## Acceptance Criteria
- [ ] **AC-1**: `start.sh` creates `proxy/data/sessions/YYYY-MM-DD_HHMMSS/` using the current UTC timestamp, with the format matching `date -u +%Y-%m-%d_%H%M%S`.
- [ ] **AC-2**: `start.sh` writes `proxy/data/sessions/<id>/session.json` with fields: `session_id` (string, matches dir name), `profile` (string, from `--profile` arg), `started_at` (ISO 8601 UTC), `stopped_at` (null), `status` ("active"), `endpoint_count` (0), `reviewed` (false), `review_notes` ("").
- [ ] **AC-3**: `start.sh` creates (or updates) a symlink at `proxy/data/current` pointing to the new session directory.
- [ ] **AC-4**: `start.sh` exports `PROXY_SESSION_DIR=/app/proxy/data/sessions/<id>` as an environment variable available to the mitmproxy container (via docker-compose.yml environment section or `.env` passthrough).
- [ ] **AC-5**: `stop.sh` reads the `current` symlink to find the active session, updates `session.json`: sets `stopped_at` to current UTC ISO 8601, `status` to `"closed"`, and `endpoint_count` to the number of lines in `endpoint-log.jsonl` (0 if the file does not exist).
- [ ] **AC-6**: `stop.sh` leaves the `current` symlink in place, pointing to the now-closed session. The symlink is only updated (never removed) by `start.sh`.
- [ ] **AC-7**: If `start.sh` is called while a `current` symlink already exists (previous session not stopped), it logs a warning but still creates the new session and updates the symlink. The previous session's `session.json` is NOT modified (it will remain `status: "active"` as a signal of unclean shutdown).
- [ ] **AC-8**: `proxy/docker-compose.yml` passes `PROXY_SESSION_DIR` to the mitmproxy container's environment.
- [ ] **AC-9**: `proxy/data/sessions/` directory is created by `start.sh` if it does not exist (via `mkdir -p`).

## Technical Approach
- Use `date -u +%Y-%m-%d_%H%M%S` in bash for the session ID. On macOS, `date -u` works natively.
- Write `session.json` using a here-doc or `jq` (jq is already a project dependency for the report scripts). `jq -n` can construct JSON without input.
- For AC-4, add `PROXY_SESSION_DIR` to the `environment` list in `proxy/docker-compose.yml` as `PROXY_SESSION_DIR=${PROXY_SESSION_DIR}` and export it in `start.sh` before calling `docker compose up`.
- For AC-5, use `jq` to update `session.json` in place: `jq '.stopped_at = "..." | .status = "closed" | .endpoint_count = N' session.json > tmp && mv tmp session.json`.
- For AC-7, check `[ -L proxy/data/current ]` and warn if true.
- The `current` symlink should be relative (e.g., `sessions/2026-03-06_143022`) so it works both on the host and inside the container (which mounts proxy/data at `/app/proxy/data`).
