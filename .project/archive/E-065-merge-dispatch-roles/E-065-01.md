# E-065-01: Rewrite Dispatch Pattern for Merged Roles

## Epic
[E-065: Merge Team Lead and PM Roles During Dispatch](epic.md)

## Status
`DONE`

## Description
After this story is complete, `/.claude/rules/dispatch-pattern.md` will describe a two-role dispatch model where the main session acts as both spawner and coordinator. The old three-role model (team lead / PM / implementers) will be replaced with a two-role model (main session coordinator / implementers). All dispatch coordination responsibilities currently assigned to PM will move to the main session.

## Context
This is the foundation story for E-065. The dispatch-pattern.md file is the canonical reference for how dispatch works -- all other files (implement skill, PM agent def, CLAUDE.md, workflow-discipline.md, multi-agent-patterns) reference its model. Getting this file right first ensures the other stories have a consistent source of truth to align with.

The current file has these major sections that need rewriting:
- "CRITICAL: Team Lead Spawning Responsibility" -- the entire framing changes
- "Team Composition" (three roles -> two roles)
- "Spawning Scenarios" (PM no longer relays spawn requests)
- "PM Responsibilities During Dispatch" (these move to the main session)
- "The Dispatch Flow" (steps 1-10 change, PM relay steps removed)
- "Closure Sequence" (PM references become main session references)
- "Task Tool vs. Agent Teams" (PM coordination reference changes)
- "Dispatch Team metadata" and "Agent Hint" paragraphs (PM routing authority references)

Sections that stay mostly unchanged:
- "Agent Selection for Dispatch" table (routing table is the same)
- "Specialist Agent Responsibilities" (implementers still work the same way)
- "Context Packaging" (format stays the same, but the sender changes from PM to main session)
- "Routing Precedence" (context-layer routing rule stays)

## Acceptance Criteria
- [ ] **AC-1**: The file describes exactly two dispatch roles: (1) main session (spawner + coordinator) and (2) implementing agents. No references to a separate PM teammate role during dispatch remain.
- [ ] **AC-2**: The dispatch flow describes the main session reading the epic, spawning implementers directly via Agent tool, assigning stories with full context blocks, monitoring completion, verifying ACs, and cascading to newly unblocked stories -- with no SendMessage relay through PM.
- [ ] **AC-3**: The closure sequence retains all current steps (validate, update epic, docs assessment, archive, memory update, ideas review, summary, commit offer) but assigns them to the main session instead of PM.
- [ ] **AC-4**: The "Dispatch Team metadata" paragraph explains that the main session reads the Dispatch Team section to determine which implementers to spawn (no PM in the spawn list). The epic template's Dispatch Team section comment (`/.project/templates/epic-template.md`) is updated to remove the "PM is always included automatically" language -- PM is not part of the dispatch team in the merged model.
- [ ] **AC-5**: The Agent Selection routing table is unchanged. The "Routing Precedence" rule for context-layer files is unchanged.
- [ ] **AC-6**: The "Context Packaging" section retains the requirement for full story file text + full Technical Notes + Handoff Context from upstream dependencies. The sender is now the main session, not PM.
- [ ] **AC-7**: The "Task Tool vs. Agent Teams" section clarifies that Agent Teams dispatch is now coordinated by the main session directly, while the Task tool remains available for consultations.
- [ ] **AC-8**: Multi-wave spawning is simplified: the main session reviews the dependency graph and spawns wave-1 agents immediately, then spawns later-wave agents directly as dependencies complete (no relay through PM).
- [ ] **AC-9**: The Dispatch Failure Protocol reference is preserved (report to user, ask how to proceed, never improvise).
- [ ] **AC-10**: No references to "team lead" as a separate idle role remain. If the term "team lead" is used, it refers to the main session in its combined spawner+coordinator capacity.

## Technical Approach
The current file at `/.claude/rules/dispatch-pattern.md` is approximately 150 lines. This is a near-complete rewrite -- the structure can be preserved but most section content changes. The implementing agent should read the current file in full, understand every section's purpose, and rewrite each section to reflect the merged model described in the epic's Technical Notes.

Key constraint: this file is loaded by all agents via the `paths: ["**"]` frontmatter. It must be clear, unambiguous, and self-contained. Other files (implement skill, PM agent def, workflow-discipline.md) will be updated in parallel stories to reference the model defined here.
