# E-065-03: Simplify PM Agent Definition -- Remove Dispatch Mode

## Epic
[E-065: Merge Team Lead and PM Roles During Dispatch](epic.md)

## Status
`DONE`

## Description
After this story is complete, the PM agent definition at `/.claude/agents/product-manager.md` will no longer contain a Dispatch Mode section. All dispatch coordination responsibilities (story assignment, AC verification, status updates during dispatch, closure sequence) will have been removed. The PM retains its five task types (discover, plan, clarify, triage, close) and all other capabilities unchanged.

## Context
The PM agent definition currently contains a large Dispatch Mode section (lines 159-239) that describes the PM's role as "Standing Team Coordinator" during dispatch. With the merged model, the main session handles all dispatch coordination directly. The PM is no longer spawned as a teammate during dispatch.

Sections to remove or simplify:
- "Dispatch Mode" header and "Your Role: Standing Team Coordinator" -- remove entirely
- "Dispatch Procedure" (steps 1-11g) -- remove entirely (this moves to the implement skill)
- "Context Block Format" -- remove (lives in dispatch-pattern.md and the implement skill)
- References to "team lead" as a separate agent the PM messages during dispatch -- remove
- Skill loading triggers that reference "Entering Dispatch Mode" -- update or remove

Sections to preserve unchanged:
- Identity, Core Principle, Philosophy
- Task Types (all five: discover, plan, clarify, triage, close)
- Technical Delegation Boundaries
- Anti-Patterns (except any that reference dispatch coordination)
- Consultation Triggers
- Numbering Scheme, File Organization
- System of Work (Epic/Story Statuses, How Work Flows, Parallel Execution Rules)
- Atomic Status Update Protocol (PM still owns status updates during planning/close)
- Decision Gates, Ideas Workflow, Context Awareness
- Quality Checklist, Memory Instructions

## Acceptance Criteria
- [ ] **AC-1**: The "Dispatch Mode" section (including "Your Role: Standing Team Coordinator", "Dispatch Procedure" steps 1-11g, and "Context Block Format") is removed from the PM agent definition.
- [ ] **AC-2**: The PM's five task types (discover, plan, clarify, triage, close) remain intact and unchanged.
- [ ] **AC-3**: Anti-Pattern 4 ("Never dispatch context-layer stories to general-purpose") is preserved but reworded if it references PM's dispatch procedure. The routing rule itself stays.
- [ ] **AC-4**: The Skills section's `filesystem-context` and `multi-agent-patterns` loading triggers are updated to remove references to "Entering Dispatch Mode." If these skills are still relevant for planning/refinement contexts, keep those triggers.
- [ ] **AC-5**: The "Atomic Status Update Protocol" sections for "Completing a story", "Completing an epic", and "Pre-dispatch" are revised. "Pre-dispatch" is rewritten for close-mode use only: PM still reads the epic directory and verifies story statuses during close mode, but the steps that mark stories IN_PROGRESS and dispatch them are removed (those now live in the implement skill). "Completing a story" and "Completing an epic" are preserved -- PM still uses these during close mode.
- [ ] **AC-6**: References to "requesting the team lead to spawn additional agents" or "messaging the team lead" are removed from the PM agent definition.
- [ ] **AC-7**: The "How Work Flows" section's step 5 (Execution) is updated to reflect that the main session coordinates dispatch, not PM. The PM's role in this step is limited to setting the epic to READY and presenting it to the user.
- [ ] **AC-8**: No references to PM being spawned as a teammate, coordinating via SendMessage, or acting as a "Standing Team Coordinator" remain in the file. The YAML frontmatter `description` field is updated to remove "dispatches implementation work" (PM plans and closes, the main session dispatches).

## Technical Approach
The current file at `/.claude/agents/product-manager.md` is approximately 337 lines. The Dispatch Mode section spans roughly lines 159-239 (about 80 lines). Removing this section and updating references throughout the file is the primary work.

The implementing agent should read the rewritten dispatch-pattern.md from E-065-01 to understand what dispatch responsibilities have moved to the main session, ensuring no accidental duplication or contradiction.

Key constraint: the PM agent definition's YAML frontmatter (name, description, model, color, memory, tools) must be preserved. The `description` field may need updating if it mentions dispatch coordination.
