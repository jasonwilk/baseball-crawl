# E-065-02: Rewrite Implement Skill for Direct Coordination

## Epic
[E-065: Merge Team Lead and PM Roles During Dispatch](epic.md)

## Status
`DONE`

## Description
After this story is complete, the implement skill at `/.claude/skills/implement/SKILL.md` will provide a complete dispatch workflow where the main session reads the epic, spawns implementers directly, assigns stories, monitors completion, verifies ACs, manages statuses, and runs the closure sequence. The skill absorbs PM's dispatch coordination procedure and becomes the primary reference for how the main session executes an epic.

## Context
The current implement skill is deliberately thin -- it handles trigger recognition, prerequisite checks, team composition extraction, and spawning PM + implementers. The actual coordination logic lives in dispatch-pattern.md and the PM agent definition.

With the merged model, the implement skill must absorb the coordination logic that was previously in the PM agent definition's Dispatch Mode section. The skill becomes the step-by-step procedure the main session follows during dispatch. It references dispatch-pattern.md for the canonical model but contains the full operational procedure.

Key changes:
- Phase 2 (Dispatch) no longer spawns PM -- it spawns implementers only
- New coordination phases: story assignment, monitoring, AC verification, cascade
- Closure sequence moves here from PM's dispatch procedure
- The "and review" chain stays but is handled by the main session directly
- Anti-patterns section updates to remove "do not bypass PM" and "do not read story files"
- The main session now owns story file reading, routing decisions, and status updates

## Acceptance Criteria
- [ ] **AC-1**: Prerequisites section is unchanged (epic exists, status is READY or ACTIVE).
- [ ] **AC-2**: Phase 1 (Team Composition) reads the Dispatch Team section. PM is NOT listed as an agent to spawn. Only implementer agent types are extracted.
- [ ] **AC-3**: Phase 2 (Dispatch) describes the main session creating the team and spawning implementers directly via Agent tool -- no PM teammate is spawned.
- [ ] **AC-4**: A new coordination phase describes the full dispatch procedure: reading eligible stories (TODO with satisfied deps), marking them IN_PROGRESS, assigning to implementers with full context blocks (full story file text + full Technical Notes + Handoff Context), monitoring completion, verifying all ACs, marking DONE, and cascading to newly unblocked stories.
- [ ] **AC-5**: The context block format matches the format defined in `/.claude/rules/dispatch-pattern.md` (full story file, full Technical Notes, completed dependencies, Handoff Context).
- [ ] **AC-6**: The closure sequence includes all steps: validate all work, update epic to COMPLETED, documentation assessment, archive epic, update PM memory, review ideas backlog, present summary to user, offer to scan and commit.
- [ ] **AC-7**: Multi-wave spawning is described: main session reviews the dependency graph, spawns wave-1 implementers, then spawns later-wave implementers directly as dependencies complete.
- [ ] **AC-8**: The "and review" modifier chain is preserved. The main session chains into the review-epic workflow after all stories are verified DONE but before the closure sequence.
- [ ] **AC-9**: Edge cases section covers: epic not found, epic is DRAFT/COMPLETED/ABANDONED, no TODO stories with satisfied dependencies, implementer spawn failure (follow Dispatch Failure Protocol).
- [ ] **AC-10**: Anti-patterns section reflects the merged model. Removes outdated anti-patterns about "not bypassing PM" and "not reading story files." Removes the old "do not modify the epic file" anti-pattern (the main session now owns epic file updates during dispatch -- status changes, history entries, closure). Retains anti-patterns about not implementing stories yourself (the coordinator must not also be an implementer).
- [ ] **AC-11**: The workflow summary diagram reflects the merged model (main session reads epic, spawns implementers, coordinates, runs closure -- no PM in the dispatch flow).

## Technical Approach
