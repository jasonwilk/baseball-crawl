## Research Question
What is the right API/backend layer for baseball-crawl, evaluated separately for each
deployment path?

- **Option A (Native Cloudflare)**: The API layer is Cloudflare Workers (TypeScript or
  JavaScript). What does this mean for code organization, developer experience, and
  Python-to-TypeScript context switching for the team?
- **Option B (Docker + Cloudflare Access)**: The API layer is a Python HTTP framework
  (FastAPI, Flask, or another). Which framework fits best given the requirements?

## Context

### Requirements for the API Layer (Both Options)
1. Serves JSON endpoints for machine consumption (crawlers posting data, future clients)
2. Serves rendered HTML pages for the coaching dashboard (coaches on mobile)
3. Runs in local dev without a full cloud deploy
4. Reads from the project database (D1 for Option A, SQLite/Postgres for Option B)
5. Stays simple -- no microservices, no message queues
6. The existing codebase is Python -- keeping the API in Python (Option B) minimizes
   the language surface area

### Why This Question Exists
The current plan (Cloudflare Workers) cannot run locally without Miniflare, which is a
proprietary simulator that lags behind production. Option B eliminates this entirely by
using a standard Python HTTP framework that runs identically in Docker everywhere.

### Candidates to Evaluate

#### Option A Candidates
- **Cloudflare Workers (TypeScript)**: The existing plan. JavaScript/TypeScript runtime,
  D1 bindings built in, Pages for the frontend. Local dev via `wrangler dev`.
- **Python Workers (experimental)**: Cloudflare has an experimental Python runtime for
  Workers. Evaluate its maturity and D1 compatibility.

#### Option B Candidates
- **FastAPI + Jinja2**: Python, minimal, auto-generates OpenAPI, supports both JSON
  endpoints and server-rendered HTML. The PM's prior for Option B.
- **Flask + Jinja2**: Older but simpler than FastAPI; synchronous by default; very
  mature. Good if async complexity is not needed.
- **Django**: Full-featured, admin panel included. Significant overhead for this scope.

The note must answer:

1. **Option A -- TypeScript Workers in 2026**: What is the current developer experience
   for Cloudflare Workers + D1 + Pages? Does `wrangler dev` give a reliable local
   environment for API and dashboard development? What is the Python-to-TypeScript context
   cost for a team that writes all crawlers in Python?

2. **Option A -- Python Workers viability**: Is Cloudflare's Python Workers runtime
   production-ready as of 2026? Does it support D1 bindings? Can it serve Jinja2-rendered
   HTML? If yes, this would be a strong argument for Option A (Python end-to-end on
   Cloudflare). If no, note it as not viable.

3. **Option B -- FastAPI vs. Flask for this use case**: Is FastAPI's async model a
   meaningful benefit or an unnecessary complication for a read-mostly dashboard API?
   Does Flask's simplicity win, or does FastAPI's auto-docs provide enough value?

4. **Option B -- Async + SQLite risk**: FastAPI's async event loop can be blocked by
   synchronous SQLite calls. Describe the risk and the standard mitigation (thread pool
   executor or aiosqlite). Is this a meaningful complexity cost, or a one-time setup?

5. **Option B -- Single process vs. two processes**: Should the API (JSON endpoints) and
   the dashboard (HTML pages) be the same FastAPI app or separate services?
   - Single app: simpler Docker Compose, one port, shared database connection, no CORS
   - Two services: explicit API/UI boundary, easier to replace one side independently
   For this scale and team size, which is right?

6. **Option B -- Docker image size and startup**: What is a typical Docker image size
   and cold-start time for a FastAPI + Jinja2 app? Does this matter for VPS deployment
   where the container is always running (not scale-to-zero)?

7. **Recommendation per path**: One clear API layer recommendation for Option A and one
   for Option B, with a paragraph each explaining why.
