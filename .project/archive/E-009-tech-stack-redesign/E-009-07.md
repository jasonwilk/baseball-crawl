# E-009-07: Production Deployment Runbook

## Epic
[E-009: Tech Stack Redesign -- Portable, Agent-Browsable, Cloudflare-Integrated](epic.md)

## Status
`DONE` (agent phase complete; AC-3/4/5/6 operator verification deferred to user)

## Description
After this story is complete, a self-contained production deployment runbook exists at
`docs/production-deployment.md`. A developer who has never touched this project can follow
the runbook to deploy the full baseball-crawl stack to any Linux server with Docker and
reach the dashboard over HTTPS via Cloudflare Access. The runbook is verified against a
real server -- not just written from memory.

## Context
E-009-02 and E-009-03 established the `docker-compose.yml` and application code. E-009-04
set up Cloudflare Tunnel and Zero Trust Access. E-009-05 set up database seeding and a simple backup script. This
story assembles all of that into a single, verified deployment procedure.

The defining characteristic of Option B (Docker + Cloudflare Access) is that the same
`docker-compose.yml` runs locally and in production. The only differences are environment
variables and whether the `cloudflared` service is active. This story confirms that
invariant actually holds.

## Acceptance Criteria

- [ ] **AC-1**: `docs/production-deployment.md` is written and committed. It covers every
  step from "Linux server with Docker not yet installed" to "dashboard accessible over HTTPS":
  1. Server prerequisites (Linux server with Ubuntu 24.04 LTS or equivalent, Docker installed)
  2. Docker and Docker Compose installation
  3. Repository clone
  4. `.env` file creation from `.env.example`
  5. Cloudflare Tunnel token retrieval and configuration
  6. `docker compose up -d` startup
  7. Verification steps (health check, dashboard reachable, backup script available)

- [ ] **AC-2**: The runbook includes a troubleshooting section covering at least three
  common failure modes: (a) tunnel not connecting (Cloudflare dashboard check), (b)
  database not initializing (migrations log check), (c) app container crashing on startup
  (docker logs inspection command).

- [ ] **AC-3** (Operator-verified): The full stack is verified against a real server. The
  verification result is recorded in the runbook as a "Verified on:" line with the date,
  server description, and OS version. This is not optional -- the runbook is not DONE
  until it is verified.

- [ ] **AC-4**: `docker compose up -d` on the production server starts all
  services (app, traefik, cloudflared) within 60 seconds. All services are
  `healthy` or `running` after startup (verified with `docker compose ps`). All services
  have `restart: unless-stopped` set (verify with `docker compose config`).

- [ ] **AC-5** (Operator-verified): The dashboard is reachable at `https://baseball.<domain>`
  from an external browser (not localhost). The operator can log in via WARP or email OTP
  and see the seeded team stats page.

- [ ] **AC-6** (Operator-verified): The health check endpoint
  `https://api.baseball.<domain>/health` (or `https://baseball.<domain>/health`) returns
  HTTP 200 with `{"status": "ok", "db": "connected"}` from an external curl command with
  a valid CF Access service token header.

- [ ] **AC-7**: The runbook documents the data directory backup situation: the host-mounted
  `./data/app.db` is backed up via `python scripts/backup_db.py` (E-009-05). The runbook
  notes what to do in a server migration: stop the stack, copy `./data/`, start the stack
  on the new server.

- [ ] **AC-8**: A `docker-compose.override.yml.example` file is committed. It uses
  Docker Compose profiles to move the `cloudflared` service behind a profile (e.g.,
  `production-only`), so that `docker compose up` without `--profile production-only`
  skips cloudflared. This lets developers without a CF tunnel token run the stack locally.
  The override is not automatically active -- developers copy it to
  `docker-compose.override.yml` if needed.

### Server Minimum Specs

Any Linux server with Docker installed. The user's deployment target is a home server.
- **Minimum**: 1+ CPU, 1+ GB RAM, 10+ GB disk (the app footprint is tiny)
- **OS**: Ubuntu 24.04 LTS (or equivalent Linux with Docker support)
- **Firewall**: All inbound closed except SSH (port 22). No ports 80 or 443 exposed.
  Cloudflare Tunnel handles all application traffic via outbound connections.
- **Cost**: None -- runs on existing home infrastructure.

### docker-compose.override.yml.example

```yaml
# Copy to docker-compose.override.yml for local dev without Cloudflare Tunnel.
# This removes the cloudflared service so the stack runs without a tunnel token.
services:
  cloudflared:
    profiles:
      - production-only  # Only starts when --profile production-only is passed
```

## Notes

- **Two-phase execution**: This story has an agent phase and an operator phase.
  - **Agent phase**: Write the runbook (AC-1, AC-2, AC-7, AC-8) and commit the
    `docker-compose.override.yml.example`. These are agent-completable.
  - **Operator phase**: The operator verifies the runbook against a real server
    (AC-3, AC-4, AC-5, AC-6). The agent reports completion of the agent phase and
    the PM coordinates with the operator for verification. The story is not DONE
    until operator verification is recorded.
- The "verified against a real server" requirement (AC-3) is non-negotiable. It is the
  difference between a runbook that works and a runbook that is aspirational. Do not
  mark this story DONE without a real deployment test.
- SSH key setup for the server should be covered in the runbook if remote access is used.
  For home servers, the operator likely has direct or SSH access already configured.
- Include the exact Docker install commands for Ubuntu 24.04 LTS (the Docker apt repo
  commands change occasionally; include the current ones at time of writing).
- The `docker compose up -d` command is the production startup. Use `docker compose logs
  -f` to monitor startup. Use `docker compose ps` to verify health. Include these in
  the verification step.
- Add a note about automatic server restarts: configure Docker to start on system boot
  (`sudo systemctl enable docker`), and verify `restart: unless-stopped` on all services.
  **Important**: The E-009-02 implementation did NOT set `restart: unless-stopped` on the
  `app` or `traefik` services. E-009-04 should add it when modifying docker-compose.yml
  for the cloudflared service. If it is still missing when this story runs, add it.
- The production startup command is `docker compose up -d`. Backups are handled separately
  via `python scripts/backup_db.py` (E-009-05).
- Size: Medium. Mostly documentation, but verification requires actually running the
  deployment on a real server. Budget time for Docker setup and Cloudflare tunnel
  verification steps.
