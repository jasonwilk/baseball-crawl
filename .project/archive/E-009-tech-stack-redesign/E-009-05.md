# E-009-05: Database Seeding and Reset Workflow

## Epic
[E-009: Tech Stack Redesign -- Portable, Agent-Browsable, Cloudflare-Integrated](epic.md)

## Status
`DONE`

## Description
After this story is complete, any developer (or agent) can reset the development database
to a clean seeded state with a single command, and back up the production database with a
simple local copy script. The seed data is realistic enough to exercise all dashboard views
when E-004 development begins.

## Context
E-009-02 bootstrapped the migration runner and created a minimal `seed_dev.sql`. This
story extends that foundation in two directions:

1. **Development workflow**: Richer seed data, a clean reset command, and a dev-only
   database reset script. The goal is to make local dashboard development fast -- a
   developer can break their database and recover in seconds.

2. **Production backup**: A simple `scripts/backup_db.py` script copies the database to a
   timestamped file under `data/backups/`. No external services or credentials needed.

Key decisions from the decision record:
- SQLite WAL mode is already enabled by `apply_migrations.py` (from E-009-02)
- Host-mounted database at `./data/app.db`
- Backup via local file copy (Litestream + R2 removed 2026-03-02 -- see Notes)

## Technical Approach

### reset_dev_db.py

```python
# scripts/reset_dev_db.py
# Usage: python scripts/reset_dev_db.py [--db-path PATH]
# Default path: data/dev.db (or data/app.db depending on APP_ENV)
# 1. Remove existing database file if it exists
# 2. Run apply_migrations.py against fresh database
# 3. Execute data/seeds/seed_dev.sql
# 4. Print summary
```

### backup_db.py

```python
# scripts/backup_db.py
# Usage: python scripts/backup_db.py [--db-path PATH]
# Copies ./data/app.db to ./data/backups/app-<timestamp>.db
# Creates data/backups/ if it does not exist.
```

### Seed Data Requirements

The `seed_dev.sql` file should produce a database state that exercises dashboard views:
- `teams`: Lincoln Varsity (owned), Opponent HS (not owned)
- `players`: 5 Lincoln Varsity players + 5 Opponent players (realistic names)
- `games`: 5 games, mix of home/away, with scores
- `player_game_batting`: rows for all players in all games they appeared in
- `player_game_pitching`: rows for 1-2 pitchers per team per game
- `player_season_batting`: season aggregate rows for all Lincoln players

## Notes

- **Litestream removed (2026-03-02)**: The original implementation included Litestream
  continuous WAL replication to Cloudflare R2. This was removed per user direction because
  the project is hosted on home Linux servers, not a VPS. Off-site cloud backup to R2 added
  unnecessary complexity (R2 bucket, API credentials, Litestream sidecar container) for a
  database that is a single file on locally-owned hardware. Replaced with Option C: a simple
  `scripts/backup_db.py` that copies the database to a timestamped local file. The
  `litestream` service was removed from `docker-compose.yml`, `litestream/litestream.yml`
  was deleted, and R2 env vars were removed from `.env.example`.
- The `reset_dev_db.py` script operates on whatever `DATABASE_PATH` points to (default:
  `./data/app.db`). There is no separate `dev.db` convention -- E-009-02 established
  `./data/app.db` as the single database path for all environments. The script includes
  a safety check: if `APP_ENV=production`, require a `--force` flag to proceed.
- SQLite WAL mode: `apply_migrations.py` (E-009-02) already enables WAL mode.
- Size: Medium. The reset script and seed data are straightforward; the backup script is
  a simple file copy.
