## Research Question

Could Model Context Protocol (MCP) servers provide meaningful value as an integration
layer between Claude Code agents and external tools or services in this project? Or do
simpler alternatives (CLI tools, environment variables, direct Python calls) cover the
same ground with less complexity?

This is **awareness and fit-assessment research only**. The output is a factual report.
No architectural commitment is made until the PM explicitly authorizes a follow-on story
or epic based on these findings.

## Context

### What MCP Is

Model Context Protocol (MCP) is an open standard from Anthropic for connecting AI models
to external data sources and tools. An MCP server exposes a typed API over a local socket
(or HTTP) that a Claude Code session can call directly as a tool. From the agent's
perspective, an MCP tool looks identical to a built-in tool -- the agent simply calls it
by name without worrying about subprocess management, output parsing, or protocol details.

Claude Code ships with first-party MCP support. MCP servers can be registered in
`.claude/settings.json` under the `mcpServers` key, making them available to any agent in
the project automatically.

### Why This Is Worth Investigating

The project currently relies on:
- Direct bash calls for filesystem and git operations
- Python subprocesses for scripts
- Environment variables for credentials
- WebFetch/WebSearch for external data

This is simple and works. But as the agent ecosystem grows (currently 7 agents with more
anticipated), patterns that require every agent to know how to invoke a subprocess, parse
its output, and handle errors could become repetitive boilerplate. MCP servers, if well-
chosen, could provide a cleaner tool boundary -- at the cost of additional infrastructure
to configure and maintain.

The question is: **does that trade-off pay off at this project's scale?**

### Specific MCP Servers to Evaluate

The following servers have been flagged for evaluation. Investigate each one. Do NOT
assume any of them is the right choice -- the goal is to understand what each one does
and whether it solves a real problem the project currently has or will likely have.

1. **docker/mcp** (canonical MCP by Docker Inc.)
   - GitHub: https://github.com/docker/mcp-servers
   - What it claims: MCP servers for Docker-related tooling (container management,
     registry access, compose orchestration). Part of Docker's official MCP catalog.
   - Question for this project: Agents currently interact with Docker Compose via bash.
     Would a Docker MCP server provide meaningfully cleaner access to container state,
     logs, or compose operations? Or is `docker compose up/down/logs` via bash sufficient?

2. **ChromeDevTools/chrome-devtools-mcp**
   - GitHub: https://github.com/ChromeDevTools/devtools-mcp
   - What it claims: MCP server exposing Chrome DevTools Protocol (CDP) -- allows an AI
     agent to control a Chrome browser, inspect DOM, run JS, take screenshots, access
     network traffic, etc.
   - Question for this project: E-009-R-03 concluded that `WebFetch` is sufficient for
     agent browsability of the coaching dashboard. Is that still true? Would CDP-level
     access (DOM inspection, screenshot, JS execution) ever be needed for dashboard
     development or debugging? What would it cost to add vs. the value it provides?

3. **CodeGraphContext/CodeGraphContext**
   - GitHub: https://github.com/CodeGraphContext/CodeGraphContext
   - What it claims: MCP server that builds and queries a semantic graph of a codebase --
     allowing an agent to ask structural questions about the code (what calls what, where
     is this symbol defined, what are the dependencies of this module) without reading
     files one at a time.
   - Question for this project: The codebase is currently small (under 20 files). Would a
     code graph provide meaningful value now? At what codebase size or complexity would it
     start paying for itself? What is the setup cost?

4. **GlitterKill/sdl-mcp** (or equivalent SDL tooling MCP)
   - GitHub: https://github.com/GlitterKill/sdl-mcp
   - What it claims: SDL (Service Dependency Layer or similar) integration via MCP.
   - Note to researcher: This one is the least familiar. Investigate what this server
     actually does before evaluating fit. If the repository is sparse or the tool is
     unmaintained, document that finding and move on. Time-box investigation to 20 minutes.

### The Simpler Alternative to Benchmark Against

Before concluding any MCP server is worth adding, compare it against the simplest
alternative that achieves the same outcome:

- **CLI + bash**: Can the agent accomplish the same task by running a command via the
  Bash tool? If yes, what does MCP add?
- **Environment variables + Python**: Can the agent call a Python module directly? If
  yes, what does MCP add?
- **Existing Claude Code tools**: Does WebFetch, WebSearch, Glob, Grep, Read, Edit, or
  Bash already cover the use case? If yes, what does MCP add?

The null hypothesis is: **simple bash calls are sufficient, and MCP adds complexity
without proportional benefit at this project's scale.** The research should test this
hypothesis, not assume it is wrong.

### Constraints

- This is a small, focused project (4 teams, ~60 players, single operator).
- Agent infrastructure is already complex (7 agents, custom skills, memory system,
  epic/story workflow). New complexity must justify itself.
- The project has no CI/CD, no remote repository yet, no external services beyond
  GameChanger API and Cloudflare. MCP integration points are limited.
- MCP servers that require cloud accounts, API keys, or ongoing maintenance have a higher
  bar to clear than ones that run entirely locally.

## Notes

This spike was requested as awareness and fit-assessment only. The PM has no prior on

and no obligation to find value.

The existing agent infrastructure (skills, memory, rules, dispatch workflow) was built
without MCP and works. Any MCP adoption must clear the bar: "does this make the agent
system meaningfully better in a way that justifies the added configuration and dependency?"

useful research outcome.
