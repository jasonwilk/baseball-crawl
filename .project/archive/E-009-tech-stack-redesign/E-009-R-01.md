## Research Question
What is the right database technology for baseball-crawl, evaluated separately for each
deployment path? The two candidates are:

- **Option A (Native Cloudflare)**: Cloudflare D1 (SQLite syntax, edge-distributed,
  managed service). No local database process; D1 is emulated locally via `wrangler dev`.
- **Option B (Docker + Cloudflare Access)**: SQLite file in a Docker volume (embedded
  in the app process, no separate service) or PostgreSQL as a separate Docker service.

## Context

### Scale Reality
- ~60 player records total (4 teams x ~15 players)
- ~120 game records per season, ~360 across 3 seasons
- ~1,800 player-season stat rows across all history
- Total data comfortably under 10 MB
- Read-mostly workload; coaches browse, they do not write
- 1-5 concurrent users at peak

### Current State
- Existing code in E-003 assumes Cloudflare D1 (SQLite syntax)
- Local dev already uses `data/dev.db` (SQLite file)
- Migrations are written in SQLite-compatible SQL
- If Option B is chosen, D1-specific bindings in any existing code must be replaced with
  standard sqlite3 or SQLAlchemy calls -- but E-003 migrations (SQL files) transfer directly

### The Core Question Per Path
- **Option A**: How reliable and capable is Cloudflare D1 in 2026? Is the local emulation
  via `wrangler dev` faithful enough for tight development loops? Any known limitations?
- **Option B**: SQLite embedded in the app process vs. PostgreSQL as a separate service.
  At this scale (under 10 MB, read-mostly), is there any reason to choose Postgres over
  SQLite?

The note must answer:

1. **Cloudflare D1 current state (Option A)**: What are D1's current limits (storage,
   reads/writes per day, row counts)? Is the free tier adequate for this workload? How
   faithful is D1 local emulation via `wrangler dev` compared to production? Are there

2. **SQLite in Option B**: For Docker + VPS deployment, what is the production durability
   story for SQLite in a Docker volume? Is a backup strategy required (e.g., periodic
   file copy, Litestream)? For a coaching app where data can be re-crawled if lost, what
   is the minimum acceptable backup approach?

3. **SQLite WAL mode concurrency**: Does SQLite in WAL mode handle 1-5 simultaneous
   HTTP connections reading the database without blocking? Any known issues with Python's
   sqlite3 module and concurrent reads from multiple FastAPI worker threads?

4. **Postgres entry cost (Option B alternative)**: If SQLite presents a real problem in
   Option B, what does a Postgres service in Docker Compose look like? Additional cost,
   complexity, and migration effort compared to SQLite?

5. **Schema migration tooling (both options)**: For Option A, what is the D1 migration
   workflow (`wrangler d1 migrations apply`)? For Option B, is a migration runner like
   Alembic justified, or are numbered SQL files applied via a bootstrap script sufficient?
   We already have numbered `.sql` files in `migrations/`.

6. **Recommendation per path**: One clear database recommendation for Option A and one
   for Option B, with a paragraph each explaining why.

## Notes
The PM's prior for Option B is SQLite in a Docker volume. This research should confirm
or challenge that prior. The key risk to check: Python's sqlite3 module and FastAPI's
async event loop -- does reading SQLite in an async context block the event loop? If yes,
note the mitigation (run DB queries via `run_in_executor`, or use `aiosqlite`).

For Option A, the PM has less of a prior -- D1 is already in use and the existing E-003
migrations work. The question is whether D1's local dev story has improved enough in 2026
to make it a viable tight-loop development environment.
