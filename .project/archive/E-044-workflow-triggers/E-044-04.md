# E-044-04: Implement workflow skill

## Epic
[E-044: Workflow Trigger Phrases](epic.md)

## Status
`DONE`

## Description
After this story is complete, a skill file at `.claude/skills/implement/SKILL.md` defines the full workflow for dispatching an epic when the user says "implement E-NNN". The team lead reads the epic's Dispatch Team section to determine which agents to include, spawns PM, and PM coordinates the dispatch as usual. The skill also supports a "and review" modifier that chains into the review-epic workflow after implementation completes.

## Context
Today when the user says "start epic E-NNN" or "execute E-NNN", the team lead must remember to spawn PM and relay the request. The existing dispatch-pattern.md and workflow-discipline.md define the rules, but there is no consolidated skill file that the team lead can follow step-by-step. This skill codifies the trigger recognition and initial routing, while delegating all coordination to PM per existing patterns.

## Acceptance Criteria
- [ ] **AC-1**: The file `.claude/skills/implement/SKILL.md` exists and follows the ingest-endpoint structural pattern: Activation Triggers, Purpose, Prerequisites, Phases, Workflow Summary (ASCII diagram), Edge Cases, Anti-Patterns.
- [ ] **AC-2**: Activation Triggers section lists: "implement E-NNN", "implement epic E-NNN", "start epic E-NNN", "execute E-NNN", "dispatch E-NNN", and similar natural variations. Also includes the chaining modifier: "implement E-NNN and review", "implement E-NNN and codex review".
- [ ] **AC-3**: Prerequisites section requires: (a) the target epic exists in `/epics/`, (b) the epic status is `READY` or `ACTIVE`, (c) if status is `DRAFT`, refuse and explain the READY gate.
- [ ] **AC-4**: Phase 1 (team composition) instructs the team lead to read the epic's `## Dispatch Team` section. If present, note the listed agents. If absent or empty, note "no team composition specified -- PM will determine agents from story domains".
- [ ] **AC-5**: Phase 2 (dispatch) instructs the team lead to spawn PM as a teammate, providing: the epic ID, the epic directory path, the team composition from Phase 1 (if any), and whether the "and review" modifier was specified. The team lead then steps back and lets PM coordinate per `/.claude/rules/dispatch-pattern.md`.
- [ ] **AC-6**: Phase 3 (optional review chain) documents that if the "and review" modifier was specified, PM should chain into the review-epic workflow (`.claude/skills/review-epic/SKILL.md`) after all stories are DONE but before the closure sequence. The skill references the review-epic skill file by path.
- [ ] **AC-7**: The skill explicitly reinforces the Team Lead Dispatch Boundary: the team lead spawns PM and relays the request -- it does NOT create dispatch teams, spawn implementing agents, update statuses, or verify acceptance criteria.
- [ ] **AC-8**: Edge Cases section covers: (a) epic not found, (b) epic is DRAFT (refuse with explanation), (c) epic is COMPLETED or ABANDONED (refuse), (d) no TODO stories with satisfied dependencies (PM will report this), (e) "and review" modifier with no uncommitted changes at the end (review skill handles this gracefully).
- [ ] **AC-9**: Anti-Patterns section includes: (a) do not bypass PM -- the team lead is a relay, not a coordinator, (b) do not read story files to make routing decisions -- that is PM's job, (c) do not start implementing before PM has updated statuses.

## Technical Approach
1. Create `.claude/skills/implement/SKILL.md` following the ingest-endpoint structure at `/.claude/skills/ingest-endpoint/SKILL.md`.
2. The skill is deliberately thin -- most of the coordination logic already lives in dispatch-pattern.md and the PM agent definition. The skill's job is trigger recognition, team composition extraction, and correct PM handoff.
3. The "and review" chain is a flag passed to PM in the spawn prompt, not a separate workflow invocation by the team lead.
4. Reference the Dispatch Team section format from `/.project/templates/epic-template.md` (added in E-044-01).
