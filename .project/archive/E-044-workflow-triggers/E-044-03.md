# E-044-03: Post-dev review workflow skill

## Epic
[E-044: Workflow Trigger Phrases](epic.md)

## Status
`DONE`

## Description
After this story is complete, a skill file at `.claude/skills/review-epic/SKILL.md` defines the full workflow for running a codex code review after an epic's implementation is complete, then having the implementing team review the findings with PM. The team lead loads this skill when the user says "review epic" and follows its procedure.

## Context
After all stories in an epic are DONE but before the closure sequence, the user may want a codex code review of the changes. Today this requires manually running the script and explaining who to spawn for the review. This skill formalizes it as a trigger-phrase workflow. The trigger phrase "review epic" is chosen to be distinct from "spec review" (pre-implementation) and from the raw "codex-review.sh" command.

## Acceptance Criteria
- [ ] **AC-1**: The file `.claude/skills/review-epic/SKILL.md` exists and follows the ingest-endpoint structural pattern: Activation Triggers, Purpose, Prerequisites, Phases, Workflow Summary (ASCII diagram), Edge Cases, Anti-Patterns.
- [ ] **AC-2**: Activation Triggers section lists: "review epic", "review epic E-NNN", "codex review epic E-NNN", "post-dev review E-NNN", and similar natural variations.
- [ ] **AC-3**: Prerequisites section requires: (a) the target epic exists, (b) all stories are DONE or the user explicitly requests a mid-implementation review, (c) if no epic ID is specified, ask the user which epic to review.
- [ ] **AC-4**: Phase 1 instructs the team lead to determine the appropriate codex-review mode. Default is `uncommitted` for changes not yet committed. If the user specifies a mode (e.g., "review epic E-042 against main"), use `base main`. The team lead runs `./scripts/codex-review.sh <mode>` via Bash and captures the output.
- [ ] **AC-5**: Phase 2 instructs the team lead to spawn PM as a teammate, providing the codex output, the epic ID, and the epic directory path. PM reads the epic's Dispatch Team section (if present) or infers the implementing agents from the epic's stories. PM spawns those agents as teammates. The team reviews codex findings together: fix, dismiss, or defer each finding. Fixes are implemented by the appropriate agent. PM documents deferred findings in the epic's Technical Notes or History.
- [ ] **AC-6**: The skill explicitly states that implementing agents apply code fixes while PM tracks which findings were addressed, dismissed, or deferred.
- [ ] **AC-7**: Edge Cases section covers: (a) codex not installed, (b) codex returns no findings ("No findings" is a valid result -- report it and end), (c) no uncommitted changes (script exits cleanly), (d) epic is already archived (read from archive path instead).
- [ ] **AC-8**: Anti-Patterns section includes: (a) do not auto-apply codex suggestions without team review, (b) do not run review before implementation is meaningfully complete (a few stories DONE is fine for incremental review, but the primary use case is post-epic), (c) do not conflate "review epic" with "spec review" -- they serve different purposes.

## Technical Approach
1. Create `.claude/skills/review-epic/SKILL.md` following the ingest-endpoint structure at `/.claude/skills/ingest-endpoint/SKILL.md`.
2. Phase 1 is the team lead running the codex-review script directly.
3. Phase 2 is a team-based review: team lead spawns PM, PM reads the epic to determine which agents to spawn, PM coordinates the review.
4. The skill references the Dispatch Team epic section (from E-044-01) as a way to determine which agents to include, with fallback to the dispatch-pattern.md routing table.
