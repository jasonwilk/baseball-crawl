# E-091-03: Scope Duplicate public_id Check to our_team_id

## Epic
[E-091: Fix E-088 Opponent Data Model Defects](epic.md)

## Status
`DONE`

## Description
After this story is complete, the duplicate `public_id` check will be scoped to the current team (`our_team_id`), eliminating false duplicate warnings when different teams share the same opponent. Currently, the queries lack an `our_team_id` filter, so cross-team reuse of the same opponent `public_id` triggers spurious warnings.

## Context
Codex code review finding #4 (P5). Warning-only (no data corruption), but confusing when multiple Lincoln teams (e.g., Varsity and JV) both link to the same opponent. DE confirmed the fix is two queries in `db.py` plus threading `our_team_id` through the call chain.

## Acceptance Criteria
- [ ] **AC-1**: Given two different teams that both link to the same opponent `public_id`, when either team's opponent link is being connected (confirm page or save), then no duplicate warning is produced
- [ ] **AC-2**: Given the same team linking to an opponent `public_id` that is already linked for that team, when the link is being connected (confirm page or save), then the duplicate warning fires as expected
- [ ] **AC-3**: Tests cover at least the save POST path; both call sites (confirm GET and save POST) use the same scoped function. Tests cover the cross-team same-`public_id` scenario (no false positive) and the same-team duplicate scenario (warning fires)

## Technical Approach
The duplicate `public_id` queries are in `src/api/db.py`. Add `AND our_team_id = ?` to each query and thread the `our_team_id` parameter through the call chain from `src/api/routes/admin.py`. See `src/api/db.py` for the current duplicate check queries and `src/api/routes/admin.py` for the call sites. Note: `is_duplicate_opponent_public_id` (db.py:976) has zero callers anywhere in the codebase -- it is dead code. The implementing agent should either delete it or fix it alongside `get_duplicate_opponent_name`; either approach is acceptable.
