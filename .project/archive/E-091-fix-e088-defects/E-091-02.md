# E-091-02: Store Hidden Opponents Instead of Skipping Them

## Epic
[E-091: Fix E-088 Opponent Data Model Defects](epic.md)

## Status
`DONE`

## Description
After this story is complete, the opponent resolver will store hidden opponents in the `opponent_links` table with `is_hidden=1` instead of silently skipping them. Currently, opponents with `is_hidden=True` in the API response trigger an early return that bypasses the database upsert entirely, even though the `is_hidden` column already exists in migration 006. This affects 13 of 70 observed opponents.

## Context
Codex code review finding #2 (P2). Also identified by the code-reviewer during E-088 dispatch and recorded in epic History as a SHOULD FIX item. The `is_hidden` column was added to the schema in E-088 but the resolver logic prevents it from ever being set to 1. Hidden opponents are confirmed as duplicates or inactive entries in GameChanger but should still be recorded for completeness.

## Acceptance Criteria
- [ ] **AC-1**: Given an opponent with `is_hidden=True` in the API response, when the resolver processes opponents, then the opponent is stored in `opponent_links` with `is_hidden=1`
- [ ] **AC-2**: Given an opponent with `is_hidden=False` (or not set), when the resolver processes opponents, then the opponent is stored with `is_hidden=0` (existing behavior preserved)
- [ ] **AC-3**: Tests cover hidden opponent upsert with correct `is_hidden` value
- [ ] **AC-4**: Previously stored non-hidden opponents are not affected by the change
- [ ] **AC-5**: `ResolveResult.skipped_hidden` is renamed to `stored_hidden` (or equivalent), increments when a hidden opponent is stored, and the summary log line at the end of `resolve()` reflects the new name

## Technical Approach
The early return on `is_hidden=True` is in `src/gamechanger/crawlers/opponent_resolver.py`. Remove the early return and ensure the existing upsert logic passes the `is_hidden` value through to the database write. The `is_hidden` column already exists in the schema -- no migration needed. See `opponent_resolver.py` for the current early-return logic and the upsert function. Note: after removing the early return, `ResolveResult.skipped_hidden` will never increment and becomes misleading. Rename it to `stored_hidden` (or similar) and increment it when a hidden opponent is stored via either the resolved or unlinked path. Update the log line at the end of `resolve()` accordingly.
