# E-091-01: Guard Connect Endpoint Against Overwriting Resolved Links

## Epic
[E-091: Fix E-088 Opponent Data Model Defects](epic.md)

## Status
`DONE`

## Description
After this story is complete, the `connect_opponent` admin endpoint will reject attempts to manually link an opponent that has already been auto-resolved. Currently, a direct POST can overwrite an auto-resolved link's `public_id` and `resolved_team_id`, silently destroying the auto-resolution with no audit trail. The endpoint will return HTTP 400 when the target link already has a resolution.

## Context
Codex code review finding #1 (originally P1, downgraded to P2 by SE). The endpoint is admin-only and the UI correctly hides the connect button for resolved opponents, but the server-side guard is missing. Single operator reduces practical risk, but defense-in-depth requires the server to enforce the constraint independently of the UI.

## Acceptance Criteria
- [ ] **AC-1**: Given an opponent link that is already resolved (has a non-NULL `public_id`, regardless of resolution method), when a POST is made to the connect endpoint for that link, then the server returns HTTP 400 with a descriptive error message and the existing link is unchanged
- [ ] **AC-2**: Given an opponent link that is unresolved (NULL `public_id`), when a POST is made to the connect endpoint, then the link is updated successfully (existing behavior preserved)
- [ ] **AC-3**: Tests cover both cases -- auto-resolved link rejected with 400, unresolved link accepted

## Technical Approach
The `connect_opponent` route in `src/api/routes/admin.py` currently fetches the link row and proceeds unconditionally. Add a check after fetching the link -- if the link already has a resolution, return HTTP 400 before calling the save function. The specific field to check (e.g., `public_id`, `resolution_method`) is left to the implementing agent. See `src/api/routes/admin.py` for the current `connect_opponent` function and `src/api/db.py` for `save_manual_opponent_link`.
