# E-089-01: Add Test Scope Discovery Rule to testing.md and SE Agent Definition

## Epic
[E-089: Test Scope Discovery Rule](./epic.md)

## Status
`DONE`

## Description
After this story is complete, `testing.md` will contain a mandatory test scope discovery rule that requires implementers to find and run all test files importing from any source module they modified -- not just tests named in the story's "Files to Create or Modify." The SE and DE agent definitions will reference this rule so implementers encounter it during story execution. The `testing.md` frontmatter will be expanded to include `src/**` so the rule auto-loads when agents edit source files.

## Context
During E-085, modifying `check_single_profile()` in `src/gamechanger/credentials.py` broke `tests/test_check_credentials.py`, but the implementer only ran `tests/test_credentials.py` and `tests/test_cli_creds.py` (story-scoped). The broken test was in a different file that also imports from the same module. No rule required the implementer to look beyond story-scoped tests.

## Acceptance Criteria
- [ ] **AC-1**: `/.claude/rules/testing.md` contains a new section describing the test scope discovery requirement. The section must: (a) state that when an implementer modifies a function in an existing source module, they must discover all test files that import from that module; (b) include a concrete example showing the E-085 scenario (modifying `src/gamechanger/credentials.py` and discovering `tests/test_check_credentials.py`); (c) specify that discovered test files must be run in addition to story-scoped tests before reporting completion.
- [ ] **AC-2**: `/.claude/agents/software-engineer.md` and `/.claude/agents/data-engineer.md` each reference the test scope discovery rule. The reference can be a sentence in a testing-related subsection or a new bullet in a code standards section -- placement is at the implementer's discretion, but it must be present in both files and point to the rule in `testing.md`.
- [ ] **AC-3**: The rule does NOT mandate a full `pytest` run. It describes targeted discovery of test files that import from modified modules.
- [ ] **AC-4**: No existing rule content in `testing.md` is modified or removed. Frontmatter path expansion is permitted.
- [ ] **AC-5**: `/.claude/rules/testing.md` frontmatter paths are expanded to include `src/**`, so the test scope discovery rule auto-loads when agents edit source files (not just test files).

## Technical Approach
The problem is that no existing rule requires test discovery beyond story scope. The solution is a new section in `/.claude/rules/testing.md` describing the mandatory discovery step, a frontmatter expansion to `src/**` so the rule auto-loads for source file edits, plus cross-references in both `/.claude/agents/software-engineer.md` and `/.claude/agents/data-engineer.md`. The implementing agent (claude-architect) should read all three files in full before making changes, and should design the rule text to be actionable by an implementer following it step-by-step.

Key constraint: the rule must include a concrete, reproducible discovery technique -- not just "make sure you run all relevant tests." The E-085 scenario (modifying `credentials.py`, discovering `test_check_credentials.py`) is a good concrete example to embed.
