# E-228-14: Generator orchestration + renderer + docs

## Epic
[E-228: Matchup Strategy Report (v1)](./epic.md)

## Status
`DONE`

## Description
After this story is complete, the matchup section is wired into the standalone scouting report end-to-end. `generate_report()` invokes `build_matchup_inputs()`, `compute_matchup()`, and (non-fatally) `enrich_matchup()` after reconciliation runs. The Jinja2 template renders a single new "Game Plan" section between the existing exec summary and the existing scouting content, with 6 sub-sections (top-3 hitters with full-roster pull-tendency notes inline, opposing eligible pitchers, SB profile, first-inning scoring tendency, 3-bucket loss recipe, eligible LSB pitchers). Inline citations within prose are LLM-authored (preserved verbatim by the renderer); the single deterministic citation exception is pull-tendency notes (renderer formats from raw `PullTendencyNote` fields). When `our_team_id` is None or `confidence == "suppress"`, the section does not render (no placeholder). Coach-voice and operator-voice user-facing documentation ships in this story.

## Context
This is the integration story. Earlier stories produce dataclasses and DB queries; this story makes the section visible to coaches and operators. Light visual treatment per v1: each sub-section uses the existing report typography. Heavier visual polish (AVOID inline sub-bullet typography, byte-identical fixture machinery, per-section break-inside:avoid, mobile responsive at 375px) is deferred to E-231.

The non-fatal try/except around `enrich_matchup()` mirrors the existing predicted-starter Tier 2 pattern at `generator.py:1186-1209`. When the LLM call fails or `OPENROUTER_API_KEY` is unset, the section renders with deterministic engine output only -- no error surfaces to the operator.

## Acceptance Criteria

### Behavior

- [ ] **AC-1 (generator orchestration)**: `generate_report()` in `src/reports/generator.py` invokes the matchup pipeline when `is_matchup_enabled()` AND `our_team_id` is not None:
  - **Step**: After reconciliation completes, call `build_matchup_inputs(conn, opponent_team_id, our_team_id, season_id, reference_date=...)` to get `MatchupInputs`.
  - **Step**: Call `compute_matchup(inputs)` to get `MatchupAnalysis`.
  - **Step**: When `analysis.confidence != "suppress"`, attempt `enrich_matchup(analysis, inputs)` inside try/except. On `LLMError` or any exception: log a warning and pass the un-enriched `MatchupAnalysis` to the renderer. On success: pass the `EnrichedMatchup`. The exception handling MUST be non-fatal -- the report continues to render.
  - **Step**: When `confidence == "suppress"` OR `our_team_id` is None, no matchup section is included in the rendered output.
- [ ] **AC-2 (placement -- pinned section order)**: When the matchup section renders (i.e., `our_team_id` is set AND `confidence != "suppress"`), the rendered report sections appear in this exact order: `Header → Exec summary → Game Plan → Predicted Starter (when enabled) → Recent form chips → Key players → Pitching → Batting → Spray → Roster → Footer`. Game Plan goes BEFORE Predicted Starter, both immediately after the exec summary; Predicted Starter shifts down by one slot when matchup renders. When matchup does NOT render (`our_team_id` is None or `confidence == "suppress"`), Predicted Starter renders in its current position (immediately after the exec summary) and no other section reorders. This is the single authoritative section-order statement for v1.
- [ ] **AC-3 (template renders all sub-sections; LLM parentheticals preserved verbatim)**: The Jinja2 template renders the Game Plan section with the section opener `game_plan_intro` immediately after the section header `GAME PLAN` and before sub-section 1's content. When `game_plan_intro` is absent (LLM unavailable or LLM-failure fallback per AC-6), the section opener is omitted entirely -- the renderer does NOT synthesize a deterministic substitute. After the opener, the section has 6 sub-sections, rendered in this fixed order:
  1. Top-3 dangerous hitters (with per-hitter cues and full-roster pull-tendency notes as a single hitter-focused sub-section):
     - Top-3 rows -- per hitter: name + jersey + PA badge + LLM cue. The cue arrives from `enrich_matchup` with the inline parenthetical citation already embedded by the LLM (per E-228-13 AC-3); the renderer outputs the cue string verbatim and does NOT append, reformat, or substitute citations from the engine's `supporting_stats`. **LLM-failure fallback**: when only a bare `MatchupAnalysis` is available (LLM unavailable or `enrich_matchup` failed -- see AC-6), the renderer skips per-hitter cue prose entirely and renders just `name + jersey + PA badge + raw stats` (e.g., PA, OBP, SLG drawn directly from `ThreatHitter.supporting_stats` formatted as a deterministic display row). No prose, no synthesized cue. This mirrors the predicted-starter LLM-prose-block degrade-by-hiding pattern.
     - Pull-tendency notes -- list of full-roster pull-heavy hitters (not limited to top-3) rendered inline within this sub-section when non-empty; omitted entirely when empty. Renderer formats citations from raw fields per AC-4; see exception note there.
  2. Opposing eligible pitchers -- list with rest math + recent workload (light treatment, no per-pitcher prose).
  3. Stolen-base profile -- short prose summary from `sb_profile_summary` + LLM `sb_profile_prose`. The renderer preserves any LLM-authored parentheticals verbatim and does not add its own.
  4. First-inning scoring tendency -- short prose summary from `first_inning_summary` + LLM `first_inning_prose`. Same pass-through rule for any LLM-authored parentheticals.
  5. 3-bucket loss recipe -- per-bucket counts + LLM `loss_recipe_prose` summarizing what the patterns reveal. LLM-authored parentheticals are preserved verbatim.
  6. Eligible LSB pitchers -- same shape as sub-section 2 (only renders when `our_team_id` is set; light treatment).
  General rule: the LLM is the source of truth for citations in prose narrative output; the renderer preserves LLM parentheticals verbatim and does NOT supply prose citations from deterministic stats. The single documented exception is pull-tendency notes (AC-4).

  **`data_notes` rendering (Option A -- per-sub-section, italic gray):** The renderer iterates over `MatchupAnalysis.data_notes` (per E-228-12 AC-1) and, for each entry, renders an italic gray "Note" line at the bottom of the sub-section identified by the entry's `subsection` field. Format: `Note: <data_notes[i].message>`. Mapping is one-to-one with the canonical sub-section identifiers: `"top_hitters"` -> sub-section 1, `"opposing_pitchers"` -> sub-section 2, `"sb_profile"` -> sub-section 3, `"first_inning"` -> sub-section 4, `"loss_recipe"` -> sub-section 5, `"lsb_pitchers"` -> sub-section 6. Multiple notes for the same sub-section render as separate italic gray lines in input order. When a sub-section has zero `data_notes` entries, no note line renders for that sub-section. For sub-section 1, the in-band PA badges and "early read only" softening on top-3 hitter rows remain as currently specified (per AC-3 sub-section 1); per-hitter `data_notes` (e.g., the low-PA softening note from E-228-12 AC-6, or any other hitter-specific note tagged `subsection="top_hitters"`) ALSO render as the italic gray note line at the bottom of sub-section 1. `data_notes` are deterministic engine output and render regardless of LLM availability (they are NOT part of the LLM-prose fallback surface in AC-6). Reviewer meaningfulness: a reviewer can identify, for any thin-sample warning produced by the engine per E-228-12 AC-6 or AC-16, the exact sub-section where the corresponding "Note: ..." line will appear in the rendered output.
- [ ] **AC-4 (inline citations -- LLM cites prose; renderer cites deterministic engine output)**: Each prescriptive claim about a hitter or a pitcher carries an inline parenthetical with the supporting data point. The split is:
  - **Prose claims (LLM-authored)**: For top-3 hitter cues and any prose in `sb_profile_prose`, `first_inning_prose`, `loss_recipe_prose`, the LLM embeds the inline parentheticals as part of its output (per E-228-13 AC-3, drawing from grounding tuples on `MatchupAnalysis` and stats on `MatchupInputs`). The renderer preserves these parentheticals verbatim and does NOT supply or reformat citations from the engine's `supporting_stats` -- the engine's `supporting_stats` field exists to ground the LLM's prompt, not to drive renderer-formatted parentheticals.
  - **Deterministic engine output (renderer-authored, the single documented exception)**: Pull-tendency notes are produced by the engine deterministically (not LLM prose), so the renderer formats their parentheticals from the raw `PullTendencyNote.pull_pct` and `PullTendencyNote.bip_count` fields directly, inline as `(NN% pull on MM BIP)` or equivalent (e.g., "Watch the pull from Smith #14 (62% pull on 24 BIP)."). No `supporting_stats` field is added to `PullTendencyNote`.
  Reviewer meaningfulness: every prose claim about a player has a parenthetical citation; for LLM-authored prose, the citation is preserved verbatim from the LLM output; for pull-tendency notes (the single exception), citations reference raw `PullTendencyNote` fields formatted by the renderer.
- [ ] **AC-5 (suppress hides without placeholder)**: When the engine returns `confidence == "suppress"`, the rendered HTML contains NO trace of the Game Plan section -- not even a hidden div, not a placeholder message. The existing scouting report renders as if `our_team_id` were never provided. Reviewer meaningfulness: assertion verifies the rendered HTML does not contain the `Game Plan` section header string.
- [ ] **AC-6 (LLM unavailable falls back gracefully -- degrade by hiding LLM-prose)**: When `OPENROUTER_API_KEY` is unset OR the LLM call fails, the renderer receives a bare `MatchupAnalysis` (not an `EnrichedMatchup`) and the section still renders with deterministic engine output. The LLM-prose fields (`game_plan_intro`, per-hitter cues, `sb_profile_prose`, `first_inning_prose`, `loss_recipe_prose`) are OMITTED ENTIRELY -- the renderer does NOT synthesize deterministic substitutes for any LLM-prose surface. Specifically:
  - Section opener (`game_plan_intro`): omitted; the section header is followed directly by sub-section 1.
  - Top-3 hitter rows (sub-section 1): no per-hitter cue prose; renders just `name + jersey + PA badge + raw stats` from `ThreatHitter.supporting_stats` (deterministic display per AC-3 sub-section 1 fallback).
  - Stolen-base profile (sub-section 3): renders only the deterministic `sb_profile_summary` (counts/rates); no `sb_profile_prose`.
  - First-inning pattern (sub-section 4): renders only the deterministic `first_inning_summary` (rates); no `first_inning_prose`.
  - Loss recipe (sub-section 5): renders only the deterministic per-bucket counts; no `loss_recipe_prose`.
  - Eligible-pitcher lists (sub-section 2 = opposing pitchers, sub-section 6 = LSB pitchers when `our_team_id` is set): unchanged (these are deterministic from the start; not LLM-driven).
  - Pull-tendency notes (rendered inline within sub-section 1, NOT a standalone sub-section): unchanged -- these are deterministic engine output (per E-228-12 AC-7 + E-228-14 AC-4) and render with renderer-formatted citations regardless of LLM availability. They are NOT part of the LLM fallback surface.
  - `data_notes` italic gray "Note" lines (rendered at the bottom of each corresponding sub-section per AC-3): unchanged -- these are deterministic engine output (per E-228-12 AC-1, AC-6, and AC-16) and render regardless of LLM availability. They are NOT part of the LLM fallback surface.
  This is the degrade-by-hiding pattern: LLM-prose surfaces hide on failure; deterministic surfaces render unchanged. The renderer's docstring documents which fields are LLM-prose (and hide on failure) versus deterministic (and always render).

## Technical Approach
The orchestration block in `generate_report()` follows the same shape as the existing predicted-starter block (lines 1186-1209). The matchup runs after reconciliation completes; reconciliation may correct pitcher attribution boundaries that the matchup engine relies on. Try/except around `enrich_matchup()` is the standard non-fatal fallback pattern.

The Jinja2 template adds a single new section. Use the existing report's typography classes for the sub-sections (light treatment per v1; heavier visual hierarchy is E-231). Each sub-section is a `<div class="game-plan-subsection">` with consistent styling.

For inline citations, the spec splits authorship by content type (per AC-3 and AC-4):
- **LLM-authored prose** (top-3 hitter cues, `sb_profile_prose`, `first_inning_prose`, `loss_recipe_prose`): the LLM embeds inline parentheticals in its output (per E-228-13 AC-3), drawing from grounding tuples on `MatchupAnalysis` and stats on `MatchupInputs`. The renderer preserves these parentheticals verbatim and does NOT format prose citations from the engine's `supporting_stats`. `ThreatHitter.supporting_stats` exists to ground the LLM's prompt, not to drive the renderer.
- **Renderer-formatted citations (the single exception)**: Pull-tendency notes are deterministic engine output, not LLM prose. The renderer formats the parenthetical from `PullTendencyNote.pull_pct` and `PullTendencyNote.bip_count` directly (e.g., `(62% pull on 24 BIP)`) rather than expecting a pre-formatted string or a `supporting_stats` list.

For documentation: software-engineer (per Agent Hint) authors both pages as part of this story per the consolidated v1 spine -- coach doc focuses on "what does this section tell me?", operator doc focuses on "how do I make this section appear?" If the implementing main session prefers a closure-time handoff to docs-writer after the code lands, that is acceptable as long as the coach doc and operator doc both ship before the epic closes.
