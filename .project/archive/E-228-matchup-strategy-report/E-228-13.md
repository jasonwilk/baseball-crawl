# E-228-13: LLM Tier 2 wrapper -- `enrich_matchup()` with hallucination guardrail

## Epic
[E-228: Matchup Strategy Report (v1)](./epic.md)

## Status
`DONE`

## Description
After this story is complete, the matchup section can be enriched with a Tier 2 LLM narrative that wraps the deterministic engine output. `enrich_matchup(analysis, inputs)` calls OpenRouter with a structured prompt that includes both the `MatchupAnalysis` dataclass (output structure to guide narrative shape) and the `MatchupInputs` dataclass (raw grounding tables the LLM cannot deviate from). The wrapper enforces the player-name round-trip hallucination guardrail: the LLM may NEVER name a player not present in the inputs. v1 ships a multi-field JSON payload (per AC-5): `game_plan_intro` (section opener), `hitter_cues` (one cue per top-3 hitter), `sb_profile_prose`, `first_inning_prose`, and `loss_recipe_prose`. Each prose field carries inline citations embedded by the LLM. Pull-tendency notes are NOT covered by the LLM -- they are deterministic engine output formatted by the renderer (per E-228-12 AC-7 + E-228-14 AC-4); the LLM has no `pull_tendency_prose` responsibility.

## Context
Mirrors the Tier 2 pattern from E-212 predicted-starter (`src/reports/llm_analysis.py`). The wrapper's role is narrative; the engine remains authoritative for structure and confidence. When `confidence == "suppress"`, the wrapper is not invoked (the renderer simply hides the section per epic Technical Notes).

The wrapper signature takes `(MatchupAnalysis, MatchupInputs)` together: inputs needed for the prompt's grounding tables; analysis needed for output structure. This was a v1 spec contradiction resolution (Codex spec review found the original story had `enrich_matchup(MatchupAnalysis)` but the prompt rendered `MatchupInputs` tables -- the wrapper now takes both explicitly).

## Acceptance Criteria

### Behavior

- [ ] **AC-1 (signature)**: `src/reports/llm_matchup.py::enrich_matchup(analysis: MatchupAnalysis, inputs: MatchupInputs, *, client=None) -> EnrichedMatchup` is defined. The optional `client` parameter accepts an injected OpenRouter client for test mocking; default is the module-level OpenRouter client from `src/llm/openrouter.py`. The function returns an `EnrichedMatchup` dataclass that wraps the original `MatchupAnalysis` with added narrative fields.
- [ ] **AC-2 (suppress short-circuit)**: When `analysis.confidence == "suppress"`, `enrich_matchup` does NOT call OpenRouter and returns the original analysis wrapped with empty narrative fields. Verified by a test that mocks `query_openrouter` and asserts it was never called.
- [ ] **AC-3 (prompt construction -- LLM is the citation author for prose)**: The system prompt instructs the model: "high school baseball coaching analyst producing matchup-strategy intelligence. Lead with the recommendation, support with evidence in parenthetical. Direct, decisive, conclusion-first. For every prescriptive claim about a hitter or a pitcher, embed an inline parenthetical citing the supporting data point. Do NOT invent statistics, names, or game results not present in the structured data above." Mirrors `_SYSTEM_PROMPT_TEMPLATE` in `llm_analysis.py:40-68`. The LLM is the source of truth for citations in prose narrative output: it draws values from the grounding tuples and structured stats supplied in `MatchupAnalysis` and `MatchupInputs` (per E-228-12) and embeds them inline as parentheticals. The renderer (E-228-14) preserves these LLM-authored parentheticals verbatim and does NOT supply citations from deterministic stats for prose content. Pull-tendency notes are the documented exception (deterministic engine output, not LLM prose) -- the renderer formats their citations directly from raw `PullTendencyNote` fields per E-228-14 AC-4.
- [ ] **AC-4 (user prompt grounding)**: The user prompt renders `MatchupInputs` as ASCII tables: top-3 hitters with all signals (PA, OBP, SLG, BB%, K%, FPS swing rate, chase rate, swing rate by count), pull-tendency rows, stolen-base profile, first-inning pattern, 3-bucket loss recipe, eligible opposing pitchers, eligible LSB pitchers (when present). The user prompt also embeds the engine's deterministic output (cue_kind per hitter, bucket counts, sb_profile_summary, first_inning_summary) so the narrative cannot contradict the deterministic core.
- [ ] **AC-5 (output schema)**: The user prompt asks the model for strict JSON output with named fields: `game_plan_intro` (1-2 sentences setting the table), `hitter_cues` (list of {`player_id`, `cue`} -- one cue per top-3 hitter, in input order), `sb_profile_prose` (string), `first_inning_prose` (string), `loss_recipe_prose` (2-3 sentences summarizing what the bucket pattern reveals). The schema does NOT include `pull_tendency_prose` -- pull-tendency notes are deterministic engine output (per E-228-12 AC-7) formatted by the renderer directly from raw `PullTendencyNote.pull_pct` and `PullTendencyNote.bip_count` fields (per E-228-14 AC-4); the LLM is not responsible for any pull-tendency narrative. Wrapper validates every required field is present and is a string (or list-of-dicts for `hitter_cues`). Failure raises `LLMError`.
- [ ] **AC-6 (hallucination guardrail -- name round-trip)**: After parsing the LLM JSON, the wrapper validates that every `hitter_cues[i].player_id` matches one of `inputs.opponent_top_hitters[*].player_id`. ANY name or player_id not in the inputs triggers the documented policy: when `FEATURE_MATCHUP_STRICT=1` is set, raise `LLMError`; when unset, filter the offending entry while preserving the rest of the response. The policy choice mirrors the predicted-starter precedent.
- [ ] **AC-7 (OpenRouter call parameters)**: The wrapper uses the existing `query_openrouter` client. Parameters: `model = os.environ.get("OPENROUTER_MODEL", "anthropic/claude-haiku-4-5-20251001")`, `max_tokens=1500`, `temperature=0.3`, single-pass JSON response. Failure modes (timeout, network error, malformed JSON) raise `LLMError` -- the generator (E-228-14) catches and falls back to deterministic-only rendering.
- [ ] **AC-8 (no-op when feature flag off)**: When `FEATURE_MATCHUP_ANALYSIS` is unset/falsy, the wrapper is never reached (the generator gates on `is_matchup_enabled()` before invoking). The wrapper itself does NOT re-check the feature flag; it assumes the caller has gated.
- [ ] **AC-9 (output dataclass)**: `EnrichedMatchup` extends `MatchupAnalysis` (composition, not inheritance -- carries the original analysis as a field, plus the new narrative fields). This mirrors `EnrichedPrediction` from `llm_analysis.py`.

## Technical Approach
Mirror `src/reports/llm_analysis.py` exactly. Build helpers `_format_hitter_table(inputs)`, `_format_pitching_table(inputs)`, etc. that produce ASCII-table strings. The user prompt is the concatenation of these tables plus the engine's analysis output. The system prompt is a module-level constant string. JSON parse + field validation use a small helper to keep error messages consistent.

The hallucination guardrail's player-name round-trip operates on `player_id` (deterministic identifier), not on the raw name string -- this avoids fuzzy-matching headaches. The engine's `MatchupAnalysis.threat_list` carries `player_id` for each top-3 hitter; the LLM is instructed to output `hitter_cues[i].player_id` matching one of those values; the wrapper validates after parsing.

OpenRouter call uses the existing `src/llm/openrouter.py` client via `query_openrouter()`. No new HTTP code. Cost is ~$0.01/call at Haiku pricing for ~1500 tokens output.
