# E-001-03: Write GameChanger API spec document

## Epic
[E-001: GameChanger API Foundation](epic.md)

## Status
`DONE`

## Description
After this story is complete, the project will have a living API specification document at `/.project/research/gamechanger-api-spec.md` that documents every GameChanger endpoint discovered so far, their request shapes, response shapes, authentication requirements, and any known quirks. This document is the shared contract between the ingestion code (E-002) and the API, replacing guesswork with facts.

## Context
This is a collaborative, human-in-the-loop story. The user will provide curl commands captured from GameChanger's web or mobile app. The implementing agent's job is to: (1) analyze those curl commands to infer base URLs, auth schemes, and endpoint patterns; (2) make test requests where possible to capture real response shapes; (3) write them up in a structured spec document.

The spec document is not written once and forgotten -- it will be updated whenever new endpoints are discovered or existing ones change. Think of it as the project's API reference manual.

This story is not blocked by E-001-01 or E-001-02 -- it can run in parallel. However, if E-001-02 is already done, use the client to make discovery requests rather than raw curl.

## Acceptance Criteria
- [x] **AC-1**: `/.project/research/gamechanger-api-spec.md` exists and contains a section for each of the following endpoint categories (or a documented note if an endpoint was not found): team roster, game schedule, game stats (box score level), player stats (season aggregates), opponent/away team data.
- [x] **AC-2**: Each documented endpoint entry includes: full URL pattern (with placeholder variables like `{team_id}`), HTTP method, required headers/auth, an example request, an example response (real or realistic mock), and any known pagination scheme.
- [x] **AC-3**: The document includes a "Credential Scheme" section that describes exactly what headers/cookies are required, where they come from, and how to detect expiry.
- [x] **AC-4**: The document includes a "Known Limits and Quirks" section with at least: observed rate limit behavior (or "unknown, assumed 500ms safe"), any fields that appear in some responses but not others, and any endpoints that require specific team/season scoping.
- [x] **AC-5**: The document includes a "Discovery Log" section -- a reverse-chronological list of when each endpoint was first documented and by what method (curl capture, live request, inference).
- [x] **AC-6**: All team IDs, player IDs, and auth tokens in the document are replaced with placeholder values (e.g., `TEAM_ID_REDACTED`, `TOKEN_REDACTED`). No real credentials or PII appear in the document.

## Technical Approach
This is primarily a research and documentation task. Steps:

1. Ask the user to provide any curl commands they have captured from GameChanger. Analyze each for URL patterns, headers, and query parameters.
2. For each distinct base path discovered, document the endpoint shape.
3. If E-001-02 is available, use `GameChangerClient` to make a few exploratory GET requests to map the response shape.
4. When response shape is ambiguous, document what was observed and note fields that may be optional.
5. Use a consistent markdown structure for each endpoint (see template below).

**Endpoint entry template:**
```markdown
### GET /v2/teams/{team_id}/stats

**Description**: Returns season aggregate stats for a team.

**Auth**: Bearer token required (`Authorization: Bearer {token}`)

**Path parameters**:
- `team_id` (string): GameChanger team identifier

**Query parameters**:
- `season` (string, optional): e.g., `2025`

**Example request**:
\`\`\`
GET https://api.vg.scorekeepersports.com/v2/teams/TEAM_ID_REDACTED/stats?season=2025
Authorization: Bearer TOKEN_REDACTED
\`\`\`

**Example response** (condensed):
\`\`\`json
{
  "team_id": "TEAM_ID_REDACTED",
  "season": "2025",
  "stats": { ... }
}
\`\`\`

**Pagination**: None observed.
**Notes**: The `stats` object shape varies by season.
```
