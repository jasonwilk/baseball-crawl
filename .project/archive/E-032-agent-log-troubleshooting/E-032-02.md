# E-032-02: Verify Agent Error Diagnosis from Logs

## Epic
[E-032: Agent Log Access and Troubleshooting Verification](../E-032-agent-log-troubleshooting/epic.md)

## Status
`DONE`

## Description
After this story is complete, we will have confirmed that an agent can trigger a real application error, find the corresponding traceback or error message in the Docker Compose logs, identify the root cause, and propose a fix -- proving the full troubleshooting loop works end-to-end.

## Context
E-032-01 verifies the basics: starting the stack, hitting health, reading logs. This story goes further by testing the *diagnostic* workflow: when something goes wrong, can an agent find the problem by reading logs? This is the real value of the troubleshooting setup -- not just reading logs, but using them to understand and fix errors.

## Acceptance Criteria
- [x] **AC-1**: The agent triggers a real application error that produces a visible error or traceback in the container logs. The agent documents which error scenario was used and why.
- [x] **AC-2**: The agent reads the error from `docker compose logs app` (or `docker compose logs --tail=N app`) and identifies the root cause from the log output alone -- without reading the application source code first.
- [x] **AC-3**: The agent proposes a specific fix for the error based on the log output. The fix must be correct (i.e., it would resolve the error if applied).
- [x] **AC-4**: The agent reverts any temporary changes made to trigger the error (e.g., restoring a renamed file, restarting a stopped service). The app is left in a working state.
- [x] **AC-5**: The agent produces a written summary documenting: (a) the error scenario used, (b) the relevant log output, (c) the root cause identified, (d) the proposed fix, and (e) any gaps or friction encountered in the workflow. If the agent identifies context layer adjustments or CLAUDE.md improvements, these are documented as specific recommendations.

## Technical Approach
1. With the stack running (confirmed by E-032-01), trigger an error. Suggested approaches (choose one or more):
   - **Database unavailable**: Rename `data/app.db` to `data/app.db.bak`, then hit `GET /health` or `GET /dashboard`. The health check should return 503, and the logs should show a database connection error.
   - **Missing template or route error**: Hit `GET /dashboard` which queries a table that may not have data, and observe what the logs show.
   - **Any other approach** that produces a genuine traceback or error log entry.
2. Read `docker compose logs --tail=30 app` to find the error.
3. From the log output alone, identify what went wrong and propose a fix.
4. Revert the error trigger (e.g., `mv data/app.db.bak data/app.db`).
5. Verify the app recovers (re-run health check).
6. Document everything in the summary.

### Important Constraint
For AC-2, the agent must attempt to diagnose the error **from the logs first**, before looking at source code. This simulates the real troubleshooting workflow: logs are the first signal, source code is the second. The agent may look at source code after forming an initial diagnosis from logs, to confirm or refine the fix.

## Notes

### Execution Summary (2026-03-03)

#### (a) Error Scenario Used

**Database file removal** -- the `data/app.db` file (and its WAL companions `app.db-shm` and `app.db-wal`) were temporarily renamed to `app.db.bak`, `app.db-shm.bak`, and `app.db-wal.bak`. This caused SQLite to create a new empty database file when the app next connected, producing `OperationalError: no such table` for every query.

**Why this scenario was chosen:** Renaming the DB file is the most realistic real-world failure -- it mimics a missing or misconfigured volume mount, a corrupted DB file, or a deployment where migrations have not run yet. It produces a real Python traceback in the logs (not just an HTTP error code in the response body), and it exercises both the health endpoint and the dashboard endpoint simultaneously.

#### (b) Relevant Log Output


#### (c) Root Cause Identified (from logs alone, before reading source)

The database file at `/app/data/app.db` was missing or empty. SQLite automatically created a new blank file when the app opened the connection, but that file had no schema. The `_migrations` table (used by the health check to confirm schema initialization) and `player_season_batting` table (queried by the dashboard) did not exist in the empty file.

Two specific symptoms:
1. `check_connection()` at `db.py:114` failed because `_migrations` table was absent -- health endpoint returned 503.
2. `get_team_batting_stats()` at `db.py:91` failed because `player_season_batting` table was absent -- dashboard returned HTTP 200 with "No stats available" fallback (error was swallowed gracefully).

The log output was sufficient to identify both the precise location in code (`db.py:114`, `db.py:91`), the exception type (`sqlite3.OperationalError`), and the missing objects (`_migrations`, `player_season_batting`). No source code reading was needed for the diagnosis.

#### (d) Proposed Fix

If this occurred in production or during a real incident:

1. **Immediate**: Restore `app.db` from backup (if one exists) -- `mv app.db.bak app.db`. Verify health returns `{"status":"ok","db":"connected"}`.
2. **If no backup**: Restart the container (`docker compose restart app`). The container entrypoint runs `python migrations/apply_migrations.py` on startup, which will create and populate the schema on the new empty DB file. Data from before the failure will be lost, but the app will be functional.
3. **If the volume mount is misconfigured**: Check that `./data` on the host is correctly mapped to `/app/data` in the container (`docker compose.yml` volume section). A misconfigured mount would cause a fresh empty file on every container start.

The proposed fix was confirmed correct -- restoring the three renamed files immediately resolved the error (`{"status":"ok","db":"connected"}` on the next health check, no restart required).

#### (e) Gaps and Friction Encountered

**No blocking gaps.** The full troubleshooting loop worked end-to-end:
- `docker compose ps` confirmed stack status immediately.
- `curl http://localhost:8001/health` showed the error response.
- `docker compose logs --tail=50 app` surfaced a full Python traceback within seconds.
- The traceback was sufficiently detailed (file path, line number, exception type, table name) to diagnose the root cause without reading source code.
- Revert and recovery were straightforward.

**Minor observations (not blockers):**

1. **Dashboard swallows errors silently (HTTP 200 on DB failure)**: The dashboard returned HTTP 200 even though the DB query failed. The template renders "No stats available" as a fallback. This is a defensible UX choice, but from a monitoring perspective it means that dashboard errors are only visible in logs -- not in HTTP response codes. An operator relying solely on HTTP status codes would miss dashboard failures. Consider logging a structured `ERROR` event with a request correlation ID so dashboard failures are linkable to HTTP requests. Currently the only log link is timing proximity.

2. **WAL files needed renaming too**: The DB has WAL mode enabled (`app.db-shm` and `app.db-wal` companion files). I had to rename all three files, not just `app.db`, to fully remove the database. If only `app.db` is renamed, SQLite may still find and use the WAL files. This is an edge case for the error scenario setup but not relevant to real incidents (a missing DB in production would typically mean the entire volume is missing, not just the main file).

3. **No log-level filtering command in CLAUDE.md**: All log output is mixed together (uvicorn access logs, app-level ERROR/WARNING, INFO). For long-running apps with many requests, finding a specific traceback in `--tail=50` output may require `--tail=N` with a larger N or grep filtering. Recommendation: Add a grep example to the CLAUDE.md troubleshooting section:
   ```bash
   docker compose logs app 2>/dev/null | grep -A 10 "ERROR\|Traceback"
   ```
   This surfaces only error blocks without manual scanning.

**Recommended CLAUDE.md addition** (for claude-architect consideration):

In the "App Troubleshooting" section, add a "Finding errors in logs" subsection:
```bash
# Find ERROR lines and the tracebacks that follow them
docker compose logs app 2>/dev/null | grep -A 10 "ERROR\|Traceback\|OperationalError"

# Find errors from a specific time window
docker compose logs --since=5m app 2>/dev/null | grep ERROR
```

This would make the log-diagnosis step faster without requiring the agent to know grep syntax from memory.
