# E-145-01: Fix env-var fallback to cover all credential keys

## Epic
[E-145: Fix Docker Credential Loading for Optional Env Vars](epic.md)

## Status
`DONE`

## Description
After this story is complete, `GameChangerClient._load_credentials()` will fall back to `os.environ` for all credential keys -- not just those in `_required_keys()`. This ensures optional keys like `GAMECHANGER_USER_EMAIL` and `GAMECHANGER_USER_PASSWORD` are available inside Docker containers where env vars are injected by Docker Compose and no `.env` file exists.

## Context
The login fallback path in `trigger.py` relies on email and password being present in the credentials dict. These are optional keys not in `_required_keys()`, so the current env-var fallback loop skips them. This is the sole bug blocking E-143's UI Sync feature in production Docker.

## Acceptance Criteria
- [ ] **AC-1**: Given a Docker environment where `dotenv_values()` returns an empty dict (no `.env` file) and `GAMECHANGER_USER_EMAIL` + `GAMECHANGER_USER_PASSWORD` are set as process env vars, when `_load_credentials("web")` is called, then both keys appear in the returned dict with their env-var values.
- [ ] **AC-2**: Given a `.env` file that contains all keys (including optional ones), when `_load_credentials()` is called, then `.env` values take precedence over `os.environ` values for the same key (existing behavior preserved).
- [ ] **AC-3**: Given a Docker environment, when `_load_credentials("web")` is called and `GAMECHANGER_ACCESS_TOKEN_WEB` is set as a process env var, then it appears in the returned dict (verifies the fix covers profile-scoped optional keys beyond email/password).
- [ ] **AC-4**: All existing tests in `tests/test_client.py` and `tests/test_token_manager.py` pass. If AC-6 error message changes cause existing assertions to fail, update those specific assertions to match the new wording (verified: current tests use `match="GAMECHANGER_USER_EMAIL"` and `"Auto-recovery" in str(...)` patterns that should survive the wording change, but the implementing agent should confirm).
- [ ] **AC-5**: New tests verify that optional credential keys are populated from `os.environ` when `dotenv_values()` returns an empty dict. Tests must cover representative keys from each category: login credentials (`GAMECHANGER_USER_EMAIL`), profile-scoped optional keys (`GAMECHANGER_ACCESS_TOKEN_WEB`), and proxy keys (`PROXY_ENABLED`).
- [ ] **AC-6**: Error messages in `src/gamechanger/token_manager.py` that reference ".env" as the sole credential source (lines 316 and 650) are updated to say ".env or environment" (or equivalent) so they are accurate in both devcontainer and Docker contexts.

## Technical Approach
The bug is in `_load_credentials()` at `src/gamechanger/client.py:628-636`. The env-var fallback loop currently iterates only over `_required_keys(profile)`. It needs to cover all keys that downstream consumers read from the credentials dict. The implementing agent determines the exact code pattern; the constraint is that `.env` values must still take precedence over `os.environ` when both are present. See epic Technical Notes for the full list of affected consumers and keys.
