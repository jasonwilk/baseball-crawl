# E-145-03: Document Docker credential auto-recovery in admin docs

## Epic
[E-145: Fix Docker Credential Loading for Optional Env Vars](epic.md)

## Status
`DONE`

## Description
After this story is complete, the admin operations guide and production deployment guide will explain that keeping `GAMECHANGER_USER_EMAIL` and `GAMECHANGER_USER_PASSWORD` in the Docker environment enables automatic login fallback when refresh tokens expire -- reducing the need for manual credential intervention.

## Context
Docs consultation found that neither `docs/admin/operations.md` nor `docs/production-deployment.md` mentions the auto-recovery benefit of retaining email/password env vars. Operations.md describes a fully manual recovery path; production-deployment.md frames these vars as one-time bootstrap tools. After E-145-01 fixes the code to actually load these vars in Docker, the docs should reflect the operational benefit.

## Acceptance Criteria
- [ ] **AC-1**: `docs/admin/operations.md` Credential Rotation section includes a note that `GAMECHANGER_USER_EMAIL` and `GAMECHANGER_USER_PASSWORD` in the Docker environment enable automatic login fallback when refresh tokens expire.
- [ ] **AC-2**: `docs/production-deployment.md` Step 2.4 (or equivalent bootstrap section) includes a note recommending that operators retain email/password vars in production `.env` for ongoing auth resilience, not just one-time bootstrap.
- [ ] **AC-3**: `docs/admin/credential-refresh.md` references to ".env" as the sole credential source (lines 85, 95, 109, 227) are updated to acknowledge that credentials can also come from environment variables (Docker context). These are targeted wording updates, not structural changes.
- [ ] **AC-4**: `docs/api/auth.md` line 210 login fallback description updated from "in `.env`" to include environment variable context.
- [ ] **AC-5**: `docs/admin/credential-refresh.md` line 229 and `docs/api/auth.md` line 210 both contain the stale claim "`force_refresh()` does not attempt login." This is factually incorrect -- `force_refresh(allow_login_fallback=True)` does attempt login (added for E-143's `trigger.py:99`). Both statements must be corrected to reflect that `force_refresh()` supports an optional login fallback when called with `allow_login_fallback=True`.
- [ ] **AC-6**: All updates are additive or targeted wording corrections (no structural rewrites, no removal of existing content).
