# E-006-05: Pre-Commit Hook Tests

## Description
After this story is complete, a pytest test suite at `tests/test_pii_scanner.py` covers all scanner behaviors: blocking on each PII pattern, passing clean files, respecting the `synthetic-test-data` annotation, and handling edge cases (binary extensions, encoding errors, empty files, skip paths). The test suite runs in any environment without network access or a configured Git repository, using `tmp_path` for all file fixtures.

## Context
The PII scanner (E-006-04) is a security control. Security controls require tests that prove they work -- "I ran it manually and it seemed fine" is not sufficient. This test suite is the proof.

All test data is synthetic. Fake emails (`test@example.com`), fake phone numbers (`(555) 867-5309`), fake names (`Jane Fakename`) are used throughout. Test source files that contain these patterns include the `synthetic-test-data` marker in their own source (as a comment) so that if the scanner is ever run against the test suite itself, the tests are not blocked.

Tests exercise the scanner's Python functions directly (import and call), not via `git commit` subprocess calls. This keeps tests fast, hermetic, and independent of Git state.

## Acceptance Criteria
- [ ] **AC-1**: `pytest tests/test_pii_scanner.py` exits 0 with all tests green; no network calls are made
- [ ] **AC-2**: A test verifies that a file containing `test@example.com` triggers a block with the correct output format (`[PII BLOCKED] {file}:{line}: matched 'email' pattern`)
- [ ] **AC-3**: A test verifies that a file containing `(555) 867-5309` triggers a block with the correct output format (`[PII BLOCKED] {file}:{line}: matched 'us_phone' pattern`)
- [ ] **AC-4**: A test verifies that a file containing only clean Python code (no PII patterns) produces exit code 0
- [ ] **AC-5**: A test verifies that a file with `synthetic-test-data` in its first 5 lines passes through even when it contains email and phone patterns
- [ ] **AC-6**: A test verifies that a file with an extension not in `SCANNABLE_EXTENSIONS` (e.g., `.png`) is skipped without any read attempt and exits 0
- [ ] **AC-7**: A test verifies that an empty file (zero bytes) does not cause an exception and exits 0
- [ ] **AC-8**: A test verifies that a file containing both an email pattern and a phone pattern on different lines reports BOTH violations before exiting 1 (not just the first match)
- [ ] **AC-9**: A test verifies that when two files are scanned -- one clean, one with PII -- the clean file is not mentioned in the output and the dirty file is blocked
- [ ] **AC-10**: A test verifies that a file with invalid UTF-8 bytes does not raise an exception; the scanner reads it with `errors='replace'` and either scans it or skips it gracefully
- [ ] **AC-11**: A test verifies that a file path matching a prefix in `SKIP_PATHS` (e.g., a path under `.claude/`) is skipped without being read
- [ ] **AC-12**: No real PII appears anywhere in the test file or in any fixture -- all names, emails, and phone numbers are obviously fake

## Technical Approach
Use `tmp_path` (pytest built-in fixture) to create all test files on the fly. Do NOT create committed fixture files in `tests/fixtures/`. The design doc is explicit: `tmp_path` is simpler and eliminates fixture drift risk.

```python
# tests/test_pii_scanner.py
# synthetic-test-data -- this file contains fake PII patterns for testing only

import pytest
from pathlib import Path
from src.safety import pii_scanner  # import the scanner module

def test_email_detected(tmp_path):
    f = tmp_path / "response.json"
    f.write_text('{"coach_email": "test@example.com"}')
    result = pii_scanner.scan_files([f])
    assert result.has_violations
    assert any("email" in v.pattern for v in result.violations)
    assert any(v.line_number == 1 for v in result.violations)

def test_synthetic_annotation_skips_file(tmp_path):
    f = tmp_path / "fixture.json"
    f.write_text("# synthetic-test-data\n{\"email\": \"test@example.com\"}")
    result = pii_scanner.scan_files([f])
    assert not result.has_violations
```

The exact scanner API (function signatures, return types) is established by E-006-04. If E-006-04 implements a different interface, adjust the tests accordingly. The behavior described in the ACs is what matters; the exact call syntax is secondary.

Every test method that constructs a string containing a fake email or phone number should include a comment `# synthetic-test-data` at the top of the test function or in the test file header.

Do not test the git hook mechanism (no `subprocess.run(['git', 'commit', ...])`). Test the scanner logic only.
