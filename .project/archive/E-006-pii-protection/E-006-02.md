## Description
After this story is complete, a written PII taxonomy document exists at `docs/pii-taxonomy.md`. It defines exactly what constitutes PII in the baseball-crawl context, how each PII type appears in GameChanger API responses, what regex or detection patterns identify it, and what the permitted handling rules are. The document is the authoritative reference for the PII scanner (E-006-04) and the developer guide (E-006-06).

## Context
"PII" is not self-defining. A phone number is obviously PII; a batting average is obviously not. But in between there are judgment calls: Is a GameChanger user ID PII if it resolves to a real person via the API? Is a team name PII if it reveals the school? This document resolves those judgment calls explicitly so that the scanner and the developer guide are consistent with each other, and so any agent or developer can make a correct ruling on an unfamiliar data field without asking.

The taxonomy also defines the synthetic data convention: how to mark test fixtures so the scanner knows a fake name or fake phone number is intentionally in a committed file and should not trigger an alert.

This story has no blocking dependencies and can run in parallel with E-006-01.

## Acceptance Criteria
- [ ] **AC-1**: Given `docs/pii-taxonomy.md`, when it is read, then it defines at minimum these categories: full names, email addresses, phone numbers, mailing addresses, GameChanger user IDs, and photos/avatars
- [ ] **AC-2**: For each PII category, the document specifies: (a) why it is PII in this context, (b) where it appears in GameChanger API responses (JSON field names if known, or "field names TBD" if API spec is incomplete), (c) a detection approach (regex, field name match, or heuristic), and (d) permitted handling (ephemeral only, redact before committing, never store, etc.)
- [ ] **AC-3**: The document defines a "non-PII" category list: data types that are explicitly NOT PII and are safe to commit (e.g., batting statistics, game scores, team names, season identifiers, synthetic test data)
- [ ] **AC-4**: The document documents the canonical synthetic data convention, exactly as specified: a committed file is exempted from PII scanning if the string `synthetic-test-data` appears anywhere in its first 5 lines. The document explains when and how to use this annotation, and warns against misuse (annotating real data as synthetic).
- [ ] **AC-5**: The document contains a "Quick Reference" section: a one-page lookup table mapping data types to their PII status and permitted handling, formatted so an agent can make a ruling in under 30 seconds

## Technical Approach
This story is primarily a domain analysis and documentation task, not a coding task. The implementing agent should:

1. Read the existing API spec at `docs/gamechanger-api.md` to enumerate known data fields
2. Use the domain knowledge in CLAUDE.md and the epic context to reason about each field's sensitivity
3. Draft `docs/pii-taxonomy.md` following the structure in the Acceptance Criteria
4. For any GameChanger-specific fields where PII status is unclear, err on the side of classifying as PII and note the uncertainty in the document

The document does not need to be perfect on first pass -- it should be "good enough to build the scanner against." It can be updated as the API spec grows.

## Notes
- If `docs/gamechanger-api.md` does not yet enumerate enough fields to fully populate the PII taxonomy, the implementing agent should note which sections are "pending API spec completion" and provide the framework so those gaps can be filled when E-001-03 (API spec story) delivers more detail.
- The synthetic data convention is already decided: the string `synthetic-test-data` in the first 5 lines of a file exempts it from scanning. This is established in the architect's design doc and in the epic technical notes. Document it exactly as specified -- do not invent an alternative convention.
- Do not invent GameChanger field names speculatively. Use what is in the API spec, and mark unknowns explicitly.
- The question of whether to detect full names via regex is left open for this story to resolve. Simple regex is unreliable for names (too many false positives on common words, place names, etc.). The recommendation is to classify names as PII in the taxonomy but note that automated detection is not attempted in the initial scanner -- names are covered by the `ephemeral/` directory convention instead. Document this tradeoff explicitly.
