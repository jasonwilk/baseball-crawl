# E-006-04: PII Scanner Implementation

## Description
After this story is complete, the pre-commit hook and Claude Code hook both call a real PII scanner at `src/safety/pii_scanner.py` that examines staged files and blocks the commit if any file contains patterns identified in the PII taxonomy (E-006-02). The scanner produces clear, actionable output: which file triggered, which line, and which pattern matched. It respects the `synthetic-test-data` marker defined in E-006-02, allowing properly annotated files to pass through without triggering a false positive.

## Context
This story implements the scanning logic -- the intelligence that decides whether a staged file contains PII. It is invoked by the pre-commit hook scaffold (E-006-03) via two modes: `--stdin` (file paths piped from the Git hook) and `--staged` (scanner discovers staged files itself, used by the Claude Code hook). Both modes must be implemented.

The scanner is stdlib-only. No pip dependencies. Patterns are stored in a Python module (`src/safety/pii_patterns.py`), not a YAML file. This keeps the scanner installable on any Python 3 environment without a prior `pip install`.

False positives are acceptable at low frequency; false negatives are not. The scanner errs toward blocking. The legitimate bypass for developers is `git commit --no-verify`. The legitimate bypass for committed test fixtures is the `synthetic-test-data` marker.

Read `/.project/research/E-006-precommit-design.md` in full before implementing. It contains the exact interface contract, pattern format, and scanner design requirements.

## Acceptance Criteria
- [ ] **AC-1**: Given a staged file containing a string matching the email regex pattern (e.g., `coach@example.com`), when `git commit` is run, then the commit is blocked and stderr names the file, the line number, and the pattern (`matched 'email' pattern`)
- [ ] **AC-2**: Given a staged file containing a string matching the US phone number regex pattern (e.g., `(555) 867-5309`), when `git commit` is run, then the commit is blocked and stderr names the file, the line number, and the pattern (`matched 'us_phone' pattern`)
- [ ] **AC-3**: Given a staged Python source file that contains no PII patterns, when `git commit` is run, then the commit proceeds without interference and the scanner exits 0
- [ ] **AC-4**: Given a staged file whose first 5 lines contain the string `synthetic-test-data`, when `git commit` is run, then the commit proceeds without interference even if the file contains email or phone patterns -- the file is skipped entirely
- [ ] **AC-5**: Given a staged file with a binary extension (e.g., `.png`, `.pyc`, `.db`), the scanner skips the file without attempting to read it and exits 0
- [ ] **AC-6**: Given a staged file containing BOTH an email pattern and a phone pattern on different lines, when `git commit` is run, then BOTH violations are reported before the commit is blocked (not just the first match)
- [ ] **AC-7**: Given any scanner invocation, the exit code is 0 (clean or skipped) or 1 (PII detected), with no unhandled exceptions that produce an ambiguous exit code -- encoding errors are handled with `errors='replace'`
- [ ] **AC-8**: The scanner completes in under 500ms for a changeset of 20 typical files

## Technical Approach
Read `/.project/research/E-006-precommit-design.md` -- the Implementation Contract section for E-006-04 is the spec. Key points:

### File Structure
```
src/safety/__init__.py          # empty package marker
src/safety/pii_scanner.py       # core scanner logic
src/safety/pii_patterns.py      # pattern definitions (Python module, NOT YAML)
```

### `src/safety/pii_patterns.py`
A plain Python module containing constants: `PATTERNS` (list of dicts with `name`, `regex`, `description`), `SYNTHETIC_MARKER` (`"synthetic-test-data"`), `SCANNABLE_EXTENSIONS` (set of extensions to scan), and `SKIP_PATHS` (set of path prefixes to always skip). Initial patterns: email and US phone. See the design doc for exact regex strings.

**Do NOT create `pii_patterns.yaml`.** The design doc considered and rejected YAML in favor of this Python module to maintain stdlib-only status.

### `src/safety/pii_scanner.py`
CLI interface:
- `--staged`: runs `git diff --cached --name-only --diff-filter=ACM` internally to get staged files
- `--stdin`: reads file paths from stdin, one per line
- Positional args: scan specific files directly

Per-file logic:
1. Check if the file's extension is in `SCANNABLE_EXTENSIONS`. If not, skip silently.
2. Check if any segment of the file path starts with a prefix in `SKIP_PATHS`. If so, skip silently.
3. Read the file with `open(..., errors='replace')`.
4. Check first 5 lines for `SYNTHETIC_MARKER`. If found, skip the file.
5. Apply each pattern in `PATTERNS` to every line. Collect all matches.
6. After scanning all files: if any violations, print all to stderr and exit 1. Otherwise exit 0.

Output format on violation (to stderr):
```
[PII BLOCKED] path/to/file.json:42: matched 'email' pattern
[PII BLOCKED] path/to/file.json:87: matched 'us_phone' pattern

2 violation(s) found in 1 file(s).
```

### Style Requirements (from CLAUDE.md)
- Type hints on all function signatures
- Docstrings on public functions
- Use `logging` module for operational messages (debug, info); use `sys.stderr.write` or `print(..., file=sys.stderr)` for blocking messages that must appear regardless of log level
- Use `pathlib.Path` for file path operations
- No `print` statements for non-blocking output

## Notes
- The `src/safety/` module is new. Do not add it to any existing package structure without checking for conflicts with E-001 through E-005 work.
- The scanner must never import from `src/gamechanger/` or any other project module -- it must remain self-contained so it runs during the pre-commit phase before the full project environment is necessarily available.
- Synthetic data convention: the marker is `synthetic-test-data` (exact string, no special prefix required). It must appear anywhere in the first 5 lines. This is case-sensitive. The E-006-02 taxonomy document is the canonical reference.
- Keep the scanner fast. Reading files line-by-line (not loading entire file into memory) and applying compiled regex patterns is the correct approach. Pre-compile all patterns at module load time using `re.compile()`.
- The design doc notes that full name detection via regex is unreliable. Do not attempt name detection unless E-006-02 taxonomy explicitly specifies a viable pattern. Email and US phone are the initial pattern set.
