## Description
After this story is complete, a developer guide exists at `docs/safe-data-handling.md` that any agent or developer can read to understand exactly how to work safely with sensitive GameChanger data. It covers where to put API response files, what can and cannot be committed, how to install the pre-commit hook, what to do when the hook fires, and how to create test fixtures safely. It references the PII taxonomy and the ephemeral directory convention without duplicating them.

## Context
The technical controls (hook, scanner, gitignore) are necessary but not sufficient. A developer who does not understand why the controls exist, or what to do when something goes wrong, will work around them or be confused when they fire. This guide ensures the controls are reinforced by understanding.

This guide is also the onboarding document for any new agent joining the project that will touch real data. It is written for an audience that knows Python and Git but may not be familiar with this project's specific data handling conventions.

The guide is written AFTER the scanner (E-006-04) exists, so it can reference real behavior rather than speculative behavior. It can be written in parallel with E-006-05 (tests), but must wait for E-006-04.

## Technical Approach
Write `docs/safe-data-handling.md` as a practical, scannable guide. Avoid walls of text. Use numbered lists for procedures, tables for quick reference, and code blocks for commands.

Suggested structure:
```
# Safe Data Handling Guide

## The One Rule
## Where Sensitive Data Lives (ephemeral directory)
## What Can Be Committed
## Installing the Pre-Commit Hook
## When the Hook Fires
  ### Understanding the Output
  ### Fixing a Violation
  ### Legitimate Bypass (if applicable)
## Working with Test Fixtures
## Quick Reference
## Related Documents
```

The "The One Rule" section should state the principle in a single sentence, e.g.: "If a file was produced by a real API call, it goes in `/ephemeral/`, never in a commit."

## Notes
- Write in second person ("you") addressed to an agent or developer who just cloned the repo
- The bypass procedure is decided: `git commit --no-verify` is the only legitimate bypass. It skips ALL Git hooks, not just the PII scanner. The guide must include a prominent warning about this. Two legitimate reasons exist for using it: (1) a confirmed false positive that cannot be quickly resolved, (2) an emergency hotfix. Using `--no-verify` "because the hook is slow" or "because you're sure the file is clean" is explicitly not legitimate. The guide should state this.
- Include the actual error message format that the scanner produces: `[PII BLOCKED] path/to/file:42: matched 'email' pattern` -- copied from E-006-04's output spec. Developers need to recognize this message when it fires.
- The Claude Code hook cannot be bypassed by an agent. Only a human can disable it by modifying `.claude/settings.json`. Include a brief note about this for any agents reading the guide.
- This document will evolve as the API spec grows and new PII categories are discovered. Include a revision note at the bottom with the date and any known gaps.
