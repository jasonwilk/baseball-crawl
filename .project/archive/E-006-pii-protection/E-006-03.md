## Description
After this story is complete, both hook layers of the two-layer PII defense are installed and configured: a Git pre-commit hook at `.githooks/pre-commit` and a Claude Code PreToolUse hook at `.claude/hooks/pii-check.sh`. Both are wired to call `src/safety/pii_scanner.py`, but if the scanner does not yet exist they pass silently. A one-command install script at `scripts/install-hooks.sh` configures the Git hook for any developer or fresh clone. The Claude Code hook requires no installation -- it fires automatically from `.claude/settings.json`.

## Context
The architect's design document at `/.project/research/E-006-precommit-design.md` specifies all implementation details. Read that document in full before starting. The key decisions:

- **Git hook mechanism**: `.githooks/` directory with `git config core.hooksPath`, NOT raw `.git/hooks/` (not version-controlled) and NOT the `pre-commit` Python framework (overkill, adds dependencies). The hook script is committed so it is version-controlled and visible in code review.
- **Claude Code hook**: A `PreToolUse` hook on the `Bash` matcher, configured in `.claude/settings.json`. It intercepts only `git commit` commands -- all other Bash calls pass through with a single `grep` check overhead.
- **Scanner not yet installed**: E-006-04 (scanner implementation) follows this story. At scaffold time, both hooks should detect the scanner's absence and exit 0 silently (or with a brief warning). This allows the scaffold to be committed and verified before the scanner exists.
- **Scoped rule**: A `.claude/rules/pii-safety.md` file protects the safety code from accidental weakening.

This story owns files in `.githooks/`, `.claude/hooks/`, `.claude/rules/`, `scripts/`, and `.claude/settings.json`. It does NOT touch `src/safety/` (that is E-006-04) or `docs/` (that is E-006-06).

## Technical Approach
Read `/.project/research/E-006-precommit-design.md` in full. It contains exact pseudocode and file contents for all deliverables. The summary:

### `.githooks/pre-commit`
Shell script. Gets staged files from `git diff --cached --name-only --diff-filter=ACM`. Passes them to the scanner via stdin (`echo "$STAGED_FILES" | python3 "$SCANNER" --stdin`). If the scanner does not exist, prints `[pii-hook] Scanner not installed yet. Skipping PII check.` and exits 0. Must be executable (`chmod +x`).

### `.claude/hooks/pii-check.sh`
Shell script. Reads JSON from stdin (`INPUT=$(cat)`). Extracts `tool_input.command` via `jq`. If the command does not match `^git\s+commit`, exits 0 immediately. If it does match, runs `python3 "$CLAUDE_PROJECT_DIR/src/safety/pii_scanner.py" --staged`. If the scanner exits non-zero, outputs a JSON denial object (`permissionDecision: "deny"`) and exits 0 (denial is signaled via JSON, not exit code). If the scanner does not exist, exits 0. Must be executable.

### `scripts/install-hooks.sh`
Shell script. Runs `git config core.hooksPath "$REPO_ROOT/.githooks"` and `chmod +x "$REPO_ROOT/.githooks/pre-commit"`. Prints confirmation. One-command execution.

### `.claude/settings.json`
Add the `hooks` key with a `PreToolUse` array. The command value is `"$CLAUDE_PROJECT_DIR"/.claude/hooks/pii-check.sh` with a timeout of 10 seconds and a statusMessage of `"Checking staged files for PII..."`. Merge with existing keys -- do not overwrite `agent`, `env`, or `statusLine`.

### `.claude/rules/pii-safety.md`
YAML frontmatter scoping to the three path patterns, followed by advisory rules: never weaken regex patterns without approval, never add blanket exclusions, always run tests before committing scanner changes, scanner must stay under 1 second for 20 files.

### Verification steps (manual, part of Definition of Done)
1. Run `./scripts/install-hooks.sh`
2. Run `git commit --allow-empty -m "test: verify hook fires"` -- must see the "Scanner not installed" warning and succeed
3. Confirm `.claude/settings.json` has the hooks block
4. In a Claude Code session, confirm the PreToolUse hook appears (check `/hooks` if available)

## Notes
- The design document is the spec. If anything in this story conflicts with the design doc, the design doc wins. Flag any genuine contradiction to the project manager rather than guessing.
- Do NOT create `src/safety/pii_scanner.py` in this story. That is E-006-04. The hook should detect scanner absence and pass silently.
- The `pii-check.sh` hook exits 0 even when it issues a denial -- the denial is communicated via JSON output (`permissionDecision: "deny"`), not via exit code. This follows the Claude Code hooks protocol.
- Merging the `hooks` block into `.claude/settings.json`: read the existing file first, then add the new key. Do not reformat or reorder other settings.
- `jq` is required for `pii-check.sh`. The design doc assumes `jq` is available in the developer environment. If it is not, the script should detect its absence and exit 0 (fail open, do not block commits).
