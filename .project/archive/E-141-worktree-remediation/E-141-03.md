# E-141-03: Rule File Updates for Single-Tier Model

## Epic
[E-141: Single Epic Worktree Model and Hook Enforcement](epic.md)

## Status
`DONE`

## Description
After this story is complete, rule files, agent definitions, and skills that reference worktree mechanics will describe the single-tier epic worktree model consistently with the rewritten implement skill. The two-tier worktree documentation, story worktree references, and epic worktree exception framing will be removed. Agent definitions will reflect that PM works in the epic worktree during dispatch and that stories execute serially.

## Context
Three rule files, two agent definitions, and two skills reference worktree mechanics: `worktree-isolation.md` (the primary worktree reference), `dispatch-pattern.md` (team roles and permitted operations), `workflow-discipline.md` (gates and routing), `code-reviewer.md` (review instructions referencing story worktrees), `product-manager.md` (dispatch execution, parallel execution rules), `codex-review/SKILL.md` (references "epic worktree exception" framing), and `workflow-help/SKILL.md` (dispatch workflow references). All seven must be updated to match the single-tier model established by E-141-02's implement skill rewrite.

- [ ] **AC-1**: `worktree-isolation.md` describes a single-tier model with only epic worktrees. The "Two Worktree Tiers" section, "Story-level worktrees" subsection, and "Story Worktree Constraints" section are removed or replaced. Per Technical Notes TN-3.
- [ ] **AC-2**: `worktree-isolation.md` contains epic worktree constraints: no Docker, no credentials, no commits, no cd to main, use absolute paths. Only one path pattern: `/tmp/.worktrees/baseball-crawl-E-NNN/`.
- [ ] **AC-3**: `worktree-isolation.md` includes a brief note about hook enforcement referencing the Write/Edit path guard from E-141-01.
- [ ] **AC-4**: `dispatch-pattern.md` Team Roles and Permitted Orchestration sections reference the epic worktree (not story worktrees). Per-story patch-apply and story worktree cleanup are removed from permitted operations.
- [ ] **AC-5**: `workflow-discipline.md` Workflow Routing Rule no longer mentions per-story merge-back or story worktree lifecycle.
- [ ] **AC-6**: No Serial Dispatch Gate is added to `workflow-discipline.md` (parallel epics are safe under the single-tier model — this gate is no longer needed).
- [ ] **AC-7**: All seven files (`worktree-isolation.md`, `dispatch-pattern.md`, `workflow-discipline.md`, `code-reviewer.md`, `product-manager.md`, `codex-review/SKILL.md`, `workflow-help/SKILL.md`) use consistent terminology matching the rewritten implement skill (e.g., "epic worktree" not "story worktree," "staging boundary" not "patch-apply").
- [ ] **AC-8**: `worktree-isolation.md` is shorter than its current version (~42 lines today → ~25 lines target).
- [ ] **AC-9**: `code-reviewer.md` no longer references story worktree paths, `git diff --cached` from story worktrees, or per-story reviews from individual story worktrees. Instead, it references the epic worktree and the staging boundary protocol (`git diff` for unstaged current-story changes).
- [ ] **AC-10**: `codex-review/SKILL.md` no longer references "epic worktree exception" framing. Working in the epic worktree is the default, not an exception.
- [ ] **AC-11**: `workflow-help/SKILL.md` reflects the single-tier worktree model. Any dispatch or worktree references are updated to match the new architecture.
- [ ] **AC-12**: `product-manager.md` Execution step (line 113) reflects that PM works IN the epic worktree during dispatch, not just receives the path for AC verification. - [ ] **AC-13**: `product-manager.md` Parallel Execution Rules section (lines 117-121) acknowledges that stories execute serially under the single-tier model. Dependency ordering still matters (determines execution ORDER), but the rules are about sequencing, not concurrency. File-conflict detection (rule 1) remains valid as a planning constraint.
- [ ] **AC-14**: All agent definitions in `.claude/agents/` have been audited for references to the old two-tier worktree model, story worktrees, or working on main during dispatch. Only `code-reviewer.md` and `product-manager.md` require updates (verified by grep — no other agent defs contain worktree-specific content).

## Technical Approach
Update three rule files, two agent definitions, and two skills to match the single-tier model. The implement skill (rewritten in E-141-02) is the authoritative source — rule files, agent definitions, and skills cross-reference it rather than duplicating procedures. The changes are primarily deletions and simplifications. Audit all agent definitions in `.claude/agents/` for old-model references (grep confirms only `code-reviewer.md` and `product-manager.md` need updates).

## Notes
- The rule files should be shorter after this story, not just different. If a file is growing, the change is likely duplicating content that belongs in the implement skill.
- The Serial Dispatch Gate (from the earlier E-141 design) is NOT added. Parallel epic dispatch is safe under the single-tier model — each epic has its own worktree with its own file scope.
- `workflow-discipline.md` changes are minimal — most worktree-specific content lives in `worktree-isolation.md` and `dispatch-pattern.md`.
- `code-reviewer.md` Worktree Review section (~20 lines) requires a full rewrite, not find-and-replace — the review workflow changes from story-worktree `git diff --cached` to epic-worktree staging boundary protocol (`git diff` for unstaged).
- `codex-review/SKILL.md` change is minimal (~1 line): remove "epic worktree exception" phrasing from the spawning mechanics section.
