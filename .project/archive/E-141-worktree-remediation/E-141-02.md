# E-141-02: Implement Skill Rewrite for Single-Tier Model

## Epic
[E-141: Single Epic Worktree Model and Hook Enforcement](epic.md)

## Status
`DONE`

## Description
After this story is complete, the implement skill will describe a single-tier worktree model where all agents work in the epic worktree, stories execute serially, and per-story diff isolation uses a staging boundary protocol. The skill will be ~30% shorter (~620 lines vs. ~880 today) with no references to story worktrees, per-story patch-apply, or the context-layer exception.

## Context
The implement skill is the authoritative source for all dispatch procedures. The current skill was designed around a two-tier model (story worktrees for agents, epic worktree for patch accumulation) with a context-layer exception. The new single-tier model eliminates the story worktree lifecycle, per-story patch-apply flow, context-layer exception, and multi-wave parallel assignment — replacing them with serial story execution and a staging boundary protocol for CR diff isolation.

## Acceptance Criteria
- [ ] **AC-1**: The implement skill contains no references to story worktrees, story worktree creation, story worktree cleanup, or story worktree constraints. Per Technical Notes TN-3 (removed sections).
- [ ] **AC-2**: The implement skill contains no per-story patch-apply flow (`git diff --binary --cached main` in story worktree, `git apply --3way` in epic worktree). Per Technical Notes TN-3 (removed sections).
- [ ] **AC-3**: The implement skill contains no context-layer exception (isolation-depends-on-file-mix, merge-back skip for context-layer stories, separate context-layer commit). Per Technical Notes TN-3 (removed sections).
- [ ] **AC-4**: The implement skill contains no multi-wave parallel assignment logic. Stories are assigned and executed serially within an epic. Per Technical Notes TN-3 (removed sections).
- [ ] **AC-5**: All agents (implementers, CR, PM) are spawned WITHOUT `isolation: "worktree"` and receive a universal spawn context with the epic worktree path, instruction to use absolute paths, and epic/story file locations relative to the worktree. Per Technical Notes TN-4.
- [ ] **AC-6**: The implement skill describes the staging boundary protocol: after each story passes review, `git add -A` in the epic worktree; CR reviews `git diff` (unstaged) for per-story isolation. Per Technical Notes TN-5.
- [ ] **AC-7**: The code-reviewer skip for context-layer-only stories is preserved (PM verifies ACs alone). This is review routing, not isolation.
- [ ] **AC-8**: Closure merge produces a single commit from the epic worktree. No targeted staging caveats, no separate context-layer commit. Per Technical Notes TN-3 (simplified closure).
- [ ] **AC-9**: Epic worktree constraints are documented in the skill (no Docker, no credentials, no commits, no cd to main, use absolute paths). These replace the longer story worktree constraints block. Per Technical Notes TN-3 (added sections).
- [ ] **AC-10**: The implement skill references the Write/Edit path guard hook (E-141-01) as the primary enforcement mechanism for implementation isolation. For the Bash bypass gap (AC-11), it uses an instruction-based backup. The enforcement model is hook-first, instruction backup for edge cases.
- [ ] **AC-11**: Epic worktree constraints include an explicit prohibition on Bash-based file writes (`echo >`, `sed -i`, `cat >`, `cp`, `mv`) to protected paths (`src/`, `tests/`, `migrations/`, `scripts/`). This is an instruction-based backup for the Bash bypass gap that Write/Edit hooks cannot enforce.
- [ ] **AC-12**: The implement skill includes post-story path verification: when an implementer reports completion, any reported changed file path that does NOT match the epic worktree pattern (`/tmp/.worktrees/baseball-crawl-E-NNN/`) triggers escalation before proceeding. Per Technical Notes TN-3 (added sections).
- [ ] **AC-13**: The `handoff_from_plan` path (Phase 1 and Phase 2 conditionals) is preserved and adapted for the single-tier model: plan skill handoff still skips Steps 1-3 when the planning team and epic worktree already exist; agent spawning uses the universal spawn context (no `isolation: "worktree"`). Per Technical Notes TN-3 (kept unchanged).
- [ ] **AC-14**: Epic worktree constraints document the pytest limitation: pytest from the epic worktree tests main's code due to the editable install's meta path finder. Implementers run tests for verification but report results; CR verifies through file inspection. Per Technical Notes TN-8.
