# E-154-03: Backfill migration for teams.public_id

## Epic
[E-154: Opponent Resolution Pipeline Hardening](epic.md)

## Status
`DONE`

## Description
After this story is complete, a new numbered SQL migration will exist that backfills `teams.public_id` from `opponent_links.public_id` for any resolved opponents where the teams row is missing its `public_id`. This heals existing data inconsistencies and documents the healing intent in the migration log.

## Context
The manual SQL fix applied on 2026-03-25 patched 6 rows for the Freshman Grizzlies. Other member teams may have the same gap. This migration ensures all existing resolved opponents have consistent `teams.public_id` values, regardless of when they were resolved. The migration is idempotent and safe to re-run.

While E-154-01 fixes the code path for future resolutions, this migration heals historical data. Both are needed for completeness.

## Acceptance Criteria
- [ ] **AC-1**: A new numbered migration file exists in `migrations/` with the next sequential number.
- [ ] **AC-2**: The migration contains an idempotent UPDATE statement that copies `public_id` from `opponent_links` to `teams` for resolved opponents where `teams.public_id IS NULL`, per Technical Notes TN-4.
- [ ] **AC-3**: The migration does not modify rows where `teams.public_id` is already populated.
- [ ] **AC-4**: The migration handles the case where multiple `opponent_links` rows point to the same `resolved_team_id` with different `public_id` values — it uses deterministic selection via `ORDER BY resolved_at DESC, id DESC LIMIT 1`, per Technical Notes TN-4.
- [ ] **AC-5**: The migration skips rows where the target `public_id` already exists on a different `teams` row (UNIQUE collision guard), per Technical Notes TN-4.
- [ ] **AC-6**: A test applies the migration to a seeded test database (with teams rows missing `public_id` and corresponding `opponent_links` rows that have `public_id`) and verifies the backfill populates `teams.public_id` correctly.

## Technical Approach
Create the next numbered migration file following the project convention (e.g., `migrations/005_backfill_teams_public_id.sql`). Use the SQL from TN-4 in the epic Technical Notes. Determine the correct migration number by checking existing files in `migrations/`.
