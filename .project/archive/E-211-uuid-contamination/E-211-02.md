# E-211-02: Fix Report Generator gc_uuid Resolution and Plays Scoping

## Epic
[E-211: Fix Opponent-Perspective UUID Contamination](epic.md)

## Status
`DONE`

## Description
After this story is complete, the report generator will always resolve gc_uuid via `POST /search` (never trusting stored values on tracked teams), and the plays stage will only process games from the report's own crawled schedule (not games loaded by other teams' pipelines).

## Context
Two downstream bugs in the report generator are caused by trusting contaminated gc_uuid values. First, the gc_uuid resolution (lines 970-987) skips search when a gc_uuid is already stored, but that stored value may be opponent-perspective. Second, the plays query (lines 553-562) uses a broad `WHERE home_team_id = ? OR away_team_id = ?` that picks up games from other pipelines, causing opponent-perspective player UUIDs to leak in. See epic Technical Notes TN-3.

## Acceptance Criteria
- [ ] **AC-1**: Given a tracked team has a `gc_uuid` stored in the DB, when the report generator resolves gc_uuid for spray chart access, then it always calls `_resolve_gc_uuid` via search for tracked teams (ignoring the stored gc_uuid). Member teams are excluded -- their gc_uuid from the authenticated API is authoritative. The search-resolved UUID is used for spray crawl/load.
- [ ] **AC-2**: Given the report generator's plays stage runs, when it discovers games to process, then it uses filesystem-only game discovery -- deriving game IDs from boxscore filenames in the report's scouting directory (`data/raw/{crawl_season_id}/scouting/{public_id}/boxscores/*.json`), not a DB query. Each filename stem is the `event_id` (= `game_id` in the DB). The previous `WHERE home_team_id = ? OR away_team_id = ?` DB query is replaced entirely.
- [ ] **AC-3**: Given a game exists in the DB because it was loaded by a member team's pipeline (the scouted team appears as an opponent), when that game has no boxscore file in the report's scouting directory, then the plays stage does not process it (it is not in the filesystem-derived game list).
- [ ] **AC-4**: Existing tests pass. New tests verify the search-always behavior and the scoped plays query.
- [ ] **AC-5**: The `_query_plays_pitching_stats` function (`src/reports/generator.py`) is scoped to the report's game set -- it no longer uses the broad `WHERE (g.home_team_id = ? OR g.away_team_id = ?)` pattern. Plays from cross-pipeline games are excluded from the pitching stats aggregation.

## Technical Approach
Two changes in `src/reports/generator.py`, per TN-3. The gc_uuid resolution block should be restructured to always search-resolve for tracked teams (ignoring stored gc_uuid). The `_crawl_and_load_plays` method should replace the DB query entirely with filesystem-driven game discovery -- derive game IDs from boxscore filenames in the scouting directory.

## Notes
- The search-resolved gc_uuid should be written back unconditionally for tracked teams (`UPDATE teams SET gc_uuid = ? WHERE id = ? AND membership_type = 'tracked'`). This overwrites any previously contaminated value and caches the correct resolution. The `membership_type` guard ensures member teams (whose gc_uuid comes from the authenticated API) are never touched.
- The scouting directory path pattern is `data/raw/{crawl_season_id}/scouting/{public_id}/boxscores/*.json` where each filename stem is the `event_id` (= `game_id` in the DB).
