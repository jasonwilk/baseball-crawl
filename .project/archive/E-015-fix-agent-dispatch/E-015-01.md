# E-015-01: Document the Dispatch Constraint in a Rules File

## Epic
[E-015: Fix Agent Dispatch -- Nested Session Error](epic.md)

## Status
`DONE`

## Description
After this story is complete, a rules file at `/.claude/rules/dispatch-pattern.md` will exist documenting the subagent nesting constraint, the two PM operating modes (main session vs. subagent), and the Dispatch Manifest format. Any agent or user reading this file will understand why nested dispatch fails and what to do instead.

## Context
The root cause of the E-011 dispatch failure is a platform constraint: `CLAUDECODE=1` is set in the environment of any active Claude Code session, and the Task tool (which launches a new `claude` subprocess) fails immediately if that variable is inherited. Subagents cannot spawn subagents. The PM's Dispatch Mode procedure calls the Task tool -- which only works when the PM is the top-level session, not when it is itself a subagent invoked by the orchestrator.

This rules file is the foundation for the two subsequent stories (E-015-02 updates the orchestrator, E-015-03 updates the PM). Both stories reference this document for the exact constraint and format definitions. It must exist first.

## Technical Approach

Write the file as a rules file following the same pattern as `/.claude/rules/workflow-discipline.md`:
- YAML frontmatter at top: `paths: ["**"]`
- Section structure: The Nesting Constraint / PM Operating Modes / Dispatch Manifest Format / Orchestrator Routing Rule
- Keep each section tight -- 3-8 lines per section is sufficient

The Dispatch Manifest Format section should include a literal markdown code block showing the format:

```markdown
## Dispatch Manifest for E-NNN

The PM is running in a subagent context and cannot dispatch directly.
Execute these Task tool calls from your main session:

### Task 1: E-NNN-01 -- [Story Title]
Agent: [agent-name]
Prompt:
  [Full standard context block]
```

Reference the Technical Notes in the parent epic (`epic.md`) for the full context on why this pattern exists.
