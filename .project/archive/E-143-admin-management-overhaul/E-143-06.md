# E-143-06: Crawl Trigger UI with Status Display

## Epic
[E-143: Admin Management Overhaul](epic.md)

## Status
`DONE`

## Description
After this story is complete, the operator can trigger a data refresh for any team directly from the admin teams table and see when each team was last crawled and whether it succeeded. This eliminates the CLI bottleneck for the most time-sensitive operator workflow: refreshing opponent data the night before a game.

## Context
E-143-05 provides per-team crawl+load scoping. E-143-06a provides the `crawl_jobs` tracking table. This story adds the HTTP endpoint, background execution, and UI surface. The baseball-coach emphasized that data freshness is a trust issue -- coaches stop using the system if data is stale and they can't tell when it was last updated. The api-scout confirmed auth tokens should be refreshed at crawl start to avoid mid-run expiration. Per TN-2, membership type determines which pipeline to invoke.

## Acceptance Criteria
- [ ] **AC-1**: `POST /admin/teams/{id}/sync` enqueues a per-team crawl as a FastAPI `BackgroundTasks` task, creates a `crawl_jobs` row with `status='running'`, and redirects to the teams page with a flash: "Sync started for [team name]."
- [ ] **AC-2**: The background task dispatches to the correct pipeline per TN-2: member teams → `crawl.run(source="db", team_ids=[id])` then `load.run(source="db", team_ids=[id])`, tracked teams → scouting pipeline's combined crawl+load flow. Both steps are always invoked -- without the load step, crawled data stays in `data/raw/` and dashboards remain stale. For tracked teams, the scouting pipeline's existing `scouting_runs` update behavior is preserved -- `crawl_jobs` is an additional status record for the UI surface, not a replacement. The scouting freshness check that reads `scouting_runs.status='completed'` must continue to work.
- [ ] **AC-2b**: The background task calls the auth token refresh at the start of each crawl run before invoking the pipeline, per TN-2 auth-freshness requirement.
- [ ] **AC-2c**: On completion (success or failure), the `crawl_jobs` row is updated to `completed` or `failed` with `completed_at` and optional `error_message`.
- [ ] **AC-2d**: On successful completion, `teams.last_synced` is updated for the synced team.
- [ ] **AC-3**: The Sync button (POST form, like the existing Discover button pattern) is shown ONLY for: (a) active member teams, and (b) active tracked teams where `public_id IS NOT NULL`. Inactive teams and unresolved tracked teams (`public_id IS NULL`) show no Sync button. Unresolved tracked teams display a muted "Unresolved -- map first" indicator instead. The `POST /admin/teams/{id}/sync` route enforces the same eligibility server-side: return a flash error and redirect if the team is inactive or is a tracked team with `public_id IS NULL`.
- [ ] **AC-4**: Each team row displays a "Last synced" value from `teams.last_synced` (updated on job completion per AC-2). If never synced, display "Never." Current sync status comes from the latest `crawl_jobs` row's `status` field: completed (green/checkmark "Success" text), failed (red/warning text), or running (yellow/"running" text). These are two separate data sources -- `teams.last_synced` is never null for previously synced teams even when a new sync is running.
- [ ] **AC-5**: The Sync button is disabled or visually indicates "Running..." if a crawl is currently in progress for that team (`status='running'` in the latest `crawl_jobs` row).
- [ ] **AC-6**: The sync route requires admin access.
- [ ] **AC-7**: Existing CLI crawl tests in `tests/test_cli_data.py`, `tests/test_scripts/test_crawl_orchestrator.py`, and `tests/test_scripts/test_load_orchestrator.py` pass without modification. `bb data crawl`, `bb data sync`, and `bb data load` commands function as before.
- [ ] **AC-8**: The post-add flash message on the teams page that currently references `bb data sync` is updated to reference the new UI Sync button instead, removing the CLI instruction.

## Technical Approach
Add the POST sync route that creates a tracking row and enqueues the crawl via `BackgroundTasks`. The background task function must be self-contained (own DB connection, error handling) since it runs after the HTTP response. The teams list query should LEFT JOIN against `crawl_jobs` (latest per team) to get status/timestamp for display. Check how `scouting_runs` is queried for a pattern to follow. The Sync button uses the same POST form pattern as the existing Discover button.

For tracked-team sync, `src/pipeline/trigger.py` should implement a `run_scouting_sync(team_id, public_id)` function that directly invokes `ScoutingCrawler` and the scouting loader to replicate the CLI's `_run_scout_pipeline` behavior — including the `scouting_runs` status update. Do NOT import from `src/cli/` (import boundary: cli imports from pipeline, not the reverse). Mirror only the orchestration logic needed.
