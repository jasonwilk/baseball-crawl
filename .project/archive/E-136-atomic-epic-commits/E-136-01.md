# E-136-01: Update Implement Skill for Atomic Commits

## Epic
[E-136: Atomic Epic Commits](epic.md)

## Status
`DONE`

## Description
After this story is complete, the implement skill will describe an atomic commit workflow where implementers stage but do not commit, code review uses `git diff --cached`, merge-back uses patch-apply instead of git merge, and closure produces a single commit per epic.

## Context
This is the primary story of the epic -- it modifies the authoritative dispatch procedure document. The implement skill (`SKILL.md`) is the single source of truth for how dispatch works. Every phase of dispatch is affected: implementer instructions (Phase 3 Step 4), code-reviewer templates (Phase 3 Step 5), merge-back (Phase 3 Step 5a), worktree verification (Phase 5 Step 1), and closure (Phase 5 Step 10). The workflow summary diagram and anti-patterns section must also be updated.

## Acceptance Criteria
- [ ] **AC-1**: The worktree context block in Phase 3 Step 4 tells implementers to run `git add -A` to stage all changes but NOT to run `git commit`, per TN-1.
- [ ] **AC-2**: The "Use Git for Inspection" section in the worktree context block removes the line about committing being fine, and instead states that staging (`git add -A`) is required before reporting completion.
- [ ] **AC-3**: The code-reviewer template in Phase 3 Step 5 uses `git diff --cached main` instead of `git diff main..HEAD`, per TN-2.
- [ ] **AC-4**: Phase 3 Step 5a (merge-back) replaces `git merge --no-ff <branch>` with the patch-apply sequence from TN-3: generate patch in worktree, apply in main checkout, remove worktree, delete branch with `-D`.
- [ ] **AC-5**: Phase 3 Step 5a includes conflict/failure handling per TN-3: story stays IN_PROGRESS, worktree stays active, escalate to user, recovery via `git checkout -- . && git clean -fd`.
- [ ] **AC-6**: Phase 5 Step 1 (worktree verification) checks that no worktrees remain rather than checking that branches are merged, per TN-5.
- [ ] **AC-7**: Phase 5 Step 10 replaces "Offer to scan and commit" with the atomic commit sequence from TN-4: `git add -A`, PII scan, user approval, single `git commit` with conventional message.
- [ ] **AC-8**: The workflow summary diagram reflects the new flow: staging instead of committing, patch-apply instead of git merge, atomic commit at closure.
- [ ] **AC-9**: Anti-patterns section adds "committing in worktrees" as an anti-pattern and updates any existing anti-patterns that reference `git merge` for merge-back to reference patch-apply.
- [ ] **AC-10**: The Purpose section (including the permitted operations list in the Purpose section) and any other references to `git merge` for merge-back are updated to reflect patch-apply.
- [ ] **AC-11**: The "Read and Write Source Code" subsection in the worktree context block replaces language about changes being "on an isolated branch" that "will be merged by the main session after review" with language reflecting the staging/patch-apply workflow (no commits, no branch divergence, no merge).

## Technical Approach
The implement skill at `/.claude/skills/implement/SKILL.md` is a large file with multiple interrelated sections. Each AC targets specific sections. The changes are consistent with the Technical Notes in the parent epic -- use TN-1 through TN-7 as the design specification. Read the full file before making changes to understand cross-references between sections.
