# E-136-03: Add No-Committing Prohibition to Worktree Isolation

## Epic
[E-136: Atomic Epic Commits](epic.md)

## Status
`DONE`

## Description
After this story is complete, the worktree isolation safety net will prohibit `git add` and `git commit` in worktrees, with an explanation that implementers stage changes via `git add -A` as part of their completion workflow but do not commit -- the main session handles the atomic commit at closure.

## Context
The worktree isolation file (`worktree-isolation.md`) is a safety-net rule loaded for all agents. It currently prohibits Docker commands, credential access, and branch management. The atomic commit workflow adds a new prohibition: implementers must not commit in worktrees. They stage changes (`git add -A`) as the final step before reporting completion, but `git commit` is prohibited because the main session produces a single atomic commit at epic closure.

Note: `git add -A` is part of the implementer's completion workflow (stage everything before reporting), so `git add` itself is NOT prohibited -- only `git commit` is. The prohibition should make this distinction clear.

## Acceptance Criteria
- [ ] **AC-1**: A new bullet point prohibits `git commit` in worktrees, with a brief explanation that the main session handles commits atomically at epic closure.
- [ ] **AC-2**: The prohibition does NOT prohibit `git add` -- it clarifies that `git add -A` is required before reporting completion (staging for diff visibility) but `git commit` is not allowed.
- [ ] **AC-3**: The existing "No branch/worktree management" bullet is unchanged (it already covers `git merge`, `git rebase`, etc.).

## Technical Approach
The worktree isolation file at `/.claude/rules/worktree-isolation.md` is a short safety-net rule (~14 lines). Add one new bullet point between the existing prohibitions. Keep the same terse style as existing bullets.
