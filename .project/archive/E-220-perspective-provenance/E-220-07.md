# E-220-07: Season Aggregate Perspective Filtering

## Epic
[E-220: Perspective-Aware Data Architecture](epic.md)

## Status
`DONE`

## Description
After this story is complete, the season aggregate computation in ScoutingLoader filters by `perspective_team_id` when summing per-game stats, preventing double-counting when the same game has been loaded from multiple perspectives.

## Context
`ScoutingLoader._compute_season_aggregates()` sums all `player_game_batting`/`player_game_pitching` rows for a `(team_id, season_id)` pair. After E-220-02 and E-220-05, the same game may have stat rows from multiple perspectives (e.g., Team A's scouting run loaded the game, then Team B's scouting run also loaded it). Without filtering, a player's stats would be double-counted.

## Acceptance Criteria
- [ ] **AC-1**: Given a game loaded from two perspectives (perspective A and perspective B), when `_compute_season_aggregates(team_id=A, season_id=X)` runs, then it sums only rows where `perspective_team_id = A`
- [ ] **AC-2**: Given a player with 3 hits in a game loaded from perspective A and the same 3 hits from perspective B, when season aggregates are computed for team A, then the player shows 3 hits (not 6)
- [ ] **AC-3**: The `_compute_batting_aggregates` and `_compute_pitching_aggregates` SQL queries include `AND pgb.perspective_team_id = ?` / `AND pgp.perspective_team_id = ?` filter clauses
- [ ] **AC-4**: A code comment at the aggregate query site documents the other known query sites that reference `player_game_batting`/`player_game_pitching` and may need perspective filtering assessment: `src/api/db.py` (pitching workload, pitcher profiles), `src/db/player_dedup.py` (recompute_batting_aggregates, recompute_pitching_aggregates). These are read-for-display or dedup-recompute queries -- perspective filtering for them is deferred to a follow-up assessment.
- [ ] **AC-5**: All existing scouting loader tests pass (updated for new query behavior)

## Technical Approach
Add `WHERE perspective_team_id = ?` to the aggregate queries in `_compute_batting_aggregates()` and `_compute_pitching_aggregates()`. The `team_id` parameter (the team being aggregated) is the correct value for `perspective_team_id` in the scouting context -- scouting runs load data from the scouted team's perspective.

## Notes
- The member-team pipeline computes season stats via the season-stats API endpoint (not aggregation), so it is unaffected.
- The `player_season_batting`/`player_season_pitching` tables do NOT have a `perspective_team_id` column (TN-1). They represent the definitive season totals for a team, computed from that team's own-perspective data.
- **Assumption: `team_id == perspective_team_id` in scouting context.** The aggregate queries use `perspective_team_id = team_id` because scouting runs always load data from the scouted team's own perspective. This equivalence holds for all current scouting flows. If a future flow loads data from a different team's perspective into the same aggregation scope, this assumption must be revisited. Document this assumption in a code comment at the query site.
