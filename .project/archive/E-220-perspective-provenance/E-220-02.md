# E-220-02: GameLoader Perspective Tagging

## Epic
[E-220: Perspective-Aware Data Architecture](epic.md)

## Status
`DONE`

## Description
After this story is complete, the GameLoader passes `perspective_team_id` on every batting and pitching INSERT, inserts a row into `game_perspectives` for each game loaded, and all existing game loader tests pass with the new column.

## Context
The GameLoader is the primary contamination vector. It loads both `own_data` and `opp_data` from every boxscore. After this story, both sides are still loaded but every row carries `perspective_team_id = owned_team_ref.id` (the team whose API call fetched the boxscore). The `game_perspectives` junction table tracks which team perspectives have been loaded for each game.

## Acceptance Criteria
- [ ] **AC-1**: Given a boxscore loaded via `GameLoader.load_file()`, when batting and pitching rows are inserted, then every row has `perspective_team_id` set to `owned_team_ref.id`
- [ ] **AC-2**: Given the same game loaded from two different team perspectives (two separate `GameLoader` instances with different `owned_team_ref`), when both loads complete, then the database contains two sets of batting/pitching rows with different `perspective_team_id` values for the same `game_id`
- [ ] **AC-3**: Given a game loaded via `GameLoader`, when the load completes, then a `game_perspectives` row exists with `(game_id, perspective_team_id)` matching the load
- [ ] **AC-4**: All existing `test_game_loader.py` tests pass (updated where they assert on schema changes)
- [ ] **AC-5**: Given a boxscore with both `own_data` and `opp_data`, when loaded, then both sides' stat rows carry the same `perspective_team_id` (the fetching team, not the team the stats belong to)
- [ ] **AC-6**: Given a game loaded via the member pipeline path (`GameLoader.load_all()`), when the load completes, then every batting/pitching row has `perspective_team_id` set to the member team's integer PK
- [ ] **AC-7**: Given a cross-perspective game where `_find_duplicate_game()` redirects to a canonical `game_id`, when the load completes, then the `game_perspectives` row uses the canonical `game_id` and the loading team's `perspective_team_id`
- [ ] **AC-8**: All `ON CONFLICT` clauses in batting/pitching upserts are updated from `ON CONFLICT(game_id, player_id)` to `ON CONFLICT(game_id, player_id, perspective_team_id)` per TN-13

## Technical Approach
The `perspective_team_id` value is `self._team_ref.id`, already available in `GameLoader.__init__`. Pass it through `_upsert_game_and_stats` to `_load_team_stats` and into every batting/pitching INSERT. After game upsert, INSERT OR IGNORE into `game_perspectives`. Update test fixtures to include `perspective_team_id` in assertions.

## Notes
- The `team_id` column on stat rows still means "which team this player plays for." The new `perspective_team_id` means "which team's API call produced this row." These are different concepts.
- `opp_data` rows get `perspective_team_id = owned_team_ref.id` (the fetching team), NOT the opponent's team_id.
