# E-220-03: PlaysLoader Perspective Tagging

## Epic
[E-220: Perspective-Aware Data Architecture](epic.md)

## Status
`DONE`

## Description
After this story is complete, the PlaysLoader passes `perspective_team_id` on every `plays` INSERT, and the UNIQUE constraint `(game_id, play_order, perspective_team_id)` allows the same game's plays to be loaded from multiple perspectives without conflict.

## Context
The plays loader currently uses whole-game idempotency (`SELECT 1 FROM plays WHERE game_id = ? LIMIT 1`). With the new schema, idempotency must also consider `perspective_team_id` -- a game may have plays from perspective A but not yet from perspective B. The `batter_id` and `pitcher_id` in plays data are perspective-specific (different UUIDs for the same human depending on the fetching team).

## Acceptance Criteria
- [ ] **AC-1**: Given plays loaded via `PlaysLoader`, when plays rows are inserted, then every row has `perspective_team_id` set to `owned_team_ref.id`
- [ ] **AC-2**: Given plays for a game already loaded from perspective A, when the same game's plays are loaded from perspective B, then both sets coexist (different `perspective_team_id` values)
- [ ] **AC-3**: Whole-game idempotency check includes `perspective_team_id`: `SELECT 1 FROM plays WHERE game_id = ? AND perspective_team_id = ? LIMIT 1`
- [ ] **AC-4**: All existing `test_plays_loader.py` tests pass (updated where they assert on schema changes)
- [ ] **AC-5**: Given plays loaded via the member pipeline path (`PlaysLoader.load_all()`), when the load completes, then every plays row has `perspective_team_id` set to the member team's integer PK
- [ ] **AC-6**: Any `ON CONFLICT` or `INSERT OR IGNORE` clauses on plays table are updated to include `perspective_team_id` in the conflict target per TN-13

## Technical Approach
The `perspective_team_id` value is `self._team_ref.id`, already available in `PlaysLoader.__init__`. Add it to every plays INSERT. Update the idempotency check to include `perspective_team_id`. Update test fixtures.
