# E-220-05: ScoutingLoader In-Memory Pipeline

## Epic
[E-220: Perspective-Aware Data Architecture](epic.md)

## Status
`DONE`

## Description
After this story is complete, the scouting boxscore pipeline (ScoutingCrawler + ScoutingLoader) passes data in-memory from crawl to load rather than writing JSON files to disk. The scouting crawl+load cycle produces zero new boxscore/roster/games files in `data/raw/`. The crawler returns boxscore data, roster data, and games metadata as in-memory structures that the loader consumes directly.

## Context
The disk cache (`data/raw/{season}/scouting/{public_id}/`) is the primary vector for run-isolation failures. A file saved by one scouting run gets silently consumed by a subsequent run without perspective metadata. The scouting pipeline already runs crawl+load as a single operation (`run_scouting_sync` in trigger.py, `_run_scout_pipeline` in CLI), making in-memory handoff straightforward.

This story covers the boxscore pipeline only (ScoutingCrawler + ScoutingLoader). The spray chart pipeline (ScoutingSprayChartCrawler + ScoutingSprayChartLoader) is converted in E-220-10.

## Acceptance Criteria
- [ ] **AC-1**: Given a scouting sync triggered via `run_scouting_sync()`, when the boxscore crawl+load cycle completes, then no new files are written under `data/raw/{season}/scouting/` for games.json, roster.json, or boxscores/
- [ ] **AC-2**: `ScoutingCrawler.scout_team()` returns an in-memory result per TN-12 containing games list, roster data, and boxscore dicts
- [ ] **AC-3**: `ScoutingLoader.load_team()` accepts the in-memory crawl result instead of a filesystem path
- [ ] **AC-4**: The CLI `bb data scout` command uses the in-memory pipeline (same as web trigger)
- [ ] **AC-5**: Given a scouting sync for a team, when the crawl returns boxscores, then `perspective_team_id` is set to the scouted team's integer PK on all loaded stat rows (inherited from E-220-02 via GameLoader delegation)
- [ ] **AC-6**: `run_scouting_sync()` in trigger.py continues to maintain `scouting_runs` status tracking and `crawl_jobs` row updates despite the removal of filesystem intermediaries
- [ ] **AC-7**: The current `CrawlResult` dataclass (with `files_written`, `files_skipped`) is replaced or adapted per TN-14. All callers that branch on `CrawlResult` fields (`trigger.py`, CLI) are updated for the new return type.
- [ ] **AC-8**: Crawler DB side effects are preserved per TN-15: `scouting_runs`, `teams`, and `seasons` table writes continue to function in the in-memory pipeline
- [ ] **AC-9**: All existing scouting loader/crawler tests pass (updated for new API signatures)

## Technical Approach
Modify `ScoutingCrawler.scout_team()` to return a dataclass or dict containing the fetched data instead of writing files. Modify `ScoutingLoader.load_team()` to accept this in-memory result. Update `run_scouting_sync()` in `trigger.py` and the CLI scout command to pass data in-memory. The `scouting_dir` existence check in trigger.py (line ~719-724) will need to be removed or replaced since no directory is created. The `scouting_runs` and `crawl_jobs` tracking must be preserved.
