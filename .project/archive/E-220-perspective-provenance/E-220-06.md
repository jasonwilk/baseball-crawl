# E-220-06: Report Generator In-Memory Pipeline

## Epic
[E-220: Perspective-Aware Data Architecture](epic.md)

## Status
`DONE`

## Description
After this story is complete, the report generator (`src/reports/generator.py`) uses an in-memory crawl-to-load pipeline instead of discovering games from filesystem globs and checking file existence. Game IDs come from crawl results, not boxscore filenames on disk. The report generation cycle produces zero new scouting/plays/spray files in `data/raw/`.

## Context
The report generator currently discovers games from boxscore filenames on disk (line 568) and skips plays/spray fetch if a file already exists (lines 588-590). This creates cross-report contamination: Report A's cached files are silently consumed by Report B's load. After this story, each report generation is self-contained with in-memory data flow.

## Acceptance Criteria
- [ ] **AC-1**: Given `generate_report()` is called, when the scouting crawl+load completes, then no new files are written under `data/raw/` for scouting, plays, or spray data
- [ ] **AC-2**: Game discovery uses crawl results (in-memory game list) instead of `boxscores_dir.glob("*.json")`
- [ ] **AC-3**: Plays and spray stages do not check for file existence on disk -- they fetch and load in-memory unconditionally (DB idempotency handles duplicates)
- [ ] **AC-4**: Given two reports generated for teams that played each other, when both complete, then neither report's data contaminates the other
- [ ] **AC-5**: All existing report generator tests pass (updated for new flow)
- [ ] **AC-6**: Every stat row loaded during report generation has `perspective_team_id` set to the report's target team
- [ ] **AC-7**: The report generator uses the in-memory `ScoutingCrawler`/`ScoutingLoader` APIs introduced in E-220-05 (restoring functionality broken by E-220-05's signature change)

## Technical Approach
Refactor the report generator's five-stage pipeline (scouting crawl, scouting load, plays crawl+load, spray crawl+load, reconciliation) to use in-memory data handoff. The crawlers return data dicts; the loaders consume them directly. The report generator no longer creates or reads files in `data/raw/` for its scouting stages.
