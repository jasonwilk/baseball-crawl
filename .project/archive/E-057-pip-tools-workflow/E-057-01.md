# E-057-01: Create .in files and compile pinned outputs

## Epic
[E-057: pip-tools Dependency Management](epic.md)

## Status
`DONE`

## Description
After this story is complete, the project will have a pip-tools workflow with `requirements.in` and `requirements-dev.in` declaring direct dependencies using compatibility ranges, and `requirements.txt` and `requirements-dev.txt` generated by `pip-compile` with full transitive pinning. The Dockerfile continues to work unchanged, and all existing tests pass.

## Context
The project currently has a single `requirements.txt` with exact pins for 14 direct dependencies but no transitive pinning. This story introduces the pip-tools two-file workflow. The `.in` files become the human-edited source of truth; the `.txt` files become generated artifacts (still committed to version control for reproducibility). The devcontainer is updated to install pip-tools and dev dependencies.

## Acceptance Criteria
- [ ] **AC-1**: `requirements.in` exists and contains all 10 runtime dependencies (httpx, fastapi, uvicorn[standard], starlette, jinja2, aiofiles, python-multipart, webauthn, python-dotenv, pyyaml) with `~=` compatible-release ranges matching current major.minor versions. Each dep line has an inline `#` comment stating what the dep is for (e.g., `httpx~=0.28  # HTTP client for GameChanger crawlers`). Starlette has an additional comment noting its FastAPI coupling (e.g., `# Pinned explicitly -- FastAPI transitive; we import run_in_threadpool directly`).
- [ ] **AC-2**: `requirements-dev.in` exists, starts with `-r requirements.in`, and contains the 4 dev-only dependencies (pytest, pytest-asyncio, pytest-timeout, respx) with `~=` ranges.
- [ ] **AC-3**: `requirements.txt` is the output of `pip-compile requirements.in` and contains exact pins (`==`) for all direct and transitive dependencies. A header comment marks it as generated (e.g., `# This file is autogenerated by pip-compile`). Every direct dependency's resolved version matches the exact-version pins in the `requirements.txt` at HEAD of `main` before this story begins (no version drift). Baseline reference: `git show HEAD:requirements.txt`.
- [ ] **AC-4**: `requirements-dev.txt` is the output of `pip-compile requirements-dev.in` and contains exact pins for all dev + runtime dependencies and their transitives. A header comment marks it as generated.
- [ ] **AC-5**: The Dockerfile builds successfully with no changes to the `pip install` line. Verify: `docker compose build app` succeeds.
- [ ] **AC-6**: `pip install -r requirements-dev.txt` in the devcontainer installs all runtime + dev dependencies. All existing tests pass (`pytest` exits 0 with no test failures).
- [ ] **AC-7**: `.devcontainer/devcontainer.json` `postCreateCommand` is updated: (a) `pip install pip-tools` is added before the requirements install, and (b) the existing `pip install -r requirements.txt` is changed to `pip install -r requirements-dev.txt` so developers get dev deps automatically.
- [ ] **AC-8**: If `--generate-hashes` works for all packages, it is used in the pip-compile invocation. If any package lacks hashes, hashes are omitted and the implementer reports which package(s) blocked it in the completion message to PM. (PM will update epic Technical Notes -- implementers do not edit epic files.)
- [ ] **AC-9**: The old section-header comments in `requirements.txt` (e.g., `# HTTP client`, `# Testing`) are removed -- pip-compile's own annotations replace them. The comments from `requirements.in` serve as the human-readable documentation.

## Technical Approach
1. Install pip-tools: `pip install pip-tools`
2. Create `requirements.in` with the 10 runtime deps using `~=` ranges derived from current exact pins (e.g., `httpx==0.28.1` becomes `httpx~=0.28`). Add starlette with a comment about FastAPI coupling.
3. Create `requirements-dev.in` with `-r requirements.in` and 4 dev deps.
4. Run `pip-compile requirements.in --output-file=requirements.txt --strip-extras` (try `--generate-hashes` first).
5. Run `pip-compile requirements-dev.in --output-file=requirements-dev.txt --strip-extras` (same hash flag decision).
6. Verify direct dependency versions in output match the old file exactly. If pip-compile resolves a different patch version for any direct dep, pin that dep exactly in the `.in` file to maintain parity.
7. Run `pytest` -- all tests must pass.
8. Run `docker compose build app` -- must succeed.
9. Update devcontainer.json postCreateCommand.

- `/workspaces/baseball-crawl/.devcontainer/devcontainer.json` (update postCreateCommand)

## Notes
- pip-tools itself must NOT appear in any `.in` file -- it is a build tool, not a project dependency.
- The `respx` package is a test-only HTTP mocking library (pairs with httpx). It belongs in dev, not runtime.
- The starlette pin exists because the project imports `run_in_threadpool` directly. If FastAPI upgrades starlette internally, our pin must be updated to match. The comment in `requirements.in` documents this coupling.
