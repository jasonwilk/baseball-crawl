# E-121-03: DE Agent Definition + DE Memory Fixes

## Epic
[E-121: Style Guide and Context-Layer Remediation](epic.md)

## Status
`DONE`

## Description
After this story is complete, the data-engineer agent definition will accurately describe the project's FK-safe orphan handling (stub-row mechanism, not "insert anyway"), will distinguish new-table indexing from performance-tuning indexing, and will acknowledge aggregate tables as valid alongside the event-storage pattern. DE memory will have corrected timestamp format and appropriately scoped ip_outs guidance.

## Context
The triage audit found several DE agent definition statements that no longer reflect actual project practice. The "insert anyway" anti-pattern is impossible with FK constraints enabled; the "every index needs EXPLAIN" rule is too strict for new tables; the "store events, compute on read" philosophy contradicts the existence of aggregate season stats tables. DE memory has a minor timestamp format error and an overly absolute ip_outs statement. These fixes are grouped because they share the DE agent def and DE memory files.

Findings addressed: #4 (FK insert-anyway), #5 (index EXPLAIN), #8 (events vs aggregates), #12 (timestamp format), #14 (ip_outs scope).

## Acceptance Criteria
- [ ] **AC-1**: `.claude/agents/data-engineer.md` Anti-Pattern #5 no longer says "insert anyway." It describes the FK-safe orphan handling pattern: insert a stub player row (first_name='Unknown', last_name='Unknown') before the stat row, log WARNING for operator backfill. This matches the pattern documented in CLAUDE.md and the SE agent def.
- [ ] **AC-2**: `.claude/agents/data-engineer.md` Error Handling #4 is consistent with the updated Anti-Pattern #5 (both describe the same stub-row mechanism).
- [ ] **AC-3**: `.claude/agents/data-engineer.md` Anti-Pattern #3 ("Never add speculative indexes") distinguishes two modes: (a) new-table creation, where indexes should be justified by known query patterns from story ACs or coaching requirements, and (b) performance tuning of existing tables, where EXPLAIN QUERY PLAN evidence is required.
- [ ] **AC-4**: `.claude/agents/data-engineer.md` Schema Design section #1 ("Store events, compute aggregates on read") is updated to acknowledge that aggregate tables (season stats) are valid when query-time computation is impractical, while preserving the principle that event-level data is the source of truth.
- [ ] **AC-5**: `.claude/agent-memory/data-engineer/MEMORY.md` timestamp format example matches what SQLite's `datetime('now')` actually produces: `YYYY-MM-DD HH:MM:SS` (space-separated, no T, no Z). The current example `2026-03-01T14:30:00Z` is misleading -- correct it to `2026-03-01 14:30:00`.
- [ ] **AC-6**: `.claude/agent-memory/data-engineer/MEMORY.md` ip_outs section: "No exceptions" is clarified to apply to storage format only (always integer outs in the database). The existing display formula (`ip_outs // 3` for full innings, `ip_outs % 3` for partial) is retained as a read-time reference.
- [ ] **AC-7**: DE agent def Core Entities table and Schema Design section are mutually consistent after changes.
