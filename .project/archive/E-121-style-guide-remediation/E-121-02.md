# E-121-02: http-discipline.md + testing.md Scoping

## Epic
[E-121: Style Guide and Context-Layer Remediation](epic.md)

## Status
`DONE`

## Description
After this story is complete, http-discipline.md's rate limiting rules will be scoped to production/operator sessions (not test code), and testing.md will have appropriately narrowed path scoping and a resolved self-contradiction between targeted test discovery and full pytest guidance.

## Context
The triage audit found that http-discipline.md rate limits technically apply to test code (which mocks HTTP anyway), and testing.md has both overly broad path scoping and contradictory guidance about targeted vs full test runs. These fixes are grouped because they share the testing.md file and both involve rule-scoping adjustments.

Findings addressed: #7 (rate limits in tests), #9 (testing.md paths), #15 (testing.md self-contradiction).

## Acceptance Criteria
- [ ] **AC-1**: `.claude/rules/http-discipline.md` Rate Limiting section includes a scoping note clarifying that rate limits apply to production/operator HTTP sessions, not to test code (which mocks HTTP at the transport layer).
- [ ] **AC-2**: `.claude/rules/testing.md` `paths:` frontmatter retains `src/**` (needed for test-scope-discovery to fire when agents edit source files) but removes `**/*test*.py` (redundant with `tests/**` for this project's layout). A brief inline comment in the frontmatter explains why `src/**` is included.
- [ ] **AC-3**: `.claude/rules/testing.md` "Test Scope Discovery" section and the "Scope" subsection within it are consistent -- no contradiction between "targeted discovery" and "consider a full pytest run" guidance. The guidance clearly states the default (targeted) and the escalation condition (10+ discovered files).
- [ ] **AC-4**: No orphaned cross-references in modified files.

## Technical Approach
Two files are modified: http-discipline.md (add scoping note to Rate Limiting section) and testing.md (adjust frontmatter paths, reword the Scope subsection for clarity). The implementer should read both files, apply changes per Technical Notes TN-1 findings #7, #9, #15, then verify internal consistency per TN-4.
