# E-121-01: CR Memory Cleanup + python-style.md Fixes

## Epic
[E-121: Style Guide and Context-Layer Remediation](epic.md)

## Status
`DONE`

## Description
After this story is complete, the code-reviewer's memory will no longer contain the fabricated "50-line function limit" rule, and python-style.md will be internally consistent and aligned with CLAUDE.md's core principle. The duplicate "Prefer dataclasses" line will be removed, the "not plain dicts" prohibition will be softened to a preference with a local-use exception, the print() rule will include a CLI escape hatch, type hints will be scoped to production code, and the docstring mandate will apply only when purpose is not obvious from the signature.

## Context
The triage audit found that CR memory fabricated a function length rule that contradicts python-style.md, and that python-style.md itself contains several rules that are too absolute for the project's actual patterns. These fixes are grouped because they share the python-style.md file and the CR memory file.

Findings addressed: #1 (CR 50-line rule), #2 (not plain dicts), #3 (never print), #10 (type hints scope), #11 (docstring mandate).

## Acceptance Criteria
- [ ] **AC-1**: `.claude/agent-memory/code-reviewer/MEMORY.md` no longer contains any reference to a "50-line" function length limit. Specifically: the "Function Length Violations (50-line rule)" subsection (lines 42-45) is deleted, and the "Function length limit: 50 lines" bullet (line 64) is deleted.
- [ ] **AC-2**: `.claude/rules/python-style.md` line 13 is changed from "Use dataclasses or Pydantic models for structured data, not plain dicts" to language that prefers dataclasses/Pydantic for data passed across module boundaries while permitting plain dicts for local/transient use.
- [ ] **AC-3**: `.claude/rules/python-style.md` line 21 (duplicate "Prefer dataclasses or Pydantic models for structured data") is deleted.
- [ ] **AC-4**: `.claude/rules/python-style.md` line 12 print() rule is updated to: "Use `logging` module for operational output. For CLI user-facing output, `typer.echo()`, `rich.Console.print()`, or `print()` are acceptable." (SE authority: `rich.Console.print()` is the established CLI pattern in `src/cli/`.)
- [ ] **AC-5**: `.claude/rules/python-style.md` line 8 type hints rule is updated to: "Use type hints for all function parameters and return types in `src/`. In tests, use hints for complex helpers and fixtures with non-obvious signatures; simple test functions and `-> None` fixtures do not require annotation." (SE authority: matches actual test patterns.)
- [ ] **AC-6**: `.claude/rules/python-style.md` line 20 docstring rule is softened to apply when purpose or behavior is not obvious from the function signature. Functions where the non-obvious part is side effects, error handling, or a non-trivial precondition are the priority cases. (SE authority: self-evident name + signature needs no docstring.)
- [ ] **AC-7**: No orphaned references to deleted content remain in the CR memory file.
- [ ] **AC-8**: The modified python-style.md does not contradict CLAUDE.md's core principle ("A dict is better than a class -- until it isn't").
