# E-163-01: Scouting Spray Chart Crawler

## Epic
[E-163: Full-Season Opponent Spray Charts and Dashboard Display](epic.md)

## Status
`DONE`

## Description
After this story is complete, the scouting pipeline will fetch player-stats (spray chart) JSON for every completed game on an opponent's schedule, writing one file per game to the scouting spray directory. This extends the scouting pipeline to cover spray chart data for opponents, using the opponent's `gc_uuid` and the game's `event_id`.

## Context
The existing `SprayChartCrawler` only processes member teams. The scouting pipeline fetches schedule, roster, and boxscores but not player-stats. This story adds the spray chart crawl step to the scouting pipeline. The `event_id` for the API call equals the `id` field in the opponent's `games.json` (confirmed: same as `games.game_id` in DB).

## Acceptance Criteria
- [ ] **AC-1**: Given an opponent with `gc_uuid` and cached `games.json` containing completed games, when the scouting spray crawler runs, then `GET /teams/{gc_uuid}/schedule/events/{event_id}/player-stats` is called for each completed game and the raw JSON is written to `data/raw/{season_id}/scouting/{public_id}/spray/{event_id}.json`, per TN-1.
- [ ] **AC-2**: Given an opponent without `gc_uuid` (NULL in teams table), the crawler logs an INFO message and skips that opponent without error.
- [ ] **AC-3**: Given a spray file already exists for a game (`event_id`), the crawler skips that game (existence-only idempotency, per TN-1).
- [ ] **AC-4**: `CredentialExpiredError` propagates immediately (abort the run). Other `GameChangerAPIError` exceptions are caught, logged, and counted (per-game error tolerance).
- [ ] **AC-5**: The crawler is integrated into the `bb data scout` CLI flow (runs after scouting crawl+load, per TN-9) and is independently callable via `bb data crawl --crawler scouting-spray`.
- [ ] **AC-6**: Rate limiting: minimum 1-second delay between requests with jitter, per HTTP discipline rules.

## Technical Approach
Create a `ScoutingSprayChartCrawler` class (or extend the scouting crawler) that reads the already-cached `games.json` from the scouting directory, resolves the opponent's `gc_uuid` from the DB, and fetches player-stats for each completed game. The file layout follows TN-8.

The crawler needs: opponent's `public_id` (for directory path), opponent's `gc_uuid` (for API call), and `games.json` content (for event_ids). The `gc_uuid` lookup is: `SELECT gc_uuid FROM teams WHERE public_id = ?` (`teams.public_id` is UNIQUE â€” do not use `opponent_links.public_id` which is not unique).

Reference existing patterns in `src/gamechanger/crawlers/spray_chart.py` (member crawler) and `src/gamechanger/crawlers/scouting.py` (scouting crawler).
