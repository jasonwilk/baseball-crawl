# E-018-R-01: Define the Frontmatter Standard

## Epic
[E-018: Agent Frontmatter Refinement](./epic.md)

## Status
`DONE`

## Description
After this spike is complete, the project will have a documented frontmatter standard that defines what "complete and correct" YAML frontmatter looks like for every agent in this project. The standard will be recorded as a decision artifact that the four implementation stories (E-018-01 through E-018-04) can reference as their specification.

## Context
Four agent configuration files have incomplete frontmatter -- bloated multi-paragraph descriptions with embedded routing examples, missing `tools` restrictions, and no use of available frontmatter capabilities. Before we can fix them, the claude-architect needs to define what "good" looks like. This spike produces that definition.

The current state: the orchestrator has a clean one-sentence description and explicit `tools` list. The product-manager has a clean description but no `tools` field. The four target agents (api-scout, baseball-coach, general-dev, data-engineer) all have multi-paragraph descriptions with 5 routing examples crammed into the YAML string, no `tools` field, and no `skills` preloading.

## Acceptance Criteria
- [ ] **AC-1**: A decision artifact exists at `/.project/research/E-018-frontmatter-standard.md` that defines the canonical frontmatter standard for this project
- [ ] **AC-2**: The standard specifies what the `description` field should contain (length, style, content) and explicitly states that multi-paragraph routing examples do not belong in the YAML description
- [ ] **AC-3**: The standard specifies recommended `tools` allowlists for each of the four target agents (api-scout, baseball-coach, general-dev, data-engineer), with rationale for each restriction or explicit rationale for inheriting all tools
- [ ] **AC-4**: The standard addresses whether `skills` should be preloaded via frontmatter or continue with on-demand loading via system prompt instructions, with rationale
- [ ] **AC-5**: The standard addresses whether `maxTurns`, `permissionMode`, `hooks`, `isolation`, or `background` should be set for any of the four target agents, with rationale for each decision (even if the decision is "not needed")
- [ ] **AC-6**: The standard addresses what should happen to the routing examples currently embedded in the description fields -- relocate to system prompt body, remove entirely, or other
- [ ] **AC-7**: The decision artifact is self-contained -- an implementing agent reading only the artifact and the target agent files can execute stories E-018-01 through E-018-04 without ambiguity
