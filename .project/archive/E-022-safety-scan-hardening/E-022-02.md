# E-022-02: Harden PreToolUse Hook Intercept Pattern

## Epic
[E-022: Safety Scan Hardening](./epic.md)

## Status
`DONE`

## Description
After this story is complete, the Claude Code PreToolUse hook at `.claude/hooks/pii-check.sh` reliably intercepts `git commit` commands regardless of their position in a shell command string -- whether they appear at the start, after `&&`, after `||`, after `;`, or after `|`. The current pattern only matches `git commit` at the start of the command, missing common multi-command patterns that agents use.

## Context
The current intercept pattern in `pii-check.sh` is:

```bash
if ! echo "$COMMAND" | grep -qE '^git\s+commit'; then
```

This is anchored to the start of the command string (`^`). When an agent runs `git add . && git commit -m "msg"`, the full command string is sent to the hook as a single value. The `git commit` portion is not at the start, so the hook exits 0 without scanning. This is a real gap -- agents routinely chain `git add` and `git commit` together.

Note: the Git pre-commit hook (Layer 1) still fires in these cases because Git always runs its hooks regardless of how the commit command was invoked. So this is a defense-in-depth fix for Layer 2, not a critical vulnerability. But the two-layer design exists precisely so that both layers work independently.

This story has no dependencies and can execute in parallel with E-022-01 and E-022-03.

## Acceptance Criteria
- [ ] **AC-1**: Given a Bash tool call with command `git commit -m "test"`, the hook intercepts and runs the scanner (existing behavior preserved)
- [ ] **AC-2**: Given a Bash tool call with command `git add . && git commit -m "test"`, the hook intercepts and runs the scanner
- [ ] **AC-3**: Given a Bash tool call with command `cd /path && git commit -m "test"`, the hook intercepts and runs the scanner
- [ ] **AC-4**: Given a Bash tool call with command `git commit -m "test"; git push`, the hook intercepts and runs the scanner
- [ ] **AC-5**: Given a Bash tool call with command `echo done || git commit -m "test"`, the hook intercepts and runs the scanner
- [ ] **AC-6**: Given a Bash tool call with command `ls -la`, the hook does NOT intercept (non-commit commands pass through unchanged)
- [ ] **AC-7**: Given a Bash tool call with command `echo "git commit is a command"`, the hook does NOT intercept (git commit in a quoted string argument to a non-git command should not trigger -- best effort, not guaranteed)
- [ ] **AC-8**: The hook script remains under 60 lines and adds no new dependencies

## Technical Approach
Replace the grep pattern in `.claude/hooks/pii-check.sh`:

**Current:**
```bash
if ! echo "$COMMAND" | grep -qE '^git\s+commit'; then
```

**New:**
```bash
if ! echo "$COMMAND" | grep -qE '(^|[;&|]\s*)git\s+commit'; then
```

The new pattern matches `git commit` when preceded by:
- Start of string (`^`) -- standalone command
- `;` -- sequential chaining
- `&` -- covers both `&&` (logical AND) and `&` (background)
- `|` -- covers both `||` (logical OR) and `|` (pipe)

The `\s*` after the character class allows optional whitespace between the operator and `git`.

**Why this is sufficient**: Claude Code sends the full Bash command as a single string in `tool_input.command`. The most common multi-command patterns are:
- `cmd1 && git commit` -- matched by `&` in the class
- `cmd1; git commit` -- matched by `;`
- `cmd1 || git commit` -- matched by `|`

**Edge case: quoted strings**: The pattern could false-positive on `echo "git commit"` -- but this is harmless. The scanner will run on staged files and find nothing, adding negligible overhead. The alternative (parsing shell quoting) is far too complex for a safety hook. AC-7 is best-effort.

**Testing**: This change is in a shell script and cannot be unit-tested in pytest. Verification is manual:
1. In a Claude Code session, create a test scenario where the command is `echo "test" && git commit --allow-empty -m "hook test"`
2. Verify the `statusMessage` "Checking staged files for PII..." appears (confirming the hook fired)

## Notes
- This is a one-line change to a grep pattern. Keep the PR focused.
- The false-positive risk (matching `git commit` inside quoted strings) is acceptable. The scanner runs quickly on clean files and the cost is a few hundred milliseconds of unnecessary scanning. The alternative (shell-aware parsing) would make the hook fragile and complex.
- The Git pre-commit hook (Layer 1) is not affected by this change -- it fires on all commits regardless of how they were invoked. This fix ensures Layer 2 (agent defense) matches Layer 1's coverage.
