# E-022-01: Add Success Confirmation to PII Scanner

## Epic
[E-022: Safety Scan Hardening](./epic.md)

## Status
`DONE`

## Description
After this story is complete, the PII scanner prints a visible confirmation line on clean scans (e.g., `[pii-scan] Scanned 5 files, 0 violations.`) and the git pre-commit hook prints a confirmation banner (`[pii-hook] PII scan passed.`) on successful commits. This provides evidence that the safety scan actually ran, replacing the current silent-success behavior that leaves no audit trail.

## Context
The scanner at `src/safety/pii_scanner.py` currently exits 0 with no output on clean scans. Both hooks (git pre-commit and Claude Code PreToolUse) call this scanner, so the scanner's output is the single point where confirmation can be added. The git pre-commit hook adds its own wrapper message around the scanner output.

The current behavior:
- Clean scan: exit 0, no output at all
- Violation: exit 1, stderr shows `[PII BLOCKED] ...` lines plus a summary

The desired behavior:
- Clean scan: exit 0, stderr shows `[pii-scan] Scanned N files, 0 violations.`
- Violation: exit 1, stderr shows existing `[PII BLOCKED] ...` lines (unchanged)

This change is purely additive -- no existing behavior changes, only a new success output line.

This story has no dependencies and can execute in parallel with E-022-02 and E-022-03.

## Acceptance Criteria
- [ ] **AC-1**: Given staged files with no PII patterns, when the scanner runs via `python3 src/safety/pii_scanner.py --staged`, then stderr contains a line matching `[pii-scan] Scanned N files, 0 violations.` where N is the count of files that were actually scanned (not skipped)

- [ ] **AC-6**: The scanner's success confirmation format is `[pii-scan] Scanned {n} file(s), 0 violations.` printed to stderr (consistent with violation output going to stderr)

## Technical Approach

### Scanner Changes (`src/safety/pii_scanner.py`)
Modify the `main()` function. After scanning all files and finding zero violations, print the confirmation line to stderr before returning 0. Count only files that were actually scanned (not skipped due to extension, skip path, or synthetic marker).

The count should reflect files that were opened and scanned -- not the total input count. Add a scanned-file counter to `scan_files()` or return additional metadata. Suggested approach: modify `scan_files()` to return a tuple `(violations, scanned_count)` or add a wrapper. Alternatively, add a counter in `main()` that counts files where `is_scannable()` and `not should_skip_path()` both pass.

The simplest approach: add a `scanned_count` parameter to the return path. Since `scan_file()` returns an empty list for both "skipped" and "clean," the caller cannot distinguish them. Options:
1. Have `scan_file()` return `None` for skipped files and `[]` for clean files, then count non-None results.
2. Add a separate `count_scannable()` function.
3. Track the count in `main()` by checking the same skip conditions before calling `scan_file()`.

Option 1 is the cleanest -- it lets `scan_files()` naturally report how many files it actually examined. But it changes the return type. Option 3 duplicates logic. **Recommended: Option 1**, changing `scan_file()` to return `None` when skipped and `[]` when clean.

However, this changes the public API that tests import. To minimize disruption: keep `scan_file()` returning `list[Violation]` as-is. Instead, add the confirmation line in `main()` by computing the scanned count separately. Use a simple helper:

```python
def _count_scannable(file_paths: list[str]) -> int:
    """Count files that will be scanned (not skipped)."""
    return sum(
        1 for f in file_paths
        if not should_skip_path(f.strip()) and is_scannable(f.strip())
    )
```

Then in `main()`, after `violations = scan_files(file_paths)`:
```python
if not violations:
    scanned = _count_scannable(file_paths)
    if scanned > 0:
        print(
            f"[pii-scan] Scanned {scanned} file(s), 0 violations.",
            file=sys.stderr,
        )
```

Note: `_count_scannable` does not check for file existence or synthetic markers (those require reading the file). The count is "files that were candidates for scanning" -- close enough for a confirmation message. The alternative (reading files twice) is wasteful.

### Git Pre-Commit Hook Changes (`.githooks/pre-commit`)
Add a success message after the scanner exits 0. After the `exit 0` at the end:

```bash
# Replace the final exit 0 with:
echo "[pii-hook] PII scan passed."
exit 0
```

### Test Changes (`tests/test_pii_scanner.py`)
Add tests for the confirmation output. Test the `main()` function by calling it and capturing stderr (use `capsys` or redirect). Tests to add:

1. **Clean scan confirmation**: Run `main()` with args pointing to a clean file. Verify stderr contains the `[pii-scan] Scanned` line.
2. **Empty file list**: Run `main()` with `--staged` in a repo with no staged files (mock `get_staged_files` to return `[]`). Verify no confirmation line.
3. **Violations suppress confirmation**: Run `main()` with args pointing to a file with PII. Verify stderr does NOT contain `[pii-scan] Scanned`.

Note: Testing `main()` requires careful handling since it uses `sys.argv`. Use `monkeypatch` to set `sys.argv` and provide file paths via positional args mode.

## Notes
- The confirmation goes to stderr, not stdout, consistent with the existing violation output. This ensures it appears in terminal output for both hooks.
- The `_count_scannable()` helper slightly over-counts because it does not check for file existence or synthetic markers. This is acceptable -- the purpose is confirmation that the scan ran, not a precise accounting.
- Do not change the scanner's exit code behavior. Exit 0 still means clean, exit 1 still means violations found.
- The PreToolUse hook (`.claude/hooks/pii-check.sh`) does NOT need changes for this story. The scanner's stderr output flows through the git pre-commit hook to the terminal. The PreToolUse hook runs the scanner separately (via `--staged`) but its output goes to the agent via the hook mechanism.
