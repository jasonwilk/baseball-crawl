# E-071-02: Integrate review loop into dispatch workflow

## Epic
[E-071: Code Review Agent](epic.md)

## Status
`DONE`

## Description
After this story is complete, the dispatch workflow will automatically route every implementer's completed work through the code-reviewer agent before marking stories DONE. The implement skill spawns a persistent code-reviewer at team creation, Phase 3 Step 5 sends completed work to the reviewer instead of rubber-stamping ACs, and a review loop operates until the reviewer approves or the circuit breaker triggers. dispatch-pattern.md documents the review step, and CLAUDE.md lists code-reviewer in the Agent Ecosystem.

## Context
E-071-01 creates the code-reviewer agent definition. This story integrates it into the dispatch workflow so the review happens automatically during every epic dispatch. The current flow (implementer completes -> main session rubber-stamps ACs -> DONE) is replaced with (implementer completes -> reviewer examines -> findings loop -> reviewer approves -> DONE). See epic Technical Notes for the full CA assessment on flow insertion, persistence model, and circuit breaker.

When an implementer reports completion (with `## Files Changed`), the main session checks whether the story modifies ONLY context-layer files (`.claude/agents/`, `.claude/rules/`, `.claude/skills/`, `.claude/hooks/`, `.claude/settings.json`, `.claude/settings.local.json`, `.claude/agent-memory/`, `CLAUDE.md`) and no Python code. If context-layer-only: the main session verifies ACs directly and marks DONE (reviewer is skipped). Otherwise: the main session sends the work to the code-reviewer using the review assignment template:
  ```

  Story file: /absolute/path/to/E-NNN-SS.md
  [Full story file text]

  Epic Technical Notes:
  [Full Technical Notes]

  Implementer-reported files changed:
  [Files Changed section from implementer's completion message]

  Review round: 1 of 2 (circuit breaker)

  Review this story's implementation against all acceptance criteria and the review rubric. The implementer's Files Changed list is the primary scope. Cross-reference it against the story's "Files to Create or Modify" section to flag any missing or unexpected files (divergence is a SHOULD FIX finding -- implementers may legitimately touch unlisted files, but it should be called out). Note: `git diff --name-only` is repo-wide and may include changes from parallel stories or untracked files -- use it as advisory context, not as the authoritative scope for this story. Report findings using the structured format.
  ```
- [ ] **AC-5**: When the reviewer returns APPROVED (no MUST FIX findings), the main session marks the story DONE. Any SHOULD FIX findings from the reviewer are recorded in the epic's History section during closure -- they are not relayed to the implementer.
- [ ] **AC-6**: When the reviewer returns NOT APPROVED (MUST FIX findings only -- SHOULD FIX items are not included in the feedback to implementers), the main session routes the MUST FIX findings to the implementer with the review round number (e.g., "Round 1 of 2 -- MUST FIX items below"). The implementer fixes and reports completion again (with updated `## Files Changed`). The main session sends the updated work back to the reviewer for Round 2. Round 2 is a full re-review of all changed files, but the reviewer focuses on unresolved Round 1 items and any new issues introduced by the fixes.
- [ ] **AC-7**: The circuit breaker is documented: max 2 review rounds per story. If the 2nd review still has MUST FIX findings, the main session escalates to the user with the findings summary and presents options: (a) fix it themselves, (b) tell implementer to try again (resets the circuit breaker), (c) override the reviewer and mark DONE (explicit user override -- the user assumes responsibility for unresolved findings), or (d) abandon the story. The main session does not mark the story DONE and does not loop further without user direction. Note: option (c) is a user override, not an autonomous main-session action -- AC-9's anti-pattern governs the main session's behavior absent user direction.

## Notes
- The reviewer is automatically spawned -- it is infrastructure, not opt-in. This is what makes it a structural guardrail rather than a suggestion.
- The existing Phase 4 (optional codex review via "and review" modifier) is NOT changed. It remains as Layer 2 defense in depth. The two layers serve different purposes: per-story review (Layer 1, automatic) vs. whole-epic review with external tool (Layer 2, user-triggered).
- The main session's role shifts from "verify ACs" to "route to reviewer and manage the loop." The main session still does all status updates (IN_PROGRESS, DONE) and the closure sequence.
- When the circuit breaker fires (2nd review still has MUST FIX), the main session presents findings to the user and waits for direction. Options the user might choose: fix it themselves, tell implementer to try again (resets the circuit breaker), override the reviewer and mark DONE (user assumes responsibility), or abandon the story. The "override" option is an explicit user action -- the main session never autonomously marks DONE with unresolved MUST FIX findings.
- SHOULD FIX items are never relayed to implementers. The fix loop is clean: only MUST FIX items flow between reviewer -> main session -> implementer. SHOULD FIX items are preserved in epic History during closure for future reference and pattern detection by Layer 2 codex review.
- Context-layer-only stories bypass the reviewer entirely. The main session verifies ACs directly for these stories. This avoids wasting reviewer context on non-code changes.
- Round 2 re-review scope: full re-review of all changed files, but focus on unresolved Round 1 MUST FIX items plus any new issues introduced by the fixes. SHOULD FIX findings cannot be escalated to MUST FIX between rounds unless the implementer's fix attempt introduces new evidence (e.g., a fix that creates a new convention violation).
- Closure-time memory updates: PM MEMORY.md agent ecosystem count ("8 agents" -> "9 agents") and CA MEMORY.md agent count are updated during the dispatch closure sequence, not by the implementing agent. These are standard closure-sequence responsibilities.
