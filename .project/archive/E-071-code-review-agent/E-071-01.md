# E-071-01: Create code-reviewer agent definition

## Epic
[E-071: Code Review Agent](epic.md)

## Status
`DONE`

## Description
After this story is complete, a new code-reviewer agent definition will exist at `.claude/agents/code-reviewer.md` with an embedded review rubric, structured findings format, adversarial review stance, and tool restrictions (Read/Glob/Grep/Bash only -- no Write/Edit). The agent is sonnet-class with a focused mandate: find issues in implementer work before stories are marked DONE.

## Context
The current dispatch workflow has no dedicated quality gate between implementer completion and DONE status. The main session's AC verification is a rubber stamp. E-064 proved this -- 5 issues survived past DONE into codex/manual review. This agent fills that gap. See epic Technical Notes for the full CA assessment including the rubric priorities, structured findings format, and circuit breaker design.

## Acceptance Criteria

### Agent definition file
- [ ] **AC-1**: `.claude/agents/code-reviewer.md` exists with valid YAML frontmatter containing: `name: code-reviewer`, `model: sonnet`, a descriptive `description` field, `color: magenta` (distinct from existing agents -- `magenta` must be added to the valid color list in claude-architect.md's agent-format standard), `memory: project`.
- [ ] **AC-2**: The `tools` field in frontmatter lists ONLY: `Read`, `Glob`, `Grep`, `Bash`. No `Write`, no `Edit`, no `WebFetch`. The reviewer finds issues; it does not fix them.

### Identity and stance
- [ ] **AC-3**: The agent definition establishes an adversarial/skeptical review stance. The reviewer's job is to find what's wrong, not to confirm what's right. The definition explicitly contrasts this with the implementer's constructive stance.
- [ ] **AC-4**: The definition states that the reviewer verifies BOTH code quality AND acceptance criteria satisfaction. It is the quality gate, not a style checker.

### Embedded rubric
- [ ] **AC-5**: The agent definition contains an embedded review rubric with 7 priority levels (in order): (1) AC verification, (2) bugs and regressions, (3) missing/inadequate tests, (4) credential and security risks, (5) schema drift, (6) convention violations (referencing CLAUDE.md, python-style.md, testing.md), (7) planning/implementation mismatch. The rubric includes a MUST FIX classification guardrail: any finding that violates a documented convention (CLAUDE.md, `.claude/rules/python-style.md`, `.claude/rules/testing.md`) is MUST FIX, not SHOULD FIX. SHOULD FIX is reserved for genuinely optional improvements not mandated by project rules. Convention-violation findings must be scoped to code written or modified in the current story -- do not flag pre-existing code.
- [ ] **AC-6**: The rubric instructs the reviewer to: (a) run `pytest` as the first step before reading changed files -- test failures are automatic MUST FIX findings (for large test suites, target changed modules first: `pytest tests/test_<module>.py`), (b) read CLAUDE.md, `.claude/rules/python-style.md`, `.claude/rules/testing.md` (baseline rules), plus any additional glob-triggered rules relevant to the story's modified files, (c) read the story file and epic Technical Notes. These are the reviewer's context sources.

### Structured findings format
- [ ] **AC-7**: The agent definition specifies the structured findings format with three sections: MUST FIX (blocks DONE), SHOULD FIX (recommended, does not block), and AC VERIFICATION (per-AC verdict with evidence). The output ends with an explicit APPROVED or NOT APPROVED verdict.
- [ ] **AC-8**: The format requires file:line citations for every finding.

### Circuit breaker
- [ ] **AC-9**: The agent definition documents the circuit breaker: max 2 review rounds per story. If the 2nd review still has MUST FIX findings, the reviewer reports this to the main session for escalation to the user. The reviewer does not loop indefinitely.

### Work authorization and context isolation
- [ ] **AC-10**: The agent definition includes a work authorization section: the reviewer only operates when assigned a review by the main session via SendMessage. It does not self-initiate reviews. Each review assignment is self-contained -- the reviewer does not reference previous reviews. It reads the story file and changed files fresh for every assignment.

### Anti-patterns
- [ ] **AC-11**: The agent definition includes anti-patterns: (1) never write or edit code -- find issues, don't fix them, (2) never mark stories DONE or update status files -- the main session handles statuses, (3) never approve work that has MUST FIX findings -- if MUST FIX items remain after 2 rounds, escalate to the main session for user override (the user may override, but the reviewer never approves), (4) never review without reading the story file and CLAUDE.md first, (5) never use Bash to modify files (no sed, awk, tee, redirect) -- Bash is for read-only commands: pytest, git diff, git log, (6) never escalate a SHOULD FIX to MUST FIX between rounds unless new evidence emerges from the implementer's fix attempt.

### Agent-format standard update
- [ ] **AC-12**: The valid color list in `.claude/agents/claude-architect.md` (agent-format standard section) is updated to include `magenta` (e.g., `blue|green|red|orange|yellow|cyan|purple|magenta`).

## Notes
- Color: `magenta`. Existing agent colors: yellow (claude-architect), green (PM), red (baseball-coach), orange (api-scout), blue (SE, data-engineer), purple (docs-writer), cyan (ux-designer). CA recommended `magenta` over `white` -- `white` may blend into light terminal backgrounds. `magenta` is ANSI-standard, visually distinct, and safe on all backgrounds. E-071-01 scope includes updating the color enum to add `magenta`.
- The reviewer is sonnet-class (not opus) because it does focused reads and pattern matching, not deep reasoning. This bounds cost.
- The "no Write/Edit" tool restriction is critical. If the reviewer could edit files, the separation of concerns collapses -- the reviewer becomes a second implementer.
- The "no file-modifying Bash" anti-pattern reinforces the tool restriction. The reviewer has Bash for `pytest`, `git diff`, `git log`, and similar read-only commands -- not for `sed`, `awk`, or redirects that would modify files.
