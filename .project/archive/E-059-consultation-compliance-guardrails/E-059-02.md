# E-059-02: Add Consultation Compliance Gate to workflow-discipline.md

## Epic
[E-059: Consultation Compliance Guardrails](epic.md)

## Status
`DONE`

## Description
After this story is complete, `workflow-discipline.md` will contain a new "Consultation Compliance Gate" section that prevents the PM from marking an epic READY when a user-directed consultation was requested but not performed. This provides defense-in-depth beyond the PM agent definition -- the rule exists in the shared rules layer where all agents can reference it, not just inside one agent's definition.

## Context
The PM agent definition already has a user-directed override rule, but when it was violated during E-058 formation, nothing else in the system caught it. `workflow-discipline.md` has gates for dispatch authorization and work authorization but no gate for consultation compliance. Adding one here creates a second enforcement point: even if the PM skips the pre-step in its own definition, the workflow-discipline rule should trigger when the PM attempts to set READY.

- [ ] **AC-1**: A new section titled "Consultation Compliance Gate" exists in `/workspaces/baseball-crawl/.claude/rules/workflow-discipline.md`, placed between "Dispatch Authorization Gate" and "Work Authorization Gate".
- [ ] **AC-2**: The section states that when the user explicitly requests PM collaboration with a specific agent during epic formation, the PM MUST invoke that agent before writing stories -- or if unable to spawn (spawning is one-level-deep), MUST escalate to the team lead/user with specific questions for the named agent. The PM MUST NOT set the epic to READY until the consultation is complete and its findings are incorporated.

- [ ] **AC-4**: The section does NOT apply to domain-triggered consultations from the PM's Consultation Triggers table (those are advisory). It applies only to explicit user directives.
- [ ] **AC-5**: The section explains the defense-in-depth rationale: this gate exists in workflow-discipline.md (loaded for all agents) in addition to the PM agent definition (loaded only for PM), so the team lead can also flag violations.
- [ ] **AC-6**: Existing sections in workflow-discipline.md are not modified or removed. Only the new section is added.
- [ ] **AC-7**: The file remains well-formed markdown with consistent heading levels relative to surrounding sections.

## Technical Approach
Read `/workspaces/baseball-crawl/.claude/rules/workflow-discipline.md` in full. Add the new section after "Dispatch Authorization Gate" (line 9) and before "Work Authorization Gate" (line 13). Use the same heading level (##) as the other gates.

Suggested content:

```
## Consultation Compliance Gate

When the user explicitly requests that PM collaborate with a specific agent during epic formation (e.g., "work with SE on this," "consult data-engineer before writing stories"), the PM MUST invoke that agent via Task tool and incorporate their input before writing stories. If the PM cannot spawn the requested agent (spawning is one-level-deep -- a platform constraint), the PM MUST escalate to the team lead/user with specific questions for the named agent. The PM MUST NOT substitute its own judgment for the requested expert's input. The PM MUST NOT skip the consultation because spawning is unavailable. The PM MUST NOT set the epic to READY until the requested consultation is complete.

This gate applies to explicit user directives only -- not to the domain-triggered consultations in the PM's Consultation Triggers table, which are advisory.

This gate exists in workflow-discipline.md (loaded for all agents) in addition to the PM agent definition (loaded only for PM), so the team lead can also flag violations. This is intentional defense-in-depth.
```
