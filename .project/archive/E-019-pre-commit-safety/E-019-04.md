# E-019-04: Developer Guide for Safe Data Handling

## Epic
[E-019: Pre-Commit Safety Gates](./epic.md)

## Status
`DONE`

## Description
After this story is complete, a practical guide exists at `docs/safe-data-handling.md` that any agent or developer can read to understand how to work safely with sensitive GameChanger data. It covers where to put API response files, what can and cannot be committed, how to install the pre-commit hook, what to do when the hook fires, and how to create test fixtures safely. CLAUDE.md is updated with the hook install command.

## Context
Technical controls (hooks, scanner, gitignore) are necessary but not sufficient. An agent that does not understand why the controls exist or what to do when they fire will be confused or work around them. This guide turns enforcement into understanding.

This guide is written AFTER the scanner (E-019-03) exists so it can reference real behavior and real output messages. It is the onboarding document for any new agent that will touch real data.

## Acceptance Criteria
- [ ] **AC-1**: The guide answers "Where do I put this JSON file from an API call?" in one paragraph or less, pointing to `/ephemeral/`
- [ ] **AC-2**: The guide answers "Can I commit this test fixture that contains a player's name?" with a specific yes-or-no-with-conditions answer and a reference to the `synthetic-test-data` convention
- [ ] **AC-3**: The guide explains how to install the pre-commit hook on a fresh clone with numbered steps that an agent can follow without additional context
- [ ] **AC-4**: The guide explains what happens when the hook fires: what the output looks like (includes the real `[PII BLOCKED]` message format), and what the developer should do next (fix, move to ephemeral, annotate as synthetic, or legitimately bypass)
- [ ] **AC-5**: The guide explains the bypass procedure (`git commit --no-verify`) with a prominent warning about when bypassing is and is not appropriate. Lists the two legitimate reasons (confirmed false positive, emergency hotfix) and the illegitimate ones ("I'm sure it's clean", "the hook is slow")
- [ ] **AC-6**: The guide contains a "Quick Reference" section: a short table or bulleted list covering the most common scenarios (store API response, commit a script, add a test fixture, share data with a teammate)
- [ ] **AC-7**: The guide notes that the Claude Code PreToolUse hook cannot be bypassed by an agent -- only a human can disable it by modifying `.claude/settings.json`
- [ ] **AC-8**: `CLAUDE.md` Commands section contains: `- \`./scripts/install-hooks.sh\` -- one-time setup for PII pre-commit hook (run after cloning)`

## Technical Approach
Write `docs/safe-data-handling.md` as a practical, scannable guide. Avoid walls of text. Use numbered lists for procedures, tables for quick reference, code blocks for commands.

Suggested structure:
```
# Safe Data Handling Guide

## The One Rule
## Where Sensitive Data Lives
## What Can Be Committed
## What Gets Scanned (pattern categories)
## Installing the Pre-Commit Hook
## When the Hook Fires
  ### Understanding the Output
  ### Fixing a Violation
  ### Legitimate Bypass
## Working with Test Fixtures
## Quick Reference
## Related Documents
```

"The One Rule" should be a single sentence, e.g.: "If a file was produced by a real API call or contains real credentials, it goes in `/ephemeral/`, never in a commit."

The "What Gets Scanned" section should briefly list the pattern categories (emails, phones, Bearer tokens, API keys) without duplicating the full pattern definitions -- point to `src/safety/pii_patterns.py` for the authoritative list.

Include the real scanner output format:
```
[PII BLOCKED] path/to/file.json:42: matched 'email' pattern
```
