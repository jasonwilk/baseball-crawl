# E-019-01: Ephemeral Data Directory and Gitignore Rules

## Epic
[E-019: Pre-Commit Safety Gates](./epic.md)

## Status
`DONE`

## Description
After this story is complete, the project has a `/ephemeral/` directory that is safe by default: gitignored so no data file placed in it can be accidentally committed, documented with a README explaining its purpose, and ready for any agent or developer to use as a landing zone for raw API responses and scratch data.

## Context
During API exploration and ETL development, agents pull raw GameChanger API responses to disk for inspection. These files contain PII and credentials. Without a designated safe zone, they end up scattered across the project tree where they risk being staged. The `/ephemeral/` directory is the canonical safe zone -- never committed, never permanent, no expectations of longevity.

This directory is NOT the same as `data/` (which holds processed pipeline output). Ephemeral means: raw exploration data that never enters version control.

This story has no dependencies and can execute immediately, in parallel with E-019-02.

## Acceptance Criteria
- [ ] **AC-1**: A directory `/ephemeral/` exists at the repository root with a `scratch/` subdirectory
- [ ] **AC-2**: The root `.gitignore` contains entries that cause Git to ignore all files under `/ephemeral/` except `README.md` and `.gitkeep` files
- [ ] **AC-3**: Given a file written to `/ephemeral/scratch/test.json`, when `git status` is run, that file does not appear as untracked
- [ ] **AC-4**: Given `/ephemeral/README.md`, it explains: (a) the directory's purpose as a safe zone for sensitive data, (b) that contents are never committed, (c) that agents should create epic-specific subdirectories as needed (e.g., `/ephemeral/E-001/`), and (d) what belongs here vs. what does not
- [ ] **AC-5**: `/ephemeral/scratch/.gitkeep` exists so the directory structure is preserved in the repository

## Technical Approach
1. Create `/ephemeral/` and `/ephemeral/scratch/`
2. Add `/ephemeral/scratch/.gitkeep` (empty file, committed)
3. Write `/ephemeral/README.md` with content per AC-4
4. Add gitignore rules to root `.gitignore`. The pattern must ignore everything under `/ephemeral/` except `README.md` and `.gitkeep` files. Recommended approach:
   ```
   # Ephemeral data directory -- safe zone for raw API responses and scratch data
   /ephemeral/*
   !/ephemeral/README.md
   !/ephemeral/scratch/
   /ephemeral/scratch/*
   !/ephemeral/scratch/.gitkeep
   ```
   The implementing agent should verify with `git check-ignore -v ephemeral/scratch/test.json` that the ignore rule works, and with `git ls-files ephemeral/` that README.md and .gitkeep are tracked.
5. Do NOT pre-create per-epic subdirectories (E-001/, E-002/, etc.). Agents create these as needed. The README explains the convention.

### README Content Guidance
- **Purpose**: Safe landing zone for any data file that contains or might contain real data from the GameChanger API. Gitignored. Never committed.
- **What belongs here**: Raw API responses, intermediate JSON/CSV from exploration, any real-data file from an API call.
- **What does NOT belong here**: Test fixtures with synthetic data (those go in `tests/` with the `synthetic-test-data` marker), processed output for the database, source code.
- **Convention**: Create a subdirectory named after your current epic (e.g., `/ephemeral/E-005/`). Use `/ephemeral/scratch/` for one-off experiments not tied to an epic.
- **Warning**: Files here are never backed up or versioned. To share an exploration artifact, describe the API call that produced it, not the file itself.

## Notes
- Keep the gitignore pattern simple. If the first approach does not satisfy both AC-3 and AC-5, try an alternative and document which pattern was chosen in the commit message.
- The README is the only documentation this story produces. Keep it short and practical -- an agent should be able to read it in 30 seconds and know exactly what to do.
