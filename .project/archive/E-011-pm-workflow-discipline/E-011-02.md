## Description
After this story is complete, a Python script at `scripts/audit_epic_board.py` will read all active epics under `/epics/` and verify that story file statuses match the Stories table entries in each `epic.md`. The script detects mismatches, orphaned story files (files with no table row), dangling rows (table rows with no file), and epics where all stories are DONE but the epic itself is not marked COMPLETED. It outputs a structured text report to stdout and exits with code 0 if clean, 1 if issues found. This script is the enforcement mechanism for the PM's atomic status update protocol.

## Context
The E-009 failure mode was silent drift: story files reached DONE status while their epic table rows still showed IN_PROGRESS, or story files were written without a corresponding table entry. No existing tool could detect this. The PM discovered the drift only when reviewing files manually.

The audit script makes drift visible on demand. The PM runs it as a mandatory step before dispatching stories (Dispatch Mode step 3) and after completing any story (completing-a-story protocol step 4). The script does not fix inconsistencies -- it reports them. The PM resolves them manually. This keeps the tool simple and avoids automated changes to tracked markdown files.

The script must use Python stdlib only (no pip dependencies). Python is the project's chosen end-to-end language per the E-009 tech stack decision (FINAL, 2026-02-28). ## Technical Approach

### Parsing Story File Status
The status field in a story file appears as:
```
## Status
`TODO`
```
Parse: find the line matching `^## Status$`, take the next non-empty line, extract the backtick-quoted value. Handle `DONE`, `TODO`, `IN_PROGRESS`, `BLOCKED`, `ABANDONED` as valid statuses.

For research spikes (`E-NNN-R-NN.md`), apply the same parsing logic.

### Parsing the Stories Table
The Stories table in `epic.md` appears after the `## Stories` heading. It is a markdown table with columns: `ID | Title | Status | Dependencies | Assignee`. Parse rows by splitting on `|`, stripping whitespace. Skip the header row and separator row. Extract the `ID` and `Status` columns.

Story IDs in the table match filenames: `E-011-02` -> `E-011-02.md`. Research spike IDs: `E-011-R-01` -> `E-011-R-01.md`.

### Matching Files to Table Rows
For each epic directory:
1. Collect all `.md` files except `epic.md` -> this is the set of story/spike files
2. Parse the Stories table from `epic.md` -> this is the set of table rows
3. For each table row: look for a corresponding file by ID. If not found: dangling row.
4. For each story file: look for a corresponding table row by ID. If not found: orphaned file.
5. For matched pairs: compare statuses. If different: mismatch.

### Epic Status Check
After checking all stories in an epic:
- If every table row has status `DONE`, check the epic's own `## Status` field (same parse pattern as story files). If the epic is not `COMPLETED`, flag it.

### Output Format
```
=== Epic Board Audit ===
Checking 8 epics...

E-001-gamechanger-api-foundation
  [MISMATCH] E-001-02: file=DONE, table=IN_PROGRESS
  [ORPHAN]   E-001-99.md has no row in Stories table

E-011-pm-workflow-discipline
  OK

--- Summary ---
Epics checked: 8
Issues found: 2
  Mismatches: 1
  Orphaned files: 1
  Dangling rows: 0
  Epic status flags: 0
```

When clean:
```
=== Epic Board Audit ===
Checking 8 epics...

All 8 epics are consistent.
Exit code: 0
```

### Script Structure
Use a `main()` function with `argparse` for the optional `--epics-dir` argument. Keep all logic in well-named functions: `parse_story_status(file_path)`, `parse_stories_table(epic_md_path)`, `parse_epic_status(epic_md_path)`, `audit_epic(epic_dir)`, `main()`.
