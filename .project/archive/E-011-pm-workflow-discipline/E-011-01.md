## Description
After this story is complete, a new rules file at `/.claude/rules/pm-workflow-standards.md` will contain the complete, authoritative atomic status update protocol for the project-manager agent. The protocol specifies the exact ordered checklist of updates the PM must make at the end of every action type: creating a story, completing a story, completing a research spike, and dispatching stories in Dispatch Mode. Any agent or human reading this file will know precisely what the PM must update, and when, with no ambiguity.

## Context
E-009 exposed a recurring PM failure mode: status fields in individual story files, the Stories table in `epic.md`, and MEMORY.md drift out of sync during sessions where multiple files are written or statuses change. The rules scattered across `project-management.md` (one line: "keep the story table in sync"), `workflow-discipline.md` (covers dispatch gate only), and MEMORY.md (carries summary notes) are insufficient -- they describe intent but not the specific ordered steps.

The canonical document this story creates is the fix. It consolidates all PM workflow rules into a single enforceable location. The document will be auto-loaded in all agent contexts via the `paths: "**"` frontmatter selector, making it available wherever the PM operates.

The decisions recorded in E-011-R-01 determine the content of this document exactly. The implementing agent should read E-011-R-01 before writing any content.

## Acceptance Criteria

- [ ] **AC-1**: A file exists at `/.claude/rules/pm-workflow-standards.md`.
- [ ] **AC-2**: The file begins with YAML frontmatter `paths: ["**"]` so it is auto-loaded in all agent contexts.
- [ ] **AC-3**: The document contains a section titled "Atomic Status Update Protocol" with four named sub-sections corresponding to the four action types: "Creating a new story file", "Completing a story", "Completing a research spike", and "Dispatch Mode pre-dispatch".
- [ ] **AC-4**: Each sub-section contains a numbered checklist where every step is specific enough that a PM agent executing it cannot misinterpret the step. Each step names the exact file(s) to update and the exact field to change. No step is vague (e.g., "update statuses" is not acceptable; "update the `Status` field in the story file to `DONE`" is acceptable).
- [ ] **AC-5**: The "Completing a story" checklist includes a step that says to run `scripts/audit_epic_board.py` to verify board consistency after updating statuses.
- [ ] **AC-6**: The "Dispatch Mode pre-dispatch" checklist includes a step that says to run `scripts/audit_epic_board.py` before identifying eligible stories.
- [ ] **AC-7**: The document contains a section titled "Board State Invariants" that lists the following three invariants as rules the PM must never violate:
  - The `Status` field in every story file must match the status shown for that story in the parent epic's Stories table.
  - Every story file in an epic directory must have a corresponding row in the parent epic's Stories table.
  - The PM MEMORY.md epic summaries must not contradict the epic file statuses on disk.
- [ ] **AC-8**: Given a new agent encounters only this document and no other context, it can determine the complete set of files it must update in response to any of the four action types without asking a clarifying question.

## Technical Approach

- YAML frontmatter at the top with `paths: ["**"]`
- Markdown body with named sections
- Numbered checklists for each action type (not bulleted -- ordered steps must be numbered)
- Brief rationale note at the top explaining why this document exists (one paragraph pointing to E-009 as the motivating incident)

Read E-011-R-01 (at `/Users/jason/Documents/code/baseball-crawl/epics/E-011-pm-workflow-discipline/E-011-R-01.md`) for the exact protocol content. Do not improvise or simplify the checklists -- use the steps as specified in E-011-R-01, Q2 section.

The document should be concise -- target 80-120 lines. It is a reference document, not a tutorial. Each checklist step should fit in 1-2 lines.
