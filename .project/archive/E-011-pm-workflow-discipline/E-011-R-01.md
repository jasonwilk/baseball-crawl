# E-011-R-01: Architect Consultation -- PM Workflow Standards Design

## Epic
[E-011: PM Workflow Discipline and Status Update Standards](epic.md)

## Status
`DONE`

## Purpose
This spike records the design decisions required before E-011 story files can be written with testable acceptance criteria. Four questions were posed in the epic's Technical Notes. Answers are recorded here and incorporated into Technical Notes.

## Context
E-011 was created after E-009 exposed a recurring pattern: story and epic statuses drift out of sync during PM sessions. The fix is a canonical PM workflow standards document plus a verification script. Before writing stories, the PM needed answers to four architectural questions. This spike delivers those answers.

This is a PM-internal consultation. The epic is entirely in the workflow/process domain -- no agent infrastructure changes, no schema changes, no code architecture. The PM has sufficient context from existing conventions (E-007, E-006) to resolve all four questions without a separate Task tool delegation. Answers are grounded in established project patterns.

## Decisions

### Q1: Canonical Document Location

**Decision: `/.claude/rules/pm-workflow-standards.md`**

Rationale:
- Rules files with `paths: "**"` are auto-loaded in all agent contexts. This makes the standard enforceable anywhere, not just when a human thinks to look it up.
- The agent definition (`/.claude/agents/project-manager.md`) is already 487 lines. Embedding a full protocol checklist inline would make it unwieldy and harder to update independently.
- `/docs/` is not agent-loaded -- it is appropriate for human reference (API specs) but not for enforced agent behavior.
- `/.project/standards/` would be a new directory with no existing convention. Prefer established patterns.
- Precedent: E-007 created `/.claude/rules/workflow-discipline.md` for cross-agent workflow rules. PM-specific rules follow the same pattern.
- The rules file will use `paths: "**"` so it loads in all contexts (matching `workflow-discipline.md`).

### Q2: Atomic Status Update Protocol

**Decision: Adopt the PM's proposal from Technical Notes verbatim.**

The protocol is well-specified and covers all action types. No changes from the proposal. The full protocol will be the body of the canonical document created in E-011-01.

Summary of protocol action types and their checklists:

**A. Creating a new story file (4 steps)**
1. Write the story file with `Status: TODO` (or `BLOCKED` if deps unresolved)
2. Immediately add the story row to the parent epic's Stories table
3. Set status to `TODO` if all deps DONE, `BLOCKED` if any dep outstanding -- in both file and table
4. Update MEMORY.md if the new story changes the epic summary

**B. Completing a story (5 steps)**
1. Update story file `Status` to `DONE`
2. Update the corresponding row in the parent epic's Stories table to `DONE`
3. Check whether any BLOCKED stories are now unblocked; update their status in file and table
4. Run the board audit script to verify consistency
5. Update MEMORY.md if the completion changes epic summary or unblocks significant work

**C. Completing a research spike (4 steps)**
1. Update spike file `Status` to `DONE`
2. Update the corresponding row in the parent epic's Stories table to `DONE`
3. Note key findings in the epic's Technical Notes section if decision-relevant
4. Update MEMORY.md with summary of findings and artifact location

**D. Dispatch Mode pre-dispatch (6 steps)**
1. Read the epic directory -- all story files
2. Verify epic status is `READY` or `ACTIVE`
3. Run the board audit script to surface pre-existing inconsistencies
4. Identify TODO stories whose dependencies are all DONE
5. For each eligible story: update file Status to `IN_PROGRESS`, update epic table row, then dispatch
6. After each story completes: update file to `DONE`, update table row, check for newly unblocked stories, repeat

### Q3: Audit Script Scope

**Decision: Adopt PM's proposal. Script lives at `scripts/audit_epic_board.py`.**

The `scripts/` directory does not yet exist in the project root. E-011-02 will create it. The script:
- Reads all epic directories under `/epics/`
- For each epic, reads the Stories table from `epic.md`
- For each row in the Stories table, finds the corresponding story file by ID
- Compares the `Status` field in the story file against the status in the table row; reports mismatches
- Finds story files in epic directories that have no row in the Stories table (orphaned files)
- Finds Stories table rows that reference a non-existent file (dangling rows)
- Checks: if all stories are DONE, is the epic status COMPLETED? Reports if not.
- Output: structured text report to stdout; exit code 0 if no issues, exit code 1 if issues found
- Runs without external dependencies (stdlib only: `pathlib`, `re`, `sys`)
- Story file status is parsed from the line matching `^## Status` followed by the next non-empty line containing a backtick-quoted value
- Stories table rows are parsed from the markdown table in the `## Stories` section of `epic.md`

**MEMORY.md consistency check** (Q1 open question): Deferred to a potential follow-on. Parsing MEMORY.md is significantly more complex and fragile than parsing structured markdown tables. The audit script focuses on file-system consistency, which is the primary failure mode from E-009.

### Q4: Enforcement Mechanism

**Decision: Manual with mandatory pre-dispatch inclusion in PM agent definition.**

Rationale:
- A pre-commit hook for PM board auditing would be noisy (runs on every commit, including code commits) and risks false positives during active story writing.
- E-006 already occupies the pre-commit hook mechanism for PII scanning. Adding a second check increases complexity.
- A Claude Code `PreToolUse` hook would require modifying `.claude/settings.json` and scoping it correctly to PM sessions only -- significant complexity for a single workflow rule.
- The simpler, sufficient solution: add the audit as a mandatory named step in the PM's Dispatch Mode procedure (in `project-manager.md`). The Dispatch Mode checklist already has 8 steps; the audit becomes step 3 (before identifying eligible stories).
- Additionally, the PM standards document (E-011-01) will instruct the PM to run the audit at the end of every story completion (step 4 in the "Completing a story" protocol). This provides two enforcement points with no new infrastructure.
