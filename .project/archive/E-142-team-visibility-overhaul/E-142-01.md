# E-142-01: Access Fan-Out on Member Team Create

## Epic
[E-142: Team Visibility Overhaul](epic.md)

## Status
`DONE`

## Description
After this story is complete, when a team is created via the admin UI with `membership_type='member'`, all existing users will automatically receive `user_team_access` rows for the new team. The new member team will appear in `permitted_teams` on the user's next request, with no logout/login or session re-creation required.

## Context
`_insert_team_new()` in `src/api/routes/admin.py` creates the team row but does not insert `user_team_access` entries. The dev-bypass backfill in `auth.py` (`_assign_member_teams`) only fires when a user has zero team assignments, so adding a new member team when the user already has other teams leaves the new team invisible. This is the prerequisite fix -- without access, the user can't reach the dashboard for the new team.

## Acceptance Criteria
- [ ] **AC-1**: Given a user with existing team access, when a new team is created via `POST /admin/teams/confirm` with `membership_type='member'`, then `user_team_access` rows exist for ALL users for the new team immediately after the insert completes.
- [ ] **AC-2**: Given a user with an active session, when they load any dashboard page after a new member team was created, then `request.state.permitted_teams` includes the new team's id (no logout/login required).
- [ ] **AC-3**: Given a team created with `membership_type='tracked'`, when the insert completes, then NO `user_team_access` rows are created for that team (tracked teams are not auto-assigned).
- [ ] **AC-4**: The `user_team_access` fan-out uses `INSERT OR IGNORE` to be idempotent per TN-2.
- [ ] **AC-5**: Existing tests continue to pass. New test covers the fan-out for member teams and the no-fan-out for tracked teams.

## Technical Approach
The core change is in `src/api/routes/admin.py`. The `_insert_team_new()` function currently inserts the team and commits without returning the new id. It needs to: (1) capture `lastrowid` after the INSERT, (2) when `membership_type == 'member'`, query all user ids from `users` and insert `user_team_access` rows with `INSERT OR IGNORE`, (3) return the new team id. All within the same connection/transaction. See TN-2 for details.
