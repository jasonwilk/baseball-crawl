# E-142-02: Year Map Fallback for No-Data Teams

## Epic
[E-142: Team Visibility Overhaul](epic.md)

## Status
`DONE`

## Description
After this story is complete, `get_team_year_map()` will return a year for every team in its input list, falling back to the current calendar year for teams with no stat data. A companion function `get_teams_with_stat_data()` will return only team ids with actual stat rows, providing a reliable `has_stat_data` signal for downstream consumers. The year selector will always include the current calendar year when any active team exists. Teams added without stat data will appear in the correct year's team pills.

## Context
`get_team_year_map()` in `src/api/db.py` only returns teams that have rows in `player_season_batting` or `player_season_pitching`. Teams with no stat data are silently omitted from the result dict, making them invisible in the year selector and team pill list. The dashboard route's `_resolve_year_and_team()` then cannot find these teams, so they never appear. See TN-1 for the fallback strategy.

## Acceptance Criteria
- [ ] **AC-1**: Given a team with no rows in `player_season_batting` or `player_season_pitching`, when `get_team_year_map([team_id])` is called, then the result dict includes `team_id → current_calendar_year`.
- [ ] **AC-2**: Given a mix of teams (some with stat data, some without), when `get_team_year_map()` is called, then teams WITH stat data return their actual year from `seasons`, and teams WITHOUT stat data return the current calendar year.
- [ ] **AC-3**: Given active teams exist but none have stat data for the current year, when `get_team_year_map()` is called with those team ids, then the current calendar year appears in the result dict's values (making it available for the year selector's `available_years` list).
- [ ] **AC-4**: Given a team with stat data in 2025 and a team with no stat data (fallback to 2026), when the dashboard is loaded with `?year=2026`, then the no-data team appears in the team pills for 2026.
- [ ] **AC-5**: A new function `get_teams_with_stat_data(team_ids: list[int]) → set[int]` returns only team ids that have actual rows in `player_season_batting` or `player_season_pitching`. This is the authoritative `has_stat_data` signal per TN-1.
- [ ] **AC-6**: Existing tests continue to pass. New tests cover the fallback behavior and `get_teams_with_stat_data()`.

## Technical Approach
Two changes to `src/api/db.py`: (1) Modify `get_team_year_map()` -- after the existing UNION query, iterate through the input `team_ids` list and for any id not present in the result dict, add `team_id → datetime.date.today().year`. This is a post-query backfill, not a query change. (2) Add `get_teams_with_stat_data(team_ids) → set[int]` -- a lightweight query that returns the set of team ids with at least one row in `player_season_batting` or `player_season_pitching`. This provides a reliable `has_stat_data` signal that E-142-03 consumes (since after the fallback, every team is in `team_year_map`). See TN-1 for rationale.
