# E-159-01: Print Route and Template

## Epic
[E-159: Print-Optimized Scouting Report View](epic.md)

## Status
`DONE`

## Description
After this story is complete, `GET /dashboard/opponents/{opponent_team_id}/print` will render a self-contained, print-optimized scouting report. Page 1: report header, context bar (last meeting + team batting tendencies), full pitching table, and full batting table. Page 2+: "Batter Tendencies" spray chart placeholder grid (may span additional pages for large rosters). The page uses inline CSS (no Tailwind CDN), landscape orientation, and includes a print button that hides itself during printing.

## Context
The existing `opponent_detail` route and its helper functions (`_fetch_opponent_detail_data`, `_compute_opponent_pitching_rates`, `_compute_team_batting`, `_sort_batting`, `_sort_pitching`) already produce all the data needed. This story adds a new route that reuses those functions and a new template with a UXD-specified print layout. See epic Technical Notes TN-1 through TN-7 for the complete design spec.

## Acceptance Criteria

### Route & Auth
- [ ] **AC-1**: `GET /dashboard/opponents/{opponent_team_id}/print` returns 200 with the same auth checks as the existing `opponent_detail` route (permitted_teams authorization, 403 if unauthorized).
- [ ] **AC-2**: The route accepts `team_id`, `year`, and `season_id` query params. `team_id` and `season_id` drive active-team/season selection, Last Meeting lookup, and empty-state detection. `year` is optional round-trip context — accepted when present and passed to the template for the "View online" back-link URL, but not used in routing or data-fetching logic. Consistent with the existing `opponent_detail` route.

### Page 1: Header + Context Bar + Pitching
- [ ] **AC-3**: The report header shows opponent team name (18pt bold), record (W-L), "Scouting Report" label, and today's date.
- [ ] **AC-4**: The context bar shows last meeting info (date, score, W/L) and team batting tendencies (OBP, K%, BB%, SLG) in a single compact row (9pt, no card chrome), per Technical Notes TN-2. If last meeting is null, show "No previous meetings". If `team_batting.has_data` is false, show dashes for OBP/K%/BB%/SLG. The print template does NOT include the "Their Pitchers" card or "Team Batting summary" card from the interactive view — these are replaced by the context bar and full tables.
- [ ] **AC-5**: The pitching table shows exactly 10 columns: `#, Player, ERA, K/9, WHIP, GP, IP, BB, SO, Strike%` — per Technical Notes TN-4. No inline "(X GP)" subtext in stat cells. The Player cell includes pitcher handedness after the name (e.g., "Name (R)") when `throws` is available.

### Page 1 (continued): Batting Table + Page 2+: Spray Chart Placeholders
- [ ] **AC-6**: A CSS page break (`page-break-before: always`) separates the Batter Tendencies section from the batting section (not between pitching and batting), so pitching and batting tables appear together on page 1 and the spray chart grid begins on page 2.
- [ ] **AC-7**: The batting table shows exactly 12 columns: `#, Player, AVG, OBP, SLG, GP, AB, BB, SO, HR, SB, RBI` — per Technical Notes TN-4. No inline "(X GP)" subtext in stat cells.
- [ ] **AC-8**: Below the batting table, a "Batter Tendencies" section renders a grid of placeholder cards (3 per row), each showing player name + jersey number + a 160x160px placeholder box with "Spray chart coming soon" centered and a light border. Row order matches batting table sort. A CSS page break forces the Batter Tendencies section to always start on a new page. Per Technical Notes TN-5.

### Template & CSS
- [ ] **AC-9**: The template is self-contained — does not extend `base.html`, does not load any external CSS/JS CDN. All styles in a `<style>` block. Font stack: `Arial, Helvetica, sans-serif`, per Technical Notes TN-3.
- [ ] **AC-10**: `@media print` rules include: `@page { size: landscape; margin: 0.5in; }`, print button and "View online" link hidden, `page-break-inside: avoid` on table rows. Tables use `width: 100%` with no `overflow-x-auto` wrappers, per Technical Notes TN-6.
- [ ] **AC-11**: Table headers use dark text + thick bottom border (no dark background). Zebra rows alternate with `#f5f5f5`. No `overflow-x-auto` wrappers.

### Interactive Elements
- [ ] **AC-12**: The page includes a "Print / Save as PDF" button (`onclick="window.print()"`) and a "View online" back-link beside it. Both are hidden during printing. The "View online" back-link preserves `team_id`, `year` (when non-None), and `season_id` query params, ensuring the user returns to the same context they printed from.
- [ ] **AC-13**: Batting and pitching tables use default sort order (batting: AVG descending, pitching: ERA ascending) with no interactive sort links in table headers.

### Empty States & Footer
- [ ] **AC-14**: The three empty states (unlinked, linked_unscouted, full_stats) are handled — unlinked shows "Stats not available.", linked_unscouted shows "Stats not loaded yet." (reusing the message strings from the existing `opponent_detail.html` template); full_stats shows all sections. Within full_stats, if the pitching list is empty, show "No pitching data available." instead of an empty table; same for batting ("No batting data available.").
- [ ] **AC-15**: A footer `<div>` at the document end contains the opponent name. It is visible on screen and prints on the last page. (CSS `@page` margin boxes are not supported in Chromium, so per-page repeating footers are not possible.)

### Tests
- [ ] **AC-16**: All existing tests pass. Tests verify: (a) the route returns 200 for full_stats and the response contains the print template's distinctive elements (e.g., "Scouting Report" text, `window.print()`), and (b) the route returns 200 for unlinked and linked_unscouted states with the corresponding empty-state message ("Stats not available." / "Stats not loaded yet.").

## Technical Approach
Add a new route function in `src/api/routes/dashboard.py` that mirrors `opponent_detail` but renders a different template with default sorts and no sort query params. The template (`src/api/templates/dashboard/opponent_print.html`) is standalone HTML with a `<style>` block — not extending `base.html`. Reuse all existing data-fetching and computation functions. The context bar replaces the separate "Their Pitchers" and "Team Batting summary" cards from the screen view.

## Notes
- Reference the existing `opponent_detail` route (`src/api/routes/dashboard.py:1267`) for the full data pipeline
- The `_sort_batting` and `_sort_pitching` functions accept sort key and direction — use `"avg"`/`"desc"` and `"era"`/`"asc"` as defaults
- Template filters `ip_display`, `format_avg`, `format_date` are available in the Jinja2 environment and should be used in the print template
- The `_compute_team_batting` function returns a dict with `obp`, `k_pct`, `bb_pct`, `slg`, `has_data` — use these for the context bar
- The `_get_top_pitchers` function is NOT used in the print view (no top pitchers card) — the full pitching table replaces it
- The Batter Tendencies section always renders a placeholder card for each player with "Spray chart coming soon" text (no conditionals). A future story will wire spray chart data into the print route context when available — E-158 does not touch the print template. Jinja2 handles absent dict keys as `Undefined` (falsy) so no pre-population is needed.
