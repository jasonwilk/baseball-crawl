# E-046-01: Python Crawler Proxy Support

## Epic
[E-046: Upstream Proxy Support](epic.md)

## Status
`DONE`

## Description
After this story is complete, `create_session()` will automatically read `PROXY_ENABLED` and the profile-specific proxy URL (`PROXY_URL_WEB` or `PROXY_URL_MOBILE`) from environment variables and configure proxy routing when enabled. A helper function `get_proxy_config(profile)` encapsulates the env var reading, mapping the session profile to the correct proxy zone URL. When `PROXY_ENABLED=true` but the relevant URL is empty, a WARNING is logged and proxy is skipped for that profile. Proxy URLs are never logged. All existing callers of `create_session()` gain proxy support automatically -- no call-site changes required.

## Context
All Python HTTP traffic flows through `create_session()` in `src/http/session.py`. The function already accepts a `profile` parameter (`"web"` or `"mobile"`) that selects the header fingerprint. Adding profile-aware proxy support here covers all crawlers and the GameChanger client automatically. The profile parameter now also determines which proxy zone to use:
- `profile="web"` -> `PROXY_URL_WEB` (Bright Data residential zone)
- `profile="mobile"` -> `PROXY_URL_MOBILE` (Bright Data mobile zone)

The env var reading is separated into a helper so callers can inspect proxy config without creating a session, and so the session factory remains testable without environment manipulation.

## Acceptance Criteria
- [ ] **AC-1**: `create_session(profile="web", proxy_url="http://proxy:7777")` returns an `httpx.Client` configured with `proxy="http://proxy:7777"`. Same for `profile="mobile"`.
- [ ] **AC-2**: `create_session()` with `PROXY_ENABLED` unset or `false` returns an `httpx.Client` with no proxy configured (current behavior preserved), regardless of profile.
- [ ] **AC-3**: `create_session(profile="web")` with `PROXY_ENABLED=true` and `PROXY_URL_WEB` set automatically configures the residential proxy. `create_session(profile="mobile")` with `PROXY_ENABLED=true` and `PROXY_URL_MOBILE` set automatically configures the mobile proxy. No explicit `proxy_url` argument needed. All existing callers gain proxy support without code changes.
- [ ] **AC-4**: A `get_proxy_config(profile: str) -> str | None` function in `src/http/session.py` reads `PROXY_ENABLED` and `PROXY_URL_{profile.upper()}` from `os.environ`. Returns the URL string when enabled and set; returns `None` when disabled or unset. `create_session()` calls this internally when no explicit `proxy_url` is passed, forwarding its own `profile` parameter.
- [ ] **AC-5**: When `PROXY_ENABLED=true` and the profile's URL var is empty or unset, `get_proxy_config()` logs a WARNING (message includes the env var name, e.g., "PROXY_ENABLED is true but PROXY_URL_WEB is not set") and returns `None`.
- [ ] **AC-6**: Proxy URL values never appear in any log message from `src/http/session.py`. Test this: call `get_proxy_config("web")` with a known URL in `PROXY_URL_WEB`, capture logs, assert the URL string is absent.
- [ ] **AC-7**: `.env.example` has a new "Upstream Proxy (Bright Data)" section with `PROXY_ENABLED`, `PROXY_URL_WEB`, and `PROXY_URL_MOBILE` entries (commented out), with comments explaining the profile-to-zone mapping and that both Python crawlers and mitmproxy respect these vars.
- [ ] **AC-8**: The existing `profile` parameter (web/mobile) and all its tests continue to work unchanged. Proxy support is additive and does not alter header profile behavior.
- [ ] **AC-9**: New tests cover AC-1 through AC-6 and AC-11: explicit proxy kwarg (both profiles), auto env-var wiring (web profile reads PROXY_URL_WEB, mobile reads PROXY_URL_MOBILE), no-proxy default, env var reading per profile, graceful degradation per profile, log safety, and malformed URL rejection.
- [ ] **AC-10**: When both profiles have proxy URLs set, each session gets only its own profile's proxy. A web session never uses the mobile proxy URL and vice versa.
- [ ] **AC-11**: When `PROXY_ENABLED=true` and the profile's URL var contains a value that does not start with `http://` or `https://`, `get_proxy_config()` logs a WARNING (message includes the env var name and notes the invalid scheme, but does NOT include the URL value) and returns `None`. This prevents `httpx.Client` from raising `ValueError` on malformed URLs.

## Technical Approach
1. Add `get_proxy_config(profile: str) -> str | None` to `src/http/session.py`. It reads `os.environ.get("PROXY_ENABLED", "").lower()` and `os.environ.get(f"PROXY_URL_{profile.upper()}", "")`. Logic: if enabled != "true", return None. If enabled == "true" and URL is empty, log WARNING naming the missing env var, return None. If URL is non-empty but does not start with `http://` or `https://`, log WARNING (name the env var and note invalid scheme, but do NOT log the URL value) and return None. Otherwise return the URL.
2. Add `proxy_url: str | None = _UNSET` parameter to `create_session()`. When `proxy_url` is not explicitly passed (sentinel), call `get_proxy_config(profile)` internally. Pass the result as `proxy=proxy_url` to `httpx.Client()`. When `proxy_url` is explicitly passed (including `None`), use the explicit value and skip env var reading.
3. Add the "Upstream Proxy (Bright Data)" section to `.env.example` after the MCP Server Tokens section. Include `PROXY_ENABLED`, `PROXY_URL_WEB`, and `PROXY_URL_MOBILE` with placeholder comments. See the `.env.example` format in the epic Technical Notes for the exact content.
4. Write tests using `monkeypatch.setenv()` for env var tests. To verify `proxy=` is passed correctly to `httpx.Client`, use `unittest.mock.patch("src.http.session.httpx.Client")` and assert the `proxy` kwarg value — this is more stable than inspecting httpx private APIs like `_mounts`. Use `caplog` to verify log safety and WARNING messages. Test both profiles independently to verify correct env var mapping.
5. Preserve the existing `profile` parameter and its tests. The `proxy_url` parameter is orthogonal to header profile selection.

## Notes
- httpx 0.28.x uses singular `proxy` kwarg (not `proxies`). httpx handles `http://user:pass@host:port` proxy auth natively.
- `create_session()` must pass `trust_env=False` to `httpx.Client()` to prevent httpx from reading `HTTP_PROXY`/`HTTPS_PROXY` from the shell environment. Proxy routing must be exclusively controlled by `PROXY_ENABLED` + `PROXY_URL_*` env vars, not by unrelated system proxy settings.
- `get_proxy_config(profile)` is intentionally separate from `create_session()` so that startup checks or future callers can inspect proxy config without creating an HTTP client, and so `create_session()` remains testable without environment manipulation.
- Use a sentinel default (e.g., `_UNSET = object()`) for the `proxy_url` parameter so callers can explicitly pass `proxy_url=None` to disable proxy even when env vars are set, distinguishing "not provided" from "explicitly None". The type annotation `str | None` won't cover the sentinel — use a `# type: ignore` on the default or a typed sentinel class to satisfy type checkers.
- The env var naming convention `PROXY_URL_{PROFILE}` is deliberately parallel to the `profile` parameter in `create_session()`. Adding a new profile in the future means adding a new env var -- no code changes to the proxy routing logic.
