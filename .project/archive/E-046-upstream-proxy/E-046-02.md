# E-046-02: mitmproxy Upstream Mode

## Epic
[E-046: Upstream Proxy Support](epic.md)

## Status
`DONE`

## Description
After this story is complete, the host-based mitmproxy stack (`proxy/`) will start in upstream proxy mode when `PROXY_ENABLED=true` and the selected profile's proxy URL is set. The operator selects which proxy profile to use at start time via `./start.sh --profile mobile` (default) or `./start.sh --profile web`. The selected profile determines which upstream proxy URL is used:
- `mobile` (default) -> `PROXY_URL_MOBILE` (Bright Data mobile zone, for iPhone traffic capture)
- `web` -> `PROXY_URL_WEB` (Bright Data residential zone, for browser traffic capture)

When proxy is disabled (or the profile's URL is empty despite `PROXY_ENABLED=true`), mitmproxy runs in its current regular intercepting mode. The `proxy/status.sh` script will display whether upstream mode is active and which profile is in use.

## Context
E-048 migrated mitmproxy from a Docker Compose profile in the project stack to a standalone `proxy/` directory on the Mac host. The proxy has its own `proxy/docker-compose.yml` with a static `mitmweb` entrypoint and command block. Upstream mode is activated by adding `--mode upstream:URL` to the command. A wrapper entrypoint script replaces the static entrypoint and checks env vars at container startup to build the appropriate command line.

The proxy stack serves two use cases:
1. **iPhone capture** (mobile profile): mitmproxy sits between the iPhone and GameChanger's servers, capturing mobile API traffic. Uses the Bright Data mobile zone (`PROXY_URL_MOBILE`).
2. **Browser capture** (web profile): mitmproxy sits between a browser (e.g., Chrome) and GameChanger's servers, capturing web API traffic. Uses the Bright Data residential zone (`PROXY_URL_WEB`).

Only one instance runs at a time (same ports 8080/8081). To switch profiles, stop and restart with the new profile argument.

## Acceptance Criteria
- [ ] **AC-1**: `./start.sh` (no arguments) starts mitmproxy with profile `mobile` (backward compatible). When `PROXY_ENABLED=true` and `PROXY_URL_MOBILE` is set, mitmproxy runs with `--mode upstream:${PROXY_URL_MOBILE}`.
- [ ] **AC-2**: `./start.sh --profile web` starts mitmproxy with profile `web`. When `PROXY_ENABLED=true` and `PROXY_URL_WEB` is set, mitmproxy runs with `--mode upstream:${PROXY_URL_WEB}`.
- [ ] **AC-3**: `./start.sh --profile mobile` is equivalent to `./start.sh` (explicit mobile selection).
- [ ] **AC-4**: When `PROXY_ENABLED` is unset, empty, or `false`, mitmproxy starts without `--mode` regardless of which profile is selected (current behavior -- regular intercepting proxy).
- [ ] **AC-5**: When `PROXY_ENABLED=true` but the selected profile's URL is empty or unset, mitmproxy starts without `--mode` (graceful degradation, same as AC-4). The entrypoint logs a warning to stderr naming the missing env var (e.g., "PROXY_ENABLED is true but PROXY_URL_WEB is not set").
- [ ] **AC-6**: The proxy URL values do not appear in the `command:` or `entrypoint:` blocks of `cd proxy && docker compose config` output. They will appear under `environment:` (expected -- env_file vars are resolved by compose config). The credential protection is that URLs are never baked into compose command args or visible in host process listings (`ps -ef`).
- [ ] **AC-7**: `proxy/status.sh` displays whether upstream proxy mode is active and which profile is in use (e.g., "Upstream proxy: enabled (mobile zone)" or "Upstream proxy: enabled (web/residential zone)" or "Upstream proxy: disabled"). Use `docker inspect` to check for `--mode upstream` in the running container's process args.
- [ ] **AC-8**: `PROXY_ENABLED`, `PROXY_URL_WEB`, and `PROXY_URL_MOBILE` reach the container via `env_file`, not via the `environment:` block in `proxy/docker-compose.yml`. `MITMPROXY_PROFILE` may appear in the `environment:` block (no credential risk -- just the string `mobile` or `web`).
- [ ] **AC-9**: `proxy/docker-compose.yml` reads both `../.env` (project root) and `proxy/.env` via `env_file`, so the operator only needs to set proxy vars in the root `.env`. Both env_file entries are `required: false`. `proxy/.env` listed second for override precedence.
- [ ] **AC-10**: `./start.sh` with an invalid profile (e.g., `./start.sh --profile foo`) prints an error message listing valid profiles and exits without starting the container.
- [ ] **AC-11**: `./start.sh` startup output shows which profile is active (e.g., "Profile: mobile" or "Profile: web").

## Technical Approach
1. Update `proxy/start.sh` to accept `--profile <name>` (default: `mobile`). Validate that the profile is `mobile` or `web`; exit with an error for invalid profiles. Export `MITMPROXY_PROFILE` so `docker compose up` inherits it. Update the startup output to show which profile is active.
2. Create `proxy/proxy-entrypoint.sh` -- a bash wrapper that:
   - Reads `MITMPROXY_PROFILE` (default: `mobile`)
   - Derives the proxy URL env var name: `proxy_var="PROXY_URL_${MITMPROXY_PROFILE^^}"` (bash uppercase)
   - Reads `PROXY_ENABLED` and the derived proxy URL var using indirect variable reference (`${!proxy_var}`)
   - If `PROXY_ENABLED=true` and the URL is set, execs `mitmweb --web-host 0.0.0.0 --scripts /app/proxy/addons/loader.py --set "web_password=${MITMWEB_PASSWORD:-}" --mode "upstream:${proxy_url}"` (double-quote expansions to prevent word splitting/globbing on passwords with special characters)
   - If `PROXY_ENABLED=true` but the URL is empty, logs a warning to stderr naming the missing env var and execs mitmweb without `--mode`
   - Otherwise, execs mitmweb without `--mode` (current behavior)
   - mitmproxy's `--mode upstream:http://user:pass@host:port` supports embedded proxy auth natively
3. Update `proxy/docker-compose.yml`:
   - Change `entrypoint` from `["mitmweb"]` to `["/app/proxy/proxy-entrypoint.sh"]`
   - Remove the `command:` block (args move into the entrypoint script)
   - Do NOT add `PROXY_ENABLED`, `PROXY_URL_WEB`, or `PROXY_URL_MOBILE` to the `environment:` block -- they flow through `env_file` only
   - Add `MITMPROXY_PROFILE` to the `environment:` block with a default (e.g., `MITMPROXY_PROFILE=${MITMPROXY_PROFILE:-mobile}`). This is safe -- no credentials.
   - Add `../.env` to the `env_file` list (before the existing `proxy/.env` entry so proxy/.env can override). Both entries must be `required: false`
4. Update `proxy/status.sh` to check the running container for `--mode upstream` via `docker inspect` and display upstream proxy status. Also check `MITMPROXY_PROFILE` env var from the container to determine the profile label (mobile zone vs web/residential zone).

## Notes
- The entrypoint wrapper approach keeps proxy URLs out of command-line args visible in `ps -ef`. `docker compose config` will still show env_file values under `environment:` -- this is expected and acceptable for a local dev tool on the operator's own machine.
- No automated tests for this story -- it is Docker infrastructure verified by manual inspection of container config. AC verification uses `docker inspect` on the running container.
- The `proxy/` stack volume-mounts `..:/app`, so the entrypoint script at `proxy/proxy-entrypoint.sh` is accessible inside the container at `/app/proxy/proxy-entrypoint.sh`.
- The profile selection pattern (`PROXY_URL_${PROFILE^^}`) is deliberately parallel to `get_proxy_config(profile)` in E-046-01. Same naming convention, same mapping logic.
