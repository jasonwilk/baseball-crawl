# E-098-05: Fix Roster-Fail Commit Gap, Load-Path completed_at, and Test Assertions

## Epic
[E-098: Fix Scouting Pipeline Bugs](epic.md)

## Status
`DONE`

## Description
After this story is complete, three remaining status lifecycle gaps are closed: (1) the roster-fail path commits its 'failed' status write so it persists instead of being rolled back, (2) `update_run_load_status()` writes `completed_at` alongside the status transition so the E-098-04 timestamp contract is honored for the load phase, and (3) load-phase tests assert `completed_at` values to prevent regression.

## Context
A Codex code review of the E-098-03/04 implementation found three gaps in the status lifecycle work:

1. **P1-A: Missing commit on roster-fail path** (`scouting.py:153`): When `_fetch_and_write_roster()` returns `None`, `_upsert_run_end()` writes `status='failed'` but there is no `self._db.commit()` before the `return CrawlResult(errors=1)`. The UPDATE is rolled back when the connection closes, leaving the row stuck as 'running'. DE confirmed this is the only call site that breaks the commit pattern -- the zero-boxscore fail path (line 179-180) and the success path (line 182-183) both commit correctly.

2. **P1-B: `update_run_load_status()` missing `completed_at`** (`scouting.py:305-313`): E-098-04 established the contract that terminal statuses ('completed', 'failed') get a `completed_at` timestamp and non-terminal ('running') gets NULL. But `update_run_load_status()` -- the CLI's path for setting the final status after load -- only updates `status` and `last_checked`, not `completed_at`. The load-phase 'completed' transition never gets a timestamp.

3. **P2: Load-phase tests missing `completed_at` assertions**: Existing tests for the load-phase status transitions don't assert `completed_at` values, so P1-B was invisible to the test suite.

All three confirmed FIX by SE and DE. Single story because they're tightly coupled around the same status lifecycle.

## Acceptance Criteria
- [ ] **AC-1**: The roster-fail path in `scout_team()` commits the 'failed' status write before returning. Specifically, `self._db.commit()` is called after `_upsert_run_end(..., "failed", ...)` at the roster-fail branch (around line 153).
- [ ] **AC-2**: A test verifies that when roster fetch fails, the `scouting_runs` row has `status='failed'` (not 'running') when read from a separate connection or after the method returns. The test must not rely on the same connection's uncommitted state.
- [ ] **AC-3**: `update_run_load_status()` sets `completed_at` in the UPDATE: current UTC timestamp when `status` is 'completed', `NULL` when `status` is 'failed'.
- [ ] **AC-4**: Load-phase tests (both success and failure paths) assert `completed_at` values: non-null for 'completed', null for 'failed'.
- [ ] **AC-5**: All existing scouting tests continue to pass.

## Technical Approach
Read `src/gamechanger/crawlers/scouting.py`. Three changes:
1. Add `self._db.commit()` after the `_upsert_run_end` call at the roster-fail branch (around line 153). This is the only missing commit site.
2. In `update_run_load_status()` (line 290-314), add `completed_at` to the UPDATE SET clause. Use the same conditional pattern established by E-098-04 in `_upsert_run_end()`: timestamp for 'completed', NULL for 'failed'.
3. In test files, add `completed_at` assertions to load-phase tests. For the roster-fail test, use a fresh connection to verify the commit persisted (classic SQLite test isolation pattern).

## Notes
- P1-A is a one-line fix (`self._db.commit()`) but the test requires care: SQLite in-process connections share uncommitted state, so verifying the commit requires either a new connection or checking after the method returns and the connection commits. DE flagged this as a classic SQLite test isolation trap.
- P1-B mirrors the E-098-04 pattern in `_upsert_run_end()` but applied to the separate `update_run_load_status()` method. The two methods should have consistent `completed_at` behavior.
- The `update_run_load_status()` docstring should note the `completed_at` behavior for clarity.
