# E-098-02: Fix Silent Load Failures in bb data scout + CLI Failure Test

## Epic
[E-098: Fix Scouting Pipeline Bugs](epic.md)

## Status
`DONE`

## Description
After this story is complete, load failures during `bb data scout` will be visible to the operator through error messages and non-zero exit codes. Currently, load failures are silently swallowed: the crawler marks scouting runs as "completed" before loading occurs, and the CLI ignores `LoadResult` return values. This story fixes both the premature status marking and the missing error propagation, and adds CLI tests that verify failure handling.

## Context
Two distinct issues combine to make load failures invisible:
1. The crawler (`scouting.py:160`) calls `_upsert_run_end` with `status=completed` before any loading happens. If loading subsequently fails, future scouting runs see the "completed" status and skip re-scouting (freshness check).
2. The CLI helpers `_load_scouted_team` and `_load_all_scouted` (`data.py:297, :325`) ignore the return value from `loader.load_team()`. The exit code in `_run_scout_pipeline` (line 261) checks only `crawl_result.errors`, so the operator sees "Load complete" even when DB writes failed.

## Acceptance Criteria
- [ ] **AC-1**: Given a successful crawl followed by a load failure (e.g., `loader.load_team()` raises an exception or returns a `LoadResult` with errors > 0), when `bb data scout --team <id>` completes, then the exit code is non-zero.
- [ ] **AC-2**: Given a load failure, when the operator checks `scouting_runs`, then the run status is NOT "completed" (it should reflect the load failure -- e.g., "load_failed" or similar distinguishable status).
- [ ] **AC-3**: Given a load failure for one team during `bb data scout` (all teams mode), when other teams load successfully, then the exit code is still non-zero (partial failure = failure) and the successful teams are still loaded.
- [ ] **AC-4**: The `_load_scouted_team` and `_load_all_scouted` helper functions propagate load errors back to `_run_scout_pipeline` so they are reflected in the exit code.
- [ ] **AC-5**: A new test in `tests/test_cli_scout.py` verifies that `bb data scout` exits non-zero when loading fails (mocked loader that returns errors or raises).
- [ ] **AC-6**: All existing scouting CLI and crawler tests continue to pass.

## Technical Approach
Three files need changes:

1. `/workspaces/baseball-crawl/src/gamechanger/crawlers/scouting.py` -- The crawler currently marks runs as "completed" (line 160) before the CLI calls the loader. The status needs to reflect the actual end-to-end outcome. The implementing agent should determine the cleanest way to defer or update the status based on load results.

2. `/workspaces/baseball-crawl/src/cli/data.py` -- `_load_scouted_team` (line 264) and `_load_all_scouted` (line 301) need to capture and propagate load errors. `_run_scout_pipeline` (line 261) needs to incorporate load errors into the exit code calculation.

3. `/workspaces/baseball-crawl/tests/test_cli_scout.py` -- Add tests that mock load failures and verify non-zero exit codes and error messaging.

The `LoadResult` dataclass is defined in the loaders -- check its structure at `/workspaces/baseball-crawl/src/gamechanger/loaders/scouting_loader.py` to understand the return type from `load_team()`.
