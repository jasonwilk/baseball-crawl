# E-004-06: Player Profile Page

## Epic
[E-004: Coaching Dashboard](epic.md)

## Status
`DONE`

## Description
After this story is complete, `GET /dashboard/players/{player_id}` will render a player profile page showing the player's stats broken down by season and team. This is the longitudinal tracking view -- coaches can see how a player has developed across freshman, JV, varsity, and legion teams over multiple seasons.

## Context
Players appear in `team_rosters` for each team and season they were on. Their stats are in `player_season_batting` and `player_season_pitching`, each scoped to a team and season. This story assembles a player's full history into a single page, giving coaches the career arc view they need for player development conversations and multi-year planning.

- [ ] **AC-6**: Season names display as the human-readable `seasons.name` (e.g., "Spring 2026 High School"), not the season_id slug.

## Technical Approach
1. Add `get_player_profile()` to `src/api/db.py`. Three queries: (a) player info from `players`, (b) batting seasons from `player_season_batting` joined with `seasons` and `teams`, (c) pitching seasons from `player_season_pitching` joined with `seasons` and `teams`.
2. Add a fourth query for recent games: union of `player_game_batting` and `player_game_pitching` joined with `games` and `teams`, limited to 5 most recent by game_date. **Two-way player edge case**: a player can appear in both `player_game_batting` and `player_game_pitching` for the same game. The recent games list should show one row per game (not per appearance type). Use `SELECT DISTINCT game_id, game_date, ...` in each branch of the UNION, then deduplicate by game_id in the outer query (or use `GROUP BY game_id`) before applying `ORDER BY game_date DESC LIMIT 5`.
3. Add `GET /dashboard/players/{player_id}` route to `src/api/routes/dashboard.py`. The route must verify the player has at least one `team_rosters` entry for a team in `request.state.permitted_teams` before rendering; return 403 otherwise.
4. Create `src/api/templates/dashboard/player_profile.html`.
5. Reference: `/workspaces/baseball-crawl/migrations/001_initial_schema.sql` for all table schemas.

## Notes
- The player profile page does not use the team_id query parameter, but it IS auth-guarded: the player must appear on at least one of the user's permitted teams. This supports the core use case of tracking players across the Lincoln program (freshman -> JV -> varsity) while maintaining consistency with the E-023 team-scoped authorization model. Opponent players (who are NOT on any of the user's permitted teams) are not viewable via player profile -- their stats appear only in the opponent scouting report (E-004-05).
- The "Recent Games" section provides a quick game log without leaving the profile. Coaches use this to check a player's recent form before making lineup decisions. Game detail links (`/dashboard/games/{game_id}`) will 404 until E-004-04 lands. This is acceptable -- same pattern as player profile links in E-004-02. No hard dependency needed.
- The "Current Season Summary" card serves the most common coaching use case: "How is this player doing right now?" The season-by-season tables below serve the longitudinal tracking use case: "How has this player developed over time?"
