# E-004-04: Game Log Page

## Epic
[E-004: Coaching Dashboard](epic.md)

## Status
`DONE`

## Description
After this story is complete, `GET /dashboard/games` will display a list of games for the selected team in the current season, showing dates, opponents, scores, and W/L results. Clicking a game opens `GET /dashboard/games/{game_id}` showing a box score with per-player batting and pitching lines for both teams.

## Context
The `games` table stores game_id, date, home/away team IDs, and scores. The `player_game_batting` and `player_game_pitching` tables store per-player per-game stat lines. This story creates the game log view that coaches use to review recent results and individual game performances -- essential for tracking player workload, recent form, and preparing for rematches.

## Acceptance Criteria
- [ ] **AC-1**: `GET /dashboard/games` returns 200 and renders a game list for the active team in the current season. Accepts optional `?season_id=` query parameter (same pattern as E-004-01 AC-9). Returns 403 if `?team_id=` specifies a team not in `request.state.permitted_teams`.
- [ ] **AC-2**: Each game row shows: date (formatted as "Mar 4"), opponent name, score (e.g., "7-3"), W/L indicator, and home/away indicator.
- [ ] **AC-3**: W/L is computed from the scores and team perspective (home vs. away). Display "W" in green, "L" in red, "-" if scores are null.
- [ ] **AC-4**: Games are sorted by date descending (most recent first).
- [ ] **AC-5**: Each game row is a clickable link to `/dashboard/games/{game_id}`. Game rows have minimum 44px tap height (`py-3` or equivalent) for dugout usability.
- [ ] **AC-6**: `GET /dashboard/games/{game_id}` returns 200 and renders a box score page.
- [ ] **AC-7**: The box score page shows two sections: one for each team. Each section is wrapped in a `<details>` element (with the user's team open by default, opponent collapsed by default) with a `<summary>` showing the team name and score. Each section contains a batting table (Player, AB, H, 2B, 3B, HR, RBI, BB, SO, SB) and a pitching table (Player, IP, H, ER, BB, SO, HR). This keeps the box score scannable on mobile without hiding data.
- [ ] **AC-8**: IP in the box score pitching table uses the `ip_display` filter.
- [ ] **AC-9**: The box score page shows the game date, opponent, and final score in a header.
- [ ] **AC-10**: Team selector appears on the game list page with `base_url="/dashboard/games"`.
- [ ] **AC-11**: The game detail page does NOT require team_id in the URL -- the game_id is sufficient. But it verifies the game involves at least one of the user's permitted teams (return 403 otherwise).
- [ ] **AC-12**: Active nav item is "Games".
- [ ] **AC-13**: `db.get_team_games(team_id, season_id)` returns game list with opponent name resolved. `db.get_game_box_score(game_id)` returns batting and pitching lines for both teams with player names.
- [ ] **AC-14**: Tests verify: game list returns 200, game list returns 403 for unauthorized team_id, game detail returns 200, game detail returns 403 for game involving no permitted team, empty season shows "No games found," and `?season_id=` override filters to the specified season.

## Technical Approach
1. Add `get_team_games()` to `src/api/db.py`. Query `games` joined with `teams` (for opponent name). Filter where `home_team_id = ? OR away_team_id = ?`. Use a CASE expression to resolve the opponent: `CASE WHEN home_team_id = :team_id THEN away_team_id ELSE home_team_id END AS opponent_team_id` (and similarly for opponent name join and home/away indicator).
2. Add `get_game_box_score()` to `src/api/db.py`. Query `player_game_batting` and `player_game_pitching` joined with `players`, grouped by team.
3. Add two routes to `src/api/routes/dashboard.py`: game list and game detail. The game detail route must pass `active_team_id` (from `?team_id=` or the user's first permitted team) to the template so the template can set the `open` attribute on the correct `<details>` element (user's team open by default, opponent collapsed).
4. Create `src/api/templates/dashboard/game_list.html` and `src/api/templates/dashboard/game_detail.html`.
5. For game detail, organize data as two team dicts each containing batting_lines and pitching_lines lists. The template uses `active_team_id` to determine which `<details>` gets the `open` attribute.
6. Reference: `/workspaces/baseball-crawl/migrations/001_initial_schema.sql` for `games`, `player_game_batting`, `player_game_pitching` schemas.
