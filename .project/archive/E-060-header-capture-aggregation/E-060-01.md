# E-060-01: First-seen-wins header aggregation with conflict logging

## Epic
[E-060: Stabilize Header-Capture Aggregation](epic.md)

## Status
`DONE`

## Description
After this story is complete, `HeaderCapture.request()` will aggregate headers across multiple requests per source using a first-seen-wins strategy instead of overwriting the entire dict on each request. When two requests provide different values for the same header key, the first-seen value is retained and a WARNING-level log message is emitted identifying the conflict. This makes `header-report.json` output deterministic regardless of request order.

## Context
The current implementation at line 209 of `proxy/addons/header_capture.py` does `self._captured_by_source[source] = headers`, which overwrites the entire captured dict per source on every request. This creates a race condition: the final report depends on which request arrived last. The fix is small (~15-20 lines) but eliminates non-deterministic output that would propagate through `proxy-refresh-headers.py` into `src/http/headers.py`.

## Acceptance Criteria
- [ ] **AC-1**: Given two requests from the same source where request A has headers `{a: 1, b: 2}` and request B has headers `{b: 3, c: 4}`, the aggregated dict is `{a: 1, b: 2, c: 4}` (first-seen wins for key `b`, union for all keys).
- [ ] **AC-2**: Given the same two requests arriving in reversed order (B then A), the set of keys in the aggregated dict is identical to AC-1's result, and each conflicting key retains whichever value was seen first in that ordering. The testable invariant: same key set regardless of order, first-seen value wins for conflicting keys.
- [ ] **AC-3**: Given a conflict (same key, different values across requests), a WARNING-level log message is emitted containing the source name, the conflicting key, the kept value, and the ignored value.
- [ ] **AC-4**: Given requests from different sources (e.g., `web` and `ios`), each source maintains its own independent aggregation -- no cross-source interference.
- [ ] **AC-5**: At least 3 new unit tests in `tests/test_proxy/test_header_capture.py` covering: (a) union aggregation with no conflicts, (b) first-seen-wins on conflicting values, (c) order-independence (same requests in two different orders produce the same key set).
- [ ] **AC-6**: All existing tests in `tests/test_proxy/test_header_capture.py` continue to pass, with one exception: `test_latest_headers_overwrite_previous` (lines 305-316) explicitly asserts the old overwrite behavior and WILL break. This test must be updated or replaced to assert first-seen-wins behavior instead. All other existing tests must pass without modification.

## Technical Approach
Replace the single-line overwrite at line 209 of `proxy/addons/header_capture.py` with a merge loop. See the "Target Behavior" pseudocode in the epic Technical Notes at `/workspaces/baseball-crawl/epics/E-060-header-capture-aggregation/epic.md`. Update the class docstring (line 176-180) to reflect the new aggregation strategy ("first-seen-wins per key" instead of "latest seen header set").

For tests, construct `HeaderCapture` instances directly and feed them mock flow objects with controlled header dicts. Use `caplog` (pytest) to assert the WARNING log on conflicts.
