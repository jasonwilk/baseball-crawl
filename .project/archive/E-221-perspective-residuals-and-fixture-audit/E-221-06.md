## Description
After this story is complete, the test suite has a standalone regression test for R8-P1-2: admin delete of a team that owns `reconciliation_discrepancies` rows must succeed without FK violation, and the team's reconciliation rows must be cleaned. The functional fix is already in place via E-221-05's canonical helper consolidation; this story adds the explicit test bulwark that TN-9 RED-test discipline requires. No production code changes.

## Context
**The fix is already shipped.** E-221-05's Option 2 refactor (2026-04-13) replaced `src/api/routes/admin.py::_delete_team_cascade`'s in-place Phase 1a/1b/2/3/4 DELETE blocks with a single delegation call to `src/reports/generator.py::cascade_delete_team`. The canonical helper cleans `reconciliation_discrepancies` through three independent DELETE paths:

1. **Perspective pass (unbounded)** — `src/reports/generator.py:1445-1448`:
   ```python
   conn.execute(
       "DELETE FROM reconciliation_discrepancies WHERE perspective_team_id = ?",
       (team_id,),
   )
   ```
   Inside `_delete_team_anchor_and_orphan_data` Pass 1. Cleans any row where the deleted team produced the reconciliation record, across any game.

2. **Anchor pass (unbounded)** — `src/reports/generator.py:1476-1478`:
   ```python
   conn.execute(
       "DELETE FROM reconciliation_discrepancies WHERE team_id = ?", (team_id,)
   )
   ```
   Pass 2. Cleans any row where the deleted team is the anchor via `team_id` (covers rows another team's perspective produced about the deleted team's players).

3. **Game-scoped perspective pass** — `src/reports/generator.py:1356-1363`:
   Participant-bounded version inside `_delete_game_scoped_data_for_perspectives`; hit transitively by `cascade_delete_team` for the team's games.

These three passes together ensure R8-P1-2 (admin delete FK violation when deleting a team that owns reconciliation rows) cannot reach the `DELETE FROM teams` step with unreferenced FK survivors.

**Why this story still exists.** E-221-05's test coverage for reconciliation_discrepancies is transitive: `tests/test_admin_delete_cascade.py::TestCascadePreservesOtherPerspectiveRows::test_delete_team_cascade_preserves_other_perspective_rows` (the E-221-04 RED test) seeds reconciliation rows for both team A and team B perspectives (lines 1148-1154 and 1199-1205) and asserts correct survival/deletion as part of its cross-perspective preservation check. But there is no standalone test specifically for "admin delete of a team that owns reconciliation_discrepancies rows in the simple single-perspective case" — the R8-P1-2 surface in isolation.

Epic TN-9 is explicit: *"Every bug-fix story (05, 06, 07) MUST add its RED test BEFORE the fix, run the suite to see it fail, then land the fix and see it pass. This is the round 7 pattern that proved the fix addresses the bug and not a symptom."* E-221-06 was specifically listed as a bug-fix story requiring RED-test discipline. Transitive coverage via the cross-perspective test is not the same as an explicit R8-P1-2 regression gate.

**RED-vs-regression framing.** In strict TN-9 terms, a RED test is "write the test, watch it fail against pre-fix code, then land the fix and watch it pass." Under this reframe, the test lands against already-fixed code and passes immediately — it is technically a **regression test**, not a RED test. The red-then-green transition happened historically in E-221-05's dispatch (where SE watched the E-221-04 RED test transition from fail to pass). This story captures the R8-P1-2 bulwark as a regression test with the red-then-green history preserved in E-221-05's History entries, rather than replaying the ceremony against code that is already correct. The bulwark's function is identical; only the historical framing differs.

## Historical Record (original pre-reframe spec, preserved 2026-04-13)

The sections below preserve the original E-221-06 charter as written on 2026-04-09. They are retained for provenance so future readers can see both the original charter and the post-E-221-05 reframe. The content above this line is authoritative.

### Original title
> E-221-06: R8-P1-2 Phase 1b reconciliation_discrepancies cleanup

### Original Description
> After this story is complete, `_delete_team_cascade` Phase 1b will include a DELETE against `reconciliation_discrepancies` scoped by `perspective_team_id = team_id`. Currently Phase 1b cleans 5 tables this way but omits the sixth (reconciliation_discrepancies, added during E-220 round 7), which causes team deletes to fail with a FK violation when the deleted team owns reconciliation rows.

### Original Context
> E-220 round 7 added `perspective_team_id INTEGER NOT NULL REFERENCES teams(id)` to `reconciliation_discrepancies`. This makes the column part of the FK graph — if you delete a team row, any reconciliation_discrepancies rows pointing to it via `perspective_team_id` block the delete with a FK violation.
>
> `_delete_team_cascade` Phase 1b currently cleans `plays`, `spray_charts`, `player_game_batting`, `player_game_pitching`, and (one other) via `WHERE perspective_team_id = ?`. The sixth table (`reconciliation_discrepancies`) was added AFTER Phase 1b's current implementation and was never backfilled.
>
> Impact: admin delete fails silently (exception caught and logged, same pattern as round 7's P1-1 pre-fix for the reports path) any time the deleted team has reconciliation rows.
>
> This is a one-line fix (one additional DELETE statement) plus a RED test.

### Original Acceptance Criteria
> - AC-1: Given a team has rows in `reconciliation_discrepancies` with `perspective_team_id = T`, when `_delete_team_cascade(conn, T, ...)` is called, then the cascade completes without raising `sqlite3.IntegrityError`, and all the team's reconciliation_discrepancies rows are deleted.
> - AC-2: Given a team with no reconciliation_discrepancies rows, when `_delete_team_cascade` runs, its behavior is unchanged. The new DELETE executes as a no-op.
> - AC-3: A RED test is added before the fix that demonstrates the FK violation. The test MUST use `load_real_schema(db)` from Story 01 so that FK enforcement is active (without FK enforcement, the bug cannot be reproduced).
> - AC-4: The RED test asserts both: (a) the cascade completes without exception, and (b) the reconciliation rows for the deleted team's perspective are gone. The test passes only after the fix lands.
> - AC-5: The fix is a single new DELETE statement in Phase 1b, added alongside the existing 5. No refactoring of Phase 1b's structure.

### Original Technical Approach
> Per TN-6 in the epic. The fix is:
>
> ```sql
> DELETE FROM reconciliation_discrepancies WHERE perspective_team_id = ?
> ```
>
> Added to Phase 1b in the existing order of DELETEs (alongside the other 5 tables). Bind parameter is the team_id being deleted.
>
> The RED test seeds a team and a minimal game, loads reconciliation_discrepancies rows with `perspective_team_id = team_id`, then calls `_delete_team_cascade`. Pre-fix: IntegrityError. Post-fix: succeeds and rows are gone.
