## Description
After this story is complete, `src/api/routes/admin.py::_delete_team_cascade` correctly handles cross-perspective games during a single-team delete by **delegating cleanup to the canonical helper** in `src/reports/generator.py::cascade_delete_team`. The admin-specific wrapping (cross-perspective probe, informed-consent exception, audit log) is preserved; the per-phase DELETE blocks are removed in favor of the single canonical path. The three failing tests in `tests/test_admin_delete_cascade.py` all go GREEN. R8-P1-1 is closed and IDEA-069 (cascade consolidation) is absorbed into this story's scope.

## Context
Round 8's R8-P1-1 finding: Phase 1a of `_delete_team_cascade` cascades stat-table DELETEs by `game_id IN (...)` — deleting every row for that game, regardless of which perspective tagged it. This is a round 6 incompleteness: cluster 3 added the informed-consent UI, cluster 2 added game-row preservation, but Phase 1a was never updated to scope by `perspective_team_id`.

When an admin deletes team A (informed-consent confirmation accepted), Phase 1a currently wipes ALL stat rows for any game team A participated in, including rows tagged with team B's perspective. Team B is an unrelated tracked team that happened to play team A. Its scouting data for those shared games vanishes.

Story 04 lands the RED test that catches this. This story lands the fix.

**Scope correction (2026-04-12, discovered during E-221-04 dispatch)**: TN-5 originally claimed that "game-level cleanup was already correct from round 6 cluster 2." Investigation during Story 04 showed that comment applied to `cascade_delete_team` in `src/reports/generator.py`, NOT to `_delete_team_cascade` in `src/api/routes/admin.py`. The admin cascade's Phase 2 (around `admin.py:1024-1028`) unconditionally deletes the games row without a `NOT EXISTS (game_perspectives)` guard. Story 05's scope is therefore expanded to fix Phase 2 in addition to the original Phase 1a fix. Both E-221-04's AC-1 (Phase 1a) and AC-2 (Phase 2) assertions will be exercised by the same RED test.

### Scope Expansion 2 — 2026-04-13 — Option 2 Refactor-delegate

After the 2026-04-13 session restart, SE re-ran the full test suite against the checkpointed state and empirically discovered that **three tests fail in `tests/test_admin_delete_cascade.py`, not one**. The prior checkpoint handoff doc's fixture-anchoring diagnosis (claiming the RED test's fixture set `team_id = team_a_id` incorrectly on team B's perspective rows) was wrong. SE verified on disk that the fixture seeds are already semantically correct — `team_id = team_b_id, perspective_team_id = team_b_id` for team B's scouting records. CR independently verified all four empirical claims (fixture state, pytest output, admin.py code state, generator.py helper contract).

**Real root cause**: The three failures all trace to missing production logic in `src/api/routes/admin.py::_delete_team_cascade`, not fixture bugs:

1. **Missing teams-row retention logic**: When another perspective still owns `game_perspectives` rows referencing the team being deleted, the teams row cannot be deleted (FK violation from perspective-tagged rows anchored via `team_id`). The admin cascade has no teams-row retention branch for this case.
2. **Missing cross-perspective orphan cleanup**: The admin cascade's in-place Phase 1a+2 fix (from 2026-04-12's Scope Expansion 1) still does not match the canonical pattern in `src/reports/generator.py::cascade_delete_team`, which correctly handles orphan cleanup through `_delete_game_scoped_data_for_perspectives` and related helpers.

**User decision (2026-04-13)**: **Option 2 — refactor-delegate**. Rather than continue patching `admin.py` in place (Option 1, which had already required two scope expansions), absorb IDEA-069 (cascade consolidation) into E-221-05 and replace `_delete_team_cascade`'s in-place cleanup phases with a single delegation call to `src/reports/generator.py::cascade_delete_team`. The admin route retains the thin wrapping: probe for cross-perspective sharing, raise the informed-consent exception when applicable, emit the audit log entry, then delegate. This creates a single source of truth for cascade delete logic and eliminates the drift risk class that caused R8-P1-1 in the first place. IDEA-069 is PROMOTED into this story rather than carried forward as a follow-up.

## Technical Approach
Per TN-5 in the epic. Two options:

**Option A (preferred if reuse is clean)**: Replace Phase 1a's DELETEs with a call to `_delete_game_scoped_data_for_perspectives(conn, game_ids, [team_id])`. That helper already handles per-perspective scoping correctly.

**Option B**: Add `AND perspective_team_id = ?` to each DELETE in Phase 1a, mirroring how round 6 cluster 3 scoped Phase 1b.

Consult DE via SendMessage before choosing. The helper at `generator.py:1306-1389` was written for the report cleanup path — verify it handles the admin cascade context correctly (specifically: does it call any report-specific state that would break in the admin path?). If reuse is clean, go with A. If not, go with B.

**Do NOT** touch the game_perspectives cleanup or games-row preservation — those are already correct.

**Technical Approach (updated 2026-04-12, scope expansion)**: Two fixes in `src/api/routes/admin.py::_delete_team_cascade`:

1. **Phase 1a perspective scoping (R8-P1-1)**: Scope each DELETE to the deleted team's perspective. The current code at roughly `admin.py:955-990` uses `DELETE FROM <table> WHERE game_id IN (SELECT game_id FROM games WHERE home_team_id = ? OR away_team_id = ?)` with no perspective filter. Change each DELETE to add `AND perspective_team_id = ?` (bound to the deleted team's id). Applies to: `plays`, `spray_charts`, `player_game_batting`, `player_game_pitching`, `reconciliation_discrepancies`. Do NOT touch Phase 1b (it already filters correctly from round 7).

2. **Phase 2 games-row conditional DELETE**: The current code at roughly `admin.py:1024-1028` does `DELETE FROM games WHERE home_team_id = ? OR away_team_id = ?` without any game_perspectives guard. Change this to add a `NOT EXISTS (SELECT 1 FROM game_perspectives gp2 WHERE gp2.game_id = games.game_id)` clause so the games row is preserved when another perspective still owns data for the shared game. Pattern reference: `src/reports/generator.py::_delete_game_scoped_data_for_perspectives` (lines 1306-1390). Do NOT refactor admin.py to delegate to the generator.py helper — that cleaner refactor is deliberately deferred to a follow-up idea (IDEA-069). Story 05 keeps the fix in-place so it stays surgical.

Implementation note: the `game_perspectives` delete for team A happens in Phase 1b (existing round 7 behavior, correct). By the time Phase 2 runs, `game_perspectives` has already been trimmed to exclude team A. So the NOT EXISTS subquery correctly evaluates "is there any remaining perspective other than team A that still owns this game?"

### Option 2 (selected 2026-04-13): Refactor-delegate

**Supersedes Options A/B/C above.** After Option C's in-place fix from 2026-04-12 proved insufficient (three failing tests, not one; missing teams-row retention; missing cross-perspective orphan cleanup), the user authorized the refactor-delegate path. IDEA-069 is absorbed into this story.

**Strategy**:
1. **Keep in `admin.py`** (the thin admin wrapper):
   - The `confirm_cross_perspective` probe that decides whether to raise the informed-consent exception.
   - The informed-consent exception raise (raised by `_delete_team_cascade` when cross-perspective sharing exists and the caller has not yet confirmed).
   - The audit log entry (operator identity, deleted team id, timestamp).
   - A single delegation call to `src/reports/generator.py::cascade_delete_team` for the actual cleanup.

2. **Remove from `admin.py`** (move to canonical helper):
   - Phase 1a stat-table DELETE blocks (`plays`, `spray_charts`, `player_game_batting`, `player_game_pitching`, `reconciliation_discrepancies`).
   - Phase 1b `game_perspectives` DELETE and the 5-table perspective-scoped cleanup.
   - Phase 2 shared games-row DELETE (and its `NOT EXISTS` guard).
   - Phase 3 and Phase 4 team/roster cleanup blocks (whatever remains after Phase 1-2 in the current admin code).

3. **Potentially extend** `src/reports/generator.py::cascade_delete_team` to support the admin caller:
   - If the reports-path signature already handles every admin requirement, delegation is a straight call.
   - If the reports path is scoped to a single game and the admin path needs team-wide scope, extend the canonical helper to accept either scope, or add a thin sibling (e.g., `cascade_delete_team_all_games`) that reuses `_delete_game_scoped_data_for_perspectives`.
   - SE determines the minimum extension during implementation; CR verifies no divergent code paths are re-introduced.

**Consequence for AC-5**: The "do NOT modify Phase 1b" constraint in the prior AC-5 is retired. Phase 1b's DELETE blocks move out of `admin.py` as part of the delegation handoff — they do not survive in the admin route.

**Consequence for AC-6**: The `NOT EXISTS (game_perspectives)` guard is no longer added inside `admin.py`. It is inherited from the canonical helper via delegation.

**IDEA-069 absorption**: The cascade consolidation refactor that was deferred to IDEA-069 (CANDIDATE) on 2026-04-12 is now load-bearing work inside E-221-05.
