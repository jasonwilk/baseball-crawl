# E-221-02: Migrate 7 inline-fixture test files to helper

## Epic
[E-221: Test Fixture Schema Parity Audit + Post-E-220 Perspective Residuals](../E-221-perspective-residuals-and-fixture-audit/epic.md)

## Status
`DONE`

## Description
After this story is complete, all 7 test files that currently define `games` and related tables inline will use the shared `load_real_schema(conn)` helper from Story 01 instead. Tests in these files will run against a schema identical to production, with foreign key enforcement active, catching schema violations that the old inline fixtures silently hid.

## Context
Round 7 of E-220 identified 7 files with inline `CREATE TABLE` statements whose definitions drifted from `migrations/001_initial_schema.sql`. The most severe example: `tests/test_report_generator.py:97-109` declared `games.home_team_id` and `games.away_team_id` as plain `INTEGER` instead of `INTEGER NOT NULL REFERENCES teams(id)`. This is why round 7's P1-1 regression (a FK violation in `cascade_delete_team`) was not caught by the existing test suite — the fixture said FK violations were impossible.

This story migrates every inline fixture to the shared helper. The migration will surface additional test hygiene bugs similar to the round 7 P2-cascade findings (5 tests with hardcoded sentinel team_ids like `999` that were never seeded). Those fixes are part of this story's scope — if migrating a fixture breaks a test because the test was asserting schema-invalid state, fix the test to operate on valid state.

**Target files** (7):
1. `tests/test_report_generator.py` — LARGEST, 41 inline CREATE TABLE statements, migrate LAST per TN-2
2. `tests/test_e211_report_generator.py`
3. `tests/test_game_ordering.py`
4. `tests/test_gc_uuid_resolver.py`
5. `tests/test_pitching_workload.py`
6. `tests/test_player_dedup.py`
7. `tests/test_reconciliation.py`

## Acceptance Criteria
- [ ] **AC-1**: Given each of the 7 target files, when the file's inline `CREATE TABLE` statements are replaced with `load_real_schema(db)`, then every test in that file passes. Tests that fail due to schema tightening must be fixed (seed missing FK targets, remove sentinel values, correct assertions against now-enforced columns) per Technical Notes.
- [ ] **AC-2**: Given the 7 files after migration, when `grep -rn "CREATE TABLE" tests/` is run, then no results are returned from any file listed above. The only remaining `CREATE TABLE` in `tests/` (if any) lives in files explicitly out of scope for this story (Story 03 will lock this down).
- [ ] **AC-3**: Given the full test suite is run after migration, the baseline is 72F/(4254+N)P/16E where N is the count of any new tests added during cleanup. No NEW failures beyond the existing 72. No tests that previously passed now fail.
- [ ] **AC-4**: Given `test_report_generator.py` is the last file migrated, when its migration completes, then its current post-round-7 state (65P/0F on the subset exercised by E-220 round 7) is preserved or improved.
- [ ] **AC-5**: The migration is done file-by-file per TN-2, not all 7 at once. For each file, the test file is fully green before moving to the next. If a commit granularity is used, each file's migration is a distinct change (makes bisect tractable if a regression appears later).

## Technical Approach
Per TN-2 in the epic. For each file:
1. Identify the inline schema block and replace with `load_real_schema(db)`.
2. Walk every INSERT in the file. For each one, check whether it references a column that the real schema requires as NOT NULL. If so, supply a value.
3. For hardcoded sentinel team_ids (round 7 pattern), seed a real opponent team and substitute its id.
4. Run the file's tests. Fix any failures by correcting the test to operate on schema-valid state. Do NOT fix by relaxing the schema — the whole point is that the schema is now enforced.
5. Move to the next file only after the current file is green.

Leave `test_report_generator.py` for last. Its inline `reconciliation_discrepancies` fixture is severely drifted and will surface the most cascade failures.
