# Code Review: Web Application & Auth

**Reviewer**: code-reviewer agent
**Date**: 2026-03-17
**Scope**: `src/api/`, auth templates, admin routes, dashboard routes, and all corresponding test files

## Critical Issues

### C1: No CSRF protection on any POST route
**Files**: `src/api/routes/admin.py`, `src/api/routes/auth.py`

The application uses cookie-based sessions with server-rendered HTML forms but has zero CSRF protection. Every state-mutating POST endpoint is vulnerable:
- `POST /admin/users` (create user)
- `POST /admin/users/{id}/edit` (modify team assignments)
- `POST /admin/users/{id}/delete` (delete user and cascade auth artifacts)
- `POST /admin/teams` (add team)
- `POST /admin/teams/confirm` (create team in DB)
- `POST /admin/teams/{id}/toggle-active` (deactivate team)
- `POST /admin/teams/{id}/discover-opponents` (trigger external API calls)
- `POST /admin/opponents/{id}/connect` (link opponent)
- `POST /admin/opponents/{id}/disconnect` (unlink opponent)
- `POST /auth/login` (issue magic link)

An attacker could craft a page that submits a form to any of these endpoints while a logged-in admin visits it. The `SameSite=Lax` cookie attribute mitigates cross-site POST attacks in modern browsers (Lax blocks cross-origin POST), but this is a defense-in-depth concern -- older browsers and edge cases (e.g., top-level navigations) may still be vulnerable. The standard mitigation is a CSRF token in each form.

**Severity**: Critical for the admin delete-user and modify-team endpoints, which perform destructive operations. The `SameSite=Lax` cookie setting provides partial mitigation.

### C2: Cascade delete omits `coaching_assignments` table -- FK violation
**File**: `src/api/routes/admin.py:279-294`

`_delete_user()` deletes from `user_team_access`, `sessions`, `magic_link_tokens`, and `passkey_credentials` before deleting from `users`. However, the `coaching_assignments` table also has `user_id INTEGER NOT NULL REFERENCES users(id)`. If a user has coaching assignments and foreign key enforcement is on (`PRAGMA foreign_keys=ON`), the `DELETE FROM users` will raise an `IntegrityError`, causing the entire delete to fail silently (the function does not handle this exception for the caller).

This is a data integrity bug: either the delete fails unexpectedly, or (if FK enforcement is off in the connection) it leaves orphaned `coaching_assignments` rows.

**Fix**: Add `conn.execute("DELETE FROM coaching_assignments WHERE user_id = ?", (user_id,))` before the `DELETE FROM users` statement.

### C3: XSS in JavaScript context -- admin user delete confirmation
**File**: `src/api/templates/admin/users.html:57`

```html
onsubmit="return confirm('Delete {{ user.email }}? This cannot be undone.');"
```

The `{{ user.email }}` value is Jinja2 auto-escaped for HTML context, but it appears inside a JavaScript string within an `onsubmit` HTML attribute. Jinja2's autoescaping produces HTML entities (`&amp;`, `&lt;`) which are decoded back by the browser's HTML parser before being passed to the JS engine. A crafted email containing `'); alert('xss` would be HTML-safe but would break out of the JS string after HTML-to-JS processing.

**Practical risk**: Low, because emails are admin-controlled and normalized to lowercase. But the pattern is architecturally wrong -- user-controlled data in inline JS event handlers should use `|tojson` filter or be passed via `data-*` attributes.

## High Priority

### H1: In-memory passkey challenge stores do not survive restarts and break with multiple workers
**File**: `src/api/routes/auth.py:81-88`

`_PASSKEY_LOGIN_CHALLENGES` and `_PASSKEY_REG_CHALLENGES` are module-level dicts. The code comments acknowledge the single-worker constraint but there is no enforcement. If the app is started with `--workers 2`, passkey login will silently fail (challenge generated by worker A, verified by worker B -- not found). There is also no upper bound on dict size; a malicious actor could exhaust memory by repeatedly requesting `/auth/passkey/login/options` without verifying.

**Recommendation**:
1. Add a size cap to the challenge dicts (e.g., max 1000 entries, evict oldest on overflow).
2. Add a startup check or configuration guard that warns/fails if `--workers > 1` is detected.
3. Migrate to SQLite-backed challenge storage (as the comments suggest).

### H2: No session garbage collection
**Files**: `src/api/auth.py`, `src/api/db.py`

Expired sessions are never cleaned up from the `sessions` table. The `_resolve_session_from_cookie` function filters by `expires_at > datetime('now')` so expired sessions are ignored, but the rows accumulate indefinitely. Over time (months/years of operation), the table will grow unboundedly.

Similarly, expired `magic_link_tokens` rows are only cleaned up when a new token is issued for the same user (the `DELETE FROM magic_link_tokens WHERE user_id = ?` statement). Tokens for users who never request a second magic link remain forever.

**Recommendation**: Add a periodic cleanup job or on-request cleanup (e.g., delete expired sessions/tokens on every Nth request, or via a `bb db cleanup` command).

### H3: `_is_excluded_path` uses prefix matching with `/health` -- overly broad
**File**: `src/api/auth.py:44`

`_EXCLUDED_PREFIXES` includes `"/health"` as a bare prefix. This means any path starting with `/health` bypasses auth -- including hypothetical future routes like `/health-dashboard` or `/healthcheck`. The intent is to exclude exactly `/health`.

**Fix**: Change to `"/health/"` in `_EXCLUDED_PREFIXES` and add `"/health"` to `_EXCLUDED_EXACT`, or use exact matching for the health endpoint.

### H4: `email.py` uses raw `httpx.AsyncClient()` without configured headers
**File**: `src/api/email.py:59`

The Mailgun API call uses `httpx.AsyncClient()` with no User-Agent or other headers configured. While the HTTP discipline rules (`.claude/rules/http-discipline.md`) are scoped to `src/gamechanger/**` paths and don't technically apply to Mailgun calls, using the bare httpx client sends `python-httpx/x.x.x` as the User-Agent. This is acceptable for the Mailgun API specifically (it's a first-party API call, not a third-party site impersonation scenario), but is worth noting for consistency.

### H5: Duplicate schema SQL across test files
**Files**: `tests/test_auth.py`, `tests/test_auth_routes.py`, `tests/test_passkey.py`, `tests/test_admin.py`, `tests/test_dashboard.py`

The E-100 schema SQL is copy-pasted verbatim across at least 5 test files (`_SCHEMA_SQL` constant). Each copy must be kept in sync with the actual migration. Some test files use `run_migrations()` directly (e.g., `test_dashboard_auth.py`, `test_coaching_assignments.py`), which is the correct approach. The copy-paste schema approach is fragile -- if the schema changes, some test files may be updated and others missed, leading to false passes against stale schema.

Note: several test files that use the copy-paste schema omit certain tables (e.g., `coaching_assignments` is missing from most test schemas). This is acceptable when the test doesn't need those tables, but creates drift risk.

**Recommendation**: Migrate all test files to use `run_migrations()` (as `test_dashboard_auth.py` already does), or extract the schema SQL into a shared test fixture.

### H6: `get_team_batting_stats` silently returns empty list on DB error
**File**: `src/api/db.py:111-113`

All query functions in `db.py` catch `sqlite3.Error` and return empty results (`[]`, `{}`, `None`). While this prevents crashes, it means the dashboard silently shows "no data" when there is a database error. The user has no way to distinguish "the team genuinely has no stats" from "the database is broken." The health check endpoint still functions independently, but a query-level error (e.g., schema drift, disk full) would produce a confusing silent failure on the dashboard.

**Recommendation**: Consider returning a distinct error indicator (e.g., `None` vs `[]`) so the route handler can render an appropriate error message, at least for the primary dashboard queries.

## Medium Priority

### M1: `_compute_pitching_rates` duplicated three times
**File**: `src/api/routes/dashboard.py:152-180`, `396-425`, `782-807`

The ERA/K9/BB9/WHIP computation logic is implemented in three nearly identical functions:
- `_compute_pitching_rates` (team pitching page)
- `_compute_opponent_pitching_rates` (opponent scouting page -- adds `avg_pitches`)
- `_compute_player_pitching_rates` (player profile page)

All three use the same formula: `(er * 27) / ip_outs` for ERA, `(so * 27) / ip_outs` for K/9, `(bb + h) * 3 / ip_outs` for WHIP. The only difference is that `_compute_opponent_pitching_rates` also computes `avg_pitches`.

**Recommendation**: Extract a single `_compute_base_pitching_rates(pitchers)` function and call it from each location, adding the `avg_pitches` computation only where needed.

### M2: Team ID authorization check repeated in every dashboard route
**File**: `src/api/routes/dashboard.py` -- lines 88-98, 222-232, 331-341, 508-518

The pattern of parsing `team_id` from query params, validating as int, checking against `permitted_teams`, and defaulting to `permitted_teams[0]` is duplicated across all four list-view routes (`team_stats`, `team_pitching`, `game_list`, `opponent_list`). Each copy uses slightly different variable names (`active_team_id`, `active_team_id_p`, `active_team_id_g`, `active_team_id_o`) to avoid collision.

**Recommendation**: Extract a `_resolve_active_team(request, permitted_teams) -> int | Response` helper that returns either the validated team ID or an error response.

### M3: `_check_opponent_authorization` and `_check_player_authorization` use inline imports
**File**: `src/api/routes/dashboard.py:669-670`, `823-824`

Both functions contain `import sqlite3 as _sqlite3` and `from contextlib import closing as _closing` inside the function body. These modules are already available at module scope (used elsewhere in the file). The inline imports appear to be a workaround for something, but there is no comment explaining why.

**Recommendation**: Use the module-level imports instead of inline imports.

### M4: No test for `_delete_user` with coaching_assignments present
**File**: `tests/test_admin.py`

The `TestCascadeDelete` class tests that deleting a user removes sessions, magic link tokens, and passkey credentials. However, there is no test for the case where the user has `coaching_assignments` rows -- which, per finding C2 above, would actually cause a failure.

### M5: `format_avg` treats negative denominators the same as zero
**File**: `src/api/helpers.py:67`

`if not denominator:` returns `-` for both zero and negative values. While negative at-bats are unlikely, using `if denominator is None or denominator == 0:` would be more explicit and correct.

### M6: Stale expired magic link tokens not cleaned up on verification failure
**File**: `src/api/routes/auth.py:415-423`

When a token is expired, the code deletes it and commits, which is good. But this only happens when the specific expired token is presented. Expired tokens for other users remain indefinitely. This is a minor data hygiene issue (see H2).

### M7: `get_player_profile` recent games deduplication logic could lose pitching data
**File**: `src/api/db.py:749-752`

The deduplication logic for recent games prefers the batting row over the pitching row when a player has both for the same game:
```python
if gid not in games_by_id or row["appearance_type"] == "batting":
    games_by_id[gid] = row
```
This means pitching appearance data (ip_outs, er, so_pitched) is lost for two-way players in their recent games. A more complete approach would merge both rows into one combined dict per game.

## Low Priority

### L1: `_SESSION_COOKIE_NAME` and `_SESSION_MAX_AGE` defined in two places
**Files**: `src/api/auth.py:39-41`, `src/api/routes/auth.py:68-69`

The session cookie name (`"session"`) and max age (`604800`) are defined as constants in both the middleware module and the auth routes module. If one is changed without the other, session handling would break.

**Recommendation**: Import from a single location.

### L2: `test_api_health.py` uses `sys.path` manipulation
**File**: `tests/test_api_health.py:24-26`

Most test files use `sys.path.insert(0, str(_PROJECT_ROOT))` for path setup, which is acceptable in test files per project conventions. However, this is noted as a pattern -- the editable install should make this unnecessary.

### L3: `format_date` uses `%-d` which is platform-dependent
**File**: `src/api/helpers.py:99`

`dt.strftime("%b %-d")` uses `%-d` for day without leading zero, which is a Linux/macOS extension and not available on Windows. This is fine for the Docker deployment target but worth noting for development portability.

### L4: `get_teams_by_ids` parameter type annotation says `list[int]` but receives it from template context
**File**: `src/api/db.py:116-138`

The function signature expects `list[int]`, but the `conn.execute(query, team_ids)` call passes the list directly. SQLite's `?` placeholder binding requires the list to be a tuple or list of individual values. This works correctly because `team_ids` is already a list, but the intermediate `placeholders` construction with `f"SELECT ... WHERE id IN ({placeholders})"` is a safe dynamic SQL pattern (not user-controlled).

### L5: `_APP_URL_DEFAULT` hardcodes port 8001
**File**: `src/api/routes/auth.py:70`

The default APP_URL is `http://baseball.localhost:8001`, but the Dockerfile comment and app factory reference port 8000. This may cause confusion if APP_URL is not set and the app runs on the default uvicorn port.

### L6: `500` exception handler may not fire for all 500 errors
**File**: `src/api/main.py:117-122`

The `@app.exception_handler(500)` only catches unhandled exceptions that FastAPI converts to 500 responses. If middleware raises an exception that Starlette catches before FastAPI, the custom 500 template won't render.

## Positive Observations

### P1: Solid session security model
The session architecture is well-designed: tokens are hashed with SHA-256 before storage, cookies are `HttpOnly` + `SameSite=Lax`, session creation uses `secrets.token_hex(32)` (64 hex characters = 256 bits of entropy), and the DEV_USER_EMAIL bypass has a production guard that raises `RuntimeError` if `APP_ENV=production`.

### P2: User enumeration prevention
The magic link login flow renders the same "If this email is registered..." page for both known and unknown email addresses, preventing email enumeration attacks. This is correctly tested.

### P3: Comprehensive auth test coverage
The auth test suite covers valid sessions, invalid sessions, expired sessions, dev bypass, production guard, missing auth tables (503 fail-closed), rate limiting, stale token invalidation, cookie properties, passkey registration/login flows, and logout. This is thorough.

### P4: Team access control with proper 403 handling
Dashboard routes consistently validate `team_id` against `permitted_teams` and return 403 for unauthorized access. The game detail and opponent detail routes have additional authorization checks (game must involve a permitted team, opponent must appear in games for a permitted team).

### P5: Single-use magic link enforcement with TOCTOU awareness
The magic link verification flow deletes the token row (single-use enforcement via DELETE) and checks `cursor.rowcount == 0` to detect race conditions where another request consumed the token between SELECT and DELETE. This is a good defensive pattern.

### P6: XSS escaping verified by test
`tests/test_admin.py::TestXSSEscaping::test_msg_param_is_escaped_in_team_list` explicitly tests that query parameter values are HTML-escaped in the response, confirming Jinja2 auto-escaping is working.

### P7: Clean `run_in_threadpool` usage for sync DB calls
All synchronous SQLite operations in async route handlers are properly wrapped in `run_in_threadpool()` to avoid blocking the event loop. This is consistently applied across all dashboard and admin routes.

### P8: Graceful degradation on missing auth tables
The session middleware handles `OperationalError("no such table")` by returning 503 instead of crashing or silently allowing access. This is properly tested with dedicated fixtures.
