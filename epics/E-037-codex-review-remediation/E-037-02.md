## Description
After this story is complete, AC-4 in both E-002-07a and E-002-07b will describe an orphan-player handling strategy that is compatible with SQLite FK enforcement. The current ACs are impossible to implement because they require inserting stat rows for player IDs that do not exist in the `players` table, but FK constraints reject such inserts.

## Context
The schema in `migrations/001_initial_schema.sql` defines FK constraints on `player_id` in `player_game_batting`, `player_game_pitching`, `player_season_batting`, and `player_season_pitching`. The `get_connection()` function in `src/api/db.py` enables `PRAGMA foreign_keys=ON`. Both E-002-07a AC-4 and E-002-07b AC-4 say "the stat row is still written" for orphaned player IDs -- this will raise an `IntegrityError` at runtime.

The fix: rewrite AC-4 in both stories to require inserting a stub `players` row (with `first_name='Unknown'`, `last_name='Unknown'`) before inserting the stat row. This preserves referential integrity while not losing data. A WARNING is still logged.

## Acceptance Criteria
- [ ] **AC-1**: E-002-07a AC-4 is rewritten to state: "Given a game record where a player_id does not exist in the `players` table, the loader inserts a stub player row (`player_id`, `first_name='Unknown'`, `last_name='Unknown'`) before inserting the stat row, and logs a WARNING including the player_id. The stat row is written successfully. (This maintains FK integrity while preserving all stat data. Stub players are backfilled by a later roster crawl.)"
- [ ] **AC-2**: E-002-07b AC-4 is rewritten with the same stub-player strategy: "Given a player_id that does not exist in the `players` table, the loader inserts a stub player row (`player_id`, `first_name='Unknown'`, `last_name='Unknown'`) before inserting the season stat row, and logs a WARNING. (Same rationale as E-002-07a AC-4.)"

## Notes
- This story modifies spec files, not code. No tests to run.
- The stub-player pattern (`INSERT OR IGNORE INTO players ...` before the stat INSERT) is the simplest approach that satisfies FK constraints. The "Unknown/Unknown" name makes stub players easy to find with `SELECT * FROM players WHERE first_name = 'Unknown'`.
- The project's CLAUDE.md previously mentioned "Soft referential integrity in stats tables (orphaned player IDs accepted with WARNING)" in the Key Architectural Decisions. That CLAUDE.md note was about the aspiration, not the schema reality. The schema uses hard FKs, so the stub-player pattern is the correct implementation of the aspiration.
