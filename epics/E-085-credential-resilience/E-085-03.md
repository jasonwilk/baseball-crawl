## Description
After this story is complete, `bb status` will include a "Proxy (Bright Data)" section showing whether the IP anonymization proxy is configured and routing correctly for each profile. This completes the `bb status` command as a single-command system health dashboard.

## Context
`bb status` currently shows four sections: credentials (per-profile), last crawl, database, and proxy sessions (mitmproxy). It does not show Bright Data proxy connectivity status, which is only available via the separate `bb proxy check` command. Operators asking "is my setup working?" should get the full picture from `bb status` without needing to run a second command.

The proxy check logic already exists in `src/http/proxy_check.py` and is already used by the `bb proxy check` CLI command in `src/cli/proxy.py`.

## Acceptance Criteria
- [ ] **AC-1**: `bb status` output includes a "Proxy" section after the credentials section, showing per-profile Bright Data proxy status
- [ ] **AC-2**: For each profile where proxy is configured (`PROXY_ENABLED=true` + `PROXY_URL_{PROFILE}` set), the section shows routing status using `[OK]`/`[XX]`/`[!!]` indicators: `[OK]` (IP differs from direct), `[XX]` (IP matches direct), `[!!]` (error with brief description)
- [ ] **AC-3**: For profiles where proxy is not configured, the section shows "not configured" (informational, not an error)
- [ ] **AC-4**: The proxy section uses the shared visual language: `[OK]` green, `[!!]` yellow, `[XX]` red, `[--]` dim. Compact one-line-per-subsystem format consistent with `bb status` style (not the expanded panel view from `bb creds check`)
- [ ] **AC-5**: The proxy check does not block `bb status` from completing -- if the IP-echo service is unreachable, the section shows the error and the rest of the output is unaffected
- [ ] **AC-6**: No proxy URLs or credentials appear in the output -- only status indicators and anonymized IPs
- [ ] **AC-7**: All new code has test coverage. Tests mock the proxy check calls.
- [ ] **AC-8**: Existing `bb status` tests continue to pass

## Technical Approach
Reuse `check_proxy_routing()` and `get_direct_ip()` from `src/http/proxy_check.py`. The `bb proxy check` command in `src/cli/proxy.py` already demonstrates how to call these functions and format the results -- the pattern can be adapted for the `bb status` output format.

The proxy check makes real HTTP calls to an IP-echo service. This adds latency to `bb status`. The IP-echo call for direct IP is shared across profiles (call once, reuse), so the overhead is 1 direct call + 1 call per configured profile (typically 1-2 seconds total).

## Notes
- Consider ordering: credentials -> proxy -> last crawl -> database -> proxy sessions. Proxy connectivity is logically related to credentials (both are prerequisites for API access), so placing it after credentials makes sense.
- The `get_direct_ip()` call is the slowest part. If latency is a concern, it could be made concurrent with the credential checks, but serial is fine for an operator diagnostic tool.
- `ProxyCheckOutcome.PASS_UNVERIFIED` (proxy succeeded but direct baseline failed, so anonymization is unconfirmed) should render as `[!!]` (yellow) with message "routing unverified (direct baseline unavailable)". This aligns with the shared visual language in E-085-02.
