# E-085-01: TokenManager Login Fallback

## Epic
[E-085: Credential Resilience and Diagnostic Improvements](epic.md)

## Status
`TODO`

## Description
After this story is complete, `TokenManager` will automatically recover from expired refresh tokens by performing the full 3-step login flow (client-auth, user-auth, password) when a refresh attempt returns HTTP 401. The refresh-first strategy remains the primary path; login is the fallback. The login flow produces the same output as refresh (access + refresh tokens) and the rotated refresh token is persisted to `.env` via the existing atomic write-back mechanism.

## Context
The 4-step login flow was confirmed working programmatically on 2026-03-09 (step 1, logout, is skipped for the recovery use case since the refresh token is already expired). `signing.py` already supports signature chaining via the `previous_signature_b64` parameter. `TokenManager` currently raises `CredentialExpiredError` on HTTP 401 during refresh -- this story adds the login fallback before that error is raised.

The login flow requires two additional `.env` variables: `GAMECHANGER_USER_EMAIL` and `GAMECHANGER_USER_PASSWORD`. These are already documented in `docs/api/auth.md` but not currently consumed by `TokenManager`. When these variables are absent, the login fallback is not available and the existing `CredentialExpiredError` behavior is preserved (graceful degradation).

## Acceptance Criteria
- [ ] **AC-1**: Given a web profile `TokenManager` with valid email/password/client_id/client_key in `.env`, when `_do_refresh()` receives HTTP 401 (expired refresh token), then the login fallback executes steps 2-4 of the login flow and returns a valid access token
- [ ] **AC-2**: Given a successful login fallback, the new refresh token is persisted to `.env` via `_persist_refresh_token()` (same atomic write-back as the refresh path)
- [ ] **AC-3**: Given a successful login fallback, the new access token is cached in memory with correct expiry tracking (same as the refresh path)
- [ ] **AC-4**: Given a web profile `TokenManager` without email or password in `.env`, when `_do_refresh()` receives HTTP 401, then `CredentialExpiredError` is raised with a message explaining that login credentials are needed for auto-recovery
- [ ] **AC-5**: Given a mobile profile `TokenManager`, the login fallback is NOT attempted (mobile client key is unknown; existing behavior preserved)
- [ ] **AC-6**: Given a login fallback where any step (2, 3, or 4) returns a non-200 response, a `LoginFailedError` is raised with diagnostic context (which step failed, HTTP status). Given a 400 response (clock skew), `AuthSigningError` is raised instead.
- [ ] **AC-7**: Signature chaining works correctly across the 3-step login flow: step 2 uses `usePreviousSignature: false` (no `previous_signature_b64`), steps 3 and 4 chain from the previous response's `gc-signature` header
- [ ] **AC-8**: The step 2 response shape (`{type: "client-token", token: "...", expires: N}`) is handled correctly -- it differs from the refresh response shape (`{type: "token", access: {...}, refresh: {...}}`)
- [ ] **AC-9**: `TokenManager.__init__` accepts optional `email` and `password` parameters for the login fallback. These parameters are optional -- their absence does not prevent construction or the refresh-only path.
- [ ] **AC-10**: All new code has test coverage. Tests mock at the HTTP layer. The 3-step chain is tested end-to-end (mocked): step 2 returns client token + gc-signature header, step 3 returns gc-signature header, step 4 returns access + refresh tokens.
- [ ] **AC-11**: `force_refresh()` does NOT trigger login fallback -- it raises `CredentialExpiredError` on 401 as it does today (login fallback is only available through the `get_access_token()` initial fetch path)
- [ ] **AC-12**: If the `gc-signature` response header is absent on any login step, a WARNING is logged (indicates the chaining contract may have changed)
- [ ] **AC-13**: The step 2 response shape is validated (assert `type == "client-token"`) -- if the shape is unexpected, log an error and raise `LoginFailedError` with diagnostic context
- [ ] **AC-14**: Existing `TokenManager` tests continue to pass without modification

## Technical Approach
The login flow is documented in `docs/api/auth.md` under "Complete Login Flow (4 Steps)". The signing module at `src/gamechanger/signing.py` already supports the `previous_signature_b64` parameter needed for signature chaining. The `_PROFILE_HEADERS` dict in `token_manager.py` already has the profile-specific Content-Type headers.

Key constraint: the step 2 (client-auth) request does NOT send a `gc-token` header -- it is the only auth step without one. Steps 3 and 4 send the client token (from step 2's response) as `gc-token`.

Key constraint: the step 2 response shape is different from the refresh response. Step 2 returns `{type: "client-token", token: "...", expires: N}`. Step 4 returns the standard `{type: "token", access: {...}, refresh: {...}}`.

The `GameChangerClient` at `src/gamechanger/client.py` constructs `TokenManager` and passes credentials from `.env`. It will need to pass email and password through if they are present.

## Notes
- The step 2 `client-auth` body includes `client_id` as a field: `{"type": "client-auth", "client_id": "{CLIENT_ID}"}`. This is the same value as the `gc-client-id` header.
- Step 2 explicitly sets `usePreviousSignature: false` in the JS source -- meaning no `previous_signature_b64` argument to `build_signature_headers()`.
- Steps 3 and 4 both use the client token from step 2 as their `gc-token` header.
- The gc-signature response header format is `{nonce}.{hmac}` -- extract the part after the dot for the `previous_signature_b64` parameter.
- The step 3 response shape is not well-documented (it just confirms user identity). The response body is not needed -- only the `gc-signature` response header matters for chaining to step 4.
- Read `/workspaces/baseball-crawl/data/raw/auth-login-flow-findings.json` before implementing -- contains the observed response shapes from the 2026-03-09 login flow test.
- `LoginFailedError` recovery message should say "check email/password in .env" (distinct from `CredentialExpiredError`'s "run bb creds import").
