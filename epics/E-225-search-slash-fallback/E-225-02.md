# E-225-02: Migrate four call sites to shared helper

## Epic
[E-225: POST /search Punctuation Normalization Fallback](./epic.md)

## Status
`TODO`

## Description
After this story is complete, all four current `POST /search` by-team-name call sites will delegate the raw search step to `search_teams_by_name` from `src/gamechanger/search.py` (created in Story 1). The four duplicate `_SEARCH_CONTENT_TYPE` constants are removed. Each call site keeps its own downstream filtering logic (public_id match, name exact-match, season year match) unchanged. After this story, a user regenerating the standalone report for a team whose name contains a trigger character (e.g., Lincoln Northwest JV/Reserve Falcons, `public_id=yecaUcoSVpJa`) will see the `gc_uuid` resolved and spray charts populated.

## Context
Story 1 created the shared helper and proved its behavior in isolation (including the four confirmed trigger characters and the straight-vs-curly apostrophe trap). This story is the migration step: mechanically replace the four inline search calls with calls to the helper, delete the duplicate content-type constants, and add regression tests that exercise the punctuation-fallback path at the specific call sites confirmed to be user-impacting.

See epic Technical Notes TN-1 (helper contract), TN-4 (content-type consolidation), TN-7 (exact-string assertion requirement), TN-8 (Unicode apostrophe trap coverage), TN-9 (call-site filtering preserved), TN-9a (multi-page fallback semantics and short-circuit rationale), and TN-10 (`_gc_search_teams` shape preservation).

## Technical Approach
Four mechanical substitutions, each with its existing filtering preserved.

### Call site 1: `src/reports/generator.py::_resolve_gc_uuid`
- Remove `_SEARCH_CONTENT_TYPE` module-level constant.
- Inside the `for page in range(_SEARCH_MAX_PAGES)` loop, replace the `client.post_json(...)` block with a call to `search_teams_by_name(client, team_name, start_at_page=page)`.
- Unwrap the response: the helper returns `hits` directly as a list (not `result.get("hits", [])` -- the helper already unwraps). The rest of the loop body -- public_id match, `logger.info` on resolve, short-circuit on partial page -- is unchanged.
- The short-circuit `if len(hits) < _SEARCH_PAGE_SIZE: break` still works because the helper returns the raw hits list. Note: when the fallback fires, the returned hits are the normalized-attempt hits, not the first-attempt hits. The short-circuit semantics stay correct because either branch returns whatever the backend produced for the query that last ran.
- **Add the AC-1a short-circuit** per Technical Notes TN-9a: on page 0, if the helper returned zero hits AND `team_name` is not gate-clean (contains at least one `[^\w ]` character), `break` out of the pagination loop after the helper's own page-0 fallback has fired. This bounds the worst-case API cost at 2 calls per not-found paginated lookup (one raw, one normalized on page 0) instead of 10 (5 pages x 2 attempts). The gate-check can be implemented inline with `re.search(r"[^\w ]", team_name)` at the top of the loop or cached outside the loop.

### Call site 2: `src/gamechanger/resolvers/gc_uuid_resolver.py::_tier3_search`
- Remove `_SEARCH_CONTENT_TYPE` module-level constant.
- Replace the `client.post_json(...)` call with `hits = search_teams_by_name(client, shortened)`.
- Both downstream filter branches (public_id-preferred, season_year fallback) use `hits` directly.

### Call site 3: `src/gamechanger/crawlers/opponent_resolver.py::_search_resolve_opponent`
- Remove `_SEARCH_CONTENT_TYPE` module-level constant.
- Replace the `self._client.post_json(...)` call with `hits = search_teams_by_name(self._client, opponent_name)`.
- The downstream filter (name case-insensitive + season year match, single-match requirement) uses `hits` directly. Note: if the opponent name contains a straight apostrophe and the helper recovers hits via fallback, the returned `hit.result.name` may use GC's canonical form (typically curly apostrophe). The existing `name.lower() == opponent_name.lower()` comparison will therefore fail for that specific case -- this is a pre-existing edge case in the filter logic (called out in epic TN-9) and is explicitly out of scope for this story. Do NOT modify the filter. Leave it as-is; the epic's improvement is limited to recovering hits where none were recovered before. (This is why AC-6's regression test deliberately uses a `#`- or `%`-containing name rather than a straight-apostrophe name -- to exercise the fallback path on a trigger character that does not interact with the apostrophe-canonicalization edge case.)

### Call site 4: `src/api/routes/admin.py::_gc_search_teams`
- Remove `_SEARCH_CONTENT_TYPE` module-level constant.
- Replace the `client.post_json(...)` call with `hits = search_teams_by_name(client, name)`.
- The normalization loop over `hits` (building flat dicts with `name`, `gc_uuid`, `public_id`, `city`, etc.) is preserved verbatim.

### Tests
- **AC-1a short-circuit test**: Add a test to the report-generator test file that mocks `post_json` to return empty hits for ALL calls (both raw slash-form and space-normalized form) on page 0, and asserts `mock_post_json.call_count == 2` (NOT 10) after `_resolve_gc_uuid` returns None. This pins the short-circuit behavior -- removing the gate-clean-check would cause the count to climb to 10.
- **AC-5 regression test**: In the appropriate report-generator test file, add a test that patches `GameChangerClient.post_json` to return empty hits for any body where the name contains a `[^\w ]` character, and the canonical match payload otherwise. Invoke `_resolve_gc_uuid` with `team_name="Lincoln Northwest JV/Reserve Falcons"` and `public_id="yecaUcoSVpJa"`; assert it returns `"ac053e2c-ee27-4f55-9b16-ed77c1bdfebb"` AND assert exactly 2 `post_json` calls with body names `"Lincoln Northwest JV/Reserve Falcons"` and `"Lincoln Northwest JV Reserve Falcons"` respectively (per AC-5 exact-string requirement).
- **AC-6 regression test**: In `tests/test_crawlers/test_opponent_resolver.py`, add a parallel test at the `OpponentResolver._search_resolve_opponent` level (or one level up, whichever test scaffolding the file already uses). Use a `#`- or `%`-containing opponent name (e.g., `"Cornhusker #1 Varsity"`) to exercise a different trigger character than the report-generator regression. The mock returns empty hits for the raw `#`-containing name and the canonical payload for the space-normalized form (e.g., `"Cornhusker 1 Varsity"`). The returned `result.name` must be compatible with the existing `name.lower() == opponent_name.lower()` filter. Assert exactly 2 `post_json` calls with the exact body strings on each.
- **AC-7 regression test (tier3 compositional coverage)**: In `tests/test_gc_uuid_resolver.py`, add a test that invokes `_tier3_search` with a team name whose trigger character is mid-name (not a slash within a JV/Reserve classification suffix). Example: `"Acme%Baseball Varsity"` (the `%` is inside the "Acme" portion and survives `_strip_classification_suffix()`). Mock returns empty hits for the raw `%`-form and canonical payload for the space-normalized form. Assert exactly 2 `post_json` calls with the exact body strings (raw form after suffix-stripping, then space-normalized form).
- **Test scope discovery**: For each modified `src/` module, run `grep -rl <module-import-path> tests/` and run the discovered test files in addition to any named above.
