# E-204-03: GS/GR Display + Appearance-Order Sorting

## Epic
[E-204: Starter vs. Relief Appearance Tracking](epic.md)

## Status
`TODO`

## Description
After this story is complete, a combined GS/GR column (e.g., `8/5`) appears on all pitching season surfaces (per TN-4) in both dashboard and standalone reports, and game boxscore pitching lines are sorted by appearance order (starter first). The scouting aggregation pipeline computes GS from `appearance_order` for tracked teams. GR is derived as `g - gs` at the display layer. Delivery parity is maintained between dashboard and reports.

## Context
Member teams already have `player_season_pitching.gs` from the GC API (per TN-3). Scouting/tracked teams do not -- their GS must be computed from `appearance_order` in the scouting aggregation query. All pitching season surfaces (per TN-4) need the GS/GR column added per the format in TN-7. Game boxscores currently sort pitchers alphabetically within each team; sorting by `appearance_order` gives coaches the natural game flow view.

## Acceptance Criteria
- [ ] **AC-1**: The scouting aggregation query in `_compute_pitching_aggregates()` computes `gs` as the count of `player_game_pitching` rows where `appearance_order = 1`. The `gs` value appears in both the INSERT column list and the ON CONFLICT SET clause of the upsert into `player_season_pitching`.
- [ ] **AC-2**: The dashboard team pitching table (`/dashboard/teams/{id}/pitching`) displays a GS/GR column per TN-7 format: header `GS/GR`, value `{gs}/{g - gs}`, positioned after G and before IP.
- [ ] **AC-3**: The dashboard opponent detail and opponent print pages display GS/GR in their pitching tables, consistent with AC-2 format and positioning.
- [ ] **AC-4**: The standalone report pitching table displays a GS/GR column with correct values, consistent with dashboard format and positioning.
- [ ] **AC-5**: The game detail page (`/dashboard/games/{id}`) sorts pitching lines by `pgp.team_id, pgp.appearance_order ASC NULLS LAST, p.last_name` (preserving per-team grouping, with appearance order within each team).
- [ ] **AC-6**: Zero values display explicitly per TN-7: `0/5` for a pure reliever, `8/0` for a pure starter. When `gs IS NULL` for a pitcher, display `—`.
- [ ] **AC-7**: GR is derived at the display layer as `g - gs`, not stored as a separate column.

## Technical Approach
The display surfaces are inventoried in TN-4 and the format specification is in TN-7. Three independent query sites need `gs` added to their SELECT lists:

1. **Dashboard team pitching**: `get_team_pitching_stats()` in `src/api/db.py` -- add `gs` to SELECT (not currently included). Serves the `/dashboard/pitching` page.
2. **Dashboard opponent detail/print**: `get_opponent_scouting_report()` in `src/api/db.py` -- add `gs` to the pitching SELECT (not currently included). This is a separate function from `get_team_pitching_stats()`. Serves `/dashboard/opponents/{id}` and its print view.
3. **Standalone reports**: `_query_pitching()` in `src/reports/generator.py` -- add `gs` to SELECT (not currently included). Serves the standalone report pitching table.

GR is computed at the template/rendering layer as `g - gs`. Templates need a new `<th>GS/GR</th>` + `<td>` pair rendering the combined format per TN-7.

For game boxscores, the ORDER BY clause in `get_game_box_score()` changes from `pgp.team_id, p.last_name` to `pgp.team_id, pgp.appearance_order ASC NULLS LAST, p.last_name` -- preserving the per-team grouping while sorting by appearance order within each team.

The scouting aggregation in `_compute_pitching_aggregates()` adds a count expression for `appearance_order = 1` and includes `gs` in both the INSERT column list and the ON CONFLICT SET clause.
