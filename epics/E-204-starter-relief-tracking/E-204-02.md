# E-204-02: Backfill Existing Rows

## Epic
[E-204: Starter vs. Relief Appearance Tracking](epic.md)

## Status
`TODO`

## Description
After this story is complete, a CLI command (`bb data backfill-appearance-order`) walks cached boxscore JSON files on disk and updates existing `player_game_pitching` rows that have `appearance_order IS NULL`. Games without cached boxscore files are logged as warnings and skipped. The command is idempotent and re-runnable.

## Context
E-204-01 populates `appearance_order` for all new loads, but hundreds of existing `player_game_pitching` rows predate the column. The cached boxscore JSON files on disk contain the pitcher order information needed to backfill these rows. The reconciliation engine already has path resolution and pitcher order parsing logic (see TN-5) that can be reused.

## Acceptance Criteria
- [ ] **AC-1**: `bb data backfill-appearance-order` queries all distinct `(game_id, team_id)` pairs from `player_game_pitching` where `appearance_order IS NULL` and attempts to resolve cached boxscore JSON for each per TN-5 path patterns.
- [ ] **AC-2**: For each resolved boxscore, the script parses pitcher order (per TN-1 structure) and UPDATEs the corresponding `player_game_pitching` rows with correct `appearance_order` values.
- [ ] **AC-3**: The script is idempotent: only rows where `appearance_order IS NULL` are updated. Re-running after a successful run produces no changes.
- [ ] **AC-4**: Games without cached boxscore JSON are logged (WARNING level) with the game_id and team info, then skipped. The script continues processing remaining games.
- [ ] **AC-5**: The script prints a summary on completion: total games processed, rows updated, games skipped (no cached file), games with errors. The summary includes a reminder to re-run `bb data scout` to recompute scouting season aggregates (so tracked-team GS reflects the backfilled data).

## Technical Approach
The script needs to: (1) query NULL-appearance-order game/team pairs from the DB (joining through `games` for `season_id` and `teams` for `gc_uuid`/`public_id` needed for path resolution), (2) for each, find the cached boxscore JSON using the path patterns in TN-5, (3) parse pitcher order from the JSON `stats` array to get an ordered list of `player_id` values, (4) match each parsed `player_id` against `player_game_pitching.player_id` for the same `(game_id, team_id)` and UPDATE with the corresponding `appearance_order`. The reconciliation engine's `_extract_pitcher_order()` and `_parse_pitching_order()` handle steps 2-3 already -- consider importing them or extracting the logic into a shared utility if they are currently module-private.
