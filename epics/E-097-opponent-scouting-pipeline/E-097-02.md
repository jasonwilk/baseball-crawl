# E-097-02: Opponent Scouting Schema Migration

## Epic
[E-097: Opponent Scouting Data Pipeline](epic.md)

## Status
`TODO`

## Description
After this story is complete, the database will have a `scouting_runs` table that tracks when each opponent was last scouted, what data was collected, and whether the run succeeded. The table includes `first_fetched` and `last_checked` timestamps for re-fetch support. The existing data tables (`players`, `team_rosters`, `games`, `player_game_batting`, `player_game_pitching`, `player_season_batting`, `player_season_pitching`) already support opponent data -- no new stat tables are needed. This story adds only the crawl-metadata tracking layer with the fetch-timestamp pattern established for future use.

## Context
The existing schema was designed to hold data for both owned teams and opponents. The `teams` table has `is_owned` to distinguish them. Players, rosters, games, and stats tables all reference `teams` via FK and do not restrict to owned teams. What is missing is a way to track scouting crawl runs: when was opponent X last scouted? How many boxscores were collected? This metadata is needed by the crawler (story 03) for idempotency and incremental crawling.

`teams.last_synced` tracks our own-team crawl cycles. Scouting runs are a different concept -- they track per-opponent data collection and should be separate.

The user also wants fetch timestamps (`first_fetched`, `last_checked`) on crawled records to support re-fetch workflows. This epic establishes the pattern on the `scouting_runs` table; existing tables will get these columns in a future migration.

## Acceptance Criteria
- [ ] **AC-1**: A new migration file `migrations/007_scouting_runs.sql` creates a `scouting_runs` table with at minimum: `id` (INTEGER PK), `team_id` (TEXT FK to teams, the opponent being scouted), `season_id` (TEXT FK to seasons), `run_type` (TEXT -- e.g., 'roster', 'boxscores', 'full'), `started_at` (TEXT), `completed_at` (TEXT, nullable), `status` (TEXT -- 'running', 'completed', 'failed'), `games_found` (INTEGER, nullable), `games_crawled` (INTEGER, nullable), `players_found` (INTEGER, nullable), `error_message` (TEXT, nullable), `first_fetched` (TEXT -- ISO 8601, `DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))`, set on first insert, never updated), and `last_checked` (TEXT -- ISO 8601, `DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ', 'now'))`, updated on every re-run).
- [ ] **AC-2**: The migration is idempotent (`CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`).
- [ ] **AC-3**: The migration includes an index on `(team_id, season_id)` for efficient "when was this opponent last scouted?" lookups.
- [ ] **AC-4**: The migration includes a `UNIQUE(team_id, season_id, run_type)` constraint to prevent duplicate concurrent runs for the same opponent/season/type. The crawler (story 03) must use `ON CONFLICT(team_id, season_id, run_type) DO UPDATE` to update `last_checked`, `status`, `started_at`, counts, and `error_message` on re-runs — but NOT `first_fetched` (which stays as the original insert timestamp). Do NOT use `INSERT OR REPLACE` as it would delete-and-reinsert the row, losing `first_fetched`.
- [ ] **AC-5**: Season rows are created dynamically by the crawler/loader at runtime (following the pattern in `game_loader.py`'s `_ensure_season_row()`). The migration does NOT seed any season rows. This matches the existing project convention.
- [ ] **AC-6**: `migrations/apply_migrations.py` picks up the new migration without code changes (it auto-discovers SQL files by naming convention).
- [ ] **AC-7**: Tests verify the migration applies cleanly to a fresh database and the table schema matches expectations (including `first_fetched` and `last_checked` columns).

## Technical Approach
Follow the pattern of `migrations/006_opponent_links.sql`. The migration numbering convention says the next number is `007` (see `/.claude/rules/migrations.md`). The `scouting_runs` table is a crawl-metadata table, not a data table -- it tracks the scouting process, not the scouted data itself.

The season question: opponent teams play in their own seasons. The simplest approach is to use the same `seasons` table with a season_id that matches the opponent's season context (e.g., `2025-spring-hs` for a spring 2025 high school team). The crawler will need to determine or create the appropriate season_id. Document the approach chosen.

This pattern will be extended to existing tables in a future migration -- this epic establishes the convention.
