## Description
After this story is complete, the system will have a scouting crawler (`src/gamechanger/crawlers/scouting.py`) that, given a resolved opponent, fetches their player roster, game summaries, and per-game boxscores via the authenticated GameChanger API. A loader (`src/gamechanger/loaders/scouting_loader.py`) will persist the crawled data into the existing database tables and compute season aggregates from boxscores. A new `bb data scout` CLI command will let the operator trigger scouting runs for all resolved opponents or a specific team.

## Context
The scouting chain has been verified end-to-end (2026-03-12). The `players`, `game-summaries`, and `boxscore` endpoints work with opponent team UUIDs. The `season-stats` endpoint returns Forbidden for non-owned teams, so season aggregates must be computed from boxscores. The existing crawler/loader patterns in `src/gamechanger/crawlers/` and `src/gamechanger/loaders/` provide the architectural template. The existing `players`, `team_rosters`, `games`, `player_game_batting/pitching`, and `player_season_batting/pitching` tables already support opponent data. The `scouting_runs` table (story 02) provides idempotency tracking with `first_fetched`/`last_checked` timestamps.

## Acceptance Criteria

### Crawler
- [ ] **AC-1**: A module `src/gamechanger/crawlers/scouting.py` exists with a `ScoutingCrawler` class that accepts a `GameChangerClient`, config, and database connection. It exposes a `scout_team(team_id: str, season_id: str)` method and a `scout_all()` method.
- [ ] **AC-2**: `scout_team()` executes the scouting chain: (1) fetch roster via `GET /teams/{team_id}/players`, (2) fetch game summaries via `GET /teams/{team_id}/game-summaries` (with pagination), (3) fetch boxscores for each game via `GET /game-stream-processing/{game_stream_id}/boxscore`. Season-stats is NOT called (returns Forbidden for non-owned teams).
- [ ] **AC-3**: `scout_all()` queries `opponent_links` for all resolved opponents (WHERE `resolved_team_id IS NOT NULL`) and calls `scout_team()` for each. Opponents with recent scouting runs (check `scouting_runs` table, configurable freshness threshold) are skipped.
- [ ] **AC-4**: The crawler writes raw API responses to `data/raw/{season}/scouting/{team_id}/` (roster.json, game-summaries.json, boxscores/{game_stream_id}.json) following the existing raw-data pattern.
- [ ] **AC-5**: The crawler creates/updates a `scouting_runs` row (status: running -> completed/failed, with counts of games found, games crawled, players found). On first run, `first_fetched` is set; on every run, `last_checked` is updated to the current timestamp.
- [ ] **AC-6**: The crawler handles `CredentialExpiredError` and `ForbiddenError` gracefully -- logs a warning and skips the opponent rather than crashing the entire run.

### Loader
- [ ] **AC-7**: A module `src/gamechanger/loaders/scouting_loader.py` exists that delegates per-game loading to the existing `GameLoader` (`src/gamechanger/loaders/game_loader.py`) rather than reimplementing boxscore parsing, player stub creation, game record upsert, or batting/pitching line upsert. Pass the scouted opponent's `resolved_team_id` as `owned_team_id` to `GameLoader`. The scouting loader's additional responsibilities: roster loading into `players`/`team_rosters`, `scouting_runs` tracking, and season aggregate computation (AC-8a).
- [ ] **AC-8**: The loader uses the FK-safe stub-player pattern for any unknown player_ids encountered in boxscores (consistent with existing loaders -- see CLAUDE.md).
- [ ] **AC-8a**: The loader computes season aggregate stats by querying `player_game_batting` and `player_game_pitching` rows for the scouted team's players, summing counting stats (batting: AB, H, doubles, triples, HR, RBI, BB, SO, SB; pitching: ip_outs, H, ER, BB, SO, HR), and upserting into `player_season_batting` and `player_season_pitching`. Rate stats (AVG, OBP, ERA, WHIP) are NOT stored — they are computed at display time from counting stat columns. Home/away split columns are left NULL for scouting-computed season stats (deferred to a future story).
- [ ] **AC-9**: The loader is idempotent -- re-running with the same raw data does not create duplicate rows (upsert via INSERT OR REPLACE or ON CONFLICT).

### CLI

## Technical Approach
Follow the patterns established by the existing crawlers and loaders. Key references:
- Crawler pattern: `src/gamechanger/crawlers/opponent.py` (two-phase crawl, raw file output, freshness skip)
- **Game loader (PRIMARY reference)**: `src/gamechanger/loaders/game_loader.py` — reuse `GameLoader` directly for per-game boxscore loading. It handles boxscore parsing, player stubs, game records, batting/pitching line upsert, asymmetric team key detection, and home/away assignment. Pass `resolved_team_id` as `owned_team_id`.
- Season stats loader: `src/gamechanger/loaders/season_stats_loader.py` — reference for `_ip_to_ip_outs()` and season stat upsert patterns (but the scouting loader aggregates from DB rows, not from the API)
- CLI pattern: `src/cli/data.py` (existing `bb data crawl` and `bb data load` commands)
- Client: `src/gamechanger/client.py` (`GameChangerClient` with `get()`, `get_paginated()`)

**Boxscore structure**: The response uses `players[]` + `groups[]` (lineup/pitching categories) with per-player `stats[]` arrays. Player identity is in `players[]`, stat entries reference by `player_id`. The existing `game_loader.py` handles this correctly — do NOT reimplement the parsing.

**Season aggregate computation**: Query `player_game_batting` and `player_game_pitching` tables for the scouted team's players (filter by `team_id = resolved_team_id`), sum counting stats across all games, and upsert into `player_season_batting`/`player_season_pitching`. Rate stats are NOT stored — they are computed at display time. Home/away split columns are left NULL (deferred).

For season_id determination: the crawler should derive it from the opponent's team metadata or the game dates. The simplest approach is to use a season_id matching the pattern `{year}-spring-hs` based on the game dates in the schedule.

## Notes
- The operator's verified test data: Lincoln Southwest Varsity 2025 -- 28 games, full boxscores, ~15 players on roster.
- Rate limiting: the crawler will make 2-3 + N API calls per opponent (1 roster + 1-2 game-summaries + N boxscores, where N = games). For 10 opponents with 25 games each, that is ~270 calls. Existing rate limiting in `GameChangerClient` (1-2s delay between requests) should be sufficient. The crawler should log progress so the operator can monitor.
- Boxscores are NOT optional in this pipeline -- they are the source of truth for season aggregates since `season-stats` is Forbidden for non-owned teams. Every scouting run must fetch boxscores to compute season stats.
- **IP is float decimal innings** in the boxscore response (e.g., 3.333... = 3⅓ IP = 10 outs). Convert via `round(float(IP) * 3)`. See AC-11a for the pre-existing bug fix.
- **GameLoader reuse**: The `owned_team_id` parameter passed to `GameLoader` for scouting is the scouted team's UUID (`resolved_team_id`), while the key that identifies their data in boxscore JSON is their `public_id` slug. These are different identifiers for the same team, handled transparently by `_detect_team_keys()`.
