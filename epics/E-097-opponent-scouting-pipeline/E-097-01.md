## Status

## Description
After this story is complete, `docs/api/flows/` will contain a new `opponent-scouting.md` integration guide that documents the verified public-endpoint scouting chain end-to-end. This flow doc will be the authoritative reference for how to go from an opponent's `public_id` to game schedules, player rosters, per-game boxscores, and locally computed season aggregates -- all via public endpoints without needing UUIDs or following.

## Context
The scouting chain was verified live on 2026-03-12 using both Lincoln Southwest Varsity 2025 and an unfollowed team (`8O8bTolVfb9A`). The entire flow works via public endpoints: schedule (no auth), roster (gc-token), boxscore (gc-token). The public `/games` endpoint's `id` field IS the `game_stream_id` for boxscores -- no bridge call needed. Originally the chain used authenticated endpoints requiring UUIDs, but the public-endpoint discovery simplifies everything. Currently this knowledge exists only in the epic's Technical Notes and conversation history. It needs a permanent home in `docs/api/flows/` alongside the existing `opponent-resolution.md` (which covers resolution but not stat retrieval). Several existing endpoint docs also need corrections to reflect the public-endpoint scouting path.

## Acceptance Criteria
- [ ] **AC-1**: A file `docs/api/flows/opponent-scouting.md` exists with the complete public-endpoint scouting chain documented step-by-step: starting entity (opponent with `public_id` in `opponent_links`), each API call with endpoint path, auth requirements, and which field feeds the next step, the local aggregation step (compute season totals from boxscores), and the ending state (schedule + roster + boxscores + computed season aggregates available).
- [ ] **AC-2**: The document presents the public-endpoint chain as the PRIMARY and RECOMMENDED scouting path. An "Authenticated Fallback" section notes that `GET /teams/{team_id}/game-summaries` exists as an alternative for game discovery (returns in-progress/upcoming games, has pagination, requires gc-token + UUID), but is NOT the primary scouting path. No references to `best-game-stream-id` bridge calls (the public `/games` `id` field IS the `game_stream_id` directly).
- [ ] **AC-3**: Each step in the flow links to its corresponding endpoint doc in `docs/api/endpoints/` using relative paths.
- [ ] **AC-4**: Edge cases are documented: opponents without `public_id` (skip), public `/games` returns only completed games (limitation noted), boxscore asymmetric keys (slug vs UUID), player UUID cross-referencing between roster and boxscore lines, inverted URL pattern (`/teams/public/` vs `/public/teams/`).
- [ ] **AC-4a**: The document includes a "Season-Stats Forbidden" section explaining that `GET /teams/{team_id}/season-stats` returns HTTP 403 for non-owned teams (web profile, verified 2026-03-12) — even for teams followed as fan (confirmed 3/3 attempts, likely requires coaching staff membership). Include the note about the unresolved mobile observation discrepancy, and that this is why aggregates are computed locally from boxscores.
- [ ] **AC-4b**: The document includes a "Following Not Required" section stating clearly: following is NOT required for any step in the public-endpoint scouting chain (confirmed on unfollowed team 2026-03-12). Notes that `POST /teams/{team_id}/follow` exists for future use cases, with a cross-reference to the endpoint doc.
- [ ] **AC-5**: The existing `docs/api/flows/opponent-resolution.md` is updated: (a) add a `see_also` cross-reference to the new scouting flow doc, and (b) correct/remove the statement "following is a prerequisite for the scouting pipeline" (now incorrect).
- [ ] **AC-6**: `docs/api/README.md` is updated to list the new flow doc (if a flows section exists) or the flow doc is discoverable from the existing index.
- [ ] **AC-7**: The boxscore endpoint doc (`docs/api/endpoints/get-game-stream-processing-game_stream_id-boxscore.md`) is updated to correct the "CRITICAL: Must crawl game-summaries first" caveat -- note that the public `/games` endpoint's `id` field is an alternative (and preferred) source for `game_stream_id`.
- [ ] **AC-8**: The public games endpoint doc (`docs/api/endpoints/get-public-teams-public_id-games.md`) is updated to add a cross-reference noting that the `id` field in the response IS the `game_stream_id` used by the boxscore endpoint.

## Technical Approach
The new flow doc follows the pattern established by `docs/api/flows/opponent-resolution.md`. It should document the chain from the perspective of a developer or agent implementing the scouting crawler -- what endpoints to call, in what order, which response fields feed the next call, and what edge cases to handle. Reference the endpoint docs in `docs/api/endpoints/` by path but do not duplicate their full schemas.
