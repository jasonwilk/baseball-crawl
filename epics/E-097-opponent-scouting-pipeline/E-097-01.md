# E-097-01: Document Opponent Scouting Flow

## Epic
[E-097: Opponent Scouting Data Pipeline](epic.md)

## Status
`TODO`

## Description
After this story is complete, `docs/api/flows/` will contain a new `opponent-scouting.md` integration guide that documents the verified multi-endpoint scouting chain end-to-end. This flow doc will be the authoritative reference for how to go from a resolved opponent's `resolved_team_id` to player rosters, per-game boxscores, and locally computed season aggregates.

## Context
The scouting chain was verified live on 2026-03-12 using Lincoln Southwest Varsity 2025 as the test case. The flow crosses 3 endpoints with specific ID field mappings, plus a local aggregation step. Originally the chain included `season-stats`, but live testing revealed it returns Forbidden for non-owned teams -- so aggregates are computed from boxscores instead. Currently this knowledge exists only in the epic's Technical Notes and conversation history. It needs a permanent home in `docs/api/flows/` alongside the existing `opponent-resolution.md` (which covers resolution but not stat retrieval).

## Acceptance Criteria
- [ ] **AC-1**: A file `docs/api/flows/opponent-scouting.md` exists with the complete scouting chain documented step-by-step: starting entity (resolved opponent in `opponent_links`), each API call with endpoint path and which field feeds the next step, the local aggregation step (compute season totals from boxscores), and the ending state (roster + boxscores + computed season aggregates available).
- [ ] **AC-2**: The document includes the two alternative paths for game discovery: (a) authenticated `game-summaries` (preferred -- returns `game_stream.id` directly) and (b) public `games` + `best-game-stream-id` bridge (extra call per game).
- [ ] **AC-3**: Each step in the flow links to its corresponding endpoint doc in `docs/api/endpoints/` using relative paths.
- [ ] **AC-4**: Edge cases are documented: opponents without `resolved_team_id` (skip), paginated `game-summaries` (follow `x-next-page`), boxscore asymmetric keys (slug vs UUID), player UUID cross-referencing between roster and boxscore lines.
- [ ] **AC-4a**: The document includes a "Season-Stats Forbidden" section explaining that `GET /teams/{team_id}/season-stats` returns HTTP 403 for non-owned teams (web profile, verified 2026-03-12) — even for teams followed as fan (confirmed 3/3 attempts, likely requires coaching staff membership). Include the note about the unresolved mobile observation discrepancy, and that this is why aggregates are computed locally from boxscores.
- [ ] **AC-4b**: The document includes a "Follow-Gating and Auto-Follow" section noting: (1) the reverse bridge is follow-gated (403 without following, 200 after), (2) this scouting chain does NOT require following because it starts from `resolved_team_id` and does not use the reverse bridge, (3) `POST /teams/{team_id}/follow` exists for auto-following if needed for future use cases, with a cross-reference to the endpoint doc.
- [ ] **AC-5**: The existing `docs/api/flows/opponent-resolution.md` is updated to add a `see_also` cross-reference to the new scouting flow doc (resolution is the prerequisite for scouting).
- [ ] **AC-6**: `docs/api/README.md` is updated to list the new flow doc (if a flows section exists) or the flow doc is discoverable from the existing index.

## Technical Approach
The new flow doc follows the pattern established by `docs/api/flows/opponent-resolution.md`. It should document the chain from the perspective of a developer or agent implementing the scouting crawler -- what endpoints to call, in what order, which response fields feed the next call, and what edge cases to handle. Reference the endpoint docs in `docs/api/endpoints/` by path but do not duplicate their full schemas.
