# E-001-02: Build authenticated GameChanger HTTP client

## Epic
[E-001: GameChanger API Foundation](epic.md)

## Status
`DONE`

## Description
After this story is complete, the system will have a reusable Python client class (`src/gamechanger/client.py`) that loads credentials from the `.env` file, attaches them to every request, handles error responses correctly, and raises a clear `CredentialExpiredError` when credentials have expired. All other ingestion code in E-002 will use this client rather than making raw HTTP calls.

## Context
This is the HTTP layer that sits between the credentials (written by E-001-01) and the data ingestion code (written in E-002). Every request to the GameChanger API will go through this client. Getting it right here means E-002 never has to think about authentication, retries, or error handling for auth failures.

The authentication scheme is confirmed: GameChanger uses a custom `gc-token` header carrying a JWT, plus `gc-device-id` and `gc-app-name` on every request. The client must also set endpoint-specific vendor-typed `Accept` headers (e.g., `application/vnd.gc.com.game_summary:list+json; version=0.1.0`). See `docs/gamechanger-api.md` for the full header reference.

The client must build on the existing HTTP session factory at `src/http/session.py` (`create_session()`), which already provides browser-realistic headers, cookie persistence, and rate limiting with jitter. See `docs/http-integration-guide.md` for the integration pattern.

## Acceptance Criteria
- [x] **AC-1**: Given valid credentials in `.env`, when `GameChangerClient().get("/teams/{team_id}/game-summaries")` is called, then it returns a parsed JSON response and logs the request URL and response status at DEBUG level.
- [x] **AC-2**: Given credentials in `.env` that return a 401 or 403 response, when any client method is called, then it raises `CredentialExpiredError` with a message that includes: the endpoint that failed, the response status code, and the instruction `Refresh by running: python scripts/refresh_credentials.py`.
- [x] **AC-3**: Given a 429 response, when the client receives it, then it logs a WARNING with the endpoint and waits for the `Retry-After` header value (or 60 seconds if not present) before raising a `RateLimitError`. It does not silently retry.
- [x] **AC-4**: Given a 5xx response, when the client receives it, then it retries up to 3 times with exponential backoff (1s, 2s, 4s) and raises `GameChangerAPIError` if all retries fail.
- [x] **AC-5**: Given no `.env` file or missing any of `GAMECHANGER_AUTH_TOKEN`, `GAMECHANGER_DEVICE_ID`, or `GAMECHANGER_BASE_URL`, when the client is instantiated, then it raises `ConfigurationError` with a message listing the missing keys.
- [x] **AC-6**: The client delegates rate limiting to the `create_session()` factory from `src/http/session.py`. The `GameChangerClient` constructor accepts `min_delay_ms` (default: 1000) and `jitter_ms` (default: 500) parameters that are forwarded to `create_session()`.
- [x] **AC-7**: All client methods accept an optional `timeout` parameter (default: 30 seconds).
- [x] **AC-8**: The client's `get()` method accepts an optional `accept` parameter to set the endpoint-specific `Accept` header. When not provided, the default `Accept` from `BROWSER_HEADERS` is used.

## Technical Approach
Build on the existing HTTP session factory (`src/http/session.py:create_session()`). Structure:

```python
# src/gamechanger/client.py

from src.http.session import create_session

class CredentialExpiredError(Exception): ...
class RateLimitError(Exception): ...
class GameChangerAPIError(Exception): ...
class ConfigurationError(Exception): ...

class GameChangerClient:
    def __init__(self, min_delay_ms: int = 1000, jitter_ms: int = 500):
        self._credentials = self._load_credentials()
        self._session = create_session(min_delay_ms=min_delay_ms, jitter_ms=jitter_ms)
        # Inject GC auth headers into the session
        self._session.headers["gc-token"] = self._credentials["GAMECHANGER_AUTH_TOKEN"]
        self._session.headers["gc-device-id"] = self._credentials["GAMECHANGER_DEVICE_ID"]
        self._session.headers["gc-app-name"] = self._credentials.get("GAMECHANGER_APP_NAME", "web")

    def get(self, path: str, params: dict | None = None, timeout: int = 30, accept: str | None = None) -> ...: ...
    def _load_credentials(self) -> dict[str, str]: ...
    def _handle_response(self, response: httpx.Response, path: str) -> ...: ...
```

Credential loading: use `python-dotenv` to load `.env`. Required keys: `GAMECHANGER_AUTH_TOKEN`, `GAMECHANGER_DEVICE_ID`, `GAMECHANGER_BASE_URL`. Optional: `GAMECHANGER_APP_NAME` (defaults to `"web"`). Raise `ConfigurationError` immediately if any required key is missing.

The base URL comes from `GAMECHANGER_BASE_URL` in `.env` (confirmed: `https://api.team-manager.gc.com`).

The `get()` method accepts an optional `accept` parameter for endpoint-specific vendor-typed Accept headers. When provided, it overrides the session's default `Accept` header for that request. See the per-endpoint Accept header table in `docs/gamechanger-api.md`.

GET requests should also set `Content-Type: application/vnd.gc.com.none+json; version=undefined` to match observed browser behavior.

Keep `_load_credentials` as a separate, testable method so it can be unit tested without making HTTP calls.
