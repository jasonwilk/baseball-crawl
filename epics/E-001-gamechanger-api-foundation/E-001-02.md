# E-001-02: Build authenticated GameChanger HTTP client

## Epic
[E-001: GameChanger API Foundation](epic.md)

## Status
`TODO`

## Description
After this story is complete, the system will have a reusable Python client class (`src/gamechanger/client.py`) that loads credentials from the `.env` file, attaches them to every request, handles error responses correctly, and raises a clear `CredentialExpiredError` when credentials have expired. All other ingestion code in E-002 will use this client rather than making raw HTTP calls.

## Context
This is the HTTP layer that sits between the credentials (written by E-001-01) and the data ingestion code (written in E-002). Every request to the GameChanger API will go through this client. Getting it right here means E-002 never has to think about authentication, retries, or error handling for auth failures.

The exact authentication scheme (bearer token, cookie, or both) will be known once E-001-01 has parsed a real curl command. This story should be written to handle both cases -- the auth mechanism should be configurable based on what credentials are present in the `.env` file.

## Acceptance Criteria
- [ ] **AC-1**: Given valid credentials in `.env`, when `GameChangerClient().get("/v2/teams/abc123/stats")` is called, then it returns a parsed JSON dict and logs the request URL and response status at DEBUG level.
- [ ] **AC-2**: Given credentials in `.env` that return a 401 or 403 response, when any client method is called, then it raises `CredentialExpiredError` with a message that includes: the endpoint that failed, the response status code, and the exact command to run to refresh credentials (`python scripts/refresh_credentials.py --curl "..."`).
- [ ] **AC-3**: Given a 429 response, when the client receives it, then it logs a WARNING with the endpoint and waits for the `Retry-After` header value (or 60 seconds if not present) before raising a `RateLimitError`. It does not silently retry.
- [ ] **AC-4**: Given a 5xx response, when the client receives it, then it retries up to 3 times with exponential backoff (1s, 2s, 4s) and raises `GameChangerAPIError` if all retries fail.
- [ ] **AC-5**: Given no `.env` file or missing required credential keys, when the client is instantiated, then it raises `ConfigurationError` immediately (fail fast, not on first request).
- [ ] **AC-6**: The client includes a configurable `request_delay_ms` parameter (default: 500) that adds a sleep between every request to avoid hammering the API.
- [ ] **AC-7**: All client methods accept an optional `timeout` parameter (default: 30 seconds).

## Technical Approach
Use `httpx` (sync client for now; async can come later if needed). Structure:

```python
# src/gamechanger/client.py

class CredentialExpiredError(Exception): ...
class RateLimitError(Exception): ...
class GameChangerAPIError(Exception): ...
class ConfigurationError(Exception): ...

class GameChangerClient:
    def __init__(self, request_delay_ms: int = 500): ...
    def get(self, path: str, params: dict | None = None, timeout: int = 30) -> dict: ...
    def _load_credentials(self) -> dict[str, str]: ...
    def _build_headers(self, credentials: dict[str, str]) -> dict[str, str]: ...
    def _handle_response(self, response: httpx.Response, path: str) -> dict: ...
```

Credential loading: use `python-dotenv` to load `.env`. Check for `GAMECHANGER_AUTH_TOKEN` (bearer) and/or `GAMECHANGER_COOKIE_*` keys. If bearer token is present, add `Authorization: Bearer <token>` header. If cookies are present, reconstruct cookie string.

The base URL comes from `GAMECHANGER_BASE_URL` in `.env`.

Keep `_load_credentials` and `_build_headers` as separate, testable methods so they can be unit tested without making HTTP calls.

import httpx
import respx

@respx.mock
