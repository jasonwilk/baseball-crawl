## Description
After this story is complete, the report generation pipeline will crawl, load, and reconcile plays data for the scouted team's games, then query plays-derived aggregates (FPS%, QAB%, pitch counts) and merge them into the existing batting/pitching data dicts passed to the renderer. Reconciliation corrects pitcher attribution errors before FPS% is queried, ensuring accurate stats. The renderer receives plays stats as additional keys in the player dicts it already processes.

## Context
The report generator (`src/reports/generator.py`) runs a multi-stage pipeline ending with stat queries and HTML rendering. Plays data needs to be crawled, loaded, and reconciled as a new pipeline stage, then queried and merged into existing data structures. The plays endpoint is NOT ownership-gated, so it works for opponent games. Existing indexes on the plays table are sufficient for aggregation queries. No new migrations needed.

**Critical constraint**: `PlaysCrawler` cannot be reused -- it requires `CrawlConfig` and iterates member teams. A new helper function must be written that reads game IDs from the DB and fetches plays directly. See TN-1 for the full constraint and recommended approach. `PlaysLoader` CAN be reused but needs the scouting directory path and a constructed `TeamRef`. See TN-1 for directory layout details.

**Reconciliation**: After plays are loaded, `reconcile_game(conn, game_id, dry_run=False)` must be called for each game to correct pitcher attribution errors. This is an automatic pipeline step, not a manual operator action. See TN-1 for reconciliation dependency on boxscore files.

## Acceptance Criteria
- [ ] **AC-1**: Given a report generation for a team with completed games, when the pipeline runs, then plays data is fetched from the plays endpoint for each completed game that doesn't already have plays rows in the DB. Plays files are written to the scouting data directory (`data/raw/{season}/scouting/{public_id}/plays/{game_id}.json`) and loaded via `PlaysLoader`, per TN-1.
- [ ] **AC-2**: Given plays data has been loaded for a team's games, when the pipeline continues, then `reconcile_game(conn, game_id, dry_run=False)` is called for each game with plays data, correcting pitcher attribution errors before stats are queried. Per TN-1.
- [ ] **AC-3**: Given reconciled plays data exists in the DB for a team's games, when the generator queries stats, then each pitcher dict includes `fps_pct` (float or None) and `pitches_per_bf` (float or None) keys, and each batter dict includes `qab_pct` (float or None) and `pitches_per_pa` (float or None) keys.
- [ ] **AC-4**: Given no plays data exists for a team's games, when the generator queries stats, then all plays-derived keys are None and the report generates successfully without error.
- [ ] **AC-5**: Failure handling is per-game, not all-or-nothing, per TN-1. Given a per-game failure (crawl, load, or reconciliation error for one game), when the pipeline continues, then the failure is logged as a warning and other games proceed normally. Given a stage-level failure (e.g., auth expiration before any games are fetched), when the pipeline continues, then the report generates without plays data (all plays columns show "—"). In both cases, report generation completes successfully.
- [ ] **AC-6**: FPS% is computed per CLAUDE.md definition: `SUM(is_first_pitch_strike) / COUNT(*)` excluding outcomes 'Hit By Pitch' and 'Intentional Walk' from the denominator. Pitching stats are scoped to games involving the team and matched to the team's roster during merge -- NOT filtered by `batting_team_id`, per TN-2.
- [ ] **AC-7**: QAB% is computed as `SUM(is_qab) / COUNT(*)`, scoped by `batting_team_id = team_id` per TN-2.
- [ ] **AC-8**: The data dict passed to `render_report()` includes `team_fps_pct` (float or None), `team_pitches_per_pa` (float or None), `has_plays_data` (bool), and `plays_game_count` (int) per TN-2. `has_plays_data` is True when at least one game has plays rows in the DB for this team; False otherwise.
- [ ] **AC-9**: Tests verify query functions return correct aggregates from plays data, including the FPS% exclusion filter, the pitching-vs-batting scoping asymmetry, and graceful handling of empty results.

## Technical Approach
The generator already has a pattern for pipeline stages (`_crawl_and_load_spray()`) and query functions (`_query_batting()`, `_query_pitching()`). The plays stage follows the same non-fatal wrapper pattern but cannot delegate to `PlaysCrawler` (see TN-1). Instead, it queries game IDs from the `games` table, fetches plays directly via the client, writes to the scouting directory, uses `PlaysLoader` for DB insertion, and then runs `reconcile_game()` for each game. The `games.game_id` value is the correct path parameter for the plays endpoint (confirmed by the scouting crawler using the same ID for boxscores).

## Notes
- The plays endpoint path parameter is `event_id` from game-summaries, NOT `game_stream_id`. In the scouting flow, `games.game_id` (from the public schedule's `game["id"]` field) serves as the equivalent. See CLAUDE.md critical ID mapping note.
- The FPS% exclusion filter (`WHERE outcome NOT IN ('Hit By Pitch', 'Intentional Walk')`) matches the partial index `idx_plays_fps` on the plays table.
- Team-level aggregates are computed from the full plays result set, not averaged across per-player rates.
- Reconciliation runs automatically as part of the pipeline -- no confidence gating needed because the data IS reconciled by the time stats are queried.
- `reconcile_game()` needs cached boxscore JSON on disk. The scouting crawler writes these before the plays stage runs, so the dependency is naturally satisfied. The reconciliation engine's `_extract_pitcher_order()` already has a scouting path fallback (`data/raw/{season}/scouting/{public_id}/boxscores/`) -- confirmed by SE review.
- `PlaysLoader` derives `season_id` from the `games` table internally (queries `games.season_id` for each game_id). No external `season_id` needs to be passed to the loader.
