## Description
After this story is complete, the operator can permanently delete teams that are both deactivated and have no associated game/stat data. This covers the primary use case: removing mis-entered or duplicate tracked teams before any data has been crawled for them.

## Context
The user wants to delete deactivated teams. Per TN-3, 12 tables reference `teams(id)` with no `ON DELETE CASCADE`, making cascade deletion of data-bearing teams dangerous. Expert consensus (SE + DE): allow hard delete ONLY for zero-data teams. Teams with historical game/stat data should remain deactivated (`is_active = 0`). The `toggle-active` route already exists; this adds the final delete step.

## Acceptance Criteria
- [ ] **AC-1**: A "Delete" action appears on the teams table ONLY for rows where `is_active = 0` (deactivated teams). Active teams show no delete option.
- [ ] **AC-2**: Clicking "Delete" shows a browser confirmation dialog (`confirm()`) with the team name before submitting.
- [ ] **AC-3**: `POST /admin/teams/{id}/delete` checks two preconditions: (a) `is_active = 0`, (b) zero dependent rows in data tables per TN-3 (games, player_game_batting, player_game_pitching, player_season_batting, player_season_pitching, scouting_runs, spray_charts). If either check fails, redirect to `/admin/teams` with an error flash (e.g., `?error=Team+has+associated+data+and+cannot+be+deleted`) using the standard admin UI redirect + query-parameter flash pattern. Do not return a 400 response body.
- [ ] **AC-4**: When deletion proceeds (both checks pass), the route deletes in a single transaction per TN-3: junction/access rows first (`team_opponents`, `team_rosters`, `opponent_links`, `user_team_access`, `coaching_assignments`, `crawl_jobs`), then the `teams` row.
- [ ] **AC-5**: After successful deletion, redirect to the teams list with a success flash confirming which team was deleted.
- [ ] **AC-6**: The delete route uses POST method (not GET) and requires admin access.

## Technical Approach
Add a POST route for team deletion. The pre-delete check queries each data table listed in TN-3 for rows referencing the team ID (check both FK columns for tables that have two, like `games.home_team_id` and `games.away_team_id`). If all counts are zero, proceed with cascading delete in a transaction. The UI conditionally renders the Delete action based on `is_active` status. Use the same `confirm()` pattern as the existing opponent disconnect action for consistency.
