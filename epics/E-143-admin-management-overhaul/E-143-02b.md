# E-143-02b: Admin Role Enforcement in UI

## Epic
[E-143: Admin Management Overhaul](epic.md)

## Status
`TODO`

## Description
After this story is complete, the admin UI enforces role-based access. Admin users can manage teams, programs, users, and trigger crawls. Regular users can view dashboards but cannot access management routes. The `_require_admin()` check switches from the `ADMIN_EMAIL` env var to also accept database-backed `role = 'admin'`.

## Context
E-143-02 adds the `role` column to `users`. This story wires it into the application. Per TN-4, `ADMIN_EMAIL` is retained as a bootstrap fallback alongside the new database role check. The baseball-coach confirmed that coaches (as "users") should have read-only dashboard access -- they consume output, not operate the system.

## Acceptance Criteria
- [ ] **AC-1**: `_require_admin()` accepts EITHER `ADMIN_EMAIL` env var match OR `users.role = 'admin'` per TN-4. Both paths grant admin access. If `ADMIN_EMAIL` is unset AND the user does not have `role = 'admin'`, access is denied (403). The current dev-mode fallback that grants admin to all authenticated users when `ADMIN_EMAIL` is unset is removed.
- [ ] **AC-2**: Application-layer validation rejects any role value other than `'admin'` or `'user'` on user create/edit.
- [ ] **AC-3**: The Users list table displays a "Role" column showing the role per row.
- [ ] **AC-4**: The "Add User" form includes a Role field (radio or select: Admin / User, default: User).
- [ ] **AC-5**: The "Edit User" page includes a Role field allowing role changes.
- [ ] **AC-6**: Self-demotion guard: an admin cannot set their own role to "user" via the edit form (server-side validation with flash error). This prevents accidental lockout.
- [ ] **AC-7**: All management routes (team CRUD, program CRUD, user CRUD) require admin access. Read-only dashboard routes remain accessible to all authenticated users.
- [ ] **AC-8**: Any existing admin bootstrap guidance visible in the admin UI (e.g., a note about setting `ADMIN_EMAIL`) is updated to reflect the new dual-path access model: admin access is granted via `ADMIN_EMAIL` env var (bootstrap/fallback) OR by setting `role = 'admin'` in the database.

## Technical Approach
Modify `_require_admin()` to query the user's role from the database in addition to checking `ADMIN_EMAIL`. Add the role field to user create/edit forms and the users list template. The self-demotion guard compares the edited user's ID against the current session user's ID. Check the existing user form templates for the pattern to follow.
