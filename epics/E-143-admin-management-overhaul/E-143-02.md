# E-143-02: User Role Schema Migration

## Epic

## Status

## Description
After this story is complete, the `users` table has a `role` column that distinguishes admin users from regular users. All existing users default to `user` role. This migration is the prerequisite for admin role enforcement in the UI (E-143-02b).

## Context
Currently admin access is gated entirely by the `ADMIN_EMAIL` env var checked in `_require_admin()`. There is no `role` column on `users`. Per TN-4, a single `ALTER TABLE` migration adds the column. Application-layer validation handles role value enforcement since SQLite cannot add CHECK constraints to existing columns via ALTER TABLE.

## Acceptance Criteria
- [ ] **AC-1**: A migration file adds `role TEXT NOT NULL DEFAULT 'user'` to the `users` table.
- [ ] **AC-2**: The migration follows the project's numbered migration convention (next sequential number after the highest existing migration).
- [ ] **AC-3**: After migration, all existing user rows have `role = 'user'` via the DEFAULT clause.
- [ ] **AC-4**: The migration file is discovered and executed by `apply_migrations.py` on container startup via the `_migrations` tracker. It runs exactly once per database. No per-statement IF-NOT-EXISTS guard is needed or expected.
- [ ] **AC-5**: The migration includes a comment documenting the two valid role values (`admin`, `user`) for future reference.

## Technical Approach
Create the next numbered migration SQL file. The migration is a single `ALTER TABLE users ADD COLUMN role TEXT NOT NULL DEFAULT 'user'` statement. Check the existing migration files to determine the correct sequence number. The migration runner (`migrations/apply_migrations.py`) applies migrations at container startup via the Dockerfile CMD.
