## Description
After this story is complete, both the crawl pipeline and the load pipeline support running against a single team instead of all active member teams. This is the backend prerequisite for the crawl trigger UI (E-143-06). A UI "Sync" must run both crawl and load (per TN-2) -- without the load step, crawled data stays in `data/raw/` and dashboards remain stale. The scouting pipeline already supports per-team invocation; this story adds the same capability to the main crawl and load pipelines.

## Context
Per TN-2, both `src/pipeline/crawl.py:run()` and `src/pipeline/load.py:run()` currently iterate all active member teams with no per-team filter. Both functions have a `source` parameter (default `"yaml"`) but neither accepts `team_ids`. The scouting pipeline (`ScoutingCrawler.scout_team(public_id)`) is already per-team. This story is a pure Python refactor with no HTTP endpoints or UI changes.

## Acceptance Criteria
- [ ] **AC-1**: `crawl.run()` accepts an optional `team_ids: list[int] | None` parameter that works with `source="db"`. When `team_ids` is provided with `source="db"`, only the specified teams (matched via `TeamEntry.internal_id`) are processed by the roster, schedule, opponent, player-stats, and game-stats crawlers. When `team_ids` is `None`, all active member teams are processed (preserving existing behavior).
- [ ] **AC-2**: `load.run()` accepts an optional `team_ids: list[int] | None` parameter that works with `source="db"`. When `team_ids` is provided with `source="db"`, only the specified teams are loaded. When `team_ids` is `None`, all active member teams are loaded (preserving existing behavior).
- [ ] **AC-3**: The per-team filter works correctly with `crawler_filter` / `loader_filter` parameters (e.g., `source="db", team_ids=[5], crawler_filter="game-stats"` processes only game-stats for team 5).
- [ ] **AC-4**: Existing CLI behavior (`bb data crawl`, `bb data sync`, `bb data load`) is not broken -- invoking without `team_ids` processes all teams as before. The default `source="yaml"` path is unchanged.
- [ ] **AC-5**: Tests verify both the filtered (single-team) and unfiltered (all-teams) code paths for both crawl and load.

## Technical Approach
Read both `src/pipeline/crawl.py` and `src/pipeline/load.py` to understand the current iteration patterns. Both have a `source` parameter that defaults to `"yaml"`. The `source="db"` path reads teams from the database where `TeamEntry.internal_id` is populated -- this is the path that supports `team_ids` filtering. Add the `team_ids` parameter to both `crawl.run()` and `load.run()`, filtering the team list early in each function's `source="db"` branch (before any crawlers/loaders run). Verify the parameter propagates through whatever abstraction layers exist.
