## Description
After this story is complete, a pure Python function will analyze structured pitching history and produce a `StarterPrediction` dataclass containing a named prediction (or committee assessment), a rest/availability table for the top arms, rotation pattern classification, confidence tier, and recent game logs for each candidate. This engine has no external dependencies and always produces a result (at minimum a rest table) when pitching data is available.

## Context
This is the core intelligence of the predicted starter feature. The engine must work standalone (Tier 1) -- reports always include a prediction section when pitching data exists, regardless of whether LLM credentials are configured. The algorithm detects rotation patterns (ace-dominant, 2-man, 3-man, committee), computes rest-based availability, and assigns confidence tiers per the coach's domain model. It must handle HS-level baseball patterns: small rotations (ace-plus-committee is most common), inconsistent rest, matchup-driven deviations (~20-30% of the time), and limited data early in the season.

## Acceptance Criteria
- [ ] **AC-1**: A module `src/reports/starter_prediction.py` exists with a function `compute_starter_prediction(pitcher_profiles: dict[str, dict], pitching_history: list[dict], workload: dict[str, dict] | None = None) -> StarterPrediction` that returns a prediction dataclass. The `workload` parameter accepts `get_pitching_workload()` output for rest table population; when None, rest table workload fields are omitted.
- [ ] **AC-2**: `StarterPrediction` dataclass contains per TN-2:
  - `confidence`: one of `"high"`, `"moderate"`, `"low"`, `"suppress"`
  - `predicted_starter`: dict with player info and reasoning string (None if low/suppress)
  - `alternative`: dict with player info (None if high/suppress)
  - `top_candidates`: list of up to 3 dicts, each with `player_id`, `name`, `jersey_number`, `likelihood` (float 0-1, internal -- not for display), `reasoning` string, `games_started` int, `recent_starts` list of game log dicts
  - `rotation_pattern`: string (e.g., `"ace-dominant"`, `"2-man rotation"`, `"3-man rotation"`, `"committee"`)
  - `rest_table`: list of dicts for up to 3 arms (1-2 starters by GS + highest-appearance relievers; always populated when data exists) with name, GS, days_rest, last_outing_date, last_outing_pitches, workload_7d
  - `bullpen_order`: list of up to 3 dicts for relievers ranked by first-relief-appearance frequency (name, jersey_number, frequency, games_sampled)
  - `data_note`: string or None (granularity per TN-2: 1-2 games / 3 games / tournament density; engine is never called at 0 rows -- that case is handled by callers passing `starter_prediction = None`)
- [ ] **AC-3**: **Minimum data gate**: The engine is never called with 0 pitching rows -- callers check `get_pitching_history()` result length and pass `None` downstream when empty (see E-212-04 AC-3, E-212-05 AC-10). When called with 1-3 completed games, returns `confidence = "suppress"` with only `rest_table`, `bullpen_order`, and `data_note` populated. No rotation detection is attempted. The `data_note` distinguishes between 1-2 games ("Rest intervals not yet available -- N game(s) played") and 3 games ("Rotation pattern unclear -- 3 games played, rest data accumulating") per TN-2.
- [ ] **AC-4**: Role classification works per TN-2 -- pitchers are classified as primary starter, spot starter, or reliever based on their GS/G ratio and total starts from the pitcher profiles data.
- [ ] **AC-5**: Rotation detection identifies the pattern type per TN-2: ace-dominant (one pitcher 60%+ of starts), two-man, three-man, or committee (no clear pattern). Given a clear 3-man rotation (A-B-C-A-B-C), the engine predicts A next with highest internal likelihood.
- [ ] **AC-6**: Rest and availability assessment per TN-2: **Within-1-day exclusion**: a pitcher whose most recent appearance was within 1 calendar day (0 or 1 days rest) is excluded from the starter candidate list. **75+/4-day exclusion**: 75+ pitches with fewer than 4 days rest = pitcher is excluded from the starter candidate list (behavioral prediction heuristic per TN-2). **10+ day gap**: flags "availability unknown" (not positive rest) in the candidate's reasoning string. Recent 7-day workload (from `workload` parameter, sourced externally via `get_pitching_workload()`) factors into availability assessment. High pitch count last outing (above pitcher's own average, sourced from the pitcher's most recent appearance row in the pitching history data): the candidate's `reasoning` string contains a workload flag (e.g., "high pitch count last outing"). Low pitch count last outing (below 50 pitches): the candidate's `reasoning` string contains an early-availability note (e.g., "low pitch count last outing"). Both are observable in the `reasoning` field by substring matching in tests.
- [ ] **AC-7**: Confidence tiers are assigned per TN-2:
  - High: top candidate has 2x the signal weight of next. Output names a starter with no qualifier.
  - Moderate: rotation pick clear but a plausible matchup alternative exists. **Deterministic trigger**: any other rested starter -- defined as a `primary_starter` or `spot_starter` (per AC-4) who is not excluded by the within-1-day rule or the 75+/4-day rule (per AC-6) -- with K/9 more than 2.0 higher than the rotation leader triggers moderate confidence with the high-K pitcher named as alternative. Output names with caveat + alternative.
  - Low / Committee: 3+ similar candidates or no clear rotation. Does NOT name a single pick. A 30% top candidate = committee.
  - Suppress: fewer than 4 games or no pitching data.
- [ ] **AC-8**: Edge cases per TN-2 -- each testable by verifying the observable output field noted:
  - Fewer than 4 completed games: `confidence = "suppress"`, `predicted_starter = None`, `rest_table` populated
  - Ace-dominant: `confidence = "high"`, and when GS% >= 70% (pitcher's GS / total team games), `predicted_starter.reasoning` contains a workload warning string (e.g., "heavy usage")
  - Spot-starting reliever: `predicted_starter` or candidate `reasoning` contains an anomaly note; pitcher is not classified as `primary_starter` until 2+ starts accumulated
  - All NULL appearance_order: engine still produces a prediction (not suppress) by inferring starters from most-IP-per-game fallback per TN-2. Only returns `confidence = "suppress"` when there is genuinely insufficient data (1-3 games)
  - 10+ day gap for a rotation member: candidate's `reasoning` contains "availability unknown"
  - Committee (3+ similar candidates): `confidence = "low"`, `predicted_starter = None`, `rotation_pattern = "committee"`
  - 75+ pitches with fewer than 4 days rest: pitcher is excluded from `top_candidates` (not just deprioritized) -- behavioral prediction heuristic per TN-2
- [ ] **AC-9**: Each candidate's `recent_starts` contains their last 3-5 starts (appearance_order=1) with: game_date, ip_outs, pitches, so, bb, decision, rest_days_from_previous_start.
- [ ] **AC-10**: `rest_table` is always populated when pitching data exists, regardless of confidence tier. **Selection**: top 1-2 starters by GS + fill remaining slots (up to 3 total) with highest-total-appearance relievers. Each entry has: name, jersey_number, games_started, last_outing_date, days_since_last_appearance, last_outing_pitches, workload_7d. **Data sources**: `last_outing_date`, `days_since_last_appearance`, and `workload_7d` are sourced from `get_pitching_workload()` output (existing shared function in `src/api/db.py`). `last_outing_pitches` is sourced from the pitcher's most recent appearance row in the pitching history data (from `build_pitcher_profiles()`), NOT from `get_pitching_workload()`. `games_started` comes from `build_pitcher_profiles()`. Do not reimplement workload computation for the fields that come from `get_pitching_workload()`.
- [ ] **AC-11**: `bullpen_order` is populated when pitching data exists. Ranks relievers by frequency of first-relief-appearance (appearance_order=2) across games. Up to 3 relievers with name, jersey_number, frequency count, and games sampled. Empty list when no relief data is available.
- [ ] **AC-12**: **Tournament density flag**: When 3+ games in the last 7 days of game history fall on consecutive or near-consecutive days (gaps of 0-1 days), `data_note` includes "Compressed schedule detected -- rotation predictions less reliable."
- [ ] **AC-13**: Tests cover: clear 3-man rotation, 2-man rotation, ace-dominant pattern (with GS% >= 70% workload flag), committee (4 pitchers with similar starts), team with only 3 games (suppress with appropriate data_note granularity), all NULL appearance_order with IP data (infers starters from most-IP fallback), 10+ day gap flagged as availability unknown, mixed starts/relief for same pitcher affecting 7-day workload, within-1-day exclusion, 75+ pitch / short rest gate, high pitch count reasoning flag in candidate string, low pitch count (<50) reasoning flag in candidate string, spot-starting reliever flagged as anomalous, moderate confidence triggered by K/9 delta, bullpen order ranked correctly, tournament density flag triggered. (The 0-row case is not tested at the engine level -- callers are responsible for checking empty history and passing `starter_prediction = None` per AC-3.)

## Technical Approach
The engine is a pure function with no DB access, no HTTP calls, no side effects. It receives the pitcher profiles dict from E-212-01's `build_pitcher_profiles()`. The algorithm follows the steps in TN-2: minimum data gate, classify roles, detect rotation pattern, assess rest/availability, compute internal likelihoods, assign confidence tier, build rest table. Internal likelihoods drive the ranking but are not displayed in the default report view -- they exist for tie-breaking and for the LLM tier to reference.

## Notes
- HS rotation patterns per coach consultation: mostly ace-plus-committee or 2-man; true 3-man at competitive varsity; 4-man rare (mostly Legion). Matchup deviations ~20-30% of the time. Weight: 70% rotation sequence, 30% matchup factors.
- The `reasoning` string for each candidate is a brief algorithmic explanation (e.g., "Next in 2-man rotation, 4 days rest, 67 pitches last outing"). It is NOT LLM-generated prose -- it is constructed from the data fields.
- Internal `likelihood` floats are never displayed to coaches in the primary report view. They exist to rank candidates and for the LLM tier to reference. The coach sees plain-English reasoning, not percentages.
