## Description
After this story is complete, shared query functions in `src/api/db.py` will extract a team's complete pitching appearance history for a season -- including computed rest days, team game numbers, and all fields needed by both the deterministic rotation analysis engine (Tier 1) and the LLM prompt builder (Tier 2). Both the report generator and the dashboard opponent detail route will import these functions. A Python post-processing function will reshape the raw query results into a structured per-pitcher format ready for downstream consumption.

## Context
This is the data foundation for the predicted starter feature. The query lives in `src/api/db.py` as a shared function, following the established pattern (`get_pitching_workload()` is the precedent for cross-surface shared queries). The query must serve two consumers: the deterministic engine (needs chronological data grouped by pitcher with computed fields) and the LLM prompt builder (needs human-readable formatted data). One SQL query returns the canonical fact set; Python reshapes it into the appropriate views. The query uses the game ordering convention.

## Acceptance Criteria
- [ ] **AC-1**: A function `get_pitching_history(team_id: int, season_id: str, *, db: sqlite3.Connection | None = None) -> list[dict]` exists in `src/api/db.py` that returns all pitching appearances for the team/season ordered chronologically per TN-1. The `db` parameter follows the `get_pitching_workload()` pattern (optional pre-existing connection). No `reference_date` parameter -- all computed fields (LAG rest_days, DENSE_RANK team_game_number) are relative to the data itself. 7-day workload data comes from `get_pitching_workload()` separately.
- [ ] **AC-2**: Each row includes: `player_id`, `first_name`, `last_name`, `jersey_number`, `game_id`, `game_date`, `start_time`, `ip_outs`, `pitches`, `so`, `bb`, `h`, `r`, `er`, `bf`, `decision`, `appearance_order`, `rest_days` (via LAG, partitioned by player_id, ordered by game_date ASC / start_time ASC NULLS LAST), and `team_game_number` (via DENSE_RANK, ordered by game_date ASC / start_time ASC NULLS LAST only -- no appearance_order, so all appearances within the same game share the same team_game_number).
- [ ] **AC-3**: `rest_days` is computed as days since the same pitcher's previous appearance of ANY type (start or relief). NULL for each pitcher's first appearance of the season.
- [ ] **AC-4**: A function `build_pitcher_profiles(history: list[dict]) -> dict[str, dict]` (public, in `src/api/db.py`) groups the raw query results by `player_id`. Each per-pitcher dict has the following keys:
  - `player_id`: str
  - `first_name`: str
  - `last_name`: str
  - `jersey_number`: str | None
  - `appearances`: list[dict] -- all appearances, chronological
  - `starts`: list[dict] -- filtered to `appearance_order = 1` when appearance_order data is available; when ALL rows for a pitcher have `appearance_order IS NULL`, falls back to the most-IP-per-game heuristic (the pitcher with the highest `ip_outs` in each game is treated as the starter for that game). This fallback is critical for scouted teams before backfill.
  - `total_games`: int
  - `total_starts`: int
  - `season_ip_outs`: int (sum of ip_outs across all appearances)
  - `season_k9`: float | None (computed from total so and season_ip_outs; None if 0 IP)
  - `start_to_start_rest`: list[int] -- days between consecutive starts (empty if fewer than 2 starts)
  
  The function does NOT compute 7-day workload -- that is provided by the existing `get_pitching_workload()` function (see E-212-02 AC-10).
- [ ] **AC-5**: Tests cover: normal team with 10+ games and 4-5 pitchers, team with a single pitcher (all starts), team with NULL appearance_order rows, team with no completed games (empty result).

## Technical Approach
The query uses `LAG()` and `DENSE_RANK()` window functions per TN-1. Both functions are public in `src/api/db.py` (shared between reports and dashboard). The post-processing function groups rows by player_id and computes derived fields (start-to-start rest, role indicators) that the deterministic engine needs. Both functions are pure data extraction with no business logic -- the rotation analysis lives in E-212-02.
