# E-220-10: Scouting Spray In-Memory Pipeline

## Epic
[E-220: Perspective-Aware Data Architecture](epic.md)

## Status
`TODO`

## Description
After this story is complete, the scouting spray chart pipeline (ScoutingSprayChartCrawler + ScoutingSprayChartLoader) passes data in-memory from crawl to load rather than writing JSON files to disk. The scouting spray crawl+load cycle produces zero new spray files in `data/raw/`.

## Context
This story mirrors E-220-05 (scouting boxscore in-memory) for the spray chart sub-pipeline. The spray stages run as part of the scouting sync (`run_scouting_sync` -> `_run_spray_stages` in trigger.py) and the CLI scout command. They currently write spray JSON files to `data/raw/{season}/scouting/{public_id}/spray/` and the loader reads from disk. After this story, the spray pipeline uses in-memory handoff following the same pattern established in E-220-05.

## Acceptance Criteria
- [ ] **AC-1**: Given a scouting sync triggered via `run_scouting_sync()`, when the spray crawl+load cycle completes, then no new files are written under `data/raw/{season}/scouting/{public_id}/spray/`
- [ ] **AC-2**: `ScoutingSprayChartCrawler` returns spray data in-memory (per TN-12 pattern)
- [ ] **AC-3**: `ScoutingSprayChartLoader` accepts in-memory spray data instead of a filesystem path
- [ ] **AC-4**: The `_run_spray_stages()` function in trigger.py passes data in-memory between crawler and loader
- [ ] **AC-5**: Given scouting spray data loaded for a team, then every spray_charts row has `perspective_team_id` set to the scouted team's integer PK (inherited from E-220-04)
- [ ] **AC-6**: Standalone CLI commands `bb data crawl --crawler spray-chart` and `bb data load --loader spray-chart` work with the in-memory pipeline (the `crawl_all()` path and any callers that previously used disk intermediary and `CrawlResult` fields are updated)
- [ ] **AC-7**: All existing scouting spray tests pass (updated for new API signatures)

## Technical Approach
Follow the same pattern as E-220-05: modify the crawler to return data in-memory, modify the loader to accept in-memory data, update the orchestration in trigger.py's spray section and CLI. The member-team `SprayChartLoader` continues to use path-based loading (unchanged by this story).
