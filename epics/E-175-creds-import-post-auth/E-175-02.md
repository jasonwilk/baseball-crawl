# E-175-02: Execute POST /auth Curls and Import Credentials

## Epic
[E-175: Fix `bb creds import` for POST /auth Curl Commands](epic.md)

## Status
`TODO`

## Description
After this story is complete, `bb creds import` will detect POST /auth curls (using the mechanism from E-175-01), execute the HTTP request, parse the response through the existing gc_auth JSON handler, extract non-token credentials from request headers, merge everything into `.env`, and provide clear error messages for expired signatures, auth failures, and non-token body types.

## Context
E-175-01 adds POST /auth detection and the `PostAuthCurlDetected` exception to the parser. This story adds the HTTP execution layer in `creds.py` (the CLI layer that already imports httpx). When `import_creds()` catches `PostAuthCurlDetected` from `parse_credentials()`, it checks the body type for safety (TN-2: logout is destructive and must be rejected pre-execution), executes the HTTP request for safe body types, parses the response, and merges credentials from both the response body (tokens) and request headers (gc-device-id, gc-client-id, gc-app-name) into `.env`. Error handling must distinguish gc-signature expiration (HTTP 400) from auth failure (HTTP 401) per TN-6. The request body may contain a password and must never be logged or displayed (TN-9).

## Acceptance Criteria
- [ ] **AC-1**: Given a valid POST /auth curl with `{"type":"refresh"}` body, when `bb creds import` runs, then the HTTP request is executed and the response's access and refresh tokens are written to `.env` as `GAMECHANGER_REFRESH_TOKEN_{PROFILE}` and (for mobile) `GAMECHANGER_ACCESS_TOKEN_{PROFILE}`.
- [ ] **AC-2**: Given a valid POST /auth curl, when the HTTP request succeeds (200), then gc-device-id, gc-client-id, gc-app-name from the curl's request headers AND `GAMECHANGER_BASE_URL` derived from the curl's URL (scheme + host) are also written to `.env` alongside the response tokens -- per TN-7.
- [ ] **AC-3**: Given a POST /auth curl whose gc-signature has expired, when `bb creds import` runs and receives HTTP 400, then the error message says the signature has expired and to copy a fresh curl -- per TN-6.
- [ ] **AC-4**: Given a POST /auth curl that receives HTTP 401, when `bb creds import` runs, then the error message indicates authentication failure and to copy a fresh curl -- per TN-6.
- [ ] **AC-5a**: Given a POST /auth curl with `{"type":"logout"}` body, the tool rejects the curl WITHOUT making an HTTP call and displays: "This is a logout curl. Running it would end your GameChanger session. Copy a POST /auth curl with `{"type":"refresh"}` body instead." -- per TN-2 (destructive category).
- [ ] **AC-5b**: Given a POST /auth curl with `{"type":"client-auth"}` or `{"type":"user-auth"}` body, when the response does not contain importable access/refresh tokens (including non-JSON responses), then a clear error message tells the user to copy a refresh-type POST /auth curl instead -- per TN-2 (non-token, safe category) and TN-6.
- [ ] **AC-6**: Given a POST /auth curl with `{"type":"password","password":"secret"}` in `--data-raw`, the password value never appears in console output, log output, or `.env` at any point during execution -- per TN-9.
- [ ] **AC-7**: Given a POST /auth curl, when a network error occurs (connection refused, timeout), then a clear error message about connectivity is displayed -- per TN-6.
- [ ] **AC-8**: Existing import paths (JSON response body, GET curls, bare JWTs) continue to work unchanged. All existing tests in `test_credential_parser.py` and `test_cli_creds.py` pass.
- [ ] **AC-9**: Given a POST /auth curl that receives an unexpected HTTP status (not 200/400/401, e.g., 403 or 500), the error message includes the status code and tells the user to copy a fresh curl and try again -- per TN-6 "Other 4xx/5xx" row.
- [ ] **AC-10**: New tests cover (using `respx` for httpx mocking): successful execution with mocked httpx response (refresh body type), successful execution with password body type (same response shape, verify tokens imported), 400 error handling, 401 error handling, unexpected HTTP status (e.g., 500) produces generic error per TN-6, logout pre-execution rejection, non-token response handling (including non-JSON 200 body), header credential extraction alongside response tokens, GAMECHANGER_BASE_URL written from request URL, password-in-body not logged, and network error handling.
- [ ] **AC-11**: Run test scope discovery per `.claude/rules/testing.md` -- find and run all test files that import from `credential_parser` or `creds`.

## Technical Approach
In `import_creds()`, catch `PostAuthCurlDetected` from `parse_credentials()`. Before making any HTTP call, check the body type (parse the body JSON for `"type"` field) -- if `"logout"`, reject immediately per TN-2 without executing. For all other body types, execute the request per TN-5: use a plain `httpx.post()` (NOT `create_session()` -- see TN-5 HTTP discipline exception), pass all extracted headers verbatim, send body as raw content, 30-second timeout, no redirects. Parse the response: on 200, feed the JSON body through `_parse_json_credentials()` (existing gc_auth handler); on 400/401/other, raise `CredentialImportError` with the appropriate message per TN-6. On 200, catch ALL `CredentialImportError` variants from `_parse_json_credentials()` and map to the "no tokens" guidance.

For credential merging per TN-7: combine tokens from the response with non-token headers from the request (gc-device-id, gc-client-id, gc-app-name). Pass the merged dict to `merge_env_file()`.

Use `respx` for httpx mocking in tests. Verify error messages match the TN-6 matrix. Verify no credential values appear in captured log output.

## Notes
- The existing `_parse_json_credentials()` function handles the gc_auth response shape (`{"type":"token","access":{"data":"jwt..."},"refresh":{"data":"jwt..."}}`) and routes tokens through `_route_gc_token()`. No new JSON shape handling should be needed.
- For body types that return 200 but without importable tokens: `_parse_json_credentials()` may raise various `CredentialImportError` variants -- `"Invalid JSON"` (non-JSON body like a bare JWT), `"unrecognised shape"` (unexpected JSON structure), or `"No credentials could be extracted"` (valid shape but no tokens for profile). The CLI must catch ALL of these on a 200 response and map to the "copy a refresh-type curl" guidance per TN-6, not expose raw parse errors.
- Logout curls are rejected pre-execution (no HTTP call) per TN-2 -- this is a safety check before any network activity.
- httpx is already a project dependency (used in `creds.py` for API health checks). `respx` should be used for test mocking. No new production dependencies needed.
