# E-175-01: Detect POST /auth Curls and Extract Request Components

## Epic
[E-175: Fix `bb creds import` for POST /auth Curl Commands](epic.md)

## Status
`TODO`

## Description
After this story is complete, `credential_parser.py` will detect POST /auth curl commands and return the extracted request components (URL, headers, body, method) as a structured result instead of attempting to parse gc-token as a credential. The parser remains a pure parser -- no I/O or network calls. The misleading access-token warning message in `_resolve_web_token_key()` will also be corrected.

## Context
Currently, `parse_curl()` processes all curls identically: extract headers, route gc-token to env keys, validate. POST /auth curls fail because gc-token contains a client token (type="client") which is silently discarded. The parser needs a new code path that detects POST /auth curls early (per TN-3), extracts all request components needed for HTTP execution (per TN-4), and signals the caller that execution is required. The warning message fix (TN-8) is included here because it's a one-line change in the same file with no dependencies.

## Acceptance Criteria
- [ ] **AC-1**: Given a curl command where the URL path is exactly `/auth` and the method is POST (via `-X POST` or implied by `--data-raw`/`-d`/`--data`/`--data-binary`), `parse_credentials()` raises `PostAuthCurlDetected` (a subclass of `CredentialImportError`) carrying the extracted URL, headers dict, and body string -- per TN-1, TN-3, and TN-4.
- [ ] **AC-2**: Given a curl command where the URL path does NOT end with `/auth` (e.g., a normal GET request), `parse_credentials()` behavior is unchanged from the current implementation.
- [ ] **AC-3**: Given a POST /auth curl with a `--data-raw '{"type":"password","password":"secret"}'` body, the body string is captured in the structured result and is not logged at any log level.
- [ ] **AC-4**: Given a POST /auth curl with headers gc-device-id, gc-client-id, gc-app-name, gc-signature, gc-timestamp, gc-token, and Content-Type, all headers are captured in the `PostAuthCurlDetected` exception's headers dict by raw string splitting (NOT routed through `_process_header()`). In particular, Content-Type must be preserved -- it is in `_SKIP_HEADERS` and would be silently dropped if processed through the normal header pipeline.
- [ ] **AC-5**: The warning message in `_resolve_web_token_key()` no longer directs users to "copy a curl command from a POST /auth request" -- updated per TN-8.
- [ ] **AC-6**: All existing tests in `test_credential_parser.py` continue to pass without modification (no regressions in GET curl, JSON, or bare JWT paths).
- [ ] **AC-7**: New tests cover: POST /auth curl detection (positive and negative), POST /auth curl with no gc-token header (client-auth type) still triggers `PostAuthCurlDetected`, body extraction from `--data-raw`/`-d`/`--data`/`--data-binary`, header extraction completeness (including Content-Type preservation), password non-logging (verify via `caplog` that password value does not appear in log output), and the updated warning message text.
- [ ] **AC-8**: Run test scope discovery per `.claude/rules/testing.md` -- find and run all test files that import from `credential_parser`.

## Technical Approach
The parser needs to detect POST /auth curls per TN-3, extract request components per TN-4, and raise `PostAuthCurlDetected` per TN-1. The `--data-raw`/`-d`/`--data`/`--data-binary` flags are currently consumed and discarded (lines 202-208 of `credential_parser.py`). For POST /auth curls, the body value must be captured instead. For non-POST-auth curls, the current skip behavior should be preserved. Headers must be captured by raw string splitting, NOT through `_process_header()` (which filters out Content-Type and other non-credential headers needed for execution).

The warning message fix in `_resolve_web_token_key()` is a text change at line ~270.

## Notes
- The `gc-signature` and `gc-timestamp` headers are not in `_SKIP_HEADERS` -- they currently fall through to "Ignoring unrecognised header" debug log. For POST /auth curls they are captured in the headers dict. For non-POST-auth curls, the current behavior (debug log, skip) is acceptable.
- The body may contain a password (TN-9). The parser captures it but must not log it. The `logger.debug("Parsed credentials: ...")` call should not include the body.
