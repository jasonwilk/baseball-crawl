# E-211-01: Stop UUID Contamination at All Three Pipeline Sources

## Epic
[E-211: Fix Opponent-Perspective UUID Contamination](epic.md)

## Status
`TODO`

## Description
After this story is complete, no pipeline path will store boxscore-derived opponent-perspective UUIDs as `gc_uuid` on tracked team rows. The game loader's `_ensure_team_row` wrapper will pass `gc_uuid=None` to the shared `ensure_team_row`, using the boxscore identifier only as a name fallback. The UUID opportunism methods in scouting_loader and scouting_crawler (`_record_uuid_from_boxscore`) will be removed entirely -- the game loader already creates team rows during boxscore loading.

## Context
The GC API returns different UUIDs for the same team depending on the caller's perspective. Three pipeline paths currently store boxscore keys as `gc_uuid`, contaminating the teams table. This is the root cause of missing spray charts and duplicate players in standalone reports. See epic Technical Notes TN-1 and TN-2.

## Acceptance Criteria
- [ ] **AC-1**: Given a member team's boxscore is processed by the game loader, when an opponent team is encountered, then `ensure_team_row` is called with `gc_uuid=None` (not the boxscore key or `summary.opponent_id`). The opponent name and season_year are still passed for dedup. When `opponent_name` is None, the original boxscore identifier (UUID string) is passed as the `name` parameter to preserve unique team row naming (never `name=None`).
- [ ] **AC-2**: The scouting loader's `_record_uuid_from_boxscore` method (`src/gamechanger/loaders/scouting_loader.py`) is removed entirely. The game loader already creates team rows via name+season_year during boxscore loading, making this UUID opportunism pattern redundant.
- [ ] **AC-3**: The scouting crawler's `_record_uuid_from_boxscore` method (`src/gamechanger/crawlers/scouting.py`) is removed entirely. Same rationale as AC-2 -- the game loader handles team row creation before any FK-dependent inserts.
- [ ] **AC-4**: Existing tests pass. New tests verify that opponent-perspective UUIDs are not passed as `gc_uuid` to `ensure_team_row` in any of the three paths.
- [ ] **AC-5**: The game loader still correctly resolves own-team ID (no change to own-team handling in `_resolve_team_ids`).

## Technical Approach
Three changes to three files, per TN-1 and TN-2. The game loader's `_ensure_team_row` wrapper is the primary change -- it should stop passing the boxscore-derived UUID as `gc_uuid` and instead use the identifier only as a name fallback. The two `_record_uuid_from_boxscore` methods in scouting_loader and scouting_crawler should be removed entirely -- the game loader already creates team rows during boxscore loading, making the UUID opportunism pattern redundant. `ensure_team_row` itself is not modified.
