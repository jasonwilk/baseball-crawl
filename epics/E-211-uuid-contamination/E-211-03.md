# E-211-03: Document and Execute Data Cleanup

## Epic
[E-211: Fix Opponent-Perspective UUID Contamination](epic.md)

## Status
`DONE`

## Description
After this story is complete, all contaminated opponent-perspective gc_uuids will be NULLed on tracked team rows, duplicate player records from wrong-perspective plays loading will be removed, and the cleanup procedure will be documented as a research artifact for operator reference.

## Context
Existing data has been contaminated by the bug fixed in E-211-01. Waverly (team 93) has a confirmed wrong gc_uuid; other tracked teams may also be affected. The Waverly plays load created ~20 duplicate player records with opponent-perspective UUIDs. Cleanup is a one-time operation via documented SQL, not a permanent CLI command. See epic Technical Notes TN-4.

## Acceptance Criteria
- [ ] **AC-1**: A research artifact at `/.project/research/E-211-uuid-cleanup.md` contains a documented procedure with SQL queries for: (a) identifying tracked teams with likely contaminated gc_uuids, (b) NULLing contaminated gc_uuids, (c) identifying opponent-perspective games via a hybrid approach (operator lists `game_id`s from `boxscores/*.json` filenames in the scouting directory, then SQL identifies DB games NOT in that list), (d) deleting data from opponent-perspective games in FK-safe order, (e) removing stale plays JSON files from the scouting directory.
- [ ] **AC-2**: The cleanup SQL for gc_uuid NULLing targets tracked teams that have a `public_id` (can be re-resolved by search). It does NOT affect member teams or tracked teams without a public_id.
- [ ] **AC-3**: The duplicate player cleanup SQL uses a game-based identification approach: identify opponent-perspective games (games where the scouted team appears as an opponent in another team's boxscore, not in the scouted team's own `games.json`), then delete all data from those games in FK-safe order per TN-4 Phase 2: `play_events` -> `plays` -> `player_game_batting` -> `player_game_pitching` -> `spray_charts` -> `reconciliation_discrepancies` -> `player_season_batting` -> `player_season_pitching` -> `team_rosters` -> `players`. Orphaned players (no remaining stat rows) are deleted last.
- [ ] **AC-4**: The research artifact includes a verification query that confirms no tracked team gc_uuid matches an opponent-perspective UUID (post-cleanup validation).
- [ ] **AC-5**: The research artifact includes operator instructions for re-running the scouting pipeline after cleanup (`bb data scout`) to re-resolve gc_uuids and regenerate clean data.

## Technical Approach
Write a research artifact with a documented operator procedure. For gc_uuid cleanup: query tracked teams where gc_uuid is non-NULL and public_id is non-NULL (the search resolver can re-resolve them). For opponent-perspective game cleanup: use a game-based approach -- the operator lists `game_id`s from boxscore filenames in the scouting directory, then SQL identifies games in the `games` table (via `game_id`, `home_team_id`, `away_team_id`) that involve the scouted team but are NOT in the boxscore file list. Those are the opponent-perspective games. Delete all downstream data from those games (`plays`, `play_events`, `player_game_batting`, `player_game_pitching`, etc.) using the `game_id` as the selection key. Then delete orphaned `players` rows (those with no remaining rows in any stat table). Remove corresponding stale JSON files from the plays directory. All steps should be reviewable by the operator before execution.
