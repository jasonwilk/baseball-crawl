## Description
After this story is complete, the reconciliation engine will correct pitcher attribution errors by reassigning `plays.pitcher_id` using boxscore pitcher order (from cached JSON) and BF boundary detection. The `bb data reconcile --execute` CLI command applies corrections, logs them to the discrepancy table with full audit trail, and reports summary statistics showing the accuracy improvement. A retroactive run against all existing games raises FPS% match rate from ~55% to 90%+.

## Context
E-198-01 builds the detection engine that identifies which games have pitcher attribution errors and which signals are affected. This story adds the correction algorithm -- the highest-value component. The algorithm uses the boxscore's authoritative pitcher appearance order (extracted from cached JSON) and per-pitcher BF counts to locate misattribution boundaries and reassign plays to the correct pitcher. Expert consensus: BF + pitcher order covers ~90% of boundary errors.

## Acceptance Criteria
- [ ] **AC-1**: `reconcile_game(conn, game_id, dry_run=False)` applies pitcher attribution corrections when `dry_run=False` per the correction algorithm in Technical Notes:
  - Extracts boxscore pitcher order and per-pitcher BF counts from cached boxscore JSON (per Pitcher Order Extraction in Technical Notes; backward-compat: tries `{game_id}.json` then `{game_stream_id}.json`)
  - Walks through plays in `play_order` sequence, assigning the first `BF[pitcher_1]` plays to pitcher_1, the next `BF[pitcher_2]` plays to pitcher_2, etc. (this naturally handles starter mismatches -- no separate starter rule needed)
  - Updates `plays.pitcher_id` in place via UPDATE for any plays whose pitcher_id changed
  - Writes discrepancy rows with `status = 'CORRECTED'` and `correction_detail` JSON containing old_pitcher_id, new_pitcher_id, and play_order range
- [ ] **AC-2**: After correction, the engine re-checks BF, SO, and BB counts per pitcher to verify improvement. If post-correction BF still doesn't match boxscore, the discrepancy status is `AMBIGUOUS` (not `CORRECTED`).
- [ ] **AC-3**: `bb data reconcile --execute` runs fresh detection and applies corrections in a single pass (per AC-5 self-contained semantics) and prints a summary:
  - Games processed, games corrected, games unchanged, games with remaining ambiguity
  - Per-signal match rates before and after correction (pitcher signals)
  - Total plays reassigned
- [ ] **AC-4**: `bb data reconcile --summary` shows aggregate statistics across ALL reconciliation records in the `reconciliation_discrepancies` table (not just the most recent run): per-signal match rates for pitcher and batter categories, correction counts, remaining gaps. Note: actual FPS% accuracy improvement requires re-running the E-195 validation query against the corrected plays data -- this is a manual verification step, not part of the `--summary` output.
- [ ] **AC-5**: `--execute` is self-contained: each invocation runs fresh detection AND correction in a single pass with its own `run_id`. It does NOT require or consume a prior dry-run. Corrections are idempotent -- re-running `--execute` on an already-corrected game produces the same result without double-applying corrections (correcting an already-correct pitcher_id is a no-op).
- [ ] **AC-6**: Edge cases handled:
  - Pitcher re-entry (same pitcher_id appears twice in boxscore order): detect and log as AMBIGUOUS
  - Games with only one pitcher per team: skip correction (no boundary to find)
  - Games with no boxscore pitching data: skip with warning
  - Games with no cached boxscore JSON: fall back to `player_game_pitching` ordered by `id` with a warning (consistent with story 01 detection fallback)
  - Games where plays and boxscore pitcher counts differ (more/fewer pitchers in plays than boxscore): log as AMBIGUOUS
- [ ] **AC-7**: Tests cover: correction of a known misattributed game (plays reassigned, BF matches post-correction), idempotency (second run doesn't change data), edge case handling (single pitcher, re-entry, missing JSON), summary output format.

## Technical Approach
The correction algorithm follows the procedure described in epic Technical Notes: extract boxscore pitcher order from cached JSON, compare against plays pitcher sequence, use BF counts to locate boundaries, and reassign via UPDATE. The `correction_detail` JSON in each discrepancy row provides the audit trail (original pitcher_id, corrected pitcher_id, play_order range). The `--summary` command queries the discrepancy table for aggregate statistics across all runs. See epic Technical Notes for the full correction algorithm, edge cases, pitcher order extraction, and CLI specification.

- BF is the primary boundary signal because it's a hard count (not computed). Pitch count (#P) is secondary confirmation. SO/BB are tertiary.
- The retroactive run against ~101 games should complete in seconds (all data is local SQLite queries, no API calls).
- After the retroactive run completes, re-running the E-195 FPS validation query (from `/.project/research/E-195-validation-results.md`) will show the accuracy improvement. This validates the epic's success criteria.
- Accuracy thresholds (per Technical Notes): 90%+ for FPS%, 95%+ for pitch counts. These are advisory targets for downstream dashboard gating -- the engine reports match rates, it does not enforce thresholds. The `--summary` output should call out pitch count match rates alongside other signals so the operator can assess readiness.
