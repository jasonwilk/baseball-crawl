# E-195-02: Plays Schema and Parser

## Epic
[E-195: Plays Data Ingestion Pipeline -- FPS% and QAB](epic.md)

## Status
`TODO`

## Description
After this story is complete, the database will have `plays` and `play_events` tables (migration 009), and a `PlaysParser` module will parse raw plays JSON into structured data ready for database insertion. The parser extracts batter/pitcher IDs, classifies each event (pitch/baserunner/substitution), counts pitches, and computes `is_first_pitch_strike` and `is_qab` flags per plate appearance. The parser is a pure function with no database dependency -- it takes JSON and returns structured dataclass records.

## Context
This is the most complex story in the epic. The parser must correctly handle template string patterns, UUID extraction, pitcher state tracking across innings, and all 7 QAB conditions. Separating the parser from the loader (E-195-03) enables thorough unit testing against sample data without database setup. The migration and parser are independent of the crawler -- tests use sample data files already on disk.

Data exploration (165 games, 9,398 plays) confirmed 24 distinct outcome types and 486 unique `final_details` patterns. Key simplifications: "Sacrifice Bunt" and "Sacrifice Fly" are explicit outcomes (no heuristic needed), "Intentional Walk" is distinct from "Walk" (IBBs detectable). Full exploration report at `data/raw/plays-exploration/FINDINGS.md`.

## Acceptance Criteria
- [ ] **AC-1**: Migration `009_plays_play_events.sql` creates `plays` and `play_events` tables with all columns, constraints, and indexes per TN-3. Migration is idempotent (`CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`).
- [ ] **AC-2**: `PlaysParser.parse_game(raw_json: dict, game_id: str, season_id: str, home_team_id: int, away_team_id: int) -> list[ParsedPlay]` accepts a raw plays API response with game context and returns a list of dataclass records. Each `ParsedPlay` contains typed fields matching the `plays` table columns, plus a list of `ParsedEvent` dataclass records matching the `play_events` columns (minus auto-generated IDs).
- [ ] **AC-3**: Each `ParsedPlay` record includes: `play_order`, `inning`, `half`, `season_id`, `batting_team_id` (derived from `half` + home/away team IDs), `batter_id`, `pitcher_id` (nullable), `outcome`, `pitch_count`, `is_first_pitch_strike`, `is_qab`, score fields, and a list of classified events.
- [ ] **AC-4**: Pitch events are classified per TN-4 into `pitch_result` values (`ball`, `strike_looking`, `strike_swinging`, `foul`, `foul_tip`, `in_play`). Non-pitch events are classified as `baserunner`, `substitution`, or `other`. "Foul bunt" is classified as a pitch event with `pitch_result = foul` (same as regular "Foul").
- [ ] **AC-5**: `is_first_pitch_strike` is computed accurately for ALL plate appearances per TN-1, including HBP and Intentional Walk outcomes. The flag records what actually happened on the first pitch. HBP and Intentional Walk exclusion from FPS% is handled at query time, not parse time.
- [ ] **AC-6**: `is_qab` is computed per TN-2: any one of the 7 conditions triggers `is_qab = 1`. All 7 conditions are implemented. BB trigger uses `outcome == "Walk"` (distinct from "Intentional Walk"). SAC trigger uses `outcome in ("Sacrifice Bunt", "Sacrifice Fly")`. HHB detection uses case-insensitive substring matching on `final_details` for "line drive" and "hard ground ball". Intentional Walk, Dropped 3rd Strike, and Catcher's Interference are explicitly excluded from QAB.
- [ ] **AC-7**: The 2S+3 QAB condition correctly counts pitches after reaching a 2-strike count per TN-2: all pitches after the second strike count, including the terminal pitch. Fouls on a 2-strike count count as pitches seen but do not advance the strike count. 3 or more pitches after 2 strikes = QAB. Minimum total pitches for this condition: 5.
- [ ] **AC-8**: Batter ID is extracted from the first `${uuid}` in the first `final_details` template. Pitcher ID is tracked via per-half game state per TN-5 (two state variables persisting across innings, updated by substitution events, overridden by explicit "pitching" references in final_details when available). Explicit pitcher references are not present for all K/BB/HBP plays (~45 known instances without); tracked pitcher state is the fallback. Pitcher ID is NULL when unidentifiable (never silently wrong).
- [ ] **AC-9**: Abandoned plate appearances (empty `final_details`) are excluded from the parsed output.
- [ ] **AC-10**: Unknown template patterns in `at_plate_details` are classified as `event_type = 'other'` with `pitch_result = NULL` (graceful degradation, not crash). The `final_details` parser uses regex-based pattern matching (486 unique patterns observed -- not enumerable as a fixed vocabulary).
- [ ] **AC-11**: Unit tests validate parsing against `data/raw/game-plays-fresh.json` and synthetic test cases for conditions not present in the sample fixture (sacrifice bunt/fly, intentional walk, dropped 3rd strike, foul bunt). Real-game fixtures from `data/raw/plays-exploration/` may also be used. Tests cover at least: correct play count (excluding abandoned), correct pitch counts for sampled plays, correct FPS flag for sampled plays, correct QAB flag for plays matching known conditions (walk, XBH, 6+ pitches, sacrifice bunt/fly). Tests also cover: abandoned play skipping, unknown template handling, pitcher state tracking across innings within the same half, Intentional Walk exclusion from QAB.
- [ ] **AC-12**: The parser handles all 24 confirmed outcome types from TN-4 without error. Unrecognized outcome values (future-proofing) are stored as-is in `outcome` without crashing.
- [ ] **AC-13**: Unknown `at_plate_details` templates (classified as `event_type = 'other'`) are logged at WARNING level with `game_id`, `play_order`, and the raw template string, creating a reviewable trail for vocabulary expansion.

## Technical Approach
Create `src/gamechanger/parsers/plays_parser.py` as a pure-function module with dataclass return types. The parser builds a player lookup dict from `team_players`, then iterates through the `plays` array maintaining per-half pitcher state. For each play: classify events via regex, extract batter/pitcher UUIDs, count pitches, evaluate FPS and QAB conditions. Reference `docs/api/flows/plays-ingestion.md` Sections 4-6 for response structure and template vocabulary, and `data/raw/plays-exploration/FINDINGS.md` for the complete outcome vocabulary and pattern catalog. Use `data/raw/game-plays-fresh.json` as the primary test fixture.

## Notes
- The `src/gamechanger/parsers/` package is new. Only `plays_parser.py` lives here for now. Future parsers (if any) would follow the same pattern.
- The parser maintains per-half pitcher state within a game (two state vars: `current_pitcher_top`, `current_pitcher_bottom`) that persists across innings until changed by substitution. See TN-5.
- For the 2S+3 condition: a "Foul" on a 2-strike count does NOT advance the strike count (can't strike out on a foul) but DOES count as a pitch seen after 2 strikes.
- The `final_details` field has 486 unique observed patterns. The parser must use regex-based extraction, not fixed vocabulary matching. See `data/raw/plays-exploration/FINDINGS.md` for the full catalog.
