# E-092-01: Create worktree isolation rule

## Epic
[E-092: Git Worktree Isolation for Agent Teams](epic.md)

## Status

## Description
After this story is complete, a new rule file at `.claude/rules/worktree-isolation.md` will exist with broad glob scope (`**`), providing all agents with guidance on working in git worktrees. The rule covers what agents can and cannot do from a worktree, path awareness, Docker/CLI boundaries, and the expectation that tests are safe to run.

## Context
When implementing agents are spawned with `isolation: "worktree"`, they work in a temporary directory outside the main checkout. They need to understand the constraints: no Docker interaction, no `bb` CLI commands that touch the app or credentials, tests are safe, and file paths in reports should be absolute. Without this guidance, agents may attempt Docker health checks, `bb data sync`, or other operations that assume the main checkout.

## Acceptance Criteria
- [ ] **AC-1**: A new file exists at `.claude/rules/worktree-isolation.md` with YAML frontmatter `paths: ["**"]`
- [ ] **AC-2**: The rule documents that agents in worktrees MUST NOT interact with Docker (no `docker compose` commands, no health checks, no `curl localhost`)
- [ ] **AC-3**: The rule documents that agents in worktrees MUST NOT run `bb` CLI commands that interact with the app, credentials, or database (e.g., `bb data sync`, `bb creds check`, `bb db reset`)
- [ ] **AC-4**: The rule documents that `pytest` is safe to run from worktrees (tests use `tmp_path` / `:memory:` SQLite, mocked HTTP)
- [ ] **AC-5**: The rule documents that agents should use absolute paths in `## Files Changed` reports
- [ ] **AC-6**: The rule documents that agents should NOT modify context-layer files from a worktree unless explicitly assigned a context-layer story
- [ ] **AC-7**: The rule documents that branch management, merging, and worktree cleanup are handled by the main session -- agents should not attempt these operations
- [ ] **AC-8**: The rule documents that `.env` and `data/` are gitignored and therefore ABSENT from worktrees -- agents must not attempt to access credentials, the app database, or data files from a worktree
- [ ] **AC-9**: The rule documents that proxy commands (`bb proxy *`, `./scripts/proxy-*.sh`) should not be run from worktrees -- they assume main-checkout paths for `proxy/data/`

## Technical Approach
Create a new rule file following the established pattern of existing rules (e.g., `.claude/rules/workflow-discipline.md`). The file should be concise and actionable -- agents need to know what to do and what not to do, not the architectural rationale. Use the same YAML frontmatter format with `paths: ["**"]` for broad scope.

Reference the existing Proxy Boundary section in CLAUDE.md as a model for documenting "what runs where" constraints.
