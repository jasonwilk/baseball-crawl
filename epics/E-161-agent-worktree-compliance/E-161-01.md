# E-161-01: Expand Worktree-Guard Hook to Allowlist Mode with Dispatch Detection

## Epic
[E-161: Agent Worktree Compliance Audit and Hardening](epic.md)

## Status
`TODO`

## Description
After this story is complete, the worktree-guard hook will block ALL Write/Edit operations to the main checkout (`/workspaces/baseball-crawl/`) when an epic worktree exists, except for the `.claude/agent-memory/` allowlist. When no worktree exists (non-dispatch work), the hook retains its current always-on denylist for implementation paths (`src/`, `tests/`, `migrations/`, `scripts/`) but allows everything else. This closes the gap that allowed E-158's docs-writer to write to `docs/` in the main checkout during dispatch.

## Context
The current hook protects only four implementation directories. During E-158, docs-writer bypassed the hook by writing to `docs/` -- an unprotected path. Rather than expanding the denylist (which would miss future paths), this story switches to an allowlist model during dispatch. The dispatch-detection mechanism is necessary because agents like claude-architect legitimately Write/Edit to `.claude/rules/`, `.claude/agents/`, etc. in the main checkout outside of dispatch.

CA consultation (2026-03-26) confirmed: allowlist with dispatch detection is the correct approach. Detection uses the existence of `/tmp/.worktrees/baseball-crawl-E-*` directories. Agent memory (`.claude/agent-memory/`) is the sole allowlisted path during dispatch -- the main session uses git/Bash commands (not Write/Edit tools) for its main-checkout operations.

## Acceptance Criteria
- [ ] **AC-1**: Given an active epic worktree exists at `/tmp/.worktrees/baseball-crawl-E-*/`, when an agent issues a Write or Edit to any path under `/workspaces/baseball-crawl/` that does NOT match `.claude/agent-memory/*`, then the hook returns a `deny` permission decision with an error message referencing the epic worktree path.
- [ ] **AC-2**: Given an active epic worktree exists, when an agent issues a Write or Edit to `/workspaces/baseball-crawl/.claude/agent-memory/<any-agent>/MEMORY.md`, then the hook allows the operation (no deny).
- [ ] **AC-3**: Given NO epic worktree exists, when an agent issues a Write or Edit to `/workspaces/baseball-crawl/docs/foo.md` or `/workspaces/baseball-crawl/.claude/rules/bar.md`, then the hook allows the operation (non-dispatch mode).
- [ ] **AC-4**: Given NO epic worktree exists, when an agent issues a Write or Edit to `/workspaces/baseball-crawl/src/foo.py`, then the hook denies the operation (always-on denylist unchanged).
- [ ] **AC-5**: Given any worktree state, when an agent issues a Write or Edit to `/tmp/.worktrees/baseball-crawl-E-NNN/docs/foo.md`, then the hook allows the operation (worktree writes always pass).
- [ ] **AC-6**: The hook's comment header documents both modes (always-on denylist and dispatch-active allowlist) and the detection mechanism.

## Technical Approach
The hook already has the structure: read `file_path` from JSON input, check prefix, decide. The change adds a dispatch-detection step (glob for `/tmp/.worktrees/baseball-crawl-E-*`) and a branching path: if dispatch is active, block everything in main checkout except the agent-memory allowlist; if not, fall through to the existing implementation-path denylist. The deny message should include the detected worktree path so the agent knows where to redirect.

## Notes
- Detection mechanism: `ls /tmp/.worktrees/baseball-crawl-E-* 2>/dev/null` (glob). A stale worktree from a crashed dispatch safely enforces stricter mode; the user can clear it by removing the worktree directory.
- The hook intercepts Write and Edit tool calls only (not Bash), so the main session's git operations are unaffected regardless of mode.
