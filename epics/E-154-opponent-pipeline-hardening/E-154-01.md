# E-154-01: Fix resolver to write public_id and season_year to teams

## Epic
[E-154: Opponent Resolution Pipeline Hardening](epic.md)

## Status
`TODO`

## Description
After this story is complete, `OpponentResolver._ensure_opponent_team_row()` will write `public_id` and `season_year` to the `teams` table during auto-resolution, in addition to the existing `gc_uuid` and `name` fields. The `_resolve_opponent` method will use defensive `public_id` extraction. Both the CLI (`scout_all`) and admin UI (`run_scouting_sync`) scouting paths will work correctly because `teams.public_id` is populated at resolution time.

## Context
This is the core bug fix. Today, `_ensure_opponent_team_row` only writes `gc_uuid` and `name` to the `teams` row. The `public_id` extracted from `GET /teams/{progenitor_team_id}` is passed to `_upsert_resolved` (which writes it to `opponent_links`) but never to the teams table. The admin UI reads `teams.public_id` for scouting sync, which is NULL, breaking the admin scouting path.

The team detail API response also includes `season_year` (confirmed NOT nullable by api-scout), which can be captured opportunistically at zero extra API cost. Note: `_heal_season_year_scouting` in `trigger.py` is still needed for tracked teams scouted via admin UI before opponent resolution runs — this story does not replace that function.

## Acceptance Criteria
- [ ] **AC-1**: Given an opponent with a valid `progenitor_team_id`, when `OpponentResolver.resolve()` runs, then the `teams` row for that opponent has `public_id` set to the value from the team detail API response.
- [ ] **AC-2**: Given an opponent with a valid `progenitor_team_id`, when `OpponentResolver.resolve()` runs, then the `teams` row for that opponent has `season_year` set to the value from the team detail API response (when present).
- [ ] **AC-3**: Given an existing `teams` row whose `public_id` column is already non-NULL (whether matching or different from the fetched value), when `_ensure_opponent_team_row` runs, then the existing `public_id` is preserved unchanged — no overwrite, no error (only-write-when-NULL behavior, per Technical Notes TN-1).
- [ ] **AC-4**: Given an existing `teams` row where a *different* row already has the same `public_id` (UNIQUE collision), when `_ensure_opponent_team_row` runs, then a WARNING is logged with both team IDs and the `public_id` write is skipped — no crash, no UNIQUE constraint violation, per Technical Notes TN-2.
- [ ] **AC-5**: Given a team detail API response where `public_id` is missing or null, when `_resolve_opponent` runs, then a WARNING is logged and resolution continues: the `teams` row is still created/updated with `gc_uuid`, `name`, and `season_year` (when present); `opponent_links` still gets an auto-resolved row with `public_id=NULL`. Per Technical Notes TN-3.
- [ ] **AC-6**: Given an existing `teams` row whose `season_year` column is NULL, when `_ensure_opponent_team_row` runs with a non-NULL `season_year` from the API, then `season_year` is updated. If `season_year` is already non-NULL, it is preserved unchanged (only-write-when-NULL, per Technical Notes TN-1).
- [ ] **AC-7**: Existing tests for `OpponentResolver` continue to pass. New tests cover AC-1 through AC-6.

## Technical Approach
The `_ensure_opponent_team_row` method needs its signature expanded to accept `public_id` and `season_year`. The INSERT statement should include these columns. The existing-row fallback path (when INSERT OR IGNORE fires due to `gc_uuid` conflict) should UPDATE `public_id` and `season_year` when the existing row has NULL values. UNIQUE collision detection for `public_id` should be checked before the UPDATE, per TN-2.

The `_resolve_opponent` method should switch from `team_data["public_id"]` to `team_data.get("public_id")` with a null guard, per TN-3, and pass the extracted fields to `_ensure_opponent_team_row`.
