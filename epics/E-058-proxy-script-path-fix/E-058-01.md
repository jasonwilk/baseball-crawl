## Description
After this story is complete, the three proxy scripts (`proxy-review.sh`, `proxy-report.sh`, `proxy-endpoints.sh`) will resolve `proxy/data/sessions` and `proxy/data/current` relative to the repository root instead of the caller's CWD. This eliminates "does not exist" errors when scripts are invoked from directories other than the repo root.

## Context
All three scripts were created in E-052 and share the same bug: hardcoded relative paths (`proxy/data/sessions`, `proxy/data/current`) that only work when invoked from the repo root. The fix is identical across all three files -- add a `SCRIPT_DIR`/`REPO_ROOT` preamble and prefix the two path variables.

## Acceptance Criteria
- [ ] **AC-1**: `proxy-review.sh` resolves `SESSIONS_DIR` and `CURRENT_LINK` relative to the repo root, not CWD
- [ ] **AC-2**: `proxy-report.sh` resolves `SESSIONS_DIR` and `CURRENT_LINK` relative to the repo root, not CWD
- [ ] **AC-3**: `proxy-endpoints.sh` resolves `SESSIONS_DIR` and `CURRENT_LINK` relative to the repo root, not CWD
- [ ] **AC-4**: Usage/help text in all three scripts continues to show the user-friendly relative path (e.g., `proxy/data/sessions/`) rather than an absolute path
- [ ] **AC-5**: All existing tests continue to pass (`pytest`)

## Technical Approach
Add a two-line preamble after `set -euo pipefail` in each script:
```bash
SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
REPO_ROOT="$(cd "${SCRIPT_DIR}/.." && pwd)"
```
Then change the path assignments:
```bash
SESSIONS_DIR="${REPO_ROOT}/proxy/data/sessions"
CURRENT_LINK="${REPO_ROOT}/proxy/data/current"
```
Leave usage text unchanged -- it shows relative paths for documentation purposes.
