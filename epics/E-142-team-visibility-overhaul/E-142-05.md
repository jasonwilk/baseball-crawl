# E-142-05: Post-Add Flash With Next-Step Hint

## Epic
[E-142: Team Visibility Overhaul](epic.md)

## Status
`TODO`

## Description
After this story is complete, when a team is successfully created via the admin UI, the success flash message includes an inline next-step hint telling the operator what command to run to load data for the new team. This closes the information gap between "team added" and "where's my data?"

## Context
Currently, after `POST /admin/teams/confirm` succeeds, the user is redirected to `/admin/teams?msg=Team added: {team_name}`. Jason reported confusion about the overall flow -- the success message doesn't explain what to do next. Adding a one-liner hint with the `bb` command reduces friction. See TN-5 for design details.

## Acceptance Criteria
- [ ] **AC-1**: Given a successful team creation via `POST /admin/teams/confirm`, when the user is redirected to `/admin/teams`, then the success flash includes the team name AND a next-step hint referencing `bb data sync` styled with `<code>` tags.
- [ ] **AC-2**: The flash message is dismissible via page navigation (disappears on next load) -- no JS required.
- [ ] **AC-3**: No `| safe` filter is used on any query-parameter-sourced value. The `<code>` markup is rendered from the template, not from the query string, per TN-5 and `.claude/rules/jinja-safety.md`.
- [ ] **AC-4**: Exactly one success banner is rendered (no duplicate from the `msg` path). The banner uses the existing green styling (`bg-green-100 text-green-800 border-green-300`). The team name is rendered with normal Jinja2 autoescaping (HTML special characters in team names display correctly, not as raw HTML).

## Technical Approach
Two coordinated changes per TN-5. In `src/api/routes/admin.py`: change the redirect after successful team creation to pass `?added=1&team_name={name}` INSTEAD OF the current `?msg=...` parameter (do not pass both -- the existing `msg` block renders unconditionally and would produce a duplicate banner). In `src/api/templates/admin/teams.html`: add a conditional block that checks for the `added` parameter and renders the enhanced flash with the team name (autoescaped) and the `bb data sync` command hint (static `<code>` markup in the template). No `| safe` filter is used on any query-parameter value.

## Notes
- UXD specified: use `<code>` styling inline within the flash, keep it a one-liner within the existing green success flash pattern.
- CLI command: `bb data sync` is the correct subcommand for pulling season stat data.
- Security: `.claude/rules/jinja-safety.md` prohibits `| safe` on user-controlled data. Query parameters are user-visible in the URL and could be manipulated. The `added` + `team_name` approach keeps HTML in the template (trusted) and data in query params (autoescaped).
