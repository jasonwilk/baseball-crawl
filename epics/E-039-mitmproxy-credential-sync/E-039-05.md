# E-039-05: iOS vs. Web Traffic Tagging

## Epic
[E-039: mitmproxy Credential Sync and API Discovery](epic.md)

## Status
`TODO`

## Description
After this story is complete, all mitmproxy addons consistently tag GameChanger traffic as `ios`, `web`, or `unknown` based on the User-Agent header. The tagging logic lives in a shared filter module (`gc_filter.py`) that all other addons import. The credential extractor, header capture, and endpoint logger all use this shared source detection rather than implementing their own. Additionally, the credential extractor writes source-tagged credential sets so the operator can see which credentials came from iOS vs. web.

## Context
The earlier addon stories (E-039-02, 03, 04) each need to detect whether traffic comes from the iOS app or web browser. Rather than duplicating that logic, this story extracts it into a shared module and ensures consistency. It also adds source-aware credential storage -- when credentials arrive from iOS, they are tagged differently than web credentials, allowing the operator to choose which source to use for the project's HTTP client configuration.

## Acceptance Criteria
- [ ] **AC-1**: A shared module at `mitmproxy/addons/gc_filter.py` exports: (a) `is_gamechanger_domain(host: str) -> bool`, (b) `detect_source(user_agent: str) -> str` returning `"ios"`, `"web"`, or `"unknown"`.
- [ ] **AC-2**: `detect_source()` returns `"ios"` for User-Agents containing `GameChanger/`, `CFNetwork/`, or `Darwin/` (case-insensitive).
- [ ] **AC-3**: `detect_source()` returns `"web"` for User-Agents containing `Chrome/`, `Firefox/`, or `Safari/` (case-insensitive) that do NOT match any iOS pattern.
- [ ] **AC-4**: `detect_source()` returns `"unknown"` for User-Agents matching neither pattern (or empty/missing).
- [ ] **AC-5**: The credential extractor (`credential_extractor.py`), header capture (`header_capture.py`), and endpoint logger (`endpoint_logger.py`) all import and use `gc_filter.is_gamechanger_domain()` and `gc_filter.detect_source()` instead of inline implementations.
- [ ] **AC-6**: The credential extractor logs the source alongside credential updates (e.g., "Credentials updated from ios source: GAMECHANGER_AUTH_TOKEN, GAMECHANGER_DEVICE_ID").
- [ ] **AC-7**: Unit tests exist for `is_gamechanger_domain()` and `detect_source()` covering: gc.com, api.gc.com, gamechanger.com, non-GC domains, iOS User-Agents, Chrome User-Agents, Safari-on-iOS User-Agents (should be `ios`), empty User-Agent.
- [ ] **AC-8**: Existing unit tests for E-039-02, 03, 04 addons continue to pass after the refactor.

## Technical Approach
1. Create `mitmproxy/addons/gc_filter.py` with `is_gamechanger_domain()` and `detect_source()` as pure functions (no mitmproxy dependency -- they take strings, return strings).
2. Refactor `credential_extractor.py`, `header_capture.py`, and `endpoint_logger.py` to import from `gc_filter` instead of implementing domain/source checks inline.
3. Add source logging to the credential extractor.
4. Write unit tests in `tests/test_mitmproxy/test_gc_filter.py`.
5. Run all existing mitmproxy tests to verify no regressions.

## Notes
- Safari on iOS includes "Safari/" in the User-Agent but also includes "Mobile/" and platform indicators. The `detect_source()` function should check iOS patterns FIRST, then web patterns, to correctly classify Safari-on-iOS as `ios`.
- The `gc_filter.py` module has no mitmproxy imports -- it is pure Python utility functions. This makes it easy to test without mocking mitmproxy internals.
