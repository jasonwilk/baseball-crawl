## Description
After this story is complete, the operator has simple commands to start and stop the mitmproxy proxy, view the header parity report, and summarize the endpoint discovery log. A concise operations guide documents the full workflow: starting the proxy, configuring the iPhone, installing the CA certificate, and interpreting the captured data.

## Context
The mitmproxy integration is only valuable if the operator can use it without remembering Docker Compose flags or file paths. This story wraps the common operations in simple scripts and provides a guide covering the end-to-end workflow -- from "I want to refresh credentials" to "credentials are in .env and I can see which endpoints were hit."

## Acceptance Criteria
- [ ] **AC-1**: A script at `scripts/proxy.sh` supports three subcommands: `start` (runs `docker compose --profile proxy up -d`), `stop` (runs `docker compose --profile proxy down`), and `status` (shows whether the mitmproxy service is running and its port bindings).
- [ ] **AC-2**: `scripts/proxy.sh start` prints a short message with: proxy address (`<host-ip>:8080`), mitmweb UI address (`http://localhost:8081`), and a reminder to configure the iPhone to use the proxy.
- [ ] **AC-3**: A script at `scripts/proxy-report.sh` reads `data/mitmproxy/header-report.json` and prints a human-readable summary of header differences (missing, extra, changed) for each traffic source. If the report file does not exist, it prints a helpful message ("No traffic captured yet. Start the proxy and generate some traffic.").
- [ ] **AC-4**: A script at `scripts/proxy-endpoints.sh` reads `data/mitmproxy/endpoint-log.jsonl`, deduplicates by (method, path), and prints a sorted summary table of unique endpoints with their most recent status code and hit count. If the log file does not exist, it prints a helpful message.
- [ ] **AC-5**: A documentation file at `docs/admin/mitmproxy-guide.md` covers: (a) what mitmproxy does in this project, (b) starting and stopping the proxy, (c) configuring an iPhone to use the proxy (LAN IP, port, CA cert installation via `mitm.it`), (d) configuring a web browser to use the proxy, (e) where captured data goes (`.env`, `data/mitmproxy/`), (f) reading the header parity report, (g) reading the endpoint log, (h) troubleshooting (proxy unreachable, cert not trusted, no traffic captured).
- [ ] **AC-6**: The CLAUDE.md `Commands` section includes entries for `scripts/proxy.sh`, `scripts/proxy-report.sh`, and `scripts/proxy-endpoints.sh` with one-line descriptions.

## Technical Approach
1. Create `scripts/proxy.sh` as a bash script. Use `docker compose --profile proxy` for all operations. For the `start` subcommand, detect the host's LAN IP (e.g., via `hostname -I` or `ip route`) and print it in the startup message.
2. Create `scripts/proxy-report.sh` as a bash script that uses `jq` to format the JSON report. `jq` is already installed in the devcontainer (listed in devcontainer.json apt-packages).
3. Create `scripts/proxy-endpoints.sh` as a bash script that uses `jq` to process the JSONL log. Group by method+path, count occurrences, sort alphabetically.
4. Create `docs/admin/mitmproxy-guide.md` following the project's documentation patterns.
5. Update CLAUDE.md Commands section.
