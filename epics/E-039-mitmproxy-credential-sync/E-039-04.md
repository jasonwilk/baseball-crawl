## Description
After this story is complete, a mitmproxy addon logs every unique GameChanger API endpoint observed in proxied traffic to an append-only JSONL file. Each entry records the HTTP method, URL path, query parameter keys, request and response content-types, response status code, and timestamp. This provides a passive, always-growing record of API surface area that feeds into the project's API documentation at `docs/gamechanger-api.md`.

## Context
API endpoint discovery today is manual: the user or api-scout identifies endpoints one at a time through browser inspection. With this addon, every time the user interacts with GameChanger through the proxy, the system automatically captures which endpoints were hit. Over time, this builds a comprehensive map of the API surface -- including endpoints we have not explicitly investigated yet. The log is append-only JSONL so it can be processed, deduplicated, and summarized by downstream tools.

## Acceptance Criteria
- [ ] **AC-1**: Given the mitmproxy service is running with the endpoint logger addon loaded, when a request to a GameChanger domain flows through the proxy and a response is received, then a JSONL entry is appended to `data/mitmproxy/endpoint-log.jsonl`.
- [ ] **AC-2**: Each JSONL entry contains: `timestamp` (ISO 8601), `method` (GET/POST/etc.), `host`, `path` (URL path without query string), `query_keys` (sorted list of query parameter names, no values), `request_content_type`, `response_content_type`, `status_code`, `source` (ios/web/unknown).
- [ ] **AC-3**: Given a non-GameChanger request flows through the proxy, then no entry is logged.
- [ ] **AC-4**: Given the same endpoint (same method + path) is hit multiple times, then each occurrence is logged (append-only -- no deduplication in the log file itself). Deduplication is a downstream concern.
- [ ] **AC-5**: Query parameter **values** are NOT logged (they may contain tokens or user-specific data). Only the parameter **names** are recorded.
- [ ] **AC-6**: Request and response **bodies** are NOT logged.
- [ ] **AC-7**: The JSONL file is created automatically on the first logged request if it does not exist.
- [ ] **AC-8**: Unit tests exist for the entry formatting logic (given a flow object, produce the correct JSONL dict). Tests do NOT require a running mitmproxy instance.

## Technical Approach
1. Create `mitmproxy/addons/endpoint_logger.py` as a mitmproxy addon class.
2. Implement a `response()` hook (we want the status code, so we log after the response arrives):
   a. Filter to GameChanger domains.
   b. Extract method, host, path, query keys, content-types, status code.
   c. Detect traffic source from User-Agent.
   d. Format as a dict, serialize to JSON, append to the JSONL file.
3. Use `pathlib.Path` and append mode (`"a"`) for file writes. Ensure the parent directory exists.
4. Write unit tests in `tests/test_mitmproxy/test_endpoint_logger.py`.
