# E-086-01: Proxy Addon: gc-client-id and Response Body Capture

## Epic
[E-086: Mobile Credential Capture](epic.md)

## Status
`TODO`

## Description
After this story is complete, the mitmproxy credential extractor addon will capture `gc-client-id` from request headers and extract access + refresh tokens from POST /auth response bodies. Both captures are written to profile-scoped `.env` keys via the existing `merge_env_file()` mechanism. This provides the plumbing for `bb creds capture` (E-086-03) to find mobile credentials in `.env` after a proxy session.

## Context
The credential extractor (`proxy/addons/credential_extractor.py`) currently has only a `request()` handler that captures `gc-token` (refresh tokens only), `gc-device-id`, and `gc-app-name`. Two gaps exist:
1. `gc-client-id` is not captured (needed for mobile profile completeness -- the mobile client ID `0f18f027-...` differs from web).
2. POST /auth response bodies contain fresh access and refresh tokens, but the addon has no `response()` handler.

The mobile client ID is already known (`0f18f027-c51e-4122-a330-9d537beb83e0`) from prior research (E-075-R-01), but capturing it from traffic ensures it stays correct if GameChanger changes it. More importantly, the response body capture gives us the access token (~12 hours for mobile) that is the primary credential for mobile API access.

This story supersedes E-075-02 (which had the same scope but within the now-stale E-075 epic).

## Acceptance Criteria
- [ ] **AC-1**: Given iOS GameChanger traffic through the proxy, the `gc-client-id` request header is captured and written to `GAMECHANGER_CLIENT_ID_{PROFILE}` in `.env` (where PROFILE is determined by `detect_source()`)
- [ ] **AC-2**: Given a POST /auth response with `{type: "token", access: {data: "..."}, refresh: {data: "..."}}`, the response body tokens are extracted and written to `GAMECHANGER_ACCESS_TOKEN_{PROFILE}` and `GAMECHANGER_REFRESH_TOKEN_{PROFILE}` in `.env`
- [ ] **AC-3**: Given a POST /auth response with a different shape (e.g., `{type: "client-token", ...}` from step 2 of login flow), the response is ignored (only `type: "token"` responses contain access + refresh tokens)
- [ ] **AC-4**: The `response()` handler only processes POST /auth responses on GameChanger domains -- all other responses are ignored
- [ ] **AC-5**: Deduplication works for the new captures: if `gc-client-id` or response tokens have not changed since the last write, no `.env` write occurs
- [ ] **AC-6**: The access token from response bodies is written to `GAMECHANGER_ACCESS_TOKEN_{PROFILE}` -- this is a new env key (not the same as `GAMECHANGER_REFRESH_TOKEN_{PROFILE}`)
- [ ] **AC-7**: No credential values (tokens, IDs) appear in log output -- only key names and status messages
- [ ] **AC-8**: All new code has test coverage. Tests mock mitmproxy flow objects. Existing credential extractor tests continue to pass.

## Technical Approach
Two changes to `proxy/addons/credential_extractor.py`:

1. Add `"gc-client-id": "GAMECHANGER_CLIENT_ID"` to `_BASE_CREDENTIAL_HEADERS`. This is a one-line change -- the existing `request()` handler already iterates over this dict.

2. Add a `response()` handler to the `CredentialExtractor` class. This handler fires on every response but immediately returns unless: (a) the host is a GameChanger domain, (b) the request method is POST, (c) the request path is `/auth`. It then parses the JSON response body, checks for `type == "token"`, and extracts the access and refresh tokens.

The access token env key (`GAMECHANGER_ACCESS_TOKEN_{PROFILE}`) is new -- it is not in `_BASE_CREDENTIAL_HEADERS` because it comes from response bodies, not request headers.

## Notes
- The `request()` handler currently rejects access tokens in `gc-token` (only saves refresh tokens). This is correct for request-header capture -- the `gc-token` header on standard API requests carries the access token, which we don't want to overwrite the refresh token. The new `response()` handler captures access tokens from a DIFFERENT source (response bodies), writing them to a DIFFERENT key (`GAMECHANGER_ACCESS_TOKEN_*`), so there is no conflict.
- The POST /auth response shape for token responses is: `{"type": "token", "access": {"data": "...", "expires": N}, "refresh": {"data": "...", "expires": N}}`. Step 2 (client-auth) returns `{"type": "client-token", ...}` -- the response() handler must distinguish these.
- The `detect_source()` function in `gc_filter.py` uses the request's User-Agent, which is available on the flow's request object even in the `response()` handler.
