# E-086-02: `bb creds import --profile mobile`

[E-086: Mobile Credential Capture](epic.md)

## Status

## Description
After this story is complete, `bb creds import` will accept a `--profile` flag to import credentials from a curl command as either web or mobile profile. For mobile, access tokens are saved (not rejected) to `GAMECHANGER_ACCESS_TOKEN_MOBILE`, and all credential keys use the `_MOBILE` suffix. This provides a manual fallback path for operators who prefer to copy a curl command from mitmweb rather than using the automated `bb creds capture`.

## Context
`parse_curl()` in `credential_parser.py` currently produces only `_WEB` suffixed keys. The `bb creds import` CLI command has no `--profile` flag. For the mobile profile, the rules are different:
- Access tokens (`type == "user"`) are valuable and should be saved to `GAMECHANGER_ACCESS_TOKEN_MOBILE` (mobile access tokens last ~12 hours)
- Refresh tokens (no `type` field) are saved to `GAMECHANGER_REFRESH_TOKEN_MOBILE` as usual
- The `gc-client-id` header maps to `GAMECHANGER_CLIENT_ID_MOBILE`

The existing `_decode_jwt_type()` function already distinguishes token types, so the logic for routing access vs. refresh tokens to different env keys is straightforward.

## Acceptance Criteria
- [ ] **AC-1**: `bb creds import --profile mobile --curl "..."` parses a mobile curl command and writes credentials with `_MOBILE` suffix to `.env`
- [ ] **AC-2**: For mobile profile, a `gc-token` header containing an access token (`type == "user"`) is saved to `GAMECHANGER_ACCESS_TOKEN_MOBILE` (not rejected as it would be for web)
- [ ] **AC-3**: For mobile profile, a `gc-token` header containing a refresh token (no `type` field) is saved to `GAMECHANGER_REFRESH_TOKEN_MOBILE`
- [ ] **AC-4**: For web profile (default), the existing behavior is unchanged -- access tokens in `gc-token` are rejected, only refresh tokens are saved to `GAMECHANGER_REFRESH_TOKEN_WEB`
- [ ] **AC-5**: `bb creds import --profile web` is equivalent to the current `bb creds import` (web is the default)
- [ ] **AC-6**: `bb creds import` with no `--profile` flag defaults to `web` (backward compatible)
- [ ] **AC-7**: The `gc-client-id` header is captured for both profiles when present in the curl command, written to `GAMECHANGER_CLIENT_ID_{PROFILE}`
- [ ] **AC-8**: After import, the output shows the token type and remaining lifetime (e.g., "Access token valid for ~N hours" or "Refresh token expires in N days")
- [ ] **AC-9**: No credential values appear in the output -- only key names and metadata
- [ ] **AC-10**: All new code has test coverage. Existing `parse_curl` and CLI tests continue to pass.

## Technical Approach
Two modules need changes:

1. `src/gamechanger/credential_parser.py`: Add a `profile` parameter to `parse_curl()`. The profile determines the env key suffix (`_WEB` or `_MOBILE`) and the token handling rules (mobile: save access tokens to a separate key; web: reject access tokens).

2. `src/cli/creds.py`: Add `--profile` option to the `import_creds` command. Pass the profile through to `parse_curl()`. After import, decode the token to show type and remaining lifetime.

The `_decode_jwt_type()` function in `credential_parser.py` already handles token type detection. The `_decode_jwt_exp()` function in `creds.py` handles expiry decoding. Both are available for the enhanced output.
