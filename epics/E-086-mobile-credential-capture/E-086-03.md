## Description
After this story is complete, `bb creds capture --profile mobile` will scan the most recent mitmproxy session for iOS GameChanger credentials, validate them against the API, write them to `.env`, and display a summary with token health. This is the "system takes over" command -- the operator starts mitmproxy, uses the iOS app, stops the proxy, and runs this one command. If something went wrong (no session, no mobile traffic, invalid tokens), the command provides clear, actionable guidance instead of a cryptic error.

## Context
After E-086-01 (addon upgrade), the proxy credential extractor will automatically write mobile credentials to `.env` during a proxy session. This story adds a CLI command that:
1. Verifies the credentials were captured (checks `.env` for mobile keys)
2. Validates them against the API (`GET /me/user`)
3. Reports token health (access token lifetime, refresh token lifetime)
4. If credentials are missing, scans proxy session data to detect whether iOS auth traffic occurred and provides actionable guidance

The command is designed for the common case: the addon already wrote the credentials during the proxy session, and this command validates and reports on them. The fallback (scanning endpoint logs) provides detection and guidance only -- the endpoint log contains metadata (`method`, `path`, `source`, `status_code`) but not credential values, so it cannot recover credentials directly.

## Technical Approach
New `capture` command in `src/cli/creds.py`. The command follows a decision tree:

1. Read `.env` for mobile credential keys
2. If credentials present: validate access token via `GET /me/user` -> report success or expiry
3. If credentials missing: scan `proxy/data/sessions/` -> find most recent session -> check endpoint log for iOS traffic -> provide guidance

For proxy session scanning, the endpoint log JSONL files contain a `source` field (from `detect_source()`) that identifies iOS vs. web traffic. The command reads the log, filters for `source == "ios"`, and checks whether any POST /auth entries exist. This is detection only -- the endpoint log does not contain credential values, so the fallback provides guidance (e.g., "iOS auth traffic detected but credentials missing from .env -- re-run the proxy session") rather than credential recovery.

The inline setup guide (AC-5) should reference the mitmproxy guide: "See docs/admin/mitmproxy-guide.md for detailed setup instructions."

## Notes
- The proxy session directory structure is: `proxy/data/sessions/{timestamp}/endpoint-log.jsonl`. The most recent session is determined by sorting directory names (they are timestamp-formatted: `YYYY-MM-DD_HHMMSS`).
- The endpoint log JSONL contains one JSON object per line with fields including `method`, `path`, `source`, `status_code`. The `source` field comes from `detect_source()` in `gc_filter.py`.
- The proxy runs on the Mac host, not in the devcontainer (see CLAUDE.md Proxy Boundary). The `capture` command only READS proxy data files from `proxy/data/` -- it does not attempt to start or stop the proxy.
- The access token for mobile lasts ~12 hours (~43,997 seconds). The command should show this in human-readable format (e.g., "~11 hours remaining").
- Consider adding `--session` flag to scan a specific session instead of the most recent. But for MVP, most-recent-session is sufficient.
