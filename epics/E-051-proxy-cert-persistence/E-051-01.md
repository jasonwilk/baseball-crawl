# E-051-01: Fix cert directory permissions in proxy entrypoint

## Epic
[E-051: Fix mitmproxy CA Certificate Persistence](epic.md)

## Status
`TODO`

## Description
After this story is complete, mitmproxy CA certificates will persist in `proxy/certs/` across container stop/start cycles. The proxy entrypoint script will ensure the cert directory has correct permissions before mitmproxy starts, so the generated CA keypair is written to the bind-mounted volume and survives container recreation.

## Context
The `proxy/docker-compose.yml` bind-mounts `./certs:/home/mitmproxy/.mitmproxy`, but the host-created directory may not be writable by the container's `mitmproxy` user due to UID mismatch. This causes mitmproxy to either fail to write certs to the mount or regenerate them each time. The fix must ensure the directory is writable before `mitmweb` starts.

## Acceptance Criteria
- [ ] **AC-1**: Given `proxy/certs/` is empty, when `./start.sh` is run, then mitmproxy generates CA cert files (`mitmproxy-ca-cert.pem`, `mitmproxy-ca.pem`, etc.) into `proxy/certs/` on the host
- [ ] **AC-2**: Given CA certs exist in `proxy/certs/`, when `./stop.sh && ./start.sh` is run, then the CA cert files in `proxy/certs/` are unchanged (same content, same fingerprint)
- [ ] **AC-3**: Given the container is started, when inspecting the mitmproxy log output (`./logs.sh`), then there are no permission-denied errors related to `/home/mitmproxy/.mitmproxy`
- [ ] **AC-4**: The fix works with the stock `mitmproxy/mitmproxy` Docker image (no custom Dockerfile added)
- [ ] **AC-5**: The `proxy/README.md` "Install CA certificate" section still accurately describes the workflow (no doc changes needed unless the fix changes the user-facing flow)

## Technical Approach
1. **Modify `proxy/proxy-entrypoint.sh`** to fix permissions on `/home/mitmproxy/.mitmproxy` before running `mitmweb`. The simplest approach is `chmod a+rwx /home/mitmproxy/.mitmproxy` at the top of the script, but this requires the entrypoint user to own the directory or have sufficient privileges.

2. **If the default container user cannot chmod**: modify `proxy/docker-compose.yml` to add `user: "0:0"` (run as root), then have the entrypoint fix permissions and use `exec su-exec mitmproxy mitmweb ...` (or `exec gosu mitmproxy mitmweb ...`) to drop privileges. Check which privilege-dropping tool is available in the stock image first. An alternative: use `exec chown` + `exec setpriv --reuid` or just run as root (acceptable for a local dev proxy tool).

3. **Update `proxy/start.sh`** if the `mkdir -p certs` line needs adjusted permissions (e.g., `mkdir -p -m 777 certs`).

4. **Verify** by checking that `proxy/certs/mitmproxy-ca-cert.pem` exists after first start, and has the same SHA256 fingerprint after a stop/start cycle.
