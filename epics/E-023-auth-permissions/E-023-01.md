## Description
After this story is complete, the SQLite database has four new tables -- `users`, `magic_link_tokens`, `passkey_credentials`, and `sessions` -- plus the `user_team_access` join table. These tables support the full auth lifecycle: magic link login, passkey registration, session management, and team-scoped access control. A migration file (`003_auth.sql`) applies cleanly via the existing `apply_migrations.py` tooling.

## Context
This is the data foundation for the entire auth system. Every subsequent story (magic link flow, passkeys, dashboard scoping, admin page) depends on these tables existing. The migration must be a single file that creates all auth-related tables atomically.

Migration 002 is reserved for the stats schema (E-003-02). This migration uses number 003.

## Technical Approach
Create `migrations/003_auth.sql` following the existing migration conventions established in `001_initial_schema.sql`:
- Same header comment style
- Same naming conventions (`_id` suffix for PKs, `TEXT NOT NULL DEFAULT (datetime('now'))` for timestamps)
- Same `INTEGER` for booleans pattern
- Use `CREATE TABLE IF NOT EXISTS` and `CREATE INDEX IF NOT EXISTS` for idempotency

The migration file contains only DDL (CREATE TABLE, CREATE INDEX). No seed data in the migration itself -- seeding is handled by the dev bypass in E-023-02 and the admin page in E-023-05.

## Notes
- The `email` column uses TEXT with a UNIQUE constraint. Emails are case-insensitive in practice; the middleware (E-023-02) normalizes to lowercase before querying. The schema does not enforce case at the SQL level.
- `credential_id` and `public_key` in `passkey_credentials` are BLOB because `py_webauthn` works with bytes for these values. Storing as BLOB avoids base64 encode/decode overhead.
- No seed data in the migration. The dev auto-create feature in E-023-02 handles local dev. Production seeding is a manual step by Jason via the admin page (E-023-05).
- Migration 002 (`002_stats_schema.sql`) is claimed by E-003-02 but may not exist yet when 003 is applied. The migration runner should handle this gracefully (apply migrations in numeric order for whatever files are present).
