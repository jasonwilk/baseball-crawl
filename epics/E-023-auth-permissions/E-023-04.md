# E-023-04: Team-Scoped Dashboard

## Epic
[E-023: Authentication and Team-Level Permissions](epic.md)

## Status
`TODO`

## Description
After this story is complete, the `/dashboard` route filters data based on the authenticated user's permitted teams. If a user has access to exactly one team, the dashboard shows that team's stats directly. If the user has access to multiple teams, the dashboard includes a team selector that lets them switch between teams. The route rejects requests for teams the user is not authorized to view. The dashboard header shows the logged-in user's name and a logout link.

## Context
The current `/dashboard` route is hardcoded to `lsb-varsity-2026`. This story makes it dynamic, driven by the user's permissions from the session middleware (E-023-02). The middleware has already attached `request.state.permitted_teams` (list of team_id strings) and `request.state.user` to the request by the time the route handler runs.

This is the story that delivers the core user-visible value: a coach opens the dashboard and sees exactly the teams they are assigned to, nothing more, nothing less.

## Acceptance Criteria
- [ ] **AC-1**: `GET /dashboard` requires an active session (the session middleware from E-023-02 handles redirection to `/auth/login` for unauthenticated users). Once authenticated, it renders stats for the user's first permitted team by default (when no `team_id` query parameter is provided).
- [ ] **AC-2**: `GET /dashboard?team_id=<id>` renders stats for the specified team, but only if `team_id` is in `request.state.permitted_teams`. If not permitted, return a 403 response.
- [ ] **AC-3**: When the user has access to multiple teams, the dashboard page includes a team selector (dropdown, tabs, or links) showing the display names of all permitted teams. Clicking a team navigates to `/dashboard?team_id=<id>`.
- [ ] **AC-4**: When the user has access to exactly one team, the team selector is not displayed (no unnecessary UI clutter).
- [ ] **AC-5**: The dashboard page header displays the authenticated user's display name (e.g., "Logged in as Coach Smith") and a "Logout" link pointing to `/auth/logout`.
- [ ] **AC-6**: The hardcoded `lsb-varsity-2026` team_id and `2026` season in the existing route are replaced with the dynamic team_id from the query parameter or default permitted team. The season should still default to the current year.
- [ ] **AC-7**: If the user has no permitted teams (edge case: user exists but has no team assignments), the dashboard shows a message: "You have no team assignments. Contact your administrator."
- [ ] **AC-8**: Tests verify that a user with access to team A cannot view team B's dashboard by passing `?team_id=B`.
- [ ] **AC-9**: Tests verify that a user with multiple teams sees the team selector, and a user with one team does not.
- [ ] **AC-10**: Tests verify the logout link is present in the rendered HTML.

## Technical Approach
Modify the existing `src/api/routes/dashboard.py` to:

1. Read `request.state.permitted_teams` and `request.state.user` (set by session middleware).
2. Determine the active team: use the `team_id` query parameter if provided and permitted, otherwise default to the first permitted team.
3. If the requested `team_id` is not in `permitted_teams`, return 403.
4. If `permitted_teams` is empty, render the "no assignments" message.
5. Fetch batting stats using the existing `db.get_team_batting_stats()` function with the dynamic team_id.
6. Pass the team selector data (list of `{team_id, name}` dicts for permitted teams) to the template.

For the team display names, query the `teams` table to get the `name` for each permitted team_id.

Modify `src/api/templates/dashboard/team_stats.html` to:
1. Add a user greeting and logout link in the page header.
2. Add a team selector (simple `<select>` with `onchange` navigation, or a row of links/tabs) when multiple teams are available.
3. Highlight the currently selected team.

## Notes
- The season is not parameterized in this story. The route continues to default to the current year. Season selection can be added in a future story.
- The `db.get_team_batting_stats()` function already accepts `team_id` and `season` parameters. This story just makes the route pass dynamic values instead of hardcoded ones.
- The team selector should be simple HTML. No JavaScript framework needed. A `<select>` element with an `onchange="window.location=..."` is sufficient, or a list of `<a>` tags styled as tabs.
- This story runs in parallel with E-023-03 (passkeys). No file conflicts: this story touches dashboard routes/templates, E-023-03 touches auth routes/templates. (The passkey registration CTA uses a dedicated interstitial in the auth template directory, not the dashboard template.)
- **Existing test compatibility**: `tests/test_dashboard.py` should already be updated by E-023-02 to work with auth middleware. If E-023-04 changes the dashboard response structure (team selector, user greeting), the new `tests/test_dashboard_auth.py` should cover those behaviors. If any existing tests in `test_dashboard.py` need further updates (e.g., asserting on changed HTML structure), include those changes too.
