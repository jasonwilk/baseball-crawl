# E-147-01: Migration, Backfill, and Year-Map Rewrite

## Epic
[E-147: Team Season Year and Cohort-Based Dashboard Navigation](epic.md)

## Status
`TODO`

## Description
After this story is complete, the `teams` table has a `season_year` column, team 8 is set to 2025 and team 78 to 2026 (all other existing teams start NULL for self-healing), and `get_team_year_map()` reads from this column instead of performing an expensive UNION+JOIN on stat tables.

## Context
This is the foundation story. Every other story in the epic depends on `season_year` existing in the schema. The current `get_team_year_map()` derives year from stat data, which is both slow and incorrect when multiple teams share a season record. The migration includes explicit corrections for the two known-wrong teams; all other teams start NULL and are filled by the self-healing pipeline (E-147-03) on the next sync.

## Acceptance Criteria
- [ ] **AC-1**: Migration `004_add_team_season_year.sql` exists in `migrations/` and adds `season_year INTEGER` to the `teams` table. The migration includes explicit UPDATE statements for the known corrections per TN-2 in the epic Technical Notes. The migration does NOT use stat-derived year logic.
- [ ] **AC-2**: After migration runs, `SELECT id, season_year FROM teams WHERE id IN (8, 78)` returns `(8, 2025)` and `(78, 2026)`. All other existing teams have `season_year = NULL` (to be filled by self-healing pipeline).
- [ ] **AC-3**: `get_team_year_map()` in `src/api/db.py` reads from `teams.season_year` instead of the stat-table UNION. NULL values fall back to current calendar year. Function signature is unchanged. Per TN-3 in the epic Technical Notes.
- [ ] **AC-4**: Existing tests pass. New tests cover: (a) `get_team_year_map` returns `season_year` when set, (b) `get_team_year_map` falls back to current year when `season_year` is NULL, (c) empty input returns empty dict.

## Technical Approach
The migration adds the column and includes explicit UPDATE statements for the two known-wrong teams (8 and 78). It does NOT use stat-derived year logic because that logic is the root cause of the bug. All other teams start NULL -- the self-healing pipeline (E-147-03) fills them from the API on the next sync cycle.

The `get_team_year_map()` rewrite replaces the UNION+JOIN query with a simple `SELECT id, season_year FROM teams WHERE id IN (...)`. The NULL fallback logic remains.
