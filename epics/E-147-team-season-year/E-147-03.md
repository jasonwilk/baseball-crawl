# E-147-03: Pipeline Self-Healing: Propagate season_year on Sync

## Epic
[E-147: Team Season Year and Cohort-Based Dashboard Navigation](epic.md)

## Status
`TODO`

## Description
After this story is complete, the member crawl and scouting pipeline UPDATE `teams.season_year` when they encounter the value from the API, filling NULLs left by non-admin INSERT paths. A warning guard logs mismatches between `teams.season_year` and loaded data year without blocking the load.

## Context
Six of the seven team INSERT paths (TN-5, items 2-7) insert with `season_year = NULL` because the year value isn't readily available at INSERT time. This story adds UPDATE-on-sync logic so that subsequent pipeline runs fill the gap. The member crawl (`season_year` from authenticated endpoints) and scouting pipeline (`team_season.year` from public endpoints) are the two self-healing paths. Per TN-5 and TN-7 in the epic Technical Notes.

## Acceptance Criteria
- [ ] **AC-1**: The member sync pipeline (`src/pipeline/trigger.py:run_member_sync()`) adds a call to `GET /teams/{team_id}` (authenticated) to fetch `season_year`, then conditionally UPDATEs `teams.season_year` WHERE `season_year IS NULL`. No existing code path fetches this value -- this is a new API call.
- [ ] **AC-2**: The scouting pipeline adds a call to `resolve_team(public_id)` (from `src.gamechanger.team_resolver`) in its per-team flow to fetch the team profile and extract `profile.year`, then conditionally UPDATEs `teams.season_year` WHERE `season_year IS NULL`. The scouting crawler does not currently call `resolve_team()` -- this is a new call.
- [ ] **AC-3**: When a loader processes data for a team where `teams.season_year` is set and the loaded data's year (from season_id or game date) doesn't match, a WARNING-level log message is emitted. The load is NOT blocked. Per TN-7 in the epic Technical Notes.
- [ ] **AC-4**: Tests verify: (a) NULL `season_year` is updated on sync, (b) non-NULL `season_year` is not overwritten, (c) mismatch warning is logged.

## Technical Approach
For the member crawl: no existing code path in the member pipeline fetches `season_year`. Add a new `GET /teams/{team_id}` call in `src/pipeline/trigger.py:run_member_sync()` (the background pipeline entry point for member teams). Extract `season_year` from the response. Conditionally UPDATE: `UPDATE teams SET season_year = ? WHERE id = ? AND season_year IS NULL`. This is a new API call, not a piggyback on an existing one.

For the scouting pipeline: the scouting crawler (`src/gamechanger/crawlers/scouting.py`) does not currently call `resolve_team()`. Add a call to `resolve_team(public_id)` (from `src.gamechanger.team_resolver`) in the per-team scouting flow to fetch the public team profile. Extract `profile.year`. Conditionally UPDATE using the same pattern as the member crawl.

For the warning guard: in loader entry points (game loader, season stats loader, scouting loader, roster loader), compare the loaded data's year against `teams.season_year` and log if they differ.

## Notes
- The "don't overwrite" pattern (UPDATE WHERE `season_year IS NULL`) is intentional. If a team was manually corrected (e.g., team 8 = 2025), the pipeline should not revert it.
- **GC API identifier**: The `GET /teams/{team_id}` endpoint (AC-1) takes the `gc_uuid` as its path parameter, NOT our integer `teams.id`. `run_member_sync()` receives only the integer PK — the implementer must look up `gc_uuid` from the `teams` table before making the API call. Member teams should always have `gc_uuid` set, but if `gc_uuid IS NULL` (e.g., bridge returned 403), skip the API call gracefully rather than failing.
- **Skip API call when unnecessary**: For both AC-1 and AC-2, check `teams.season_year IS NULL` before making the API call. If `season_year` is already populated, skip the call entirely — don't just skip the UPDATE. This avoids unnecessary API calls on every sync run after the first successful fill.
- Warning guards are observability only. A mismatch could be legitimate (e.g., a team loaded into the wrong season during data migration). The operator investigates warnings; the system does not enforce.
