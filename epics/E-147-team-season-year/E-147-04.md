# E-147-04: Cohort-Based Dashboard Navigation

## Epic
[E-147: Team Season Year and Cohort-Based Dashboard Navigation](epic.md)

## Status
`TODO`

## Description
After this story is complete, the dashboard navigation groups teams by year-cohort. The default view shows only the current year's teams. A year dropdown lets coaches view historical cohorts. When viewing a past year, a "← Current season" back-link appears. Two identically-named teams from different years never appear simultaneously.

## Context
The UX designer specified a cohort selector pattern: each year shows that year's ~3-4 teams as pills. The `_team_selector.html` macro already handles `available_years` and renders a year dropdown. The route-level filtering at line ~369 of `dashboard.py` already filters `permitted_team_infos` by year -- but it uses the stat-derived year map. After E-147-01, `get_team_year_map()` reads from the column, so the filtering logic is already structurally correct. The main changes are: deriving `available_years` from the column, adding `current_year` for the "(current)" label and back-link, and applying the cohort pattern consistently across all dashboard routes. Per TN-6 in the epic Technical Notes.

## Acceptance Criteria
- [ ] **AC-1**: `/dashboard` with no `?year=` param shows only teams where `teams.season_year` matches the current year (max `season_year` among permitted teams, with NULL values falling back to current calendar year per TN-3). Teams whose `season_year` is NULL participate in the current-year cohort. Team pills do not include teams from other years.
- [ ] **AC-2**: `/dashboard?year=2025` shows only teams with `season_year = 2025`. The year dropdown reflects the selected year. If `?year=` specifies a year with no matching teams (e.g., stale bookmark), fall back silently to `current_year` rather than rendering an empty page.
- [ ] **AC-3**: When viewing a historical year (not the current/max year), a "← Current season" link is visible that navigates back to the current year's view.
- [ ] **AC-4**: The year dropdown shows "(current)" next to the most recent year value.
- [ ] **AC-5**: The cohort filtering behavior is consistent across all dashboard routes: batting (`/dashboard`), pitching (`/dashboard/pitching`), games (`/dashboard/games`), and opponents (`/dashboard/opponents`).
- [ ] **AC-6**: When only one year of data exists, the year dropdown is hidden and a static year `<span>` is shown (existing behavior preserved). No "(current)" label is needed in this case -- the absence of the dropdown already implies it's the only/current year.
- [ ] **AC-7**: Tests verify: (a) cohort filtering returns only same-year teams, (b) `current_year` template variable is set correctly, (c) historical year shows back-link.

## Technical Approach
The route-level year filtering already exists in `dashboard.py` (~line 369: `year_team_infos = [t for t in team_infos if team_year_map.get(t["id"]) == active_year]`). After E-147-01, this works correctly because `get_team_year_map()` returns column-based values.

The changes:
1. Add `current_year` to template context (computed as `max(team_year_map.values())` or fallback to `datetime.date.today().year`).
2. Update `_team_selector.html` macro to accept and display `current_year`: show "(current)" next to the max year in the dropdown, and render a "← Current season" link when `active_year < current_year`.
3. Ensure all dashboard route handlers pass `current_year` to their templates.
4. Verify the year-filtering pattern is applied consistently in all route handlers (batting, pitching, games, opponents).

## Notes
- The `_team_selector.html` macro signature gains a `current_year` parameter. Existing call sites that don't pass it get `None` (backward compatible via Jinja2 default).
- The "← Current season" link is a simple anchor, not a component -- keep it minimal.
- **NULL-fallback behavior**: Teams with `season_year = NULL` map to the current calendar year via `get_team_year_map()` (E-147-01 AC-3). This means they appear in the current-year cohort alongside teams that genuinely have `season_year` set. This is correct: until the self-healing pipeline (E-147-03) fills their `season_year`, showing them in the current view is the safest default. No special handling needed -- the fallback in `get_team_year_map()` handles it transparently.
