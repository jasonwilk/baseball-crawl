# E-098-01: Fix Cross-Season Aggregate Bleed + Regression Test

## Epic
[E-098: Fix Scouting Pipeline Bugs](epic.md)

## Status
`DONE`

## Description
After this story is complete, the season aggregate computation in `ScoutingLoader` will produce correct per-season stats even when the same opponent has been scouted across multiple seasons. Currently, `_compute_batting_aggregates` and `_compute_pitching_aggregates` filter only on `team_id`, causing all prior-year game rows to be summed into the current season's stats. A new regression test will verify that multi-season scouting produces isolated per-season aggregates.

## Context
The `player_game_batting` and `player_game_pitching` tables have no `season_id` column, so filtering by `season_id` requires a JOIN through the `games` table (which does have `season_id`). Both aggregate methods already receive `season_id` as a parameter but do not use it in the query. This is a data-correctness bug that produces wrong scouting stats when an opponent is scouted in consecutive seasons.

## Acceptance Criteria
- [ ] **AC-1**: Given an opponent scouted in two different seasons (e.g., "2025-spring" and "2026-spring") with different game data in each, when `_compute_batting_aggregates` runs for season "2026-spring", then the resulting `player_season_batting` rows contain only stats from games in the "2026-spring" season (no data from "2025-spring" games).
- [ ] **AC-2**: Same as AC-1 but for `_compute_pitching_aggregates` and `player_season_pitching`.
- [ ] **AC-3**: Both aggregate queries JOIN through the `games` table to filter on `season_id`, since `player_game_batting`/`player_game_pitching` have no `season_id` column.
- [ ] **AC-4**: A new test in `tests/test_scouting_loader.py` sets up game rows for the same team across two seasons, runs aggregation for one season, and asserts that only that season's stats appear in the season aggregate table. The test covers both batting and pitching.
- [ ] **AC-5**: All existing scouting loader tests continue to pass.

## Technical Approach
Both `_compute_batting_aggregates` (line 322) and `_compute_pitching_aggregates` (line 367) in `/workspaces/baseball-crawl/src/gamechanger/loaders/scouting_loader.py` need their SELECT queries updated to JOIN `player_game_batting`/`player_game_pitching` through the `games` table and add a `season_id` filter. The `season_id` parameter is already passed to both methods but unused in the query. The schema is at `/workspaces/baseball-crawl/migrations/001_initial_schema.sql`. Existing tests are at `/workspaces/baseball-crawl/tests/test_scouting_loader.py`.
