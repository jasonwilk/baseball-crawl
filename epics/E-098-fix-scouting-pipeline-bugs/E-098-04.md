# E-098-04: Fix completed_at Timestamp on Intermediate Status Rows

## Epic
[E-098: Fix Scouting Pipeline Bugs](epic.md)

## Status
`DONE`

## Description
After this story is complete, `_upsert_run_end()` will set `completed_at` to `NULL` when writing a non-terminal status ('running'), and to the current timestamp only when writing a terminal status ('completed' or 'failed'). This corrects a metadata contract violation where 'running' rows have a `completed_at` timestamp.

## Context
E-098-03 changed the crawler to write 'running' instead of 'completed' as the intermediate status after the crawl phase. However, `_upsert_run_end()` unconditionally sets `completed_at = ?` with a timestamp regardless of the status value. A 'running' row should have `completed_at IS NULL` because the run hasn't completed yet. The timestamp is only meaningful for terminal states ('completed' or 'failed'). This is a small metadata correctness fix discovered during the final Codex code review.

## Acceptance Criteria
- [ ] **AC-1**: `_upsert_run_end()` sets `completed_at = NULL` when `status` is 'running', and `completed_at` to the current UTC timestamp when `status` is 'completed' or 'failed'.
- [ ] **AC-2**: A test verifies that after a successful crawl (status='running'), the `scouting_runs` row has `completed_at IS NULL`.
- [ ] **AC-3**: Existing tests for 'completed' and 'failed' status rows continue to have a non-null `completed_at` value.
- [ ] **AC-4**: All existing scouting tests continue to pass.

## Technical Approach
Read `src/gamechanger/crawlers/scouting.py`, specifically `_upsert_run_end()`. The `completed_at` value at line 521 is currently set unconditionally to `now`. Change it to be conditional on the status parameter: `None` for 'running', `now` for terminal states.
