# E-098-03: Fix Crawler Status Lifecycle and Freshness Gate

## Epic
[E-098: Fix Scouting Pipeline Bugs](epic.md)

## Status
`DONE`

## Description
After this story is complete, the scouting crawler's `scouting_runs` status lifecycle will be correct end-to-end: the crawler writes an intermediate status ('running') instead of 'completed', the CLI owns the final 'completed'/'failed' transition after load completes, the freshness gate filters by season when an explicit `--season` is provided, and crawls that fetch zero boxscores are marked 'failed' instead of 'completed'.

## Context
A Codex code review of the E-097+E-098 changeset found three additional bugs in the scouting crawler's status and freshness logic. E-098-02 fixed the CLI side (ignored LoadResult, exit code), but the crawler side still has issues:

1. **Freshness gate missing season_id** (`scouting.py:406-428`): `_is_scouted_recently()` queries only `WHERE team_id = ?`. If the same opponent is scouted for a different season, the freshness gate incorrectly skips it. The complication: `scout_all()` calls the freshness check BEFORE `scout_team()` derives the season, so full season filtering is only possible when `--season` is explicit.
2. **Premature 'completed' status** (`scouting.py:160-163`): `scout_team()` calls `_upsert_run_end(... "completed" ...)` at the end of the crawl phase. Loading happens afterward in the CLI. If loading fails, the run is permanently marked 'completed' and future freshness checks skip it -- locking the team into an un-loaded state.
3. **Zero-boxscore crawl marked 'completed'** (`scouting.py:158-165`): When `_fetch_boxscores()` returns 0 (all boxscore fetches failed), the run is still marked 'completed'. The operator sees success, but no data was actually crawled. Future runs skip the team due to freshness.

All three confirmed FIX by SE and DE. Single story because all three touch the same state machine in the same files.

## Acceptance Criteria
- [ ] **AC-1**: `scout_team()` writes 'running' (not 'completed') as the intermediate status in `_upsert_run_end()` at the end of the crawl phase. The 'completed' status is never written by the crawler.
- [ ] **AC-2**: A new public method `update_run_load_status(team_id, season_id, status, error_message=None)` exists on `ScoutingCrawler` (or a standalone function accessible to the CLI). The CLI calls this after the load phase to set the final status ('completed' on success, 'failed' on load errors).
- [ ] **AC-3**: `_is_scouted_recently()` accepts an optional `season_id` parameter. When provided, the query filters on `WHERE team_id = ? AND season_id = ?`. When `None`, the query filters on `team_id` only (current behavior, acceptable for auto-derive path).
- [ ] **AC-4**: `scout_all()` passes `season_id` to `_is_scouted_recently()` when an explicit `season_id` was provided to `scout_all()`. When `season_id` is `None` (auto-derive), it passes `None` (team-only freshness, current behavior).
- [ ] **AC-5**: When `games_crawled == 0` and `len(completed_games) > 0` (total boxscore fetch failure), `scout_team()` marks the run as 'failed' with an appropriate error message, and returns a `CrawlResult` with `errors >= 1`.
- [ ] **AC-6**: The CLI's load-phase code (in `data.py`) calls the new status update method after loading, setting 'completed' or 'failed' based on the `LoadResult`.
- [ ] **AC-7**: Tests exist for: (a) freshness gate with explicit season_id filters correctly (scouted for season A does not skip season B), (b) freshness gate with `season_id=None` uses team-only check, (c) crawler writes 'running' not 'completed' after crawl phase, (d) zero-boxscore crawl returns errors and marks 'failed', (e) CLI sets final 'completed' after successful load, (f) CLI sets 'failed' after load errors.
- [ ] **AC-8**: All existing scouting tests continue to pass.

## Technical Approach
The three fixes are tightly coupled around the `scouting_runs` state machine. The crawler currently owns the full lifecycle (start -> completed/failed). After this fix, the lifecycle is split: crawler owns start -> running/failed (crawl-phase failure), CLI owns running -> completed/failed (load-phase outcome). Read `src/gamechanger/crawlers/scouting.py` and `src/cli/data.py` for the current code. The `_is_scouted_recently` query at `scouting.py:415-420` needs the optional `season_id` filter. The `_upsert_run_end` call at `scouting.py:160-163` needs to write 'running' instead of 'completed'. The zero-boxscore check goes between `_fetch_boxscores` (line 158) and `_upsert_run_end` (line 160). The CLI integration point is wherever E-098-02 already wired up LoadResult checking in `data.py`.

## Notes
- No schema migration needed. The `scouting_runs.status` column is TEXT -- 'running' is a new value but requires no DDL change.
- The split lifecycle (crawler writes 'running', CLI writes final status) means the CLI must handle the case where it crashes between crawl and load -- the run stays 'running' permanently. This is acceptable: a 'running' status does not pass the freshness gate (which checks `status = 'completed'`), so the team will be re-scouted on the next run.
- E-098-02 already made the CLI check LoadResult and propagate errors. This story extends that by wiring the load outcome back into `scouting_runs.status`.
