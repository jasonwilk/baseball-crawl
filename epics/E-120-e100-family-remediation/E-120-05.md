## Epic
[E-120: E-100 Family Code Review Remediation](epic.md)

## Status

## Description
After this story is complete, admin and dashboard auth test files will use the real migration-based schema (catching constraint violations), a misleadingly-named test file will have a correct name, and a vacuously true FK test will actually verify its claim.

## Context
Four test infrastructure issues: (1) `test_admin_teams.py` and `test_admin_opponents.py` define inline `_SCHEMA_SQL` strings that are incomplete subsets of the real schema — missing CHECK constraints and partial unique indexes. Tests can pass with data the real schema would reject. Other test files (e.g., `test_e100_schema.py`) already use `run_migrations()`. (2) `test_dashboard_auth.py` also uses an inline `_SCHEMA_SQL` with a `_migrations` table structure but no seed rows — if any code checks migration state, it will behave incorrectly. Switching to `run_migrations()` solves both the incomplete schema and missing seed data. (3) `test_migration_003.py` tests 001 auth tables but is named after old migration 003 (which no longer exists after the E-100 schema consolidation). (4) `test_e100_schema.py:160` `test_foreign_keys_enabled` is vacuously true because the fixture already sets `PRAGMA foreign_keys = ON` — the test verifies the fixture, not the application behavior.

## Acceptance Criteria
- [ ] **AC-1**: `tests/test_admin_teams.py` uses `run_migrations()` (from `migrations.apply_migrations`) instead of the inline `_SCHEMA_SQL` at line ~57. The inline `_SCHEMA_SQL` string is removed. All existing tests in the file still pass with the real schema. Test INSERT statements are updated as needed to satisfy CHECK constraints the inline schema was missing.
- [ ] **AC-2**: `tests/test_admin_opponents.py` uses `run_migrations()` instead of the inline `_SCHEMA_SQL` at line ~48. Same requirements as AC-1.
- [ ] **AC-2a**: `tests/test_dashboard_auth.py` uses `run_migrations()` instead of the inline `_SCHEMA_SQL` at line ~46. The inline schema string is removed. All existing tests in the file still pass with the real schema.
- [ ] **AC-3**: `tests/test_migration_003.py` is renamed to accurately reflect its content (it tests 001 auth tables — the header at line 2 says so). The new name clearly references 001 or auth (e.g., `test_migration_001_auth.py`). All tests pass under the new filename. The old filename no longer exists. The internal `pytest` run-with line (line 16) is updated.
- [ ] **AC-4**: `tests/test_e100_schema.py:160` `test_foreign_keys_enabled` is restructured so it is not vacuously true. Currently the `migrated_db` fixture (line 54-62) sets `PRAGMA foreign_keys=ON` at line 58, and the test merely reads that pragma back. The fix: either (a) the test attempts an INSERT that violates a FK constraint and asserts `sqlite3.IntegrityError` is raised, or (b) the test creates its own fresh connection without the pragma and verifies migration behavior.
- [ ] **AC-5**: The dead `_patch_all` helper in `tests/test_bootstrap.py` (line ~29) is removed. It is defined but never called by any test function in the file.
- [ ] **AC-6**: All existing tests pass

## Technical Approach
1. For AC-1/AC-2/AC-2a: Follow the pattern in `tests/test_e100_schema.py:54-62` — create a `tmp_path` fixture, call `run_migrations(db_path=...)`, connect. The `_SCHEMA_SQL` inline strings in both files are incomplete subsets of `001_initial_schema.sql` (missing CHECK constraints on `membership_type`, `classification`, and partial unique indexes). Test data INSERTs that use invalid values (e.g., missing required columns, invalid enum values) will need to be updated.
2. For AC-3: `git mv tests/test_migration_003.py tests/test_migration_001_auth.py`. Grep the repo for references to the old filename.
3. For AC-4: The simplest approach is option (a): add an INSERT into a child table with a nonexistent parent FK value, wrapped in `pytest.raises(sqlite3.IntegrityError)`. This proves FK enforcement is active via observed behavior, not just pragma state.
