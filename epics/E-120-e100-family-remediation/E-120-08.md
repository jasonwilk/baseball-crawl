# E-120-08: Admin team insert IntegrityError handling

## Status

## Description
After this story is complete, a concurrent team insert that violates a UNIQUE constraint will return a user-friendly error response instead of an unhandled 500.

## Context
The `confirm_team_submit` handler at `admin.py:1138` performs a check-then-insert pattern: it checks for duplicates, then calls `_insert_team_new` in a separate transaction. A concurrent request between the check and the insert can produce an `sqlite3.IntegrityError` on the UNIQUE constraint (`public_id`), which is currently unhandled and surfaces as a raw 500 error. The fix is to catch `IntegrityError` on the insert call and return the user to the confirm page with an error message.

## Acceptance Criteria
- [ ] **AC-1**: The `_insert_team_new` call (or its surrounding code in `confirm_team_submit`) catches `sqlite3.IntegrityError` and returns a user-facing error response (redirect with error message or re-rendered confirm page) instead of propagating as a 500
- [ ] **AC-2**: A test verifies that when `_insert_team_new` raises `IntegrityError`, the response is not a 500 (it should be a redirect with an error message or a 400-level response)
- [ ] **AC-3**: All existing tests pass

## Technical Approach
Wrap the `_insert_team_new` call in `confirm_team_submit` with a try/except for `sqlite3.IntegrityError`. On catch, redirect back to the team list with an error message indicating the team already exists (the most likely UNIQUE violation). This is the simplest fix -- it doesn't change the TOCTOU window, just handles the race gracefully.

## Notes
This does not eliminate the TOCTOU race -- it makes the race harmless by handling the resulting error gracefully. A same-transaction check+insert would eliminate the race entirely but would require restructuring the handler, which is disproportionate to the risk (single-user admin UI).
