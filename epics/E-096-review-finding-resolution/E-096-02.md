# E-096-02: Update Dispatch Review Loop in dispatch-pattern and implement skill

## Epic
[E-096: Resolve All Review Findings During Stories](./epic.md)

## Status
`TODO`

## Description
After this story is complete, the dispatch review loop in `dispatch-pattern.md` and `implement/SKILL.md` will implement the resolution-first model: when the code-reviewer returns findings, the main session triages ALL findings (not just MUST FIX) and routes accepted SHOULD FIX items to the implementer alongside MUST FIX items. No findings are deferred to epic History. Every finding reaches a terminal state of FIXED or DISMISSED during the story.

## Context
The current dispatch loop explicitly instructs the main session to ignore SHOULD FIX findings: "SHOULD FIX findings from the reviewer are recorded in the epic's History section during closure -- they are NOT relayed to the implementer" (dispatch-pattern.md) and anti-pattern 10 in implement SKILL.md says "Do not relay SHOULD FIX findings to implementers." This story replaces that pattern with an active triage step where the main session decides fix-or-dismiss for each SHOULD FIX item.

This story depends on E-096-01 because the code-reviewer's updated output format (from that story) defines the contract that this story's triage step consumes.

## Acceptance Criteria
- [ ] **AC-1**: `dispatch-pattern.md` Step 6/7 (or equivalent) includes a triage step where the main session evaluates each SHOULD FIX finding and decides: accept (route to implementer) or dismiss (record one-line reason, finding closed). No deferral path exists.
- [ ] **AC-2**: `dispatch-pattern.md` no longer contains language about recording SHOULD FIX findings in epic History or deferring them to closure.
- [ ] **AC-3**: `implement/SKILL.md` Phase 3 Step 5 reflects the same triage model: after receiving reviewer findings, the main session triages ALL findings before routing to the implementer. Accepted SHOULD FIX items are included in the implementer feedback alongside MUST FIX items.
- [ ] **AC-4**: Anti-pattern 10 in `implement/SKILL.md` ("Do not relay SHOULD FIX findings to implementers") is replaced with guidance that reflects the triage model.
- [ ] **AC-5**: The Workflow Summary diagram in `implement/SKILL.md` is updated to show the triage step (the "SHOULD FIX -> epic History" annotation is removed or replaced).
- [ ] **AC-6**: Triage heuristics from the epic's Technical Notes are included in the dispatch-pattern or implement skill (whichever is the right home -- CA to decide) so the main session has guidance on when to accept vs dismiss SHOULD FIX items.
- [ ] **AC-7**: Both files are internally consistent -- no contradictory guidance about SHOULD FIX handling remains anywhere in either file.

## Technical Approach
Two files need coordinated updates. The changes are structural (rewriting the review loop handling) rather than additive.

**`/.claude/rules/dispatch-pattern.md`** -- The Dispatch Flow steps 6-7 currently describe APPROVED and NOT APPROVED paths. A triage step needs to be inserted between receiving the reviewer's findings and routing to the implementer. The triage step is where the main session reads ALL findings, accepts or dismisses each SHOULD FIX item, and then routes the combined set (MUST FIX + accepted SHOULD FIX) to the implementer.

**`/.claude/skills/implement/SKILL.md`** -- Phase 3 Step 5 mirrors the dispatch-pattern logic and needs the same structural update. Anti-pattern 10 needs replacement. The Workflow Summary diagram needs the "SHOULD FIX -> epic History" annotation replaced.

The triage heuristics from the epic's Technical Notes section should land in one of these two files. CA should decide the right home based on where the triage step is most naturally described.

## Notes
- The code-reviewer's behavior is upstream of this story -- it classifies findings the same way. What changes here is how the main session handles those findings after receiving them.
- The circuit breaker (2 rounds max) is unchanged.
- The APPROVED/NOT APPROVED verdict from the reviewer is unchanged -- APPROVED still means no MUST FIX items. What changes is that SHOULD FIX items in an APPROVED review are also triaged (not silently recorded in History).
