# E-096-01: Revise Code-Reviewer Finding Categories and Output Format

## Epic
[E-096: Resolve All Review Findings During Stories](./epic.md)

## Status
`TODO`

## Description
After this story is complete, the code-reviewer agent definition will reflect a resolution-first model where every finding reaches a terminal state (FIXED or DISMISSED) during the story. The SHOULD FIX category remains in the reviewer's rubric and output, but its semantics change: SHOULD FIX findings are no longer deferred -- they are triaged by the main session and either accepted (routed to implementer) or dismissed (with reason). The reviewer's structured output format and anti-patterns are updated to support this model.

## Context
The code-reviewer currently classifies findings as MUST FIX or SHOULD FIX. The downstream handling of SHOULD FIX (defer to epic History) is defined in implement SKILL.md, but the reviewer's own definition reinforces the deferral model in its structured format header ("recommended, does not block DONE") and anti-pattern 6. This story updates the reviewer's side; E-096-02 updates the coordinator's side.

## Acceptance Criteria
- [ ] **AC-1**: The SHOULD FIX section header in the Structured Findings Format no longer says "does not block DONE" -- it reflects that the main session will triage each item to a terminal state (fix or dismiss).
- [ ] **AC-2**: Anti-pattern 6 ("Never escalate a SHOULD FIX to MUST FIX between rounds") is revised to account for the new triage model -- the reviewer still does not escalate between rounds, but the anti-pattern text acknowledges that the main session may accept SHOULD FIX items for fixing.
- [ ] **AC-3**: The Structured Findings Format, Review Rubric, and all related sections in `code-reviewer.md` are internally consistent with the resolution-first model. No references to deferral, "epic History recording," or "does not block" semantics for SHOULD FIX findings.
- [ ] **AC-4**: The reviewer's core behavior is unchanged: it still classifies findings using the same rubric (priorities 1-7), still outputs MUST FIX and SHOULD FIX sections, still follows the circuit breaker. Only the framing of what happens to SHOULD FIX findings changes.

## Technical Approach
The code-reviewer agent definition (`.claude/agents/code-reviewer.md`) needs targeted edits to the Structured Findings Format section and Anti-Patterns section. The Review Rubric itself is unchanged -- the classification criteria stay the same. The change is in how the reviewer describes what happens to its findings after it produces them.

Key areas to update in `code-reviewer.md`:
- **Structured Findings Format**: The SHOULD FIX header currently reads "recommended, does not block DONE." This framing should indicate that the main session triages these items.
- **Anti-pattern 6**: Currently says "Never escalate a SHOULD FIX to MUST FIX between rounds unless new evidence emerges." The reviewer-side rule stays (the reviewer does not re-classify), but the text should acknowledge the main session's triage authority.
- **Any other references** to SHOULD FIX deferral semantics -- grep the file for "SHOULD FIX" and "epic History" to find all instances.
