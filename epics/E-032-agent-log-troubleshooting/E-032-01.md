## Description
After this story is complete, we will have confirmed that an agent can reliably start the Docker Compose stack from inside the devcontainer, verify the app is running via the health endpoint, and read application logs -- the foundational steps of the troubleshooting workflow documented in CLAUDE.md.

## Context
E-027 established the devcontainer-to-compose networking and documented troubleshooting commands in CLAUDE.md. This story exercises those commands for the first time from an agent's perspective. It is a smoke test: can the documented workflow actually be followed by an agent, and does it produce useful output?

## Acceptance Criteria
- [ ] **AC-1**: Given the devcontainer is running, when the agent runs `docker compose up -d` from `/workspaces/baseball-crawl`, then the app container starts successfully (verified by `docker compose ps` showing `app` status as `Up` or `running`).
- [ ] **AC-2**: Given the app container is running, when the agent runs `curl -s http://localhost:8001/health`, then the response is `{"status": "ok", "db": "connected"}` with HTTP 200.
- [ ] **AC-3**: Given the app container is running, when the agent runs `docker compose logs app`, then the output contains the uvicorn startup message and the `baseball-crawl API starting` log line.
- [ ] **AC-4**: Given the app container is running, when the agent runs `docker compose logs --tail=10 app`, then the output is limited to approximately 10 lines (demonstrating the `--tail` flag works as expected).
- [ ] **AC-5**: The agent produces a brief written report (as a comment in the story file under Notes, or as direct output) confirming each step worked, including the actual command output observed. If any step failed or was harder than expected, document the gap with a specific recommendation.

## Technical Approach
1. From `/workspaces/baseball-crawl`, run `docker compose up -d` and wait for the stack to stabilize.
2. Run `docker compose ps` and verify the `app` service is healthy.
3. Run `curl -s http://localhost:8001/health` and verify the response.
4. Run `docker compose logs app` and verify startup messages appear.
5. Run `docker compose logs --tail=10 app` and verify output is limited.
6. Record all command outputs and observations.

All commands are documented in CLAUDE.md lines 87-122. No new tooling or scripts are needed.
