# E-005-02: Build HTTP session factory with rate limiting and jitter

## Epic
[E-005: HTTP Request Discipline](epic.md)

## Status
`DONE`

## Description
After this story is complete, the project will have a `create_session()` factory in `src/http/session.py` that returns a configured httpx Client. The client comes pre-loaded with `BROWSER_HEADERS`, maintains a persistent cookie jar, and enforces a per-request delay with jitter between sequential requests. Any code in the project that needs to make HTTP requests calls `create_session()` rather than constructing its own httpx Client.

## Context
E-005-01 defined the header profile as data. This story makes that data operational: a factory function that stamps it onto every request, manages cookie state across a session, and silently enforces the rate limiting rules from CLAUDE.md.

The rate limiting is enforced inside the session via an httpx event hook -- it fires after every response. The caller does not need to add sleeps; the discipline is automatic. This is the key design choice: callers get the right behavior for free, and there is one place to tune it.

## Acceptance Criteria
- [x] **AC-1**: Given `session = create_session()`, when `session.get("https://example.com")` is called (mocked), then the outgoing request contains all keys from `BROWSER_HEADERS` with their correct values.
- [x] **AC-2**: Given `create_session()` with defaults, when two sequential requests are made, then the second request is sent no sooner than 1000ms after the first response (the default `min_delay_ms`).
- [x] **AC-3**: Given `create_session(min_delay_ms=1000, jitter_ms=500)`, when two sequential requests are made, then the elapsed time between them is between 1000ms and 1500ms (jitter is applied -- not always exactly 1000ms).
- [x] **AC-4**: Given a server that sets a `Set-Cookie` header in response, when a subsequent request is made on the same session, then the cookie is included in the `Cookie` header of the next request (cookie jar persistence).
- [x] **AC-5**: Given `create_session()`, when a request is made, then no log record at DEBUG, INFO, WARNING, ERROR, or CRITICAL level contains the substring `"Authorization"`, `"Bearer"`, or `"Cookie"`.
- [x] **AC-6**: `create_session()` signature: `def create_session(min_delay_ms: int = 1000, jitter_ms: int = 500) -> httpx.Client`. The returned client is a standard httpx.Client and supports all httpx.Client methods including context manager usage.
- [x] **AC-7**: Given `create_session()` is used as a context manager (`with create_session() as s:`), then it closes cleanly without error.

## Technical Approach
`src/http/session.py`:

```python
import random
import time
import httpx
from src.http.headers import BROWSER_HEADERS

def create_session(
    min_delay_ms: int = 1000,
    jitter_ms: int = 500,
) -> httpx.Client:
    """
    Returns a configured httpx.Client for all project HTTP requests.

    The client:
    - Sends BROWSER_HEADERS on every request (browser fingerprint)
    - Maintains a persistent cookie jar across requests
    - Enforces a per-request delay of min_delay_ms + random(0, jitter_ms) ms

    Auth credentials (Authorization header, Cookie overrides) must be
    injected by the caller after receiving the client -- do not pass
    them here.
    """
    ...
```

Rate limiting via httpx event hooks:

```python
def _make_rate_limit_hook(min_delay_ms: int, jitter_ms: int):
    def response_hook(response: httpx.Response) -> None:
        delay_s = (min_delay_ms + random.uniform(0, jitter_ms)) / 1000
        time.sleep(delay_s)
    return response_hook

client = httpx.Client(
    headers=BROWSER_HEADERS,
    cookies=httpx.Cookies(),
    event_hooks={"response": [_make_rate_limit_hook(min_delay_ms, jitter_ms)]},
)
```

httpx.Client already maintains a cookie jar automatically when `cookies=httpx.Cookies()` is passed. Set-Cookie response headers are picked up and replayed on subsequent requests to the same domain without any additional code.

Do NOT log header values anywhere in this module. The module may use `logging.debug("GET %s -> %d", url_path, response.status_code)` but never `logging.debug("Headers: %s", request.headers)`.

## Notes
Testing the jitter timing: make two mocked requests and measure elapsed wall time. Assert `1.0 <= elapsed <= 1.6` (1500ms max plus a small buffer for test execution overhead). Run the timing assertion in a loop of 10 iterations to catch cases where jitter accidentally rounds to zero.

Testing log safety: use Python's `logging.handlers.MemoryHandler` or `caplog` fixture to capture all log output during a request, then assert none of the captured messages contain credential-pattern strings.

Cookie jar test: mock server responds with `Set-Cookie: gc_session=abc123` on request 1, then assert request 2's `Cookie` header contains `gc_session=abc123`.
