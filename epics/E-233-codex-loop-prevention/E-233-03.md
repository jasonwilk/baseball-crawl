## Description
After this story is complete, the codex-review skill will detect regression candidates by comparing current findings against prior dispositions in the disposition log written by E-233-02 (and E-233-04's mirror). The implement skill's Phase 4b Step 3 will document the canonical fresh-eyes escape hatch protocol (Task-tool spawn with no shared prior-judgment context with the dispatch team CR) that fires on the regression-candidate tag. The codex-review skill's Step 4 (advisory triage) will reference that protocol so the same response fires when the user invokes codex-review directly outside dispatch. Together, these consume the disposition log and break out of the framing-flip loop pattern by requiring a no-prior-commitment judgment before remediation, regardless of which call site detected the regression candidate.

## Context
This story is the consumer half of the regression-detection mechanism. The producer half — the disposition log writer — ships in stories E-233-02 (codex-review skill side) and E-233-04 (implement skill Phase 4b mirror side); both write to the same shared state files per the canonical contract at `.claude/rules/codex-review-breaker.md`. The detector and the escape hatch are bundled here because they are meaningless apart: the detector emits the `[REGRESSION CANDIDATE]` tag; the escape hatch consumes it. Splitting them at any other boundary would create an intermediate state where one side is wired and the other isn't.

This story touches two skill files: `codex-review/SKILL.md` (for the Step 3.5 detector AND the Step 4 cross-skill protocol reference) and `implement/SKILL.md` (for the canonical escape hatch protocol at Phase 4b Step 3 item 7). The escape hatch protocol is defined ONCE in the implement skill and referenced from codex-review Step 4 — single protocol, two call sites. This widens the trigger surface to cover both the in-dispatch automated codex path and the user-directly-invoked codex-review skill path (the loop hot zone).

- [ ] **AC-2**: Step 3.5 documents the file-overlap rule (string equality on at least one path) and the summary-overlap rule per TN-5: case-insensitive token overlap >= 50% where the threshold applies to the SHORTER summary's filtered token set; tokens are whitespace-split, lowercased, leading/trailing-punctuation-stripped, AND filtered to remove tokens shorter than 3 characters and a small stop-word set (the, a, an, and, or, but, of, in, on, to, for, with, is, was, be, by, at, as, that, this, it, its). The stop-word filter prevents short summaries from over-matching on common words.

## Technical Approach
Reference Technical Notes sections TN-5 (regression detector logic with stop-word filter), TN-6 (fresh-eyes escape hatch mechanism + spawn prompt + single-protocol-two-call-sites pattern), TN-7 (standalone exemption — already installed by E-233-02, this story cross-references it), and TN-11 (diff-context resolution algorithm).

For Step 3.5 in `codex-review/SKILL.md`: the new step inserts cleanly between Steps 3 and 4. The Workflow Summary diagram update is consolidated under AC-14 — this story owns ALL diagram changes for this epic (covering Story 02's breaker section AND this story's Step 3.5/Step 4 changes).

For item 7 in `implement/SKILL.md` Phase 4b Step 3: the existing items 1-6 remain unchanged. Item 7 is appended after the circuit breaker (item 6) because its trigger is independent of round count — the escape hatch can fire on round 2 if round 2 produces a regression candidate against round 1's dispositions.

For the cross-skill reference (AC-13): the codex-review skill's Step 4 (advisory triage) gets a small addition that detects the `[REGRESSION CANDIDATE]` tag and invokes the canonical protocol in `implement/SKILL.md` Phase 4b Step 3 item 7. Implementer should write Step 4's addition as a brief documented branch ("If any finding carries `[REGRESSION CANDIDATE]` AND an epic worktree path is available, follow the fresh-eyes escape hatch protocol in `implement/SKILL.md` Phase 4b Step 3 item 7 before continuing triage. The protocol's verdict (a/b/c) determines whether the tagged finding proceeds to triage, is dismissed, or is escalated to the user.") rather than duplicating item 7's text.

The summary-overlap rule (50% case-insensitive token overlap with stop-word/min-length filter) is intentionally simple. A more sophisticated similarity metric (e.g., embedding-based) is out of scope; if the simple rule produces too many false positives or false negatives in practice, that's a follow-up tuning epic.

The Task-tool spawn for the escape hatch is NOT a teammate on the dispatch team — it's a one-shot Task-tool invocation. The Task tool intrinsically provides a fresh instance; the load-bearing property is that the spawn prompt MUST NOT include any context from the dispatch team CR's prior judgments. The lack of carried prior-judgment context is the design intent: confirmation bias is the failure mode the spawn guards against.
