## Technical Approach
Reference Technical Notes sections TN-2 (state file locations), TN-3 (log entry shape including `R{round}-{codex_id_or_ordinal}` namespacing), TN-4 (circuit breaker semantics), TN-7 (standalone exemption), and `/workspaces/baseball-crawl/.claude/rules/codex-review-breaker.md` (canonical contract for both consumers — load this rule file verbatim before drafting the skill section).

The two state files are at:
- `<epic-worktree-path>/.codex-review-rounds` — single integer
- `<epic-worktree-path>/.codex-dispositions.jsonl` — append-only JSON lines

The `<epic-worktree-path>` placeholder refers to the epic worktree path created by `implement/SKILL.md` Phase 2 Step 1 (`/tmp/.worktrees/baseball-crawl-E-NNN/`). Both consumers (this skill section AND the implement Phase 4b mirror in Story E-233-04) MUST use the dispatch-context-provided path; neither may glob-detect worktrees on disk. (See `.claude/rules/codex-review-breaker.md` "Worktree-Path Resolution" for the canonical statement.)

The standalone-mode exemption (TN-7) means: when the skill is invoked without an epic worktree path, both the counter mechanism and the disposition log writer are skipped entirely. The existing prose remediation log to `.project/research/codex-review-YYYY-MM-DD-remediation.md` continues to run in standalone mode unchanged.

The circuit breaker hard-stop is a documented skill behavior, not a code path the implementer writes. The skill is markdown that the main session executes — the implementer writes the documented procedure that the main session will follow on future invocations.

The Workflow Summary diagram update is consolidated into Story E-233-03 AC-14, which covers all diagram changes for both this story's breaker entry gate AND Story 03's Step 3.5 detector. This avoids duplicate diagram-edit ACs across stories.
