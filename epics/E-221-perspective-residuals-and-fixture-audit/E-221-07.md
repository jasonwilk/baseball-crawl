## Description
After this story is complete, `bb data reconcile --game-id X` will process all loaded perspectives for the given game, not just the default (which currently silently reconciles only one perspective for cross-perspective games). Operators running the CLI against a specific game will see discrepancies from both perspectives instead of losing half the data.

## Context
E-220 round 6 cluster 4 updated `reconcile_all` to iterate `(game_id, perspective_team_id)` pairs from `plays`. The single-game CLI path at `src/cli/data.py:1160` was not updated — it still calls `reconcile_game(conn, game_id, dry_run=...)` without a `perspective_team_id` kwarg, which means the default path reconciles only the first perspective (typically home-first).

For any cross-perspective game (the modal case for LSB-adjacent tracked opponents), this means `bb data reconcile --game-id X` silently reports half the discrepancies. Operators have no way to know this from the output.

This is the "helper knows, caller doesn't" pattern — exactly what the stop-gate trigger was designed to catch, and exactly why we created E-221 instead of grinding round 9.

## Acceptance Criteria
- [ ] **AC-1**: Given a game has plays data loaded from two perspectives (A and B), when `bb data reconcile --game-id X` is invoked, then `reconcile_game` is called for each perspective that has plays data for X. Both perspectives' discrepancies land in `reconciliation_discrepancies` during the same CLI invocation.
- [ ] **AC-2**: Given a game has plays data loaded from one perspective only, when the CLI is invoked, the behavior is unchanged from current — one reconciliation pass runs.
- [ ] **AC-3**: A RED test is added to `tests/test_reconciliation.py` (or a CLI-specific test file if one exists) that loads a game from two perspectives, invokes the CLI path for that game_id, and asserts that both perspectives' discrepancies are persisted (distinguishable by the 6-column UNIQUE from E-220 round 7). The test uses `load_real_schema(db)` from Story 01.
- [ ] **AC-4**: If DE recommends adding a `--perspective-team-id N` flag per TN-7 Option B, the flag is implemented. When omitted, iteration across all perspectives happens (Option A default). When supplied, only the named perspective is reconciled. DE call must be routed via SendMessage before SE implements; document the choice in the PR.
- [ ] **AC-5**: The `--dry-run` vs `--execute` semantics are preserved unchanged. The per-perspective iteration must work consistently in both modes.

## Technical Approach
Per TN-7 in the epic. Two options:

**Option A**: Inside the CLI handler, query `SELECT DISTINCT perspective_team_id FROM plays WHERE game_id = ?` and iterate, calling `reconcile_game(conn, game_id, dry_run=..., perspective_team_id=ptid)` for each. Mirrors `reconcile_all`'s pattern from engine.py:455-466.

**Option B**: Add a `--perspective-team-id N` flag. When supplied, call once with that perspective. When omitted, iterate as in Option A.

**Before implementing, SE must consult DE** on which option fits the operator context. Option A alone is sufficient for correctness. Option B is a UX improvement for debugging (operator can say "reconcile only my perspective for this game"). DE's call determines which.

Look at `src/cli/data.py` around line 1160 for the current single-game path. `reconcile_game` already accepts a `perspective_team_id` kwarg (round 6 cluster 4 added it), so the plumbing change is local to the CLI handler.
