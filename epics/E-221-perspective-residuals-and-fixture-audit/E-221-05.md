## Description
After this story is complete, `_delete_team_cascade` in `src/api/routes/admin.py` will correctly scope Phase 1a DELETEs to the deleted team's perspective only. Rows belonging to other participants' perspectives of shared games will survive, consistent with E-220 round 6 cluster 2's "preserve what other perspectives own" principle.

## Context
Round 8's R8-P1-1 finding: Phase 1a of `_delete_team_cascade` cascades stat-table DELETEs by `game_id IN (...)` — deleting every row for that game, regardless of which perspective tagged it. This is a round 6 incompleteness: cluster 3 added the informed-consent UI, cluster 2 added game-row preservation, but Phase 1a was never updated to scope by `perspective_team_id`.

When an admin deletes team A (informed-consent confirmation accepted), Phase 1a currently wipes ALL stat rows for any game team A participated in, including rows tagged with team B's perspective. Team B is an unrelated tracked team that happened to play team A. Its scouting data for those shared games vanishes.

Story 04 lands the RED test that catches this. This story lands the fix.

## Acceptance Criteria
- [ ] **AC-1**: Given `_delete_team_cascade(conn, team_id, confirm_cross_perspective=True)` is called on a team that shares games with other perspectives, when the cascade completes, then Phase 1a has deleted only rows where `perspective_team_id = team_id` across `plays`, `spray_charts`, `player_game_batting`, `player_game_pitching`, and `reconciliation_discrepancies`. Rows tagged with other teams' perspectives for the same game_id MUST survive.
- [ ] **AC-2**: Given the RED test from Story 04 (`test_delete_team_cascade_preserves_other_perspective_rows` or equivalent), when this story's fix is applied, then the test PASSES.
- [ ] **AC-3**: Given a team has no cross-perspective sharing (the common case), when `_delete_team_cascade` runs, then its behavior is unchanged from pre-fix — all rows for the team's games are deleted. No performance regression or behavioral change for the common path.
- [ ] **AC-4**: The cascade still handles the round 6 cluster 2 games-row preservation correctly (delete the games row only when no `game_perspectives` entries remain). This story does NOT touch the games-row logic — verify by running the existing `test_cascade_delete_team_preserves_games_row_when_other_perspective_remains` and confirming it still passes.
- [ ] **AC-5**: The fix is scoped to `_delete_team_cascade`'s Phase 1a. Do NOT modify Phase 1b (already correct from round 6 cluster 3) or the informed-consent UI layer (already correct from round 6 cluster 3). If DE consultation suggests reusing `_delete_game_scoped_data_for_perspectives` from `src/reports/generator.py:1306-1389`, prefer reuse over duplication.

## Technical Approach
Per TN-5 in the epic. Two options:

**Option A (preferred if reuse is clean)**: Replace Phase 1a's DELETEs with a call to `_delete_game_scoped_data_for_perspectives(conn, game_ids, [team_id])`. That helper already handles per-perspective scoping correctly.

**Option B**: Add `AND perspective_team_id = ?` to each DELETE in Phase 1a, mirroring how round 6 cluster 3 scoped Phase 1b.

Consult DE via SendMessage before choosing. The helper at `generator.py:1306-1389` was written for the report cleanup path — verify it handles the admin cascade context correctly (specifically: does it call any report-specific state that would break in the admin path?). If reuse is clean, go with A. If not, go with B.

**Do NOT** touch the game_perspectives cleanup or games-row preservation — those are already correct.
