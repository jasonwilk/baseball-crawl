# E-221-05: R8-P1-1 Phase 1a perspective scoping in admin delete

## Epic
[E-221: Test Fixture Schema Parity Audit + Post-E-220 Perspective Residuals](../E-221-perspective-residuals-and-fixture-audit/epic.md)

## Status
`IN_PROGRESS`

## Description
After this story is complete, `src/api/routes/admin.py::_delete_team_cascade` correctly handles cross-perspective games during a single-team delete: (1) Phase 1a's stat-table DELETEs are scoped by `perspective_team_id` so only the deleted team's rows are removed; (2) Phase 2's games-row DELETE is guarded by a `NOT EXISTS (game_perspectives)` check so the shared game row is preserved while any other perspective still owns data for it. The RED test from E-221-04 (`test_delete_team_cascade_preserves_other_perspective_rows`) goes GREEN. R8-P1-1 is closed.

## Context
Round 8's R8-P1-1 finding: Phase 1a of `_delete_team_cascade` cascades stat-table DELETEs by `game_id IN (...)` — deleting every row for that game, regardless of which perspective tagged it. This is a round 6 incompleteness: cluster 3 added the informed-consent UI, cluster 2 added game-row preservation, but Phase 1a was never updated to scope by `perspective_team_id`.

When an admin deletes team A (informed-consent confirmation accepted), Phase 1a currently wipes ALL stat rows for any game team A participated in, including rows tagged with team B's perspective. Team B is an unrelated tracked team that happened to play team A. Its scouting data for those shared games vanishes.

Story 04 lands the RED test that catches this. This story lands the fix.

**Scope correction (2026-04-12, discovered during E-221-04 dispatch)**: TN-5 originally claimed that "game-level cleanup was already correct from round 6 cluster 2." Investigation during Story 04 showed that comment applied to `cascade_delete_team` in `src/reports/generator.py`, NOT to `_delete_team_cascade` in `src/api/routes/admin.py`. The admin cascade's Phase 2 (around `admin.py:1024-1028`) unconditionally deletes the games row without a `NOT EXISTS (game_perspectives)` guard. Story 05's scope is therefore expanded to fix Phase 2 in addition to the original Phase 1a fix. Both E-221-04's AC-1 (Phase 1a) and AC-2 (Phase 2) assertions will be exercised by the same RED test.

## Technical Approach
Per TN-5 in the epic. Two options:

**Option A (preferred if reuse is clean)**: Replace Phase 1a's DELETEs with a call to `_delete_game_scoped_data_for_perspectives(conn, game_ids, [team_id])`. That helper already handles per-perspective scoping correctly.

**Option B**: Add `AND perspective_team_id = ?` to each DELETE in Phase 1a, mirroring how round 6 cluster 3 scoped Phase 1b.

Consult DE via SendMessage before choosing. The helper at `generator.py:1306-1389` was written for the report cleanup path — verify it handles the admin cascade context correctly (specifically: does it call any report-specific state that would break in the admin path?). If reuse is clean, go with A. If not, go with B.

**Do NOT** touch the game_perspectives cleanup or games-row preservation — those are already correct.

**Technical Approach (updated 2026-04-12, scope expansion)**: Two fixes in `src/api/routes/admin.py::_delete_team_cascade`:

1. **Phase 1a perspective scoping (R8-P1-1)**: Scope each DELETE to the deleted team's perspective. The current code at roughly `admin.py:955-990` uses `DELETE FROM <table> WHERE game_id IN (SELECT game_id FROM games WHERE home_team_id = ? OR away_team_id = ?)` with no perspective filter. Change each DELETE to add `AND perspective_team_id = ?` (bound to the deleted team's id). Applies to: `plays`, `spray_charts`, `player_game_batting`, `player_game_pitching`, `reconciliation_discrepancies`. Do NOT touch Phase 1b (it already filters correctly from round 7).

2. **Phase 2 games-row conditional DELETE**: The current code at roughly `admin.py:1024-1028` does `DELETE FROM games WHERE home_team_id = ? OR away_team_id = ?` without any game_perspectives guard. Change this to add a `NOT EXISTS (SELECT 1 FROM game_perspectives gp2 WHERE gp2.game_id = games.game_id)` clause so the games row is preserved when another perspective still owns data for the shared game. Pattern reference: `src/reports/generator.py::_delete_game_scoped_data_for_perspectives` (lines 1306-1390). Do NOT refactor admin.py to delegate to the generator.py helper — that cleaner refactor is deliberately deferred to a follow-up idea (IDEA-069). Story 05 keeps the fix in-place so it stays surgical.

Implementation note: the `game_perspectives` delete for team A happens in Phase 1b (existing round 7 behavior, correct). By the time Phase 2 runs, `game_perspectives` has already been trimmed to exclude team A. So the NOT EXISTS subquery correctly evaluates "is there any remaining perspective other than team A that still owns this game?"
