# E-221-03: CI guardrail forbidding inline CREATE TABLE in tests/

## Epic
[E-221: Test Fixture Schema Parity Audit + Post-E-220 Perspective Residuals](../E-221-perspective-residuals-and-fixture-audit/epic.md)

## Status
`TODO`

## Description
After this story is complete, a CI-enforced check will fail any PR that introduces a new inline `CREATE TABLE` statement in a file under `tests/`. This prevents future drift from re-creating the fixture-schema-parity problem that E-221 was built to solve.

## Context
Story 02 migrates the 7 current offenders. Without a guardrail, future contributors (agents or humans) will re-introduce the pattern out of convenience. The enabling condition for round-by-round discovery during E-220 was precisely this drift. Without a check, we will hit it again.

The check should be noisy and clear: a failing test or lint step with a message naming the offending file, line number, and an explanation pointing at the shared helper.

## Acceptance Criteria
- [ ] **AC-1**: Given a PR that adds a file `tests/test_something.py` containing `CREATE TABLE foo (...)` outside of approved exceptions, when the test suite runs, then the guardrail check fails with a message that identifies the file and line.
- [ ] **AC-2**: Given the current repo state (post-Story-02), when the guardrail check runs, it passes. No false positives on legitimate usages in the shared helper or explicitly-exempted files.
- [ ] **AC-3**: The guardrail is implemented as a pytest test (not a shell script, not a pre-commit hook) so it runs in the same harness as the rest of the suite and cannot be skipped by a caller who forgets to run pre-commit hooks.
- [ ] **AC-4**: The guardrail allows a file-level opt-out pragma comment (e.g., `# noqa: fixture-schema` at the top of a file) for legitimate exceptions, documented in the test's docstring. No files should require the opt-out after Story 02.
- [ ] **AC-5**: The guardrail test's own failure message suggests the fix: "Use `load_real_schema(conn)` from `tests/conftest.py` instead. See E-221 for context."

## Technical Approach
Per TN-3 in the epic. Implement as `tests/test_no_inline_schemas.py` (or equivalent name). Walk `tests/**/test_*.py` with `Path.rglob`, read each file, grep for `CREATE TABLE` via a simple substring or regex check, assert none found except in allowed locations.

Keep the test under 50 lines. Do NOT attempt to parse SQL or handle complex edge cases — a substring check on the literal `CREATE TABLE` is sufficient because agents and humans write it literally, not via string concatenation.
