## Description
After this story is complete, `tests/test_admin_delete_cascade.py` will have a test case that explicitly asserts cross-perspective row preservation during a single-team delete. The test will run against the real schema (via Story 01's helper), will FAIL against current production code (because R8-P1-1 wipes cross-perspective rows), and will unblock Story 05's fix with a RED test already in place.

## Context
Round 8's R8-P1-1 finding (Phase 1a of `_delete_team_cascade` cascades by `game_id IN (...)` without perspective scoping) was not caught by the existing round 6 cluster 2 test because that test only asserts own-team rows disappear — it never asserts the OTHER participant's perspective-tagged rows survive. Had it, R8-P1-1 would have been caught in round 6.

This story adds the missing assertion shape. It lands BEFORE Story 05 (the fix) because the round 7 RED-GREEN precedent is: prove the bug exists via a failing test, then fix it. Code-reviewer will verify the commit sequence during dispatch.

## Acceptance Criteria
- [ ] **AC-1**: Given a game is loaded from perspective A and perspective B via GameLoader (producing two sets of stat rows with different `perspective_team_id` values), when `_delete_team_cascade` is called on team A with the informed-consent confirmation flag set, then perspective A's stat rows are deleted AND perspective B's stat rows for the SAME game_id are still present in `plays`, `spray_charts`, `player_game_batting`, `player_game_pitching`, and `reconciliation_discrepancies`.
- [ ] **AC-2**: Given the same scenario, the `games` row for the shared game_id is preserved (round 6 cluster 2 behavior), because `game_perspectives` still has an entry for team B.
- [ ] **AC-3**: The test uses `load_real_schema(db)` from Story 01, so FK constraints are enforced during the test (without this, the test is meaningless — it would pass against a fixture that doesn't care about FKs).
- [ ] **AC-4**: Before Story 05 lands, this test FAILS when run against the current working tree. The failure mode is: perspective B's rows are missing after the cascade. Capture the failure output in the commit message for the story landing this test.
- [ ] **AC-5**: The test is named descriptively (e.g., `test_delete_team_cascade_preserves_other_perspective_rows`) and its docstring names E-221 and R8-P1-1 so future readers understand the provenance.

## Technical Approach
Per TN-4 in the epic. Pattern off the existing `test_cascade_delete_team_preserves_games_row_when_other_perspective_remains` in `tests/test_admin_delete_cascade.py` — that test asserts the `games` row survives; this new test extends the assertion to the stat tables for the other perspective.

Setup: two teams (A, B), both with `membership_type='tracked'`. Load one game from each perspective via GameLoader (or seed rows directly with correct `perspective_team_id` values — seeding is faster and the fixture exercises are the point of this test, not the loader path). Call `_delete_team_cascade(conn, team_a_id, confirm_cross_perspective=True)`. Then run SELECT queries scoped to B's perspective across the 5 stat tables and assert rows still present.

Note: this story is explicitly a test-only change. It does NOT fix R8-P1-1. The test is expected to fail when run. The test does NOT go green until Story 05 lands. During dispatch, code-reviewer should see a failing test in this story's commit that then passes in Story 05's commit.
