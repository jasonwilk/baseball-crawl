`TODO`

## Description
After this story is complete, the operator can run `python scripts/bootstrap.py` to execute the full pipeline in one command: validate credentials, check team configuration, crawl all data, and load into the database. The script provides clear status output at each stage and exits early with actionable messages when a pre-flight check fails.

## Context
Today, the operator must run three separate commands in sequence (`check_credentials.py`, `crawl.py`, `load.py`) with no unified error handling between them. If credentials expire mid-crawl, the error is buried in crawler log output. The bootstrap script chains these steps with validation gates, providing a single entry point for the "capture credentials, then go" workflow.

## Acceptance Criteria
- [ ] **AC-1**: `scripts/bootstrap.py` exists and is executable via `python scripts/bootstrap.py`
- [ ] **AC-2**: The script runs four stages in order, printing a clear header for each:
  1. "Checking credentials..." -- calls `check_credentials()` from `scripts/check_credentials.py`
  2. "Checking team configuration..." -- validates teams are configured
  3. "Crawling data..." -- calls `crawl.run()`
  4. "Loading data..." -- calls `load.run()`
- [ ] **AC-3**: If credential check fails (exit code != 0), the script prints the credential check message and exits immediately without proceeding to crawl
- [ ] **AC-4**: Team configuration check reads `config/teams.yaml` and verifies that at least one team has an `id` that does NOT start with `REPLACE_WITH_`. If all teams have placeholder IDs, the script prints "No teams configured -- add teams via the admin UI at /admin/teams or edit config/teams.yaml" and exits with code 1
- [ ] **AC-5**: If team config is valid, the script prints the number of configured teams and their names
- [ ] **AC-6**: The crawl stage calls `crawl.run()` directly (imported function, not subprocess). If it returns non-zero, the script prints a warning but continues to the load stage (partial crawl data is still loadable)
- [ ] **AC-7**: The load stage calls `load.run()` directly. If it returns non-zero, the script reports the error
- [ ] **AC-8**: After all stages, the script prints a summary: credentials (valid/expired), teams (count), crawl (success/partial/failed), load (success/failed)
- [ ] **AC-9**: The script accepts `--check-only` flag that runs only the credential check and team config check, then exits (no crawl or load)
- [ ] **AC-10**: The script accepts `--profile web|mobile` flag (default: `web`) which is passed to `GameChangerClient` when instantiated by crawl. If E-050-02 is not yet merged, the `--profile` flag is accepted but only `web` works (mobile raises an error from the client). Implementation: the profile is passed by setting `CRAWL_PROFILE` in the environment or by monkey-patching `GameChangerClient` default -- use whichever approach is cleanest.
- [ ] **AC-11**: The script accepts `--dry-run` flag that passes through to both `crawl.run(dry_run=True)` and `load.run(dry_run=True)`
- [ ] **AC-12**: The script NEVER prints or logs credential values
- [ ] **AC-13**: Tests in `tests/test_bootstrap.py` cover: successful full run (all stages pass), credential failure (early exit), team config failure (early exit), check-only mode, crawl failure with load continuing, and dry-run passthrough. All tests mock `check_credentials()`, `crawl.run()`, and `load.run()` -- no real API calls or file I/O.

## Technical Approach
1. Create `scripts/bootstrap.py` following the same structure as `scripts/crawl.py`:
   - Project root on `sys.path`
   - `logging.basicConfig` for structured output
   - `_build_arg_parser()` for CLI flags
   - `run()` function with all logic (importable for testing)
   - `main()` entry point
2. Import `check_credentials` from `scripts.check_credentials` (AC-9 in E-050-01 ensures it is importable)
3. Import `run` from `scripts.crawl` and `scripts.load` (both already have importable `run()` functions)
4. Use `load_config()` from `src/gamechanger/config` for team config validation
5. For `--profile` passthrough: the simplest approach is to modify `scripts/crawl.py` to accept an optional `profile` parameter in its `run()` function, which gets passed to `GameChangerClient(profile=profile)`. This is a minor, backward-compatible change to `crawl.py`.
6. Status output uses `print()` for human-readable stage headers (not logging) -- these are operator-facing status messages, not log entries
