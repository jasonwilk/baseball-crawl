# E-050-02: Profile-aware GameChangerClient

## Epic
[E-050: Credential Validation and Crawl Bootstrap](epic.md)

## Status
`TODO`

## Description
After this story is complete, `GameChangerClient` will accept a `profile` parameter that selects between the web browser and mobile (iOS Odyssey) header profiles when creating its HTTP session. This connects the dual-header infrastructure from E-049-05 to the API client that all crawlers use, enabling the operator to run crawls with either web or mobile credentials.

## Context
E-049-05 adds `profile: Literal["web", "mobile"]` to `create_session()` and exports both `BROWSER_HEADERS` and `MOBILE_HEADERS`. However, `GameChangerClient.__init__` currently calls `create_session()` without a profile argument, so it always uses web headers. This story is the integration point: after it lands, any code that instantiates `GameChangerClient(profile="mobile")` will send the iOS Odyssey header profile on all requests. The bootstrap script (E-050-03) and crawl orchestrator (`scripts/crawl.py`) can then accept a `--profile` flag and pass it through.

## Acceptance Criteria
- [ ] **AC-1**: `GameChangerClient.__init__` accepts a `profile` parameter with type `str` and default value `"web"`
- [ ] **AC-2**: The `profile` parameter is passed through to `create_session(profile=profile)` when building the internal HTTP session
- [ ] **AC-3**: `GameChangerClient(profile="web")` behaves identically to the current `GameChangerClient()` -- backward compatible, no existing callers need changes
- [ ] **AC-4**: `GameChangerClient(profile="mobile")` creates a session with mobile headers (the `User-Agent` on outgoing requests matches `MOBILE_HEADERS["User-Agent"]`)
- [ ] **AC-5**: `GameChangerClient(profile="invalid")` raises `ValueError` (propagated from `create_session()`)
- [ ] **AC-6**: The `gc-app-name` header behavior respects the captured value from `.env` (`GAMECHANGER_APP_NAME`). If the env var is set, it is used regardless of profile. If the env var is absent, `gc-app-name` is not sent (the `.get()` with default `"web"` is updated to not set a default when profile is `"mobile"`, since mobile traffic may not send `gc-app-name`)
- [ ] **AC-7**: The class docstring is updated to document the `profile` parameter and its effect
- [ ] **AC-8**: Tests in `tests/test_gamechanger_client.py` cover: default profile uses web UA, explicit `profile="mobile"` uses mobile UA, invalid profile raises `ValueError`, and `gc-app-name` behavior for both profiles
- [ ] **AC-9**: All existing `GameChangerClient` tests pass without modification (backward compatible default)

## Technical Approach
1. In `src/gamechanger/client.py`:
   - Add `profile: str = "web"` to `__init__` signature
   - Change `create_session(min_delay_ms=..., jitter_ms=...)` to `create_session(min_delay_ms=..., jitter_ms=..., profile=profile)`
   - Update `gc-app-name` logic: use `GAMECHANGER_APP_NAME` from `.env` if present; if absent AND profile is `"mobile"`, do not set a default. Current code: `self._credentials.get("GAMECHANGER_APP_NAME", "web")` -- change to: set `gc-app-name` only if the env var is present, OR if profile is `"web"` (in which case default to `"web"`)
   - Update docstring
2. In tests:
   - Add parametrized tests for profile selection
   - Mock `create_session` to verify the `profile` argument is passed through
   - Test `gc-app-name` conditional logic

## Notes
- This story requires E-049-05 to be complete. If E-049-05 is not yet done, this story remains BLOCKED.
- The `profile` parameter type is `str` (not `Literal`) at the client level to avoid importing typing concerns. `create_session()` handles the validation.
