## Description
After this story is complete, the project will have a comprehensive analysis of mobile (iOS) proxy traffic compared against web profile documentation. This includes: header parity between mobile and web profiles, endpoint coverage differences, response behavior differences, and updated `profiles.mobile` status fields in endpoint docs where mobile traffic was observed.

## Context
The GameChanger iOS app ("Odyssey") uses a different header profile than the web app: different User-Agent, gc-app-version header, different Accept-Encoding preferences, and potentially different gc-app-name values. The project has `MOBILE_HEADERS` in `src/http/headers.py` based on a 2026-03-05 proxy capture, but most endpoint docs have `profiles.mobile.status: unverified`.

Mobile credentials are NOT currently captured in `.env` (no `_MOBILE` suffix env vars for refresh token, client key, etc.). This story requires the user to run a mobile proxy session first. If no mobile proxy data exists, the story produces the analysis framework and a checklist for the user.

The proxy header_capture addon already diffs captured headers against `MOBILE_HEADERS` for iOS traffic. The endpoint_logger captures all traffic paths regardless of source.

## Acceptance Criteria
- [ ] **AC-1**: Given mobile proxy session data exists in `proxy/data/sessions/` with iOS traffic, when the analysis script runs, then it reads the endpoint-log.jsonl entries with `source: "ios"` and compares mobile-observed endpoints against the documented endpoint set.
- [ ] **AC-2**: Given mobile proxy session header-report.json data, when analyzed, then the script reports header parity between iOS captured headers and `MOBILE_HEADERS` from `src/http/headers.py`, identifying: headers in traffic but not in the canonical dict, headers in the canonical dict but not in traffic, and value differences.
- [ ] **AC-3**: Given mobile and web proxy traffic for the same endpoints, when compared, then the script reports per-endpoint behavioral differences: different status codes, different response shapes, different Accept header requirements, or different auth behavior.
- [ ] **AC-4**: Given endpoints observed in mobile traffic, when the analysis completes, then the report includes recommended `profiles.mobile.status` updates (from `unverified` to `observed`) for each observed endpoint, with the proxy session ID and date. Doc file updates are deferred to Story 05.
- [ ] **AC-5**: Given no mobile proxy session data exists, when the script runs, then it outputs a clear message: "No mobile proxy sessions found. Run a mobile proxy session on the Mac host first." and exits cleanly with a non-zero exit code. The script must NOT fail with an unhandled exception.
- [ ] **AC-6**: The script has tests covering the mobile/web comparison logic and header parity analysis (using mock data -- no live proxy sessions required for tests).

## Technical Approach
Build an analysis script that:
1. Scans `proxy/data/sessions/` for sessions containing iOS traffic (check endpoint-log.jsonl for `source: "ios"` entries)
2. Aggregates mobile-observed endpoints and headers across sessions
3. Compares mobile endpoints against documented endpoint set
4. Reads header-report.json for mobile header parity data (this is already generated by the `HeaderCapture` addon)
5. If Story 02 has run, compares mobile behavior against confirmed web behavior
6. Generates a parity report and optionally updates endpoint doc `profiles.mobile` fields

The proxy addons already capture source detection (ios/web/unknown) in endpoint-log.jsonl and per-source header diffs in header-report.json. This story builds on that existing infrastructure.

## Notes
- The iOS app may access endpoints not seen in web traffic (e.g., sync endpoints, push notification endpoints). These should be flagged as "mobile-only" in the report, not treated as errors.
- The three HTTP 500 endpoints (IDEA-011) may work from mobile. If they return 200 in mobile traffic, that is significant and should be prominently noted in the report.
- Mobile proxy captures require the iOS device to trust the mitmproxy CA certificate. See `docs/admin/mitmproxy-guide.md` for setup instructions.
- The `source` field in endpoint-log.jsonl is set by `gc_filter.detect_source()` based on the User-Agent header. iOS traffic has source "ios", web traffic has source "web".
