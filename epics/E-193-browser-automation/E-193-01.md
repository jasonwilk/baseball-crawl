## Description
After this story is complete, a headless Chrome browser runs as a Docker Compose sidecar service alongside the existing app, traefik, and cloudflared services. The Chrome service is dev-only (defined in `docker-compose.override.yml.example`, the tracked template) and exposes a CDP WebSocket endpoint at `localhost:3000` that `agent-browser` can connect to from the devcontainer shell.

## Context
Agents need a headless browser to verify UI changes. Currently, Playwright's Chromium binary (928MB) is downloaded into the devcontainer. A Docker sidecar is lighter, more maintainable, and provides crash recovery. The `agent-browser` tool connects to Chrome via CDP (Chrome DevTools Protocol) over WebSocket. Note: `agent-browser` has zero npm dependencies -- Playwright is NOT a transitive dependency. The Chromium was installed separately.

## Acceptance Criteria
- [ ] **AC-1**: Given a fresh `docker compose up -d` (with override present), when the stack is healthy, then a `chrome` service is running and its health check passes.
- [ ] **AC-2**: Given the Chrome sidecar is running, when `./node_modules/.bin/agent-browser --connect-over-cdp ws://localhost:3000 open "http://localhost:8001"` is executed from the devcontainer shell, then the command succeeds and a browser session is established. *(User-verified manually in main checkout post-merge, not in epic worktree -- see Technical Notes "Worktree Constraint.")*
- [ ] **AC-3**: Given the Chrome sidecar is running, when `./node_modules/.bin/agent-browser snapshot` is executed after AC-2, then an accessibility tree snapshot of the app's page is returned. *(User-verified manually in main checkout post-merge, not in epic worktree -- see Technical Notes "Worktree Constraint.")*
- [ ] **AC-4**: The Chrome service is defined in `docker-compose.override.yml.example` (the tracked template, NOT in the base `docker-compose.yml`). When copied to `docker-compose.override.yml` (gitignored), Chrome starts in dev. In production (no override), it does not exist. Per Technical Notes "Chrome Sidecar Configuration."
- [ ] **AC-5**: `PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD=1` is set in `containerEnv` in `devcontainer.json` (NOT `remoteEnv`) per Technical Notes "Playwright Chromium Removal." No step in `postCreateCommand` triggers a Playwright browser download. Observable outcome: `~/.cache/ms-playwright/` does not exist in a fresh container build.

## Technical Approach
The Chrome sidecar uses `browserless/chrome` v2 (multi-arch). It needs a host-port binding (`"127.0.0.1:3000:3000"`) because the devcontainer shell is the Docker-in-Docker host, not a compose service -- Docker service-name DNS does not resolve from the devcontainer shell. The Chrome service is defined in `docker-compose.override.yml.example` (tracked template); users copy to `docker-compose.override.yml` (gitignored) for dev. Production has no override, so Chrome does not exist. The Playwright skip env var in `containerEnv` prevents any accidental browser download during container setup.

## Notes
- SE recommends `browserless/chrome` v2 for multi-arch support (amd64 + arm64), built-in health check, and CDP WebSocket on port 3000.
- `npx agent-browser` does NOT work (misparses flags) -- always use `./node_modules/.bin/agent-browser`.
- The app must be healthy before browser verification is meaningful. Consider whether the Chrome service should `depends_on` the app service.
- AC-2 and AC-3 cannot be verified in the epic worktree (Docker services unreachable from worktrees). The implementer writes the config; the user verifies manually post-merge in the main checkout.
