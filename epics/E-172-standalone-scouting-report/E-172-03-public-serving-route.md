# E-172-03: Public Serving Route

## Epic
[E-172: Standalone Scouting Report Generator](epic.md)

## Status
`TODO`

## Description
After this story is complete, anyone with a report link can open it in a browser and see the scouting report — no login required. The route serves the self-contained HTML file generated by E-172-02, checks for expiration, and returns 404 for unknown or expired reports.

## Context
The generation pipeline (E-172-02) produces HTML files and DB rows. This story adds the public-facing route that makes those files accessible. The route must be outside the auth middleware so no login is required. Expiration is enforced at serve time (lazy check against `expires_at`).

## Acceptance Criteria
- [ ] **AC-1**: `GET /reports/<slug>` serves the report HTML file with `Content-Type: text/html; charset=utf-8`. The route does NOT require authentication — no session check, no login redirect.
- [ ] **AC-2**: Given a valid slug for a report with `status='ready'` and `expires_at` in the future, the route returns HTTP 200 with the HTML content.
- [ ] **AC-3**: Given a slug that does not exist in the `reports` table, the route returns HTTP 404.
- [ ] **AC-4**: Given a slug for a report where `expires_at` is in the past, the route returns HTTP 404. The response does not reveal that a report once existed at this URL.
- [ ] **AC-5**: Given a slug for a report with `status='generating'` or `status='failed'`, the route returns HTTP 404. Only `ready` reports are served.
- [ ] **AC-6**: The route is registered in the FastAPI app without auth middleware. It does not interfere with existing routes (no path conflicts with `/reports/` if such paths existed).
- [ ] **AC-7**: Tests verify: (a) 200 for valid ready report, (b) 404 for unknown slug, (c) 404 for expired report, (d) 404 for generating/failed status, (e) response Content-Type is text/html.

## Technical Approach
A new router module registers the public route. The route handler queries the `reports` table by slug, checks status and expiration, reads the HTML file from `data/<report_path>`, and returns an `HTMLResponse`. The router is included in the FastAPI app without the auth dependency that protects admin and dashboard routes. This follows the precedent of public endpoints in the app (if any exist) or simply includes the router at the app level without middleware.

## Notes
- The 404 responses for expired reports should NOT include a message like "this report has expired" — treat it the same as a nonexistent slug to avoid information leakage.
- Cache headers: `Cache-Control: public, max-age=3600` is reasonable since reports are immutable snapshots. No need for cache-busting.
- The route reads the HTML file from disk on each request. For the expected volume (a handful of reports), this is fine — no caching layer needed.
