# E-046-02: mitmproxy Upstream Mode

## Epic
[E-046: Upstream Proxy Support](epic.md)

## Status
`TODO`

## Description
After this story is complete, the mitmproxy Docker service will start in upstream proxy mode when `PROXY_ENABLED=true` and `PROXY_URL` is set. When proxy is disabled (or `PROXY_URL` is empty despite `PROXY_ENABLED=true`), mitmproxy runs in its current regular intercepting mode. The `proxy.sh` status command will display whether upstream mode is active.

## Context
The mitmproxy service in `docker-compose.yml` currently uses a fixed `mitmweb` command. Upstream mode is activated by adding `--mode upstream:URL` to the command. Since Docker Compose does not support conditional command arguments natively, a small entrypoint wrapper script will check the env vars at container startup and build the appropriate command line.

## Acceptance Criteria
- [ ] **AC-1**: When `PROXY_ENABLED=true` and `PROXY_URL=http://proxy:7777` are set in `.env`, `docker compose --profile proxy up` starts mitmproxy with `--mode upstream:http://proxy:7777` in its command line. Verify by inspecting `docker compose --profile proxy config` output or `docker inspect` on the running container.
- [ ] **AC-2**: When `PROXY_ENABLED` is unset, empty, or `false`, mitmproxy starts without `--mode` (current behavior -- regular intercepting proxy).
- [ ] **AC-3**: When `PROXY_ENABLED=true` but `PROXY_URL` is empty or unset, mitmproxy starts without `--mode` (graceful degradation, same as AC-2). The wrapper logs a warning to stderr.
- [ ] **AC-4**: The `PROXY_URL` value does not appear in `docker compose --profile proxy config` output (Docker Compose must not inline the URL into the command; it should remain an env var reference resolved at runtime by the wrapper).
- [ ] **AC-5**: `scripts/proxy.sh status` displays whether upstream proxy mode is active (e.g., "Upstream proxy: enabled" or "Upstream proxy: disabled").
- [ ] **AC-6**: `PROXY_ENABLED` and `PROXY_URL` are passed as environment variables to the mitmproxy service in `docker-compose.yml`.

## Technical Approach
1. Create `scripts/proxy-entrypoint.sh` -- a small bash wrapper that:
   - Checks `PROXY_ENABLED` and `PROXY_URL`
   - If both are set and valid, execs `mitmweb --web-host 0.0.0.0 --scripts /app/proxy/addons/loader.py --mode upstream:${PROXY_URL}`
   - If `PROXY_ENABLED=true` but `PROXY_URL` is empty, logs a warning to stderr and execs mitmweb without `--mode`
   - Otherwise, execs mitmweb without `--mode` (current behavior)
2. Update `docker-compose.yml` mitmproxy service:
   - Change `entrypoint` to `["/app/scripts/proxy-entrypoint.sh"]`
   - Remove the current `command` block (args move into the entrypoint script)
   - Add `PROXY_ENABLED` and `PROXY_URL` to the `environment` list (as `${PROXY_ENABLED:-}` and `${PROXY_URL:-}` to handle unset gracefully)
3. Update `scripts/proxy.sh` `cmd_status()` to check `PROXY_ENABLED` env var and display upstream proxy status.

## Notes
- The entrypoint wrapper approach is preferred over Docker Compose variable interpolation in `command` because Compose would inline the URL (containing credentials) into the resolved config, violating credential safety.
- No automated tests for this story -- it is Docker infrastructure verified by manual inspection of container config. AC verification uses `docker compose config` and `docker inspect`.
