## Description
After this story is complete, all endpoints from the "Proxy-Discovered Endpoints" sections of the monolith (2026-03-05 and 2026-03-07 batches) that are not already covered by E-062-02 or E-062-03 will exist as individual files under `docs/api/endpoints/`, following the format spec from E-062-R-01. These are endpoints observed in mitmproxy traffic with method, path, status code, and observational notes.

## Context
The monolith's "Proxy-Discovered Endpoints" sections contain endpoints observed passively during mitmproxy traffic capture. These range from simple method+path+status observations to endpoints with brief behavioral notes. Some proxy-discovered endpoints were later confirmed via curl and appear in the "Confirmed Endpoints" section -- the spike inventory assigns those to E-062-03 (not this story). This story handles endpoints that remain at the OBSERVED documentation tier.

The "Write Operations Catalog" section lists POST/PUT/PATCH/DELETE endpoints that are observed but explicitly out of scope for data ingestion. These still get individual files (we document what we see) but their frontmatter should reflect their status accurately.

## Acceptance Criteria
- [ ] **AC-1**: Every endpoint assigned to this story by the spike inventory (E-062-R-01) exists as an individual file in `docs/api/endpoints/`
- [ ] **AC-2**: Each file follows the YAML frontmatter schema and body structure from the format spec at `/workspaces/baseball-crawl/.project/research/E-062-format-spec.md`
- [ ] **AC-3**: File naming follows the convention from the format spec
- [ ] **AC-4**: The `status` field in frontmatter is set to `OBSERVED` for endpoints that have not been independently confirmed via curl
- [ ] **AC-5**: All information from the monolith is preserved for each endpoint -- discovery dates, status codes, observational notes, and any cross-references
- [ ] **AC-6**: Cross-references to global content use relative links to reference files in `docs/api/`
- [ ] **AC-7**: No endpoint file created in this story duplicates one assigned to E-062-02 or E-062-03 per the spike inventory

## Technical Approach
Read the format spec and endpoint inventory from E-062-R-01 (at `/workspaces/baseball-crawl/.project/research/E-062-format-spec.md`). The inventory assigns each endpoint to exactly one story. For each endpoint assigned to this story, extract its information from the monolith and write to `docs/api/endpoints/`.

These are typically the simplest files -- sparse frontmatter and a short body. The main risk is completeness: ensure every proxy-discovered endpoint in the monolith appears in the inventory and gets a file.

The monolith is not modified or deleted in this story -- that happens in E-062-04.
