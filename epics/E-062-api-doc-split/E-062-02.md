# E-062-02: Migrate Fully-Documented Endpoints

## Epic
[E-062: Split API Documentation into Per-Endpoint Files](epic.md)

## Status
`TODO`

## Description
After this story is complete, all fully-documented endpoints (~24 endpoints with complete headers, response schemas, query params, and behavioral notes) will exist as individual files under `docs/api/endpoints/`, following the format spec established by E-062-R-01.

## Context
The current monolith's "Endpoints" section contains the richest endpoint documentation -- complete header blocks, response schema references, query parameter tables, behavioral notes, and confirmation dates. These are the endpoints most frequently referenced by implementing agents (SE, DE) and are the highest-value targets for per-file access. They also have the most complex cross-reference patterns (schema references, "see also" links to other endpoints).

## Acceptance Criteria
- [ ] **AC-1**: Every endpoint from the monolith's main "Endpoints" section (before "Confirmed Endpoints" and "Proxy-Discovered" sections) exists as an individual file in `docs/api/endpoints/`
- [ ] **AC-2**: Each file follows the YAML frontmatter schema and body structure from the format spec at `/workspaces/baseball-crawl/.project/research/E-062-format-spec.md`
- [ ] **AC-3**: File naming follows the convention established in the format spec
- [ ] **AC-4**: All content from each endpoint's section in the monolith -- including schemas documented in the separate "Response Schemas" section that belong to these endpoints (as flagged in the spike inventory) -- appears in the corresponding file with zero information loss (headers, schemas, query params, behavioral notes, confirmation dates, response examples, caveats)
- [ ] **AC-5**: Cross-references to global content (authentication, pagination, content-type) use relative links to the reference files in `docs/api/`
- [ ] **AC-6**: Cross-references between endpoints use relative links to other endpoint files
- [ ] **AC-7**: Web vs. mobile profile information, where present, follows the profile convention from the format spec
- [ ] **AC-8**: The endpoint inventory from E-062-R-01 is used as the migration checklist -- every endpoint marked as "fully documented" in the inventory has a corresponding file

## Technical Approach
Read the format spec from E-062-R-01 (at `/workspaces/baseball-crawl/.project/research/E-062-format-spec.md`) and the endpoint inventory for the list of fully-documented endpoints and their proposed filenames. For each endpoint, extract its complete section from `docs/gamechanger-api.md`, convert to the per-file format, and write to `docs/api/endpoints/`. The prototype files from the spike (at `/workspaces/baseball-crawl/.project/research/E-062-endpoint-prototypes/`) serve as reference examples.

The monolith is not modified or deleted in this story -- that happens in E-062-04.

## Notes
Can execute in parallel with E-062-01 (reference files), E-062-03 (confirmed endpoints), and E-062-06 (proxy-discovered endpoints) since they create different files. The boundary between stories is defined by the endpoint inventory from E-062-R-01, which assigns each endpoint to exactly one story.
