# E-148-03: Atomic Session Commits for Plan and Implement Skills

## Epic
[E-148: Review Cycle Reordering for Plan and Implement Skills](epic.md)

## Description
After this story is complete, both the plan and implement skills will have explicit commit steps that capture ALL session artifacts -- not just primary outputs (epic files or implementation code) but also agent memory updates, vision signals, research artifacts, and idea captures. The plan skill's Phase 5 gains a new Step 2a (atomic planning commit). The implement skill's Phase 5 gains a new Step 7a (ancillary file sweep) that clears main-checkout artifacts before the existing closure merge.

## Context
Currently, planning workflows commit their primary artifacts but leave ancillary files uncommitted. Evidence from E-147 and E-148 planning sessions: after committing epic files, these remained uncommitted:

```
 M .claude/agent-memory/claude-architect/MEMORY.md
 M .claude/agent-memory/product-manager/MEMORY.md
 M .claude/agent-memory/product-manager/archived-epics.md
 M docs/vision-signals.md
?? .claude/agent-memory/claude-architect/review-cycle-reordering.md
```

All were produced during the same planning sessions and should have been part of the respective commits. The plan skill has no commit step at all -- the main session commits ad-hoc after the skill completes. For dispatch, the implement skill's closure merge captures the worktree patch (which includes agent memory writes made by dispatch agents in the worktree), but any main-checkout changes (vision signals captured by the main session, leftover planning artifacts) are not included. The implement skill's clean-tree preflight (Phase 5 Step 8) would block closure if uncommitted main-checkout files exist.

The architect designed the atomic commit approach (insertion points, commit conventions, artifact classification) via live team consultation during E-148 expansion.

## Technical Approach
Two insertion points in existing workflows, each adding a commit step at the boundary where session artifacts accumulate. The plan skill's Step 2a is the primary capture mechanism -- it commits everything produced during planning (epic files, agent memory, vision signals, research, ideas). The implement skill's Step 7a is a safety net that sweeps any main-checkout changes that accumulated outside the epic worktree (vision signals captured by the main session, leftover artifacts from prior sessions). During dispatch, agents write to the epic worktree (including agent memory paths), so those changes are captured by the existing closure merge (Step 8 worktree patch) -- Step 7a does not need to capture them. After Step 7a clears any main-checkout artifacts, the existing clean-tree preflight in Step 8 serves as a final safety net (should always pass for normal workflows). No existing steps are modified, resequenced, or removed. Reference TN-8 for recognized artifact categories and TN-10 for commit message conventions.

## Notes
- The plan skill commit step is the main session's responsibility (like all git operations per dispatch-pattern.md). The skill specifies WHAT to commit; the main session executes the git commands.
- The `feat` prefix for plan commits is intentional: creating the epic IS the feature deliverable of the planning workflow. The `chore` prefix for session artifacts during dispatch is correct because they're housekeeping alongside the real deliverable (the implementation code).
- The clean-tree preflight in Step 8 stays unchanged. Step 7a clears recognized ancillary files before the preflight runs, so the preflight functions as a safety net rather than needing modification.
- The archive commit (Step 11) remains separate and unchanged -- `git mv` for archive is a distinct logical operation.
- During dispatch, agents work in the epic worktree and write agent memory to worktree paths. Those writes are captured by the closure merge patch (Step 8), not by Step 7a. Step 7a only sweeps main-checkout changes (vision signals from the main session, leftover planning artifacts).
- PM's memory update at Step 10 (after closure merge) is also in the main checkout -- it gets captured by the archive commit (Step 11) along with the archive move, which is the correct grouping.
