# E-162-01: Merge gc_uuid onto Existing public_id Stub in OpponentResolver

## Epic
[E-162: Fix OpponentResolver Duplicate Team Creation](epic.md)

## Status
`TODO`

## Description
After this story is complete, `OpponentResolver._ensure_opponent_team_row` will check for an existing team row by `public_id` before attempting to create a new row, preventing duplicate team creation when a stub team already owns the target `public_id`. When a match is found, the resolver merges `gc_uuid` onto the existing row instead of creating a new one.

## Context
The current code does `INSERT OR IGNORE INTO teams WHERE gc_uuid=?` without checking if the API-returned `public_id` already belongs to an existing team. This creates orphaned duplicate rows. The fix adds a `public_id` lookup step per the lookup order in Technical Notes TN-1. This was observed in production on 2026-03-26 for Bennington (team 280 vs team 454).

## Acceptance Criteria
- [ ] **AC-1**: Given a team row exists with `public_id=X` and `gc_uuid=NULL`, when `_ensure_opponent_team_row` is called with a new `gc_uuid` and the API returns `public_id=X`, then `gc_uuid` is written to the existing row and no new team row is created. The returned `(team_id, effective_public_id)` tuple uses the existing row's ID and `public_id`.
- [ ] **AC-2**: Given a team row exists with `public_id=X` and `gc_uuid=NULL`, and a *different* team row exists with `gc_uuid=Y`, when `_ensure_opponent_team_row` is called with `gc_uuid=Y` and `public_id=X`, then the gc_uuid write is skipped (UNIQUE collision), a WARNING is logged, and the existing `public_id` row's ID is returned without creating a new row, per TN-2.
- [ ] **AC-3**: Given a team row exists with matching `gc_uuid`, the gc_uuid-match path returns the existing row and applies the same field update rules as today: name updated only if existing name equals the gc_uuid string (UUID-as-name stub), real names preserved; season_year written only when NULL; public_id written via `_write_public_id` only when NULL (with collision check).
- [ ] **AC-4**: Given no team row exists with matching `gc_uuid` or `public_id`, a new row is created (existing INSERT behavior preserved).
- [ ] **AC-5**: Field updates during merge follow the rules in TN-3: gc_uuid only-write-when-NULL with collision check, name updated only if UUID-stub, season_year only-write-when-NULL.
- [ ] **AC-6**: Existing tests continue to pass. New tests cover TN-4 scenarios 1 (happy path merge) and 2 (gc_uuid collision on merge). Scenarios 3, 4, and 5 are already covered by existing tests (lines 364/386, 1139/1192, and 302 respectively in `tests/test_crawlers/test_opponent_resolver.py`) -- verify they still pass, do not duplicate them.

## Technical Approach
Refactor `_ensure_opponent_team_row` in `src/gamechanger/crawlers/opponent_resolver.py` to implement the three-step lookup order described in TN-1. The `_write_public_id` helper already handles public_id collision checking; create a symmetric `_write_gc_uuid` helper for gc_uuid collision checking during the merge step.

Read the existing tests at `tests/test_crawlers/test_opponent_resolver.py` to understand the current coverage. Add new tests for TN-4 scenarios 1 and 2 (the new merge paths). Scenarios 3, 4, and 5 are already covered by existing tests -- verify they still pass.
