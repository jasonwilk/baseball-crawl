## Description
After this story is complete, the implement skill's CR assignment template will include structured context fields (API endpoints, migration files, behavioral changes) so the code-reviewer knows where to cross-reference. The implementer completion report template will include a "Behavioral Changes" section that declares which functions changed signature or behavior, giving CR the signal to trigger caller audit.

## Context
The retro found that CR missed findings partly because it lacked domain context in its assignment. CR received the story file and epic TNs but not which API endpoints were involved or which migrations exist. Meanwhile, the caller audit checklist item (added in E-149-01) needs a trigger signal -- CR can discover behavioral changes by reading code, but an explicit declaration from the implementer makes it systematic. See epic TN-4/4a/4b for the full design.

## Acceptance Criteria
- [ ] **AC-1**: The CR per-story assignment template in the implement skill (Phase 3 Step 5, the `Review story E-NNN-SS` code block) includes an **API endpoints** field listing relevant `docs/api/endpoints/*.md` files when the story touches GameChanger API calls. The field is omitted when the story does not touch API calls.
- [ ] **AC-2**: The CR per-story assignment template includes a **Migration files** field listing `migrations/*.sql` files when the story touches database code. The field is omitted when the story does not touch database code.
- [ ] **AC-3**: The CR per-story assignment template includes a **Behavioral changes** field containing the implementer's declared behavioral changes (from AC-5). The field is omitted when the implementer declares "None."
- [ ] **AC-4**: The main session derives the API endpoints and migration files fields from the story's "Files to Create or Modify" section, the implementer's Files Changed list, and the story's description/Technical Approach (for selecting specific endpoint docs). The derivation heuristics are documented in the skill per TN-4a (documented heuristics, not ad-hoc judgment).
- [ ] **AC-5**: The implementer completion report format in the implement skill's Phase 3 Step 4 spawn context (the `**Completion**` paragraph at line ~214 where `## Files Changed` and `## Test Results` are defined) includes a **Behavioral Changes** section per TN-4b. The section is always present in completion reports (with "None" when no behavioral changes occurred).
- [ ] **AC-6**: The Behavioral Changes section explicitly states it is a **supplement** to CR's own caller audit (Story 01), not a replacement. CR still independently scans the diff for non-obvious behavioral changes.
- [ ] **AC-7**: The Round 2 assignment template (when routing findings back for re-review) includes the same structured context fields as Round 1, plus the implementer's updated behavioral changes declaration and updated Files Changed list. The assignment distinguishes which files are new in the remediation vs. carried forward from Round 1, so CR can focus the remediation regression guard on the new/changed files.
- [ ] **AC-8**: Existing assignment template fields (story file text, epic TNs, Files Changed, test results, review round) are unchanged.
- [ ] **AC-9**: The integration review assignment template (Phase 4a Step 3) is NOT modified by this story. Integration review scope is different from per-story review.
