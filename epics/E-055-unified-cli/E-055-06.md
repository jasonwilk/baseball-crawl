## Description
After this story is complete, the operator can run `bb status` to see a single-screen overview of the entire system's health: per-profile credential validity, last crawl timestamp, database info, and latest proxy session summary. This is the "operator dashboard" -- the first command Jason runs each morning to decide what needs attention.

## Context
Currently, checking system health requires running multiple commands: `python scripts/check_credentials.py`, manually checking crawl timestamps, etc. `bb status` consolidates all of this into one glanceable output with color-coded health indicators.

Proxy lifecycle detection (whether the proxy is running) is excluded because the proxy runs on the Mac host, not inside the devcontainer. Instead, status shows the latest proxy session info if E-052 session data is readable from the mounted `proxy/data/` volume.

## Acceptance Criteria
- [ ] **AC-1**: `src/cli/status.py` exists and defines a status command registered on the main app as `bb status`.
- [ ] **AC-2**: `bb status` displays per-profile credential health. For each profile (web, mobile), shows "valid (logged in as NAME)" in green, or "expired" / "missing" in red. Uses the `check_credentials()` function from `scripts/check_credentials.py` with the `--profile` flag per E-053.
- [ ] **AC-3**: `bb status` displays last crawl info: "Last crawl: YYYY-MM-DD HH:MM:SS (N files)" read from the most recent `data/raw/*/manifest.json`. If no manifest exists, shows "Last crawl: never" in yellow.

- [ ] **AC-5**: `bb status` displays latest proxy session info if available: session ID, timestamp, and reviewed status. Reads from E-052's session directory structure under `proxy/data/`. If no sessions exist or `proxy/data/` is not mounted, shows "Proxy sessions: none" in yellow.
- [ ] **AC-6**: Output uses color (via `rich`) when stdout is a terminal, and plain text when piped.
- [ ] **AC-7**: `bb status` exits with code 0 if all credentials are valid, 1 if any critical issue (expired/missing creds, no database).
- [ ] **AC-8**: Tests verify output formatting for each health state (valid creds per profile, expired creds, missing creds, manifest present, manifest absent, DB present, DB absent, session present, session absent). Mock all external calls.

## Technical Approach
1. Create `src/cli/status.py` with the status check logic.
2. Credential check: import and call `check_credentials()` from `scripts/check_credentials.py` for each profile (web, mobile) per E-053's profile-aware interface.
3. Last crawl: scan `data/raw/*/manifest.json` for the most recent `crawled_at` timestamp.
4. Database: check `data/app.db` existence and size via `pathlib.Path.stat()`.
5. Latest proxy session: scan `proxy/data/sessions/` (E-052 structure) for the most recent session directory. Read session metadata (timestamp, reviewed status) if available. Gracefully handle missing directory.
6. Use `rich.console.Console` for colored output. Use `Console(force_terminal=False)` to auto-detect terminal.
7. The mount in `src/cli/__init__.py` is pre-wired by E-055-01 -- do not modify `__init__.py`. Note: status is registered as a top-level command (not a subgroup), so the stub and mount in E-055-01 should use `@app.command()` or `app.add_typer()` accordingly.

## Notes
- Proxy lifecycle detection is intentionally excluded. The proxy runs on the Mac host; the CLI runs in the devcontainer. Checking whether the proxy Docker container is running is not possible from inside the container. Instead, we show the latest proxy session data from the mounted volume.
- Human-readable file size: use a simple helper (e.g., KB/MB/GB formatting) rather than adding a dependency.
- Example output:
  ```
  baseball-crawl status

    Credentials (web):     valid (logged in as Jason Smith)
    Credentials (mobile):  expired
    Last crawl:            2026-03-05 14:30:00 (47 files)
    Database:              data/app.db (2.4 MB)
    Proxy sessions:        3 total, 1 unreviewed (latest: 2026-03-05-1430)
  ```
