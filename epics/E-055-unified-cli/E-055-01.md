## Description
After this story is complete, the project will have a `src/cli/` package with a Typer-based main app, a registered `bb` entry point in `pyproject.toml`, and `typer[all]` added to `requirements.txt`. Running `bb --help` will display the top-level help with placeholders for command groups. The devcontainer will install the package in editable mode (`pip install -e .`) so `bb` is available on PATH and code changes are live immediately. The CLI is devcontainer-only -- it is NOT installed in the production Docker image (the production container runs the API server only). A `__main__.py` fallback enables `python -m src.cli` as an alternative invocation.

## Context
This is the foundation story for the unified CLI. All other stories depend on it because they add command groups to the app created here. The story is deliberately minimal: create the package, register the entry point, verify it works.

## Technical Approach
1. Add `typer[all]` to `requirements.txt`.
2. Create `src/cli/__init__.py` with a Typer app that **pre-wires all sub-app mounts**. This is critical for parallelism: stories 02-05 create their respective modules but do NOT modify `__init__.py`. The init file imports and mounts all sub-apps upfront. Use conditional imports so the app still loads even if a sub-module does not yet exist:
   ```python
   import typer
   app = typer.Typer(help="baseball-crawl operator CLI.")

   # Sub-apps are mounted here. Each module is created by a separate story.
   # Conditional imports allow the skeleton to work before all modules exist.
   try:
       from src.cli.creds import app as creds_app
       app.add_typer(creds_app, name="creds")
   except ImportError:
       pass

   # ... same pattern for data, proxy, db
   # status is a top-level command, not a sub-app:
   try:
       from src.cli.status import status
       app.command()(status)
   except ImportError:
       pass
   ```
3. Create `src/cli/__main__.py` for fallback invocation via `python -m src.cli`:
   ```python
   from src.cli import app
   app()
   ```
4. Also create stub files for each sub-module (`src/cli/creds.py`, `src/cli/data.py`, `src/cli/proxy.py`, `src/cli/db.py`, `src/cli/status.py`) containing just the empty Typer sub-app with help text. Stories 02-06 will overwrite these stubs with full implementations.
5. Add `version = "0.1.0"` and `[project.scripts]` to `pyproject.toml`. The `version` field is **required** -- `pip install -e .` will fail without it. Reference `/workspaces/baseball-crawl/pyproject.toml` for current state (has `name` and `requires-python` but no `version`).
   ```toml
   [project]
   name = "baseball-crawl"
   version = "0.1.0"
   requires-python = ">=3.13"

   [project.scripts]
   bb = "src.cli:app"
   ```
6. [REMOVED -- CLI is not installed in the production Docker image.]
7. Update `.devcontainer/devcontainer.json` `postCreateCommand`: append `&& pip install -e .` after the existing `pip install -r requirements.txt` step. No PATH changes needed -- `/home/vscode/.local/bin` is already on PATH via `remoteEnv`.
8. Write a test in `tests/test_cli.py` using `CliRunner`.

## Notes
- Typer's default behavior when invoked with no arguments on an app that has subcommands is to show help. This is the desired behavior.
- The `[all]` extra for typer includes `rich` (colored output) and `shellingham` (shell detection for completions).
- **Gotcha -- `version` field**: `pip install -e .` requires a `version` in `pyproject.toml`. The current file lacks one. Without it, the install will fail with a metadata error.
- **Devcontainer-only**: The CLI is installed only in the devcontainer via `pip install -e .` (editable -- live code changes). The production Docker image does NOT install the CLI package -- it runs `uvicorn` only.
- **`requirements.txt` remains the dependency source**: `pip install -e .` does NOT install dependencies from `requirements.txt`. The two commands are complementary. Do not move deps into `pyproject.toml` `[project.dependencies]` -- that is a larger migration out of scope.
- **Logging initialization order**: Every `scripts/*.py` file calls `logging.basicConfig()` at module level (outside any `if __name__` guard). `basicConfig()` is first-caller-wins -- once configured, subsequent calls are no-ops. If the CLI imports `scripts.check_credentials` before `scripts.crawl`, the WARNING level from `check_credentials.py` wins and silently drops all INFO messages from crawl/load/bootstrap. **The fix**: `src/cli/__init__.py` must call `logging.basicConfig(level=logging.INFO)` (or the desired root level) BEFORE any `from scripts.*` import occurs. This ensures the CLI's logging config wins regardless of import order in downstream command modules.
