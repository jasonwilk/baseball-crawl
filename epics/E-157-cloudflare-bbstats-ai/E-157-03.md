## Description
After this story is complete, the project's context layer (CLAUDE.md and related files) will reflect bbstats.ai as the canonical production domain, the current auth model (app-internal auth, CF Access passive), and any deployment-related context that agents need when working on this codebase.

## Context
CLAUDE.md's Deployment Target section currently says "Cloudflare Tunnel (no exposed ports); Zero Trust Access for auth" -- which implies CF Access is enforcing authentication. With the bbstats.ai deployment, CF Access is present but not blocking; the app handles auth via magic links and passkeys (E-023). Agents working on the codebase need accurate context about how production auth works so they don't write code that assumes CF Access headers or policies are in place.

## Acceptance Criteria
- [ ] **AC-1**: CLAUDE.md Deployment Target section includes `https://bbstats.ai` as the canonical production URL and accurately describes the auth model: Cloudflare Tunnel for network ingress, app-internal auth via magic links/passkeys, CF Access present but not enforcing.
- [ ] **AC-2**: Any references to "Zero Trust Access for auth" in CLAUDE.md are updated to reflect the passive Access model. The distinction between network-level (Cloudflare Tunnel) and application-level (magic links/passkeys) authentication is clear.
- [ ] **AC-3**: The `.env.example` reference in CLAUDE.md (if any) or the Deployment Target section notes that `CF_ACCESS_CLIENT_ID` / `CF_ACCESS_CLIENT_SECRET` are not required for the current bbstats.ai deployment.
- [ ] **AC-4**: No other context-layer files (rules, skills, agent definitions, agent memory) contain stale references to the pre-bbstats.ai deployment model that would mislead agents. Specifically, `.claude/agent-memory/product-manager/MEMORY.md` has two stale deployment references: "Cloudflare Tunnel + Zero Trust" in the tech stack line and "Cloudflare Access for /admin/*" in the auth model section -- both must be updated to reflect the passive CF Access model.
