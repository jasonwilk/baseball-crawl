# E-042-03: Admin team list and add-team page

## Epic
[E-042: Admin Interface and Team Management](epic.md)

## Status
`TODO`

## Description
After this story is complete, admins will see a team management page at `GET /admin/teams` showing all teams in the database, with a form to add new teams by pasting a GameChanger URL. The form resolves the URL to a team profile, creates the team record, and redirects back to the team list with a success message.

## Context
This is the primary user-facing story of the epic -- the "paste a URL, get a team" flow. It builds on the URL parser and team resolver from E-042-02 and the schema from E-042-01. The page follows the same patterns as the existing `/admin/users` page (E-023): admin guard, Jinja2 template, flash messages, form handling.

## Acceptance Criteria
### Team List Page (`GET /admin/teams`)
- [ ] **AC-1**: The page renders two sections: "Lincoln Program" (is_owned=1 teams) at top, "Tracked Opponents" (is_owned=0 teams) below. Each section is a separate table with columns: Name, Level, Status (Active/Inactive), Public ID, Last Synced.
- [ ] **AC-2**: If either section has no teams, show a placeholder message (e.g., "No opponents tracked yet -- use Discover Opponents on a Lincoln team").
- [ ] **AC-3**: "Status" column shows "Active" (green text) for `is_active=1` and "Inactive" (gray text) for `is_active=0`.
- [ ] **AC-4**: Each team row has an "Edit" link pointing to `/admin/teams/{team_id}/edit`.
- [ ] **AC-5**: The page is protected by `_require_admin()` -- unauthenticated users are redirected to `/auth/login`, non-admin users see 403.
- [ ] **AC-6**: The page shows a flash message when `?msg=` or `?error=` query params are present (same pattern as `/admin/users`).

### Add Team Form (on the same page)
- [ ] **AC-7**: Below the team list, a form with fields: "GameChanger URL or Public ID" (text input, required), "Level" (select: freshman/jv/varsity/reserve/legion/other, optional), "Type" (radio: Owned/Tracked, default Tracked).
- [ ] **AC-8**: The form POSTs to `POST /admin/teams`.

### Add Team Handler (`POST /admin/teams`)
- [ ] **AC-9**: The handler parses the URL input using `parse_team_url()`, resolves the team via `resolve_team()`, and inserts a row into the `teams` table with: `team_id = public_id`, `name` from API response, `public_id` from parser, `level` from form, `is_owned` from form (1 for Owned, 0 for Tracked), `source = 'gamechanger'`, `is_active = 1`.
- [ ] **AC-10**: On success, redirects to `/admin/teams?msg=Team+added:+{team_name}+({city},+{state})` so the admin can verify the resolution without a preview step. If city/state are unavailable, just show the team name.
- [ ] **AC-11**: If `parse_team_url()` raises `ValueError`, re-renders the form with an error message showing the parsing error.
- [ ] **AC-12**: If `resolve_team()` raises `TeamNotFoundError`, re-renders with error "Team not found on GameChanger. Check the URL and try again."
- [ ] **AC-13**: If `resolve_team()` raises `GameChangerAPIError`, re-renders with error "Could not reach GameChanger API. Try again later."
- [ ] **AC-14**: If a team with the same `team_id` (public_id) already exists, re-renders with error "This team is already in the system."
- [ ] **AC-15**: The handler is protected by `_require_admin()`.

### Navigation
- [ ] **AC-16**: A horizontal admin sub-nav bar appears at the top of both `/admin/users` and `/admin/teams` pages with links to "Users" and "Teams". The current page link is visually distinguished (bold or underlined). This sub-nav is added to `admin/users.html` and the new `admin/teams.html`.

### Tests
- [ ] **AC-17**: Tests in `tests/test_admin_teams.py` verify: GET /admin/teams returns 200 for admin, 403 for non-admin, 302 for unauthenticated. POST /admin/teams with valid URL creates team and redirects. POST with invalid URL shows error. POST with duplicate team shows error. Team list displays teams correctly.
- [ ] **AC-18**: Tests mock `resolve_team()` to avoid real HTTP calls.

## Technical Approach
- Add team management routes to `src/api/routes/admin.py` (extend existing file, append new routes at end).
- Add team query functions to `src/api/db.py`: `get_all_teams()`, `create_team()`, `team_exists()`.
- Create `src/api/templates/admin/teams.html` following the same layout as `admin/users.html`.
- Mock `resolve_team` in tests using `unittest.mock.patch`.
- The team resolver call happens synchronously inside `run_in_threadpool` (same pattern as existing admin DB calls).

## Notes
- The decision to use `public_id` as `team_id` for URL-added teams means these teams will have a short alphanumeric ID (e.g., `"a1GFM9Ku0BbF"`) instead of a UUID. This is fine -- `team_id` is `TEXT` in the schema, not UUID-typed.
- Teams added via the old authenticated flow have UUID `team_id` values. Teams added via URL have `public_id` values. Both coexist in the same table.
