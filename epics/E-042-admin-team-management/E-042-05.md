## Description
After this story is complete, an admin can trigger opponent discovery for any team with a `public_id`. The system fetches the team's public game schedule, extracts unique opponent names, and creates placeholder team records for any opponents not already in the database. Discovered opponents default to `is_active=0` and have no `public_id` -- the admin can later paste an opponent's GameChanger URL (via the existing add-team flow) to fully onboard it.

## Context
When a team is added via URL, its public schedule reveals opponent names via `GET /public/teams/{public_id}/games`. Each game record includes `opponent_team.name` but does NOT include any opponent identifier (no `public_id`, no UUID). This is a confirmed API limitation (see `docs/gamechanger-api.md` Known Limitations). This story uses opponent names to create placeholder team records, giving the admin visibility into the full opponent landscape. To fully onboard an opponent (enable crawling, access public API data), the admin pastes the opponent's GameChanger URL separately.

## Technical Approach
- The `GET /public/teams/{public_id}/games` response is a bare JSON array. Each element has `opponent_team.name` (always present) and optionally `opponent_team.avatar_url`. No opponent identifier (UUID or public_id) is available from this endpoint.
- Use `src/http/session.py`'s `create_session()` for HTTP (same as team resolver).
- No auth headers -- this is a public endpoint.
- The discovery function returns data; the admin route handles DB insertion. This keeps the resolver module DB-free.
- Name matching for deduplication: normalize to lowercase for comparison. This handles "Lincoln East 15U" vs "lincoln east 15u".
- Generated `team_id` slugs ensure discovered opponents get unique primary keys without needing a `public_id`. Example: `"lincoln-east-15u-a3f2"`.
- Use `INSERT OR IGNORE` in the DB to handle any race conditions.

## Notes
- **API limitation**: The public games endpoint does NOT expose opponent identifiers (no `public_id`, no UUID). Only opponent names are available. This was confirmed via api-scout consultation during the codex spec review triage (2026-03-06). See `docs/gamechanger-api.md` Known Limitations for `/public/teams/{public_id}/games`.
- Discovered opponents default to `is_active=0` with `public_id=NULL` and `source='discovered'`. They are name-only placeholders. The admin must paste the opponent's GameChanger URL (via the add-team form from E-042-03) to resolve the full profile and assign a `public_id`. Only then can the opponent be activated for crawling.
- Name-based matching is imperfect (team names can vary across seasons or have typos). This is acceptable for a discovery aid -- the admin reviews and curates the list.
- **Future enhancement**: If authenticated crawling is available, `GET /teams/{team_id}/schedule` -> `pregame_data.opponent_id` (UUID) -> `GET /teams/{opponent_uuid}/public-team-profile-id` -> `public_id` would enable full identifier-based discovery. This is a separate epic when the bridge endpoint is verified for opponent UUIDs.
