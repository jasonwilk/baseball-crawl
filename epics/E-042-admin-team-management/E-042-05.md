## Description
After this story is complete, an admin can trigger opponent discovery for any team with a `public_id`. The system fetches the team's public game schedule, extracts unique opponents, and creates team records for any opponents not already in the database. Discovered opponents default to `is_active=0` (not crawled) so the admin explicitly enables the ones they want to track.

## Context
When a team is added via URL, its public schedule reveals all opponents via `GET /public/teams/{public_id}/games`. Each game record includes `opponent_team.public_team_profile_id` -- the opponent's `public_id`. This story uses that data to populate the database with opponent teams, saving the admin from manually pasting each opponent's URL.

## Acceptance Criteria
### Opponent Discovery Function
- [ ] **AC-1**: A function `discover_opponents(public_id: str) -> list[DiscoveredOpponent]` in `src/gamechanger/team_resolver.py` calls `GET /public/teams/{public_id}/games` and extracts unique opponents.
- [ ] **AC-2**: `DiscoveredOpponent` is a dataclass with fields: `public_id: str`, `name: str`.
- [ ] **AC-3**: Opponents are deduplicated by `public_team_profile_id` (each opponent appears once regardless of how many games).
- [ ] **AC-4**: Opponents whose `public_team_profile_id` is `None` or empty are skipped (some games may not have opponent profile links).
- [ ] **AC-5**: The function uses the correct Accept header: `application/vnd.gc.com.public_team_schedule_event:list+json; version=0.0.0` (from `docs/gamechanger-api.md`).
- [ ] **AC-6**: The function raises `GameChangerAPIError` for non-200 responses.

### Admin Trigger
- [ ] **AC-7**: A "Discover Opponents" button appears on the team list page (E-042-03) for each team that has a `public_id` and `is_active=1`.
- [ ] **AC-8**: `POST /admin/teams/{team_id}/discover-opponents` calls `discover_opponents()` with the team's `public_id`.
- [ ] **AC-9**: For each discovered opponent not already in the `teams` table, inserts a row with: `team_id = opponent.public_id`, `name = opponent.name`, `public_id = opponent.public_id`, `is_owned = 0`, `is_active = 0`, `source = 'gamechanger'`.
- [ ] **AC-10**: Opponents already in the database (matched by `team_id`) are skipped (no duplicates, no updates).
- [ ] **AC-11**: Redirects to `/admin/teams?msg=Discovered+N+new+opponents+for+{team_name}` with the count of newly added opponents.
- [ ] **AC-12**: If the team has no `public_id`, redirects with error "Cannot discover opponents: team has no public ID."
- [ ] **AC-13**: Protected by `_require_admin()`.

### Tests
- [ ] **AC-14**: Tests in `tests/test_team_resolver.py` (extending from E-042-02) verify `discover_opponents()` with mocked HTTP: correct parsing, deduplication, skipping null profile IDs, error handling.
- [ ] **AC-15**: Tests in `tests/test_admin_teams.py` verify the admin endpoint: creates new opponents, skips existing, shows correct count, auth guard.

## Technical Approach
- The `GET /public/teams/{public_id}/games` response is a bare JSON array. Each element has `opponent_team.name` and `opponent_team.public_team_profile_id`.
- Use `src/http/session.py`'s `create_session()` for HTTP (same as team resolver).
- No auth headers -- this is a public endpoint.
- The discovery function returns data; the admin route handles DB insertion. This keeps the resolver module DB-free.
- Use `INSERT OR IGNORE` in the DB to handle the race condition where an opponent was added between the check and insert.

## Notes
- Discovered opponents default to `is_active=0`. The admin must explicitly activate opponents they want to crawl (via E-042-04's toggle). This prevents accidentally crawling dozens of opponents.
- The opponent `name` comes from the schedule response (`opponent_team.name`), not from a separate profile fetch. This avoids N+1 API calls. If the admin wants richer metadata, they can paste the opponent's URL to re-add it (which will resolve the full profile).
