# E-163-02: Scouting Spray Chart Loader

## Epic
[E-163: Full-Season Opponent Spray Charts and Dashboard Display](epic.md)

## Status
`TODO`

## Description
After this story is complete, scouting spray chart JSON files will be loaded into the `spray_charts` table, giving opponent players full-season spray coverage. The loader reads files from the scouting spray directory and inserts BIP events with correct team and player resolution.

## Context
E-163-01 produces raw spray JSON at `data/raw/{season_id}/scouting/{public_id}/spray/`. This story loads that data into the DB. The existing `SprayChartLoader` handles member team spray data using `gc_uuid` path resolution. Scouting spray files use `public_id` in the path instead, so team resolution differs.

## Acceptance Criteria
- [ ] **AC-1**: Given scouting spray JSON files exist for an opponent at `data/raw/{season_id}/scouting/{public_id}/spray/`, when the loader runs, then offensive and defensive BIP events are inserted into `spray_charts` with correct `game_id`, `player_id`, `team_id`, `chart_type`, coordinates, and `season_id`, per TN-2.
- [ ] **AC-2**: Team resolution uses `public_id` → `teams.id` lookup (not `gc_uuid` → `teams.id` as in the member loader), per TN-2.
- [ ] **AC-3**: Idempotency: re-loading the same files produces zero new inserts (`INSERT OR IGNORE` on `event_gc_id`).
- [ ] **AC-4**: Unknown player UUIDs get a stub row before the spray row (FK-safe orphan handling, same as existing `SprayChartLoader`).
- [ ] **AC-5**: Null `spray_chart_data` games are skipped gracefully with an INFO log.
- [ ] **AC-6**: The loader is integrated into the `bb data scout` CLI flow (runs after scouting spray crawl, per TN-9) and is independently callable via `bb data load --loader scouting-spray`.
- [ ] **AC-7**: Given a spray event for a game not present in the `games` table, the loader skips that event (logs at DEBUG level) rather than inserting a stub or failing. **Note**: In the normal `bb data scout` flow, the scouting pipeline loads boxscores for all completed games before the spray loader runs, so game rows should exist. This skip is defensive for two edge cases: (1) `bb data load --loader scouting-spray` runs independently before the main scouting load, (2) a specific boxscore fetch failed during the scouting crawl.

## Technical Approach
Create a new `ScoutingSprayChartLoader` class — the existing `SprayChartLoader.load_dir()` cannot be reused because it infers `gc_uuid` from the directory path (`spray_dir.parent.name`), but scouting paths use `public_id`. Adapt the insertion logic patterns from `SprayChartLoader` with scouting-specific resolution. Key differences:
- Path is `scouting/{public_id}/spray/` instead of `teams/{gc_uuid}/spray/`
- Team resolution via `SELECT id FROM teams WHERE public_id = ?` instead of `WHERE gc_uuid = ?`
- `season_id` derivation: extract from the directory path (`data/raw/{season_id}/scouting/...`) — the `{season_id}` component is already in the path structure (per TN-2)
- Game rows should exist after a normal scouting pipeline run; skip spray events for games not in the DB as a defensive fallback (log at DEBUG level)

Reference: `src/gamechanger/loaders/spray_chart_loader.py` for the insertion logic.

## Notes
- In the normal `bb data scout` flow, the scouting pipeline loads boxscores (and upserts game rows) for all completed opponent games before the spray loader runs, so game rows should exist for the full schedule. The AC-7 skip is defensive for two edge cases: (1) `bb data load --loader scouting-spray` runs independently before the main scouting load, (2) a specific boxscore fetch failed during the scouting crawl. It is not the expected path for opponent-vs-third-party games.
- The player-stats endpoint returns a full response; the loader extracts the `spray_chart_data` array from it. When `spray_chart_data` is null (scorekeeper did not record), the file still exists on disk (written by crawler for idempotency) but the loader skips it per AC-5.
