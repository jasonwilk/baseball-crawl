# E-071-02: Integrate review loop into dispatch workflow

## Epic
[E-071: Code Review Agent](epic.md)

## Status

## Description
After this story is complete, the dispatch workflow will automatically route every implementer's completed work through the code-reviewer agent before marking stories DONE. The implement skill spawns a persistent code-reviewer at team creation, Phase 3 Step 5 sends completed work to the reviewer instead of rubber-stamping ACs, and a review loop operates until the reviewer approves or the circuit breaker triggers. dispatch-pattern.md documents the review step, and CLAUDE.md lists code-reviewer in the Agent Ecosystem.

## Context
E-071-01 creates the code-reviewer agent definition. This story integrates it into the dispatch workflow so the review happens automatically during every epic dispatch. The current flow (implementer completes -> main session rubber-stamps ACs -> DONE) is replaced with (implementer completes -> reviewer examines -> findings loop -> reviewer approves -> DONE). See epic Technical Notes for the full CA assessment on flow insertion, persistence model, and circuit breaker.

## Acceptance Criteria

- [ ] **AC-2**: The reviewer is NOT listed in the epic's Dispatch Team section. The implement skill spawns it automatically -- it is infrastructure, not a story-specific implementer.

### implement SKILL.md -- Phase 3 Step 5 (review loop replaces rubber stamp)
- [ ] **AC-3**: Phase 3 Step 5 ("Monitor and verify") is rewritten. When an implementer reports completion, the main session sends the completed work to the code-reviewer (not a direct AC verification by the main session). The review assignment includes: the full story file, the list of files changed by the implementer, and the epic Technical Notes.
- [ ] **AC-4**: When the reviewer returns APPROVED (no MUST FIX findings), the main session marks the story DONE.
- [ ] **AC-5**: When the reviewer returns NOT APPROVED (MUST FIX findings), the main session routes the findings to the implementer with specific feedback. The implementer fixes and reports completion again. The main session sends the updated work back to the reviewer. This is the review loop.
- [ ] **AC-6**: SHOULD FIX findings are relayed to the implementer alongside MUST FIX findings but do not block DONE. The main session notes unaddressed SHOULD FIX items in the story's notes or epic History.
- [ ] **AC-7**: The circuit breaker is documented: max 2 review rounds per story. If the 2nd review still has MUST FIX findings, the main session escalates to the user with the findings summary. The main session does not mark the story DONE and does not loop further.

## Technical Approach

The key structural change is in implement SKILL.md Phase 3 Step 5, which currently has the main session verifying ACs directly. That step becomes a review-assignment to the code-reviewer, with a loop back to the implementer on findings. See epic Technical Notes for the exact flow and CA's recommendations.

## Notes
- The reviewer is automatically spawned -- it is infrastructure, not opt-in. This is what makes it a structural guardrail rather than a suggestion.
- The existing Phase 4 (optional codex review via "and review" modifier) is NOT changed. It remains as Layer 2 defense in depth. The two layers serve different purposes: per-story review (Layer 1, automatic) vs. whole-epic review with external tool (Layer 2, user-triggered).
- The main session's role shifts from "verify ACs" to "route to reviewer and manage the loop." The main session still does all status updates (IN_PROGRESS, DONE) and the closure sequence.
- When the circuit breaker fires (2nd review still has MUST FIX), the main session presents findings to the user and waits for direction. Options the user might choose: fix it themselves, tell implementer to try again, accept the risk, or abandon the story.
