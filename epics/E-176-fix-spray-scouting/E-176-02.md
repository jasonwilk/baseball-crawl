# E-176-02: Report Generator Spray Cleanup

## Epic
[E-176: Fix Spray Charts for Scouting Reports and Opponent Pipeline](epic.md)

## Status
`TODO`

## Description
After this story is complete, the report generator will contain zero spray-specific crawl, UUID resolution, or inline API fetching code. The spray-specific functions added in commit e802bae (`_resolve_and_crawl_spray`, `_build_boxscore_uuid_map`, `_crawl_spray_via_boxscore_uuids`, `_resolve_gc_uuid_via_search`) will be removed and replaced with a simple call to the scouting spray pipeline (`ScoutingSprayChartCrawler.crawl_team()` + `ScoutingSprayChartLoader.load_all()`), which now handles `gc_uuid=NULL` teams via Story 01's boxscore-UUID fallback.

## Context
The report generator's inline spray code was a workaround for the crawler's inability to handle `gc_uuid=NULL`. With Story 01 fixing the crawler, the generator can delegate spray crawl/load to the pipeline and focus on rendering. This eliminates duplicated crawl logic, the fragile exact-name search, and the architectural violation of a renderer doing API calls. See TN-3 for the target architecture.

## Acceptance Criteria
- [ ] **AC-1**: The functions `_resolve_and_crawl_spray`, `_build_boxscore_uuid_map`, `_crawl_spray_via_boxscore_uuids`, and `_resolve_gc_uuid_via_search` are removed from `src/reports/generator.py`.
- [ ] **AC-2**: The constants `_SEARCH_CONTENT_TYPE`, `_UUID_RE`, and `_PLAYER_STATS_ACCEPT` are removed from `src/reports/generator.py` (they were only used by the removed functions).
- [ ] **AC-3**: After the main scouting crawl/load (Step 4), the generator instantiates `ScoutingSprayChartCrawler` (with a DB connection and authenticated client) and calls `crawl_team(public_id, season_id=season_id)`, then instantiates `ScoutingSprayChartLoader` (with a DB connection) and calls `load_all(data_root, public_id=public_id, season_id=season_id)`, per TN-3. For teams with `gc_uuid=NULL`, the crawler's boxscore-UUID fallback (Story 01) populates spray data without inline resolution.
- [ ] **AC-4**: The spray crawl/load in the generator is wrapped in a try/except that catches all exceptions except `CredentialExpiredError` (which propagates). Spray failure is non-fatal -- the report renders without spray charts and logs a warning.
- [ ] **AC-5**: Existing report generator tests are updated to reflect the removal of the inline spray functions and the new pipeline delegation.
- [ ] **AC-6**: The `import re` statement is removed if no remaining code in the module uses it (it was added for `_UUID_RE`).

## Technical Approach
Remove the four spray-specific functions and three constants from `generator.py`. Replace the `_resolve_and_crawl_spray(...)` call in `generate_report()` (Step 4b, around line 777) with calls to `ScoutingSprayChartCrawler.crawl_team()` and `ScoutingSprayChartLoader.load_all()`. The crawler needs a DB connection and client (both already available in scope). Wrap in try/except for non-fatal failure. Update imports: remove `re`, add or adjust crawler/loader imports as needed.
