# E-002-07: Load raw game and stats JSON into database

## Epic
[E-002: GameChanger Data Ingestion Pipeline](epic.md)

## Status
`ABANDONED`

**Reason**: Split into E-002-07a (game loader) and E-002-07b (season stats loader) during 2026-03-03 refinement. This story was too large -- it covered three loader classes across three source files. See E-002-07a.md and E-002-07b.md for the replacements.

## Description
After this story is complete, the system will have loaders for game box scores and player stats that read raw JSON from `data/raw/` and upsert records into the appropriate database tables. Coaches will be able to query game results, player batting/pitching lines, and season aggregate stats from SQL.

## Context
This is the most data-dense load story. It covers three record types: games (one row per game), player game lines (one row per player per game), and player season stats (one row per player per season per team). All three come from different raw files and map to different tables.

Home/away splits and left/right pitcher splits are either computed here (if the API does not provide them) or loaded directly (if the API does). The API spec (E-001-03) determines which.

## Acceptance Criteria
- [ ] **AC-1**: Given `games/{game_id}.json` files, when `GameLoader().load_all()`, then each completed game is upserted into the `games` table with: game ID, date, home team, away team, final score.
- [ ] **AC-2**: Given the same game files, player batting and pitching lines are upserted into `player_game_stats`: one row per (player_id, game_id) per stat type (batting, pitching). Running twice is idempotent.
- [ ] **AC-3**: Given `stats.json` files (season aggregates), when `SeasonStatsLoader().load_all()`, then each player's season stats are upserted into `player_season_stats`.
- [ ] **AC-4**: All loaders return a `LoadResult` (loaded, skipped, errors counts).
- [ ] **AC-5**: Given a game record where a player ID does not exist in the `players` table (orphaned reference), the game stat row is still written with the player ID, and a WARNING is logged. (Referential integrity is soft -- do not reject records because a player wasn't in a roster file.)

## Technical Approach
Three loader classes in `src/gamechanger/loaders/`:
- `game_loader.py` -- `GameLoader`
- `player_game_stats_loader.py` -- `PlayerGameStatsLoader`
- `season_stats_loader.py` -- `SeasonStatsLoader`

Each follows the same pattern as `RosterLoader`. Extract field mapping to a separate `_map_*` method for testability.

## Notes
AC-5 (soft referential integrity) is intentional. The crawl pipeline might have gaps -- an opponent player appears in game stats but was never in a roster we could access. Rejecting the stat row would silently drop data. Accepting it with a warning keeps the data and surfaces the gap.
