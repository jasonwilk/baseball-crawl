# E-002-07b: Load raw player stats JSON into database

## Epic
[E-002: GameChanger Data Ingestion Pipeline](epic.md)

## Status
`BLOCKED`

## Description
After this story is complete, the system will have a `SeasonStatsLoader` that reads season-aggregate stats JSON from `data/raw/` and upserts records into the `player_season_batting` and `player_season_pitching` tables. Coaches will be able to query season-level player statistics from SQL.

## Context
Season-aggregate stats provide the "who is this player?" view -- batting average, OBP, SLG, strikeout rate, and splits. These come from raw files written by E-002-04 (player season stats crawler). The exact data shape depends on what E-002-R-01 discovers about the stats endpoint.

If the API does not provide home/away or L/R splits directly, those split columns will remain NULL and can be computed later from game-level data in a separate story.

## Acceptance Criteria
- [ ] **AC-1**: Given `stats.json` files (season aggregates from E-002-04), when `SeasonStatsLoader().load_all()`, then each player's season batting stats are upserted into `player_season_batting` and pitching stats into `player_season_pitching`. (Exact column names per the delivered `migrations/001_initial_schema.sql` from E-003-01.)
- [ ] **AC-2**: Running the loader twice with the same data produces the same database state (idempotent via upsert on `(player_id, team_id, season_id)` unique constraint).
- [ ] **AC-3**: The loader returns a `LoadResult` (loaded, skipped, errors counts).
- [ ] **AC-4**: Given a player_id that does not exist in the `players` table, the loader inserts a stub player row (`player_id`, `first_name='Unknown'`, `last_name='Unknown'`) before inserting the season stat row, and logs a WARNING. (Same rationale as E-002-07a AC-4.)
- [ ] **AC-5**: Given a stats response where a player has no pitching data (position player), only a `player_season_batting` row is created. Given a player with no batting data (pitcher-only in DH leagues), only a `player_season_pitching` row is created.

## Technical Approach
`src/gamechanger/loaders/season_stats_loader.py` -- `SeasonStatsLoader` class. Follows the same pattern as `RosterLoader` (E-002-06) and `GameLoader` (E-002-07a).

The implementing agent must read the delivered `migrations/001_initial_schema.sql` to get exact column names and types -- including the split column naming convention (`vs_lhp_*`/`vs_rhp_*` for batting, `vs_lhb_*`/`vs_rhb_*` for pitching). Do not rely on table/column names assumed in this story file.

## Notes
- This story does NOT load per-game player lines -- that is E-002-07a.
- The loaders do NOT make HTTP calls. DoD should focus on database interaction testing, not HTTP mocking.
- Split columns (home/away, L/R) may be NULL if the API does not provide them directly. The loader should write whatever the API provides and leave unknown splits as NULL.
- The exact shape of the season stats response depends on E-002-R-01 findings. This story cannot be fully implemented until the raw data format is known.
- Orphaned player IDs are handled via the stub-player pattern: insert a minimal player row before the stat row to satisfy FK constraints. Stub players are identifiable by `first_name='Unknown'` and `last_name='Unknown'` and are backfilled by a later roster crawl.
