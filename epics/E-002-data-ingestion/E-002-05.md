# E-002-05: Crawl opponent team data for all scheduled games

## Epic
[E-002: GameChanger Data Ingestion Pipeline](epic.md)

## Status
`TODO`

## Description
After this story is complete, the system will fetch public profile and stats data for every opponent team that appears in the schedules of our configured teams, writing raw JSON to `data/raw/{season}/teams/{opponent_team_id}/`. This is what enables the "competitive intelligence" use case: scouting opponents before games.

## Context
When Lincoln plays a game, the opponent's team ID appears in the schedule. This crawler uses those opponent IDs to fetch whatever public data GameChanger exposes for those teams -- roster (if accessible), season stats, and game stats from games they've played against other opponents.

This is where a key unknown lives: GameChanger may restrict access to opponent data. The crawl scope must match what the API actually permits. E-001-03 (API spec) must document what opponent data is accessible before this story is executed.

## Acceptance Criteria
- [ ] **AC-1**: Given schedule files for all configured teams, when `OpponentCrawler().crawl_all()`, then for each unique opponent team ID found across all schedules, an attempt is made to fetch the opponent's roster and season stats.
- [ ] **AC-2**: Data for opponents is written to the same directory structure as owned teams: `data/raw/{season}/teams/{opponent_team_id}/roster.json`, `stats.json`.
- [ ] **AC-3**: Given an opponent whose data is inaccessible (403 Forbidden), the error is logged at WARNING level (not ERROR -- this is expected behavior), and the crawl continues. The opponent is not retried.
- [ ] **AC-4**: Opponent files follow the same idempotency rules as equivalent files for owned teams (24-hour freshness for roster/stats).
- [ ] **AC-5**: After the run, the crawler logs: total unique opponents found, opponents successfully crawled, opponents with access denied, opponents with unexpected errors.

## Technical Approach
`src/gamechanger/crawlers/opponent.py`. Reads all schedule files from configured teams, builds a set of unique opponent team IDs (excluding IDs that are already in `config/teams.yaml` as owned teams), then calls the same roster/stats endpoints as the owned-team crawlers. No new API endpoints needed -- just different team IDs.

The `OpponentCrawler` should reuse `RosterCrawler` and `PlayerStatsCrawler` logic rather than duplicating it. Consider extracting a shared base class or passing team IDs as a parameter.

This story is the highest-risk story in E-002 because it depends on GameChanger's access controls for non-owned teams. If opponent data is entirely locked down, this story should be ABANDONED and a note added to E-003 that opponent stats must be sourced from game stats (i.e., what we observed when playing them) rather than their own team profile.
