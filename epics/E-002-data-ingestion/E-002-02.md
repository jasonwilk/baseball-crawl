# E-002-02: Crawl game schedule and game summaries, write raw JSON

## Epic
[E-002: GameChanger Data Ingestion Pipeline](epic.md)

## Status

## Description
After this story is complete, the system will have a `ScheduleCrawler` class that fetches the full game schedule and game summaries for each configured team from the GameChanger API and writes raw JSON to `data/raw/{season}/teams/{team_id}/schedule.json` and `data/raw/{season}/teams/{team_id}/game_summaries.json`. These two files together form the index that downstream crawlers use to identify completed games, opponent IDs, and game metadata.

## Context
The schedule and game-summaries are the spine of the ingestion pipeline. The schedule endpoint (`/teams/{id}/schedule`) returns event objects with dates and metadata. The game-summaries endpoint (`/teams/{id}/game-summaries`) returns game-level results including `game_status`, `opponent_id`, scores, and home/away status. Downstream crawlers need both: schedule for the event index, game-summaries for completion status and opponent identification.

## Acceptance Criteria
- [ ] **AC-1**: Given valid credentials and a configured team list, when `ScheduleCrawler().crawl_all()`, then `data/raw/{season}/teams/{team_id}/schedule.json` is written for each team.
- [ ] **AC-2**: The written schedule JSON is the unmodified API response from `GET /teams/{team_id}/schedule?fetch_place_details=true`. The response contains event objects (see `docs/gamechanger-api.md`). The crawler does NOT filter or transform the response -- it writes raw JSON. Downstream consumers (E-002-03, E-002-05) are responsible for extracting game IDs and opponent IDs from the raw schedule data.
- [ ] **AC-3**: Idempotency rule: schedule files less than 1 hour old are skipped; schedule files 1+ hours old are re-fetched. (Shorter window than roster because schedules change -- games get added, postponed, cancelled.)
- [ ] **AC-4**: Given an API error, the error is logged and the crawl continues to the next team.
- [ ] **AC-5**: The `freshness_hours` threshold is configurable, defaulting to 1.
- [ ] **AC-6**: Given valid credentials, when `ScheduleCrawler().crawl_all()`, then `data/raw/{season}/teams/{team_id}/game_summaries.json` is also written for each team, containing the unmodified response from `GET /teams/{team_id}/game-summaries`. Pagination is handled (all pages fetched). Same idempotency rules as schedule.json (1-hour freshness).

## Technical Approach
Same pattern as `RosterCrawler`. Create `src/gamechanger/crawlers/schedule.py` with a `ScheduleCrawler` class that fetches both endpoints:

1. `GET /teams/{team_id}/schedule?fetch_place_details=true` -- Accept: `application/vnd.gc.com.event:list+json; version=0.2.0`
2. `GET /teams/{team_id}/game-summaries` -- Accept: `application/vnd.gc.com.game_summary:list+json; version=0.1.0`. Supports pagination via `x-pagination: true` header and `start_at` query parameter (see `docs/gamechanger-api.md`).

Reuse `CrawlConfig` and `GameChangerClient` from E-002-01.

## Notes
The freshness threshold for schedule/game-summaries is shorter than roster (1 hour vs. 24 hours) because schedule data and game completion status change more frequently during the season.

The game-summaries response schema is documented in `docs/gamechanger-api.md` (Schema: game-summaries). Key fields for downstream crawlers: `game_status` ("completed"), `game_stream.opponent_id` (opponent team UUID), `game_stream.game_id` (game UUID), `home_away`.
