# E-196-01: Add start_time and timezone to games table and loaders

## Epic
[E-196: Pitching Availability and Game Chronological Ordering](epic.md)

## Status
`TODO`

## Description
After this story is complete, the `games` table will have `start_time` and `timezone` columns, and the schedule loader and scouting loader will populate them from their respective API data sources. The game loader will preserve existing values during upsert since game-summaries does not carry start time data.

## Context
Games currently store only `game_date` (date-only). The API responses already contain start times and timezones but we truncate them during loading. This story adds the schema columns and updates all three loaders to handle the new fields, establishing the data foundation for chronological game ordering (E-196-02).

## Acceptance Criteria
- [ ] **AC-1**: Migration `010_add_game_start_time.sql` exists and adds `start_time TEXT` and `timezone TEXT` columns to the `games` table per Technical Notes TN-1.
- [ ] **AC-2**: The schedule loader extracts `start_time` and `timezone` from the authenticated schedule endpoint response per Technical Notes TN-2 and writes them to the `games` table on INSERT and UPDATE.
- [ ] **AC-3**: The scouting loader extracts `start_time` and `timezone` from the public games endpoint response per Technical Notes TN-2 and writes them to the `games` table on INSERT and UPDATE.
- [ ] **AC-4**: The game loader preserves existing `start_time` and `timezone` values during upsert -- it does not overwrite non-NULL values with NULL (per Technical Notes TN-2, game-summaries lacks start time data).
- [ ] **AC-5**: Tests verify that each loader correctly handles the new columns, including the game loader's preservation behavior.

## Technical Approach
The migration adds two nullable columns. The schedule loader currently reads `event.start.datetime` but truncates to date -- it needs to pass through the full datetime and timezone. The scouting loader reads from cached public games JSON which has `start_ts` and `timezone` fields. The game loader's INSERT/UPDATE SQL needs a conditional pattern that preserves existing non-NULL values (e.g., `COALESCE(excluded.start_time, games.start_time)` in an upsert, or simply omitting the columns from the UPDATE SET clause).

Read each loader to understand its current upsert pattern before modifying.
