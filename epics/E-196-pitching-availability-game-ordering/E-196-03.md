# E-196-03: Pitching workload query function

## Epic
[E-196: Pitching Availability and Game Chronological Ordering](epic.md)

## Status

## Description
After this story is complete, `src/api/db.py` will have a query function that returns per-pitcher workload data (last outing date, days since last outing, 7-day pitch total, and 7-day span) for a given team. Both the dashboard route and standalone report renderer will use this function to achieve data parity.

## Context
Pitching availability columns (Rest/Last and P(7d)) need the same underlying data on both the dashboard and standalone report surfaces. A single shared query function in `src/api/db.py` ensures data parity (Technical Notes TN-7) and avoids duplicating SQL logic across the two display paths.

## Acceptance Criteria
- [ ] **AC-1**: A new function in `src/api/db.py` accepts a team ID and a reference date, and returns per-pitcher workload data per Technical Notes TN-4: `last_outing_date`, `last_outing_days_ago`, `pitches_7d`, `span_days_7d`.
- [ ] **AC-2**: The function returns data keyed by player ID so callers can merge it with existing pitcher stat rows.
- [ ] **AC-3**: `pitches_7d` is 0 (not NULL) when a pitcher has no appearances in the 7-day window.
- [ ] **AC-4**: `last_outing_days_ago` is NULL when a pitcher has no appearances at all, and 0 when the last outing is on the reference date.
- [ ] **AC-5**: `pitches_7d` is NULL (not 0) when a pitcher has appearances in the 7-day window but ALL pitch counts are NULL, so callers can distinguish "no pitch data" from "zero pitches".
- [ ] **AC-6**: Tests verify all edge cases: pitcher with no appearances, pitcher with appearances only outside 7d window, pitcher with multiple appearances in 7d window, pitcher whose last outing is on the reference date, pitcher with appearances but all NULL pitch counts, pitcher with mixed NULL and non-NULL pitch counts in 7d window (pitches_7d should be sum of non-NULL values).

## Technical Approach
The query joins `player_game_pitching` with `games` to compute workload metrics. The reference date parameter enables both live (dashboard passes `date('now')`) and snapshot (report passes generation date) use cases. Consider whether a single SQL query with conditional aggregation can compute all four metrics, or whether two queries (last outing + 7d window) are clearer.

## Notes
- The `player_game_pitching` table has `pitches` column (integer, nullable). Pitchers with NULL pitches in a game still count for `last_outing_date` and `span_days_7d` (the appearance happened). For `pitches_7d`, if ALL appearances in the 7d window have NULL pitches, the return value should be NULL (not 0) so callers can distinguish "no pitch data" from "zero pitches". Callers display NULL pitches as `?/{span}d` vs `0/{span}d`.
- The function should work for both member and tracked teams -- the query is team-scoped, not membership-scoped.
- **7-day window boundary**: Use `game_date >= date(reference_date, '-6 days')` for a true 7-calendar-day inclusive window. Do NOT use `reference_date - 7` which creates an 8-day window and could produce `/8d` under the "P (7d)" header.
