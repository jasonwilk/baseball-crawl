# E-196-02: Fix game chronological ordering across all surfaces

## Epic
[E-196: Pitching Availability and Game Chronological Ordering](epic.md)

## Status
`TODO`

## Description
After this story is complete, all game-listing queries will include `start_time` as a same-direction tiebreaker alongside `game_date`, ensuring same-day games (tournament doubleheaders) display in correct chronological order across both the standalone report and dashboard surfaces.

## Context
Same-day games currently display in non-deterministic order because queries sort only by `game_date`. With `start_time` now populated (E-196-01), we can add a tiebreaker. This is a targeted fix to every query that orders games -- it does not change what data is displayed, only the order.

## Acceptance Criteria
- [ ] **AC-1**: All game-ordering queries in `src/reports/generator.py` include `start_time` as a same-direction tiebreaker with `NULLS LAST` per Technical Notes TN-3 (DESC queries get `start_time DESC NULLS LAST`; ASC queries get `start_time ASC NULLS LAST`).
- [ ] **AC-2**: All game-ordering queries in `src/api/db.py` follow the same convention per Technical Notes TN-3, preserving each query's existing sort direction.
- [ ] **AC-3**: All `ORDER BY` clauses containing `game_date` in `src/` (discovered via grep) include a `start_time` tiebreaker matching the existing sort direction, with `NULLS LAST`.
- [ ] **AC-4**: Games with NULL `start_time` sort after timed games on the same date.
- [ ] **AC-5**: Tests verify ordering behavior for same-day games with different start times and with NULL start times.

## Technical Approach
Search the codebase for all SQL queries containing `ORDER BY` with `game_date` and add `start_time` as a same-direction tiebreaker with `NULLS LAST`. Preserve each query's existing sort direction (DESC stays DESC, ASC stays ASC). The change is mechanical -- add the column to the ORDER BY clause. Verify by grepping for `game_date` ordering patterns across `src/`.

## Notes
- SQLite supports `NULLS LAST` syntax natively (since 3.30.0).
- The tiebreaker preserves the existing sort direction of each query. DESC queries (recent form, game history) use `start_time DESC NULLS LAST`; ASC queries (upcoming schedule, nearest game) use `start_time ASC NULLS LAST`.
