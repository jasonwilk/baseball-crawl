# E-053-01: Profile-scoped credential extractor

## Epic
[E-053: Profile-Scoped Credentials](epic.md)

## Status
`TODO`

## Description
After this story is complete, the mitmproxy credential extractor addon will write credentials to profile-scoped `.env` keys based on the detected traffic source. Web browser traffic will write to `GAMECHANGER_AUTH_TOKEN_WEB`, `GAMECHANGER_DEVICE_ID_WEB`, etc. iOS app traffic will write to `_MOBILE` suffixed keys. Unknown traffic is logged and dropped -- there are no flat keys to write to.

## Context
The credential extractor already calls `gc_filter.detect_source(user_agent)` to classify traffic, but uses the result only for logging. This story makes the detection actionable by routing credentials to the correct env key set based on source.

## Acceptance Criteria
- [ ] **AC-0**: `detect_source()` in `gc_filter.py` returns `"ios"` for the Odyssey app User-Agent: `Odyssey/2026.7.0 (com.gc.teammanager; build:0; iOS 26.3.0) Alamofire/5.9.0`. The `ios_patterns` list includes `"odyssey/"` and `"alamofire/"` to cover this UA. A regression test in `tests/test_proxy/test_gc_filter.py` verifies this.
- [ ] **AC-1**: Given web browser traffic through mitmproxy (User-Agent contains `Chrome/`), when the credential extractor processes the request, then it writes to `GAMECHANGER_AUTH_TOKEN_WEB`, `GAMECHANGER_DEVICE_ID_WEB`, `GAMECHANGER_APP_NAME_WEB`, and `GAMECHANGER_SIGNATURE_WEB` in `.env`.
- [ ] **AC-2**: Given iOS app traffic through mitmproxy (User-Agent contains `Odyssey/` or `CFNetwork/`), when the credential extractor processes the request, then it writes to `GAMECHANGER_AUTH_TOKEN_MOBILE`, `GAMECHANGER_DEVICE_ID_MOBILE`, `GAMECHANGER_APP_NAME_MOBILE`, and `GAMECHANGER_SIGNATURE_MOBILE` in `.env`.
- [ ] **AC-3**: Given traffic with an unrecognized User-Agent (`detect_source` returns `"unknown"`), when the credential extractor processes the request, then it logs a WARNING with the User-Agent snippet and does NOT write any credentials to `.env`.
- [ ] **AC-4**: The `_CREDENTIAL_HEADERS` mapping in `credential_extractor.py` uses a helper function or lookup that applies the profile suffix, rather than duplicating the mapping three times.
- [ ] **AC-5**: The in-memory deduplication cache is keyed by the suffixed env key names, so web and mobile credentials are tracked independently.
- [ ] **AC-6**: Log messages include both the source and the suffixed key names that were written (never log credential values).
- [ ] **AC-7**: Tests cover all three source paths (web, ios, unknown) and verify the correct env keys are written (or not written, for unknown).
- [ ] **AC-8**: The old flat-key constants (`"GAMECHANGER_AUTH_TOKEN"`, etc.) are removed from `credential_extractor.py` -- no code path writes unsuffixed keys.

## Technical Approach
- Add a helper function (e.g., `_suffix_keys(source: str) -> dict[str, str] | None`) that takes the `detect_source` result and returns the header-to-env-key mapping with the appropriate suffix, or `None` for unknown sources. The source-to-suffix mapping: `"ios"` -> `"_MOBILE"`, `"web"` -> `"_WEB"`, `"unknown"` -> `None` (drop).
- Call this helper in `CredentialExtractor.request()` after calling `detect_source()`. If `None`, log WARNING and return early.
- Update the cache to use the suffixed keys so web and mobile credentials are tracked independently.
- In `credential_parser.py`, no changes are needed -- `merge_env_file()` is key-agnostic and already handles arbitrary key names.
- Note: these scripts will eventually be wrapped by E-055's unified `bb` CLI (separate epic, no dependency).
