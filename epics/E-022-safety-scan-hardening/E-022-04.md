# E-022-04: Integration Tests for Git Pre-Commit Hook Chain

## Epic
[E-022: Safety Scan Hardening](./epic.md)

## Status
`DONE`

## Description
After this story is complete, a pytest integration test suite at `tests/test_pii_hook_integration.py` proves that the git pre-commit hook chain works end-to-end: a real `git commit` in a temporary repository fires the hook, the scanner runs, PII violations block the commit, and clean commits succeed with visible confirmation output. This validates the full chain that unit tests cannot reach.

## Context
The existing `tests/test_pii_scanner.py` tests scanner functions directly by importing and calling them. It proves the scanner logic works. But it does NOT prove that:
- The git pre-commit hook at `.githooks/pre-commit` actually fires during a commit
- The hook correctly invokes the scanner with the right arguments
- The scanner's exit code correctly blocks or allows the commit
- The confirmation output from E-022-01 appears in commit output

These are integration-level concerns. A bug in the hook script (wrong path, wrong argument, missing chmod) would pass all unit tests but fail in production. This story fills that gap.

The Claude Code PreToolUse hook is NOT tested here -- it requires a running Claude Code session and cannot be automated in pytest. The git hook chain is the primary enforcement layer and can be tested with standard git commands.

This story depends on E-022-01 because one acceptance criterion verifies the success confirmation output.

- [ ] **AC-2**: A test creates a temporary git repo, configures `core.hooksPath`, stages a clean Python file, runs `git commit`, and verifies the commit succeeds (exit code 0)

## Technical Approach

### Test File: `tests/test_pii_hook_integration.py`

Use `subprocess.run` to execute git commands in a temporary directory. The key setup for each test:

```python
import subprocess
from pathlib import Path

import pytest

def _init_repo(tmp_path: Path, project_root: Path) -> Path:
    """Create a temp git repo with the project's pre-commit hook configured."""
    repo = tmp_path / "test-repo"
    repo.mkdir()
    subprocess.run(["git", "init", str(repo)], check=True, capture_output=True)
    subprocess.run(
        ["git", "config", "core.hooksPath", str(project_root / ".githooks")],
        cwd=str(repo), check=True, capture_output=True,
    )
    # Configure git user for commits
    subprocess.run(
        ["git", "config", "user.email", "test@test.local"],
        cwd=str(repo), check=True, capture_output=True,
    )
    subprocess.run(
        ["git", "config", "user.name", "Test"],
        cwd=str(repo), check=True, capture_output=True,
    )
    return repo
```

**Important**: The pre-commit hook at `.githooks/pre-commit` uses `$(git rev-parse --show-toplevel)/src/safety/pii_scanner.py` to find the scanner. In the temp repo, `--show-toplevel` returns the temp repo path, not the project root. The scanner will not be found there.

**Solution**: Set the `PYTHONPATH` or symlink. The simplest approach: the pre-commit hook needs to find the scanner relative to its own location. Since `core.hooksPath` points to the project's `.githooks/`, and the hook uses `$(git rev-parse --show-toplevel)` (which returns the temp repo), the scanner path will be wrong.

**Fix approach**: Symlink `src/safety/` into the temp repo so the hook finds the scanner at the expected path:

```python
# In _init_repo:
(repo / "src" / "safety").mkdir(parents=True, exist_ok=True)
# Symlink the scanner and patterns modules
for f in ["pii_scanner.py", "pii_patterns.py", "__init__.py"]:
    src = project_root / "src" / "safety" / f
    dst = repo / "src" / "safety" / f
    if src.exists():
        dst.symlink_to(src)
```

**Project root discovery**: Use a pytest fixture:

```python
@pytest.fixture
def project_root() -> Path:
    """Return the project root directory."""
    return Path(__file__).resolve().parent.parent
```

### Test Cases

1. **test_blocked_commit_with_pii**: Stage a `.json` file with `{"email": "test@example.com"}`. Run `git commit`. Assert exit code != 0. Assert stderr contains `[PII BLOCKED]` and `email`.

2. **test_clean_commit_succeeds**: Stage a `.py` file with `x = 42`. Run `git commit`. Assert exit code == 0. Assert commit was created (check `git log`).

3. **test_clean_commit_shows_confirmation**: Same as above but assert stderr/stdout contains `[pii-scan] Scanned` and `[pii-hook] PII scan passed.`.

4. **test_synthetic_marker_allows_commit**: Stage a `.txt` file with `# synthetic-test-data\ntest@example.com`. Run `git commit`. Assert exit code == 0.

5. **test_blocked_commit_shows_pattern_name**: Stage a file with `Bearer eyFAKETOKEN123`. Assert output contains `bearer_token`.


Or register in `pyproject.toml`:
```toml

markers = [
    "integration: integration tests that require git (deselect with '-m not integration')",
]
```

Check if `pyproject.toml` exists first. If not, use `conftest.py`.

### Git Commit Command in Tests

Use `subprocess.run` with `capture_output=True` and `text=True`. Do NOT use `check=True` for commits that should fail. Check the return code manually.

```python
result = subprocess.run(
    ["git", "commit", "-m", "test commit"],
    cwd=str(repo),
    capture_output=True,
    text=True,
)
# Combine stdout and stderr -- hook output may go to either
output = result.stdout + result.stderr
```

## Notes
- The symlink approach for making the scanner available in the temp repo is the simplest option. An alternative is to modify the pre-commit hook to accept a `SCANNER_PATH` environment variable, but that adds complexity to a production script for test convenience. Prefer the symlink.
- Git commit requires at least one staged change. Each test must stage a file before committing.
- The test user email `test@test.local` does not match the PII email pattern because `.local` is not a valid TLD in the scanner's regex. If it does match, use a non-email format for the git config.
- The Claude Code PreToolUse hook is NOT tested. That requires a running Claude Code session. The git hook chain is the primary enforcement layer and is sufficient for automated testing.
- These tests require `git` to be installed. They will fail in environments without git, which is acceptable -- this is a git-based project.
