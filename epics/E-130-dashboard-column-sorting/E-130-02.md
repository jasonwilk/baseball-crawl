## Description
After this story is complete, coaches can click any column header on the opponent scouting report's batting leaders and pitching leaders tables to sort by that column. This is the highest-value sorting surface for coaching -- before a game, coaches open the opponent report and need to quickly identify dangerous hitters (sort by AVG) or effective pitchers (sort by ERA).

## Context
The opponent detail page (`/dashboard/opponents/{opponent_team_id}`) displays two stat tables: batting leaders and pitching leaders. Both are currently in a fixed sort order. The page has a single route handler (`opponent_detail()`) that fetches data from `db.get_opponent_scouting_report()` and computes pitching rate stats via `_compute_opponent_pitching_rates()`.

This page has a unique challenge: two independently sortable tables on a single page. The sort params need to distinguish which table they apply to, per TN-7.

The sort pattern (query params, clickable headers, `▲`/`▼` indicators) is established by E-130-01 for the team stats pages. This story reuses that pattern.

## Acceptance Criteria
- [ ] **AC-1**: The `/dashboard/opponents/{opponent_team_id}` route accepts sort and direction parameters for both the batting and pitching tables on the page, per TN-7.
- [ ] **AC-2**: All column headers in the opponent batting leaders table are rendered as clickable `<a>` links that sort the batting table by that column. Column keys per TN-3 (opponent batting columns).
- [ ] **AC-3**: All column headers in the opponent pitching leaders table are rendered as clickable `<a>` links that sort the pitching table by that column. Column keys per TN-3 (opponent pitching columns).
- [ ] **AC-4**: The active sort column in each table displays a `▲` (ascending) or `▼` (descending) indicator. Non-active columns show no indicator.
- [ ] **AC-5**: Default sort (no params) remains AVG descending for opponent batting and ERA ascending for opponent pitching, matching the current behavior.
- [ ] **AC-6**: Players with zero denominators (0 AB, 0 IP) sort to the bottom regardless of direction, per TN-2.
- [ ] **AC-7**: Sorting one table does not affect the sort state of the other table on the same page.
- [ ] **AC-8**: Tests verify: (a) default sort order matches current behavior, (b) recognized sort params produce correctly ordered results for both tables, (c) sorting one table preserves the other table's sort state, (d) zero-denominator rows sort to bottom.

## Technical Approach
Read E-130-01's implementation for the sort pattern (sort key computation, header link generation, direction toggling). Apply the same pattern to the `opponent_detail()` route handler in `src/api/routes/dashboard.py` and the `opponent_detail.html` template.

The key difference from E-130-01 is the dual-table challenge. The route handler needs to parse and apply sort params for both tables independently, per TN-7. The template needs to generate header links that preserve the other table's sort state.

Rate stats for opponent pitching are already computed in Python by `_compute_opponent_pitching_rates()`. Opponent batting rate stats (AVG, OBP, SLG) follow the same Jinja2 template computation pattern as team batting -- sort keys use the raw column formulas from TN-2.

If E-130-01 extracted a reusable sort utility (sort key definitions, sort function), this story should reuse it rather than duplicating.
