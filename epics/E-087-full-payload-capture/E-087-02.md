# E-087-02: API Doc Standards for Payload Examples and PII-Safe Placeholders

## Epic
[E-087: Full Payload Capture in Endpoint Logger](epic.md)

## Status
`TODO`

## Description
After this story is complete, the API documentation rules and ingest-endpoint skill will include guidance on (1) documenting request body and response payload examples as standard sections in endpoint files, and (2) using PII-safe placeholder values in all committed example JSON. This ensures that api-scout knows to capture and document request/response examples during endpoint ingestion, and that those examples never contain real team names, player names, cities, or unredacted UUIDs.

## Context
The endpoint documentation format already has an "Example Response" section in the markdown body template (see `/.claude/skills/ingest-endpoint/SKILL.md`), but there is no explicit standard for:
1. **Request body examples** -- POST/PATCH/PUT endpoints have no guidance on documenting request payloads
2. **PII-safe placeholders** -- existing endpoint docs contain real team names ("Lincoln Rebels 14U"), real cities ("Lincoln"), and real UUIDs in example JSON (see `docs/api/endpoints/get-me-teams.md`)
3. **Explicit rules** -- neither `.claude/rules/api-docs.md` nor the ingest-endpoint skill mandate PII-safe examples

The full payload capture in E-087-01 will make raw request/response data readily available in proxy sessions. This story ensures the documentation layer is ready to receive that data in a safe form.

## Acceptance Criteria
- [ ] **AC-1**: `.claude/rules/api-docs.md` includes a new "Example JSON Safety" section (after YAML Frontmatter Schema) that defines the PII-safe placeholder taxonomy with specific approved placeholder values for each category: team names, organization/league names, cities/states, venue/field names, player names, UUIDs (including UUID-as-dict-key pattern), emails, public_id slugs, url_encoded_name, and avatar/media URLs. Also includes explicit NOT-PII declarations (jersey numbers, numeric stats, dates, boolean flags, enum values) and a semi-identifying combinations note (win-loss records, year+age_group+competition_level). A cross-reference sentence is added to the existing Accuracy Standard section.
- [ ] **AC-2**: `.claude/rules/api-docs.md` includes guidance that endpoint files SHOULD include an "Example Request Body" section (between Headers and Response sections) for POST/PATCH/PUT/DELETE endpoints that accept a JSON body.
- [ ] **AC-3**: The PII-safe placeholder taxonomy in `.claude/rules/api-docs.md` explicitly states that numeric stats, dates, boolean flags, enum values, and jersey numbers are NOT PII and may be kept as-is from real responses.
- [ ] **AC-4**: `.claude/skills/ingest-endpoint/SKILL.md` Phase 1 instructions include a step (between step 3 "verify credential safety" and step 4 "document the endpoint") to redact PII in BOTH request body AND response body example JSON per the taxonomy in `.claude/rules/api-docs.md`. The instruction references the rule file by absolute path.
- [ ] **AC-5**: `.claude/skills/ingest-endpoint/SKILL.md` markdown body template includes an "Example Request Body" section placeholder (between Headers and Response) for mutation endpoints (POST/PATCH/PUT/DELETE), with an "Omit if method is GET or no request body" annotation consistent with the existing optional-section pattern.
- [ ] **AC-6**: The placeholder taxonomy is defined in exactly one place (`.claude/rules/api-docs.md`) and referenced by path from the ingest-endpoint skill -- no duplicate taxonomy definitions.

## Technical Approach
Two context-layer files need updating. The API docs rule file (`.claude/rules/api-docs.md`) gets a new section defining the placeholder taxonomy and request body example guidance. The ingest-endpoint skill (`.claude/skills/ingest-endpoint/SKILL.md`) gets a redaction step in Phase 1 instructions and an "Example Request Body" section in the markdown body template. The taxonomy from the epic Technical Notes provides the specific placeholder values.

**Placement guidance** (from architect review):
- `api-docs.md`: "Example JSON Safety" section goes after YAML Frontmatter Schema (currently the last section). Add a cross-reference in the Accuracy Standard section.
- `ingest-endpoint skill`: PII redaction step goes in Phase 1 between step 3 and step 4. "Example Request Body" template section goes between Headers and Response in the Markdown Body Template, with "Omit if method is GET or no request body" annotation.
- Note: the tag vocabulary duplication between api-docs.md and the skill is a pre-existing issue -- do NOT replicate this pattern for the PII taxonomy. Define it once in api-docs.md and reference by path.

**Ingest workflow clarification**: For POST/PATCH endpoints, the saved raw response in `data/raw/` is the RESPONSE body only. The request body is documented in the endpoint file, not saved as a separate raw artifact.

## Notes
- This story does NOT retroactively fix existing endpoint docs that contain real team names/UUIDs. That would be a separate sweep (and could be captured as an idea if desired).
- The PII pre-commit hook catches credentials (tokens, passwords) but not team names or cities. The documentation rules are the enforcement point for this broader PII-safety concern in docs.
- Numeric stats, dates, enum values, and jersey numbers are explicitly NOT PII per the taxonomy -- this avoids over-redaction that would make examples useless for schema understanding.
- Future idea (out of scope): a `pii_risk: high | low` frontmatter field for endpoint files, enabling automated identification of high-PII-density docs (e.g., `/teams/{team_id}/users`, `/me/user`, `/teams/{team_id}/relationships`).
