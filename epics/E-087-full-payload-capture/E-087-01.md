# E-087-01: Capture Full Payloads in Endpoint Logger

## Epic
[E-087: Full Payload Capture in Endpoint Logger](epic.md)

## Status

## Description
After this story is complete, the mitmproxy endpoint logger addon will capture complete request/response pairs: bodies (inline, with size cap), all headers (no filtering by default; optional stripping when enabled via `PROXY_STRIP_AUTH_HEADERS`), and full query parameter key-value pairs. Binary content types will be skipped for bodies. The JSONL log format remains atomic -- one line per request/response pair. Session data is gitignored, secured, and will never be committed.

## Context
The current `endpoint_logger.py` captures only metadata: method, host, path, query key names (no values), content-types, status code, timestamp, and source. This was conservative but insufficient for API discovery. The operator wants full, unfiltered payloads so agents can analyze API responses without live calls AND so credentials can be extracted from session data for crawling.

SE recommended inline storage (no sidecar files) and a 2MB body cap with truncation sentinel. The default is no header filtering -- all headers including auth tokens are captured as-is (the session store is secured and gitignored). Optional stripping of sensitive headers is available via `PROXY_STRIP_AUTH_HEADERS=true` for users who want it.

## Acceptance Criteria
- [ ] **AC-1**: Given a GC API response with a JSON body, when the endpoint logger processes the flow, then the JSONL entry includes a `response_body` field containing the full response body as a string.
- [ ] **AC-2**: Given a GC API POST/PATCH/PUT/DELETE request with a JSON body, when the endpoint logger processes the flow, then the JSONL entry includes a `request_body` field containing the full request body as a string.
- [ ] **AC-3**: Given a GC API GET request with no body, when the endpoint logger processes the flow, then the `request_body` field is `null`.
- [ ] **AC-4**: Given `PROXY_CAPTURE_BODIES=true` and a GC API request with query parameters, when the endpoint logger processes the flow, then the JSONL entry includes a `query_params` field (dict) with the full key-value mapping. The existing `query_keys` field is retained in both modes for backward compatibility.
- [ ] **AC-5**: Given `PROXY_CAPTURE_BODIES=true` (the default), when the endpoint logger processes a flow, then the JSONL entry includes `request_body`, `response_body`, `request_headers`, `response_headers`, and `query_params` fields. When `PROXY_CAPTURE_BODIES=false`, these five fields are omitted (metadata-only mode, backward compatible with current behavior -- only `query_keys` remains).
- [ ] **AC-6**: Given `PROXY_STRIP_AUTH_HEADERS=true`, when the endpoint logger captures headers, then sensitive headers (`gc-token`, `gc-device-id`, `authorization`, `gc-signature`, `cookie`, `set-cookie`) are excluded. When `PROXY_STRIP_AUTH_HEADERS=false` (the default), all headers are captured unfiltered.
- [ ] **AC-7**: Given a response body larger than 2MB (or the value of `MAX_BODY_BYTES`), when the endpoint logger processes the flow, then `response_body` contains `"<truncated: N bytes>"` where N is the original byte count. Same rule applies to `request_body`.
- [ ] **AC-8**: Given a response with a binary content type (`image/*`, `video/*`, `application/octet-stream`), when the endpoint logger processes the flow, then `response_body` is `null`. Same rule for request bodies with binary content types.
- [ ] **AC-9**: Given the env var `MAX_BODY_BYTES` is set to a custom value, when the endpoint logger initializes, then the truncation threshold uses the env var value instead of the default 2MB.
- [ ] **AC-10**: The `_make_flow` test helper accepts `request_body`, `response_body`, `request_headers` (beyond content-type/user-agent), and `response_headers` (beyond content-type) parameters for constructing mock flows with body/header data.
- [ ] **AC-11**: The `test_entry_contains_exactly_expected_keys` test verifies the expanded key set including `request_body`, `response_body`, `query_params`, `request_headers`, `response_headers`. Two variants are required: one for capture mode key set (`PROXY_CAPTURE_BODIES=true`), one for metadata-only mode key set (`PROXY_CAPTURE_BODIES=false`).
- [ ] **AC-12**: Tests cover both configurations: full capture mode and metadata-only mode, plus auth header stripping on/off.
- [ ] **AC-13**: All existing tests are updated or replaced as needed (not deleted without replacement). All tests pass.

## Technical Approach
The addon file `proxy/addons/endpoint_logger.py` needs a `MAX_BODY_BYTES` module-level constant and an expanded `_build_entry()` function that extracts bodies, headers, and query params from the mitmproxy flow object. The test file needs its `_make_flow` helper extended and test cases added/updated to cover the new fields, truncation, and binary skip behavior.

Key constraint: mitmproxy flow objects expose `flow.request.content` (bytes) and `flow.response.content` (bytes) for bodies, and `flow.request.headers` / `flow.response.headers` as case-insensitive multidict-like objects. The implementation needs to decode bytes to string for JSON bodies and handle encoding errors gracefully.

**Implementation notes from SE review:**
- mitmproxy transparently decompresses gzip/deflate and reassembles chunked bodies before the `response` hook. No special handling needed.
- `flow.request.content` is `b""` (not `None`) for bodyless GET requests. Must explicitly check `not flow.request.content` and emit `null`, not empty string.
- All `.decode()` calls must use `errors="replace"` for non-UTF-8 bytes.
- `MAX_BODY_BYTES` env var parse: guard with try/except, fallback to 2MB default + warning log.
- Binary content-type matching: use `startswith("image/")`, `startswith("video/")`, exact match `"application/octet-stream"`.
- `dict(flow.request.headers)` produces lowercase keys; duplicate headers keep last value only (acceptable).

No header filtering by default. The session store is secured and gitignored -- auth tokens are captured intentionally so they can be used for credential extraction and API calls. Optional stripping is available via `PROXY_STRIP_AUTH_HEADERS=true` (see AC-6).
