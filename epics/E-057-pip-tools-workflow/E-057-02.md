# E-057-02: Update project docs and developer workflow

## Epic
[E-057: pip-tools Dependency Management](epic.md)

## Status
`TODO`

## Description
After this story is complete, CLAUDE.md and the requirements file headers will document the pip-tools workflow so any developer or agent knows how to add, upgrade, or remove dependencies. The old manual-edit workflow is replaced with clear pip-compile instructions.

## Context
E-057-01 creates the `.in` and compiled `.txt` files. This story ensures the workflow is discoverable: how to add a dep, how to upgrade, how to regenerate. Without this, developers will edit `requirements.txt` directly (defeating the purpose) or agents will not know the correct procedure.

## Acceptance Criteria
- [ ] **AC-1**: CLAUDE.md Commands section includes pip-tools commands:
  - `pip-compile requirements.in -o requirements.txt` -- regenerate runtime pins
  - `pip-compile requirements-dev.in -o requirements-dev.txt` -- regenerate dev pins
  - Include the `--strip-extras` and `--generate-hashes` flags if E-057-01 used them.
- [ ] **AC-2**: CLAUDE.md has a new "Dependency Management" subsection (under Tech Stack or as a peer section) that documents:
  - The two-file workflow (`*.in` for humans, `*.txt` for machines)
  - How to add a new dependency (add to appropriate `.in` file, run pip-compile, run tests, commit both)
  - How to upgrade a dependency (update range in `.in`, recompile, test, commit)
  - How to remove a dependency (remove from `.in`, recompile, test, commit)
  - That `requirements.txt` and `requirements-dev.txt` are generated files -- never edit directly
  - That pip-tools is a dev build tool, not listed in any `.in` file
- [ ] **AC-3**: `requirements.in` begins with a comment block (before the first dependency line) that states: (a) this file lists direct runtime dependencies, (b) `requirements.txt` is generated -- do not edit it directly, and (c) the pip-compile command to regenerate. Exact wording is implementer discretion; all three points must be present.
- [ ] **AC-4**: `requirements-dev.in` begins with a comment block (before the `-r` include line) that states: (a) this file adds dev-only dependencies on top of runtime deps, and (b) `requirements-dev.txt` is generated -- do not edit it directly. Exact wording is implementer discretion; both points must be present.
- [ ] **AC-5**: The existing CLAUDE.md comment about requirements.txt in the Architecture section (if any) is updated to reflect the new workflow. No stale references to "edit requirements.txt directly" remain.
