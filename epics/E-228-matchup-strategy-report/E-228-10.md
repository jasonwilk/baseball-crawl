# E-228-10: User-facing documentation -- coaching guide + admin guide

## Epic
[E-228: Matchup Strategy Report](./epic.md)

## Status
`TODO`

## Description
After this story is complete, the project's `docs/coaching/` and `docs/admin/` surfaces include user-facing documentation for the matchup section: a coach-voice guide explaining what the matchup section delivers and how to read it, and an operator-voice guide explaining how to generate a matchup report (admin form, "us team" dropdown, fresh-scout expectations, error states). The `docs/api/flows/opponent-scouting.md` page is updated if the canonical opponent-scouting flow now includes a plays step (verified during E-228-02).

## Context
The user established WELL-tested-and-documented as a hard quality gate: documentation is story work, not closure deferral. This story consolidates the user-facing documentation that was deliberately kept out of code stories (E-228-01, E-228-02, E-228-03, E-228-08, E-228-09) so docs-writer can produce coherent, audience-appropriate prose -- not piecemeal docstring fragments scattered across implementation commits.

This story runs AFTER all code stories complete so the docs reflect actual shipped behavior. It runs BEFORE the context-layer closure pass (E-228-11) so the rule edits can reference the documented surface.

Audiences:
- **Coaches** (`docs/coaching/`): "What is this report? When do I generate one? How do I read it during pre-game prep?"
- **Operators** (`docs/admin/`): "How do I create a matchup report? What does the form mean? What if generation fails? Where does the data come from?"
- **Developers** (`docs/api/flows/`): If the opponent-scouting flow now has a plays step, document it -- otherwise skip

## Acceptance Criteria

### Behavior

- [ ] **AC-1**: New page at `docs/coaching/matchup-report.md` explaining the matchup report in coach voice. Content: what it is, when to generate one, what each section means (head-to-head banner, top hitters, opposing pitching availability, pitch-attack notes, defensive shifts, lineup card with **explicit philosophy caveat per Coach S-3** -- 2-hole-as-second-OBP is ONE coaching philosophy and the coach is expected to override based on their preference, optional starter suggestion, loss-recipe), how to read inline citations, what the data-depth badges mean, **what "silence" on starter prediction means per Coach M-3** (system had no strong opinion — coach decides — NOT a broken section), how to override the system suggestion. **Voice**: coach-to-coach, not technical. Bullet lists welcome. Length target: 1-2 pages printed.
- [ ] **AC-2**: New page at `docs/admin/matchup-report-generation.md` explaining the operator-side flow. Content: how to generate a matchup report from the admin form, what each form field does (matchup checkbox, "us team" dropdown, optional starter input), feature-flag toggle (`FEATURE_MATCHUP_ANALYSIS`), the ~2-min generation time expectation, error states (LSB scout fails, OpenRouter unavailable, opponent has no games), CLI parity (`bb report generate <url> --our-team <id>`, `--starter <player_id>`). **Per api-scout MEDIUM-2 + MEDIUM-3**: include notes that (a) newly-tracked opponents may show frequent "decision unknown" loss-recipe entries until plays/reconcile catches up across enough games (this is normal and resolves with subsequent scout runs), (b) cross-perspective player UUID duplication is an existing operational concern — operators should run `bb data dedup-players` periodically OR after a fresh scout if matchup hitter stats look split.
- [ ] **AC-3 (per ARCH-MICRO + P1-C accepted)**: Update `docs/api/flows/opponent-scouting.md` to reflect the post-E-228-02 six-stage pipeline (now includes plays + reconcile after spray load). **Unconditional** — E-228-02 ships only the api-flow doc edit (NOT the CLAUDE.md / architecture-subsystems.md edits, which were moved to E-228-11 per P1-C); this story verifies E-228-02's update is present and cross-references it from the new admin docs.
- [ ] **AC-4**: Content cross-references between docs follow existing project conventions: coaching doc links to admin doc when the operator does the work; admin doc links to coaching doc for "what coaches see"; both link to relevant CLAUDE.md sections only when the reader needs ambient project identity, NOT for feature-specific content
- [ ] **AC-5**: Each new doc page links from the appropriate index. If `docs/coaching/README.md` exists, add an entry. If `docs/admin/README.md` exists, add an entry. If no index exists, the doc is discoverable from the existing dashboard or admin route docs.

### Test Coverage (documentation tests; this story is mostly prose review)

- [ ] **AC-T1 (link integrity)**: Run a markdown link checker (or grep-based equivalent) over the new doc files; assert all internal links resolve to existing files/sections, all cross-references match real story IDs / file paths. **Reviewer meaningfulness**: catches dead links from copy-paste of older draft content.
- [ ] **AC-T2 (PII scan)**: Run `python3 src/safety/pii_scanner.py` on the new doc files; assert no real player names, real team UUIDs, real coach names, or real venue names appear. Use only the PII-safe placeholder taxonomy from `.claude/rules/api-docs.md` (Jane Doe, Anytown Eagles 14U, etc.) for any examples. **Reviewer meaningfulness**: matchup docs naturally use "Westside Warriors" and "LSB Varsity" as natural reference frames -- those are the project's intentional public scope and OK; but invented or real opponent player names and stats must use placeholders.
- [ ] **AC-T3 (feature-currency)**: For every doc claim about a CLI flag, form field, or behavior, the doc author verifies against the actual implementation (run the CLI with `--help`, open the admin form). Discrepancies flagged in PR description. **Reviewer meaningfulness**: catches "doc described the spec, not the code" drift -- the most common failure mode of stories-deferred docs.
- [ ] **AC-T4 (audience-appropriate language sample test)**: A reviewer reads each new doc front-to-back and confirms tone matches audience: coaching doc reads in coach voice (no jargon, no Python identifiers, no rule-file references), admin doc reads in operator voice (CLI commands, env vars, error states are appropriate). **Reviewer meaningfulness**: a reviewer who is NOT the author validates voice fit.

### Documentation

- [ ] **AC-D1**: Both new doc pages include a "Last updated: YYYY-MM-DD" footer matching the docs-writer convention (if used elsewhere in `docs/`)
- [ ] **AC-D2**: The coaching doc opens with a one-paragraph summary that a head coach can read in 30 seconds and decide "do I want to read more?". The admin doc opens with a one-paragraph summary an operator can read in 30 seconds and decide "is this the page I need?"

## Technical Approach
- docs-writer reads every shipped story's commits + docstrings to ensure the docs reflect actual behavior, not the original spec
- Voice consistency is the most important deliverable -- coaches don't read like operators, operators don't read like developers. If the coach doc accidentally says "the `compute_matchup()` engine returns a `MatchupAnalysis` dataclass", that's a voice violation and a rewrite is required
- Cross-reference existing documentation patterns (e.g., `docs/coaching/` if any pages exist) to match house style
- For images / screenshots: optional. The matchup section's HTML is the visual source of truth; screenshots will go stale and aren't required for v1
