## Description
After this story is complete, the deterministic matchup engine and its input builder exist as importable, tested units. `MatchupInputs` and `MatchupAnalysis` dataclasses define the contract. `build_matchup_inputs(conn, opponent_team_id, our_team_id, season_id)` performs all DB queries (including new helpers in `src/api/db.py`). `compute_matchup(inputs) -> MatchupAnalysis` is a pure function (no DB, no HTTP) that produces the v1 signal set: top-3 opposing hitters, pull-tendency notes, stolen-base profile, first-inning scoring tendency, 3-bucket loss recipe, eligible opposing pitchers, eligible LSB pitchers. The engine carries the `confidence` field and the `data_notes` field used by downstream stories.

## Context
Mirrors the predicted-starter pattern from E-212 (`src/reports/starter_prediction.py`). Engine purity is what makes engine tests fast and DB-free. The input builder owns all DB calls; the engine owns all narrative logic. This story combines what would normally be split (engine vs builder) because v1's signal selection is tightly coupled to helper return shapes -- splitting forces excess interface churn for a small story.

v1 SCOPE (per epic):
- **Top-3 opposing hitters** ranked by OPS (or equivalent), with per-hitter signals: `pa`, `obp`, `slg`, `bb_pct`, `k_pct`, `fps_swing_rate`, `chase_rate`, `swing_rate_by_count`. Below ~20 PA the analysis softens language to "early read only" via the `data_notes` field. ONE mental cue per hitter (NOT two -- coach reduced from original two-cue plan).
- **Pull-tendency notes** for any roster member (not limited to top-3 hitters) when `pull_pct >= _PULL_THRESHOLD` (engine constant, default 0.55) AND `bip_count >= _MIN_BIP_FOR_PULL_NOTE` (default 10). Sourced from `MatchupInputs.opponent_roster_spray` (full-roster spray dataset populated by the input builder per AC-3). Renamed from "defensive shifts" -- v1 surfaces the data; coach decides positioning.
- **Stolen-base profile**: opponent team SB% + opposing catcher CS-against rate (the catcher's own perspective, derived from `caught_stealing` events in plays + boxscore SB/CS counts).
- **First-inning scoring tendency**: from `line_scores`, the rate at which the opponent scores in the 1st inning (offensive) AND allows a 1st-inning run (defensive). Already in DB.
- **3-bucket loss recipe** (HS reality, per coach):
  - `starter_shelled_early` -- starter `ip_outs < 12` AND starter `er >= 4` AND starter took the L
  - `bullpen_couldnt_hold` -- starter went `ip_outs >= 12` AND total bullpen `er >= 3` AND opponent lost
  - `close_game_lost_late` -- `abs(margin) <= 2` AND none of the above
  Other losses fall outside these buckets and are summarized as "uncategorized" without prose.
- **Eligible opposing pitchers** -- list with rest math + recent workload (reuses `get_pitching_workload`). NO per-pitcher prose elaboration (light treatment).
- **Eligible LSB pitchers** when `our_team_id` set -- same list shape, same light treatment. NO three-gate prediction. NO recipe-match suggestion. NO override input.

DROPPED (deferred to E-230 v2): three-gate G1/G2/G3 starter prediction, decision_unknown 6th bucket, plays_loaded_per_game data-completeness surfacing, recent-form arrow with `_RECENT_FORM_ARROW_THRESHOLD`, 4-template TL;DR slot-2 priority tree with 8 module constants, `_TLDR_DIRECTIVE_MAX_CHARS = 25` truncation.

## Acceptance Criteria

### Behavior

- [ ] **AC-1 (dataclasses)**: `src/reports/matchup.py` defines `MatchupInputs` and `MatchupAnalysis` dataclasses. `MatchupInputs` carries: opponent profile, opponent_top_hitters (list), opponent_pitching (list), opponent_losses (list of pre-classified loss rows for engine to bucket), opponent_sb_profile, opponent_first_inning_pattern, opponent_roster_spray (list of `PlayerSprayProfile` records covering ALL opposing roster members with sufficient spray-events data -- `player_id`, `name`, `jersey_number`, `pull_pct`, `bip_count`; this is the source the engine uses for AC-7 pull-tendency notes, NOT just the top-3 hitters), lsb_team (None when our_team_id is None), lsb_pitching (None when our_team_id is None), reference_date, season_id. `MatchupAnalysis` carries: confidence ("high" | "moderate" | "low" | "suppress"), threat_list (top-3 hitters with one cue-kind enum each + supporting_stats list), pull_tendency_notes (list), sb_profile_summary, first_inning_summary, loss_recipe_buckets (3-bucket counts + per-bucket grounding tuples; NO prose), eligible_opposing_pitchers (list), eligible_lsb_pitchers (None or list), data_notes (list of `DataNote` records, each with fields `{message: str, subsection: str}` where `subsection` is one of the canonical sub-section identifiers: `"top_hitters"` (sub-section 1), `"opposing_pitchers"` (sub-section 2), `"sb_profile"` (sub-section 3), `"first_inning"` (sub-section 4), `"loss_recipe"` (sub-section 5), `"lsb_pitchers"` (sub-section 6); the `subsection` field tells the renderer which sub-section the note attaches to per E-228-14 AC-3). Engine produces deterministic structure only; all prose is produced by the LLM Tier 2 wrapper (E-228-13).
- [ ] **AC-2 (engine purity)**: `compute_matchup(inputs: MatchupInputs) -> MatchupAnalysis` does not import `sqlite3`, `httpx`, or any file I/O module. Engine code-level purity is asserted by an explicit test that scans `src/reports/matchup.py` for those imports inside the engine's call graph (test guards regression).
- [ ] **AC-3 (input builder)**: `build_matchup_inputs(conn, opponent_team_id, our_team_id, season_id, *, reference_date) -> MatchupInputs` performs all DB queries through helpers (existing or new). Builder accepts `our_team_id` as None and produces an inputs object with `lsb_team=None`, `lsb_pitching=None`. Builder populates `opponent_roster_spray` for ALL opposing roster members (not just top-3) by querying roster IDs from `team_rosters` and batch-fetching spray events via the existing `get_players_spray_events_batch` helper (`src/api/db.py:2564` or equivalent); each entry carries `player_id`, `name`, `jersey_number`, `pull_pct`, `bip_count` derived from the spray events under the opponent's perspective. Roster members with zero spray events are excluded. Builder docstring documents which helpers it calls and the perspective each query uses (perspective-provenance compliance per `.claude/rules/perspective-provenance.md`).
- [ ] **AC-4 (top-3 hitter ranking)**: Engine ranks opposing hitters by OPS descending, filtered by `pa >= _MIN_PA_FOR_RANKING` (default 10), takes top 3. Tie-breaker: higher PA wins (more sample). Ranking is deterministic and reproducible.
- [ ] **AC-5 (per-hitter cue)**: For each of the top 3 hitters, engine emits one `ThreatHitter` record with: `name`, `jersey_number`, `pa`, supporting stats (e.g., `bb_pct`, `slg`, `fps_swing_rate`), and a `cue_kind` enum value the LLM wrapper later renders. v1 cue kinds: `attack_early` (high `fps_swing_rate` or low `bb_pct`), `pitch_around` (high `bb_pct` AND high `slg`), `expand_zone` (high `k_pct`), `default` (none of the above -- generic note). Engine does not produce prose -- prose is the LLM's job.
- [ ] **AC-6 (low-PA softening)**: When a top-3 hitter has `pa < 20`, engine adds a `data_notes` entry with `subsection="top_hitters"` naming the hitter and the PA badge. PA badges are always visible in the rendered output (never hidden).
- [ ] **AC-7 (pull-tendency notes)**: Engine produces pull-tendency notes from `MatchupInputs.opponent_roster_spray` (the full-roster spray dataset populated by the input builder per AC-3). For each `PlayerSprayProfile` entry where `pull_pct >= 0.55` (`_PULL_THRESHOLD`) AND `bip_count >= 10` (`_MIN_BIP_FOR_PULL_NOTE`), engine emits a `PullTendencyNote` with `name`, `jersey_number`, `pull_pct`, `bip_count`. The engine does NOT recompute spray data here -- it consumes the pre-built profiles from `opponent_roster_spray`. List may be empty.
- [ ] **AC-8 (3-bucket loss recipe)**: Engine classifies each opponent loss into one of three buckets per the heuristics in the Context section. Engine emits per-bucket counts AND, for buckets with at least one loss, a list of (game_date, opposing_score, opponent_score, starter_name, key_stat) grounding tuples that the LLM wrapper uses to ground its prose. Engine does NOT produce prose for any bucket -- prose is exclusively the LLM's responsibility (per E-228-13). Losses falling outside all three buckets are counted as "uncategorized" without per-game detail.
- [ ] **AC-9 (stolen-base profile)**: `MatchupAnalysis.sb_profile_summary` carries: opponent team SB attempts/successes (current season), opponent team SB success rate, opposing catcher CS-against rate (catcher's own perspective). Engine does not include the LSB side; this is opponent-only intel.
- [ ] **AC-10 (first-inning pattern)**: `MatchupAnalysis.first_inning_summary` carries: 1st-inning runs scored rate (offensive) and 1st-inning runs allowed rate (defensive) for the opponent's current season. Sourced from `line_scores`.
- [ ] **AC-11 (eligible opposing pitchers)**: Engine emits a list of the opposing team's eligible pitchers ranked by likely-availability heuristic (rest cleared + recent workload), reusing the existing `get_pitching_workload` output. NO per-pitcher prose. List shape: `[(name, jersey_number, last_outing_date, days_rest, last_outing_pitches, workload_7d), ...]` for up to 5 entries.
- [ ] **AC-12 (eligible LSB pitchers)**: When `our_team_id` is set, engine emits the same list shape for LSB pitchers (queried via `get_pitching_workload(our_team_id, ...)`). When `our_team_id` is None, this field is None.
- [ ] **AC-13 (confidence + suppress)**: The engine drives the suppress decision from existing fields on `MatchupInputs` (no new "coverage" signal field). Engine sets `confidence = "suppress"` when BOTH `len(inputs.opponent_top_hitters) == 0` AND `len(inputs.opponent_losses) == 0` -- there is nothing to analyze. Engine sets `confidence = "low"` when at least one of those is non-empty but no losses bucket out and no top-3 hitter clears `_MIN_PA_FOR_RANKING`. Otherwise `confidence` is "moderate" or "high" based on data depth (>= 8 opponent games AND >= 30 PA on each top-3 hitter qualifies as "high"; thinner data is "moderate"). The LLM Tier 2 wrapper is not invoked when `confidence == "suppress"`.
- [ ] **AC-14 (new query helpers in `src/api/db.py`)**: New helpers added to `src/api/db.py`:
  - `get_top_hitters(team_id, season_id, *, limit=5, min_pa=10)` -- joins `player_season_batting` + `players` + `team_rosters`; OPS computed in SQL (with NaN guard for zero AB).
  - `get_hitter_pitch_tendencies(player_id, season_id, perspective_team_id)` -- per-player aggregates from plays/play_events: `total_pa_with_plays`, `fps_seen`, `fps_swing_count`, `two_strike_pa`, `full_count_pa`, `chase_rate`, `swing_rate_by_count` (a small dict keyed by count).
  - `get_sb_tendency(team_id, season_id, *, perspective_team_id)` -- opponent's SB attempts/successes from plays + boxscore aggregates; opposing catcher's CS-against rate from caught_stealing events keyed on the catcher's own perspective.
  - `get_first_inning_pattern(team_id, season_id)` -- 1st-inning offensive and defensive run rates from `line_scores`.
  Existing helpers reused (no code change, but documented as matchup consumers): `get_team_games`, `get_pitching_history`, `build_pitcher_profiles`, `get_pitching_workload`, `get_player_spray_events`, `get_players_spray_events_batch` (used by `build_matchup_inputs` to populate `opponent_roster_spray` per AC-3).
- [ ] **AC-15 (perspective-provenance compliance)**: All new helpers that read per-player stat tables (`player_game_batting`, `player_game_pitching`, `spray_charts`, `plays`) MUST filter by `perspective_team_id`. The input builder's docstring explicitly names the perspective used for each query. Reviewer meaningfulness: a reader can identify the perspective for every per-player fact in the dataclass.
- [ ] **AC-16 (data_notes for non-hitter sub-sections)**: Engine emits `data_notes` entries for thin samples outside the top-3 hitters when warranted, each tagged with the appropriate `subsection` value. Suggested thresholds for v1 (engine constants):
  - `subsection="opposing_pitchers"` -- when fewer than 3 opposing pitchers cleared the eligibility heuristic (rest cleared + recent workload), emit a note like "Thin opposing-pitcher pool: only N candidates cleared rest/workload."
  - `subsection="sb_profile"` -- when opponent team has fewer than `_MIN_SB_ATTEMPTS_FOR_PROFILE` (default 5) total SB attempts in the season, emit a note like "Small SB sample: N attempts on the season."
  - `subsection="first_inning"` -- when opponent has played fewer than `_MIN_GAMES_FOR_FIRST_INNING` (default 5) games, emit a note like "Thin first-inning sample: N games."
  - `subsection="loss_recipe"` -- when opponent has fewer than `_MIN_LOSSES_FOR_RECIPE` (default 3) total losses, emit a note like "Small loss sample: N losses on the season."
  - `subsection="lsb_pitchers"` (when `our_team_id` is set) -- when fewer than 3 LSB pitchers cleared eligibility, emit a note like "Thin LSB pitching: only N candidates cleared rest/workload."
  Threshold constants live as module-level `_` -prefixed constants in `src/reports/matchup.py`. The engine MAY emit zero `data_notes` for a given sub-section (sample is sufficient); the renderer uses the absence of a note for that `subsection` to mean "no note to render" (per E-228-14 AC-3). Reviewer meaningfulness: each non-hitter sub-section with a thin sample produces a `data_notes` entry tagged with the correct `subsection` value.

## Technical Approach
The engine is a pure Python function that walks the input dataclass and produces an output dataclass; no clever tricks. The input builder is the standard "fetch all the things" pattern -- many sequential queries, all operating against the same `conn`. Cross-perspective UUID handling: opponent stats use the opponent's `perspective_team_id`; LSB stats (when present) use the LSB team's `perspective_team_id`. The engine never crosses perspectives -- each top-3 hitter's signals are all from a single perspective.

For new helpers in `src/api/db.py`: follow the existing shared-query convention (return `list[dict]`, accept optional `db=None` connection, document the perspective). OPS is computed in SQL with the standard zero-AB guard.

For the loss-recipe classification: walk `get_team_games(opponent, season)` joined in memory with `get_pitching_history(opponent, season)`. For each loss, fetch the starting pitcher's row + sum the bullpen ER, then bucket per the heuristics in Context.
