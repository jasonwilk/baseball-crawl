## Status

## Description
After this story is complete, `src/api/db.py` will contain three new query helpers (`get_top_hitters`, `get_hitter_pitch_tendencies`, `get_pitcher_fps_pct`) that the matchup input builder (E-228-06) consumes. The story REUSES five existing helpers identified by data-engineer rather than reimplementing them. A new `play_events` index is added ONLY if EXPLAIN QUERY PLAN evidence shows table-scan on the proposed pitch-tendency query; otherwise the index is deferred. Each new helper is independently testable, returns a typed structure, filters perspective-aware where applicable, and is scoped strictly by `season_id` and `team_id`.

## Context
Per DE consultation (2026-04-27, relayed via team-lead), the matchup data layer should reuse five already-existing helpers in `src/api/db.py` rather than build duplicates. The "queries fetch facts; the engine derives narratives" principle means loss-recipe classification and close-games filtering belong in the engine (E-228-05), not as separate helpers. This story is significantly smaller than initially scoped: 3 new helpers and ~100 LOC of SQL + tests, plus a conditional index.

### Behavior — New Helpers (3 helpers)

- [ ] **AC-6**: `get_top_hitters(conn, team_id, season_id, *, limit: int = 5, min_pa: int = 10) -> list[HitterRanking]` returns top hitters from `player_season_batting` joined to `players` and `team_rosters` (for jersey_number), filtered by `pa >= min_pa`, ordered by OPS computed in SQL per the formula below. Returns `pa, ab, h, bb, hbp, so, doubles, triples, hr, tb, xbh, sb, cs, ob` plus existing splits columns. OPS computation:
  ```sql
  (CAST(h + COALESCE(bb,0) + COALESCE(hbp,0) AS REAL)
     / NULLIF(ab + COALESCE(bb,0) + COALESCE(hbp,0) + COALESCE(shf,0), 0))
  + (CAST(COALESCE(tb,0) AS REAL) / NULLIF(ab, 0))
  AS ops
  ```
- [ ] **AC-7**: `get_hitter_pitch_tendencies(player_id, season_id, *, perspective_team_id, db=None) -> HitterPitchTendencies` returns: `total_pa_with_plays`, `fps_seen` (count from `plays.is_first_pitch_strike` from the batter's perspective), `two_strike_pa`, `full_count_pa`, swing rate by count, chase signals from `play_events.pitch_result`. Aggregated at the player level. Per DE B-1, indexes EXPLAIN-QUERY-PLAN'd; if table-scan visible on the production-scale fixture, add `idx_play_events_play_first_pitch ON play_events(play_id, is_first_pitch)` in `migrations/003_play_events_first_pitch_index.sql` (003 per `.claude/rules/migrations.md`); otherwise defer the index.
- [ ] **AC-8**: `get_pitcher_fps_pct(team_id, season_id, *, db=None) -> dict[str, dict]` returns per-pitcher FPS% from `plays` (NOT `play_events` -- the boolean `is_first_pitch_strike` is on `plays`). Uses the existing `idx_plays_fps` partial index (`migrations/001_initial_schema.sql:691`). Query MUST include `WHERE outcome NOT IN ('Hit By Pitch', 'Intentional Walk')` to match the partial-index predicate. Returns `{player_id: {fps, total_pa_seen, fps_pct}}`.
- [ ] **AC-8a (helper signature pattern per DE-F3)**: All three new helpers (`get_top_hitters`, `get_hitter_pitch_tendencies`, `get_pitcher_fps_pct`) use the existing `src/api/db.py` connection-management pattern: `*, db: sqlite3.Connection | None = None` keyword-only with internal `own_conn = db is None` lifecycle. Mirrors `get_pitching_history`, `get_pitching_workload` patterns. NOT positional `conn` first arg — that would be inconsistent with the existing module convention.
- [ ] **AC-8b (recent-form helper per DE-F5 + Coach S-4)**: `get_player_recent_form(player_id, season_id, *, last_n_games: int = 7, perspective_team_id: int, db: sqlite3.Connection | None = None) -> RecentForm` is added to `src/api/db.py`. Returns aggregated `pa, ab, h, bb, hbp, obp, slg, games_in_window`. Query reads `player_game_batting WHERE player_id = ? AND perspective_team_id = ? ORDER BY game_date DESC LIMIT N`. **Window is by GAME COUNT, not calendar days** — per project data philosophy. The `RecentForm` dataclass surfaces `games_in_window` so the renderer can render thin-data badges. **MUST filter by `perspective_team_id`** (per perspective-provenance rule); skipping the filter would silently double-count cross-perspective rows.
