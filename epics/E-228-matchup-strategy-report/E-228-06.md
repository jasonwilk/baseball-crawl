## Description
After this story is complete, `src/reports/matchup.py` will contain `build_matchup_inputs(conn, *, opponent_team_id, our_team_id, season_id, reference_date, lsb_starter_override, starter_prediction)` -- the DB-aware function that orchestrates calls to the query helpers from E-228-04 and assembles a populated `MatchupInputs` dataclass for the engine to consume. This is the only DB-aware piece in `src/reports/matchup.py`; the engine remains pure.

## Context
Per Technical Notes section "Engine Architecture (Mirrors Predicted-Starter)", the input builder is where ALL DB lookups happen so `compute_matchup()` can be a pure function. The builder calls the helpers added in E-228-04 once each, packages the results into the `MatchupInputs` dataclass shape defined in E-228-05, and returns it.

The builder also handles the `our_team_id is None` case -- when no LSB context exists, the builder populates LSB-side fields with `None` and the engine downgrades the analysis accordingly (no lineup card, no head-to-head, no LSB-pitcher-aware pitch-attack notes).

## Acceptance Criteria

### Behavior

- [ ] **AC-1**: `build_matchup_inputs(conn, *, opponent_team_id, our_team_id, season_id, reference_date, lsb_starter_override=None, starter_prediction=None) -> MatchupInputs` defined and exported from `src/reports/matchup.py`
- [ ] **AC-2**: Builder calls each E-228-04 helper exactly once with the correct `(team_id, season_id, perspective_team_id, reference_date)` arguments
- [ ] **AC-3**: When `our_team_id` is None, the builder populates `MatchupInputs.lsb_team`, `lsb_pitching`, `lsb_top_hitters`, `head_to_head` as None
- [ ] **AC-4**: When `our_team_id` is provided, the builder runs the LSB-side queries and populates the corresponding fields
- [ ] **AC-5**: Perspective-provenance compliance per DE D: every per-player stat read uses the correct `perspective_team_id`. Specifically:
  - **Opponent stats** (top hitters, pitching workload, hitter pitch tendencies, FPS%): use `perspective_team_id = opponent_team_id`. This requires the opponent's own-perspective boxscores to be loaded (see AC-6 pre-flight gate).
  - **LSB stats**: use `perspective_team_id = our_team_id`.
  - **Head-to-head plays**: pick ONE perspective and stay in it (per DE D-2). Default: LSB perspective on the head-to-head games. Documented in the docstring.
  - **`player_season_batting`** (season aggregates): does NOT include a perspective filter (per DE D-3). The aggregate has no `perspective_team_id` column; perspective is filtered at compute time. The builder's calls to `get_top_hitters` (E-228-04 AC-11) reflect this.
- [ ] **AC-6 (pre-flight gate per DE D-1)**: Before building the input, the builder runs a coverage check: query `game_perspectives` for rows matching `(game_id IN <opponent's games>, perspective_team_id = opponent_team_id)`; count the games with opponent-perspective coverage. When coverage < `_MIN_OPPONENT_PERSPECTIVE_COVERAGE` (module-level constant, default 0.5 -- coach-tunable), the builder REFUSES to construct full inputs and returns a `MatchupInputs` with `data_completeness.opponent_games_with_own_perspective_count` populated but `opponent_top_hitters` / `opponent_pitching` set to empty lists, accompanied by a `data_notes` flag explaining the refusal. The engine then returns `confidence="suppress"` and the matchup section degrades cleanly. The narrative layer (E-228-07) surfaces the gap: "Insufficient opponent-perspective game coverage to build a matchup analysis."
- [ ] **AC-7**: When `lsb_starter_override` is a player_id, the builder validates the player exists on the LSB roster and resolves the player's display name; on mismatch the builder logs a WARNING and clears the override (does NOT pass an unknown name to the engine)
- [ ] **AC-8**: When `starter_prediction` is provided, the builder passes it through into `MatchupInputs.starter_prediction`; when None, the field is None
- [ ] **AC-9**: Builder populates `MatchupInputs.data_completeness` per DE H-1 with the FIVE fields per E-228-05 AC-8b:
  - `opponent_games_with_own_perspective_count` (count from `game_perspectives`)
  - `opponent_games_total` (from `get_team_games`)
  - `games_missing_opponent_perspective: list[str]` (per DE-F2 partial-coverage tracking — list of `game_id` values that lack opponent-perspective coverage; narrative renders "loss-recipe based on N of M losses — K games lack opponent-perspective coverage")
  - `scouting_last_run_at: str | None` — sourced from `SELECT MAX(completed_at) FROM scouting_runs WHERE team_id = ? AND season_id = ? AND status = 'completed'`. Per-scout-run timestamp is more accurate than `teams.last_synced` because partial scouts update last_synced but the coverage gap may persist.
  - `plays_loaded_per_game: dict[str, bool]` (mapping built by checking `game_perspectives` for plays-row presence per game)

## Technical Approach
- The builder is roughly 80-150 lines of orchestration: call each helper, populate dataclass fields, handle None/empty cases. Linear and coherent per python-style rule -- no need to extract sub-helpers unless one is reused.
- For LSB team profile resolution, query `teams` table directly with `our_team_id`. Don't add a new helper for this -- direct SQL is fine.
- For `lsb_starter_override` resolution: the parameter is a string (could be a player_id UUID or a fuzzy name); E-228-01 didn't fully define the resolution scheme. SE judgment -- recommend exact `player_id` match first, fall back to first-name + last-name LIKE matching with a single-result requirement.
