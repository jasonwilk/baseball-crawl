## Description
After this story is complete, `generate_report()` in `src/reports/generator.py` will orchestrate the full matchup pipeline: when `our_team_id` is provided AND the feature flag is on, the generator runs a fresh in-memory scout for the LSB team, builds matchup inputs after reconciliation completes, calls the deterministic engine, optionally calls the LLM Tier 2 wrapper, and passes the resulting analysis dataclass to the renderer. Backward compatibility is preserved: when `our_team_id` is None, the existing flow runs unchanged.

## Context
Per Technical Notes section "Architectural Path: C" and the existing predicted-starter integration at `generator.py:1150-1209`, the matchup orchestration block sits parallel to the predicted-starter block -- after reconciliation, before render. The block is gated by the same pattern: feature flag check, `our_team_id` non-None check, `is_llm_available()` check for Tier 2.

The key sequencing concern is the LSB-side scout: when `our_team_id` is provided, we need fresh LSB data. The scouting pipeline (`run_scouting_sync` / scouting crawler+loader) is the right tool, invoked here with the LSB team's `public_id`. This is the "fresh-on-demand" decision the user locked in -- no caching threshold logic, just run it every time.

## Acceptance Criteria

### Behavior

- [ ] **AC-1**: `generate_report()` accepts the parameters defined in E-228-01/E-228-03 with their established defaults
- [ ] **AC-2**: When `our_team_id` is provided AND `is_matchup_enabled()` returns True, the generator runs the orchestration sequence: resolve LSB public_id → run LSB scout → load LSB data → `build_matchup_inputs(...)` → `compute_matchup(...)` → optional `enrich_matchup(...)` (inside try/except, mirroring `generator.py:1186-1209`) → pass both `matchup` and `enriched_matchup` to the renderer
- [ ] **AC-3**: When `our_team_id` is None, the matchup orchestration block is fully skipped

- [ ] **AC-5**: The matchup orchestration block runs AFTER reconciliation (so plays data is corrected) but BEFORE the render block
- [ ] **AC-6**: `CredentialExpiredError` raised during the LSB scout is caught and logged; the report still completes
- [ ] **AC-7 (title format with explicit cases per PM-7 + SE-C5)**: Report title format depends on which checkboxes were on:

## Technical Approach
- Read `generator.py:1150-1209` carefully -- the predicted-starter block is the structural template. The matchup block uses the same conditional layering (`if show_X and our_Y is not None:`) and the same try/except for the LLM tier.
- The LSB scout is a separate `ScoutingCrawler.scout_team()` invocation -- ONE additional pipeline run during generation. SE judgment whether to factor a shared helper (`_run_scout_for_team(public_id)`) out of the existing inline scout code or duplicate.
- The fresh-scout-on-generation user-facing time cost (~2 minutes per the user's spec) is the operator's responsibility; UX expectations are set in the admin form (E-228-09 task -- placeholder warning text).
