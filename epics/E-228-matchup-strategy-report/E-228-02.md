## Description
After this story is complete, both `run_scouting_sync` (web background pipeline) and `_scout_live` (CLI) will run plays crawl + load + reconcile after the spray load step, restoring the scouting-pipeline-parity invariant declared in CLAUDE.md. Tracked opponents will have plays data available for any downstream feature that needs FPS%, QAB%, or pitcher-of-record reconciliation -- including the matchup engine (E-228-04+) but also the existing dashboard.

## Context
Per the user's verification in the discovery transcript (2026-04-27), today's scouting pipeline runs five stages (scouting crawl + load, gc_uuid resolve, spray crawl + load, post-spray dedup) but skips plays. Plays runs only via the standalone `bb data crawl --crawler plays` + `bb data load --loader plays` command pair. The CLAUDE.md scouting-pipeline-parity invariant ("`run_scouting_sync` (web) and `bb data scout` (CLI) MUST produce equivalent data artifacts -- all five stages in both paths") is currently violated for plays.

The standalone reports flow ALREADY runs plays inline (lines 1091-1110 of `src/reports/generator.py`) so this story is purely about the dashboard/tracked-team pipeline parity. The matchup engine reads plays-derived signals (per E-228-04) for its hitter pitch-tendency component, so without this story the matchup engine cannot produce hitter pitch tendencies for tracked opponents that haven't been individually report-generated.

## Technical Approach
- Reuse the report-flow's existing helper if it can be extracted from `src/reports/generator.py`; if not, reimplement the same shape in `src/pipeline/trigger.py` and `src/cli/data.py`. SE chooses extract vs duplicate based on coupling.
- The plays crawl entry point is `PlaysCrawler` (per `architecture-subsystems.md` Plays Pipeline section), the loader is `PlaysLoader`, and the reconcile entry is `reconcile_game(conn, game_id, dry_run=False)`.
- For idempotency, rely on the loader's existing whole-game check (`SELECT 1 FROM plays WHERE game_id = ? AND perspective_team_id = ? LIMIT 1`).
- For auth handling, mirror lines 1099-1110 of `generator.py` exactly: catch `CredentialExpiredError`, log warning, continue. Catch broad `Exception` for unexpected per-game failures, log with `exc_info=True`.
