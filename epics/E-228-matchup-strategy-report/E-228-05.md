## Description
After this story is complete, `src/reports/matchup.py` will contain the pure-function deterministic matchup engine -- `compute_matchup(inputs: MatchupInputs) -> MatchupAnalysis` -- plus all output dataclasses (`MatchupAnalysis`, `ThreatHitter`, `PitcherPick`, `ShiftRecommendation`, `LineupSlot`, `BullpenPatternSummary`, `HeadToHeadSummary`, `DataDepthBadge`, `StatCitation`). The engine takes pre-resolved inputs, applies coach-locked logic (top-hitter ranking, three-gate starter prediction, shift derivation, lineup ranking, loss-recipe classification, data-depth badging), and returns a typed dataclass with NO DB or HTTP access.

## Context
Per Technical Notes section "Pure-function discipline" and SE's recommendation, the engine mirrors `compute_starter_prediction()` (lines 844-1073 of `starter_prediction.py`) -- pure function, takes pre-fetched dicts, returns a typed dataclass. This shape is what makes engine tests fast and DB-free, and it's what enables the test-fixture pattern (`tests/test_starter_prediction.py`) to be replicated cleanly here.

The engine is the centerpiece of the epic but the simplest piece architecturally -- given inputs, produce outputs. Coach-defined thresholds live as module-level constants (mirroring `_PRIMARY_STARTER_GS_RATIO`, `_HIGH_CONFIDENCE_RATIO`, etc. in `starter_prediction.py`) so they can be tuned post-ship without restructuring code.

## Acceptance Criteria

### Behavior

- [ ] **AC-1**: `MatchupInputs` and `MatchupAnalysis` dataclasses defined per Technical Notes "MatchupInputs / MatchupAnalysis Dataclass Shape". `MatchupAnalysis` has these fields: `confidence`, `threat_list`, `pitcher_recommendation`, `defensive_shifts`, `lineup_card`, `loss_recipes`, `bullpen_pattern`, `head_to_head_summary`, `tldr_directives: list[str]`, `data_notes`, `data_completeness: DataCompleteness`, `suppress_section`. Sub-types: `ThreatHitter` (with `bb_pct: float` REQUIRED for TL;DR slot 3 detection), `PitcherPick`, `ShiftRecommendation`, `LineupSlot`, `BullpenPatternSummary`, `HeadToHeadSummary`, `DataDepthBadge`, `StatCitation`, `DataCompleteness` (with fields `opponent_games_with_own_perspective_count: int`, `opponent_games_total: int`, `games_missing_opponent_perspective: list[str]`, `scouting_last_run_at: str | None`, `plays_loaded_per_game: dict[str, bool]` per Technical Notes).
- [ ] **AC-2**: `compute_matchup(inputs: MatchupInputs) -> MatchupAnalysis` is a pure function -- no DB calls (no `sqlite3` import in engine code paths), no HTTP calls, no filesystem access; verified by code review (search the module for `sqlite3`, `httpx`, `requests`, `open(`)
- [ ] **AC-3**: Engine produces `confidence="suppress"` when `inputs.opponent_top_hitters` is empty AND `inputs.opponent_pitching` is empty
- [ ] **AC-4**: Top-hitter ranking orders `inputs.opponent_top_hitters` by OPS descending; ties broken by PA descending
- [ ] **AC-5**: Three-gate starter prediction logic per Technical Notes:
  - `lsb_starter_override` set → recommendation uses override; gates bypassed
  - override None → gates run: G1 (rest cleared), G2 (≥2 GS), G3 (K/BB within 30% of `target_k_per_bb`)
  - G3 formula: `target_k_per_bb = mean(K/BB ratio of opposing-team's-loss-pitchers from inputs.opponent_losses)`. **K/BB (strikeouts-per-walk) is the canonical command-efficiency metric here, NOT K/9.** Variable name MUST be `target_k_per_bb` (no K/9 nomenclature mixing); module-level constant `_RECIPE_K_PER_BB_TOLERANCE = 0.30`.
  - All three pass → `pitcher_recommendation` populated with chosen pitcher + 3-criteria reasoning
  - Any gate fails → `pitcher_recommendation = None`
- [ ] **AC-6**: Shift derivation: for each top-hitter with `pull_pct >= _PULL_SHIFT_THRESHOLD` (default 0.55), engine emits a `ShiftRecommendation` with direction, sample size, and `supporting_stats`
- [ ] **AC-7**: Lineup-card ranking (when `inputs.lsb_top_hitters` non-None): produces a 9-slot lineup with leadoff = best OBP, 3-hole = best SLG, 2-hole = second OBP, remaining slots fill in OPS order. When fewer than 9 hitters available, truncate
- [ ] **AC-8**: Loss-recipe classification operates on `get_team_games(opponent_team_id, season_id)` joined in memory with `get_pitching_history(opponent_team_id, season_id)` outputs (per DE F). Emits one of FIVE buckets per opponent loss (HS-calibrated thresholds per Coach S-1, S-2):
  | Bucket | Heuristic |
  |--------|-----------|
  | `starter_blew_up` | starter `ip_outs < 12` AND starter `er >= 4` AND `decision = 'L'` on starter |
  | `pitching_duel_lost_late` | starter `ip_outs >= 18` AND starter `er <= 2` AND `decision = 'L'` on a reliever |
  | `bullpen_burn` | starter `ip_outs >= 12` AND `COUNT(DISTINCT pitchers) >= 3` AND total pen `er >= 3` |
  | `got_blasted` | total team `er >= 6` OR `abs(margin) >= 6` |
  | `close_loss` | `abs(margin) <= 2` AND none of the above match |
  Thresholds are HS-calibrated and live as module-level constants per Technical Notes for post-ship tuning. **`bullpen_burn` uses ≥3 distinct pitchers (NOT ≥4 MLB-norm)** because HS reality is starter+2-relievers as the bullpen-burn pattern. **`got_blasted` uses ≥6 (NOT ≥8)** because a 5-run loss is a HS blowout.
- [ ] **AC-8a (decision-NULL fallback per DE E)**: When `player_game_pitching.decision IS NULL` for an opponent loss (data-quality flag), the loss is classified as `decision_unknown` (sixth bucket) AND a flag entry is added to `MatchupAnalysis.data_notes`. Engine MUST NOT guess. The narrative layer (E-228-07) surfaces the gap to coach.
- [ ] **AC-8b (data_completeness block per DE H-1 + DE-F2 partial-coverage)**: `MatchupAnalysis.data_completeness` populated with: `opponent_games_with_own_perspective_count: int`, `opponent_games_total: int` (from `get_team_games`), `games_missing_opponent_perspective: list[str]` (per DE-F2 partial-coverage tracking — per-game ID list so narrative can say "loss-recipe based on 14 of 18 losses — 4 games lack opponent-perspective coverage"), `scouting_last_run_at: str | None` (passed in via input from `scouting_runs.completed_at` for the most recent completed run; per-scout-run timestamp is more accurate than `teams.last_synced` because partial scouts update last_synced but coverage gap may persist), `plays_loaded_per_game: dict[str, bool]`. The narrative layer surfaces gaps to coach.
- [ ] **AC-9**: Data-depth badging: engine emits a `DataDepthBadge` per thin-data component; badges live in `MatchupAnalysis.data_notes`
- [ ] **AC-10**: Suppress vs partial:
  - `confidence="suppress"` ONLY when neither hitters nor pitchers present
  - Otherwise `confidence` is `high` / `moderate` / `low` and every component renders with badges
- [ ] **AC-11**: All recommendation types carry `supporting_stats: list[StatCitation]` with `(label, value, sample_size)` tuples
- [ ] **AC-12 (deterministic TL;DR strings per UX, decision B-aware, Coach P2-T Option B)**: `MatchupAnalysis.tldr_directives: list[str]` is populated by the engine with up to three directive strings for the renderer to compose with middot separators. The engine produces:
  - **Slot 1** (LSB starter): `f"Pitch {pitcher_recommendation.name}"` when `pitcher_recommendation is not None`; OMITTED when None.
  - **Slot 2** (top hitter to attack): selected from FOUR coach-locked directive templates per Coach P2-T Option B. Engine evaluates the top entry of `threat_list` against the templates in priority order (1 → 2 → 3 → 4); first match wins. Default fallback: Template 3 ("Attack {name} early"). Always populated when `threat_list` has ≥1 entry.

    | Priority | Template | Condition | Profile |
    |---------|----------|-----------|---------|
    | 1 | `f"Pitch around {name}"` | `bb_pct > _PITCH_AROUND_BB_PCT (0.12) AND slg > _PITCH_AROUND_SLG (0.500)` | Dangerous + patient — walks more than he strikes out AND hits for extra bases. Do not compete in close games. |
    | 2 | `f"Make {name} earn it"` | `bb_pct > _MAKE_EARN_BB_PCT (0.12) AND slg <= 0.500` | Patient table-setter, not power. Work the count but challenge — different from pitch-around because he can't beat you alone. |
    | 3 | `f"Attack {name} early"` | `fps_swing_rate > _ATTACK_EARLY_FPS_SWING (0.50) OR bb_pct < _ATTACK_EARLY_BB_PCT (0.05)` | First-pitch swinger or free hitter who doesn't work counts. Go after him. |
    | 4 | `f"Expand on {name}"` | `k_pct > _EXPAND_K_PCT (0.22) AND bb_pct >= 0.05 AND bb_pct <= 0.12` | Chaser who strikes out but has some discipline. Challenge zone, then expand off-speed late in count. |

    **Priority tie-breaker**: Templates evaluated in order 1 → 2 → 3 → 4; first match wins. Template 1 takes precedence over Template 3 even when the hitter ALSO has high `fps_swing_rate` — the walk signal dominates the power level. (Per Coach P2-T: a patient hitter is a patient hitter regardless of power.)

    **Missing-data fallback**: when `fps_swing_rate` is unavailable for a hitter (no plays data — `HitterProfile.fps_swing_rate is None`), engine evaluates only against Templates 1 + 2 (`bb_pct`/`slg` only). Templates 3 + 4 (which require `fps_swing_rate` or `k_pct`) are skipped. If neither matches, fall to default ("Attack {name} early").

    **Hard 25-char constraint**: directive string MUST be ≤25 characters total including the rendered name. If the directive would exceed 25 chars, engine truncates to last name only (e.g., "Pitch around Vandermolen" 24 chars OK; "Make Smith-Johnson earn it" 26 chars → truncate to last-name → "Make Smith-Johnson earn it" still 26 → no further truncation possible, accept with warning logged). Implementation: try full name first, fall back to last name on overflow. The `_TLDR_DIRECTIVE_MAX_CHARS = 25` constant is module-level.

  - **Slot 3** (avoid walking): `f"Avoid walking {hitter.name}"` when `threat_list` has at least one hitter with `bb_pct >= _AVOID_WALK_BB_PCT_THRESHOLD` (constant, default 0.10); OMITTED otherwise. Same 25-char constraint as Slot 2; truncate name if needed.

  The engine produces the strings; the renderer composes via middot. **CRITICAL**: TL;DR is deterministic and survives LLM Tier 2 failure -- the engine is the single source of truth for these strings.

## Technical Approach
- Module-level constants for tunable thresholds: `_PULL_SHIFT_THRESHOLD = 0.55`, `_RECIPE_K_PER_BB_TOLERANCE = 0.30`, `_TOP_HITTER_PA_FLOOR = 10`, `_MIN_OPPONENT_GAMES_FOR_FULL = 8` (etc.). Mirror the constant block at lines 35-52 of `starter_prediction.py`.
- The engine reuses `_is_excluded()` (or its public equivalent) from `starter_prediction.py` for G1; do NOT duplicate the rest-tier logic.
- For the lineup-card slots, the engine returns slots in batting-order position (slot 1 = leadoff), each with a player reference + reasoning string. The LLM later wraps this with prose.
- For the suppress path, the engine returns a `MatchupAnalysis` with `confidence="suppress"`, empty lists for all components, and a `data_notes` list containing one entry explaining the suppression reason.
