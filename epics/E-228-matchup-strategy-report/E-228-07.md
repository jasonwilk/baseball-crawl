## Description
After this story is complete, `src/reports/llm_matchup.py` will contain the Tier 2 LLM wrapper that takes a `MatchupAnalysis` dataclass + auxiliary context (the predicted-starter dataclass for consistency, NSAA rest table) and produces a coach-voice prescriptive narrative. The wrapper mirrors the structure of `src/reports/llm_analysis.py` exactly: ASCII-table prompt construction, strict JSON output schema, error handling, deterministic temperature.

## Context
Per Technical Notes section "LLM Prompt Design" and SE's discovery transcript, the matchup LLM wrapper is the most prescriptive content the project has produced. The system prompt enforces three rules that don't exist in the predicted-starter prompt:

1. **Hallucination ban** -- "Do NOT invent statistics, names, or game results not present in the structured data above." Player names round-trip dataclass → LLM → renderer; tests verify.
2. **Inline citation requirement** -- for the three coach-locked recommendation categories (pitcher choice, pitch-attack per hitter, defensive shifts), every claim must embed an inline parenthetical citing the supporting data point.
3. **Inline avoid pairing** -- when a hitter has high-OBP-where-pitcher-instinct-is-to-nibble, the prose appends an inline "Do not" clause; max 2-3 per hitter; never a separate section.

The wrapper's structure mirrors `llm_analysis.py:40-283` line-by-line so that future maintenance touches both modules identically.

## Acceptance Criteria

### Behavior

- [ ] **AC-1**: `enrich_matchup(analysis: MatchupAnalysis, *, predicted_starter: StarterPrediction | None, reference_date: date | None = None) -> EnrichedMatchup` defined in `src/reports/llm_matchup.py`; `EnrichedMatchup` dataclass wraps `MatchupAnalysis` with `narrative_sections: dict[str, str]` and `model_used: str`
- [ ] **AC-2**: System prompt template (`_SYSTEM_PROMPT_TEMPLATE` constant) embeds NSAA rest table verbatim via `format_nsaa_rest_table()` and contains identity / output schema / hallucination ban / voice rules / inline citation requirement / avoid-clause rule per Technical Notes
- [ ] **AC-3**: User prompt construction renders `MatchupInputs` and `MatchupAnalysis` as ASCII tables (mirroring `_format_pitcher_table()` in `llm_analysis.py`); tables include threat list + spray + tendencies, opposing pitching, loss recipes, head-to-head summary (when present), LSB pitching (when present); includes predicted-starter dataclass output verbatim when provided
- [ ] **AC-4**: OpenRouter call uses `model=os.environ.get("OPENROUTER_MODEL", "anthropic/claude-haiku-4-5-20251001")`, `max_tokens=1500`, `temperature=0.3`
- [ ] **AC-5**: Suppress short-circuit: when `analysis.confidence == "suppress"`, the wrapper does NOT call OpenRouter; returns an `EnrichedMatchup` with empty `narrative_sections` and a structural marker the renderer can detect
- [ ] **AC-6**: Response parsing: parses with `json.loads()`, validates each required field is present and a string, raises `LLMError` on structural failure (mirrors `llm_analysis.py` lines 252-275)
- [ ] **AC-7 (Hallucination guardrail policy with full reference set per SE-H2)**: when LLM output contains a player name NOT present in `analysis`'s reference set, wrapper applies the policy chosen via `FEATURE_MATCHUP_STRICT` env var. **Reference set is the union of all dataclass-field names that could legitimately appear in narrative**: `threat_list[*].name + pitcher_recommendation.name + lineup_card[*].name + loss_recipes[*].pitcher_of_record_name + bullpen_pattern[*].name + head_to_head_summary[*].lsb_starter_name`. Names from any of those collections are valid; names absent from ALL are hallucinations. Policy: when `FEATURE_MATCHUP_STRICT=1`, raise `LLMError` (fail-loud); when unset, filter the offending recommendation field while preserving the rest of the response (fail-graceful). Document chosen behavior in the docstring.
- [ ] **AC-7b (HTML sanitization per P1-N + CR-F-09-5)**: BEFORE returning `narrative_sections` to the caller, the wrapper strips ALL HTML tags from each narrative-section string. Use `bleach.clean(text, tags=[], strip=True)` or equivalent. This prevents XSS surface when E-228-09's renderer applies `|safe` filter to the prose for citation parenthetical preservation. The wrapper is the single sanitization point — renderer trusts the wrapper's output. Test asserts: an LLM response containing `<script>alert(1)</script>` is stripped to plain text before reaching the renderer.

## Technical Approach
- The system prompt is the most important piece of this story. Treat it as a structured asset: the rules are explicit, numbered, and the JSON schema is exhaustive. Spend prompt-tuning effort here, NOT in the user prompt.
- For the user prompt, reuse the existing `_format_pitcher_table()` pattern -- separate `_format_threat_table()`, `_format_loss_recipe_table()`, `_format_shift_table()`, etc. helpers, each returning a string. Concatenate in `_build_user_prompt()`.
- For the hallucination guardrail, the simplest implementation is a post-parse validation: extract all proper-noun-looking strings (length>2, capitalized) from each narrative field and check membership against the set of names from `MatchupAnalysis.threat_list`, `pitcher_recommendation`, `lineup_card`, etc. SE may choose a more sophisticated approach (e.g., regex) but the AC requires the test to pass on the specific case.
