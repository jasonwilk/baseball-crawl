## Epic
[E-228: Matchup Strategy Report](./epic.md)

## Status

## Description
After this story is complete, the reports system will accept an optional "us team" context that travels through the schema, the generator entry point, the admin generation form, and the CLI -- WITHOUT yet rendering any matchup content. The plumbing exists end-to-end with `our_team_id` round-tripping from form submission to DB persistence to a generator parameter, and an empty-but-detectable hook in `generate_report()` where downstream stories will inject matchup logic. Backward compatibility is the most important deliverable: a report generated WITHOUT the matchup option must produce byte-identical output to today.

## Context
This story is the architectural foundation. Per Technical Notes section "Architectural Path: C" and the SE consultation transcript (2026-04-27), Path C is the lowest-risk lift -- one schema column + one generator parameter + one form field + one CLI flag. Everything else in the epic layers on top of this contract.

The "us team" dropdown source (`teams WHERE membership_type='member'`) and the optional starting-pitcher input field are added in this story but not yet consumed -- they're persisted as form fields and passed through to `generate_report()`, where downstream stories use them. The "Include pitcher predictions" checkbox is OUT of scope for this story; it ships in E-228-03.

- [ ] **AC-2a (`our_team_id` resolution layer per SE-C1)**: Public_id-or-int resolution lives in the CALLERS, not in `generate_report()`. Specifically:

## Technical Approach
- The migration is one SQL statement: `ALTER TABLE reports ADD COLUMN our_team_id INTEGER REFERENCES teams(id);`. SQLite ALTER TABLE ADD COLUMN is safe per `.claude/rules/migrations.md`.
- For the generator parameter, the existing `generate_report(gc_url: str)` signature gains keyword-only parameters. The persistence happens in the existing `_create_report_row()` helper -- pass `our_team_id` through and add the column to the INSERT.
- For the admin form, use the UX-specified vertical-stack layout with vanilla JavaScript (~4 lines per AC-5c). Existing CSS palette (`bg-gray-50`, `border-gray-200`, `text-xs text-gray-600`, etc.) per UX. The "Sections" group container is the existing pattern; the new content is the checkbox + indented dropdown wrapper + LSB starter input.
- For the CLI flag, Typer accepts optional flags via `typer.Option(None, "--our-team", help="...")`.
- For backward-compat regression, generate a baseline report on the pre-migration codebase and check it into `tests/fixtures/baseline_scouting_report.html`. Regression test renders a report against an identical fixture DB and asserts equality (modulo `generated_at` timestamp).
- The feature flag check should happen in three places: form rendering, CLI flag parsing, generator entry. A single helper (`is_matchup_enabled()` in `src/reports/matchup.py`) centralizes the env-var read; created in this story even though the rest of `matchup.py` ships in E-228-05.
- **Decision B note**: this story scaffolds the form so E-228-03's "Include pitcher predictions" checkbox slots into the established "Sections" group with no extra structural change. E-228-03 only adds the second checkbox + the gating in `generate_report()`.
