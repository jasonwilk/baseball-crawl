# E-228-01: Path C lift -- schema, generator parameter, admin checkbox, CLI flag

## Epic
[E-228: Matchup Strategy Report (v1)](./epic.md)

## Status
`TODO`

## Description
After this story is complete, the reports system will accept an optional `our_team_id` context that travels through the schema, the generator entry point, the admin generation form, and the CLI -- WITHOUT yet rendering any matchup content. The plumbing exists end-to-end with `our_team_id` round-tripping from form submission to DB persistence to a generator parameter. Backward compatibility is the most important deliverable: a report generated WITHOUT the matchup option must produce equivalent output to today (regression test passes).

## Context
This story is the architectural foundation. Per Technical Notes section "Architectural Path: C" and the SE consultation transcript (2026-04-27), Path C is the lowest-risk lift -- one schema column + one generator parameter + one form field + one CLI flag. Everything else in the epic layers on top of this contract.

The "us team" dropdown source is `teams WHERE membership_type='member'`. v1 has a single new "Include matchup section" checkbox; the predicted-starter checkbox split (originally planned as E-228-03) was dropped from v1 -- predicted-starter remains always-on as it ships today.

## Acceptance Criteria

### Behavior

- [ ] **AC-1**: Migration `migrations/002_add_our_team_id_to_reports.sql` adds `our_team_id INTEGER REFERENCES teams(id)` to the `reports` table as nullable; `apply_migrations.py` runs cleanly and existing rows have `our_team_id IS NULL` after migration. Migration number is 002, the next available per `.claude/rules/migrations.md`.
- [ ] **AC-2**: `generate_report()` in `src/reports/generator.py` accepts `our_team_id: int | None = None` as a new keyword-only parameter, defaulting to None. `our_team_id` is always an int (never a string); resolution from public_id slug or string-id happens in CALLERS (CLI + admin handler), NOT inside `generate_report()`.
- [ ] **AC-2a (resolution layer)**: Public_id-or-int resolution lives in the callers:
  - **CLI** (`src/cli/report.py`): a helper resolves the `--our-team` argument by trying integer parse first, falling back to a `public_id` lookup against `teams WHERE membership_type='member'`. Returns `None` if no match (CLI exits non-zero with a clear error message).
  - **Admin route handler** (`src/api/routes/admin.py`): the dropdown submits the team's INTEGER `id` directly; no string resolution needed.
  - `generate_report()` receives `our_team_id: int | None` only.
- [ ] **AC-3**: When `our_team_id` is non-None and the matchup section is enabled, the report row is created with `our_team_id` populated; verified by direct SQL query against the `reports` table.
- [ ] **AC-4 (backward-compat regression)**: A report generated with `our_team_id=None` produces output equivalent to a report generated against the same fixture DB on the pre-migration codebase, after deterministic substitution of non-deterministic fields (`generated_at`, `expires_at`, `slug`). The exact regression mechanism (snapshot text comparison, structural element-by-element check, or similar) is the implementer's choice; the test must FAIL LOUDLY when the existing single-team scouting report layout changes. Note: the heavier byte-identical baseline-fixture machinery (substitution scheme, regen script, checked-in baseline HTML) is deferred to E-231; v1 ships with a lighter regression check that proves "matchup-off path produces the same report it does today."
- [ ] **AC-5 (admin form -- single matchup checkbox)**: Admin form (`src/api/templates/admin/reports.html`) gains an "Include matchup section" checkbox, default unchecked. When checked, a `<select>` dropdown reveals (vanilla JS, ~4 lines) populated from `SELECT id, name FROM teams WHERE membership_type='member' ORDER BY name`, with help text "Stats from this team will be used to build the matchup section." The dropdown is `required` when the checkbox is checked (server-side validation re-renders the form with the existing red error flash if the checkbox is checked but the dropdown is empty).
- [ ] **AC-5a (empty member-team edge case)**: When zero `membership_type='member'` rows exist, the matchup checkbox is rendered with the `disabled` attribute AND a help text "Add a member team in /admin/teams to enable matchup analysis." replaces the dropdown. Route handler passes `has_member_teams: bool` into the template context.
- [ ] **AC-6 (form POST handler)**: Admin form POST handler in `src/api/routes/admin.py` parses the new `our_team_id` field (from the dropdown when matchup is checked) and passes it to `generate_report()`. Server-side validation: when matchup is checked, `our_team_id` must be a non-empty value matching one of the member-team rows. Existing form behavior unchanged when the matchup checkbox is unchecked.
- [ ] **AC-7 (CLI flag)**: `bb report generate <gc_url>` accepts `--our-team <public_id_or_id>` as a new optional flag. Omitting the flag preserves today's exact behavior.
- [ ] **AC-8 (CLI flag resolution)**: `--our-team` value resolution: try `int(value)` first, fall back to `teams.public_id` lookup; on neither match, exit non-zero with a clear error message naming the unknown value and pointing the operator at `bb status` to list members.
- [ ] **AC-9 (feature flag)**: When env `FEATURE_MATCHUP_ANALYSIS` is unset or evaluates falsy ("0", "false", "no", ""), the admin checkbox is hidden, the CLI flag emits a "feature disabled" warning and proceeds without matchup, and `generate_report()` ignores `our_team_id` even if passed.
- [ ] **AC-10 (`is_matchup_enabled()` helper)**: `src/reports/matchup.py::is_matchup_enabled() -> bool` returns True when `os.environ.get("FEATURE_MATCHUP_ANALYSIS", "").lower() in ("1", "true", "yes")` (mirrors `is_predicted_starter_enabled()`). This is the single helper used by all three feature-flag checks (form rendering, CLI flag parsing, generator entry).

- [ ] **AC-T2 (signature + persistence)**: Test calls `generate_report(..., our_team_id=42)` and asserts `SELECT our_team_id FROM reports WHERE id = ?` returns `42`; companion test with `our_team_id=None` asserts the column is NULL.
- [ ] **AC-T3 (backward-compat regression)**: Test generates a report against a fixture DB with `our_team_id=None`, applies deterministic substitution of `generated_at` / `expires_at` / `slug`, and asserts the output matches a snapshot/baseline. - [ ] **AC-T4 (admin form rendering -- matchup checkbox)**: Test covers (a) `FEATURE_MATCHUP_ANALYSIS=1` form GET renders the matchup checkbox + the conditional dropdown wrapper (initially hidden) + help text + dropdown options matching `SELECT id, name FROM teams WHERE membership_type='member' ORDER BY name`; (b) `FEATURE_MATCHUP_ANALYSIS=0` form GET does NOT render the matchup checkbox at all; (c) form POST without matchup checked passes `our_team_id=None` to a mocked `generate_report()`; (d) form POST with matchup checked AND a valid dropdown selection passes the resolved integer `our_team_id`; (e) form POST with matchup checked AND empty dropdown re-renders the form with an error flash (server-side validation, `generate_report()` not called). - [ ] **AC-T7 (feature-flag)**: Test sets `FEATURE_MATCHUP_ANALYSIS=0`, calls `generate_report(..., our_team_id=42)`, asserts the report generation succeeds AND `our_team_id` is treated as if None (the documented behavior is captured in the helper's docstring).

## Technical Approach
The migration is one ALTER TABLE ADD COLUMN statement, safe per `.claude/rules/migrations.md`. The generator parameter is a keyword-only addition; persistence happens in the existing report-row INSERT path with the new column included. The admin form uses a vanilla-JS reveal pattern (no framework). The CLI flag uses Typer's optional flag pattern. The feature flag check happens in three places (form rendering, CLI flag parsing, generator entry) and is centralized via a single helper in `src/reports/matchup.py`. The `matchup.py` module is created in this story with the `is_matchup_enabled()` helper as its first content; the engine + dataclasses ship in E-228-12.

For the backward-compat regression test, the implementer chooses the comparison mechanism (snapshot, structural diff, etc.). The hardened byte-identical machinery from the original plan is deferred to E-231; v1 ships with a lighter check sufficient to catch layout regressions.
