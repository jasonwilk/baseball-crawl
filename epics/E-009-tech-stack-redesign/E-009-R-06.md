## Research Question

When agents in this project need to access git history, GitHub data, or repository
metadata, is the `gh` CLI sufficient -- or is there a scenario where an MCP-based git
integration (such as abhigyanpatwari/GitNexus) would provide meaningfully better access?

This is **awareness and fit-assessment research only**. No tool adoption decision is made
here. The output is a factual report comparing the two approaches.

## Context

### Current State

The project is not yet on GitHub. There is no remote repository. Agents currently interact
with git via the Bash tool (e.g., `git log`, `git status`, `git diff`). This is functional
but limited -- agents have no structured way to query repository metadata, file history,
or cross-file change patterns.

When the project eventually has a remote GitHub repository, agents may need to:
- Query commit history for a specific file or directory
- Identify which files changed in a given story's implementation
- Cross-reference epic/story IDs in commit messages against story statuses
- Understand the shape of recent changes without reading every diff

The question is: when that time comes, how should agents get this data -- via `gh` CLI
calls in a bash subprocess, or via an MCP server that exposes git/GitHub data as a
structured tool?

### The Two Approaches to Evaluate

#### Approach 1: `gh` CLI (GitHub official CLI)
- GitHub: https://github.com/cli/cli
- What it is: The official GitHub CLI. Supports querying issues, PRs, commits, repos,
  actions, gists, and more. Outputs JSON (with `--json` flag) or human-readable text.
- How agents use it: Via the Bash tool (`gh repo view`, `gh pr list --json`, etc.).
  Output is parsed by the agent as text or JSON.
- Already available: `gh` is a widely installed CLI. No additional configuration needed
  beyond authentication.

#### Approach 2: abhigyanpatwari/GitNexus (MCP server for git)
- GitHub: https://github.com/abhigyanpatwari/GitNexus
- What it claims: MCP server that exposes git repository data (commits, branches, file
  history, blame, diff, etc.) as structured MCP tools. Allows an agent to query git
  history via typed tool calls rather than parsing bash output.
- How agents use it: Registered in `.claude/settings.json`. Called as a tool directly,
  with structured inputs and structured JSON output.

Note: If GitNexus turns out to be sparse, unmaintained, or hard to evaluate, the
researcher should document that finding and assess whether any other MCP git server
(e.g., those in the MCP ecosystem catalog) is a better representative of the category.

### Why This Matters for Agent Infrastructure

As more stories are executed, commit history becomes a form of project memory:
- "What files did E-005-03 touch?" -- answerable via `git log --grep="E-005-03"`
- "What changed in the last week in src/gamechanger/?" -- answerable via `git log`
- "Which stories have been marked DONE in story files but have no corresponding commits?"
  -- answerable via cross-referencing story files and commit messages

The question is whether `gh`/`git` CLI is sufficient for these queries, or whether a
structured MCP interface would make these queries easier for agents to execute reliably.

### The Simpler Alternative to Benchmark Against

The null hypothesis: **`gh` CLI and `git` CLI, called via Bash tool, are sufficient for
all realistic agent use cases at this project's scale.** MCP adds a configuration layer
(registration in settings.json, server process startup) without meaningful benefit over
piped bash output.

Test this hypothesis. Do not assume it is correct or incorrect.

### Scope Boundary

This spike covers **git and GitHub data access patterns for agents** only. It does NOT
cover:
- Code review workflows (out of scope, no CI/CD yet)
- GitHub Actions integration
- PR management or issue tracking (no GitHub remote yet, and IDEA-003 covers work
  management tooling if/when that becomes a priority)
- Any other MCP server beyond GitNexus (or a well-maintained equivalent) in the git
  category
