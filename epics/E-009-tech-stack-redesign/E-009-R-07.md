## Status

## Research Question

What is apitap (https://github.com/n1byn1kt/apitap), and would it -- or any tool in the
same category -- be useful for the baseball-crawl project?

This is **awareness and fit-assessment research only**. The researcher makes no assumptions
about what apitap does or what category it belongs to. Investigation starts from scratch.
No tool adoption decision is made here. The output is a factual report.

## Context

### How This Spike Was Requested

The user flagged the apitap repository for awareness. No description was provided beyond
the URL. The researcher should treat this as an open question: discover what apitap does,
identify its category, find the broader landscape of tools in that category, and assess
whether any of them are useful for this project.

### What the Researcher Does NOT Know Yet

- What apitap does
- What problem it solves
- What category of tool it is (e.g., HTTP mocking, API testing, traffic inspection,
  API spec generation, contract testing, something else entirely)
- Whether the project has any gap that fits that category

All of the above must be determined by reading the repository, not assumed in advance.

### Why Category Identification Matters

The value of a single tool is often better understood in the context of alternatives.
Once the researcher identifies what apitap does, they should:

1. Name the category (one sentence: "apitap is a tool for X")
2. Identify what other tools exist in that category (well-known, maintained options)
3. Assess the full landscape briefly before drilling into any single tool

This prevents the spike from producing a verdict on apitap in isolation while missing
a more widely adopted tool in the same space that would be a better fit.

### The Simpler Alternative to Benchmark Against

Before concluding any tool is worth adding, compare it against the simplest alternative
already available to the project:

- **Existing Claude Code tools**: Does WebFetch, Bash, Read, Grep, or another built-in
  tool already cover this use case?
- **Python stdlib or common libraries**: Is there a built-in Python way to do this
  (e.g., `unittest.mock`, `responses`, `httpretty`, `pytest-httpserver`)?
- **Do nothing**: Is this a problem the project actually has, or a solution looking for
  a problem?

The null hypothesis is: **the project does not need this category of tool at its current
scale, and existing tools are sufficient.** Test this hypothesis; do not assume it is
wrong.

### Project Constraints (for fit assessment)

When assessing fit, use these constraints:

- Stack: Python end-to-end. FastAPI + Jinja2. SQLite. Docker Compose. No TypeScript
  serving layer.
- Scale: 4 teams, ~60 players, ~120 games/season. Single operator. 1-5 coaching staff.
- HTTP interactions: GameChanger API (undocumented, short-lived credentials). All HTTP
  client code lives in `src/gamechanger/` and `src/http/`.
- Testing: pytest. Tests mock at the HTTP layer (no real network calls in test suite,
  per CLAUDE.md). Current mocking approach not specified -- this spike may discover
  whether there is a gap.
- Agent infrastructure: 7 specialized Claude Code agents, custom skills, epic/story
  workflow. New tools that help agents (e.g., as MCP servers or CLI tools callable
  via Bash) are in scope.
- Complexity bar: The agent ecosystem is already complex. New tools must justify
  themselves. "Interesting" is not sufficient; "solves a real current problem or
  clearly solves an anticipated near-term problem" is required.

## Deliverable

A written research report saved to:
`/Users/jason/Documents/code/baseball-crawl/.project/research/E-009-R-07-apitap.md`

The report must contain the following sections in order.

### Section 1: apitap -- What It Is

1. **What it does**: One to two paragraphs, factual description based on the actual
   repository and any documentation. Read the README, source code if needed, and any
   linked documentation. Do not summarize marketing copy -- summarize what the tool
   actually does mechanically.
2. **Category**: One sentence naming the category of tool this is (e.g., "apitap is an
   HTTP traffic recording and replay tool"). If the tool fits multiple categories, name
   the primary one and note the others.
3. **Setup and maintenance cost**: What does it take to install, configure, and keep
   running? Does it require a server process, a Docker container, a browser extension,
   a language-specific library, or something else? What is the ongoing maintenance
   burden (updates, config drift, etc.)?
4. **Current status**: Is the repository maintained? Last commit date. Stars/forks as
   a rough signal of community adoption. Any known breaking issues, open bugs, or signs
   of abandonment.
5. **How it works**: A brief technical description of the mechanism. Not a tutorial --
   just enough that the reader understands how the tool operates at a conceptual level.

### Section 2: Category Landscape

Once the researcher has identified what category apitap belongs to:

1. **Category definition**: One clear paragraph defining the problem space. What does
   a tool in this category do, and what problem does it solve?
2. **Well-known alternatives**: List at least three to five other tools in the same
   category. For each, provide:
   - Name and a one-sentence description
   - Maintenance status (actively maintained? archived?)
   - Language/ecosystem (is it Python-native, language-agnostic, etc.?)
   - Any notable differentiator from apitap
3. **Landscape summary**: Two to three sentences characterizing the category. Is it
   crowded? Mature? Dominated by a few well-known tools? Niche?

### Section 3: Fit Assessment for baseball-crawl

For apitap and for each alternative identified in Section 2 that appears at least
potentially relevant to this project:

1. **Is there a real problem this solves here?** Name the specific workflow, agent, or
   code location where this tool would apply. If no real gap exists, say so.
2. **What is the simpler alternative?** Name the simplest existing approach (Python
   library, Claude Code built-in, bash command) that covers the same ground.
3. **What does this tool add?** Specifically: what can this tool do that the simpler
   alternative cannot, or cannot do as cleanly?
4. **Verdict** (one of three):
   - **Not recommended**: The simpler alternative is sufficient, or the tool does not
     fit the project's constraints. State the reason in one sentence.
   - **Worth a follow-up spike**: Interesting enough to investigate more deeply before
     deciding. State what the follow-up spike would need to determine.
   - **Worth adopting**: Clear fit, low setup cost, defensible added complexity. State
     what a follow-on story would look like.

Assign a verdict to apitap and to each alternative assessed. Tools that clearly do not
fit (wrong language ecosystem, requires cloud account, irrelevant problem space) may be
dismissed in one sentence -- no need for a full per-tool writeup for those.

### Section 4: Synthesis

After the per-tool sections:

1. **Is there a gap here?** Does this category of tool address a real gap in the
   baseball-crawl project today, or in the near term?
2. **If yes**: Which tool (apitap or an alternative) is the best fit, and why?
3. **If no**: State clearly that no tool in this category is recommended at this time,
   and under what circumstances a future revisit would be warranted.
4. **Anything unexpected discovered?** Note any tools or patterns discovered during
   research that are outside the category but seem relevant to the project.

## Research Method

Use WebSearch and WebFetch. Key sources:
- The apitap GitHub repository (https://github.com/n1byn1kt/apitap) -- README, source,
  issues, commits
- Search terms derived from whatever category apitap belongs to (discover these during
  research, not in advance)
- Python ecosystem equivalents (PyPI, relevant GitHub repos)
- Any community discussion of apitap or the category (GitHub Issues, Hacker News,
  developer blogs, 2024-2026)

Do NOT make architectural decisions or modify any project files. This spike produces
a research report only.

## Time Box

One focused research session. Allocate roughly:
- 20 minutes: Read apitap repository, identify category, form initial understanding
- 20 minutes: Research alternatives landscape
- 20 minutes: Write fit assessment and synthesis

If apitap turns out to be sparse, unmaintained, or trivially scoped, document that
quickly and spend the bulk of the session on the category landscape and fit assessment --
that information is useful regardless of apitap's quality.
