## Description
After this story is complete, the baseball-coach agent has used WebFetch to browse the
running dashboard, confirmed that the rendered HTML contains real stat content, and
provided structured UX feedback. The verification result -- including the agent's actual
WebFetch output -- is documented as a research note. The workflow for using WebFetch
during dashboard development is documented and ready for E-004.

## Context
One of the core requirements driving E-009 was agent browsability: the baseball-coach
agent must be able to browse the coaching dashboard during development to review layouts,
data presentation, and mobile suitability. E-009-03 built the dashboard. This story
verifies the browsability mechanism actually works and captures the workflow so E-004
can rely on it.

This is a verification and documentation story, not an implementation story. The agent
browses what already exists and confirms the pipeline works. The output is a research note
and a workflow guide, not code.

From the decision record: "User provides localhost URL; agent uses WebFetch" is the
confirmed mechanism. This story validates that assumption and handles the fallback case
if WebFetch cannot reach localhost.

## Acceptance Criteria

- [ ] **AC-1**: While `docker compose up` is running, the dashboard at
  `http://localhost:8000/dashboard` is fetched by an agent tool (WebFetch preferred, Bash
  `curl` as fallback). The response is valid HTML with at least one stat table visible.
  The tool used and the result are recorded in the research note (AC-3).

- [ ] **AC-2**: The baseball-coach agent provides UX feedback on the dashboard. The
  feedback must address at minimum: (a) whether the stat layout is useful for game prep,
  (b) whether the mobile-friendliness is adequate for dugout use, (c) one specific
  improvement recommendation for E-004. The feedback is captured verbatim in the research
  note.

- [ ] **AC-3**: A research note is written at
  `/.project/research/E-009-agent-browsability-verification.md`. It contains:
  - The WebFetch URL used and the test result (success or fallback triggered)
  - The actual HTML snippet returned (first 500 characters or the stat table fragment)
  - The baseball-coach agent's UX feedback verbatim
  - A conclusion: "agent browsability via WebFetch works as expected" or "fallback required"

- [ ] **AC-4**: If WebFetch cannot reach `http://localhost:<port>` (the tool refuses
  localhost URLs or times out), the fallback procedure is documented in the research note.
  Fallback options in priority order:
  1. `curl http://localhost:8000/dashboard` via the Bash tool (simplest, always available)
  2. `mcp__ide__executeCode` running a Python `urllib.request` call
  3. Any other MCP tool that can fetch HTTP content
  The research note must confirm which approach works and recommend it for E-004.

- [ ] **AC-5**: `docs/agent-browsability-workflow.md` is written and committed. It
  documents the standard workflow for using agents to review dashboard changes during
  development: (a) start `docker compose up`, (b) invoke baseball-coach agent with the
  localhost URL, (c) review feedback, (d) iterate. This doc is the reference for E-004.

## Technical Approach

This story is executed as follows:

1. Ensure `docker compose up` is running with E-009-02 and E-009-03 complete (dashboard
   serves data).
2. In the same Claude Code session, ask the baseball-coach agent to WebFetch the dashboard
   URL and provide UX feedback.
3. Record the exact output.
4. Write the research note and workflow doc.
5. If WebFetch fails on localhost, test the `mcp__ide__executeCode` fallback immediately
   and document which approach works.

### WebFetch Test Prompt for baseball-coach

```
Please fetch http://localhost:8000/dashboard and tell me:
1. Can you see the stat table content? (If yes, quote one player row.)
2. From what you can see in the HTML, is the layout suitable for mobile use?
3. What is one specific change you would recommend for the E-004 coaching dashboard?
```

### Fallback: mcp__ide__executeCode

If WebFetch cannot reach localhost, this Python snippet (run via `executeCode`) verifies
connectivity from within the agent's execution environment:

```python
import urllib.request
response = urllib.request.urlopen("http://localhost:8000/dashboard")
html = response.read().decode("utf-8")
print(html[:1000])
```

## Notes

- This story is owned by general-dev but requires the baseball-coach agent to provide
  UX feedback. The implementing agent should invoke baseball-coach via the Task tool
  (subagent_type: baseball-coach) with the fetched HTML content and ask for the
  feedback specified in AC-2. The baseball-coach is a direct-routing exception --
  no PM intermediation needed.
- The research output from R-03 noted that WebFetch processes fetched content through an
  AI model before returning it to the calling agent. This means the baseball-coach agent
  receives a structured summary of the HTML, not raw markup. The summary should be
  sufficient for UX review purposes.
- If the dashboard has no real game data (only seed data), that is fine for this story.
  The goal is to verify the browsability mechanism works, not to evaluate production data.
- Document whether Tailwind CDN renders correctly in the WebFetch response or if the CDN
  `<script>` tag causes any issues (it should not, since WebFetch fetches server-rendered
  HTML before any client-side JS runs).
- Size: Small. The implementation is mostly navigation and documentation; the only
  uncertainty is whether WebFetch works with localhost (which the fallback resolves).
