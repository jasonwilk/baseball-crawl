## Research Question
What is the right frontend approach for the coaching dashboard, and how does it differ
between the two deployment options? Both options must satisfy the same three constraints:
(1) agents must be able to browse the dashboard via `WebFetch` or equivalent, (2) coaches
use it on mobile, (3) the dashboard is read-only and data-display focused.

This research also settles the agent browsability question: what tool or mechanism allows
a Claude Code agent to fetch and evaluate a rendered HTML page from a local development URL?

## Context

### What "Agents Browse the Dashboard" Means
The baseball-coach agent needs to review coaching dashboard layouts during development.
This is not just data validation (which hitting a JSON endpoint would satisfy). It is
visual/structural review: is the data in the right order? Is the mobile layout readable?
Is the opponent scouting report organized the way a coach would expect?

For this to work:
1. The dashboard must serve real HTML from a real HTTP server (not a file:// URL)
2. The agent must have a tool that fetches that HTML and presents it meaningfully

The `WebFetch` tool in Claude Code fetches a URL and processes it with an AI model. If
the dashboard runs locally, `WebFetch` should be able to reach it -- but this needs
verification.

**Key verification needed**: Does `WebFetch` work with `http://localhost:<port>` URLs, or
does it require a public internet URL? This question applies to both deployment options
during local dev.

### Dashboard Requirements (Same for Both Options)
- Mobile-first (coaches check this in the dugout, on a phone)
- Read-only (no forms, no edits)
- Data-display: tables, stat cards, simple lists
- No real-time updates (page refresh is fine)
- Multiple views: team roster, opponent scouting report, player profile, game log
- Loading time under 2 seconds on a mobile connection (server-rendered HTML preferred)

### Frontend Approach Differences by Option

**Option A (Native Cloudflare)**:
The dashboard lives in Cloudflare Pages. Pages supports static HTML/JS/CSS or server-side
rendering via Workers (using Workers as the backend for Pages functions). Local dev uses
`wrangler pages dev`. Evaluate whether server-rendered HTML is achievable in Pages + Workers,
or whether Pages is effectively static-only in local dev.

**Option B (Docker + Cloudflare Access)**:
The dashboard is served directly from a FastAPI container (Jinja2 templates, server-rendered
HTML). This is the simpler and more flexible path for server-rendered HTML.

### Frontend Candidates

For both options, the HTML/CSS layer choices are similar:
- **Jinja2 (server-rendered)**: Full HTML on the server; agents get real rendered content
  from `WebFetch`; no client-side JS required for data display.
- **Alpine.js**: Lightweight JS for minor interactivity; no build step; can layer over
  server-rendered HTML.
- **HTMX**: Server-side interactivity (partial page updates) without a JS framework;
  pairs well with FastAPI/Jinja2; no build step.
- **SvelteKit**: Modern, SSR capable. Adds Node.js and a build step. Probably overkill.
- **Tailwind CDN vs. Bootstrap CDN vs. minimal hand-written CSS**: CSS approach is
  independent of the JS choice.
