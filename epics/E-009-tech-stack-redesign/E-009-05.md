## Description
After this story is complete, any developer (or agent) can reset the development database
to a clean seeded state with a single command, and restore the production database from a
Litestream backup with documented steps. Litestream is configured in `docker-compose.yml`
to continuously replicate the SQLite WAL to Cloudflare R2. The seed data is realistic
enough to exercise all dashboard views when E-004 development begins.

## Context
E-009-02 bootstrapped the migration runner and created a minimal `seed_dev.sql`. This
story extends that foundation in two directions:

1. **Development workflow**: Richer seed data, a clean reset command, and a dev-only
   database reset script. The goal is to make local dashboard development fast -- a
   developer can break their database and recover in seconds.

2. **Production backup**: Litestream provides continuous SQLite replication. The backup
   target is Cloudflare R2 (keeps vendor surface consistent with the rest of the stack).
   The restore procedure must be documented and testable.

Key decisions from the decision record:
- SQLite WAL mode is already enabled by `apply_migrations.py` (from E-009-02)
- Host-mounted database at `./data/app.db`
- Litestream runs as a sidecar in Docker Compose, not as a separate process in the app
- Cloudflare R2 is the preferred backup target; S3 is acceptable fallback

## Technical Approach

### reset_dev_db.py

```python
# scripts/reset_dev_db.py
# Usage: python scripts/reset_dev_db.py [--db-path PATH]
# Default path: data/dev.db (or data/app.db depending on APP_ENV)
# 1. Remove existing database file if it exists
# 2. Run apply_migrations.py against fresh database
# 3. Execute data/seeds/seed_dev.sql
# 4. Print summary
```

### Litestream in docker-compose.yml

```yaml
  litestream:
    image: litestream/litestream:latest
    profiles:
      - backup                          # Only starts with: docker compose --profile backup up
    command: replicate
    volumes:
      - ./data:/data                    # Same host-mounted data dir as app
      - ./litestream/litestream.yml:/etc/litestream.yml
    environment:
      - LITESTREAM_ACCESS_KEY_ID=${LITESTREAM_ACCESS_KEY_ID}
      - LITESTREAM_SECRET_ACCESS_KEY=${LITESTREAM_SECRET_ACCESS_KEY}
    depends_on:
      app:
        condition: service_healthy
    restart: unless-stopped
```

In production, always use `docker compose --profile backup up -d` to include Litestream.
For local dev, plain `docker compose up` skips it (no R2 credentials needed).

### litestream.yml

```yaml
dbs:
  - path: /data/app.db
    replicas:
      - type: s3
        bucket: ${LITESTREAM_R2_BUCKET}
        path: baseball-crawl/app.db
        endpoint: ${LITESTREAM_R2_ENDPOINT}  # https://<account-id>.r2.cloudflarestorage.com
        access-key-id: ${LITESTREAM_ACCESS_KEY_ID}
        secret-access-key: ${LITESTREAM_SECRET_ACCESS_KEY}
        sync-interval: 1s
```

### Seed Data Requirements

The `seed_dev.sql` file should produce a database state that exercises dashboard views:
- `teams`: Lincoln Varsity (owned), Opponent HS (not owned)
- `players`: 5 Lincoln Varsity players + 5 Opponent players (realistic names)
- `games`: 5 games, mix of home/away, with scores
- `player_game_batting`: rows for all players in all games they appeared in
- `player_game_pitching`: rows for 1-2 pitchers per team per game
- `player_season_batting`: season aggregate rows for all Lincoln players

## Notes

- Cloudflare R2 provides S3-compatible object storage. Litestream uses the S3 replica type
  with the R2 endpoint (`https://<account-id>.r2.cloudflarestorage.com`). The bucket must
  be created manually in the Cloudflare dashboard before Litestream can replicate to it.
- R2 free tier (10 GB storage, 1M Class A ops/month) is ample for SQLite WAL replication
  at this scale. No cost concern.
- For local dev without R2 credentials, the `litestream` service is behind a Docker Compose
  profile (AC-3) and does not start by default. No override file needed for this case.
- The `reset_dev_db.py` script operates on whatever `DATABASE_PATH` points to (default:
  `./data/app.db`). There is no separate `dev.db` convention -- E-009-02 established
  `./data/app.db` as the single database path for all environments. The script should
  include a safety check: if `APP_ENV=production`, require a `--force` flag to proceed.
- SQLite WAL mode: `apply_migrations.py` (E-009-02) already enables WAL mode. Litestream
  requires WAL mode to be enabled. Verify this is the case before running Litestream.
- Size: Medium. The reset script and seed data are straightforward; Litestream config and
  R2 integration require careful credential wiring and a real connectivity test.
