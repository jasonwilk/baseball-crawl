# E-009-04: Cloudflare Tunnel + Zero Trust Access Configuration

## Epic
[E-009: Tech Stack Redesign -- Portable, Agent-Browsable, Cloudflare-Integrated](epic.md)

## Status
`DONE`

## Description
After this story is complete, all inbound traffic to the baseball-crawl stack routes
through a Cloudflare Tunnel. No ports are exposed to the public internet. Cloudflare
Zero Trust Access policies protect every service: service tokens for machine-to-machine
calls (crawlers), WARP + email OTP for human (coaching staff) access, and a public bypass
for any pages that should be openly accessible. The tunnel token is stored in `.env` and
the `cloudflared` service runs in `docker-compose.yml`.

## Context
The technology decision (E-009-01) specified Cloudflare Tunnel + Zero Trust Access as the
network and auth layer for Option B. This is the same pattern used in the n8n-wilk-io
production deployment. The user has direct experience operating this pattern.

This story is infrastructure-only. It does not require working application code in
E-009-02 or E-009-03 to configure the Cloudflare side (tunnel creation, DNS, access
policies). However, to verify end-to-end connectivity, the app must be running. Coordinate
timing: configure Cloudflare first, then test once E-009-02 is complete.

Key access policy layers (from the epic Technical Notes):
1. **Service tokens**: Machine-to-machine. Crawlers call the API using
   `Authorization: Bearer <token>` with a Cloudflare Access service token.
2. **WARP + email OTP**: Coaching staff access the dashboard via Cloudflare Access.
   Users must have the WARP client installed, or complete email OTP.
3. **Public bypass**: Specific URL patterns (e.g., a game-day scouting page) can be
   marked public in the Access policy so coaches can share links without login.

## Technical Approach

### cloudflared in docker-compose.yml

```yaml
  cloudflared:
    image: cloudflare/cloudflared:latest
    command: tunnel --no-autoupdate run
    environment:
      - TUNNEL_TOKEN=${CLOUDFLARE_TUNNEL_TOKEN}
    depends_on:
      app:
        condition: service_healthy
    restart: unless-stopped
```

### Tunnel Ingress Config (in Cloudflare Dashboard)

Set the tunnel ingress to route all traffic to Traefik:
```
baseball.<domain>     -> http://traefik:80
api.baseball.<domain> -> http://traefik:80
```

Traefik's label-based routing handles the Host header split internally.

### Zero Trust Access Policy Structure

For the dashboard application (`baseball.<domain>`):
```
Policy: Allow coaching staff
  Rule: WARP enrolled
  OR
  Rule: Email domain matches <your-domain>

Service token policy:
  Allow any request with valid CF-Access-Client-Id header
```

For the API application (`api.baseball.<domain>`):
```
Policy: Service token only (no human access needed for API-only paths)
  Rule: Service token == <baseball-crawl-crawler-token>
```

### Crawler Auth Pattern

The Python crawlers (E-001 through E-006) calling the production API must include:
```
CF-Access-Client-Id: <client-id>
CF-Access-Client-Secret: <client-secret>
```
These headers satisfy the Cloudflare Access service token check before the request
reaches the FastAPI app. Store them as environment variables, never in code.

## Notes

- This is the one story in E-009 that requires a real Cloudflare account and a real
  domain with DNS managed by Cloudflare. There is no local mock for `cloudflared` tunnel
  verification. Run the tunnel tests against the real Cloudflare environment.
- The `cloudflared` image pulls from `docker.io/cloudflare/cloudflared:latest`. Pin to a
  specific version tag in production to avoid unexpected updates.
- Public bypass: if any scouting reports need to be shared without login, configure a
  bypass rule in the Access application for that specific path pattern. Document the bypass
  approach in `docs/cloudflare-access-setup.md`.
- SSL: Cloudflare Tunnel handles SSL termination. The Docker Compose stack has no TLS
  certificates and no cert renewal concern. This is a key operational advantage.
- For local dev only (not production), the `cloudflared` service can be omitted by using
  a `docker-compose.override.yml` that removes the service. Local dev accesses the app
  directly at `localhost:8000` without going through the tunnel.
- While modifying `docker-compose.yml`, add `restart: unless-stopped` to ALL services
  (app, traefik, and the new cloudflared). This was missing from the E-009-02 implementation
  and is required for production stability. Document this in the commit message.
- The `docker-compose.override.yml.example` for local dev without tunnel is covered by
  E-009-07 AC-8, not this story. This story only adds the `cloudflared` service definition.
- Size: Medium. The Cloudflare configuration is straightforward given the reference
  n8n-wilk-io pattern; the documentation write-up is the most time-consuming part.
