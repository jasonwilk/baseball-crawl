## Description
After this story is complete, the context layer will include a migration conventions rule (guiding agents working on SQL migrations), the first API flow document (documenting the opponent resolution chain end-to-end), and an updated API docs structure that recognizes the new `flows/` directory.

## Context

The opponent resolution flow doc (first instance of IDEA-017) documents the multi-endpoint API chain that agents and operators need to understand -- knowledge that currently exists only in conversation history and epic Technical Notes.

## Technical Approach
The migrations rule should be concise and practical -- agents working on migrations need to know the numbering scheme, naming pattern, idempotency requirements, and how to handle seed data. The rule should reference the existing migration files as examples.

The flow doc follows a different structure than endpoint docs -- it documents a multi-step chain rather than a single endpoint. Use a descriptive format: starting entity, each step (endpoint + which field feeds the next step), ending entity, edge cases, and fallback paths. This is the first flow doc, so the format will establish the pattern for future flows (per IDEA-017).
