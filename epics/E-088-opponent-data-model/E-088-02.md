## Description
After this story is complete, running the resolver will automatically chain authenticated API calls to resolve opponents from the opponent list to their real GameChanger team identities, populating the `opponent_links` table. Approximately 86% of opponents will be auto-resolved via their `progenitor_team_id`. The remaining ~14% (null progenitor) will be inserted as unlinked rows for manual resolution via the admin UI.

## Context
This is the core data pipeline story. It bridges the gap between "we know our opponents exist" and "we know who they really are in GameChanger." The resolution chain was mapped during API exploration and confirmed by api-scout: opponents list -> progenitor_team_id -> team detail (which returns public_id). The `GET /teams/{progenitor_team_id}/public-team-profile-id` endpoint returns 403 for opponent UUIDs and must NOT be used.

## Acceptance Criteria
- [ ] **AC-1**: `OpponentResolver` class in `src/gamechanger/crawlers/opponent_resolver.py` follows existing crawler patterns (class with a `resolve()` method, returns structured result with counts)
- [ ] **AC-2**: Outer loop iterates over all owned teams from `config.owned_teams` (same team list used by `OpponentCrawler`). Field mapping: `our_team_id` = the team's GC UUID from config (accessed as `team.id` on a `TeamEntry` instance -- see `src/gamechanger/config.py`), `root_team_id` = the opponent's `root_team_id` from the API response. For each owned team, fetches the opponents list via `GET /teams/{team_id}/opponents`, then for each opponent with a non-null `progenitor_team_id`, resolves via `GET /teams/{progenitor_team_id}` -> extract `public_id` from team metadata -> upsert into `opponent_links`. Copy `is_hidden` from the API response into `opponent_links.is_hidden` during upsert.
- [ ] **AC-3**: Opponents with null `progenitor_team_id` are inserted as unlinked rows (resolution_method=NULL, resolved_team_id=NULL, public_id=NULL)
- [ ] **AC-4**: Upsert uses `ON CONFLICT(our_team_id, root_team_id) DO UPDATE` with COALESCE-based protection to prevent auto-resolution from overwriting manual links (see epic Technical Notes section 5)
- [ ] **AC-5**: Before inserting a resolved link, ensure the `resolved_team_id` exists in the `teams` table for FK satisfaction. Do NOT reuse `_ensure_team_row()` directly (it uses `ON CONFLICT DO NOTHING` with UUID-as-name). Instead, write a custom upsert: `INSERT INTO teams ... ON CONFLICT(team_id) DO UPDATE SET name = excluded.name WHERE teams.name = teams.team_id` -- this creates the stub with the real team name from the `GET /teams/{progenitor_team_id}` response and updates stale UUID-as-name stubs to the real name.
- [ ] **AC-6**: Error handling follows OpponentCrawler pattern: 403 -> log warning and skip, 401 -> abort with CredentialExpiredError, 5xx -> log warning and skip, 404 -> log warning and skip (defunct team)
- [ ] **AC-7**: Sequential requests with project-standard rate limiting (~1.5s delay between API calls)
- [ ] **AC-8**: Invocable via a new `bb data resolve-opponents` CLI subcommand that runs the resolver end-to-end for all owned teams from `config.owned_teams` and prints a summary (resolved count, skipped count, error count). Supports `--dry-run` (log what would be resolved without writing to the database). `--profile` inherits from GameChangerClient default (web). No `--source` flag needed -- reads from YAML config (`config.owned_teams`) directly.
- [ ] **AC-9**: Tests cover: successful resolution chain (mock API responses), null progenitor_team_id handling, manual resolution protection (manual link not overwritten by auto), error cases (403/404/5xx responses), idempotent re-runs (second run does not corrupt data)

## Technical Approach
The resolver is a new module, NOT an extension of `src/gamechanger/team_resolver.py` (that module uses public/unauthenticated endpoints; this uses authenticated `GET /teams/{progenitor_team_id}` which requires gc-token).

The existing `OpponentCrawler` in `src/gamechanger/crawlers/opponent.py` already fetches the opponents list -- the resolver can reuse that data or compose with it. `GameChangerClient.get()` already supports all needed endpoints with automatic token refresh and 401 retry.

Expected crawl timing: ~60 opponents with progenitor_team_id x 1.5s/request = ~90 seconds sequential.

## Notes
- The resolver should be designed for re-runs: safe to run nightly or on-demand without data corruption.
- The COALESCE protection pattern is critical -- without it, a coach who manually linked an opponent would lose that link on the next auto-resolution pass.
- The `public_id` stored in opponent_links enables future public endpoint data ingestion (IDEA-020) without re-resolving.
