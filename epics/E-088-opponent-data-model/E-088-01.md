# E-088-01: opponent_links Migration and Seed Data

## Epic
[E-088: Opponent Data Model and Resolution](epic.md)

## Status
`TODO`

## Description
After this story is complete, the database will have an `opponent_links` table that bridges local opponent entries to resolved GameChanger teams. The migration is purely additive (no existing tables modified) and seed data will include representative rows for all three link states (auto-resolved, manually linked, unlinked).

## Context
This is the first story in the epic and unblocks both the automated resolver (E-088-02) and the admin UI (E-088-03). The schema was designed by the data-engineer during expert consultation. This is the project's first new migration since 005_teams_public_id.sql -- migration number 006.

## Acceptance Criteria
- [ ] **AC-1**: `migrations/006_opponent_links.sql` creates the `opponent_links` table with all columns from the epic Technical Notes section 1 (id, our_team_id, root_team_id, opponent_name, resolved_team_id, public_id, resolution_method, resolved_at, is_hidden, created_at, updated_at)
- [ ] **AC-2**: Unique constraint on `(our_team_id, root_team_id)` prevents duplicate opponent entries per team
- [ ] **AC-3**: Three indexes created: `idx_opponent_links_our_team` on our_team_id, `idx_opponent_links_resolved` partial on resolved_team_id WHERE NOT NULL, `idx_opponent_links_public_id` partial on public_id WHERE NOT NULL
- [ ] **AC-4**: `data/seeds/seed_dev.sql` includes at least 3 representative opponent_links rows: 1 auto-resolved (resolution_method='auto', resolved_team_id and public_id both populated), 1 manually resolved (resolution_method='manual', public_id populated, resolved_team_id NULL -- matches the URL-paste flow which cannot obtain a UUID), and 1 unlinked (resolved_team_id NULL, public_id NULL, resolution_method NULL)
- [ ] **AC-5**: `python migrations/apply_migrations.py` applies 006 cleanly on a fresh database and idempotently on an existing database that already has 001-005
- [ ] **AC-6**: `bb db reset` works with the new migration and seed data (database resets and re-seeds without errors)

## Technical Approach
The migration file follows the same pattern as existing migrations (001-005): `CREATE TABLE IF NOT EXISTS` with `CREATE INDEX IF NOT EXISTS`. The full DDL is in epic Technical Notes section 1.

Seed data rows need valid FK references -- `our_team_id` and `resolved_team_id` must reference existing rows in the `teams` table from the existing seed data. Check `data/seeds/seed_dev.sql` for existing team_id values to reference.

Reference files:
- `/workspaces/baseball-crawl/migrations/005_teams_public_id.sql` -- previous migration for pattern reference
- `/workspaces/baseball-crawl/data/seeds/seed_dev.sql` -- existing seed data to extend
- `/workspaces/baseball-crawl/migrations/apply_migrations.py` -- migration runner (should handle 006 automatically)
