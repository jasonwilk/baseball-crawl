# E-088-03: Admin Opponents Page with Manual URL-Paste Linking

[E-088: Opponent Data Model and Resolution](epic.md)

## Status

## Description
After this story is complete, the admin interface will have a dedicated opponents page showing all opponent link states with coach-friendly labels, filter pills for quick triage, and a URL-paste flow for manually linking unlinked opponents to their real GameChanger teams. Operators can also disconnect incorrectly linked opponents.

## Context
The admin teams page (E-042) already displays tracked opponents, but without link state visibility or manual association capability. This story adds a focused opponents management page with three visual states (per UXD design) and a URL-paste connect flow. Search-based association is deferred until the search endpoint response schema is captured via ingest-endpoint.

## Technical Approach
The admin opponents page follows the same patterns as the existing admin teams page (E-042). The five routes handle listing, connect flow (two separate GET routes + POST), and disconnect. All routes use `{id}` which is the `opponent_links.id` integer primary key. Routes must return an HTML 404 response if the row doesn't exist (follow the existing admin pattern, not FastAPI's default JSON 404).

URL parsing reuses `parse_team_url()` from `url_parser.py`. Team resolution for the confirmation step should use `resolve_team()` from `team_resolver.py` (public, unauthenticated) -- it returns name, city/state, and record directly via `TeamProfile.record_wins` / `record_losses`, which is everything the confirm page needs.

The connect flow uses two separate GET routes (`/connect` for the form, `/connect/confirm` for the confirmation display), each rendering the same or separate templates via server-side rendering, following the existing admin pattern of server-rendered forms.

## Notes
- Badge colors and labels come from the coaching requirements research: coaches understand "Full stats" vs. "Scoresheet only" instantly. Yellow signals a limitation, not an error.
- The auto/manual micro-label on the green/blue badges is for the operator's benefit (distinguishing how a link was created), not for coaches.
- The disconnect flow is important for correcting wrong manual associations. After disconnect, the opponent becomes unlinked and eligible for re-linking.
- The template design accommodates future search results rendering when the search endpoint schema is captured -- but search is NOT implemented in this story.
