# E-229-02: Wire `run_plays_stage` into `_scout_live` (CLI)

## Epic
[E-229: Plays in `bb data scout` parity](./epic.md)

## Status
`TODO`

## Description
After this story is complete, `bb data scout` invokes the new shared helper after the post-spray dedup sweep, iterating per scouted team. Plays for every game in each team's scout result are crawled, loaded, and reconciled per the helper's contract. Plays-stage failures (per-game errors, auth expiry, reconcile failures) are non-fatal: the operator sees a typer.echo summary line, `crawl_jobs` is unaffected (CLI does not write to it), and the CLI exits 0 when scout's primary stages succeeded -- regardless of plays outcome.

## Context
The CLI scout entry point is `_scout_live` in `src/cli/data.py:264-385`. Today it ends with the post-spray dedup sweep at line 378 and immediately raises `SystemExit(exit_code)`. This story slots the plays stage in between, after dedup and before exit. CLI scout supports both single-team mode (`bb data scout --team <public_id>`) and bulk mode (no `--team` flag, iterates all tracked teams) -- both must invoke the plays stage per team.

The CLI re-raises `CredentialExpiredError` from spray (lines 337-338); this story does NOT propagate that pattern to plays. The shared helper catches `CredentialExpiredError` internally and returns `auth_expired=True`; the CLI wrapper logs a warning naming the deferred game count and exits per the existing exit-code logic (which already accounts for spray errors). The auth-error inconsistency between spray's re-raise and plays's catch is documented in epic Technical Notes "Auth-error parity gap" -- intentionally NOT fixed in this epic to keep scope tight.

## Technical Approach
The plays stage slots into `_scout_live` after the post-spray dedup sweep call at line 378. The wrapper iterates `crawl_results` -- the same per-team list spray already iterates (lines 327-371) -- and calls `run_plays_stage` once per team with that team's `team_id`, `public_id`, and `sorted(crawl_result.boxscores.keys())`. The `sorted(...)` call ensures deterministic ordering across CLI, web, and report paths so the parity test (E-229-05) compares well-defined row sets.

The CLI is currently inside one shared `with closing(sqlite3.connect(...))` block (line 284); the helper accepts that same connection. No new connection management is introduced.

The plays-stage result does NOT alter `exit_code`. The existing exit-code logic (line 372-373: `if total_spray_errors: exit_code = 1`) is preserved; plays additions are pure typer.echo output, mirroring spray's additive-enrichment pattern.

For the typer.echo summary, follow the format used by spray (lines 364-371) for consistency. The summary string per epic Tech Notes "Operator visibility" rule: `"Plays stage for <public_id>: loaded={loaded} skipped={skipped} errored={errored} deferred={N}"` with per-team prefix to support bulk-mode disambiguation.

The CLI does NOT write to `crawl_jobs` (it uses `scouting_runs`). No `crawl_jobs` updates are added.
