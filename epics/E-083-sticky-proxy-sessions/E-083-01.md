# E-083-01: Add sticky session injection to proxy URL resolution

## Epic
[E-083: Bright Data Sticky Proxy Sessions](./epic.md)

## Status
`TODO`

## Description
After this story is complete, the proxy URL resolution functions in `src/http/session.py` will accept an optional session ID parameter and inject it into the Bright Data proxy URL username. `GameChangerClient` will generate a unique session ID per instance and pass it through, ensuring all requests from a single client exit from the same proxy IP. The diagnostic tool (`bb proxy check`) will continue to use rotating IPs.

## Context
Bright Data sticky sessions work by appending `-session-<id>` to the proxy URL username. The session ID must be alphanumeric. Same ID = same exit IP (for up to 5 minutes idle). The injection needs to happen in the proxy URL resolution layer so it is centralized, and `GameChangerClient` needs to generate and pass a session ID per instance. The proxy check diagnostic should NOT use sticky sessions.

## Acceptance Criteria
- [ ] **AC-1**: Given a proxy URL `http://USERNAME:PASS<at>HOST:PORT` and a session_id `abc123`, when `resolve_proxy_from_dict()` is called with `session_id="abc123"`, then the returned URL has username `USERNAME-session-abc123` and the password, host, and port are unchanged.
- [ ] **AC-2**: `get_proxy_config()` in `src/http/session.py` is NOT modified by this story. It has no caller that needs sticky sessions.
- [ ] **AC-3**: Given `resolve_proxy_from_dict()` called without a session_id (or `session_id=None`), then the proxy URL is returned unmodified (backward-compatible).
- [ ] **AC-4**: Given a `GameChangerClient` instance is created, then it generates a unique alphanumeric session ID and passes it to `resolve_proxy_from_dict()`. The session ID is logged at DEBUG level (the value itself is safe to log -- it is a random identifier, not a credential).
- [ ] **AC-5**: Given two `GameChangerClient` instances created in the same process, then each has a different session ID.
- [ ] **AC-6**: `check_proxy_routing()` in `proxy_check.py` continues to call `resolve_proxy_from_dict()` without a `session_id` argument (preserving existing rotating-IP behavior for the diagnostic tool -- this is a regression guard, not new behavior).
- [ ] **AC-7**: Given a proxy URL with a session ID injected, then the proxy URL value (including the injected session suffix) never appears in any log message at any level. Only the session ID itself may be logged (at DEBUG).
- [ ] **AC-8**: All pre-existing test assertions in `tests/test_http_session.py`, `tests/test_proxy_check.py`, and `tests/test_client.py` continue to pass (no regressions). Adding new test cases to these files is expected per AC-9.
- [ ] **AC-9**: New tests cover: session ID injection into URL username (`resolve_proxy_from_dict()` only), no-session-id backward compatibility, GameChangerClient generates unique session IDs, proxy check does not use session IDs, log safety (neither the raw proxy URL nor the session-injected URL appears in any log output).

## Technical Approach
`resolve_proxy_from_dict()` in `/workspaces/baseball-crawl/src/http/session.py` needs an optional `session_id` parameter. When provided, a private `_inject_session_id(proxy_url, session_id)` helper parses the URL with `urllib.parse.urlparse()`, appends `-session-<id>` to the username, re-percent-encodes `parsed.password` with `urllib.parse.quote()` when rebuilding netloc (handles special characters safely), and returns the reconstructed URL. `get_proxy_config()` is NOT modified.

`GameChangerClient.__init__` in `/workspaces/baseball-crawl/src/gamechanger/client.py` needs to generate a unique session ID at construction time using `secrets.token_hex(8)` (16-char lowercase hex string -- satisfies Bright Data's alphanumeric-only requirement; do NOT use `secrets.token_urlsafe()` which contains `-` and `_`). The session ID is passed to `resolve_proxy_from_dict()`.

`check_proxy_routing()` in `/workspaces/baseball-crawl/src/http/proxy_check.py` must NOT pass a session_id -- verify this is the case (it currently calls `resolve_proxy_from_dict(env, profile)` with no session_id, so the new optional parameter maintains backward compatibility).

Test files to update: `/workspaces/baseball-crawl/tests/test_http_session.py` (new tests for session ID injection), `/workspaces/baseball-crawl/tests/test_client.py` (verify client generates and passes session ID), `/workspaces/baseball-crawl/tests/test_proxy_check.py` (verify proxy check does not use session ID).

## Notes
- Bright Data session ID format: alphanumeric only, appended as `-session-<id>` to the proxy URL username.
- Residential proxy IP persistence: up to 5 minutes idle. Our rate-limiting delays (1-2s) are well within this window.
- 502 handling: Bright Data returns 502 when a sticky peer becomes unavailable, then auto-assigns a new peer on the next request. Existing 5xx retry logic in `GameChangerClient` covers this.
