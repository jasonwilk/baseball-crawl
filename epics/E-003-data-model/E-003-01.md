## Description
After this story is complete, the project will have an initial SQL migration file that defines the core entity tables: `players`, `teams`, `seasons`, and `team_rosters`. These tables are the foundation every other table references. The migration applies cleanly to a SQLite database.

## Context
The core entities are the simplest and most stable part of the schema. They need to exist before any stats tables can be defined (E-003-02) and before any loader can write data (E-002-06, E-002-07).

The trickiest design decision here is player identity across teams and seasons. A player has one `player_id` (from GameChanger) that is stable across their career. Their membership on a team in a season is a separate record. This design allows querying "show me all stats for player X across all teams and seasons they played on."

## Technical Approach
Write `migrations/001_core_schema.sql` as pure SQL. Keep it simple: no triggers, no stored procedures. Add inline comments explaining non-obvious design decisions (e.g., why `is_owned` is an integer rather than boolean).

## Notes
Use `TEXT` for all ID fields in SQLite rather than `INTEGER` -- GameChanger IDs may be UUIDs or opaque strings, and `TEXT` is safe for any format. Do not assume they are integers.

`is_owned INTEGER NOT NULL DEFAULT 0` is used instead of `BOOLEAN` because SQLite stores booleans as integers. Document this convention in a comment in the SQL file.
