# E-003-04: Seed data and query validation tests

## Epic
[E-003: Data Model and Storage Schema](epic.md)

## Status
`DONE`

## Description
After this story is complete, the project has a seed data fixture (`tests/fixtures/seed.sql`) with realistic test data representing two Lincoln teams, several opponents, two seasons, a partial schedule of games, and player stats including batting and pitching season rollups with splits. Query validation tests confirm that the schema supports the coaching queries coaches will actually run: player stats lookups, team rosters by season, win-loss records, home/away splits, pitching leaderboards, and crawl configuration.

## Context
A schema is not done until you can run real queries against it and get sensible answers. This story closes the loop: it validates that the rewritten schema supports the queries coaches will actually run, and it provides a realistic test fixture that all future loader tests can use as a baseline.

The seed data covers the new schema features: `seasons` table entries with different types (spring-hs, summer-legion), `teams` with crawl configuration (is_active mix), `player_season_pitching` alongside batting, and all FK relationships through `season_id`.

This story does NOT depend on E-003-02 (coaching_assignments) -- that table is optional for query validation of the core data model.

## Technical Approach
Write `tests/fixtures/seed.sql` as a plain SQL INSERT script. Use fixed, known values (not random) so query results are deterministic and tests can assert exact values.

Write `tests/test_schema_queries.py` with pytest. Each test applies the migration + seed in a fresh in-memory SQLite database, runs a query, and asserts on the result.

```python
_PROJECT_ROOT = Path(__file__).resolve().parent.parent

@pytest.fixture
def seeded_db():
    conn = sqlite3.connect(":memory:")
    conn.execute("PRAGMA foreign_keys=ON;")
    # apply all migrations (use _PROJECT_ROOT to avoid cwd sensitivity)
    for mf in sorted((_PROJECT_ROOT / "migrations").glob("[0-9][0-9][0-9]_*.sql")):
        conn.executescript(mf.read_text())
    # apply seed
    conn.executescript((_PROJECT_ROOT / "tests/fixtures/seed.sql").read_text())
    return conn
```

## Notes
- The seed data player IDs should be human-readable strings like `PLAYER_VARSITY_01` rather than UUIDs. This makes test assertions readable at a glance.
- `ip_outs` convention: 1 inning pitched = 3 outs. A pitcher with 6.2 IP has 20 outs. K/9 = so * 27.0 / ip_outs (27 outs in 9 innings).
- The `seeded_db` fixture applies ALL migration files found in `migrations/`. If `003_auth.sql` exists at the time this test runs, it will also be applied (which is fine -- auth tables do not conflict with data model queries). If `004_coaching_assignments.sql` does not yet exist, that is also fine -- the seed data does not include coaching assignments.
- Home/away split batting average: `home_h / home_ab` vs `away_h / away_ab`. These come from the split columns in `player_season_batting`, not derived from game data.
- `player_game_pitching` and `player_season_pitching` rows should exist only for players who actually pitched, not for every player on the roster. In real baseball, only 1-3 players per team pitch in a given game. The seed data should reflect this: designate 2-3 players per owned team as pitchers.
- E-003-04 can run in parallel with E-003-02 since they touch completely different files and E-003-04 does not depend on the coaching_assignments table.
