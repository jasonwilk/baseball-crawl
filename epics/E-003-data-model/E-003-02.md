## Description
After this story is complete, the project will have a second SQL migration that defines the stats tables: `games`, `player_game_batting`, `player_game_pitching`, `player_season_batting`, and `player_season_pitching`. These are the tables coaches will query most frequently for game prep and scouting.

## Context
Stats tables are the "wide" tables -- they have many columns, most of them nullable integers. The design decisions here matter for query performance and flexibility:

- **Store raw counting stats, not rates**: Store AB, H, BB, K, not batting average. Rates are computed in queries. This avoids stale derived data.
- **Home/away and L/R splits**: If the API provides these, store them as columns in `player_season_batting`. If not, they must be derived by aggregating `player_game_batting` with the `games.home_team_id`/`away_team_id` context.
- **Innings pitched**: Store as outs (integer) to avoid decimal arithmetic edge cases with 2/3 innings.

## Technical Approach
Write `migrations/002_stats_schema.sql`. Reference the schema sketch in the epic's Technical Notes.

For splits, use this column pattern in `player_season_batting`:
```sql
home_ab INTEGER, home_h INTEGER, home_hr INTEGER, home_bb INTEGER, home_so INTEGER,
away_ab INTEGER, away_h INTEGER, away_hr INTEGER, away_bb INTEGER, away_so INTEGER,
vs_lhp_ab INTEGER, vs_lhp_h INTEGER, vs_lhp_hr INTEGER, vs_lhp_bb INTEGER, vs_lhp_so INTEGER,
vs_rhp_ab INTEGER, vs_rhp_h INTEGER, vs_rhp_hr INTEGER, vs_rhp_bb INTEGER, vs_rhp_so INTEGER
```
All nullable. If the API doesn't provide these, the columns stay NULL and can be backfilled later.

## Notes
`ip_outs` convention: 1 inning pitched = 3 outs, so a pitcher who throws 6.2 innings has 20 outs. This avoids the awkward "6.2 innings" floating point representation. Any display layer should convert back: `ip_outs / 3` whole + `ip_outs % 3` thirds.
