# E-187-01: Fix Search Pagination and Add Integration Tests

## Epic
[E-187: Threshold Calibration for Youth/HS Seasons](epic.md)

## Status
`TODO`

## Description
After this story is complete, `_resolve_gc_uuid` will paginate through POST /search results (up to 5 pages of 25 hits each) instead of only checking page 0. Two new tests will cover: (1) the full resolution → persist → spray-crawler wiring in `generate_report()`, and (2) multi-page search pagination.

## Context
E-186 Codex post-dispatch review found two gaps. First, the search endpoint returns 25 results per page, and common team names (e.g., "Lincoln") may push the matching hit to page 2+. The current implementation only queries page 0, causing silent spray chart omission. Second, no integration test covers the actual wiring between gc_uuid resolution, database persistence, and spray crawler invocation in `generate_report()`.

## Acceptance Criteria
- [ ] **AC-1**: Given POST /search returns 25 non-matching hits on page 0 and a matching hit on page 1, when `_resolve_gc_uuid` is called, then it returns the correct gc_uuid from page 1.
- [ ] **AC-2**: Given POST /search returns fewer than 25 hits on page 0 with no match, when `_resolve_gc_uuid` is called, then it returns None without requesting page 1 (short-circuit on partial page).
- [ ] **AC-3**: Given POST /search returns 25 non-matching hits on each of pages 0-4, when `_resolve_gc_uuid` is called, then it returns None after exactly 5 page requests (cap at 5 pages).
- [ ] **AC-4**: Given a team with `gc_uuid=NULL` in the database and POST /search returns a matching hit, when `generate_report()` runs, then the resolved gc_uuid is persisted to the `teams` table AND passed to `_crawl_and_load_spray`.
- [ ] **AC-5**: All existing `_resolve_gc_uuid` unit tests continue to pass (pagination is additive, not breaking).
- [ ] **AC-6**: The pagination loop uses the existing `start_at_page` parameter, per Technical Notes TN-3.

## Technical Approach
The pagination bug is in `_resolve_gc_uuid` in `src/reports/generator.py` (~line 409-457). The function currently makes a single call with `start_at_page=0`. It needs a loop that increments the page when the page is full (exactly 25 hits) and no match was found. The `_SEARCH_CONTENT_TYPE` and `params` structure already support `start_at_page`. The integration test (AC-4) requires mocking the HTTP layer (POST /search response) and `_crawl_and_load_spray`, while letting `_resolve_gc_uuid` execute with its real implementation against a mocked or in-memory database. See existing test `test_existing_gc_uuid_skips_search` for the DB setup and mocking pattern (that test mocks `_resolve_gc_uuid` because it tests the skip path; the new AC-4 test covers the execute path).
