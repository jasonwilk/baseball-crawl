# Data Model

The schema is defined in `migrations/001_initial_schema.sql` (base) with incremental migrations (`004_add_team_season_year.sql`, etc.). Key design decisions:

- **Programs**: Umbrella entity grouping teams under an organizational program (e.g., `lsb-hs` = Lincoln Standing Bear HS). Types: `hs`, `usssa`, `legion`.
- **Teams**: INTEGER PRIMARY KEY AUTOINCREMENT (`teams.id`). External GC identifiers live in dedicated UNIQUE columns: `gc_uuid` (authenticated API) and `public_id` (public URL slug). All FK references to teams use `teams(id)`. INTEGER PK applies to `teams` only -- programs, seasons, and players keep TEXT PKs.
- **Membership type**: `teams.membership_type` (`member` or `tracked`). Member teams are those the operator manages in GameChanger; tracked teams are opponents or other teams added for scouting.
- **Season year**: `teams.season_year INTEGER` -- the year this team belongs to, populated from GameChanger API (`season_year` on authenticated endpoints, `team_season.year` on public). NULL falls back to current calendar year in `get_team_year_map()`. Self-healed by pipeline on sync. **Footgun**: stub team auto-creation (in roster/season_stats loaders) must pass `season_year` to `ensure_team_row()` to avoid the current-year fallback producing wrong values for historical data.
- **Classification**: `teams.classification` column. Valid values: `varsity`, `jv`, `freshman`, `reserve` (HS); `8U`-`14U` (USSSA); `legion`. Division dropdown in the admin UI.
- **team_opponents**: Junction table linking member teams to their tracked opponents (`our_team_id` -> `opponent_team_id`), with `first_seen_year` for historical context. `is_hidden` serves dual purpose: admin dismissal ("no match" during resolution) AND pipeline filtering (hidden opponents are skipped by seeder/resolver).
- **TeamRef pattern**: Pipeline code uses a `TeamRef` dataclass (`id: int`, `gc_uuid: str | None`, `public_id: str | None`). `.id` for all DB operations; `.gc_uuid` / `.public_id` for API calls. Tracked teams may have `gc_uuid=None`.
- **Game ordering convention**: Every `ORDER BY game_date` MUST include `start_time` as a same-direction tiebreaker with `NULLS LAST` (e.g., `ORDER BY game_date DESC, start_time DESC NULLS LAST`). Doubleheaders, rescheduled games, and multi-site days all produce same-date collisions that `game_date` alone cannot resolve. For game boxscore pitching lines, additionally sort by `appearance_order ASC NULLS LAST` (starter first, then relievers in order).
- **Appearance order**: `player_game_pitching.appearance_order INTEGER` (migration 015). Value 1 = starter, 2+ = relievers in order of entry. NULL = unknown (not zero -- zero is not a valid appearance order). Populated by the game loader from boxscore JSON pitcher ordering (the `stats` array under each team's pitching group is ordered by appearance). Backfillable from cached boxscore JSON via `bb data backfill-appearance-order`. Scouting aggregation computes GS as `COUNT(*) WHERE appearance_order = 1` using a NULL-safe CASE expression (NULL `appearance_order` contributes 0, not NULL).
- **Game status values**: Completed games use `status = 'completed'` (not `'final'`). All queries across `src/` filter on `'completed'`; the `game_loader.py` inserts with this value. Do not use `'final'` in game queries.
- **Game time data ownership**: The schedule loader is authoritative for `game_date` and `start_time` -- it writes directly (no COALESCE). The game loader (game-summaries) lacks time data and uses `COALESCE(existing, new)` to preserve values already set by the schedule loader. Do not add COALESCE to schedule loader time columns; do not remove COALESCE from game loader time columns.
- **7-day rolling window**: For inclusive 7-calendar-day windows, use `date(ref, '-6 days')` (NOT `-7` which creates an 8-day window). Example: games from Mar 28 through Apr 3 inclusive = `WHERE game_date >= date('2026-04-03', '-6 days')`.
- **Enriched stat columns** (schema-ready, not yet populated): `game_stream_id` on games (links to game-streams endpoints like `GET /game-streams/{game_stream_id}/events`; NOT a path parameter for game-stream-processing endpoints -- `event_id` from game-summaries is the path parameter for both `GET /game-stream-processing/{event_id}/boxscore` and `GET /game-stream-processing/{event_id}/plays`; the game loader indexes summaries by both `event_id` and `game_stream_id` for robust file matching), `bats`/`throws` on players, home/away and vs-LHP/RHP split columns on season batting/pitching, `pitches`/`total_strikes` on game pitching, `batting_order`/`positions_played`/`is_primary` on game batting, `stat_completeness` tracking on all stat tables.
- **Plays and play events**: Two-table schema (`plays` + `play_events`, migration `009_plays_play_events.sql`). `plays` stores one row per plate appearance with batter/pitcher linkage, pitch count, outcome, and pre-computed boolean flags (`is_first_pitch_strike`, `is_qab`). `play_events` stores individual events within a PA (pitch results, baserunner events, substitutions). Idempotency keys: `UNIQUE(game_id, play_order)` on plays, `UNIQUE(play_id, event_order)` on play_events. **Whole-game idempotency**: before processing, check if any `plays` row exists for the `game_id` -- if yes, skip entirely. Never delete-and-reinsert (parent-child re-insertion problem: `INSERT OR IGNORE` on existing parent returns no `lastrowid` for children). Populated by: `bb data crawl --crawler plays` + `bb data load --loader plays`. 24 confirmed outcome types; 486 `final_details` patterns (regex-parsed, not enumerable). The plays endpoint (`GET /game-stream-processing/{event_id}/plays`) is NOT ownership-gated -- works for opponent games too (unlike season-stats).
- **Spray charts**: `spray_charts` table stores ball-in-play coordinate data (x/y, play type, result, fielder position). Populated by the spray chart pipeline (`bb data crawl --crawler spray-chart` + `bb data load --loader spray-chart`). Data quality: spray data is scorekeeper-dependent (~93% of games have offensive data, ~16% have defensive); `spray_chart_data` is `null` (not empty array) when the scorekeeper did not record. Display thresholds: 3 BIP minimum for per-player charts, 20 BIP for team aggregates.
- **Cross-perspective player UUIDs**: GameChanger returns different `player_id` UUIDs for the same human depending on which team's boxscore is viewed. This is a permanent API property, not a bug. When team A's boxscore lists team B's players, it uses different UUIDs (and often initial-only first names) than team B's own boxscore. All agents working with player data must account for this -- never assume `player_id` is globally unique per human. Same-team duplicates are detected by prefix-matching first names with matching last names on the same roster. See `.claude/rules/perspective-provenance.md` for the perspective tagging invariant and field classification table.
- **`gc_athlete_profile_id`**: The `players` table has a `gc_athlete_profile_id` column but it is never populated by any loader. It awaits E-104 probe results (athlete profile endpoint). Agents must not assume this column contains data.
- **Player dedup merge-every-run cycle**: The post-load dedup sweep in the scouting pipeline re-detects and re-merges duplicate player pairs on every run. Perspective tagging (E-220) reduces cross-perspective stub creation by isolating each perspective's data, but does not eliminate it entirely -- the plays loader still creates player stubs via `ensure_player_row()`. The dedup sweep remains necessary for both residual cross-perspective cleanup and genuine name-variant dedup (e.g., "O" vs "Oliver").
- **Plays pipeline dedup gap**: The plays pipeline (`bb data crawl --crawler plays` + `bb data load --loader plays`) runs independently from `bb data scout` and is not wired into the post-load dedup sweep. Perspective-scoped idempotency (`WHERE game_id = ? AND perspective_team_id = ?`) prevents cross-perspective play collisions -- each perspective's plays are isolated. Cross-perspective player stubs introduced by plays data are cleaned up by the next `bb data scout` run or by manual `bb data dedup-players --execute`.
- **`reports.our_team_id`**: Nullable column on `reports` referencing `teams(id)`. Identifies the LSB "us team" perspective the report was generated for. **Convention**: when `our_team_id IS NULL`, the report is opponent-only (no LSB-perspective sections render). When set, the report is LSB-aware -- currently surfaces the matchup strategy section (E-228); future LSB-perspective report features should gate on this column. This is the first us-team-aware reporting surface; downstream reporting work that needs LSB perspective should consume this column rather than re-deriving the perspective. **Resolution layer convention**: `our_team_id` is always `int` inside `generate_report()` -- the generator NEVER does string resolution. Resolution from a `public_id` slug or string id happens in CALLERS (e.g., `resolve_our_team()` in the `bb report generate` CLI, the admin handler dropdown). Add new callers by resolving to int before calling the generator; do not push slug-handling into `src/reports/generator.py`.

## Cleanup-Detection Mirror Invariant

When extending a cleanup helper's surface (adding a new table to a DELETE cascade, broadening a WHERE clause, adding a new row-deletion pass), you MUST audit every detection/gate query that mirrors the cleanup surface and update it in the same change. Detection queries include informed-consent gates, precondition checks, row-count probes, and any query that enumerates "what this delete will affect" to the operator.

**Why**: A perspective-aware cleanup gap let foreign-owned data bypass the informed-consent gate and silently delete — the canonical cascade grew to cover a new table but the gate query didn't, so a team with only that table's cross-perspective footprint walked through an empty warning screen and wiped data belonging to other perspectives.

**Concrete example**: `src/reports/generator.py::_delete_team_anchor_and_orphan_data` deletes five stat tables (`player_game_batting`, `player_game_pitching`, `spray_charts`, `plays`, `reconciliation_discrepancies`). The mirror query is `src/api/routes/admin.py::_get_delete_confirmation_data`'s `cross_persp_rows` UNION: it MUST enumerate the same five tables with the `team_id = T AND perspective_team_id != T` shape. When the cleanup adds a sixth table, the UNION must grow with it in the same commit.

**How to apply**: Any PR that modifies a DELETE statement, a cascade helper, or a cleanup pass in `src/reports/`, `src/db/`, or `src/api/` must grep for mirror queries (look for `COUNT(*)` / `SELECT ... WHERE team_id` / UNION shapes referencing the same table) and update them in lock-step. Code review checklist item: "Did the cleanup surface change? If yes, were all mirror queries updated?"
