# Codex Spec Review Findings: E-123

## 2. Dependency Correctness and Sequencing

- **P1** — **E-123 epic / E-123-01**: The epic currently treats the scouting-crawler auth fix as if it is already shipped, but `E-122` is still `READY` and `E-122-01` is still `TODO`. In `epics/E-123-full-cr-remediation/epic.md`, the Background says these findings are "NOT already addressed by" `E-122`, and the Success Criteria requires `CredentialExpiredError` during **any** crawler's `crawl_all()` loop to abort immediately. But `epics/E-122-e100-cr-remediation/epic.md` shows the scouting fix is not done yet. That leaves E-123 with an undeclared cross-epic dependency and success criteria it cannot satisfy on its own. Fix by either making `E-122-01` an explicit prerequisite/dependency or rewriting E-123's scope/success criteria to cover only the four non-scouting crawlers it actually owns.

## 1. Acceptance-Criteria Clarity and Testability

- **P1** — **E-123-06, AC-1**: AC-1 weakens the story below the epic's own technical decision. The story Description, Context, and epic TN-6 all say the successful crawl should transition from `"running"` to `"completed"`, but AC-1 also allows leaving some other status in place and merely changing `_is_scouted_recently()` to recognize it. That creates a direct inconsistency with the chosen lifecycle and would allow the misleading `"running"` state to survive. Tighten AC-1 to require `"completed"` after a successful crawl, with failed runs remaining non-fresh.

- **P1** — **E-123-06, AC-4**: The test surface is incomplete. The story's file list points to `tests/test_scouting_loader.py` (or a new file), but the current behavior being changed is already asserted in `tests/test_scouting_crawler.py`, which contains multiple tests that explicitly require the old `"running"` post-crawl status. As written, the story hides a required modification to an existing test file, so an implementer can follow the spec and still fail "All existing tests pass." Add `tests/test_scouting_crawler.py` to the file list and make the required test updates explicit.

- **P1** — **E-123-05, AC-1 / AC-3**: The story does not specify the output/display contract for the fix, and its file list omits a required UI file. AC-1 says recent games must show both batting and pitching data for the same game, but the Technical Approach leaves two incompatible implementations open: merge into one row or return two rows. The current template at `src/api/templates/dashboard/player_profile.html` renders either batting **or** pitching per row, and `tests/test_dashboard.py` currently asserts a single deduplicated row for a two-way game. The spec needs to choose one contract explicitly and include the template in "Files to Create or Modify"; otherwise the display requirement is not dispatchable without guesswork.

- **P2** — **E-123-09, AC-1**: The season-default fix is still underspecified and untested. AC-1 allows either "query the most recent season" or "derive from current date", but those rules diverge when seeded data and calendar year differ. The story also names only `tests/test_cli_status.py`, even though current repo tests for `get_team_batting_stats()` / `get_team_pitching_stats()` live in `tests/test_db.py` and never exercise the no-argument default path. Pick one default-resolution rule and require a DB-level regression test for calling these functions without `season_id`.
